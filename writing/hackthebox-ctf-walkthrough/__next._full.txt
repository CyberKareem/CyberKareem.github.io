Root's flag is leaked as a side-effect of that error.</p><h2 id="11-lessons-learned-and-key-techniques">11. Lessons Learned &amp; Key Techniques</h2><p>Vulnerability Summary<br>1. Blind LDAP Injection (Username Enum) - login.php - Username disclosure<br>2. Blind LDAP Injection (Attribute Exfil) - login.php - OTP seed disclosure<br>3. Second-Order LDAP Injection + Null Byte - login.php → page.php - Admin group bypass → RCE<br>4. Hardcoded Credentials - login.php - SSH access as ldapuser<br>5. 7za @listfile via Wildcard Expansion - honeypot.sh (cron) - Root flag disclosure</p><h2 id="key-lessons">Key Lessons</h2><p>1. re-enumerate dynamically<br>The token seed from every public writeup was wrong. The machine had been reset. Trusting static writeup values cost significant time. extract live data - especially secrets.</p><p>2. Encoding is a filter bypass technique, not just formatting<br>The server filtered *()&amp; raw but not URL-encoded. Understanding that urldecode() runs after the filter is the key insight. In real engagements, test both raw and encoded forms of injection payloads.</p><p>3. Second-order injection is about context switching<br>The injection bypassed the login check but still failed on page.php - because page.php ran a different LDAP query against the same stored username. Recognise that stored user-controlled data can be re-injected in a different context.</p><p>4. Null bytes still work in PHP 5.4 LDAP<br>Modern PHP (7+) sanitises null bytes in many contexts. PHP 5.4 does not. Version fingerprinting during recon directly informs which techniques are viable.</p><p>5. Developer comments are security findings<br>The HTML comment revealing "81-digit token in existing attribute" told us exactly what to look for in the LDAP injection. In real assessments, document every comment, error message, and debug artifact.</p><p>6. Know your tools deeply<br>stoken - use-time takes an absolute Unix timestamp. time.mktime() is timezone-aware. calendar.timegm() is not. A 5-hour timezone offset silently broke OTP generation. Reading manpages and understanding edge cases separates a stuck assessment from a completed one.</p><p>7. SELinux changes how you exploit, not whether you can<br>Standard reverse shells were blocked by SELinux's httpd_t context. Instead of fighting SELinux directly, we used the web application as the command channel - which is always permitted because it's what httpd_t is designed to allow.</p><p>8. Wildcard expansion is a systemic risk in cron jobs<br>7za&nbsp;… - * combined with @listfile support creates a privilege escalation path whenever a low-privileged user can write to the globbed directory.
