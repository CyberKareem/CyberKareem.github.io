<h2 id="summary">Summary</h2>
<h3 id="what-is-this-box">What is this box?</h3>
<p>Eighteen is a <strong>medium-difficulty Windows Active Directory machine</strong> set in a corporate environment. It simulates a domain controller (<code>DC01</code>) running a financial planning web application backed by a Microsoft SQL Server 2022 database. The machine is notable for running <strong>Windows Server 2025</strong>, which makes it vulnerable to a brand new critical AD vulnerability - <strong>BadSuccessor (CVE-2025-53779)</strong> - discovered by Akamai in 2025.</p>
<h3 id="what-does-this-box-teach">What does this box teach?</h3>
<p>This box covers a complete real-world attack chain across three distinct phases:</p>
<ul>
<li><strong>Database exploitation</strong> : abusing SQL Server privilege misconfiguration to extract sensitive data</li>
<li><strong>Credential attacks</strong> : hash cracking and password spraying to gain initial domain access</li>
<li><strong>Active Directory privilege escalation</strong> : exploiting a recent Windows Server 2025 vulnerability to fully compromise the domain</li>
</ul>
<h3 id="the-attack-story">The Attack Story</h3>
<p><strong>Phase 1 - Foothold via MSSQL</strong></p>
<p>We are given initial credentials (<code>kevin:iNa2we6haRj2gaw!</code>) and discover a SQL Server on port 1433. Kevin's account is restricted, but SQL Server's impersonation feature lets us switch context to <code>appdev</code> - a more privileged user. As appdev, we access the <code>financial_planner</code> database and dump the users table, which contains the admin account's password hash in PBKDF2 format. We convert it to hashcat-compatible format and crack it in seconds: <code>admin:iloveyou1</code>.</p>
<p><strong>Phase 2 - Domain User Access (User Flag)</strong></p>
<p>With a cracked password in hand, we enumerate every domain account by performing RID brute force through the MSSQL channel - no LDAP needed. We get a full list of domain users and spray <code>iloveyou1</code> across all of them via WinRM. The user <code>adam.scott</code> reuses this password. We log in with evil-winrm and capture the user flag.</p>
<p><strong>Phase 3 - Domain Compromise via BadSuccessor (Root Flag)</strong></p>
<p>As <code>adam.scott</code> we discover he has <code>CreateChild</code> rights on an Organizational Unit in AD. On a Windows Server 2025 domain controller, this is enough to own the entire domain. We use the BadSuccessor exploit to create a <strong>Delegated Managed Service Account (dMSA)</strong> - a new account type in Server 2025 - and configure it as the "successor" to the Administrator account. We then use impacket's getST tool with the <code>-dmsa</code> flag to request a Kerberos ticket for our dMSA. The domain controller, following its credential handoff logic, includes the <strong>Administrator's NTLM hash</strong> in the ticket response as a "previous key". We use this hash in a Pass-the-Hash attack via evil-winrm and log in as Administrator, capturing the root flag.</p>
<h3 id="why-is-badsuccessor-so-powerful">Why is BadSuccessor so powerful?</h3>
<p>The vulnerability abuses a legitimate Windows Server 2025 feature designed to help organisations migrate service accounts. When a dMSA "succeeds" another account, the DC hands over the predecessor's credentials as part of the transition. The flaw is that <strong>any user with CreateChild on any OU</strong> can trigger this process for any account in the domain - including Domain Admins. <code>CreateChild</code> is a low-sensitivity permission that is commonly delegated, making this a trivially exploitable path to full domain compromise on any Server 2025 environment.</p>

<pre><code>Weak SQL privilege model        -&gt; credential extraction

Overly permissive OU delegation -&gt; full domain compromise (BadSuccessor)
</code></pre>

<h2 id="step-3-hash-extraction-and-cracking">Step 3: Hash Extraction &amp; Cracking</h2>

<p>The hash is in Django/Werkzeug PBKDF2 format. Hashcat can't read this directly - we need to convert it to hashcat's format first.</p>
<h3 id="why-we-need-the-converter">Why we need the converter</h3>
<p>PBKDF2 hashes store the salt and hash in base64, but the raw hash from the DB uses hex encoding. The script converts between these formats.</p>
<h3 id="pbkdf2-to-hashcat-py">Pbkdf2-to-hashcat.py</h3>
<pre><code class="language-python" data-language="python">#!/usr/bin/env python3
import base64
import sys

h = ''.join(sys.argv[1:])
taa = h.split(':')[:-1]
start = len(':'.join(taa) + ':')

iterations = h[start:].split('$')[0]
salt = h[start:].split('$')[1]
sha = h[start:].split('$')[2]

salt_base64 = base64.b64encode(salt.encode()).decode()
hash_bytes = bytes.fromhex(sha)
hash_base64 = base64.b64encode(hash_bytes).decode()

print(f'{taa[1]}:{iterations}:{salt_base64}:{hash_base64}')
</code></pre>
<h3 id="commands-3">Commands</h3>
<pre><code class="language-bash" data-language="bash"># Convert the hash format
python3 pbkdf2-to-hashcat.py 'pbkdf2:sha256:600000$&lt;salt&gt;$&lt;hash&gt;' &gt; hsh

# Crack with hashcat (mode 10900 = PBKDF2-HMAC-SHA256)
hashcat -m 10900 hsh /usr/share/seclists/Passwords/LeakedDatabases/rockyou.txt

# Show result after cracking
hashcat hsh --show
</code></pre>

<pre><code>admin:iloveyou1
</code></pre>

<h2 id="step-6-badsuccessor-cve-2025-53779">Step 6: BadSuccessor (CVE-2025-53779)</h2>
<h3 id="what-is-badsuccessor">What is BadSuccessor?</h3>
<p>A critical Active Directory privilege escalation vulnerability discovered by Akamai (2025). It abuses <strong>Delegated Managed Service Accounts (dMSA)</strong> - a new account type introduced in Windows Server 2025.</p>
<p><strong>The vulnerability:</strong> If you have <code>CreateChild</code> rights on any OU, you can create a dMSA and set its <code>msDS-ManagedAccountPrecededByLink</code> attribute to point to ANY user (e.g., Administrator). The DC then treats your dMSA as the "successor" of that user, granting it access to the target's credentials.</p>
<p><strong>Requirements:</strong></p>
<ul>
<li>Windows Server 2025 DC (this box runs Server 2025)</li>
<li><code>CreateChild</code> rights on any OU (adam.scott has this on <code>OU=Staff</code>)</li>
</ul>

<ol>
<li>Upload the BadSuccessor PowerShell module via evil-winrm</li>
<li>Create a fake dMSA called <code>nory_dmsa</code> that "succeeded" the Administrator account</li>
<li>Grant adam.scott full control over the dMSA</li>
</ol>
<h3 id="commands-in-evil-winrm">Commands (in evil-winrm)</h3>
<pre><code class="language-powershell" data-language="powershell"># Upload the BadSuccessor module
upload /home/kali/BadSuccessor.ps1 C:\Users\adam.scott\Documents\BadSuccessor.ps1

# Import the module
import-module C:\Users\adam.scott\Documents\BadSuccessor.ps1

# Create the malicious dMSA
# -Path: OU where we have CreateChild rights
# -Name: name of our fake dMSA
# -DelegatedAdmin: who gets control of it (us)
# -DelegateTarget: whose credentials we want (Administrator)
BadSuccessor -mode exploit `
  -Path "OU=Staff,DC=eighteen,DC=htb" `
  -Name "nory_dmsa" `
  -DelegatedAdmin "adam.scott" `
  -DelegateTarget "Administrator" `
  -domain "eighteen.htb"
</code></pre>

<h3 id="what-happened-under-the-hood">What happened under the hood</h3>

<h2 id="key-techniques-and-lessons-learned">Key Techniques &amp; Lessons Learned</h2>
<h3 id="1-mssql-impersonation">1. MSSQL Impersonation</h3>
<p>When a SQL login has <code>IMPERSONATE</code> permission on another login, you can fully switch context using <code>EXECUTE AS LOGIN = 'target'</code>. This lets you access databases and objects that the original account couldn't reach. Check with <code>nxc -M mssql_priv</code>.</p>
<h3 id="2-pbkdf2-hash-format-conversion">2. PBKDF2 Hash Format Conversion</h3>
<p>Django/Werkzeug PBKDF2 hashes look like <code>pbkdf2:sha256:600000$salt$hexhash</code>. Hashcat mode 10900 needs <code>sha256:iterations:base64salt:base64hash</code>. A small Python script handles the conversion.</p>
<h3 id="3-rid-brute-force-via-mssql">3. RID Brute Force via MSSQL</h3>
<p>MSSQL's <code>xp_dirtree</code> and authentication mechanisms can be abused to enumerate AD users without LDAP access. NetExec's <code>--rid-brute</code> automates this through the MSSQL channel.</p>
<h3 id="4-password-reuse-pattern">4. Password Reuse Pattern</h3>
<p>A single cracked password (<code>iloveyou1</code>) from the web app database reused by a domain user (<code>adam.scott</code>) is the pivot from internal app access to domain foothold - extremely common in real environments.</p>
<h3 id="5-badsuccessor-cve-2025-53779-dmsa-abuse">5. BadSuccessor (CVE-2025-53779): dMSA Abuse</h3>
<p><strong>Core mechanic:</strong> dMSAs can designate a predecessor account. When the dMSA requests its Kerberos credentials, the DC includes the predecessor's previous keys in the response. This is by design for migration purposes - but it becomes a critical vulnerability when an unprivileged user can create a dMSA pointing to any account.</p>
<p><strong>Requirements to exploit:</strong></p>
<ul>
<li>DC must run Windows Server 2025</li>
<li>Attacker needs <code>CreateChild</code> rights on any OU</li>
</ul>
<p><strong>Why it's dangerous:</strong> <code>CreateChild</code> on an OU is not a highly sensitive permission and is often delegated broadly. Any user with this right can fully compromise the domain on Server 2025 DCs.</p>
<h3 id="6-chisel-reverse-socks-for-kerberos">6. Chisel Reverse SOCKS for Kerberos</h3>
<p>Kerberos (port 88) is often not exposed externally. Running a reverse SOCKS proxy through an already-compromised WinRM session lets you tunnel all Kerberos traffic through the victim machine. This is more reliable than sshuttle for Windows targets.</p>
<h3 id="7-getst-dmsa-self-flag">7. getST -dmsa -self Flag</h3>
<p>The <code>-dmsa</code> flag in impacket's getST tells it to perform the dMSA-specific S4U2self flow, which returns the predecessor's credentials. The <code>-self</code> flag requests the ticket for the account itself (no service specified). The NTLM hash appears in plaintext in the output - no secretsdump required.</p>
<h3 id="8-clock-skew-with-kerberos">8. Clock Skew with Kerberos</h3>
<p>Kerberos requires clocks to be within 5 minutes of the DC. sync before running Kerberos tools:</p>
<pre><code class="language-bash" data-language="bash">sudo timedatectl set-ntp false
sudo timedatectl set-time "$(date -d "$(curl -s -I http://&lt;DC_IP&gt; | grep -i '^Date:' | cut -d' ' -f2-)" '+%Y-%m-%d %H:%M:%S')"
</code></pre>
<p>If that fails (NTP locked), use <code>faketime</code> prefix on the command instead.</p>
