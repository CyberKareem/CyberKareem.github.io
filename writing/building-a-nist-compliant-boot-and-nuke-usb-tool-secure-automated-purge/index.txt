<p><em>This is Part 2 of my Secure Data Erasure Tools series. </em><a href="/writing/introducing-the-bitlocker-cryptographic-erase-utility-secure-data-destruction-made-simple/"><em>Part 1: Introducing the BitLocker Cryptographic Erase Utility: Secure Data Destruction Made Simple</em></a></p><p><em>How I evolved from a BitLocker-specific solution to creating a universal disk sanitization tool that automatically detects and attempts device-specific sanitization commands across detected internal drives</em></p><figure><img src="/writing/media/0f26363f1335d08aaef1c7cb.webp" alt="Secure Automated Purge: Building a Bootable Disk-Sanitization Utility, figure 1" width="1200" height="800" loading="lazy" decoding="async"></figure><h2 id="the-problem-secure-data-destruction-at-scale">The Problem: Secure Data Destruction at&nbsp;Scale</h2><p>Last month, I was tasked to arrange a solution prototyping a repeatable workstation-sanitization workflow informed by NIST SP 800-88 Rev.1 before disposal. The options were:</p><ol><li><strong>Manual commands: </strong>Boot each system with Linux, run different commands for different drive types. Time-consuming and error prone.</li><li><strong>Surface Data Eraser </strong>limited for Microsoft Surface machines, and<strong> Lenovo ThinkShield Secure Wipe</strong> limited for Lenovo new Gen. machines.</li><li><strong>Enterprise tools: </strong>Expensive licenses, often require network infrastructure.</li><li><strong>Physical destruction: </strong>Wasteful when drives could be reused.</li></ol><p>Initially, I tackled this by creating a <a href="/writing/introducing-the-bitlocker-cryptographic-erase-utility-secure-data-destruction-made-simple/">PowerShell-based BitLocker Cryptographic Erase tool</a>. It worked brilliantly for Windows systems with BitLocker enabled, providing a fast cryptographic-erase path for supported BitLocker systems. But I quickly realized its limitations which one of them it has only worked on Windows machines with BitLocker On.</p><p>This led me to think bigger. What I really needed was a universal tool that would:</p><ul><li>Boot from USB and run automatically</li><li>Detect all drive types (NVMe, SATA SSD, HDD)</li><li>Select a device-specific command path for each detected drive</li><li>Map the selected methods to the Purge concepts described in the then-current NIST SP 800-88 Rev.1</li><li>Provide audit logs for compliance documentation</li><li>Be completely free and open source</li></ul><p>So, I built one.</p><h2 id="introducing-secure-automated-purge-usb">Introducing Secure Automated Purge&nbsp;USB</h2><p>The <strong>Secure Automated Purge USB Utility</strong> is a bootable Linux ISO that automates device-specific sanitization commands from a bootable Linux environment. Just boot from USB, confirm the operation, and walk away. When you come back, the attempted device commands have finished and the system has shut down; the result still requires independent verification.</p><p> <strong>GitHub</strong>: <a href="https://github.com/CyberKareem/SecureAutomatedPurge-USB" rel="nofollow noopener noreferrer" target="_blank">github.com/CyberKareem/SecureAutomatedPurge-USB<span class="sr-only"> opens in a new tab</span></a></p><h2 id="main-features">Main Features</h2><ul><li><strong>Fully Automated: </strong>Automated prompts are provided, but destructive use still requires trained supervision</li><li><strong>Standards-aware prototype: </strong>Methods selected from the then-current Rev.1 guidance; current, device-specific validation is required</li><li><strong>Multi-Drive Support: </strong>NVMe, SATA SSD, and HDD</li><li><strong>Drive-Type Routing: </strong>Selects a command path by detected drive type; suitability requires validation</li><li><strong>Audit Logs: </strong>Detailed logs for compliance documentation</li><li><strong>Safety Controls Implemented: </strong>Requires explicit confirmation, excludes USB drives</li></ul><h2 id="the-evolution-from-bitlocker-to-universal-purge">The Evolution: From BitLocker to Universal Purge</h2><p>This project actually started with my <a href="https://github.com/CyberKareem/BitLocker-CryptoErase" rel="nofollow noopener noreferrer" target="_blank">BitLocker Cryptographic Erase<span class="sr-only"> opens in a new tab</span></a> tool. That PowerShell utility uses BitLocker’s encryption to achieve a fast cryptographic-erase path on supported BitLocker systems. It’s incredibly fast.</p><p>But real-world IT is messy. In 50 workstations, for example:</p><ul><li>30 had BitLocker enabled (the BitLocker tool was observed to complete its command path)</li><li>10 were Linux workstations</li><li>5 had corrupted Windows installations</li><li>5 had BitLocker disabled</li></ul><p>This led to an important realization: <strong>I needed both tools in my arsenal</strong>.</p><figure><img src="/writing/media/1f935c1805ee257c2cd3b7c4.webp" alt="" width="1200" height="800" loading="lazy" decoding="async"><figcaption>Info graph that shows BitLocker vs Boot-USB solution Comparison</figcaption></figure><p><strong>Windows with BitLocker enabled</strong><br>&nbsp;<strong>Best Tool:</strong> <em>BitLocker Crypto Erase</em><br>&nbsp;<strong>Why:</strong> Instant and can be performed remotely</p><p><strong>Corrupted or non-booting OS</strong><br>&nbsp;<strong>Best Tool:</strong> <em>Secure Automated Purge</em><br>&nbsp;<strong>Why:</strong> Doesn’t require a functioning operating system</p><p><strong>Linux or Unix systems</strong><br>&nbsp;<strong>Best Tool:</strong> <em>Secure Automated Purge</em><br>&nbsp;<strong>Why:</strong> BitLocker is not available on these platforms</p><p><strong>Mixed environment (Windows + Linux)</strong><br>&nbsp;<strong>Best Tool:</strong> <em>Secure Automated Purge</em><br>&nbsp;<strong>Why:</strong> A universal solution for all OS types</p><p><strong>Remote management needed</strong><br>&nbsp;<strong>Best Tool:</strong> <em>BitLocker Crypto Erase</em><br>&nbsp;<strong>Why:</strong> Can be triggered via PowerShell remoting</p><p><strong>Unknown drive encryption status</strong><br>&nbsp;<strong>Best Tool:</strong> <em>Secure Automated Purge</em><br>&nbsp;<strong>Why:</strong> Does not depend on BitLocker state; device behavior still requires validation</p><p>And both tools were designed around NIST SP 800-88 Rev.1 concepts; that design intent is not independent verification, they just approach it differently:</p><ul><li><strong>BitLocker tool</strong>: Uses existing encryption for cryptographic erase</li><li><strong>This tool</strong>: Uses drive controller commands or overwriting</li></ul><p>Together, they explore several common device scenarios; they do not establish complete sanitization coverage.</p><blockquote><em>Starting with a specialized tool (BitLocker) taught me the value of cryptographic erasure. But real-world complexity demanded a universal solution. Sometimes the best toolkit isn’t one perfect tool, but multiple tools that excel in different scenarios.</em></blockquote><h2 id="the-technical-challenge">The Technical Challenge</h2><p>Creating this tool involved solving several interesting technical challenges:</p><h2 id="1-automatic-drive-detection-and-classification">1. Automatic Drive Detection and Classification</h2><p>The script needs to differentiate between:</p><ul><li>NVMe SSDs (require <code>nvme-cli</code> tools)</li><li>SATA SSDs (support ATA Secure Erase)</li><li>Traditional HDDs (require overwriting)</li><li>USB drives (must be excluded for safety)</li></ul><p>Here’s how I implemented it:</p><pre><code data-language="text">identify_drive_type() {    local drive=$1    local drive_name=$(basename "$drive")        # Check if USB    if [[ $(readlink -f /sys/block/"$drive_name") =~ usb ]]; then        echo "usb"    # Check if NVMe    elif [[ "$drive" =~ nvme ]]; then        echo "nvme"    # Check if SSD    elif [[ $(cat /sys/block/"$drive_name"/queue/rotational 2&gt;/dev/null) == "0" ]]; then        echo "ssd"    else        echo "hdd"    fi}</code></pre><h2 id="2-sanitization-methods-implemented-in-the-prototype">2. Sanitization Methods Implemented in the Prototype</h2><p>NIST SP 800-88 Rev. 1 defines different sanitization levels. the prototype attempted to implement methods it mapped to the Rev.1 “Purge” category:</p><h3 id="for-nvme-drives-cryptographic-erase">For NVMe Drives: Cryptographic Erase</h3><pre><code data-language="text">nvme format "$drive" --ses=2nvme format "$drive" --ses=1nvme sanitize "$drive" --sanact=2</code></pre><p>This asks a supporting drive controller to perform cryptographic erase. The actual result depends on the controller, firmware, command support, and independent verification.</p><p>And it will fall back to block erase using NVMe format command, and it will also use the crypto erase sanitize command as the last resort, attempting an alternate controller command path.</p><h3 id="for-sata-ssds-ata-secure-erase">For SATA SSDs: ATA Secure&nbsp;Erase</h3><pre><code data-language="text">hdparm --user-master u --security-set-pass p "$drive"hdparm --user-master u --security-erase p "$drive"hdparm --sanitize-crypto-scramble "$drive"hdparm --sanitize-block-erase "$drive"</code></pre><p>This triggers the drive’s built-in secure erase function which is the most compatible, and the controller executes internally. Then tries the newer sanitize commands, as alternate controller command paths.</p><h3 id="for-hdds-3-pass-overwrite">For HDDs: 3-Pass Overwrite</h3><pre><code data-language="text">shred -v -n 3 "$drive"</code></pre><p>This overwrites the entire drive three times: random data, random data, zeros.</p><h2 id="3-building-a-bootable-environment">3. Building a Bootable Environment</h2><p>I used <a href="https://www.debian.org/download" rel="nofollow noopener noreferrer" target="_blank"><strong><em>Debian 12 netinst Live Build</em></strong><span class="sr-only"> opens in a new tab</span></a> to create a minimal Linux environment that:</p><ul><li>Boots entirely into RAM (no persistence)</li><li>Includes only necessary tools (<code>nvme-cli</code>, <code>hdparm</code>, <code>shred</code>)</li><li>Automatically starts the purge script</li><li>Has no network connectivity (security feature)</li></ul><p>The key was configuring <code>rc.local</code> to launch immediately after boot:</p><pre><code data-language="text">#!/bin/shexec &lt; /dev/console &gt; /dev/console 2&gt;&amp;1clear/usr/local/bin/secure_purge.shexit 0</code></pre><h2 id="4-safety-mechanisms">4. Safety Mechanisms</h2><p>Because the tool is destructive, The tool includes multiple safety features:</p><ul><li><strong>USB drives excluded: </strong>Prevents accidentally wiping the boot drive</li><li><strong>Explicit confirmation: </strong>Must type (all caps)<code>ERASE ALL DATA</code> to proceed</li><li><strong>Detailed drive listing: </strong>Shows model and serial numbers</li><li><strong>No network access: </strong>Prevents remote triggering</li><li><strong>Auto-shutdown: </strong>Ensures clean completion</li></ul><h2 id="limited-test-demonstration">Limited Drive Controllers Are&nbsp;Smart</h2><p>Modern drives have sophisticated controllers that can perform secure erasure far faster than any software-based overwriting. Using these built-in capabilities is key to efficient sanitization.</p><h2 id="2-one-size-doesn-t-fit-all">2. One Size Doesn’t Fit&nbsp;All</h2><p>Different drive types require different approaches. NVMe drives support cryptographic erase, SATA drives have ATA Secure Erase, and only HDDs require actual overwriting.</p><h2 id="3-automation-reduces-errors">3. Automation Reduces&nbsp;Errors</h2><p>By automating the entire process, we eliminate the risk of human error, wrong commands, skipped drives, or incomplete erasure.</p><h2 id="4-compliance-requires-documentation">4. Compliance Requires Documentation</h2><p>The tool generates detailed logs showing:</p><ul><li>Drive serial numbers</li><li>Erasure methods used</li><li>Completion timestamps</li><li>Verification results</li></ul><h2 id="building-your-own">Building Your&nbsp;Own</h2><p>A v1.0.0 ISO release is available on GitHub for controlled lab inspection; do not use it on operational media without a current source review and hardware validation: <a href="https://github.com/CyberKareem/SecureAutomatedPurge-USB/releases/tag/v1.0.0" rel="nofollow noopener noreferrer" target="_blank"><em>SecureAutomatedPurge USB utility</em><span class="sr-only"> opens in a new tab</span></a></p><p>The tool is completely open source. Reuse or disposal still requires a verified result under an approved procedure.</p><h2 id="compliance-requirements">Compliance Requirements</h2><p>Many regulations (HIPAA, GDPR, PCI-DSS) require secure data disposal. The prototype attempts sanitization commands and produces logs; neither the command result nor the logs alone establish regulatory compliance.</p><h2 id="donation-programs">Donation Programs</h2><p>Refurbishing computers for charity? Ensure previous data is completely removed first.</p><h2 id="possible-next-steps">Possible next steps</h2><p>I’m thinking of adding:</p><ul><li>Network boot support (PXE)</li><li>Parallel drive processing</li><li>Cloud logging for centralized reporting</li><li>GUI version for less technical users</li><li>Support for external drives (with safety controls)</li></ul><h2 id="what-i-learned">What I learned</h2><p>The work produced two experimental media-sanitization paths: a BitLocker workflow for supported Windows systems and a bootable Linux prototype for other device types.</p><ul><li><strong>BitLocker Crypto Erase</strong>: The power of using existing encryption</li><li><strong>Secure Automated Purge</strong>: The importance of universal solutions</li></ul><p>Together, they document two experimental approaches to media sanitization. The BitLocker tool handles the “easy” cases where Windows and encryption are already in place. This bootable prototype explores non-BitLocker device paths.</p><p>The combination of modern drive capabilities and NIST guidelines allowed me to create something that’s fast in the limited hardware tests documented here.
