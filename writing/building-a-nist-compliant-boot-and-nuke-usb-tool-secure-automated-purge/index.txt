12:T4d84,<p><em>This is Part 2 of my Secure Data Erasure Tools series. </em><a href="/writing/introducing-the-bitlocker-cryptographic-erase-utility-secure-data-destruction-made-simple/"><em>Part 1: Introducing the BitLocker Cryptographic Erase Utility: Secure Data Destruction Made Simple</em></a></p><p><em>How I evolved from a BitLocker-specific solution to creating a universal disk sanitization tool that automatically detects and securely erases all internal drives</em></p><figure><img src="/writing/media/0f26363f1335d08aaef1c7cb.webp" alt="Building a NIST-Compliant Boot-and-Nuke USB Tool: Secure Automated Purge, figure 1" width="1200" height="800" loading="lazy" decoding="async" /></figure><h2 id="the-problem-secure-data-destruction-at-scale">The Problem: Secure Data Destruction at Scale</h2><p>Last month, I was tasked to arrange a solution securely wiping workstations according to NIST SP 800–88 standards (Purge-level) before disposal. The options were:</p><ol><li><strong>Manual commands: </strong>Boot each system with Linux, run different commands for different drive types. Time-consuming and error prone.</li><li><strong>Surface Data Eraser </strong>limited for Microsoft Surface machines, and<strong> Lenovo ThinkShield Secure Wipe</strong> limited for Lenovo new Gen. machines.</li><li><strong>Enterprise tools: </strong>Expensive licenses, often require network infrastructure.</li><li><strong>Physical destruction: </strong>Wasteful when drives could be reused.</li></ol><p>Initially, I tackled this by creating a <a href="/writing/introducing-the-bitlocker-cryptographic-erase-utility-secure-data-destruction-made-simple/">PowerShell-based BitLocker Cryptographic Erase tool</a>. It worked brilliantly for Windows systems with BitLocker enabled, achieving NIST-compliant crypto erase in seconds. But I quickly realized its limitations which one of them it has only worked on Windows machines with BitLocker On.</p><p>This led me to think bigger. What I really needed was a universal tool that would:</p><ul><li>Boot from USB and run automatically</li><li>Detect all drive types (NVMe, SATA SSD, HDD)</li><li>Use the optimal erasure method for each drive</li><li>Meet Purge-level NIST 800–88 compliance requirements</li><li>Provide audit logs for compliance documentation</li><li>Be completely free and open source</li></ul><p>So, I built one.</p><h2 id="introducing-secure-automated-purge-usb">Introducing Secure Automated Purge USB</h2><p>The <strong>Secure Automated Purge USB Utility</strong> is a bootable Linux ISO that performs NIST-compliant data sanitization automatically. Just boot from USB, confirm the operation, and walk away. When you come back, all internal drives are securely erased and the system has shut down.</p><p>🔗 <strong>GitHub</strong>: <a href="https://github.com/CyberKareem/SecureAutomatedPurge-USB" rel="nofollow noopener noreferrer" target="_blank">github.com/CyberKareem/SecureAutomatedPurge-USB</a></p><h2 id="main-features">Main Features</h2><ul><li><strong>Fully Automated: </strong>No technical knowledge required</li><li><strong>NIST 800–88 Compliant: </strong>Meets federal standards for data sanitization (Purge-level)</li><li><strong>Multi-Drive Support: </strong>NVMe, SATA SSD, and HDD</li><li><strong>Optimal Methods: </strong>Uses the fastest secure method for each drive type</li><li><strong>Audit Logs: </strong>Detailed logs for compliance documentation</li><li><strong>Safety First: </strong>Requires explicit confirmation, excludes USB drives</li></ul><h2 id="the-evolution-from-bitlocker-to-universal-purge">The Evolution: From BitLocker to Universal Purge</h2><p>This project actually started with my <a href="https://github.com/CyberKareem/BitLocker-CryptoErase" rel="nofollow noopener noreferrer" target="_blank">BitLocker Cryptographic Erase</a> tool. That PowerShell utility leverages BitLocker’s encryption to achieve instant NIST-compliant sanitization (Purge-level). It’s incredibly fast.</p><p>But real-world IT is messy. In 50 workstations, for example:</p><ul><li>30 had BitLocker enabled (BitLocker tool worked perfectly)</li><li>10 were Linux workstations</li><li>5 had corrupted Windows installations</li><li>5 had BitLocker disabled</li></ul><p>This led to an important realization: <strong>I needed both tools in my arsenal</strong>.</p><figure><img src="/writing/media/1f935c1805ee257c2cd3b7c4.webp" alt="" width="1200" height="800" loading="lazy" decoding="async" /><figcaption>Info graph that shows BitLocker vs Boot-USB solution Comparison</figcaption></figure><p><strong>▪️ Windows with BitLocker enabled</strong><br /> <strong>Best Tool:</strong> <em>BitLocker Crypto Erase</em><br /> <strong>Why:</strong> Instant and can be performed remotely</p><p><strong>▪️ Corrupted or non-booting OS</strong><br /> <strong>Best Tool:</strong> <em>Secure Automated Purge</em><br /> <strong>Why:</strong> Doesn’t require a functioning operating system</p><p><strong>▪️ Linux or Unix systems</strong><br /> <strong>Best Tool:</strong> <em>Secure Automated Purge</em><br /> <strong>Why:</strong> BitLocker is not available on these platforms</p><p><strong>▪️ Mixed environment (Windows + Linux)</strong><br /> <strong>Best Tool:</strong> <em>Secure Automated Purge</em><br /> <strong>Why:</strong> A universal solution for all OS types</p><p><strong>▪️ Remote management needed</strong><br /> <strong>Best Tool:</strong> <em>BitLocker Crypto Erase</em><br /> <strong>Why:</strong> Can be triggered via PowerShell remoting</p><p><strong>▪️ Unknown drive encryption status</strong><br /> <strong>Best Tool:</strong> <em>Secure Automated Purge</em><br /> <strong>Why:</strong> Works regardless of encryption method</p><p>And both tools follow NIST 800–88 (Purge-level) guidelines, they just approach it differently:</p><ul><li><strong>BitLocker tool</strong>: Uses existing encryption for cryptographic erase</li><li><strong>This tool</strong>: Uses drive controller commands or overwriting</li></ul><p>Together, they cover virtually every secure erasure scenario an IT professional might encounter.</p><blockquote><em>Starting with a specialized tool (BitLocker) taught me the value of cryptographic erasure. But real-world complexity demanded a universal solution. Sometimes the best toolkit isn’t one perfect tool, but multiple tools that excel in different scenarios.</em></blockquote><h2 id="the-technical-challenge">The Technical Challenge</h2><p>Creating this tool involved solving several interesting technical challenges:</p><h2 id="1-automatic-drive-detection-and-classification">1. Automatic Drive Detection and Classification</h2><p>The script needs to differentiate between:</p><ul><li>NVMe SSDs (require <code>nvme-cli</code> tools)</li><li>SATA SSDs (support ATA Secure Erase)</li><li>Traditional HDDs (require overwriting)</li><li>USB drives (must be excluded for safety)</li></ul><p>Here’s how I implemented it:</p><pre><code data-language="text">identify_drive_type() {    local drive=$1    local drive_name=$(basename "$drive")        # Check if USB    if [[ $(readlink -f /sys/block/"$drive_name") =~ usb ]]; then        echo "usb"    # Check if NVMe    elif [[ "$drive" =~ nvme ]]; then        echo "nvme"    # Check if SSD    elif [[ $(cat /sys/block/"$drive_name"/queue/rotational 2&gt;/dev/null) == "0" ]]; then        echo "ssd"    else        echo "hdd"    fi}</code></pre><h2 id="2-nist-compliant-erasure-methods">2. NIST-Compliant Erasure Methods</h2><p>NIST SP 800–88 Rev. 1 defines different sanitization levels. I implemented “Purge” level:</p><h3 id="for-nvme-drives-cryptographic-erase">For NVMe Drives: Cryptographic Erase</h3><pre><code data-language="text">nvme format "$drive" --ses=2nvme format "$drive" --ses=1nvme sanitize "$drive" --sanact=2</code></pre><p>This destroys the encryption keys in the drive controller, rendering all data unrecoverable in seconds.</p><p>And it will fall back to block erase using NVMe format command, and it will also use the crypto erase sanitize command as the last resort, achieving comprehensive data sanitization.</p><h3 id="for-sata-ssds-ata-secure-erase">For SATA SSDs: ATA Secure Erase</h3><pre><code data-language="text">hdparm --user-master u --security-set-pass p "$drive"hdparm --user-master u --security-erase p "$drive"hdparm --sanitize-crypto-scramble "$drive"hdparm --sanitize-block-erase "$drive"</code></pre><p>This triggers the drive’s built-in secure erase function which is the most compatible, and the controller executes internally. Then tries the newer sanitize commands, also achieving comprehensive approach.</p><h3 id="for-hdds-3-pass-overwrite">For HDDs: 3-Pass Overwrite</h3><pre><code data-language="text">shred -v -n 3 "$drive"</code></pre><p>This overwrites the entire drive three times: random data, random data, zeros.</p><h2 id="3-building-a-bootable-environment">3. Building a Bootable Environment</h2><p>I used <a href="https://www.debian.org/download" rel="nofollow noopener noreferrer" target="_blank"><strong><em>Debian 12 netinst Live Build</em></strong></a> to create a minimal Linux environment that:</p><ul><li>Boots entirely into RAM (no persistence)</li><li>Includes only necessary tools (<code>nvme-cli</code>, <code>hdparm</code>, <code>shred</code>)</li><li>Automatically starts the purge script</li><li>Has no network connectivity (security feature)</li></ul><p>The key was configuring <code>rc.local</code> to launch immediately after boot:</p><pre><code data-language="text">#!/bin/shexec &lt; /dev/console &gt; /dev/console 2&gt;&amp;1clear/usr/local/bin/secure_purge.shexit 0</code></pre><h2 id="4-safety-mechanisms">4. Safety Mechanisms</h2><p>With great power comes great responsibility. The tool includes multiple safety features:</p><ul><li><strong>USB drives excluded: </strong>Prevents accidentally wiping the boot drive</li><li><strong>Explicit confirmation: </strong>Must type (all caps)<code>ERASE ALL DATA</code> to proceed</li><li><strong>Detailed drive listing: </strong>Shows model and serial numbers</li><li><strong>No network access: </strong>Prevents remote triggering</li><li><strong>Auto-shutdown: </strong>Ensures clean completion</li></ul><h2 id="real-world-usage">Real-World Usage</h2><p>Here’s what it looks like in action:</p><h2 id="nvme-system">NVMe Drive Controllers Are Smart</h2><p>Modern drives have sophisticated controllers that can perform secure erasure far faster than any software-based overwriting. Using these built-in capabilities is key to efficient sanitization.</p><h2 id="2-one-size-doesn-t-fit-all">2. One Size Doesn’t Fit All</h2><p>Different drive types require different approaches. NVMe drives support cryptographic erase, SATA drives have ATA Secure Erase, and only HDDs require actual overwriting.</p><h2 id="3-automation-reduces-errors">3. Automation Reduces Errors</h2><p>By automating the entire process, we eliminate the risk of human error, wrong commands, skipped drives, or incomplete erasure.</p><h2 id="4-compliance-requires-documentation">4. Compliance Requires Documentation</h2><p>The tool generates detailed logs showing:</p><ul><li>Drive serial numbers</li><li>Erasure methods used</li><li>Completion timestamps</li><li>Verification results</li></ul><h2 id="building-your-own">Building Your Own</h2><p>I have already uploaded an ISO file of the tool ready to fire available on my GitHub: <a href="https://github.com/CyberKareem/SecureAutomatedPurge-USB/releases/tag/v1.0.0" rel="nofollow noopener noreferrer" target="_blank"><em>SecureAutomatedPurge USB utility</em></a></p><p>The tool is completely open source. To build your own ISO:</p><pre><code data-language="text"># Clone the repositorygit clone https://github.com/CyberKareem/SecureAutomatedPurge-USB.gitcd SecureAutomatedPurge-USB</code></pre><pre><code data-language="text"># Run the build script (Debian/Ubuntu)sudo ./build/build_iso.sh</code></pre><p>The build process:</p><ol><li>Creates a Debian Live environment</li><li>Installs required packages</li><li>Adds the purge script</li><li>Configures auto-start</li><li>Generates a bootable ISO</li></ol><h2 id="security-considerations">Security Considerations</h2><h2 id="what-it-protects-against">What It Protects Against</h2><ul><li>Data recovery software</li><li>Filesystem undelete tools</li><li>Forensic recovery methods</li><li>Casual data thieves</li></ul><h2 id="what-it-doesn-t-protect-against">What It Doesn’t Protect Against</h2><ul><li>Nation-state level forensics</li><li>Electron microscopy on HDDs</li><li>Hidden drive areas (HPA/DCO)</li><li>Damaged sectors that can’t be overwritten</li></ul><p>For classified data, physical destruction remains the gold standard.</p><h2 id="use-cases">Use Cases</h2><h2 id="it-asset-disposal">IT Asset Disposal</h2><p>Perfect for decommissioning corporate workstations. Boot, confirm, and move to the next machine while the first one wipes.</p><h2 id="personal-privacy">Personal Privacy</h2><p>Selling your old computer? Ensure previous data is completely removed first.</p><h2 id="future-enhancements">Future Enhancements</h2><p>I’m thinking of adding:</p><ul><li>Network boot support (PXE)</li><li>Parallel drive processing</li><li>Cloud logging for centralized reporting</li><li>GUI version for less technical users</li><li>Support for external drives (with safety controls)</li></ul><h2 id="in-short">In Short</h2><p>Building this tool reminded me why I love open-source development. What started as a PowerShell script for BitLocker evolved into a comprehensive solution for secure data destruction. Each tool taught me something new:</p><ul><li><strong>BitLocker Crypto Erase</strong>: The power of leveraging existing encryption</li><li><strong>Secure Automated Purge</strong>: The importance of universal solutions</li></ul><p>Together, they form a complete toolkit for modern data sanitization. The BitLocker tool handles the “easy” cases where Windows and encryption are already in place.
