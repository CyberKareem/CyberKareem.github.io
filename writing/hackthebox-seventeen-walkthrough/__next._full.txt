<h2 id="the-hacker-mindset-for-this-box">The Hacker Mindset for This Box</h2>
<p>Before we touch a single command, let's understand what makes <em>Seventeen</em> a "Hard" box. This isn't about one big vulnerability — it's about <strong>chaining multiple smaller vulnerabilities across different virtual hosts and services</strong>. The attacker must:</p>
<ol>
<li><strong>Think in layers:</strong> Each subdomain reveals a piece of the puzzle. No single service gives you shell.</li>
<li><strong>Follow the data:</strong> SQL injection leaks credentials → credentials unlock a file manager → a PDF leaks another subdomain → that subdomain has an LFI → the LFI triggers an uploaded shell.</li>
<li><strong>Understand the environment:</strong> After getting shell, you're in Docker. The real target is the host.</li>
<li><strong>Abuse application logic:</strong> The final root isn't a kernel exploit or a SUID binary — it's abusing how a Node.js application loads dependencies via <code>npm install</code>.</li>
</ol>
<p><strong>The core philosophy:</strong> When you see a domain name (<code>seventeen.htb</code>), always assume there are subdomains. When you see a web application, always assume it talks to a database. When you see a package manager, always ask: <em>"What if I control the repository?"</em></p>
<hr />
<h2 id="phase-1-reconnaissance-and-discovery">Phase 1: Reconnaissance &amp; Discovery</h2>
<h3 id="step-1-1-nmap-scan-know-your-attack-surface">Step 1.1: Nmap Scan — Know Your Attack Surface</h3>
<pre><code class="language-bash" data-language="bash">sudo nmap -p- --min-rate 10000 -oN nmap_alltcp.txt 10.129.227.143
</code></pre>
<p><strong>Why this command?</strong></p>
<ul>
<li><code>-p-</code>: Scan all 65,535 ports. Hard boxes often hide services on non-standard ports.</li>
<li><code>--min-rate 10000</code>: Flood the target with packets to speed up the scan. HTB machines can handle it.</li>
<li><code>-oN</code>: Save output. You <em>will</em> forget port numbers.</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>"I don't know what's running yet. My job is to map the terrain before I attack."</em></p>
<p><strong>Results:</strong></p>
<pre><code>PORT     STATE SERVICE
22/tcp   open  ssh
80/tcp   open  http
8000/tcp open  http-alt
</code></pre>
<p><strong>Analysis:</strong></p>
<ul>
<li><strong>Port 22 (SSH):</strong> Usually the final destination, not the entry point. We note it but don't brute-force yet.</li>
<li><strong>Port 80 (HTTP):</strong> The main web server. Always our first target.</li>
<li><strong>Port 8000 (HTTP-ALT):</strong> Another web server! This is suspicious. It might be a different application, a development instance, or a container endpoint.</li>
</ul>
<p><strong>Baby Step:</strong> When you see multiple HTTP ports, treat them as completely separate applications. Don't assume they're the same site.</p>
<hr />
<h3 id="step-1-2-add-domains-to-etc-hosts">Step 1.2: Add Domains to <code>/etc/hosts</code></h3>
<pre><code class="language-bash" data-language="bash">sudo bash -c 'echo "10.129.227.143 seventeen.htb exam.seventeen.htb oldmanagement.seventeen.htb mastermailer.seventeen.htb" &gt;&gt; /etc/hosts'
</code></pre>
<p><strong>Why?</strong> Modern web servers use <strong>Virtual Hosts (VHosts)</strong> to serve different content based on the <code>Host</code> header. Accessing <code>http://10.129.227.143</code> might return a default page, but <code>http://seventeen.htb</code> might return the actual application.</p>
<p><strong>Hacker Mindset:</strong> <em>"IP addresses are for routers. Domain names are for web servers. If the server expects a domain name, I need to give it one."</em></p>
<hr />
<h3 id="step-1-3-vhost-fuzzing-finding-hidden-subdomains">Step 1.3: VHost Fuzzing — Finding Hidden Subdomains</h3>
<pre><code class="language-bash" data-language="bash">ffuf -c -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt \
     -u http://seventeen.htb -H "Host: FUZZ.seventeen.htb" -fw 2760
</code></pre>
<p><strong>Why this tool/command?</strong></p>
<ul>
<li><code>ffuf</code> is a fast web fuzzer.</li>
<li><code>-H "Host: FUZZ.seventeen.htb"</code>: We fuzz the <code>Host</code> header. The server might route <code>exam.seventeen.htb</code> to a completely different application.</li>
<li><code>-fw 2760</code>: Filter out responses with 2760 words (the default page size). We're looking for anomalies.</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>"The main website is just the lobby. The real secrets are in the back rooms. VHost fuzzing is how I find the doors."</em></p>
<p><strong>Result:</strong></p>
<pre><code>exam                    [Status: 200, Size: 17375, Words: 3222, Lines: 348, Duration: 18ms]
</code></pre>
<p><strong>Analysis:</strong> We found <code>exam.seventeen.htb</code>! This is a completely different application living on the same IP address.</p>
<hr />
<h2 id="phase-2-exam-management-system-sql-injection">Phase 2: Exam Management System — SQL Injection</h2>
<h3 id="step-2-1-recon-the-exam-site">Step 2.1: Recon the Exam Site</h3>
<p>Browse to <code>http://exam.seventeen.htb</code>. We see an "Exam Management System." There's a "Take Exam" link that goes to:</p>
<pre><code>http://exam.seventeen.htb/?p=take_exam&amp;id=1
</code></pre>
<p><strong>Why is this interesting?</strong></p>
<ul>
<li><code>?p=take_exam&amp;id=1</code>: The <code>id</code> parameter looks like a database query. This screams <strong>SQL injection</strong>.</li>
<li>The application is PHP (confirmed by <code>X-Powered-By: PHP/7.2.34</code> header).</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>"Anytime I see <code>?id=1</code>, I hear a database screaming. PHP + user input = SQLi until proven otherwise."</em></p>
<hr />
<h3 id="step-2-2-confirming-the-vulnerability">Step 2.2: Confirming the Vulnerability</h3>
<p>We could manually test with <code>' AND 1=1 --</code> vs <code>' AND 1=2 --</code>, but this is a <strong>boolean-based blind SQL injection</strong> — the page changes slightly based on whether the SQL condition is true or false. Manual testing is tedious.</p>

<h3 id="step-7-3-the-critical-step-removing-the-local-copy">Step 7.3: The Critical Step — Removing the Local Copy</h3>
<p>Here's where many attempts fail. If <code>/opt/app/node_modules/loglevel</code> already exists from a previous install, <code>npm -l ls | grep loglevel</code> will match, and <code>startup.sh</code> will <strong>skip</strong> the installation. It will never query our registry.</p>
<p><strong>We must delete the existing module first:</strong></p>
<pre><code class="language-bash" data-language="bash">rm -rf /opt/app/node_modules/loglevel
</code></pre>
<p><strong>Why does this work?</strong> <code>rm -rf</code> on a directory only requires <strong>write permission on the parent directory</strong>. While <code>/opt/app/node_modules</code> itself is <code>drwxr-xr-x</code> (root-owned 755), the <code>loglevel</code> subdirectory inside it was created by npm with permissions that allowed deletion in this specific case. More importantly, once deleted, <code>npm -l ls | grep loglevel</code> no longer matches, forcing <code>startup.sh</code> to reinstall.</p>
<p><strong>Hacker Mindset:</strong> <em>"I don't just run the exploit. I clear the obstacles first. If npm thinks the package is already installed, it won't fetch my malicious version. Deletion is preparation."</em></p>

<h3 id="step-7-5-getting-root">Step 7.5: Getting Root</h3>
<pre><code class="language-bash" data-language="bash">ls -la /bin/bash
# -rwsr-xr-x 1 root root 1113504 Apr 18  2022 /bin/bash

bash -p
id
# uid=1000(kavi) gid=1000(kavi) euid=0(root) groups=1000(kavi)

cat /root/root.txt
# d253d[redacted]edb892ca04
</code></pre>
<p><strong>Why <code>bash -p</code>?</strong></p>
<ul>
<li>When a binary has the <strong>SUID bit</strong> set (<code>s</code> instead of <code>x</code> in the owner permissions), running it gives you the <strong>effective UID</strong> of the owner (root).</li>
<li>Normally, bash drops privileges. <code>bash -p</code> (preserve) tells bash to <strong>keep the effective UID</strong>.</li>
<li>This gives us a root shell.</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>"I didn't need a reverse shell or a password. I just needed bash to remember it's supposed to be root. SUID is privilege elevation in its purest form."</em></p>

<li>SQLi → Credentials</li>
<li>Credentials → File Upload</li>
<li>File Upload → LFI</li>
<li>LFI → Shell</li>
<li>Shell → Docker Breakout</li>
<li>Docker Breakout → Host Creds</li>
<li>Host Creds → User Shell</li>
<li>User Shell → Malicious NPM Package</li>
<li>NPM Package → Root</li>
</ul>
<h3 id="5-npm-install-as-root-is-dangerous">5. npm install as Root is Dangerous</h3>
<p>Any script that runs <code>npm install</code> as root is a privilege escalation waiting to happen. If you control the registry (or <code>.npmrc</code>), you control the code that executes as root.</p>
<h3 id="6-clear-obstacles-before-exploiting">6. Clear Obstacles Before Exploiting</h3>
<p>The most frustrating part of this box was that <code>/opt/app/node_modules/loglevel</code> already existed, so <code>npm install</code> was skipped. <strong>Deleting the existing module was the critical step</strong> that forced npm to fetch from our rogue registry.</p>
<h3 id="7-suid-bash-is-reliable">7. SUID Bash is Reliable</h3>
<p>Once you have <code>chmod u+s /bin/bash</code>, you have a persistent root backdoor. <code>bash -p</code> is the simplest privilege escalation technique in Linux.</p>

<p><strong>Final Note:</strong> Seventeen is a masterclass in chaining. It teaches you that hard boxes aren't about one massive exploit — they're about patience, enumeration, and connecting the dots across multiple services.</p>
