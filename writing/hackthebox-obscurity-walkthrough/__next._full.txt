<h2 id="introduction-and-mindset">Introduction &amp; Mindset</h2>
<h3 id="what-is-security-through-obscurity">What is "Security Through Obscurity"?</h3>
<p>This box is literally named <strong>Obscurity</strong>, and its theme is <em>"security through obscurity."</em> This is a critical concept in cybersecurity:</p>
<blockquote>
<p><strong>Security through obscurity</strong> is the reliance on the secrecy of the design or implementation as the main method of providing security for a system.</p>
</blockquote>
<p>The website even jokes about it: <em>"you can't be hacked if attackers don't know what software you're using!"</em> This is a <strong>fallacy</strong>. Real security comes from strong design, peer review, and assuming the attacker <strong>already knows</strong> how your system works (Kerckhoffs's Principle).</p>
<h3 id="the-hacker-mindset-for-this-box">The Hacker Mindset for This Box</h3>
<p>Before we start typing commands, let's understand the mindset:</p>
<ol>
<li><strong>Assume nothing is truly hidden.</strong> The devs think their "secret" directory is safe because it's not linked anywhere. We know better.</li>
<li><strong>Source code is the ultimate target.</strong> If we can read the source code of a custom application, we can find vulnerabilities that black-box testing would never reveal.</li>
<li><strong>Custom crypto is almost always broken.</strong> The box advertises an "unbreakable encryption algorithm." In the real world, custom cryptography is a huge red flag. Standard algorithms (AES, RSA, ChaCha20) are standard for a reason -- they've been tested by thousands of cryptographers.</li>
<li><strong>Look for developer mistakes, not complex exploits.</strong> The <code>exec()</code> vulnerability we'll find is a basic Python mistake. The root escalation is a logic flaw in argument parsing. These aren't zero-days -- they're careless coding.</li>
</ol>

<h3 id="why-we-do-this">Why We Do This</h3>

<h3 id="the-hacker-mindset">The Hacker Mindset</h3>
<p>We're asking: <em>"What services are running? What versions? What can I talk to?"</em></p>
<ul>
<li><strong>Port 22 (SSH):</strong> Standard OpenSSH. Probably not our initial entry point unless we have credentials, but we'll note it for later.</li>
<li><strong>Port 8080 (HTTP):</strong> A web server running something called <code>BadHTTPServer</code>. This is <strong>not</strong> Apache, Nginx, or IIS. It's <strong>custom software</strong> -- immediately interesting because custom software often has custom vulnerabilities.</li>
<li><strong>Ports 80 and 9000 are closed:</strong> Interesting that they send RST packets instead of no-response. This tells us the host is actively rejecting connections on those ports, which might hint at firewall rules or future pivoting opportunities.</li>

<h3 id="key-takeaway">Key Takeaway</h3>
<blockquote>

<h2 id="step-4-source-code-analysis">Step 4: Source Code Analysis</h2>
<h3 id="the-commands-4">The Commands</h3>

cat SuperSecureServer.py
</code></pre>
<h3 id="why-we-do-this-4">Why We Do This</h3>
<p><strong>Source code is the attacker's best friend.</strong> When you can read the source code of a web application, you don't have to guess how it works -- you can see exactly how it processes input, where it sanitizes (or doesn't sanitize) data, and what assumptions the developer made.</p>
<h3 id="the-hacker-mindset-4">The Hacker Mindset</h3>
<p>We're reading this code like a security auditor. We're not looking for "does it work?" -- we're looking for "how can I break it?"</p>
<p>Key questions we ask:</p>
<ul>
<li>Where does user input enter the system?</li>
<li>Is that input sanitized, validated, or escaped?</li>
<li>Does it reach any dangerous functions (<code>exec</code>, <code>eval</code>, <code>os.system</code>, <code>subprocess</code>, SQL queries, etc.)?</li>
<li>Are there any logic flaws or race conditions?</li>
</ul>
<h3 id="the-vulnerability">The Vulnerability</h3>
<p>In the <code>serveDoc()</code> method, we find:</p>
<pre><code class="language-python" data-language="python">def serveDoc(self, path, docRoot):
    path = urllib.parse.unquote(path)
    try:
        info = "output = 'Document: {}'" # Keep the output for later debug
        exec(info.format(path)) # This is how you do string formatting, right?
</code></pre>
<p><strong>Let's break down why this is catastrophic:</strong></p>
<ol>
<li><strong><code>path</code> comes from the HTTP request URL.</strong> The user completely controls this.</li>
<li><strong><code>urllib.parse.unquote(path)</code></strong> URL-decodes the path. This means <code>%20</code> becomes a space, <code>%27</code> becomes <code>'</code>, etc.</li>
<li><strong><code>info.format(path)</code></strong> inserts the user-controlled path into a Python string.</li>
<li><strong><code>exec()</code> executes that string as Python code.</strong></li>
</ol>
<h3 id="the-developer-s-mistake">The Developer's Mistake</h3>
<p>The developer wanted to create a debug string like:</p>
<pre><code class="language-python" data-language="python">output = 'Document: /index.html'
</code></pre>
<p>But they used <code>exec()</code> instead of simple assignment. If a user sends:</p>
<pre><code>/'; os.system('id'); a='
</code></pre>
<p>The formatted string becomes:</p>
<pre><code class="language-python" data-language="python">output = 'Document: '; os.system('id'); a=''
</code></pre>
<p>The <code>exec()</code> runs this as three separate Python statements:</p>
<ol>
<li><code>output = 'Document: '</code> (harmless assignment)</li>
<li><code>os.system('id')</code> (<strong>REMOTE CODE EXECUTION</strong>)</li>
<li><code>a=''</code> (harmless assignment)</li>
</ol>
<h3 id="why-os-is-available">Why <code>os</code> Is Available</h3>
<p>Notice at the top of the file:</p>
<pre><code class="language-python" data-language="python">import os
</code></pre>
<p>Because <code>os</code> is already imported into the script's namespace, our injected code can use it directly without needing to import it ourselves.</p>
<h3 id="key-takeaway-4">Key Takeaway</h3>
<blockquote>
<p><strong><code>exec()</code> and <code>eval()</code> with user input is always dangerous.</strong> There is virtually no safe way to use these functions with untrusted data. If you see <code>exec()</code>, <code>eval()</code>, or <code>pickle.loads()</code> in code that processes user input, you've probably found your vulnerability.</p>
</blockquote>

<ol>

<h3 id="the-hacker-mindset-5">The Hacker Mindset</h3>

<h2 id="step-8-cryptographic-analysis-known-plaintext-attack">Step 8: Cryptographic Analysis (Known-Plaintext Attack)</h2>
<h3 id="understanding-the-encryption">Understanding the Encryption</h3>
<p>Looking at <code>SuperSecureCrypt.py</code>:</p>
<pre><code class="language-python" data-language="python">def encrypt(text, key):
    keylen = len(key)
    keyPos = 0
    encrypted = ""
    for x in text:
        keyChr = key[keyPos]
        newChr = ord(x)
        newChr = chr((newChr + ord(keyChr)) % 255)
        encrypted += newChr
        keyPos += 1
        keyPos = keyPos % keylen
    return encrypted

def decrypt(text, key):
    keylen = len(key)
    keyPos = 0
    decrypted = ""
    for x in text:
        keyChr = key[keyPos]
        newChr = ord(x)
        newChr = chr((newChr - ord(keyChr)) % 255)
        decrypted += newChr
        keyPos += 1
        keyPos = keyPos % keylen
    return decrypted
</code></pre>
<h3 id="breaking-down-the-math">Breaking Down the Math</h3>
<p>This is essentially a <strong>Vigenere cipher</strong> with modulo 255 instead of 26:</p>
<ul>
<li><strong>Encryption:</strong> <code>cipher_char = (plain_char + key_char) % 255</code></li>
<li><strong>Decryption:</strong> <code>plain_char = (cipher_char - key_char) % 255</code></li>
</ul>
<h3 id="why-custom-crypto-is-bad">Why Custom Crypto Is Bad</h3>
<p>Real-world encryption uses algorithms like AES because:</p>
<ol>
<li>They resist known-plaintext attacks</li>
<li>They use complex substitution-permutation networks</li>
<li>They have been analyzed by cryptographers for decades</li>
</ol>
<p>This custom algorithm is just <strong>repeated addition</strong>. If you know any plaintext and its ciphertext, you can solve for the key:</p>
<pre><code>key_char = (cipher_char - plain_char) % 255
</code></pre>
<h3 id="the-attack-using-the-script-against-itself">The Attack: Using the Script Against Itself</h3>
<p>We have:</p>
<ul>
<li><code>check.txt</code> = plaintext</li>
<li><code>out.txt</code> = ciphertext of check.txt</li>
</ul>
<p>If we "decrypt" <code>out.txt</code> using <code>check.txt</code> as the "key," the math works out to:</p>
<pre><code>result = (cipher_char - plain_char) % 255 = key_char
</code></pre>
<p>So we run:</p>
<pre><code class="language-bash" data-language="bash">python3 SuperSecureCrypt.py -i out.txt -k "$(cat check.txt)" -d -o /tmp/key.txt
cat /tmp/key.txt
</code></pre>
<p><strong>Output:</strong> <code>alexandrovichalexandrovichalexandrovich...</code></p>
<p>The key is <strong><code>alexandrovich</code></strong>.</p>
<h3 id="decrypting-the-password">Decrypting the Password</h3>
<p>Now that we have the key:</p>
<pre><code class="language-bash" data-language="bash">python3 SuperSecureCrypt.py -i passwordreminder.txt -k alexandrovich -d -o /tmp/passwd.txt
cat /tmp/passwd.txt
</code></pre>
<p><strong>Output:</strong> <code>SecThruObsFTW</code></p>
<h3 id="the-hacker-mindset-8">The Hacker Mindset</h3>
<p>We're thinking: <em>"The developer wanted to test their encryption, so they left a plaintext/ciphertext pair. They didn't realize this completely breaks their cipher. This is why Kerckhoffs's Principle says the security of a cipher should depend ONLY on the key, not on keeping the algorithm secret."</em></p>
<h3 id="key-takeaway-8">Key Takeaway</h3>
<blockquote>
<p><strong>Never roll your own cryptography.</strong> And if you find custom crypto on a target, look for known-plaintext or chosen-plaintext opportunities. Test files, example outputs, and encrypted data with predictable structure (like JSON, XML, or English text headers) are all potential attack vectors.</p>
</blockquote>

<h2 id="step-10-root-privilege-escalation">Step 10: Root Privilege Escalation</h2>
<h3 id="checking-sudo-privileges">Checking sudo Privileges</h3>
<pre><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>User robert may run the following commands on obscure:
    (ALL) NOPASSWD: /usr/bin/python3 /home/robert/BetterSSH/BetterSSH.py
</code></pre>
<h3 id="why-this-is-interesting">Why This Is Interesting</h3>
<p><code>sudo -l</code> tells us what commands robert can run as root without a password. The only entry is a <strong>custom Python script</strong> -- another instance of custom software. Custom software running with elevated privileges is a <strong>privilege escalation goldmine</strong>.</p>
<h3 id="analyzing-betterssh-py">Analyzing BetterSSH.py</h3>
<p>The script:</p>
<ol>
<li>Reads a username and password from the user</li>
<li>Copies <code>/etc/shadow</code> to <code>/tmp/SSH/[random_name]</code></li>
<li>Sleeps for 0.1 seconds</li>
<li>Validates the password against the shadow hash</li>
<li>If authenticated, enters a loop that runs commands via <code>sudo -u [user]</code></li>
</ol>
<p>The critical code is:</p>
<pre><code class="language-python" data-language="python">command = input(session['user'] + "@Obscure$ ")
cmd = ['sudo', '-u',  session['user']]
cmd.extend(command.split(" "))
proc = subprocess.Popen(cmd, stdout=subprocess.PIPE, stderr=subprocess.PIPE)
</code></pre>
<h3 id="finding-the-vulnerability">Finding the Vulnerability</h3>
<p>The script builds the command list as:</p>
<pre><code class="language-python" data-language="python">cmd = ['sudo', '-u', 'robert']
# Then extends with our input
cmd = ['sudo', '-u', 'robert', '-u', 'root', 'cat', '/root/root.txt']
</code></pre>
<p><strong>How <code>sudo</code> handles multiple <code>-u</code> flags:</strong> When <code>sudo</code> receives multiple <code>-u</code> options, the <strong>last one wins</strong>. This is standard behavior for many command-line tools.</p>
<p>So <code>sudo -u robert -u root cat /root/root.txt</code> executes as <strong>root</strong>.</p>
<h3 id="the-hacker-mindset-10">The Hacker Mindset</h3>
<p>We're analyzing the script as if we wrote it. We ask:</p>
<ul>
<li><em>"What assumptions did the developer make?"</em> They assumed the user would only enter simple commands.</li>
<li><em>"What can I inject?"</em> The <code>command.split(" ")</code> splits on spaces, but <code>sudo</code> parses arguments sequentially. Adding <code>-u root</code> before our real command hijacks the privilege level.</li>
<li><em>"Why does this work?"</em> Because the developer trusted user input to be a simple command, not a command with its own flags.</li>
</ul>
<h3 id="exploitation">Exploitation</h3>
<pre><code class="language-bash" data-language="bash">mkdir -p /tmp/SSH
sudo /usr/bin/python3 /home/robert/BetterSSH/BetterSSH.py
# Enter username: robert
# Enter password: null
# Authed!
# robert@Obscure$ -u root cat /root/root.txt
</code></pre>
<p><strong>Root flag:</strong> <code>ea0cac[redacted]695d4b6d</code></p>
<h3 id="why-we-created-tmp-ssh-first">Why We Created /tmp/SSH First</h3>
<p>The script writes the shadow file to <code>/tmp/SSH/[random]</code>. If <code>/tmp/SSH</code> doesn't exist, the script crashes with a <code>FileNotFoundError</code>. This is a small detail that could block exploitation if we didn't know about it.</p>
<h3 id="key-takeaway-9">Key Takeaway</h3>
<blockquote>
<p><strong>When analyzing sudo privileges, always read the source code of custom scripts.</strong> Look for:</p>
<ul>
<li><code>subprocess.Popen</code>, <code>os.system</code>, or <code>exec</code> with user input</li>
<li>Command-line argument injection opportunities</li>
<li>Race conditions (file operations with predictable paths or timing windows)</li>
<li>Path hijacking (directories in PATH that are writable by the user)</li>
</ul>
</blockquote>
<hr />
<h2 id="alternative-root-paths">Alternative Root Paths</h2>
<h3 id="path-1-race-condition-on-shadow-file">Path 1: Race Condition on Shadow File</h3>
<p>The script writes <code>/etc/shadow</code> to <code>/tmp/SSH/[random_name]</code> and leaves it there for <strong>0.1 seconds</strong>. An attacker could:</p>
<ol>
<li>Run a loop constantly reading <code>/tmp/SSH/*</code></li>
<li>Run BetterSSH simultaneously</li>
<li>Catch the shadow file before it's deleted</li>
<li>Crack the root password hash with <code>john</code> or <code>hashcat</code></li>
</ol>
<p><strong>Why this works:</strong> The developer assumed 0.1 seconds was "too fast" to exploit. On a modern CPU, 0.1 seconds is an eternity.</p>
<h3 id="path-2-directory-replacement">Path 2: Directory Replacement</h3>
<p>If robert had write access to the <code>BetterSSH</code> directory, he could:</p>
<ol>
<li><code>mv BetterSSH BetterSSH-old</code></li>
<li><code>mkdir BetterSSH</code></li>
<li>Write a malicious <code>BetterSSH.py</code> that spawns a root shell</li>
<li>Run <code>sudo /usr/bin/python3 /home/robert/BetterSSH/BetterSSH.py</code></li>
</ol>
<p><strong>Why this works:</strong> <code>sudo</code> checks the path, not the file contents. If you can replace the directory or file, you control what runs as root.</p>
<p><strong>Note:</strong> In our instance, this was patched -- the <code>BetterSSH</code> directory was owned by root and not writable by robert.</p>
<h3 id="path-3-python-path-hijacking-patched">Path 3: Python Path Hijacking (Patched)</h3>
<p>Python loads modules from the script's directory first. If <code>BetterSSH/</code> was writable, robert could create a fake <code>subprocess.py</code> in that directory. When BetterSSH imported <code>subprocess</code>, it would load robert's malicious version instead of the system library.</p>
<p><strong>Why this works:</strong> Python's module resolution prioritizes the local directory. This is well-known behavior that many developers forget about.</p>
<h3 id="path-4-lxd-group-patched">Path 4: LXD Group (Patched)</h3>
<p>At release, robert was in the <code>lxd</code> group. Members of the <code>lxd</code> group can create privileged containers that mount the host filesystem. An attacker could:</p>
<ol>
<li>Import an Alpine Linux container image</li>
<li>Create a privileged container with the host root mounted as <code>/mnt/root</code></li>
<li>Access any file on the host as root</li>
</ol>
<p><strong>Why this works:</strong> LXD containers share the kernel with the host. A privileged container has root access to the host's kernel namespaces and mounted filesystems.</p>
<hr />
<h2 id="lessons-learned">Lessons Learned</h2>
<h3 id="for-attackers">For Attackers</h3>
<ol>
<li><strong>Custom software is fertile ground for vulnerabilities.</strong> Always prioritize custom applications over standard, well-tested services.</li>
<li><strong>Source code access changes everything.</strong> A single source file leak can turn a black-box test into a code audit.</li>
<li><strong>Never underestimate developer laziness.</strong> "Secret" directories, hardcoded credentials, debug comments, and test files are everywhere.</li>
<li><strong>Custom cryptography is a vulnerability, not a feature.</strong> If you see homegrown encryption, look for known-plaintext attacks.</li>
<li><strong>Sudo on custom scripts is dangerous.</strong> Always analyze what the script does and how it handles user input.</li>
</ol>
<h3 id="for-defenders">For Defenders</h3>
<ol>
<li><strong>Never use <code>exec()</code> or <code>eval()</code> with user input.</strong> There is no safe way to do this.</li>
<li><strong>Don't rely on obscurity.</strong> Hiding your source code or using custom algorithms doesn't make you secure.</li>
<li><strong>Remove debug code and comments before deployment.</strong> That "note to devs" gave away the entire attack path.</li>
<li><strong>Use standard cryptography.</strong> AES, RSA, and ChaCha20 exist for a reason.</li>
<li><strong>Audit custom scripts running with elevated privileges.</strong> Especially any script that uses <code>subprocess</code>, <code>os.system</code>, or handles user input.</li>
<li><strong>Set strict permissions on sensitive directories.</strong> If a directory contains a script that runs as root, no unprivileged user should be able to rename, modify, or write to it.</li>
</ol>
