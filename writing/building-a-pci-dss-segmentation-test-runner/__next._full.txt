28:T3c20,<p>For an assessment, a segmentation scan needs a record of where it ran, what it targeted, which probes were used, and which raw files were retained. I built <a href="https://github.com/CyberKareem/seg-test" target="_blank" rel="nofollow noopener noreferrer"><code>seg-test</code></a> to run a defined set of reachability checks from each approved source VLAN and package the output for review.</p>
<p>The Bash runner takes an explicit cardholder data environment (CDE) target list and does not expand scope. This article covers the design in the <a href="https://github.com/CyberKareem/seg-test/blob/db8d0cce2c3a06af20d3974a9cc59611f4415099/seg-test.sh" target="_blank" rel="nofollow noopener noreferrer">audited source revision</a>, including its limits and the changes I would make next.</p>
<p><code>seg-test</code> collects evidence. It does <strong>not</strong> determine PCI DSS compliance, certify segmentation, or replace the assessor's methodology and judgment.</p>
<h2 id="why-the-evidence-matters">Why the evidence matters</h2>
<p>PCI DSS is a baseline of technical and operational requirements for protecting payment account data. PCI SSC published PCI DSS v4.0.1 as a limited revision that clarified guidance without adding or deleting requirements (<a href="https://blog.pcisecuritystandards.org/just-published-pci-dss-v4-0-1" target="_blank" rel="nofollow noopener noreferrer">PCI SSC: “Just Published: PCI DSS v4.0.1”</a>).</p>
<p>Where segmentation is used to isolate the CDE, Requirement 11.4.5 in the standard available through PCI SSC's <a href="https://www.pcisecuritystandards.org/document_library/?category=pcidss&amp;document=pci_dss" target="_blank" rel="nofollow noopener noreferrer">PCI DSS document library</a> expects segmentation penetration testing at least every 12 months and after changes to segmentation controls or methods. The work must cover the controls in use, follow the entity's methodology, confirm that isolation is effective, and be performed by a qualified resource with organizational independence. The scan is one part of that assessment.</p>
<p>Modern environments may enforce segmentation across cloud networks, service meshes, identities, and ephemeral workloads. PCI SSC's <a href="https://blog.pcisecuritystandards.org/new-information-supplement-pci-dss-scoping-and-segmentation-guidance-for-modern-network-architectures" target="_blank" rel="nofollow noopener noreferrer">modern network architecture guidance</a> covers this wider set of controls. Record the actual source and boundary exercised in every run.</p>
<h2 id="run-once-from-each-approved-source-vlan">Run once from each approved source VLAN</h2>
<p>The script, configuration, CDE list, and evidence stay together. Run the kit inside each source VLAN named in the rules of engagement (ROE) so every result has a recorded source position.</p>
<p>The normal preparation flow is:</p>
<ol>
<li>Obtain signed authorization and a precise ROE before sending a packet.</li>
<li>Scaffold a configuration with <code>--init &lt;client&gt;</code>.</li>
<li>Populate <code>cde-all.txt</code> with only approved CDE targets, one per line.</li>
<li>Define an optional traceroute anchor and any prior-finding hot spots.</li>
<li>Preview the run with <code>--dry-run</code>.</li>
<li>Enter the approved source VLAN and use <code>--expect-cidr &lt;approved-cidr&gt;</code> as a source-position guard.</li>
<li>Execute the run, review the raw output, and preserve the resulting evidence package.</li>
</ol>
<p>A sanitized command sequence looks like this:</p>
<pre><code class="language-bash" data-language="bash">./seg-test.sh --init &lt;client&gt;
./seg-test.sh --client &lt;client&gt; &lt;source-vlan&gt; --dry-run
sudo ./seg-test.sh \
  --client &lt;client&gt; &lt;source-vlan&gt; \
  --expect-cidr &lt;approved-cidr&gt;
</code></pre>
<p><code>--dry-run</code> prints the configuration, target count, source guard, scan families, hot spots, and evidence path without sending packets or requiring root. Review it before the live run.</p>
<h2 id="run-sequence">Run sequence</h2>
<h3 id="1-fail-early-on-obvious-setup-errors">1. Fail early on obvious setup errors</h3>
<p>The runner validates the client and VLAN labels, requires a matching client configuration, checks that the target list exists and is non-empty, and verifies its dependencies. It can report missing tools or install the relevant packages on Debian/Kali through <code>apt</code>. Real SYN and UDP scans require root.</p>
<h3 id="2-create-a-timestamped-evidence-workspace">2. Create a timestamped evidence workspace</h3>
<p>Each execution receives its own directory under <code>evidence/</code>, divided into baseline, Nmap, manual, packet-capture, and screenshot locations. The runner starts a transcript and records metadata such as the source label, host, operator, arguments, target count, anchor, expected CIDR, and selected ports.</p>
<p>Those empty directories are reserved for manual follow-up; they do not mean a capture or screenshot was collected.</p>
<h3 id="3-record-the-source-side-baseline">3. Record the source-side baseline</h3>
<p>Before scanning, the script records time, interface addresses, routes, neighbor state, and, when configured, a short traceroute to a CDE anchor. It prints the detected IPv4 source addresses prominently.</p>
<p>If no local interface is inside the approved source CIDR, the run stops. <code>--force</code> bypasses that check and must be documented.</p>
<h3 id="4-use-several-reachability-checks">4. Use several reachability checks</h3>
<p>The first network phase performs Nmap host discovery. The official <a href="https://nmap.org/book/man-host-discovery.html" target="_blank" rel="nofollow noopener noreferrer">Nmap host-discovery documentation</a> explains why a single ping is insufficient: <code>-sn</code> uses discovery probes without a port scan, while <code>-Pn</code> skips discovery and treats targets as online for later scanning. Firewalls can suppress probes or replies, so “did not answer discovery” is not equivalent to “unreachable.”</p>
<p>The substantive port tests therefore use <code>-Pn</code> and attempt every approved target:</p>
<ul>
<li>A TCP SYN scan checks a fixed set of common and assessment-relevant ports.</li>
<li>An optional full TCP scan covers all ports with a rate limit.</li>
<li>A selected UDP scan checks a bounded list of infrastructure and management ports unless explicitly skipped.</li>
<li>Configured TCP and HTTPS hot spots re-test paths documented in an earlier report.</li>
</ul>
<p>The runner does not silently expand scope. Nmap consumes the supplied target file, and manual hot spots come from the engagement configuration. That design is useful only if both inputs have been independently reviewed against the ROE.</p>
<h3 id="5-preserve-raw-artifacts-before-summarizing">5. Preserve raw artifacts before summarizing</h3>
<p>Nmap writes normal, XML, and grepable output. The script extracts open-port lines into a quick-look summary, but the summary is not the evidence of record; the raw files and transcript are. Finally, it archives the run and writes a SHA-256 digest beside the archive.</p>
<p>This produces a convenient integrity check: later, an assessor can detect a change if the separately trusted digest remains available. A bare hash stored beside the archive does not prove who produced the evidence or when. The package is integrity-checkable, but it is not tamper-evident or tamper-proof.</p>
<h2 id="what-seg-test-does-not-do">What seg-test does not do</h2>
<ul>
<li>It does <strong>not</strong> make pass/fail decisions; that is a human reporting task.</li>
<li>It does <strong>not</strong> capture packet traces automatically; ambiguous results may justify a controlled, interactive capture.</li>
<li>It does <strong>not</strong> brute-force, exploit, or run Nmap vulnerability scripts; those actions are outside this runner's role and must be separately authorized.</li>
<li>Assessment probes target entries in <code>cde-all.txt</code>, configured hot-spot lists, and the optional traceroute anchor. Review all three against the ROE before a run.</li>
</ul>
<h2 id="interpreting-the-results">Interpreting the results</h2>
<p>An <code>open</code> TCP port demonstrates that the tested source could complete enough of the protocol exchange for Nmap to classify the port at that time. That is strong evidence of a reachable path requiring investigation. It is not automatically a segmentation failure: the connection may be explicitly authorized, mediated, or required, and the assessor must compare it with the approved data flow and control design.</p>
<p><code>closed</code> usually means the host was reachable but no service accepted the connection. That still demonstrates network reachability; it does not prove isolation.</p>
<p><code>filtered</code>, silence, and many UDP results are ambiguous. A firewall may have dropped the request, the return path may differ, the host may be down, an intermediate control may rate-limit, or the service may simply not respond to the selected payload. Those observations need correlation with firewall logs, routing, control configuration, and, when authorized, a packet capture from an appropriate observation point.</p>
<p>Report the observation directly: “From approved source position A, target B returned a SYN/ACK on TCP port C at time D.” Then explain whether the path is expected and how it maps to policy.</p>
<h2 id="known-limitations-in-this-revision">Known limitations in this revision</h2>
<ol>
<li><strong>The source guard is IPv4 and interface based.</strong> It does not prove the actual egress path, VRF, policy identity, tunnel selection, or IPv6 position used by every packet.</li>
<li><strong><code>--force</code> can bypass the strongest safety check.</strong> An exception can turn a correctly labeled run into evidence from the wrong network.</li>
<li><strong>The target list, traceroute anchor, and hot-spot entries are trust boundaries.</strong> The runner does not cryptographically bind them to the signed ROE, enforce that the anchor is in the target list, or prove that every entry is authorized.</li>
<li><strong>The client environment file is sourced as shell code.</strong> A modified file executes with the runner's privileges.</li>
<li><strong>Root increases impact.</strong> A typo, compromised dependency, or malicious configuration executes in a highly privileged context.</li>
<li><strong><code>-Pn -sn</code> is not independent reachability proof.</strong> Because <code>-Pn</code> treats targets as online and <code>-sn</code> disables port scanning, that artifact must not be read as a successful response from every listed host.</li>
<li><strong>A fixed common-port list is incomplete.</strong> Segmentation leakage on an unlisted port will be missed unless the optional full TCP scan is approved and completed.</li>
<li><strong>Selected UDP coverage is inherently narrow and ambiguous.</strong> Silent UDP services and filtered responses can produce false reassurance.</li>
<li><strong>The full-scan rate is static.</strong> It can be too aggressive for fragile controls or unnecessarily slow elsewhere.</li>
<li><strong>No automatic packet capture exists.</strong> The evidence may be insufficient to distinguish a dropped probe, asymmetric return path, proxy response, or scanner-side problem.</li>
<li><strong>A local timestamp is not trusted time.</strong> Clock drift or manipulation weakens chronology unless time synchronization is verified and recorded.</li>
<li><strong>A sidecar SHA-256 file authenticates neither origin nor custody.</strong> Anyone able to replace the archive can also replace its digest.</li>
<li><strong>A failed command can still leave an archive that looks complete.</strong> The script does not use fail-fast execution or a machine-readable completeness manifest for every scan stage.</li>
<li><strong>VLAN checks do not cover every modern control plane.</strong> Cloud security groups, service meshes, identity-aware proxies, workloads that appear and disappear, and control-plane paths may not be exercised from a traditional source VLAN.</li>
</ol>
<p>These limits belong in the test plan and final report.</p>
<h2 id="what-i-would-change-next">What I would change next</h2>
<p><strong>Bind runs to authorization.</strong> Replace free-form targets with a validated manifest containing approved CIDRs, source positions, permitted protocols, change ticket, ROE identifier, and engagement window. Normalize every target before Nmap sees it, reject unexpected syntax, and require hot spots to fall inside the same allowlist. Store hashes of the manifest, configuration, target list, and exact script revision in run metadata.</p>
<p><strong>Record stage completion.</strong> Capture each command's start time, end time, exit code, and output files in a machine-readable run manifest. Mark interrupted or failed stages as incomplete and refuse to label their archive as a completed run. Correct the discovery phase so <code>-sn</code> is used for observed discovery and <code>-Pn</code> is reserved for scans that must test every target regardless of discovery response.</p>
<p><strong>Strengthen evidence provenance.</strong> Record Nmap and operating-system versions, scanner image or package versions, UTC time, and time-sync status. Sign the final manifest and archive with a detached signature whose public key and verification procedure are maintained outside the engagement kit. Store the signed digest in a separate evidence system so archive and checksum cannot be replaced together without detection.</p>
<p><strong>Keep result states separate.</strong> Parse Nmap XML into a structured finding table while retaining all raw artifacts. Use explicit states such as observed-open, observed-closed, filtered, no-response, tool-error, and not-tested. Never collapse those states into a generated “pass.” Add a reviewer field and require a reason for every allowed reachable path.</p>
<p><strong>Add targeted corroboration.</strong> Offer an opt-in packet-capture mode for specific approved targets and ports, with data-minimization limits, capture duration, storage warnings, and a separate authorization gate. Add collection hooks for firewall logs and policy snapshots rather than trying to infer the entire control path from scanner output.</p>
<p><strong>Cover modern controls.</strong> Extend the test plan beyond VLANs to IPv6, cloud accounts and regions, security groups, Kubernetes network policies, service-mesh identities, remote-access paths, and ephemeral assets. The runner should consume an assessor-defined coverage matrix and report which source-to-destination control paths were actually exercised.</p>
<h2 id="role-in-an-assessment">Role in an assessment</h2>
<p><code>seg-test</code> standardizes evidence collection across approved source VLANs; the assessor remains responsible for authorization, scope, interpretation, exceptions, and the final conclusion.</p>
