Nmap with -sCV runs service detection (-sV) and default scripts (-sC) which fingerprint the software versions and pull metadata like TLS certificate subjects.</p><p>What we found:<br>- Port 22: OpenSSH 8.2p1 - SSH. Noted for later (potential login vector).<br>- Port 80: nginx 1.18.0 - A web server. First target.<br>- Port 443: nginx 1.18.0 with TLS - The certificate's commonName is passbolt.bolt.htb.</p><figure><img src="/writing/media/8205b279eb560b08db4205b8.webp" alt="HackTheBox “Bolt” Walkthrough, figure 2" width="1200" height="542" loading="lazy" decoding="async"></figure><p>The TLS certificate is valuable. It reveals a virtual hostname (passbolt.bolt.htb) that wouldn't be visible otherwise. Attackers Check TLS certs for subdomains.</p><h3 id="step-2-add-virtual-hosts-to-etc-hosts">Step 2: Add Virtual Hosts to /etc/hosts</h3><p>Command:<br>echo "10.129.15.45 bolt.htb demo.bolt.htb mail.bolt.htb passbolt.bolt.htb" | sudo tee -a /etc/hosts</p><p>Why: The web server uses virtual hosting - it serves different sites depending on what Host: header you send. Without adding these to /etc/hosts, your browser and tools will send requests to the IP directly and get the wrong (or no) response. This maps the domain names to the target IP locally.</p><figure><img src="/writing/media/79cc6764ff4f1a1ba96204ad.webp" alt="HackTheBox “Bolt” Walkthrough, figure 3" width="1200" height="148" loading="lazy" decoding="async"></figure><p>Check for virtual hosts on web servers. One IP can host many sites, each with different vulnerabilities. The nmap cert leak gave us passbolt.bolt.htb for free; we guessed bolt.htb, demo.bolt.htb, and mail.bolt.htb from context clues in the writeup and the app itself.</p><h2 id="phase-2-web-enumeration">PHASE 2: WEB ENUMERATION</h2><h3 id="step-3-explore-bolt-htb">Step 3: Explore bolt.htb</h3><figure><img src="/writing/media/315682fc756ef885dfdbeac7.webp" alt="HackTheBox “Bolt” Walkthrough, figure 4" width="1200" height="1046" loading="lazy" decoding="async"></figure><p>Visiting <span class="article-lab-address">http://bolt.htb/<span> (lab-only address)</span></span> shows a web design company landing page. Marketing fluff hides breadcrumbs. The internal chat about "scrubbing the Docker image" is a direct hint that the Docker image contains sensitive data.</p><h3 id="step-4-download-the-docker-image">Step 4: Download the Docker Image</h3><figure><img src="/writing/media/f67b897d4f1d81825c00839f.webp" alt="HackTheBox “Bolt” Walkthrough, figure 5" width="1200" height="769" loading="lazy" decoding="async"></figure><p>Command:<br>wget <span class="article-lab-address">http://bolt.htb/uploads/image.tar<span> (lab-only address)</span></span> -O image.tar</p><p>Why: The site offers a Docker image for download. Docker images are layered filesystems - each layer is a snapshot of file changes. Old layers often contain deleted files, credentials, source code, and database files that were "cleaned up" from the final image but still exist in earlier ones.</p><figure><img src="/writing/media/5564b818210761f3c7aadfdc.webp" alt="HackTheBox “Bolt” Walkthrough, figure 6" width="1200" height="498" loading="lazy" decoding="async"></figure><p>Whenever you see a Docker image offered publicly, download and inspect it.
