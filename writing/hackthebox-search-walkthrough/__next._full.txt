<h2 id="1-methodology-and-hacker-mindset">1. Methodology &amp; Hacker Mindset</h2>
<h3 id="the-penetration-testing-methodology">The Penetration Testing Methodology</h3>
<p>Before typing a single command, understand the <strong>mindset</strong>:</p>
<blockquote>
<p><strong>"Every system has a weakest link. Your job is not to attack the strongest wall, but to find the crack in the foundation."</strong></p>
</blockquote>
<p>Penetration testing follows a cycle:</p>
<ol>
<li><strong>Reconnaissance</strong> — Gather information without touching the target</li>
<li><strong>Enumeration</strong> — Actively query services to understand the attack surface</li>
<li><strong>Exploitation</strong> — Use discovered weaknesses to gain access</li>
<li><strong>Post-Exploitation</strong> — Escalate privileges and achieve objectives</li>
<li><strong>Lateral Movement</strong> — Pivot through the network</li>
<li><strong>Reporting</strong> — Document findings</li>
</ol>
<h3 id="why-windows-active-directory">Why Windows Active Directory?</h3>
<p>Active Directory (AD) is the most common enterprise identity system. It is also:</p>
<ul>
<li><strong>Complex</strong>: Hundreds of permissions, trusts, and configurations</li>
<li><strong>Misconfigured by default</strong>: Many organizations deploy AD without hardening</li>
<li><strong>A trust graph</strong>: One weak user can lead to Domain Admin through permission chains</li>
</ul>
<p>The hacker mindset for AD is:</p>
<blockquote>
<p><strong>"Don't look for the one vulnerability. Look for the PATH — the chain of small misconfigurations that leads from a regular user to Domain Admin."</strong></p>
</blockquote>
<hr />
<h2 id="2-initial-enumeration-reconnaissance">2. Initial Enumeration: Reconnaissance</h2>
<h3 id="2-1-why-start-with-nmap">2.1 Why Start with Nmap?</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">echo "10.129.229.57 search.htb" | sudo tee -a /etc/hosts
sudo nmap -sC -sV -p- --min-rate=10000 10.129.229.57 -oN nmap_all.txt
</code></pre>
<p><strong>What this does:</strong></p>
<ul>
<li><code>-sC</code>: Runs default NSE scripts (service detection, banner grabbing)</li>
<li><code>-sV</code>: Probes service versions</li>
<li><code>-p-</code>: Scans ALL 65,535 TCP ports (not just the top 1,000)</li>
<li><code>--min-rate=10000</code>: Sends packets fast to speed up the scan</li>
<li><code>-oN</code>: Saves output to a file for reference</li>
</ul>
<p><strong>Why add to <code>/etc/hosts</code>?</strong></p>
<blockquote>
<p>Many Windows services (especially Kerberos, LDAP, and IIS with virtual hosts) require the <strong>fully qualified domain name (FQDN)</strong> to respond correctly. If you query <code>search.htb</code> without DNS resolution, the domain controller won't recognize the request as targeting the domain.</p>
</blockquote>
<p><strong>Hacker Mindset:</strong></p>
<blockquote>
<p><strong>"Before I attack, I must understand what I'm looking at. The ports are the doors to the building — I need to know which ones are open, what's behind them, and what version they run."</strong></p>
</blockquote>

<h2 id="4-username-enumeration-with-kerbrute">4. Username Enumeration with Kerbrute</h2>
<h3 id="4-1-what-is-kerbrute">4.1 What is Kerbrute?</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">cat &gt; users.txt &lt;&lt; 'EOF'
Dax.Santiago
Keely.Lyons
Sierra.Frye
Kyla.Stewart
Kaiara.Spencer
Dave.Simpson
Ben.Thompson
Chris.Stewart
Hope.Sharp
EOF

kerbrute userenum users.txt -d search.htb --dc 10.129.229.57
</code></pre>
<p><strong>What Kerbrute does:</strong>
Kerbrute exploits a <strong>Kerberos protocol behavior</strong>: when you send an Authentication Service Request (AS-REQ) with a valid username, the Key Distribution Center (KDC) responds differently than for an invalid username.</p>
<ul>
<li><strong>Valid username</strong>: <code>KDC_ERR_PREAUTH_REQUIRED</code> (error code 25)</li>
<li><strong>Invalid username</strong>: <code>KDC_ERR_C_PRINCIPAL_UNKNOWN</code> (error code 6)</li>
</ul>
<p><strong>Why enumerate users?</strong></p>
<blockquote>
<p>You cannot attack what you don't know exists. A list of valid usernames is the foundation of password spraying, Kerberoasting, and AS-REP roasting.</p>
</blockquote>
<p><strong>Hacker Mindset:</strong></p>
<blockquote>
<p><strong>"Kerberos is a chatty protocol. It tells you whether a user exists before you even try a password. This is gold — I can build a precise target list instead of guessing in the dark."</strong></p>
</blockquote>
<h3 id="4-2-interpreting-results">4.2 Interpreting Results</h3>
<p><strong>Output:</strong></p>
<pre><code>[+] VALID USERNAME:  Sierra.Frye@search.htb
[+] VALID USERNAME:  Keely.Lyons@search.htb
[+] VALID USERNAME:  Dax.Santiago@search.htb
[+] VALID USERNAME:  Hope.Sharp@search.htb
</code></pre>
<p>We now have <strong>4 confirmed valid accounts</strong>.</p>

<h2 id="6-kerberoasting-harvesting-service-account-hashes">6. Kerberoasting: Harvesting Service Account Hashes</h2>
<h3 id="6-1-what-is-kerberoasting">6.1 What is Kerberoasting?</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo ntpdate 10.129.229.57
GetUserSPNs.py -request -dc-ip 10.129.229.57 search.htb/Hope.Sharp:'IsolationIsKey?'
</code></pre>
<p><strong>What this does:</strong></p>
<ol>
<li><code>ntpdate</code> synchronizes your clock with the domain controller (Kerberos is extremely time-sensitive — clock skew &gt; 5 minutes causes failures)</li>
<li><code>GetUserSPNs.py</code> queries LDAP for accounts with <strong>Service Principal Names (SPNs)</strong></li>
<li>For each SPN found, it requests a <strong>Ticket Granting Service (TGS)</strong> ticket</li>
<li>The TGS ticket is encrypted with the service account's password hash</li>
<li>We save this ticket offline and crack it</li>
</ol>
<p><strong>Why Kerberoasting?</strong></p>
<blockquote>
<p>Service accounts often have:</p>
</blockquote>
<ul>
<li>Weak passwords (because they never log in interactively)</li>
<li>No password rotation policy</li>
<li>High privileges</li>
</ul>
<p><strong>Hacker Mindset:</strong></p>
<blockquote>
<p><strong>"I'm not attacking the service directly. I'm asking the domain politely, 'May I have a ticket for this service?' The domain gives me a ticket encrypted with the service's password. I take it home and crack it at my leisure. The service never knows I attacked it."</strong></p>
</blockquote>
<h3 id="6-2-the-hash">6.2 The Hash</h3>
<p><strong>Output:</strong></p>
<pre><code>ServicePrincipalName               Name     MemberOf  PasswordLastSet             LastLogon  Delegation
---------------------------------  -------  --------  --------------------------  ---------  ----------
RESEARCH/web_svc.search.htb:60001  web_svc            2020-04-09 08:59:11.329031  &lt;never&gt;

[redacted captured Kerberos material]
</code></pre>
<p><strong>Why <code>web_svc</code> matters:</strong>
The <code>&lt;never&gt;</code> last logon tells us this is a <strong>service account</strong> that doesn't log in interactively. The SPN <code>RESEARCH/web_svc.search.htb:60001</code> confirms it's used for some web service.</p>

</blockquote>

<blockquote>

<h2 id="12-gmsa-abuse-the-hidden-service-account">12. GMSA Abuse: The Hidden Service Account</h2>
<h3 id="12-1-what-is-a-gmsa">12.1 What is a GMSA?</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">python3 gMSADumper.py -d search.htb -u 'Sierra.Frye' -p '$$49=wide=STRAIGHT=jordan=28$$18'
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>BIR-ADFS-GMSA$:::[redacted retired-lab hash or flag]
BIR-ADFS-GMSA$:aes256-cts-hmac-sha1-96:[redacted encoded lab material]
BIR-ADFS-GMSA$:aes128-cts-hmac-sha1-96:[redacted retired-lab hash or flag]
</code></pre>
<p><strong>What is a GMSA?</strong></p>
<blockquote>
<p><strong>Group Managed Service Account (GMSA)</strong> is a special AD account type where:</p>
</blockquote>
<ul>
<li>The password is REDACTED generated and rotated by Active Directory</li>
<li>The password is <strong>REDACTED bytes</strong> of random data — impossible to crack</li>
<li>Only specific security principals (groups/users) can retrieve the password</li>
</ul>
<p><strong>Why GMSAs are dangerous:</strong></p>
<blockquote>
<p>The password itself is uncrackable, but <strong>who can read it</strong> is what matters. If a regular user (or group they belong to) has <code>ReadGMSAPassword</code> permission, they can retrieve the plaintext-equivalent hash and act as the GMSA.</p>
</blockquote>
<p><strong>Hacker Mindset:</strong></p>
<blockquote>
<p><strong>"The GMSA password is REDACTED fortress — uncrackable, unguessable. But the gatekeeper is just a permission. ITSEC can read the password, Sierra is in ITSEC, so Sierra can read the password. The fortress is irrelevant when the gate is open."</strong></p>
</blockquote>

<h2 id="13-domain-admin-via-genericall">13. Domain Admin via GenericAll</h2>
<h3 id="13-1-what-is-genericall">13.1 What is GenericAll?</h3>
<p><code>GenericAll</code> is the <strong>highest level of access</strong> in Active Directory ACLs. It means:</p>
<ul>
<li>Read all attributes</li>
<li>Write all attributes</li>
<li>Delete the object</li>
<li><strong>Reset the password</strong></li>
<li>Modify group membership</li>
<li>Etc.</li>
</ul>
<h3 id="13-2-resetting-tristan-davies-password">13.2 Resetting Tristan.Davies' Password</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">rpcclient -U 'BIR-ADFS-GMSA$' -W search.htb search.htb --pw-nt-hash
Password: null retired-lab hash or flag]

rpcclient $&gt; setuserinfo2 tristan.davies 23 'qwerty1234!'
</code></pre>
<p><strong>What <code>setuserinfo2</code> does:</strong></p>
<ul>
<li><code>setuserinfo2</code> is an MS-RPC function to modify user attributes</li>
<li><code>23</code> = <code>USER_PASSWORD</code> field (NT password hash / cleartext)</li>
<li><code>'qwerty1234!'</code> = The new password</li>
</ul>
<p><strong>Why does this work?</strong></p>
<blockquote>
<p>Because <code>BIR-ADFS-GMSA$</code> has <code>GenericAll</code> on <code>Tristan.Davies</code>, the GMSA can reset Tristan's password without knowing the old one.</p>
</blockquote>
<p><strong>Alternative methods (from writeups):</strong></p>
<pre><code class="language-powershell" data-language="powershell"># Via PowerShell (if you have a shell)
$cred = New-Object System.Management.Automation.PSCredential("BIR-ADFS-GMSA$", $gmsa_password)
Invoke-Command -ComputerName localhost -Credential $cred -ScriptBlock { net user Tristan.Davies qwerty1234! /domain }
</code></pre>
<p><strong>Hacker Mindset:</strong></p>
<blockquote>
<p><strong>"I didn't hack Tristan.Davies. I hacked the PERMISSION that controls Tristan.Davies. This is the essence of Active Directory attacks — you don't attack the admin, you attack the thing that has power over the admin."</strong></p>
</blockquote>

<h2 id="14-capturing-the-root-flag">14. Capturing the Root Flag</h2>
<h3 id="14-1-verifying-domain-admin-access">14.1 Verifying Domain Admin Access</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">crackmapexec smb 10.129.229.57 -u Tristan.Davies -p REDACTED
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>[+] search.htb\Tristan.Davies:qwerty1234! (Pwn3d!)
</code></pre>
<p><strong>What <code>(Pwn3d!)</code> means:</strong>
This is <code>crackmapexec</code>'s way of confirming the user has <strong>administrative access</strong> to the machine. It automatically tests if it can get a privileged session.</p>
<h3 id="14-2-grabbing-root-txt">14.2 Grabbing root.txt</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">smbclient //search.htb/C$ -U 'Tristan.Davies' -W search.htb
smb: \&gt; cd Users\Administrator\Desktop
smb: \Users\Administrator\Desktop\&gt; get root.txt
</code></pre>
<p><strong>Root Flag:</strong> <code>213fb[redacted]9e29dd3b1</code></p>
<p><strong>Hacker Mindset:</strong></p>
<blockquote>
<p><strong>"The flag is just a file. The real victory is understanding the chain: image → password → user → kerberoast → spray → excel → certificates → bloodhound → GMSA → GenericAll → Domain Admin. Every step was a small crack. Together, they shattered the foundation."</strong></p>
</blockquote>

<h3 id="why-this-machine-is-hard">Why This Machine is "Hard"</h3>
<p>This machine is rated <strong>Hard</strong> not because any single step is impossible, but because:</p>
<ol>
<li>It requires <strong>chaining 10+ steps</strong> together</li>
<li>Each step depends on the previous one</li>
<li>It tests <strong>multiple domains of knowledge</strong>: web, SMB, Kerberos, AD, certificates, PowerShell</li>
<li>The path to Domain Admin is <strong>indirect</strong> — you must follow the permission chain</li>
</ol>
<h3 id="the-most-important-lesson">The Most Important Lesson</h3>
<blockquote>
<p><strong>"Active Directory security is not about finding the ONE vulnerability. It's about understanding the GRAPH of trust, permissions, and group memberships. A chain of three 'low' vulnerabilities beats one 'critical' vulnerability when the chain leads to Domain Admin."</strong></p>
</blockquote>
<h3 id="defensive-recommendations">Defensive Recommendations</h3>
<ol>
<li><strong>Never put credentials in images or public documents</strong></li>
<li><strong>Enforce unique passwords</strong> for service accounts (no reuse)</li>
<li><strong>Monitor for Kerberoasting</strong> — alerts on TGS requests from non-service accounts</li>
<li><strong>Review BloodHound regularly</strong> — know your shortest paths to Domain Admin</li>
<li><strong>Audit GMSA permissions</strong> — limit who can read managed passwords</li>
<li><strong>Remove GenericAll</strong> where possible — use more granular permissions</li>
<li><strong>Rotate leaked credentials immediately</strong> when phishing simulations end</li>
<li><strong>Encrypt and protect backup certificates</strong> — don't store them in user folders</li>
</ol>
